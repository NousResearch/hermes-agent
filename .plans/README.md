## Findings considered and rejected

These came up during the audit but were rejected per the hermes-agent
AGENTS.md § "Contribution Rubric" rules. Recorded here so a future run
does not re-audit them.

- **Refactor `apps/desktop/electron/main.cjs` (7,614 lines) into
  smaller modules.** Reason: **speculative infrastructure**. AGENTS.md
  explicitly rejects "hooks, callbacks, or extension points with no
  concrete consumer." No open issue / no TODO marker / no `Extract me`
  signal exists in this file; refactoring on speculation would violate
  the project's own contribution rubric. Revisit only if a concrete
  consumer surfaces (e.g. a third-party plugin that needs to register
  an IPC channel from outside the desktop package).
- **`tui_gateway/entry.py` docstring-vs-import-order inconsistency.**
  The file's docstring (lines 3-9) says `hermes_bootstrap` must be
  the very first import, yet `import os` and `import sys` come before
  it on lines 3-4. Reason: **insufficient impact**. `os` and `sys`
  do not produce stdout output during import on any supported Python
  version; the documented constraint is forward-compatibility guidance,
  not a current bug. Tracked as a low-impact DX nit; not worth a
  separate plan.
- **`main.cjs:5510 spawn('hermes', ...)` uses generic shell.** Reviewed
  the surrounding code; the spawn uses an argv-array form (not
  `shell: true`), so arguments cannot be injected. By-design.
- **64 separate `.cjs` files in `apps/desktop/electron/`**. By-design
  per `vercel/eve`'s filesystem-first convention — each module has a
  single responsibility, identifiable by path. Not split-worthy.

## Why this PoC was scoped narrowly

The user asked specifically for two scopes: (a) `hermes-agent/venv/Scripts/`
(which we read broadly as "the Python runtime entry-points") and (b) `desktop
exe startup scripts` (which we read as `apps/desktop/electron/`). The full
hermes-agent monorepo has ~3500 files across dozens of packages; a `deep`
audit would have taken many hours and broken the "5-minute" expectation.
This PoC demonstrates the workflow end-to-end at `standard` effort — a
real audit with real findings, real evidence, and a real self-contained
plan. Scaling up to `deep`/monorepo is straightforward: re-run
`/improve deep --issues` after this PR lands.
