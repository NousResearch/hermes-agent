🔴 CRITICAL (2 HIGH-SEVERITY ISSUES)
────────────────────────────────────

1. PII LEAKAGE IN AUTHORIZATION LOGS
   File: gateway/run.py, Line 1800
   Issue: User IDs and names (phone numbers, emails) logged in plaintext
   
   Code:
   ```
   logger.warning("Unauthorized user: %s (%s) on %s", 
                  source.user_id, source.user_name, source.platform.value)
   ```
   
   Impact: GDPR/CCPA violation, PII enumeration attack, harassment risk
   Example Leak: "Unauthorized user: +1234567890 (john.doe) on telegram"
   
   Fix: Hash user_id before logging using _hash_sender_id() from session.py
   Effort: 5 minutes
   Risk if Not Fixed: HIGH - regulatory violation, data breach

────────────────────────────────────────────────────────────────────────────────

2. MISSING PLATFORM-WIDE RATE LIMITING ON UNAUTHORIZED MESSAGES
   File: gateway/run.py, Lines 1799-1832
   Issue: No rate limiting on incoming unauthorized messages (DoS vulnerability)
   
   Current Code Flow:
   - Line 1807: Check if THIS user is rate limited (per-user, 10 min)
   - Line 1809: Generate pairing code if rate limit passed
   - MISSING: No check to prevent multiple NEW users from hammering the gateway
   
   Attack Scenario:
   ```
   Alice sends message → generates pairing code (user A, not rate limited)
   Bob sends message   → generates pairing code (user B, not rate limited)  
   Charlie sends msg   → generates pairing code (user C, not rate limited)
   ...attacker controls 100 SIM cards...
   ```
   
   Impact: DoS via repeated pairing code generation, log spam, resource exhaustion
   
   Mitigation: MAX_PENDING_PER_PLATFORM = 3 (line 43 in pairing.py) partially blocks
   But still allows 3 concurrent unauthorized users * multiple platforms = issue
   
   Fix: Add platform-wide rate limit (10 unauthorized msgs/min max)
   Effort: 30 minutes
   Risk if Not Fixed: MEDIUM - DoS possible but capped by pairing store limits

3. UNSAFE JSON DESERIALIZATION IN WEBHOOK DYNAMIC ROUTES
   File: gateway/platforms/webhook.py, Line 249
   Issue: No validation of route names in webhook_subscriptions.json
   
   Code Path:
   - Agent writes webhook_subscriptions.json with arbitrary route names
   - _reload_dynamic_routes() loads without validation
   - No length checks, no character validation, no path validation
   
   Potential Issue (if file is malicious):
   ```json
   {
     "../../etc/passwd": {...},
     "\x00injection": {...},
     "a".repeat(100000): {...}
   }
   ```
   
   Fix: Add route name validation (alphanumeric, max 255 chars)
   Effort: 15 minutes
   Risk: LOW-MEDIUM (requires agent compromise or file tampering)

────────────────────────────────────────────────────────────────────────────────

4. WEBHOOK SIGNATURE VALIDATION - INCOMPLETE COVERAGE
   File: gateway/platforms/webhook.py, Lines 473-502
   Issue: No fallback for unknown signature header formats
   
   Supported Headers:
   - GitHub: X-Hub-Signature-256 ✓
   - GitLab: X-Gitlab-Token ✓
   - Generic: X-Webhook-Signature ✓
   - Others: REJECTED ✓
   
   Risk: If a webhook provider uses non-standard header, it's rejected
   Status: CORRECT BEHAVIOR (fail-safe, not fail-open)
   
   Minor Issue: No logging of which validation method succeeded
   Fix: Add debug log showing which signature method validated
   Effort: 5 minutes
   Risk: LOW (system is secure, just not logging properly)

────────────────────────────────────────────────────────────────────────────────

5. PAIRING RATE LIMITING - PER-USER ONLY
   File: gateway/pairing.py, Lines 250-255
   Issue: Rate limit is per-user-per-platform, not global
   
   Current:
   ```
   key = f"{platform}:{user_id}"  # Different key per user
   ```
   
   Attack: 1000 different Telegram numbers in 10 minutes = 1000 rate limit keys
   
   Mitigation: MAX_PENDING_PER_PLATFORM = 3 (line 43)
   Status: PARTIALLY MITIGATED (max 3 pending codes cap total damage)
   
   Real Risk: LOW due to the 3-code cap
   Fix: Optional - add global rate limit anyway for defense-in-depth
   Effort: 20 minutes
   Risk: LOW

────────────────────────────────────────────────────────────────────────────────

7. FALLBACK FORM-ENCODED WEBHOOK PARSING
   File: gateway/platforms/webhook.py, Lines 317-330
   Issue: Webhook handler accepts both JSON and form-encoded data
   
   Code:
   ```
   try:
       payload = json.loads(raw_body)
   except json.JSONDecodeError:
       payload = urllib.parse.parse_qsl(...)  # Fallback to form encoding
   ```
   
   Risk: Parsing multiple formats = larger attack surface
   Status: WORKING CORRECTLY (multiple parsers can be risky)
   Fix: Accept JSON only, reject form-encoded
   Effort: 5 minutes
   Risk: LOW (historical fallback, probably not needed)

────────────────────────────────────────────────────────────────────────────────
