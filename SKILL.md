# Project Manager v2.1 - 项目上下文管理系统

## 架构原则：Memory vs File
**Agent 是无状态的**，必须依靠文件系统来维持连续性。
*   **全局 Memory（`memory`）**：仅用于存储 **用户偏好**（如"不喜欢废话"）和 **跨项目的通用经验教训**。**严禁**存储具体项目的进度细节（如"CMRA 方案写到了 v3"），否则会导致记忆爆炸和上下文混乱。
*   **项目状态文件（`STATUS.md`）**：用于存储 **项目进度**。每个项目拥有独立的 `STATUS.md`，互不干扰。
*   **项目索引（`projects/index.md`）**：维护所有项目的清单，支持模糊查找和快速恢复。**首次使用时如不存在，自动创建并写入表头。**
*   **恢复机制**：当用户提到某个项目时，Agent 必须 **强制读取** 该项目的 `STATUS.md`，而不是依赖对话记忆。

## 与 Auto-Coding v3 集成
当使用 `auto-coding-v3` 进行代码开发时，该系统会在 `workspace/projects/{project_name}/status/` 下生成详细的工程状态文件（如 `design_decisions.md`, `test_results.md`）。
*   `STATUS.md` 负责 **宏观进度**（Todo, Next Step, Decision）。
*   `status/` 目录负责 **微观工程细节**（Scratchpad, Logs）。
*   两者结合，确保代码任务也能通过 Project Manager 进行断点续传。

## 核心原则

1. **物理隔离**：每个项目拥有独立的 `STATUS.md`，绝不混用。
2. **状态显式化**：Agent 没有记忆，所有"记忆"必须写入文件。没有记录在文件里的事，就是没发生。
3. **全局 Memory 瘦身**：项目进度只存在于 `STATUS.md`。
4. **分级管理**：不是所有任务都配得上完整 STATUS.md，轻重有别。
5. **意图优先**：不依赖死板的关键词，根据对话上下文自动判断项目操作意图。
6. **乐观写入 (Optimistic Write)**：写 STATUS.md 前，**必须再次读取** 文件确认无其他 Session 刚刚修改过。
   - 如果发现文件时间戳变了（被别人抢先写了），**必须重新读取并合并**，绝不盲目覆盖。

## 目录结构规范

所有项目文件存放在 `workspace/projects/` 下：

```text
workspace/projects/
├── index.md              # 📇 项目索引（自动维护）
├── {project-name}/
│   ├── STATUS.md         # 🧠 核心记忆：进度、待办、关键决策（每次对话必读/必写）
│   ├── docs/             # 方案、报告、调研数据
│   ├── src/              # 代码、产出物
│   └── status/           # auto-coding v3 微观工程文件（可选）
└── _archive/             # 📦 已归档项目（已完成/已废弃）
    └── {project-name}/
        └── STATUS.md     # 保留历史记录
```

## 📇 项目索引：`projects/index.md`

```markdown
# 项目索引
> 自动维护，按最后活跃时间倒序

| 项目 | 目录 | 最后更新 | 状态 | 备注 |
|------|------|---------|------|------|
| english-reader | english-reader/ | 2026-04-20 | 活跃中 | 英语学习工具 |
| cmra-geo | cmra-geo/ | 2026-04-18 | 已暂停 | 等需求确认 |
```

**维护规则：**
- **首次使用**：如 `index.md` 不存在，自动创建并写入表头
- **自愈机制**：如果发现项目目录存在但 index.md 中没有，自动补录
- 新建项目时：自动添加到 index.md 顶部
- 恢复项目时：更新"最后更新"时间，将该项目行移到顶部
- 暂停项目时：更新"状态"为"已暂停"并记录暂停原因
- 归档项目时：将目录移入 `_archive/`，从 index.md 移除或标记为"已归档"并移到底部

## 分级 STATUS.md 模板

根据项目复杂度选择模板，**不要杀鸡用牛刀**。
详细模板请参考：`references/templates.md`

### 模板选择规则
| 场景 | 模板 |
|------|------|
| 3天以上的多阶段项目 | A 完整模板 |
| 单任务（如"帮我把这个接口重构一下"） | B 轻量模板 |
| 调研/分析类临时任务 | B 轻量模板 |
| 需要多人协作或长期维护 | A 完整模板 |
| 不确定 | 先用 B，后续发现需要再升级到 A |

## 操作工作流 (Workflow)

### 1. 意图识别（自动激活）

**不依赖死板关键词**，根据对话上下文自动判断用户意图：

| 用户意图 | 典型表达 | 触发操作 |
|----------|---------|----------|
| 新建项目 | "新建项目 XX"、"帮我建一个XX"、"开始做XX" | 初始化 |
| 恢复项目 | "回到XX"、"继续XX"、"上次那个XX"、"XX做完了吗" | 恢复上下文 |
| 保存状态 | "先停一下"、"我去忙别的"、"保存状态"、"暂停" | 落盘 |
| 查看项目 | "有哪些项目"、"项目列表"、"看看XX进度" | 展示索引或状态 |
| 切换项目 | "切换到XX"、"不管XX了，先搞YY" | 保存当前 + 恢复目标 |
| 隐含恢复 | "上次说的那个方案"、"接着来" | 自动识别并读取对应 STATUS.md |

**识别优先级**：
1. 明确提到项目名 → 直接匹配
2. 模糊指代（"那个"、"上次"、"接着来"）→ 根据 index.md 中最近活跃的项目推断
3. **无法确定** → **展示最近 2-3 个活跃项目让用户选择**，不要盲目猜测

### 2. 初始化新项目 (Start New)

**输入**：`新建项目 CMRA GEO 方案`

**动作**：
1. 目录名用 kebab-case：`cmra-geo`（不是 `cmra_geo` 或 `CMRA-GEO`）
2. 创建目录 `workspace/projects/cmra-geo/`
3. 根据复杂度选择 A/B 模板，创建 `STATUS.md`
4. **确保 `index.md` 存在**：如不存在自动创建并写入表头
5. 更新 `projects/index.md`（添加到顶部）
6. 检查是否有 git repo，有则记录初始 commit hash
7. **回复用户**："✅ 已创建项目 `cmra-geo`。请告诉我项目目标和第一步要做什么。"

### 3. 恢复旧项目 (Resume)

**输入**：`回到 CMRA 项目` 或隐含恢复

**动作**：
1. **模糊匹配**：用户说的项目名可能不精确，先在 index.md 中查找最接近的项目
2. **强制读取**：`read(workspace/projects/{name}/STATUS.md)`
3. **摘要同步**："📖 已恢复 `cmra-geo` 上下文。当前：[简述进度]。待办：[列出]。继续哪一步？"
4. **更新 index.md**：将该项目移到最后活跃时间顶部
5. **禁止假设**：如果文件不存在或无法匹配，提示用户确认

### 4. 状态更新 (Update State)

**什么时候更新**（满足任一）：
- ✅ 完成了一个子任务
- ✅ 做出了新的技术决策
- ✅ 发现了阻塞性 Bug
- ✅ 用户给出了新的需求或约束

**什么时候不更新**：
- ❌ 只是改了个变量名 / 调整了注释
- ❌ 正在思考方案但还没确定
- ❌ 同一个任务中间尝试了不同方法但最终用的是最初方案

**更新策略**：
- **乐观写入**：写入前**再次 check** 文件状态。若发现被其他 Session 修改，**立即 abort 并重新读取**，合并后再写入。
- 完整模板（A）：更新对应模块，保持简洁
- 轻量模板（B）：更新"当前状态"、"决策"（如有）和"下一步"
- 如果 B 模板项目持续超过 3 天或变得复杂，**自动升级到 A 模板**

### 5. 暂停/切换 (Pause/Switch)

**输入**：`先停一下` / `我去忙别的` / `切换到 XX`

**动作**：
1. 总结当前对话的结论
2. 更新当前项目的 `STATUS.md`：修改"当前状态"，更新"待办"
3. 更新 `index.md`：状态改为"已暂停"，记录暂停原因
4. 如果是切换，恢复目标项目
5. **回复用户**："✅ 状态已保存。下次说'回到 [Name]'即可无缝接续。"

### 6. 项目清理与归档 (Cleanup & Archive)

**心跳期间定期检查**：
- 超过 30 天未活跃的项目 → 标记为"可能归档"，询问用户
- 已完成的项目 → 状态改为"已完成"
- 已废弃的项目 → 状态改为"已废弃"

**归档操作**：
1. 将 `projects/{name}/` 整体移动到 `projects/_archive/{name}/`
2. 保留 STATUS.md 作为历史记录
3. 从 `index.md` 中移除或标记为"已归档"并移到底部
4. 在 index.md 中保留一行记录：`| {name} | _archive/{name}/ | [最后日期] | 已归档 | [归档原因] |`

## 与日常记忆的关系

| 信息类型 | 存放位置 | 示例 |
|---------|---------|------|
| 用户偏好 | `memory` | "不喜欢废话，只要结论" |
| 跨项目经验 | `memory` | "上次用 X 方案踩了 Y 坑" |
| 每日工作流水 | `memory/YYYY-MM-DD.md` | "今天推进了 CMRA 认证模块，详情见 STATUS.md" |
| 项目进度 | `projects/{name}/STATUS.md` | "CMRA：认证模块完成，开始做权限" |
| 技术细节 | `projects/{name}/status/` | 测试日志、设计草稿 |

**原则**：
- 项目进度从 STATUS.md 读，经验教训沉淀到 memory，两者不交叉。
- `memory/YYYY-MM-DD.md` 中的项目记录**只写一句话摘要 + 引用 STATUS.md**，不重复细节。

## Git 关联（可选）

如果项目目录下有 `.git` 目录：
- STATUS.md 头部记录 `Git: [最近 commit hash]`
- 更新状态时自动执行 `git -C projects/{name} log --oneline -1` 获取最新 commit（**注意使用 -C 指定项目路径**）
- 帮助快速定位代码状态

## PR 提交最佳实践

当将 Project Manager Skill 提交到上游仓库时：
- **同时提交 `DESIGN.md`**：描述三层记忆模型的架构图、设计哲学、解决的问题清单和量化效果。维护者更愿意合并有清晰设计文档的 PR。
- **PR 描述包含架构图**：把三层记忆模型的 ASCII 图直接写在 PR Body 里，让 Reviewer 一眼看懂价值。
- **Key Features 描述完整能力**：不要带版本号前缀，列出该 Skill 的全部核心价值。

## 严格约束

1. **Memory 禁令**：绝不要用 `memory` 工具记录项目进度。项目进度只存在于 `STATUS.md`。
2. **读取优先**：回复项目相关问题前，**必须先读取** `STATUS.md`。不要依赖上一轮对话的记忆（因为可能被截断）。
3. **极简原则**：STATUS.md 只记"结论"和"下一步"，不记过程流水账。
4. **模板降级**：小任务用轻量模板，别搞过度工程化。
5. **模糊匹配**：用户说"上次那个"、"接着来"时，主动推断对应项目；**无法确定时展示最近项目让用户选，不要盲目猜测**。
6. **并发安全**：写 STATUS.md 前先 read 最新内容，合并后再 write，防止多 session 覆盖。
7. **index.md 自愈**：首次使用时如不存在自动创建；发现项目目录存在但 index 中没有时，自动补录。
