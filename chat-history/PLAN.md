# Hermes 好友模拟与个人微信接入实施方案

## 1. 目标与边界

### 1.1 目标
- 输入：你与单一好友的 JSON 聊天记录（已放在 `chat history` 目录）。
- 输出：一个在对话风格上高拟真的 Hermes 模式，能够以该好友的语气与你对话。
- 终态：通过个人微信接入层实现自动收发，与你进行“好友模拟”聊天。

### 1.2 边界
- 这是“风格模拟”而非“人格复制”，不能保证 100% 还原真实个人意图。
- 个人微信自动化存在风控与封禁风险，必须在小流量、白名单和可回滚前提下运行。
- 默认仅服务于“你自己与机器人”的对话，不用于对外冒充真人。

---

## 2. 总体架构（推荐先 RAG，后再考虑微调）

```text
JSON 聊天记录
   -> 清洗/脱敏/标准化
   -> 会话切片（按时间窗 + 轮次）
   -> 检索索引（FTS5/BM25，后续可加向量检索）
   -> Hermes 好友模式 Prompt + 检索召回
   -> 回复生成（语气控制 + 安全约束）
   -> 微信桥接（个人微信）
```

建议分 2 个阶段：
1. **阶段 A（MVP）**：关键词/BM25 检索 + 人格提示词 + 微信接入；
2. **阶段 B（增强）**：增加向量检索、自动风格评估、长期记忆压缩。

---

## 3. 数据准备（JSON）

## 3.1 建议统一字段
将原始 JSON 统一为如下结构（可通过转换脚本完成）：

```json
{
  "chat_id": "friend_xxx",
  "timestamp": "2025-01-15T20:13:00+08:00",
  "speaker": "me",
  "text": "吃饭了吗",
  "msg_type": "text",
  "reply_to": null
}
```

最少保留：`timestamp`、`speaker`、`text`。

## 3.2 清洗规则
- 删除空消息、系统提示、撤回提示、媒体占位符（如 `[图片]` 可保留文本标记）。
- 合并连续短句（同一说话人、短时间窗口内）为一个语义片段。
- 去重（防止导出重复）。
- 统一时区与时间格式（ISO8601）。

## 3.3 隐私与脱敏
- 对手机号、地址、身份证、银行卡等做掩码。
- 默认不把原始数据上传外部第三方；尽量本地或自托管向量库。

---

## 4. 会话切片与检索索引

## 4.1 切片策略
- 按“会话间隔 > N 分钟”切会话（建议 N=30）。
- 每个切片保留最近 4~12 轮问答，形成“场景片段”。
- 每个片段附加标签：`topic`（可选）、`mood`（可选）、时间段（白天/深夜）。

## 4.2 检索策略（MVP）
- 首选 SQLite FTS5 或 BM25（实现简单、稳定）。
- 每次用户输入时，召回 top-k 片段（建议 k=3~6）。
- 将召回片段作为“风格示例”，而不是直接照抄答案。

## 4.3 检索策略（增强）
- 增加向量检索（embedding）提升语义召回。
- 混合检索：BM25 + 向量重排，降低跑偏概率。

---

## 5. 人格建模（Prompt 设计）

## 5.1 人格卡（Friend Persona Card）
从历史聊天统计并固化：
- 常用称呼（例如是否会叫你昵称）；
- 句子长度偏好（短句/长句）；
- 常见口头禅、语气词、emoji 使用习惯；
- 回复节奏（直接回答/先寒暄）；
- 禁区（不聊的话题、容易拒绝的请求）。

## 5.2 系统提示词模板（示意）

```text
你正在扮演“我的某位固定好友”的聊天风格助手。
目标：高拟真语气、保持自然、避免过度表演。

约束：
1) 不要声称自己是真人，只做“风格模拟”。
2) 优先参考检索到的历史片段中的表达方式，不机械复读。
3) 对不确定事实要明确不确定，不胡编细节。
4) 回复长度默认与历史习惯一致，除非我明确要求详细解释。
5) 若触发安全/隐私边界，改为温和拒绝。
```

## 5.3 生成链路
1. 输入消息；
2. 检索相似历史片段；
3. 构造上下文（人格卡 + 召回片段 + 当前消息）；
4. 生成候选回复；
5. 规则后处理（敏感词、长度、语气一致性）；
6. 返回最终回复。

---

## 6. 在 Hermes 中的落地建议

## 6.1 新增能力建议
- 增加一个“好友模式”开关（例如 `/friend_mode on|off`）。
- 增加一个聊天检索工具（例如 `friend_chat_retrieve`）：
  - 输入：当前用户消息；
  - 输出：top-k 历史片段（JSON）。
- 将该工具放入可控 toolset，默认只在好友模式启用。

## 6.2 配置建议
- `friend_mode.enabled`: bool
- `friend_mode.identity`: 单一好友 ID
- `friend_mode.max_history_hits`: 默认 4
- `friend_mode.whitelist_user_ids`: 仅允许指定用户触发
- `friend_mode.disclosure`: 是否在首轮提示“这是模拟模式”

## 6.3 运行策略
- 普通模式与好友模式严格分离，避免污染其他会话风格。
- 历史聊天索引按好友 ID 隔离（当前为单好友，也保留扩展性）。

---

## 7. 接入个人微信（高风险环节）

## 7.1 接入方式
- 常见路径是第三方个人微信桥接（如 wechaty 生态）。
- 流程：微信消息 -> 网关回调 -> Hermes 回复 -> 回发微信。

## 7.2 风控建议（必须）
- 只对“你自己的账号 + 白名单联系人”开启自动回复。
- 增加手动急停开关（环境变量或命令）。
- 限速：最短回复间隔（例如 2~5 秒），避免高频触发风控。
- 夜间静默时段可配置（避免行为异常）。
- 完整日志可追踪（入站、召回片段 ID、出站）。

## 7.3 合规与伦理
- 先获得好友明确知情同意再使用其聊天记录训练/检索。
- 不用于冒充真人对外沟通，不用于诈骗、营销或误导。

---

## 8. 评估与验收

## 8.1 离线评估
- 从历史中抽样 200 条“你发出的问题”作为测试集。
- 指标：
  - 语气相似度（人工打分 1~5）；
  - 事实幻觉率（越低越好）；
  - 违禁回复率（越低越好）；
  - 平均响应时延。

## 8.2 在线灰度
- 先只在本地命令行测试；
- 再小流量接入微信（仅你自己可见）；
- 连续观察稳定性与风控后再扩大使用范围。

---

## 9. 最小可用版本（MVP）执行清单

1. 将 `chat history` 中 JSON 统一结构化并脱敏；
2. 建立 FTS5/BM25 检索索引；
3. 编写“人格卡”与系统提示模板；
4. 在 Hermes 增加“好友模式”与检索工具调用；
5. 加入输出后处理（长度、敏感边界、免责声明）；
6. 本地回放测试（离线样本）；
7. 微信白名单灰度接入；
8. 监控日志 + 一键急停。

---

## 10. 不建议一开始就做的事

- 不建议直接微调整个模型（成本高、迭代慢、可控性差）。
- 不建议直接全量对外开放自动回复（风控与误伤概率高）。
- 不建议跳过日志和审计（问题排查会非常困难）。

---

## 11. 你当前条件下的结论

在“**JSON + 单一好友 + 个人微信**”前提下，最佳路径是：
- **先做 RAG 风格模拟 MVP**（可落地、可控）；
- **再做微信白名单灰度接入**（控制风险）；
- 以“拟真聊天助手”定位，而不是“完全替代真人”定位。
