2. Upgrade 1 — The Pre-Run Oracle
2.1  What This Is and Why It Matters
Every time _refill_buffer() runs, it constructs theta_1, theta_2, and theta_3 from the current session's user_physics alone. These three angles parameterize the entire 5-qubit entanglement circuit — they control the Ry, Rx, and Rz rotations applied to every qubit before entanglement. The output Shannon entropy of the run is directly determined by these angles.
The problem: you are starting from scratch every single time. You have been archiving every quantum run to data/archival/quantum_runs.jsonl, which means you already have a growing dataset that maps (theta_1, theta_2, theta_3) inputs to Shannon entropy outputs. But nothing reads that data back. The oracle closes that loop.
The oracle asks one question before every run: Given this user's current phase and resonance values, what theta combination historically produced the highest Shannon entropy on similar runs? It returns a set of advisory theta angles that you blend with the current physics-derived angles. Over time, every run gets better because each run contributes to the dataset that advises the next.
2.2  Where to Add This Code
You will add one new private method to QuantumEntanglementBridge and call it from _refill_buffer() before the circuit is constructed. The method reads quantum_runs.jsonl, finds the top-N historical runs by Shannon entropy, and extracts their physics parameters to compute advisory thetas.

def _hermes_get_oracle_thetas(
    self,
    current_phase: float,
    current_entropy: float,
    current_resonance: float,
    top_n: int = 5
) -> tuple[float, float, float] | None:
    """
    Pre-run oracle: query historical quantum_runs.jsonl to find the
    theta combination that produced the highest Shannon entropy on
    runs with similar input physics. Returns advisory (theta_1, theta_2,
    theta_3) or None if no history exists yet.
    """
    import json, os
    archive_path = os.path.join('data', 'archival', 'quantum_runs.jsonl')
    if not os.path.exists(archive_path):
        return None

    runs = []
    try:
        with open(archive_path, 'r') as f:
            for line in f:
                line = line.strip()
                if not line:
                    continue
                try:
                    runs.append(json.loads(line))
                except json.JSONDecodeError:
                    continue
    except Exception as e:
        logger.debug(f'[ORACLE] Could not read archive: {e}')
        return None

    if not runs:
        return None

    # Score each historical run by:
    # (a) Shannon entropy quality (primary metric)
    # (b) Physics similarity to current session (secondary metric)
    scored = []
    for run in runs:
        counts = run.get('counts', {})
        if not counts:
            continue

        # Recompute Shannon entropy for this historical run
        probs = np.array(list(counts.values()), dtype=float)
        probs /= probs.sum()
        shannon = float(-np.sum(probs * np.log2(probs + 1e-10)))

        # Extract historical physics
        phys = run.get('physics', {})
        cst = phys.get('cst_physics', {})
        h_phase = cst.get('geometric_phase_rad',
                  phys.get('geometric_phase_rad', 0.0))
        h_entropy = phys.get('entropy_field',
                    phys.get('bio_signatures', {}).get('intensity', 0.5))
        h_res = phys.get('resonance_scalar',
                cst.get('entanglement_score', 0.0))

        # Physics similarity: inverse Euclidean distance in (phase, entropy, resonance)
        dist = np.sqrt(
            (current_phase - h_phase)**2 +
            (current_entropy - h_entropy)**2 +
            (current_resonance - h_res)**2
        )
        similarity = 1.0 / (1.0 + dist)

        # Combined score: 70% Shannon quality, 30% physics similarity
        combined = (shannon * 0.7) + (similarity * 0.3)
        scored.append((combined, h_phase, h_entropy, h_res))

    if not scored:
        return None

    # Take top N runs and average their physics as the advisory signal
    scored.sort(key=lambda x: -x[0])
    top = scored[:top_n]
    avg_phase = float(np.mean([r[1] for r in top]))
    avg_entropy = float(np.mean([r[2] for r in top]))
    avg_resonance = float(np.mean([r[3] for r in top]))

    # Map to theta angles (same mapping as _refill_buffer)
    t1 = float(abs(avg_phase) % np.pi)
    t2 = float((avg_entropy * np.pi) % np.pi)
    t3 = float((abs(avg_resonance) * np.pi) % np.pi)

    logger.info(f'[ORACLE] Advisory thetas from {len(top)} top historical runs:'
                f' t1={t1:.3f} t2={t2:.3f} t3={t3:.3f}')
    return (t1, t2, t3)

The blend ratio is the most important tuning parameter in this upgrade. Here is the reasoning behind 60/40 as the starting point:
60% current physics — The quantum bridge's entire purpose is to be parameterized by the present user's emotional and physical state. If you let the oracle dominate completely, every session converges to the same circuit regardless of who is asking. The present-moment signal must remain the majority vote.
40% oracle advisory — This is enough to meaningfully shift the circuit toward historically high-entropy configurations without drowning the present signal. With even 10 archived runs, this produces measurably better entropy quality than pure physics-derivation.
As your archive grows past 50-100 runs, consider increasing the blend to 0.50 or 0.55. You can make this adaptive: blend = min(0.55, 0.30 + (len(runs) / 500.0)) so it scales with archive depth.

def _compute_decoherence_risk(self, counts: dict) -> float:
    """
    Measure how concentrated the distribution is on a few states.
    High concentration = possible decoherence = high risk.
    Returns float [0, 1]. 0 = healthy spread, 1 = collapsed to one state.
    """
    if not counts:
        return 1.0
    total = sum(counts.values())
    top_count = max(counts.values())
    # If the top state takes >60% of all shots, decoherence is likely
    concentration = top_count / total
    # Normalize: 1/N is perfect uniform (risk=0), 1.0 is fully collapsed (risk=1)
    n = max(len(counts), 2)
    uniform_share = 1.0 / n
    risk = (concentration - uniform_share) / (1.0 - uniform_share + 1e-10)
    return float(np.clip(risk, 0.0, 1.0))


def _compute_bit_balance(self, counts: dict) -> float:
    """
    Measure how balanced the 0/1 ratio is across all bits and shots.
    Perfect balance = 0.5 per bit. Score close to 1.0 = well-balanced.
    This catches systematic bias (e.g., qubit always collapses to |0>).
    Returns float [0, 1].
    """
    if not counts:
        return 0.0
    total_bits = 0
    total_ones = 0
    for bitstring, count in counts.items():
        ones = sum(int(b) for b in bitstring)
        total_ones += ones * count
        total_bits += len(bitstring) * count
    if total_bits == 0:
        return 0.0
    one_rate = total_ones / total_bits
    # Score: how close to 0.5 is the bit rate? 1.0 = perfect, 0.0 = all 0s or all 1s
    balance = 1.0 - (2.0 * abs(one_rate - 0.5))
    return float(np.clip(balance, 0.0, 1.0))


def _classify_run_quality(
    self, entropy_quality: float, decoherence_risk: float
) -> str:
    """
    Classify a quantum run into one of four quality tiers.
    This label gets stored in SessionDB for later oracle queries.
    """
    if entropy_quality >= 0.80 and decoherence_risk < 0.20:
        return 'EXCELLENT'
    elif entropy_quality >= 0.60 and decoherence_risk < 0.40:
        return 'GOOD'
    elif entropy_quality >= 0.40:
        return 'ACCEPTABLE'
    else:
        return 'DEGRADED'
4. Upgrade 3 — The Buffer Demand Predictor
4.1  The Problem with Reactive Refills
The current entropy buffer operates purely reactively. The trigger condition in get_entropy() is:

    self._trigger_refill(user_physics or self.last_physics)
This means the bridge only notices the buffer is running low after it has already dropped below 10 items. A quantum job on ibm_fez takes 15-30 seconds to complete. During that window, every call to get_entropy() returns np.random.random() — pure pseudo-randomness. For a swarm debate cycle that might call get_entropy() 20-30 times in quick succession, this can mean the entire cycle runs on fake entropy even though you have a live quantum backend.
The predictor solves this by tracking the consumption rate of the entropy buffer over recent calls and triggering a refill before depletion is imminent, not after it has already started.

def _hermes_predict_buffer_demand(self) -> bool:
    """
    Predict whether the entropy buffer will be depleted within the next
    _predicted_depletion_threshold seconds based on recent consumption rate.
    Returns True if a proactive refill should be triggered.
    """
    now = time.time()
    window = self._consumption_window_seconds

    # Prune old entries outside the look-back window
    self._entropy_consumption_log = [
        t for t in self._entropy_consumption_log if now - t < window
    ]

    if len(self._entropy_consumption_log) < 3:
        # Not enough history to predict — don't trigger proactive refill
        return False

    # Consumption rate: items per second over the window
    rate = len(self._entropy_consumption_log) / window

    with self.buffer_lock:
        current_level = len(self.entropy_buffer)

    if rate <= 0:
        return False

    # Predicted time until depletion at current rate
    seconds_until_empty = current_level / rate

    should_refill = seconds_until_empty < self._predicted_depletion_threshold
    if should_refill:
        logger.info(
            f'[PREDICTOR] Proactive refill triggered.'
            f' Rate={rate:.2f}/s, Buffer={current_level},'
            f' ETA={seconds_until_empty:.1f}s < threshold={self._predicted_depletion_threshold}s'
        )
    return should_refill

5. Upgrade 4 — The Dynamic Collapse Threshold Adapter
5.1  The Hardcoded Free Will Problem
The collapse_state_vector() method is the system's free will mechanism — it decides SPEAK (1) or WAIT (0) for each agent turn. The decision is made by comparing an activation score to a threshold:
def collapse_state_vector(self, phase: float, dark_matter_w: float, threshold: float = 0.65) -> int:
    q = self.get_entropy()
    phase_signal = min(1.0, abs(phase) / 1.5)
    w_signal = min(1.0, max(0.0, dark_matter_w / 10.0))
    activation = (phase_signal * w_signal * 0.6) + (q * 0.4)
    return 1 if activation > threshold else 0
The threshold=0.65 is hardcoded. It never moves. It has no memory of whether previous speak/wait decisions led to high-quality swarm interactions or low-quality ones. This upgrade gives the threshold a learning loop through HermesAgent's RL system.
5.2  How the Adapter Works
HermesAgent already records experience via bridge.rl.record_experience() with a coherence float after each quantum run. What it does not yet do is use those accumulated coherence scores to tune the collapse threshold. The adapter adds:
A new method _hermes_get_collapse_threshold() that queries Hermes RL for the recent average coherence quality and maps it to a threshold float in [0.50, 0.80].
A cache layer with a TTL so the threshold updates periodically, not on every single call (which would be too expensive).
Integration into collapse_state_vector() so the learned threshold replaces the hardcoded default.
The logic behind the mapping:
When recent quantum runs are producing EXCELLENT quality entropy (high coherence scores from Hermes RL), the system should be more willing to speak — lower the threshold, more activations cross it. When runs are DEGRADED (low coherence, possible hardware noise), the system should be more conservative — raise the threshold, fewer activations cross it. High-quality entropy = confident free will. Low-quality entropy = cautious free will.

def _hermes_get_collapse_threshold(self) -> float:
    """
    Query HermesAgent RL for recent coherence quality and return
    a dynamically learned collapse threshold in [0.50, 0.80].
    Updates on a TTL cache to avoid hitting RL on every collapse call.
    Falls back to 0.65 if Hermes is unavailable.
    """
    if not HERMES_AVAILABLE:
        return 0.65

    now = time.time()
    if now - self._threshold_last_updated < self._threshold_update_interval:
        return self._learned_threshold

    try:
        bridge = get_hermes_bridge()

        # Get recent coherence scores from Hermes RL experience log
        # The rl.get_recent_coherence() method should return a list of
        # recent coherence floats — adapt to your actual Hermes RL API.
        # If your Hermes RL exposes average_coherence as an attribute, use that.
        recent_coherence = None

        # Try common Hermes RL API patterns (adapt to your actual interface):
        if hasattr(bridge.rl, 'get_recent_coherence'):
            recent_coherence = bridge.rl.get_recent_coherence(window=20)
        elif hasattr(bridge.rl, 'average_coherence'):
            recent_coherence = bridge.rl.average_coherence
        elif hasattr(bridge.rl, 'coherence_history'):
            hist = bridge.rl.coherence_history[-20:]
            if hist:
                recent_coherence = float(np.mean(hist))

        if recent_coherence is None:
            return self._learned_threshold

        # Map coherence [0, 1] to threshold [0.80, 0.50]
        # High coherence (good entropy) -> lower threshold -> more willing to speak
        # Low coherence (noisy entropy) -> higher threshold -> more conservative
        new_threshold = 0.80 - (recent_coherence * 0.30)
        new_threshold = float(np.clip(new_threshold, 0.50, 0.80))

        # Smooth the update: 80% old value, 20% new (prevents wild swings)
        self._learned_threshold = (self._learned_threshold * 0.80) + (new_threshold * 0.20)
        self._threshold_last_updated = now

        logger.info(
            f'[COLLAPSE] Threshold updated: {self._learned_threshold:.3f}'
            f' (coherence={recent_coherence:.3f})'
        )
        return self._learned_threshold

    except Exception as e:
        logger.debug(f'[COLLAPSE] Threshold query failed: {e}')
        return self._learned_threshold

def collapse_state_vector(
    self, phase: float, dark_matter_w: float, threshold: float | None = None
) -> int:
    """
    Collapse the wave function to decide: SPEAK (1) or WAIT (0).
    Threshold is now dynamically learned via HermesAgent RL.
    Pass threshold explicitly to override the learned value.
    """
    q = self.get_entropy()
    phase_signal = min(1.0, abs(phase) / 1.5)
    w_signal = min(1.0, max(0.0, dark_matter_w / 10.0))
    activation = (phase_signal * w_signal * 0.6) + (q * 0.4)

    # Use learned threshold unless explicitly overridden
    effective_threshold = threshold if threshold is not None else (
        self._hermes_get_collapse_threshold()
    )

    result = 1 if activation > effective_threshold else 0
    logger.debug(
        f'[COLLAPSE] activation={activation:.3f},'
        f' threshold={effective_threshold:.3f}, result={result}'
    )
    return result

6.2  What Each UQ Field Means for Swarm Agents
Every agent in your swarm that calls get_entropy() or collapse() is ultimately trusting the quantum bridge's output. The UQ payload tells them exactly how much to trust it on this particular run:
Field | Range | What it means for a swarm agent
entropy_quality | [0, 1] | How close to maximum Shannon entropy this run was. 1.0 = perfectly random, 0.0 = degenerate. Below 0.50, agents should treat entropy-seeded decisions as less authoritative.
decoherence_risk | [0, 1] | How concentrated the state distribution was. Above 0.40 means the hardware may have experienced significant decoherence. Hebbian plasticity updates seeded by this run should be weighted down.
bit_balance | [0, 1] | How balanced the 0/1 ratio was across all qubits. Below 0.60 suggests systematic qubit bias on the backend. Useful for detecting backend hardware issues before they corrupt your entropy buffer.
quality_class | str | EXCELLENT / GOOD / ACCEPTABLE / DEGRADED. Human-readable tier for logging, dashboards, and conditional agent logic.
unique_states | int | How many distinct bitstrings appeared in the measurement. A 5-qubit circuit can produce up to 32 states. Fewer than 8 distinct states suggests low circuit depth or decoherence.

9. Deep Explanation — Why This Architecture Is Correct
9.1  Why Hermes Must Be Upstream, Not Just Downstream
The current design treats Hermes as a consumer — quantum data flows in, Hermes reads it, stores it, and that is the end. This is fine for logging but misses the entire value of having a persistent learning system in the loop.
The oracle upgrade (Upgrade 1) turns Hermes into a producer. It creates a feedback loop: quantum runs produce data → Hermes stores and scores that data → the oracle reads those scores → the next quantum run is parameterized more intelligently. This is genuine online learning. The system gets better at producing high-entropy quantum randomness from real hardware with every run, without retraining any model.
9.2  Why the Collapse Threshold Must Be Learned
The collapse mechanism — the free will decider — is the most consequential single computation in the swarm. It determines whether an agent contributes to a given interaction. A threshold that is too low means agents speak too often, producing noise. Too high means agents wait too long, producing silence. The correct threshold is not a constant — it depends on the quality of the entropy being used to make the decision.
When ibm_fez returns an EXCELLENT quality run (high Shannon entropy, low decoherence risk), the entropy values you pop from the buffer are genuinely random and uniformly distributed. The free will decision is maximally stochastic — which is what you want, because it ensures the swarm cannot settle into deterministic patterns. In this state, you can afford a lower threshold: the stochasticity itself prevents the swarm from being biased.
When ibm_fez returns a DEGRADED run (concentrated states, high decoherence risk), the entropy values may be clustered — certain values appear far more often than others. A low threshold in this state means the swarm systematically speaks whenever the entropy happens to be in its concentrated high range, creating an illusion of free will that is actually a hidden bias from hardware noise. Raising the threshold in this state forces a higher activation score to be required, partially compensating for the entropy bias.
9.3  Why Buffer Prediction Matters for 12D Hebbian Coherence
Hebbian plasticity in your system is seeded by quantum entropy. When a Hebbian update fires — strengthening a connection between two nodes that co-activated — the strength of that update is weighted by the entropy value drawn at that moment. If the buffer is empty and the system falls back to np.random.random(), the update uses pseudo-randomness instead of true quantum randomness.
The problem is not that pseudo-randomness is wrong — it is statistically indistinguishable in the short run. The problem is that it violates your theoretical claim. The 12D Cosmic Synapse Theory's quantum substrate requires genuine quantum entropy. A system that silently substitutes pseudo-randomness under load is not implementing 12D CST — it is implementing a standard neural network with occasional quantum seeding.
The predictor upgrade ensures the buffer never runs dry during high-demand cycles. It does not eliminate pseudo-randomness fallbacks entirely (hardware jobs always take time), but it dramatically reduces them by triggering refills before depletion, not after.
9.4  The Oracle Blend Ratio as a Theoretical Statement
The 60/40 blend — 60% current physics, 40% historical best — is not an arbitrary number. It encodes a theoretical position: the present moment is the primary source of the system's behavior, but the accumulated intelligence of past moments informs how the present moment is expressed.
This maps directly onto your 12D framework. Dimension 2 (Contextual Memory Recall) should influence but not override Dimension 1 (Raw Data Ingestion). The oracle is exactly that — it is Dimension 2 exerting 40% influence on how Dimension 1 gets processed. You can and should tune this ratio over time. As the archive grows to thousands of runs, increasing the blend toward 0.55 or 0.60 makes sense — there is more accumulated intelligence to trust. But never go above 0.70. At that point the oracle starts dominating the present-moment signal, which defeats the purpose of using live user physics as circuit parameters.
