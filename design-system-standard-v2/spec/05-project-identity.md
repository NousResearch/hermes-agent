# 05 · Project Identity & Direction — ชั้นบริบทโครงการ (ทำก่อนทุกชั้น)

> เพิ่มตามคำสั่งเจ้าของ 2026-07-05: "AI ต้องรู้ภาพรวมทั้งหมด ว่า direction ไปทิศทางเดียวกันหรือเปล่า"
> หลักการ: **ทุกโปรเจกต์มีแผนธุรกิจ/แผนการตลาด/roadmap อยู่แล้ว — DS ต้องอ่านของพวกนั้นเป็น input ไม่ใช่เริ่มจากศูนย์**
> พิสูจน์ความจำเป็นจากเคสจริง: OneManFleet มี BRAND_SOURCE_OF_TRUTH.md (canonical) ระบุสี `#1A1A2E`/`#E94560`
> แต่โค้ด tailwind.config ใช้ `#231F20`/`#851914` — **แหล่งความจริงขัดกันเองโดยไม่มีใครจับ** · ชั้นนี้มีไว้จับสิ่งนี้

## H0 · Input บังคับ — เอกสารโปรเจกต์ที่ AI ต้องหาอ่านก่อน (ตามลำดับ)
1. ไฟล์แบรนด์ canonical: `BRAND_SOURCE_OF_TRUTH.md` / `BRAND_CI.md` / `.project/BusinessPlan.md`
2. แผนธุรกิจ: `BUSINESS_PLAN.md` / ผล `Use Business Plan`
3. แผนการตลาด: `MARKETING_FLOW.md` / content pillars / funnel
4. แผนงาน: `PROJECT_PLAN.md` / roadmap / เฟสปัจจุบัน (MVP หรือ scale — กำหนดว่าลงลึกแค่ไหน)
- ไม่พบ = ถามเจ้าของ 4 คำถามสั้น (พันธกิจ · กลุ่มเป้าหมาย · ปัญหาหลัก · เฟสงาน) ห้ามมโน
- **พบหลายไฟล์ขัดกัน = หยุด รายงาน conflict ให้เจ้าของตัดสิน ห้ามเลือกเอง**

## H1 · บัตรประจำตัวโครงการ (Project Identity Card — ส่วนหัวของ DS ทุกโปรเจกต์)
| ช่อง | ตัวอย่างจริง (OneManFleet) |
|---|---|
| ชื่อโครงการ/แบรนด์ | One Man Fleet (ล็อกแล้ว · ห้าม rename) |
| บริษัท/เจ้าของ | (ตาม privacy rule — แบรนด์ faceless ไม่เปิดชื่อบริษัทจริง) |
| วิสัยทัศน์/positioning | "เปลี่ยน AI จากของเล่น ให้เป็นระบบงานจริง มี agent มี QC มีหลักฐาน ทำซ้ำได้" |
| พันธกิจ/สิ่งที่ทำ | ห้องทดลองระบบ AI สำหรับคนทำธุรกิจจริง · flagship = ViberQC |
| ภาษาหลัก | English-first (ไทย secondary) |
| เฟสปัจจุบัน | Redesign — เป้า Linear/Stripe/MasterClass |

## H2 · เหตุผลการเลือกสี (Color Rationale — บังคับทุกสี ห้ามมีสีที่ตอบไม่ได้ว่า "ทำไม")
ทุกสีในพาเลตต์ต้องมี 3 อย่าง: ความหมาย · เชื่อมกับพันธกิจ/บุคลิกข้อไหน · ใช้ที่ไหน

| สี | ความหมาย | เชื่อมพันธกิจยังไง | ใช้ที่ไหน |
|---|---|---|---|
| (ตัวอย่าง) Deep Navy | ห้องแล็บ · ลึก · ไว้ใจได้ | "ระบบงานจริง มีหลักฐาน" — ความน่าเชื่อถือแบบวิศวกรรม | พื้น dark หลัก |
| (ตัวอย่าง) Bold Red | พลังงาน · จุด action | "ลงมือทดลองจริง" — จุดที่อยากให้คนกด | CTA/เน้น |
| (ตัวอย่าง) Emerald | ผลลัพธ์ · ผ่าน/สำเร็จ | "วัดผลได้" — สีของหลักฐานที่เขียว | สถานะสำเร็จ/ตัวเลขบวก |
- สีที่ไม่มีเหตุผล = ตัดออกหรือส่งกลับให้เจ้าของตัดสิน

## H3 · ปัญหากลุ่มเป้าหมาย → คำตอบเชิงดีไซน์ (Pain → Design Response)
ดึง pain จากแผนธุรกิจจริง แล้วตอบว่า DS ออกแบบอะไรมาแก้:

| Pain (จากแผนจริง OneManFleet) | DS ตอบด้วยอะไร |
|---|---|
| ใช้ AI หลาย tool แต่ไม่มีระบบกลาง | dashboard components เด่น (B9) · แดชบอร์ดคือหน้าแรก ไม่ใช่ blog |
| กลัว automation พังคุมไม่ได้ | trust/proof ชิ้นส่วน (G4) + สถานะ 5 แบบชัด (C2) + สีเตือนแยกชัด |
| อยากได้ report พิสูจน์ผล | ตัวเลข tabular-nums + การ์ด KPI + before/after component |
- ตารางนี้บังคับอย่างน้อย 3 แถวต่อโปรเจกต์ · pain ที่ไม่มีคำตอบดีไซน์ = ช่องว่างที่ต้องรายงาน

## H4 · สองภาษา (TH/EN — บังคับ)
- หน้า DS/showcase ต้องมี**ปุ่มสลับ ไทย/อังกฤษ** · microcopy สำคัญ (ปุ่ม/เมนู/หัวข้อ) มี 2 ภาษาคู่กันใน token/dictionary
- ภาษานำของแต่ละโปรเจกต์มาจาก H1 (เช่น OneManFleet = EN นำ · งานราชการ = ไทยนำ)
- ฟอนต์ต้องรองรับคู่: ไทย (Noto Sans Thai/IBM Plex Sans Thai) + ละติน จับคู่ระยะให้เนียน (กฎ C9)

## H5 · Direction Check (ด่านตรวจทิศก่อนปิดงานดีไซน์ทุกครั้ง)
ตอบ 4 คำถาม ถ้าข้อไหน "ไม่" = ยังไม่ผ่าน:
1. งานชิ้นนี้สะท้อนวิสัยทัศน์/positioning ใน H1 ไหม (คนดูรู้สึกถึงสิ่งที่แบรนด์อยากเป็นไหม)
2. สีที่ใช้ทุกสีตอบ H2 ได้ไหม (ไม่มีสีหลุดโผล่มาโดยไม่มีเหตุผล)
3. องค์ประกอบหลักของหน้า ตอบ pain ใน H3 อย่างน้อย 1 ข้อไหม
4. ภาษา/ถ้อยคำตรงกับภาษานำ + voice ของแบรนด์ไหม

## H6 · ความสอดคล้องข้ามเอกสาร (Cross-doc Consistency)
- เทียบ: สีในโค้ด vs ไฟล์แบรนด์ canonical · ชื่อแบรนด์ในโค้ด vs การตัดสินใจล่าสุด · tagline ในหน้าเว็บ vs positioning
- พบขัดกัน = รายงานเป็นตาราง conflict ให้เจ้าของตัดสิน (ห้าม AI เลือกเอง — ตามเคสจริง OneManFleet ที่สีขัดกัน 2 แหล่ง)
