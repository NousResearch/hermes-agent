## 5. แผนงานคืนนี้ — AI ทำเอง ไม่ต้องรอเจ้าของงานตอบ

> กติกาคืนนี้ (เหมือนรอบที่แล้วที่สำเร็จ): ทำเฉพาะงานในเครื่องที่ปลอดภัยและถอยกลับง่าย
> **ไม่แตะ production server · ไม่ commit · ไม่ push · ไม่ deploy** (พวกนี้ต้องให้เจ้าของงานเคาะ)
> ผลงานคืนนี้ = ไฟล์ที่แก้ค้างไว้ + รายงานเช้า ให้เจ้าของงานตรวจแล้วค่อยกด commit

### งานที่ AI จะทำคืนนี้ (เรียงตามลำดับ ทำจากบนลงล่าง)

| ลำดับ | งาน | สายงาน | ทำไมปลอดภัย | เกณฑ์ผ่าน (ตรวจจริง) |
|---|---|---|---|---|
| N1 | เริ่มเฟส 7 ข้อแรก — เขียนเทสต์ครอบ `tools/approval.py` (ตัวกัน P0) ให้แน่นขึ้น | สาย 4 ฟ.7 | เพิ่มเทสต์ใหม่ ไม่แตะโค้ดรันจริง | เทสต์ใหม่รันผ่าน + เทสต์เดิมไม่พัง |
| N2 | เริ่มเฟส 3 — หาจุดช้าในการโหลด config/skills แล้วทำแคชเล็ก ๆ | สาย 4 ฟ.3 | แก้เฉพาะจุด มีเทสต์คุม ถอยกลับได้ | วัดเวลา ก่อน/หลัง + เทสต์ผ่าน |
| N3 | ปิดงานค้างเฟส 1.10/1.12 (แยก command handlers / worktree utils ที่เลื่อนไว้) | สาย 4 ฟ.1 | ย้ายโค้ดแบบมีเทสต์คุม | import ไม่พัง + เทสต์ชุดเดิมผ่าน |
| N4 | กวาดงานเฟส 3–7 ที่ทำในเครื่องได้ ทำเพิ่มเท่าที่เวลาคืนนี้ไหว | สาย 4 | ทุกชิ้นมีเทสต์ + ถอยกลับได้ | แต่ละชิ้นมีหลักฐานตรวจของตัวเอง |
| N5 | เขียนรายงานเช้า สรุปทำอะไรไป + ไฟล์ไหนแก้ + ต้องกดอะไรต่อ | ทุกสาย | เป็นเอกสาร | ไฟล์ `.hermes/ai-relay/overnight-report-<วันที่>.md` |

### งานที่ AI **จะไม่ทำ** คืนนี้ (รอเจ้าของงาน)

| งาน | เหตุผล |
|---|---|
| ตั้ง Cloudflare Access (สาย 1-A) | ต้องบัญชี Cloudflare ของเจ้าของงาน · AI ตั้งเองไม่ได้ |
| ตรวจ role-based credential / monitoring / health แยก (สาย 1 B–E) | ต้อง SSH เข้า VPS แต่กุญแจถูกปฏิเสธ |
| commit + deploy เฟส 013 ขึ้น VPS (สาย 2) | ต้องผ่านด่านอนุมัติของเจ้าของงาน |
| อัป production จาก `287d79eed` → `5498d0487` | แผนพร้อมแล้วใน `production-upgrade-plan-2026-06-21.md` · รอเจ้าของงานเคาะ |

### เช้าพรุ่งนี้เจ้าของงานทำ 3 อย่าง

1. เปิดทาง SSH ให้ AI (หรือบอก "ตรวจเครื่องได้") → AI จะตรวจสาย 1 ข้อ B–E ให้
2. ดูโค้ดที่ AI แก้ค้างคืนนี้ → ถ้าโอเค สั่ง "commit ได้"
3. ตัดสินใจเรื่อง Cloudflare Access — จะเปิดประตูเว็บให้พนักงานเมื่อไหร่
