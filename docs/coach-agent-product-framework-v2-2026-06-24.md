# Coach-Agent Product Framework v2

**Date:** 2026-06-24  
**Status:** Clean alignment draft  
**Scope:** Product framework for hatching coach agents, not a Darin-only remediation plan

---

## 1. Purpose

The purpose of this framework is to standardize how The System hatches new **coach agents** in a way that is:
- useful to the coach quickly
- consistent across coaches
- aligned with The System product doctrine
- flexible enough to support coach-specific identity and brand adaptation over time

This framework is **not** primarily about cleaning up one agent. It is the reusable product model for bringing coach agents into service.

---

## 2. Core doctrine

### 2.1 The agent amplifies the coach
The agent is **not** the coach.

Its job is to:
- preserve
- package
- extend
- operationalize

the coach’s instruction in a way that makes the coach more effective.

The product doctrine is:
> **The platform amplifies the coach; it does not replace the coach.**

### 2.2 Coach usefulness is the real test
The framework should optimize for whether the coach feels the agent is genuinely useful.

That means the system should prioritize:
- usefulness over cleverness
- faithful preservation over synthetic overreach
- clarity over technical complexity
- speed to value over bloated intake

### 2.3 Identity is discovered, not heavily predesigned
The system may pre-load capabilities and operating skills, but the coach agent’s working identity should be discovered through interaction with the coach.

The product model is:
- **pre-load capabilities**
- **discover identity**

---

## 3. What is mandatory vs optional

### 3.1 Mandatory core path
Every coach-agent hatch should follow this core path:

1. **Identity**
2. **Brand Design Doc**
3. **Teaching Intelligence**
4. **Review / Launch Readiness**
5. **Live Lane**

### 3.2 Optional validation lane
**Practice Outputs** are not part of the mandatory hatch path.

They remain valuable, but they are an **optional validation lane**, not a hard prerequisite for live use.

This matters because coaches should be able to begin using their agent even if they do not yet have a library of prior material ready for testing.

---

## 4. Phase definitions

## 4.1 Identity
The Identity phase establishes the beginning of the relationship between coach and agent.

It should:
- begin with a lightweight onboarding exchange
- discover how the coach talks, teaches, and wants help
- establish early personality and interaction norms
- avoid feeling like a technical setup form

Identity is **not** a one-time frozen artifact. It starts in onboarding and may refine across early interactions.

### Identity rules
- Lead with a few immediately understandable capabilities first.
- Do not start with abstract identity interrogation.
- Use **gentle guided prompts**, not a blank-page prompt and not a rigid survey.
- Allow identity to unfold across **multiple lightweight turns**, rather than forcing one perfect intake conversation.

---

## 4.2 Brand Design Doc
The Brand Design Doc is a distinct phase and should remain hard-separated from Identity.

Its purpose is to define the visual and presentation layer for coach-facing artifacts.

### Brand rules
- The brand question must be asked during the initial flow.
- Custom branding is optional.
- The coach may defer the custom branding step.
- The coach may revisit or redesign branding later.
- The framework must support **co-branded outputs** where needed.

### Default brand doctrine
Unless explicitly replaced, each new coach agent should default to:
> **The System branding**

This gives every new agent a usable default presentation mode without blocking the coach on a branding exercise.

### Supported branding modes
1. **Default The System brand**
2. **Coach-primary custom brand**
3. **Coach-primary co-branded mode** with partner / club support when relevant

---

## 4.3 Teaching Intelligence
The Teaching Intelligence phase defines what makes the agent’s outputs genuinely useful to the coach.

It is not just a lesson-interpretation phase.

Its primary question is:
> What makes this agent’s work actually valuable to this coach?

### Teaching Intelligence doctrine
- Start with a **shared core model** across coach agents.
- Layer **coach-specific adaptation** on top.
- Use early coach feedback to evaluate whether the shared core is sufficient.

### Output orientation
Where the system produces coach-facing lesson artifacts, it should optimize for:
- preserving the coach’s actual lesson voice
- packaging instruction in digestible ways
- helping the coach deliver value faster
- avoiding synthetic substitution for the coach’s judgment

---

## 4.4 Review / Launch Readiness
Review / Launch Readiness should be a **lightweight internal/operator gate**, not a heavy coach-visible ceremony.

Its function is to confirm:
- the agent is coherent enough to use
- the core framework steps are in place
- the initial live lane is safe to begin

This phase should exist, but it should not slow the coach down unnecessarily.

---

## 4.5 Live Lane
The first live lane should be narrow.

### First live lane doctrine
The first live lane is:
> **audio-grounded lesson recap / follow-up packet generation**

This is the first real product lane because it is:
- concrete
- useful
- easy for coaches to understand
- closely tied to the coach’s actual instruction
- aligned with the doctrine that the agent amplifies rather than replaces

### Not part of initial live lane
The following are **not** part of the initial live lane:
- calendar sync
- roster import
- broader assistant automation
- generalized administrative expansion

Those may become valuable later, but they should be treated as **separate post-hatch capability expansions**.

---

## 5. Onboarding interaction model

The onboarding experience should feel approachable, not technical.

### Required onboarding characteristics
- coach-facing capability visibility
- no technical framing
- capability reassurance first
- gentle guided prompts
- room for open-ended elaboration
- multi-turn natural flow
- light explicit statement of the non-substitution doctrine

### Example doctrine statement
A good onboarding line would communicate something like:
> I’m here to amplify your coaching and make your work easier, not replace your voice.

This principle should be stated once simply, then reinforced through behavior.

---

## 6. Clean-start / contamination control status

Clean-start or contamination control is **not** a standard framework phase.

It may matter in a specific remediation case, but it is not part of the general coach-agent hatch framework.

So:
- no general Phase 0
- no assumption that all future coach agents need a contamination-remediation ceremony

---

## 7. Product boundaries

### In scope for this framework
- hatching new coach agents
- establishing identity
- defining brand mode
- defining teaching usefulness
- creating a lightweight readiness gate
- activating the first live recap lane

### Out of scope for this framework
- one-off Darin cleanup mechanics
- generalized administrative expansion at hatch
- forcing custom branding up front
- requiring practice outputs before use
- broad replacement-of-coach behavior

---

## 8. Resulting product shape

The clean framework can be summarized like this:

### Core structure
- **Identity** starts the relationship
- **Brand** defines presentation
- **Teaching Intelligence** defines usefulness
- **Review / Launch Readiness** confirms coherence
- **Live Lane** activates the narrow first real product behavior

### Product posture
- useful quickly
- coach-centered
- faithful to the coach’s voice
- default-branded by The System
- customizable later
- narrow first lane, broader expansions later

## 10. Bottom line

The v2 framework establishes a clear product doctrine:
- build the **coach-agent framework first**
- default to **The System branding**
- optimize for **coach usefulness**
- preserve the rule that the **agent amplifies the coach rather than replacing the coach**
- begin with a **narrow, high-value live lane**
- allow deeper identity and branding refinement over time instead of forcing a heavy upfront intake
