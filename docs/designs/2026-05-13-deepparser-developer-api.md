## Vision

### 10x Check
DeepParser becomes the standard parsing substrate for AI agents working with
enterprise documents — the way S3 became the standard for blob storage. Every
developer building a legal, financial, or construction AI tool reaches for
`pip install deepparser` first because it's the only library that handles
DWG, Excel-embedded PDFs, and scanned tables without falling back to OCR soup.
GitHub stars, HN posts, and citation in LLM tool benchmarks become the growth
driver. The hosted API funds the infra; the Docker image enables privacy-sensitive
enterprise adoption; the benchmark is the moat's proof.

### Platform Potential
The Python SDK + Docker image combination makes DeepParser consumable by any
agent system — Hermes, LangChain, LlamaIndex, custom agents. This means
DeepParser's reach extends beyond the Hermes ecosystem without requiring
integration work.

## Architecture Decisions (Section 1)

### Parse endpoint: async job polling
POST /parse → `{job_id}`. GET /parse/{job_id} → status + result.
Synchronous blocking rejected: ties a worker thread per active parse job, kills concurrency.

Parse job state machine:
```
UPLOAD ──▶ QUEUED ──▶ PARSING ──▶ INDEXED ──▶ READY
   │                     │
   ▼                     ▼
FILE_TOO_LARGE       PARSE_FAILED / TIMEOUT
UNSUPPORTED_FORMAT
```

### Database: SQLite WAL mode + aiosqlite
`PRAGMA journal_mode=WAL` + `aiosqlite` for async writes. Zero extra infra.
Handles 10–50 concurrent devs. Upgrade path to Postgres deferred.

### dp_cli interface: asyncio.create_subprocess_exec wrapper
dp_cli called via `asyncio.create_subprocess_exec('dp', ...)`. Clean isolation —
dp_cli crashes don't propagate into the FastAPI process. Stderr captured for structured error reporting.

### Concurrency cap: asyncio.Semaphore(4)
Global `asyncio.Semaphore(4)` in `run_dp_parse`. No new dependencies. Caps concurrent
dp_cli subprocesses at 4 — prevents overloading the DeepParser backend API (dp_cli is
an HTTP client; local CPU is not the constraint). Semaphore value is tunable via env var
`DP_MAX_CONCURRENT_PARSE` (default: 4).

## API Contract Minimums

- `/parse`: max 50MB, 120s timeout. Error schema: `{"error": "string", "code": "PARSE_FAILED|TIMEOUT|FILE_TOO_LARGE|UNSUPPORTED_FORMAT"}`.
- `/ask`: 30s timeout. Returns citations with source filename + page/cell reference.
- Data retention: raw files deleted after parse completes. Parsed index retained until API key is revoked. Privacy-sensitive users should run the Docker image locally.

## Performance Ceiling (Section 7)

- dp_cli subprocesses don't block the FastAPI event loop (separate OS processes)
- Real concurrency limit: DeepParser backend API capacity (dp_cli is HTTP client; local CPU not the bottleneck; Semaphore(4) protects the remote API, not the host)
- /ask: <5s per call, not a bottleneck
- Single Fly.io/Railway instance is sufficient for validation phase (3-50 developers)
- Scale-out: horizontal replicas if needed; Postgres migration only if SQLite WAL becomes the bottleneck

## Test Strategy (Section 6)

- **D8:** CI runs unit tests only (mocked dp_cli subprocess). Integration tests run manually before each release using real dp_cli credentials locally.
- Unit coverage targets: auth (401/403), parse job state transitions, UUID rename, /ask NOT_READY guard, /admin/stats password check
- dp_cli mock fixture: returns pinned JSON that matches the `DPCLIResult` Pydantic model — any model drift caught on model update, not at runtime
- Benchmark (10 docs, 50 Q&A) is a manual run — output stored in `benchmark/results.json` in the repo

## Code Quality Defaults (Section 5)

- Pydantic models for all request/response shapes — no bare dicts returned from handlers
- `DPCLIResult` Pydantic model validates dp_cli stdout on every subprocess call — silent format drift caught at parse time
- **D7:** Background tasks use `asyncio.create_task` (detached from request lifecycle) — not FastAPI `BackgroundTasks` (request-tied, risks silent cancellation on 120s parse jobs)
- No ORM — direct aiosqlite queries; SQLAlchemy deferred until Postgres migration

## Data Flow & Edge Cases (Section 4)

**Parse lifecycle:**
POST /parse → auth → save as `{uuid}.{ext}` → insert job QUEUED → background task →
PARSING → dp_cli subprocess → READY or PARSE_FAILED/TIMEOUT.
GET /parse/{job_id} scoped to `api_key_id` — cross-key lookup returns 404.

**Key edge cases:**
- Duplicate uploads: accepted, no dedup at v1 (two jobs, two indexes)
- /ask during PARSING: returns `NOT_READY` + current status
- Server restart mid-parse (D6): stuck PARSING jobs expire via 24h auto-expiry (no startup sweep — simpler)
- Disk near-full: check <100MB free before accepting upload; return 503 if not
- Concurrent /ask: dp_cli reads only, no contention

## Security Model (Section 3)

| Threat | Mitigation |
|---|---|
| API key brute-force | Rate-limit auth failures: 10/min per IP |
| Malicious upload (zip bomb, path traversal) | UUID rename on ingest; isolated tmp dir; deleted after parse |
| dp_cli command injection | **D5:** All uploaded files renamed to `{uuid4}.{ext}` before any subprocess call. Original filename stored in DB for display only. No user-controlled string ever reaches subprocess args. |
| Admin endpoint exposure | `ADMIN_PASSWORD` env var required; fail closed if unset |
| Cross-key index exfiltration | Job records and index dirs namespaced by `api_key_id` in SQLite |
| dp_cli SSRF | Flag for P0 dp_cli audit — unknown if dp_cli fetches remote URLs during parse |

## Error & Rescue Map (Section 2)

| Failure | Trigger | Code | Recovery |
|---|---|---|---|
| File >50MB | POST /parse | `FILE_TOO_LARGE` | Compress/split before upload |
| Unsupported format | POST /parse | `UNSUPPORTED_FORMAT` | Check supported format list |
| dp_cli non-zero exit | During PARSING | `PARSE_FAILED` + detail | Retry; contact support |
| Parse >120s | During PARSING | `TIMEOUT` | Split doc; use Docker locally |
| Invalid job_id | GET /parse/{id} | 404 | Resubmit |
| Missing/invalid API key | Any endpoint | 401 | Fix `X-API-Key` header |
| Revoked API key | Any endpoint | 403 | Re-register |
| /ask before READY | GET /ask | `NOT_READY` + current status | Poll until READY |
| Disk full | POST /parse | 503 | Operator: disk cleanup |
| dp_cli not on PATH | POST /parse | 500 | Operator: verify Docker image |

**Job expiry (D4):** Failed, READY, and TIMEOUT jobs auto-deleted after 24h via hourly background
task (APScheduler). Parsed indexes follow the same 24h window. Developers have 24h to retrieve
error messages before the job record is gone.

### Engineering Implementation Notes (required, no decision needed)

- **asyncio.shield is mandatory** for sync mode: `asyncio.wait_for(asyncio.shield(task), 10.0)` prevents cancellation of the running parse task on handler timeout
- **API key comparison**: `hmac.compare_digest(provided_key, stored_key)` — prevents timing attacks. Never use `==`
- **Global exception handler**: `@app.exception_handler(Exception)` returns JSON 500, not HTML
- **File extension validation**: validate extension server-side BEFORE calling dp_cli; return UNSUPPORTED_FORMAT early
- **dp_cli returncode**: explicitly check `process.returncode != 0` → PARSE_FAILED (not only exception handling)
- **Cleanup task crash recovery**: wrap cleanup loop body in `try/except Exception: logger.exception(...)` — task must survive DB errors
- **client.demo() URL versioning**: SDK constant (not hardcoded string) pinned to SDK version; if demo endpoint changes, bump SDK minor version
- **POST /ask semaphore**: P0 dp_cli audit to confirm if `/ask` also spawns dp_cli subprocess — if yes, apply same Semaphore or separate limit
