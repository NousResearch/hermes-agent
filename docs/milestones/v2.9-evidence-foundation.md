# Hermes v2.9 Evidence-First Managed Agents Foundation

**Tag:** `v2.9-evidence-foundation`  
**Date:** 2026-06-02  
**Status:** Complete  

---

## Executive Summary

Hermes v2.9 introduces the Evidence-First Managed Agents foundation: five
incremental phases that make every router decision, policy evaluation, and
agent capability explainable and auditable without changing actual execution
behavior.

The system can now tell you **why** it routed a task to a particular agent,
**what** it considered, **whether** the decision is advisory or binding, and
**which** metrics are known vs unknown. It does all of this without
auto-executing impactful actions, without inventing certainty when data is
missing, and without breaking existing workflows.

---

## What Changed

### 1. Task Taxonomy

- `TaskType` enumeration with 10 canonical types
- Legacy aliases: `tests` -> `test`, `code_review` -> `review`
- Missing `task_type` falls back to `investigation` with `taxonomy_confidence: unknown`
- Backward compatible: old tasks without task_type load without error

### 2. Ledger Event Normalization

- `LedgerEventType` enumeration with 15 canonical event types
- `_canonical_event_type()` maps legacy `run_*` events to `execution.*`, `policy.*`, etc.
- `_normalize_ledger_event()` adds `event_type` to old ledger records on read
- New ledger writes automatically include `event_type` alongside legacy `event`
- Old records without event_type fall back to `unknown`

### 3. Router Explanation

- `RouterExplanation` and `CandidateExplanation` structured dataclasses
- Every router decision now produces: selected agent, candidate list, per-agent
  scores, reasons, penalties, and metrics
- `advisory_only: true` on every explanation
- `build_router_explanation()` works without changing router ranking logic
- Missing metrics explicitly marked as `unavailable`, not invented

### 4. Policy Dry-Run

- `ProposedAction` dataclass: action_id, task_id, action_type, reason,
  requires_human_approval, evidence
- `derive_proposed_action()` maps execution policy decisions to standardized
  action types (retry, switch_agent, escalate_to_review, mark_blocked,
  request_human_approval)
- Safe actions (complete, continue) produce no proposal
- `POST /api/agents/runs/policy/propose` dry-run endpoint writes
  `event_type: policy.proposed` to ledger without executing
- CI-facing endpoint returns `advisory_only: true`

### 5. Agent Capability Contract

- Four new advisory fields per agent: `not_recommended_for`, `risk_limit`,
  `preferred_phase`, `requires_review_after`
- Contracts configured for claude, codex, deepseek-tui, opencode
- Agents without contracts load with empty defaults (no hard-block)
- Exposed in `CapabilityProfile.to_dict()` and API responses

### 6. Integration Audit

- One bug found and fixed: `LedgerEventType.unknown` was missing from the enum
- All 5 cross-feature integration checks pass
- 12 pre-existing ledger test failures fixed by the audit

### 7. Scenario Evaluation

- 6 new scenario tests added covering all 8 audit scenarios
- Scenarios include: implementation routing, missing task_type fallback,
  failed -> policy.proposed, timeout advisory-only, agent without contract,
  missing explanation as unavailable, CI safety

## Architecture Impact

No existing execution behavior was changed:

- Router ranking: **unchanged**
- Policy execution: **unchanged**
- Scheduler behavior: **unchanged**
- Agent selection: **unchanged**
- CLI execution: **unchanged**

All new capabilities are **advisory layers** that sit alongside the existing
pipeline:

```
                    Task Taxonomy
                         |
                    Router Explanation  (advisory only)
                    /           \
          Capability Matrix    Router Ranking (unchanged)
                    |
              Execution Policy
              /               \
      Policy Dry-Run       Actual Execution (unchanged)
      (proposed actions)
                    |
              Run Ledger
         (canonical event types)
```

---

## Evidence-First Behavior Now Supported

| Behavior | How |
|----------|-----|
| Router explains why it selected an agent | `RouterExplanation` with per-candidate reasons and penalties |
| Missing metrics are not invented | `timeout N/A`, `fail N/A`, `unavailable` |
| Decisions are advisory by default | `advisory_only: true` on all router decisions |
| Policy actions are proposed, not auto-executed | `ProposedAction` written as `policy.proposed` |
| Human approval is explicit | `requires_human_approval` on every proposed action |
| Old ledger records remain readable | `event_type` normalized on read, legacy `event` preserved |
| Old agent configs load without error | Empty defaults for missing contract fields |
| Dashboard shows uncertainty honestly | `unknown`, `unavailable`, `external-managed` displayed explicitly |

---

## Explicit Non-Goals

- No automatic agent demotion
- No automatic weight mutation
- No hard-blocking of agents based on capability contracts
- No CI auto-execution of impactful policy actions
- No behavior change to router ranking, scheduler, or policy execution
- No destructive migration of legacy ledger data
- No Rust performance layer (not yet needed)
