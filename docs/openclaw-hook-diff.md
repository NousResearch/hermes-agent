# OpenClaw → Hermes Hook System Gap Analysis

Generated by analysis of:
- `/mnt/projects/hermes/src/hooks/internal-hooks.ts` (OpenClaw internal hooks)
- `/mnt/projects/hermes/src/plugins/types.ts` (OpenClaw plugin hook types)
- `/mnt/projects/hermes/src/plugins/hooks.ts` (OpenClaw plugin hook runner)
- `/mnt/projects/hermes/gateway/hooks.py` (Hermes current state)
- No `gateway/plugins.py` exists — file is entirely absent.

---

## 1. ARCHITECTURE OVERVIEW

OpenClaw has TWO independent hook systems that serve different consumers:

### System A — Internal Hooks (`src/hooks/internal-hooks.ts`)
- Consumed by bundled hooks (HOOK.md / handler.ts pattern, e.g. session-memory)
- Handler signature: `(event: InternalHookEvent) => Promise<void> | void`
- Structured event object: `{ type, action, sessionKey, context, timestamp: Date, messages: string[] }`
- Registration key: `"type"` (broad) or `"type:action"` (specific); both fire on trigger
- Execution: serial, sequential, errors caught and logged

### System B — Plugin Hooks (`src/plugins/hooks.ts`)
- Consumed by OpenClawPlugin's `api.on(hookName, handler, { priority? })` registration
- Per-hook typed event + context pair (see full list below)
- Three execution modes:
  - **void/parallel**: all handlers run concurrently via `Promise.all`
  - **modifying/sequential**: handlers run in priority order (higher first), results merged
  - **claiming/sequential**: first handler returning `{ handled: true }` stops the chain
  - **sync-only**: two hooks explicitly MUST NOT return Promises
- Priority: integer `priority` field on registration (default 0), higher = earlier
- No equivalent exists in Hermes at all

Hermes `hooks.py` currently only partially implements System A.

## 2. WHAT HERMES hooks.py IS MISSING

### 2a. Missing Events (Internal Hook System)

Hermes current docstring lists these events:
```
gateway:startup, session:start, session:end, session:reset,
agent:start, agent:step, agent:end, command:*
```

OpenClaw fires these additional `type:action` events that Hermes does not:

| Missing Event | type | action | Payload fields in `event.context` |
|---|---|---|---|
| `message:received` | `message` | `received` | `from: str, content: str, channelId: str, accountId?, conversationId?, messageId?, timestamp?: int, metadata?: dict` |
| `message:sent` | `message` | `sent` | `to: str, content: str, success: bool, channelId: str, accountId?, conversationId?, messageId?, isGroup?: bool, groupId?, error?: str` |
| `message:transcribed` | `message` | `transcribed` | `transcript: str, channelId: str, from?, to?, body?, bodyForAgent?, timestamp?, conversationId?, messageId?, senderId?, senderName?, senderUsername?, provider?, surface?, mediaPath?, mediaType?` |
| `message:preprocessed` | `message` | `preprocessed` | `channelId: str, from?, to?, body?, bodyForAgent?, transcript?, timestamp?, conversationId?, messageId?, senderId?, senderName?, senderUsername?, provider?, surface?, mediaPath?, mediaType?, isGroup?: bool, groupId?` |
| `agent:bootstrap` | `agent` | `bootstrap` | `workspaceDir: str, bootstrapFiles: list, cfg?: dict, sessionKey?: str, sessionId?: str, agentId?: str` |

Notes on existing events where Hermes fires them but payload shape is undocumented:
- `agent:step` — listed in Hermes docs but the step context payload is unspecified. OpenClaw uses this for per-turn tool-calling loop events.
- `gateway:startup` — OpenClaw context: `{ cfg?, deps?, workspaceDir? }`
- `session:start` — OpenClaw context: `{ sessionId, sessionKey?, resumedFrom? }`
- `session:end` — OpenClaw context: `{ sessionId, sessionKey?, messageCount: int, durationMs?: int }`
- `agent:start` — OpenClaw context: `{ prompt: str, messages?: list }`
- `agent:end` — OpenClaw context: `{ messages: list, success: bool, error?: str, durationMs?: int }`
- `command:*` — OpenClaw fires with `event.action` = the command name (e.g. `"new"`, `"reset"`)

### 2b. Missing Execution Modes

Hermes `emit()` runs all handlers **serially** in registration order. OpenClaw has:

| Mode | OpenClaw | Hermes |
|---|---|---|
| Parallel (void) | `Promise.all(handlers)` — fire-and-forget | Not present — always serial |
| Sequential + merge (modifying) | Priority order, results accumulated | Not present |
| Sequential + first-claim | Stop on first `{ handled: true }` | Not present |
| Sync-only enforcement | Warns if handler returns a Promise | Not present |
| Priority ordering | `priority?: number` per registration | Not present |

### 2c. Missing Internal Hook `messages` Field

OpenClaw's `InternalHookEvent` has a `messages: string[]` field that hooks can push into. The gateway then forwards those strings back to the user. Hermes has no equivalent — there is no way for a hook to send a reply to the user.

```python
# OpenClaw shape (needs Python equivalent):
event = {
    "type": "command",
    "action": "reset",
    "sessionKey": "agent/default",
    "context": { ... },
    "timestamp": datetime.now(),
    "messages": [],  # hooks append strings here; caller reads and delivers them
}
```

### Merge functions required per hook:

| Hook | Merge logic |
|---|---|
| `before_model_resolve` | `modelOverride = acc.get("modelOverride") or next.get("modelOverride")` (first defined wins) |
| `before_prompt_build` | `systemPrompt` = last non-None; `prependContext/prependSystemContext/appendSystemContext` = `"\n\n".join(filter(None, [acc, next]))` |
| `before_agent_start` | union of both above |
| `message_sending` | `content = next.get("content") or acc.get("content")`, `cancel = next.get("cancel") or acc.get("cancel")` |
| `before_tool_call` | `params/block/blockReason` = last non-None wins |
| `subagent_spawning` | if either is error → propagate error; else `threadBindingReady = bool(acc.threadBindingReady or next.threadBindingReady)` |
| `subagent_delivery_target` | first non-None `origin` wins |

### Execution mode implementations:

```python
async def _run_void_hook(self, hook_name, event, ctx):
    """All handlers run in PARALLEL via asyncio.gather."""
    hooks = self._get_hooks(hook_name)  # sorted by priority descending
    if not hooks:
        return
    tasks = [self._call_handler(h, event, ctx) for h in hooks]
    await asyncio.gather(*tasks, return_exceptions=True)  # errors caught per-task

async def _run_modifying_hook(self, hook_name, event, ctx, merge_fn):
    """Handlers run SEQUENTIALLY in priority order; results merged."""
    hooks = self._get_hooks(hook_name)
    result = None
    for hook in hooks:
        next_result = await self._call_handler(hook, event, ctx)
        if next_result is not None:
            result = merge_fn(result, next_result) if result is not None else next_result
    return result

async def _run_claiming_hook(self, hook_name, event, ctx):
    """Sequential; stops at first result with handled=True."""
    hooks = self._get_hooks(hook_name)
    for hook in hooks:
        result = await self._call_handler(hook, event, ctx)
        if result and result.get("handled"):
            return result
    return None

def _run_sync_hook(self, hook_name, event, ctx, merge_fn):
    """Synchronous only. Detect and warn on async handlers."""
    hooks = self._get_hooks(hook_name)
    result = None
    for hook in hooks:
        out = hook.handler(event, ctx)
        if asyncio.iscoroutine(out):
            # Log warning: async not allowed for this hook
            out.close()
            continue
        if out is not None:
            result = merge_fn(result, out) if result is not None else out
    return result
```

## 5. PLUGIN HOOK DISCOVERY (what becomes plugins.py)

OpenClaw `resolvePluginHookDirs` does:
1. Load plugin manifest registry from workspace dir
2. Filter by: plugin enabled in config + passes exclusive slot selection (memory/contextEngine)
3. For each plugin with `hooks` paths in manifest: resolve absolute path, check exists, check path is inside plugin root (security)
4. Return list of `{ dir: str, plugin_id: str }`
