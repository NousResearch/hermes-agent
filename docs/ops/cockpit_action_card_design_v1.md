# Unified Cockpit Action Card Design v1

Date: 2026-03-21
Status: active (Wave 8 C1 Operator Cockpit UX)
Parent doctrine: `docs/ops/unified_operating_doctrine_v1.md`

## 0) Purpose
To condense complex subsystem telemetry (SLOs, layered health, continuity drift, parity validations) into a single, unambiguous status surface for the human operator. An action card explicitly separates what is wrong from how to fix it.

## 1) Card Layout
The unified action card must render in Markdown with the following structured sections:

### 1. Headline Status
A single emoji and a terse phrase:
- 🟢 `SYSTEM OPTIMAL` (SLOs pass, Gate open)
- 🟡 `DEGRADED OPERATIONS` (SLOs warn, Mutation Gate `caution`)
- 🔴 `BLOCKER: SYSTEM HALTED` (Gate `forbidden`)

### 2. Failing Constraints (if any)
A bulleted list of failing guards or SLOs. Must include the exact contract id.
Example: 
- `SLO-1_VERIFY_FRESHNESS`: `Verification evidence is stale (5000 sec old)`
- `A6_HEALTH`: `OpenClaw gateway process not found (alive layer failed)`

### 3. Immediate Action (Remediation)
A direct CLI command or a link to the relevant playbook the operator must execute to resolve the blocker.
Example: 
- `Run: bash docs/ops/incident_playbooks/blindness_recovery.md`
- `Run: openclaw gateway restart`

### 4. Telemetry Footer
Tiny text or code block containing the raw snapshot pointers.
- `chk_latest: chk_20260321T220705Z_1a5fe9`
- `gtc_latest: gt_20260321T220658Z`

## 2) Runtime Binding
Action cards are generated by `ops/openclaw/continuity/cockpit_summary.sh` from live A6 snapshots (`layered_health_snapshot.json`, `slo_snapshot.json`, `verify_last.json`).

Schemas/contracts define required input fields and status semantics, but the renderer itself is currently script-defined (not schema-templated).
