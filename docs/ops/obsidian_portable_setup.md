# Portable Setup: Obsidian Integration + Local Memory EDD Harness (OpenClaw, local-first)

This guide is for operators moving the Obsidian/OpenClaw integration to another machine.
It is deterministic, copy-pasteable, and tuned for **silent success + fail-closed safety**.

---

## 1) Prerequisites

## Required

- **OpenClaw**: `2026.3.x` recommended (tested with `2026.3.2`)
  - Must support:
    - `openclaw memory index`
    - `openclaw memory search --max-results ... --json`
- **Python 3**: `>=3.10` (stdlib-only scripts)
- **bash**

## Optional

- **pdftotext** (`poppler-utils`) for operator workflows that preprocess PDFs before putting notes into governed markdown.
  - Not required by the core scripts below.

## Quick preflight

```bash
openclaw --version
python3 --version
command -v pdftotext || echo "pdftotext not installed (optional)"
```

## 3) Governed vault schema conventions

Every governed markdown note must have YAML frontmatter with non-empty values for:

- `type`
- `id`
- `status`
- `trust_level`
- `created`
- `updated`

Minimal template:

```markdown
---
type: spec
id: spec.example.v1
status: draft
trust_level: reviewed
created: 2026-03-08
updated: 2026-03-08
---

# Title
...
```

## 5) Component map

- `ops/obsidian/vault_compiler.py`
  - Deterministic vault compiler.
  - Reads markdown/frontmatter from allowlisted roots.
  - Writes:
    - `vault/.compiled/plans/<digest>.json` (or custom output)
    - `vault/.compiled/index.json`
  - Also provides deterministic writeback helpers (`writeback-note`, `writeback-artifact`).

- `ops/obsidian/vault_validate.py`
  - Schema validator for governed notes.
  - Emits machine-readable JSON to stdout.
  - `--strict` exits non-zero on violations.

- `ops/obsidian/export_to_memory.py`
  - Structure-aware export from vault markdown to memory chunks.
  - Chunking rules:
    - Header lineage chunking (`#`, `##`, `###`...)
    - Does **not** split on headings inside fenced code blocks
    - Table-aware handling (large tables split safely, with grounding context when applicable)
    - `--max-chars` cap (default `4000`) with deterministic subchunking for oversized sections
  - Writes only to memory export folders (never inside vault).

- `ops/obsidian/retrieval_golden_queries.json`
  - Golden query set for EDD retrieval checks.
  - Supports per-query fields like `kind` (`positive`/`negative`), `expect`, optional `top_k`, optional `max_score_le`.
  - Stores strict metric thresholds.

- `ops/obsidian/retrieval_eval.py`
  - Executes `openclaw memory search` for each golden query.
  - Evaluates strict retrieval quality.
  - Multi-K metrics supported (if present in thresholds):
    - `hit_rate_at_5`, `hit_rate_at_10`, `hit_rate_at_20`, `mrr`, `precision_at_5_expected`
    - plus `negative_pass_rate` for negative traps
  - `--strict` fails if thresholds are not met.

- `ops/obsidian/vault_tick.sh`
  - Low-noise compile tick wrapper.
  - **Silent on success**.
  - On failure emits deduped `BLOCKER_JSON: ...` (new signature or reminder window).
  - Exits `0` intentionally to avoid generic cron-failed spam.

- `ops/obsidian/hourly_canary.sh`
  - End-to-end hourly canary wrapper:
    1. strict schema validate
    2. export to memory
    3. `openclaw memory index`
    4. strict retrieval eval
    5. vault tick
  - **Silent on success**.
  - On failure emits deduped `BLOCKER: step=...; reason=...`.

## 6) Manual runbook (copy/paste)

Set paths once:

```bash
export ROOT="/home/yeqiuqiu/clawd-architect"
export VAULT="$ROOT/obsvault_yq_terminal"
cd "$ROOT"
```

### A) Validate governed schema

```bash
python3 ops/obsidian/vault_validate.py --vault "$VAULT" --strict
```

Expected:
- stdout: JSON (`{"ok":true|false,...}`)
- exit `0` if clean, `1` if violations exist

### B) Compile deterministic plan

```bash
python3 ops/obsidian/vault_compiler.py compile \
  --vault "$VAULT" \
  --output "$VAULT/.compiled/plan.json"
```

Expected:
- stdout: JSON with keys like `ok`, `plan_digest`, `plan_path`, `index_path`, `summary`
- files updated in `"$VAULT/.compiled"`

### C) Export vault to memory chunks

```bash
python3 ops/obsidian/export_to_memory.py \
  --vault "$VAULT" \
  --out-root "$ROOT/memory/obsidian_export_chunks" \
  --overview "$ROOT/memory/obsidian_export_latest.md" \
  --max-chars 4000 \
  --print-summary
```

Expected:
- stdout: summary JSON
- chunk files under `memory/obsidian_export_chunks/**`
- `INDEX.json` updated

### D) Build/rebuild memory index

```bash
openclaw memory index
```

Expected:
- completes with exit `0`
- first run may take longer while downloading/loading embedding model

### E) Run retrieval harness (strict)

```bash
python3 ops/obsidian/retrieval_eval.py --strict --top-k 20
```

Expected:
- stdout: deterministic JSON summary (`metrics`, `threshold_eval`, `results`)
- exit `0` if thresholds pass
- on strict fail: exit `1` and stderr contains `STRICT_FAIL: ...`

### F) Run vault tick (single-shot)

```bash
OPENCLAW_VAULT_PATH="$VAULT" bash ops/obsidian/vault_tick.sh
```

Expected:
- success: no stdout, exit `0`
- failure: single deduped `BLOCKER_JSON: {...}` line, still exit `0`

### G) Run full hourly canary once

```bash
OPENCLAW_VAULT_PATH="$VAULT" bash ops/obsidian/hourly_canary.sh
```

Expected:
- success: no stdout, exit `0`
- failure: single deduped `BLOCKER: step=...; reason=...`, exit `0`

## 7) OpenClaw cron job example (hourly canary, delivery.mode=none, forward BLOCKER only)

Use this job payload pattern (session/agent IDs are examples):

```json
{
  "agentId": "codex-executioner",
  "sessionKey": "agent:codex-executioner:telegram:direct:5936691533",
  "name": "obsidian-vault-tick-hourly-canary",
  "enabled": true,
  "schedule": {
    "kind": "every",
    "everyMs": 3600000
  },
  "sessionTarget": "isolated",
  "wakeMode": "now",
  "payload": {
    "kind": "agentTurn",
    "timeoutSeconds": 120,
    "message": "Run hourly Obsidian full canary (silent on success).\\n\\nCommand:\\nOPENCLAW_VAULT_PATH=/home/yeqiuqiu/clawd-architect/obsvault_yq_terminal \\\\nOPENCLAW_HOURLY_CANARY_STATE_FILE=/tmp/obsidian_hourly_canary_state_canary.json \\\\nOPENCLAW_HOURLY_CANARY_REMINDER_SECS=21600 \\\\nbash /home/yeqiuqiu/clawd-architect/ops/obsidian/hourly_canary.sh\\n\\nThen:\\n- Capture stdout.\\n- Find the first non-empty line.\\n- If it starts with \"BLOCKER:\" or \"BLOCKER_JSON:\", send that line to Telegram user 5936691533 using message tool (channel=telegram,target=5936691533).\\n- Otherwise reply exactly: NO_REPLY."
  },
  "delivery": {
    "mode": "none"
  }
}
```

Key point: `delivery.mode = none` suppresses generic cron delivery; only explicit forwarded `BLOCKER*` lines are sent.

## 8) OpenClaw config snippet (local memory search + EmbeddingGemma GGUF)

Place under `agents.defaults.memorySearch` in `~/.openclaw/openclaw.json`:

```json
{
  "enabled": true,
  "provider": "local",
  "fallback": "none",
  "model": "hf:ggml-org/embeddinggemma-300m-qat-q8_0-GGUF/embeddinggemma-300m-qat-Q8_0.gguf",
  "local": {
    "modelPath": "hf:ggml-org/embeddinggemma-300m-qat-q8_0-GGUF/embeddinggemma-300m-qat-Q8_0.gguf"
  }
}
```

One-time download note:
- The `hf:...gguf` reference can trigger a one-time model fetch/cache on first embedding/index operation.
- First `openclaw memory index` may be slow; subsequent runs are faster from local cache.

## 9) Troubleshooting

## A) Schema violations (`vault_validate.py --strict` fails)

- Inspect JSON output field `violations[]`.
- Typical causes:
  - missing frontmatter block
  - malformed frontmatter delimiters (`---`)
  - missing required fields (`type,id,status,trust_level,created,updated`)

## B) Export write failures (`export_to_memory.py` exits non-zero)

Check:
- include roots are vault-relative and exist
- output path is **outside** vault (exporter refuses writes inside vault)
- filesystem permissions for `<ROOT>/memory/**`

Useful files:
- `<ROOT>/memory/obsidian_export_chunks/INDEX.json`
- `<ROOT>/memory/obsidian_export_latest.md`

## C) Memory indexing failures (`openclaw memory index`)

Common causes:
- local embedding model not available yet
- incorrect `memorySearch.local.modelPath`
- insufficient disk/memory for GGUF load

Checks:
- verify `agents.defaults.memorySearch.provider = local`
- verify `model` and `local.modelPath` are identical GGUF references
- retry `openclaw memory index` after model download completes

## D) Canary/tick blocker dedupe state inspection

State files:

- Vault tick state (default):
  - `<VAULT>/.compiled/vault_tick_state.json`
  - fallback: `/tmp/obsidian_vault_tick_state_<hash>.json`
- Hourly canary state (default):
  - `<VAULT>/.compiled/hourly_canary_state.json`
  - fallback: `/tmp/obsidian_hourly_canary_state_<hash>.json`

Other useful artifacts:
- `<VAULT>/.compiled/plan.json`
- `<VAULT>/.compiled/index.json`
- OpenClaw cron state: `~/.openclaw/cron/jobs.json` (`state.lastStatus`, `state.consecutiveErrors`)
- OpenClaw logs: `~/.openclaw/logs/`

## 10) Minimal security posture

- Do **not** store API keys/tokens/secrets in governed vault notes.
- Keep all include roots vault-relative; never allow `..` or absolute include roots.
- Enforce path containment:
  - **portable hardened policy:** do not write outside `vault/.compiled` and memory export folders
  - exporter writes only under `memory/obsidian_export_chunks` (+ overview in memory)
  - if you keep compiler writeback helpers enabled, pin `reports-root`/`artifacts-root` to approved contained paths
- Keep success path silent to reduce noise; alert only on deduped blocker signatures.
