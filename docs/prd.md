# Keystat Landing Page PRD

> 文档状态：Draft v0.1  
> 项目：Keystat Landing Page  
> 范围：只规划 landing page，不包含业务代码实现。  
> 信息来源：用户本次指令为产品定位唯一事实来源；当前仓库内容仅作为技术/目录现状参考。

## 1. 背景与目标

Keystat landing page 用于介绍 Keystat 产品、突出核心价值，并引导访客完成等待名单、下载、联系等转化动作。页面需要在用户首次访问的短时间内解释“Keystat 是什么、为谁解决什么问题、为什么值得现在留下线索或采取行动”。

### 1.1 业务目标

1. 清晰建立 Keystat 产品认知：让首次访客在首屏理解产品定位与主要价值。
2. 促进转化：引导目标用户加入等待名单、下载产品或联系团队（具体主 CTA 待确认）。
3. 支撑后续营销：提供可被搜索、分享、投放落地页复用的内容结构。
4. 降低沟通成本：为后续设计、文案、开发、埋点和验收提供统一规格。

### 1.2 用户目标

1. 快速判断 Keystat 是否与自己的需求相关。
2. 了解核心收益、主要功能、适用场景与可信依据。
3. 能够在低阻力路径下完成下一步动作：加入等待名单、下载、联系或了解更多。

### 1.3 成功指标（建议）

> 具体数值目标待用户确认，以下为建议指标口径。

- 首屏 CTA 点击率。
- 等待名单/下载/联系表单提交转化率。
- 页面滚动深度：50%、75%、100%。
- 关键区块可见率与交互率。
- 自然搜索访问占比与品牌词曝光变化。
- Core Web Vitals：LCP、INP、CLS 达标率。

## 2. 非目标

1. 不实现 Keystat 产品本体功能。
2. 不实现登录、账号体系、云同步、订阅计费、用户后台等能力；如后续需要，均需另行确认。
3. 不承诺真实价格、正式发布时间、平台支持范围或下载链接，除非用户提供已确认信息。
4. 不开发业务代码；本阶段仅产出 PRD 文档。
5. 不将项目定位改写为仓库既有产品或其他产品。

## 4. 目标用户

### 4.1 主要用户画像

1. **潜在个人用户**
   - 需要快速理解 Keystat 能带来的效率、洞察或管理价值。
   - 对安装成本、隐私安全、跨平台支持、价格敏感。

2. **潜在团队/商业用户**
   - 关注团队层面的可见性、分析能力、合规、部署方式和采购沟通。
   - 更可能点击“联系团队/预约演示”。

3. **早期采用者/产品探索者**
   - 愿意尝鲜、加入等待名单、参与 beta 或提供反馈。
   - 关注产品愿景、路线图、社区/更新节奏。

### 4.2 用户关键问题

- Keystat 是什么？
- 它解决了什么具体问题？
- 和我现在的工作流有什么关系？
- 我能立刻获得什么价值？
- 是否安全、轻量、可信？
- 现在应该加入等待名单、下载，还是联系团队？

## 5. 核心信息架构

建议采用单页 landing page 信息架构：

1. Hero 首屏：一句话定位、核心价值、主 CTA、辅助 CTA、产品视觉。
2. Problem 问题：目标用户当前痛点。
3. Solution 解决方案：Keystat 如何解决问题。
4. Value Props 核心价值：3-5 个关键收益。
5. Product Preview 产品预览：截图、动效、流程图或概念图。
6. Use Cases 使用场景：个人、团队、分析/报告等场景。
7. Trust 信任构建：隐私、安全、轻量、本地优先/数据策略（待确认）。
8. Social Proof 社会证明：用户评价、logo、数据、媒体引用（如暂无则不展示或使用“Coming soon”策略）。
9. FAQ 常见问题：降低疑虑。
10. Final CTA 末屏转化：重复主 CTA，提供替代联系入口。
11. Footer 页脚：品牌、隐私政策、条款、联系方式、社媒链接（具体链接待确认）。

## 6. 页面区块需求

### 6.1 Hero 首屏

**目标**：在 5 秒内传达产品定位与下一步动作。

建议内容：

- H1：清晰表达 Keystat 的核心价值。示例方向：`Understand your keyboard-driven workflow at a glance.`（最终文案待确认）
- Subheadline：补充说明 Keystat 如何帮助用户获得洞察、优化效率或追踪关键输入行为。
- 主 CTA：加入等待名单 / 下载 Keystat / 获取早期访问（待确认）。
- 辅助 CTA：查看功能 / 联系团队 / 观看演示（待确认）。
- 视觉：产品界面截图、仪表盘 mockup、键盘/数据可视化相关动效或静态图。

验收要点：首屏在移动端不依赖横向滚动，CTA 可见且可点击。

### 6.2 痛点区块

**目标**：让目标用户感到问题被准确描述。

建议覆盖：

- 工作流输入行为不可见，难以量化。
- 效率瓶颈难以被发现。
- 工具使用、快捷键习惯或输入模式缺乏反馈。
- 团队/个人缺少轻量的数据化回顾方式（团队场景待确认）。

### 6.3 解决方案区块

**目标**：解释 Keystat 的方法论。

建议表达：

- 捕捉关键使用信号（具体采集范围待确认）。
- 将行为转化为可理解的统计、趋势与洞察。
- 帮助用户发现模式、改善习惯、提升效率。

注意：不得在未确认前承诺后台、云端同步、企业管理台或跨设备同步能力。

### 6.4 核心价值区块

建议 3-5 个价值点：

1. **Clarity / 可见性**：把隐藏的输入与使用模式变成可读指标。
2. **Focus / 聚焦改进**：发现高频行为、低效路径或可优化习惯。
3. **Progress / 持续追踪**：查看趋势变化，理解改进是否有效。
4. **Lightweight / 轻量**：低学习成本，快速获得反馈（具体安装/权限待确认）。
5. **Privacy-aware / 隐私友好**：数据处理与存储策略需用户确认后再作为正式卖点。

### 6.5 产品预览区块

所需素材：

- Dashboard 主界面截图或 mockup。
- 关键指标卡片：例如输入量、快捷键使用、应用/任务上下文、趋势图（具体指标待确认）。
- 一段 3 步使用流程：Install → Track → Improve（最终流程待确认）。

### 6.6 使用场景区块

建议场景：

- 个人效率复盘：理解自己的输入节奏与高频操作。
- 开发者/创作者工作流：观察快捷键、编辑、切换上下文等行为模式。
- 团队生产力洞察：如团队版存在，可展示团队层面趋势；如未确认，不作为正式承诺。

### 6.7 信任与隐私区块

必须回答：

- 收集什么数据？不收集什么数据？
- 数据存储在哪里？本地、云端或混合？
- 是否记录具体输入内容？
- 用户如何删除数据？
- 是否开源、是否可审计、是否支持离线？

> 以上均为待确认问题。未确认前页面只可使用保守表述，例如“Privacy details coming soon”或“Designed with privacy in mind”，避免具体承诺。

### 6.8 FAQ 区块

建议 FAQ：

1. Keystat 适合谁？
2. Keystat 会记录我输入的具体内容吗？（答案待确认）
3. 支持哪些操作系统？（待确认）
4. 是否免费？是否有团队版？（价格待确认，不写具体价格）
5. 数据是否会上传云端？（待确认）
6. 什么时候可以使用？（发布时间待确认，不承诺日期）
7. 如何联系团队或参与早期测试？

### 6.9 Final CTA 与 Footer

- 末屏重复主 CTA，文案更直接，例如“Join the Keystat waitlist”。
- Footer 包含：品牌、联系方式、隐私政策、服务条款、社媒链接、版权信息。
- 如果政策页尚未准备，Footer 链接应标注为待补充或在上线前移除。

## 7. CTA 策略

### 7.1 CTA 类型

- 主 CTA（优先级最高）：
  - 加入等待名单（推荐默认，适合早期 landing page）。
  - 下载 Keystat（仅在安装包/发布版本确认后启用）。
  - 获取早期访问（适合 beta 招募）。

- 辅助 CTA：
  - 联系团队。
  - 观看演示。
  - 阅读隐私说明。
  - 查看产品截图/功能。

### 7.2 CTA 规则

1. 首屏必须有主 CTA。
2. 页面中段在关键价值解释后可重复一次 CTA。
3. 末屏必须有最终 CTA。
4. 移动端 CTA 点击区域需足够大，建议高度不低于 44px。
5. CTA 不应承诺未确认的上线时间、下载平台或价格。

## 8. 转化事件与埋点需求

> 事件命名仅为建议，实际埋点工具待确认。

### 8.1 页面浏览与行为事件

- `page_view_landing`
  - 属性：`referrer`、`utm_source`、`utm_medium`、`utm_campaign`、`device_type`、`locale`。
- `scroll_depth_reached`
  - 属性：`depth_percent` = 25/50/75/100。
- `section_viewed`
  - 属性：`section_name`。

### 8.2 CTA 事件

- `cta_clicked`
  - 属性：`cta_label`、`cta_type`、`section_name`、`destination`。
- `waitlist_form_started`
- `waitlist_form_submitted`
  - 属性：`email_domain`（避免采集完整邮箱作为分析属性）、`role`、`use_case`（如表单包含）。
- `download_clicked`
  - 属性：`platform`、`version`（仅在下载可用后启用）。
- `contact_clicked`
  - 属性：`contact_method`。

### 8.3 表单转化要求

- 等待名单表单最小字段建议：Email。
- 可选字段：角色、使用场景、公司、备注。
- 表单提交后需要明确成功状态与下一步提示。
- 失败状态需可恢复，避免用户输入丢失。
- 需要隐私同意提示与链接（上线前确认）。

## 9. 内容需求

### 9.1 必备文案

- 产品一句话定位。
- 3-5 个核心价值标题与说明。
- 3 个典型使用场景。
- 5-7 条 FAQ。
- 表单成功/失败状态文案。
- SEO title、description、Open Graph 标题与描述。

### 9.2 必备视觉素材

- Logo / wordmark。
- Hero 产品视觉：截图、mockup 或概念图。
- favicon。
- Open Graph 分享图。
- 如使用图标，应保持统一风格并具备可访问替代文本。

### 9.3 内容语气

- 清晰、可信、直接。
- 避免夸大承诺，如“保证提升效率”“完全安全”“零风险”等。
- 在事实未确认处使用待确认或更保守表述。

## 10. SEO 要求

1. 页面必须有唯一 `title`，建议包含 `Keystat` 与核心关键词。
2. 页面必须有 `meta description`，长度建议 120-160 英文字符或等效中文长度。
3. 使用语义化 heading：单个 H1，H2/H3 层级清晰。
4. 配置 Open Graph / Twitter Card 基础信息。
5. 图片具备描述性 `alt`。
6. 首屏核心内容不得完全依赖图片渲染。
7. 若支持多语言，需要明确 canonical/hreflang 策略（待确认）。
8. 页面 URL、站点域名、sitemap、robots 策略待确认。

## 11. 性能要求

1. Core Web Vitals 建议目标：
   - LCP ≤ 2.5s。
   - INP ≤ 200ms。
   - CLS ≤ 0.1。
2. 首屏资源应控制体积，避免无必要的大型动画/视频阻塞加载。
3. 图片需使用合适尺寸与现代格式，并提供懒加载策略。
4. 第三方脚本最小化；分析脚本需异步加载。
5. 移动端 4G 网络下仍应可快速加载和完成 CTA。
6. 若使用复杂动效，需支持 reduced motion。

## 12. 可访问性要求

1. 满足 WCAG 2.2 AA 作为目标。
2. 全站键盘可导航，焦点状态清晰。
3. CTA、表单、导航具备可读 label。
4. 文本与背景对比度达标。
5. 动效不应造成眩晕；支持 `prefers-reduced-motion`。
6. 表单错误需以文本方式呈现，不只依赖颜色。
7. 图片和图标提供合适替代文本；装饰性图标可隐藏给屏幕阅读器。

## 13. 响应式与浏览器要求

- 支持移动端、平板、桌面端布局。
- 常见断点建议：360px、768px、1024px、1440px。
- 支持现代版本 Chrome、Safari、Firefox、Edge。
- 页面不得出现横向滚动。

## 14. 验收标准

### 14.1 产品验收

1. 页面标题统一为 **Keystat Landing Page**。
2. 首屏清楚说明 Keystat 的定位与价值。
3. 至少包含：Hero、痛点、解决方案、核心价值、产品预览、使用场景、信任/隐私、FAQ、最终 CTA、Footer。
4. 主 CTA 路径明确，且全页至少出现 2 次。
5. 不出现未经确认的真实价格、发布日期、下载平台承诺。
6. 不出现云后端、账号体系、团队版等未确认能力的确定性承诺。
7. FAQ 覆盖隐私、平台、价格/计划、可用时间等关键疑虑。

### 14.2 内容验收

1. 所有待确认事实均以 TBD/待确认标记或保守文案处理。
2. 图片、图标、截图位置有明确素材需求说明。
3. 文案不混用其他项目名称。
4. 页面可独立被目标用户理解，无需额外上下文。

### 14.3 数据验收

1. CTA 点击、表单开始、表单提交、下载点击、联系点击均有事件定义。
2. UTM 参数可保留并进入分析属性。
3. 表单成功/失败状态可追踪。
4. 不在分析属性中暴露不必要的个人敏感信息。

### 14.4 SEO/性能/可访问性验收

1. `title`、`description`、OG 信息齐备。
2. Lighthouse 建议：Performance、Accessibility、Best Practices、SEO 均达到 90+（具体门槛待确认）。
3. 移动端无横向滚动，核心 CTA 可访问。
4. 键盘导航、表单错误、焦点状态通过检查。
