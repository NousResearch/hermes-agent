# Action Candidate Rubric

Updated: 2026-04-18

这个文档用于帮助 Hermes 在复盘后判断：
**下一个最值得新增的长期资产是什么。**

## 先判断问题类型

### 适合写成 skill
当问题满足以下任一条件：
- 是重复出现的工作流
- 已经有一套验证过的步骤
- 有明显前置条件 / 验证方法 / 常见坑
- 下次再遇到时，不想重新思考

### 适合写成 doc
当问题更像：
- 原则
- 规则
- 结构说明
- 多个 skill 共享的判断标准
- 长期边界定义

### 适合写成 script
当问题更像：
- 固定格式的重复处理
- 把多个已知动作串起来
- 低风险自动化
- 需要稳定产出某类文件/摘要/检查结果

### 适合写成 cron
当问题更像：
- 应该定期执行
- 不需要每次都由用户提醒
- 输出是摘要 / 复盘 / 健康检查 / 建议
- 失败不会导致高风险副作用

### 适合写成 memory
仅当它是稳定长期事实：
- 用户偏好
- 环境事实
- 稳定工作约定
- 反复会踩的坑

不要把临时任务进度写进 memory。

## 优先级判断

### High
满足任一条件：
- 同一个问题近期反复出现
- 已经浪费了多次人工解释/排查
- 修好后能明显减少下次摩擦
- 与系统健康/可靠性关系强

### Medium
满足任一条件：
- 有复用价值，但还不是最频繁
- 现在补上会让系统更完整
- 是已有资产的补强层

### Low
满足任一条件：
- 只是锦上添花
- 还没出现足够重复信号
- 现在不做也不会明显影响下次效率

## 选择规则
复盘后优先输出：
1. 一个 top priority 候选动作
2. 再给 2~3 个备选
3. 每个候选都写清：
   - type
   - name
   - priority
   - reason
   - suggested_action

## 不该输出为候选动作的内容
- 高风险不可逆操作
- 需要用户授权的敏感外发
- 只是临时想法、没有证据支撑的“大跃进”
- 完全没有复用价值的一次性事项
