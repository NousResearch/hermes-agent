# Independent Senior Review (Opus)

## Verdict: BLOCK

The merge *strategy* (merge-not-rebase, isolated worktree, staged deploy, gated sign-off) is sound and well-evidenced. I'm blocking on **verification completeness**: for an explicitly high-blast-radius 7-host production merge, the proof spine certifies the 35 conflicted files but is structurally blind to where this class of merge actually breaks — clean auto-merges, config migration, and gates that may reference tests that don't exist. As written, this PRD can ship a silent fleet regression with all-green CI. Close the holes below and it flips to APPROVE WITH CHANGES.

## Critical Blockers (severity-ordered, cite section/evidence)

1. **The verification spine trusts the conflict set; the real blast radius is the clean merge (§1 Ground-Truth, §5B, INV-1).** Every gate keys off the 35-file/83-hunk surface, but that's only what git *chose* to flag against 2,070 upstream-touched ∩ 356 fork-touched files. Two unguarded failure classes dominate production merges this size: (a) **rename-detection misses** — upstream both *relocated and refactored* `gateway/platforms/*.py` → `plugins/platforms/*/adapter.py`; if similarity drops below git's threshold, the fork's edit lands as a clean ADD at the dead old path while runtime calls the new module → fork behavior silently absent, and **INV-1's symbol-grep passes on the dead duplicate**; (b) **semantic conflicts** — a fork caller in an unconflicted file against an upstream-changed callee signature merges clean and fails only at runtime. Neither has a gate beyond "full CI green," and CI demonstrably does not cover the fork-only systems (LCM = 0 upstream matches, per INV-1). This is the outage path.

2. **Phase gating is physically non-executable as specified (§6 Phase 1–3, §5B level-1).** Phase 1 starts `git merge --no-commit`, materializing all 35 conflicts at once, then claims to run `pytest tests/plugins/memory/ -x` as the Phase-1 gate. Conflict markers (`<<<<<<<`) in `.py` files are syntax errors → any pytest run that *transitively imports* a still-conflicted core module (agent/*, run_agent, etc.) fails at collection, not on logic. The mem0 plugin almost certainly imports conflicted core. The "resolve-one-subsystem-then-test-it-green-before-moving-on" spine cannot run until *all* markers are cleared — so "level 1 per-file" verification is illusory.

3. **Config migration against real fleet `config.yaml` is never exercised (§7, INV-5, §10).** Absorbing 1,546 upstream commits almost certainly bumps `DEFAULT_CONFIG`/`_config_version`; the migration runs on first post-deploy launch against each host's *live* config carrying fork-only keys (kanban per-task-model, fallback reasoning_effort, F1/F2 settings). CI runs against temp `HERMES_HOME` (AGENTS.md hermetic rule) and therefore **structurally cannot catch a migration that drops or rejects fork-only keys.** That's a silent fleet config regression with green CI. No gate, no dry-run.

4. **"Per-system test passes" gates assume tests that the PRD never confirms exist (INV-1, INV-2, §6 Phase 2–3).** Gates cite `tests/gateway/test_telegram_network_reconnect.py`, `test_telegram_noise_filter.py`, plus dedicated suites for F1/F2 restart-cascade, send_message in-turn routing, and blackbox per-class cost. If any fork system shipped without a dedicated test, its INV gate is vapor and the revert-to-RED proof (Phase 2) has nothing to assert against. A gate that references a possibly-nonexistent test is not a gate.

5. **1,546 inherited upstream commits are un-audited; "zero behavior change" (§2) is false framing.** The merge *is* 1,546 commits of upstream behavior change — only the conflicted slice is reviewed. Nothing audits what upstream **added**: new outbound calls / telemetry (AGENTS.md: opt-in-gated only), new *core tools* shipped on every API call (changes the every-turn schema + cost + cache profile — the project's sacred constraint), or new `HERMES_*` env vars. The fleet has an `external-code-ingest-audit` posture; this is the largest ingest of the year with no diff-audit gate on the upstream side.

## Required Changes

- **Add a clean-merge audit gate (blocks #1).** Beyond the 35 conflicts: (a) run `git merge` with explicit rename detection and enumerate every rename where the fork also touched the source — confirm each fork edit re-threaded onto the new path AND the old path is gone (grep *both* old-absent and new-present, not just symbol-present); (b) build a fork-only symbol/import smoke that imports every INV-1 module and asserts its public entrypoints resolve, to catch dead-duplicate and semantic breaks the conflict set misses.
- **Re-order or re-scope the phase gates (blocks #2).** Either resolve in dependency order so the tree stays importable at each checkpoint, or explicitly state that per-system suites only run after *all* markers are cleared and replace the mid-merge "test green" checkpoints with marker-free + import-smoke checkpoints.
- **Add a config-migration dry-run on copies of real fleet configs (blocks #3).** Pre-deploy, run the merged migration against a snapshot of each host's `config.yaml`; assert every fork-only key survives. Add to §10 acceptance and the Phase-6 smoke.
- **Verify-then-cite every gate test (blocks #4).** Before execution, confirm each INV-1/INV-2 system has an existing test that fails when the fork behavior is removed; for any system without one, write the test as part of this work or downgrade its INV to "manual verification" honestly.
- **Add an upstream-diff audit step (blocks #5).** Diff `c6b0eb4..origin/main` for: new core tools added to the every-call schema, new outbound/telemetry, new `HERMES_*` env vars, new dependencies with major bumps. Record findings; flag any AGENTS.md violations inherited.
- **Fix the verification mechanics:** use `git diff --check` + `git grep -nE '^(<<<<<<<|=======|>>>>>>>)'` (not grep `<<<<<<<` alone — §10 misses `=======`/`>>>>>>>` and resolved-wrong files); run the **hermetic wrapper** `scripts/run_tests.sh`, not bare `pytest -x -q` (Phase 4 bypasses it → local-vs-CI drift, the exact thing AGENTS.md warns against); drop `-x` for the full-suite run so you get the complete failure set on a huge merge.
- **Define deploy objectivity:** name host-1 (deploy to the highest-risk/most-representative host — the gateway host — first, not "one host"); give "regresses" an objective threshold; add a soak/bake window before fanning out to 6 (a single immediate smoke won't catch turn-N / compaction / memory-growth regressions); confirm `fleet.sh deploy claude-bridge --restart` is the correct update vector for **hermes-agent core code**, not just the bridge.

## Lens Notes (one line each)

- **Architecture:** Sound strategy, but the design certifies the conflict set and is blind to rename-misses + semantic conflicts in the ~2,070-file clean-merge surface — the actual failure mode.
- **Security/identity-isolation:** Secret-scan/SAST/OSV present and required (INV-4, good), but inheriting 1,546 upstream commits with no behavioral/telemetry/new-core-tool audit violates the project's own ingest and narrow-waist posture.
- **DevOps/SRE:** Per-host SHA rollback + staged deploy are real; weakened by no soak window, unnamed host-1, no objective regression threshold, and `git revert -m 1` understated (it poisons the next merge).
- **Implementation/maintainability:** Phase ordering reads clean but is non-runnable — conflict markers break imports, so mid-merge per-subsystem gates can't execute until the whole tree is resolved.
- **QA:** Best part of the doc (revert-to-RED mutation tests, negative cases), but several gates cite tests of unconfirmed existence/coverage, and `-x` + bare-pytest undercut completeness.
- **Config-drift:** The PRD targets external drift well but ignores its own internal config-version migration against live fleet configs — the one drift CI cannot see.

## Residual Risks / Open Questions

- **Q (OQ-1 sharpened):** Freeze-at-snapshot is correct, but record the exact frozen `origin/main` SHA in the spec now; "929dd9c0d" in §1 vs "snapshot" in §10 must be the same pinned value or acceptance criteria are unverifiable.
- **Q:** Are the 4 parked live-tree changes (INV-8) part of the "220 fork-only commits preserved" count or additive? §10's `git log` acceptance won't cover uncommitted work; the interplay needs to be explicit.
- **Risk:** `git revert -m 1 <merge-sha>` as the stated branch-rollback (§7) silently poisons the *next* upstream merge (revert-of-revert footgun). Per-host `reset --hard` is the real rollback and is fine; downgrade the revert claim.
- **Risk:** Lockfile regeneration absorbs the full upstream dependency delta; `uv sync` clean proves *resolution*, not runtime compat — a transitive major bump (e.g. pydantic) can pass OSV/lockfile-check and still break the fleet. No major-bump diff step.
- **Risk:** Greptile "always blocks" (§8) + Phase-6 `--admin` merge: confirm own-fork-green `--admin` does not bypass *unresolved* Greptile threads, or INV-4's "no merge over an open gate" is quietly violated.
- **Ambiguity:** `test_mem0_v2.py (UD)` vs `test_run_agent.py (DU)` — resolution differs materially by direction; the DU/UD notation should be spelled out before Phase 4.
