What Was Broken

Hermes Agent was a Linux-first tool. Running it on native Windows meant hitting crash after crash: missing fcntl imports, hardcoded /tmp paths, signal.SIGKILL that doesn't exist, /dev/tty writes that fail, a clipboard that spawned PowerShell for every keystroke, and a code execution sandbox that flat-out refused to work because Python on Windows doesn't have Unix domain sockets.

Clipboard: 50,000x Faster

× Before

Every Ctrl+V spawned a PowerShell process to check the clipboard. Cold start: 2-15 seconds. Image paste felt completely broken.

subprocess.run(["powershell", "-Command",
  "[Clipboard]::ContainsImage()"])
# 2,000 - 15,000ms per check
✓ After

Native win32 API via ctypes. Instant clipboard format checks. PowerShell only invoked for actual PNG extraction.

ctypes.windll.user32
  .IsClipboardFormatAvailable(CF_BITMAP)
# 0.03ms per check

  • ⊕

  • ⊕

  • ⊕
    save_clipboard_image() — Gates on instant ctypes check, then PowerShell only for DIB→PNG conversion.

Code Execution Sandbox: TCP Fallback

The execute_code tool lets the AI write Python scripts that call Hermes tools via RPC. The parent process listens on a socket, the child connects and sends JSON-RPC requests. On Linux, this uses Unix domain sockets. On Windows, socket.AF_UNIX doesn't exist in stock CPython.

× Before

Tool showed red in banner. Hard error: "execute_code is not available on Windows." The entire sandbox was dead.

✓ After

Dual-transport RPC: UDS on POSIX, TCP 127.0.0.1:0 on Windows. Same protocol, different socket type. Works everywhere.

Architecture

Parent (Hermes Agent) | |-- _detect_transport() | Linux/macOS → AF_UNIX socket at /tmp/hermes_rpc_xxx.sock | Windows → AF_INET socket at 127.0.0.1:{random_port} | |-- HERMES_RPC_SOCKET env var | UDS: "/tmp/hermes_rpc_abc123.sock" | TCP: "tcp:127.0.0.1:54321" | Child (sandbox subprocess) | |-- hermes_tools.py _connect() if addr.startswith("tcp:") → AF_INET connect else → AF_UNIX connect

The CPython Story

We didn't stop at the TCP fallback. We found the upstream CPython PR that adds AF_UNIX to Windows (#137420) — open since August 2025, still awaiting review. So we applied the patch ourselves, built CPython 3.13.12 from source on Windows, and verified it works.

Total patch: 10 lines across 3 files.

  • §
    PC/pyconfig.h.in — Detect <afunix.h> header via __has_include
  • §
    Modules/socketmodule.h — Include <afunix.h> when HAVE_AFUNIX_H is defined
  • §
    Modules/socketmodule.c — Register AF_UNIX in win_runtime_flags for build 17134+, fix int cast
# Stock Python 3.13 on Windows:
>>> hasattr(socket, "AF_UNIX")
False

# Our build:
>>> socket.AF_UNIX
1
>>> s = socket.socket(socket.AF_UNIX, socket.SOCK_STREAM)
>>> s.close()  # works

All Windows Fixes

Four phases of work, from crash fixes to polish:

Phase 1 — Crash Fixes
fcntl → msvcrt file locking · SIGKILL → SIGTERM guard · /tmp → tempfile.gettempdir() · /dev/tty → CON device
Phase 2 — Security
icacls ACL enforcement (chmod is no-op on Windows) · Keyring credential manager · Cross-platform shell_quote() · SMS webhook bound to localhost
Phase 3 — Feature Parity
taskkill /F /T for process trees · Windows Task Scheduler gateway · wmic process status queries · Dual-transport sandbox (TCP + UDS)
Phase 4 — Polish
Native win32 clipboard via ctypes · WINDOWS.md documentation · Windows CI workflow · 37 compat tests + 82 clipboard tests · CPython AF_UNIX build

Install on Windows

# PowerShell (one-liner)
irm https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.ps1 | iex

# Or manual
pip install hermes-agent
hermes setup

Requires Windows 10 version 1803+ and Python 3.10+. Native Windows PowerShell, CMD, or Windows Terminal all work. WSL not required.
