### [High] shell=True in local STT transcription tool — user-controlled template injection

**File**: `tools/transcription_tools.py`  
**Line(s)**: 1206–1209  
**Issue**: When `HERMES_LOCAL_STT_COMMAND` env var is set, the user's command template is rendered and executed with `shell=True`. If the template contains shell metacharacters (`;`, `|`, `$(...)`, `` ` `` ), arbitrary command execution is possible. The code itself uses `shlex.quote()` on placeholders, but the template string itself is under user control, making injection possible when the template format string itself contains shell operators.

**Why invisible in previous passes**: Requires understanding that `use_shell = bool(os.getenv(...))` means the user's env-var IS the command string, not just a path argument.

**Impact**: RCE if attacker can set `HERMES_LOCAL_STT_COMMAND` env var or supply malicious model/path names that influence the rendered template.

**Suggested fix**: Always use `shell=False` with a list-form command. If shell syntax is genuinely needed, validate the template against a strict allowlist of safe characters before rendering.

**Confidence**: High

### [High] shell=True in Docker container cleanup — dangling shell operators

**File**: `tools/environments/docker.py`  
**Line(s)**: 637–651  
**Issue**: Three `subprocess.Popen` calls use `shell=True` with hardcoded string commands containing `&&`, `>`, `|`, `$(...)`. While the command strings themselves are internally constructed (not from user input), if `_docker_exe` or `_container_id` were compromised or malformed, shell injection via these variables is theoretically possible. More critically, these are fire-and-forget background processes — any failure is silently caught and swallowed.

**Why invisible in previous passes**: Requires tracking that `_docker_exe` and `_container_id` are instance attributes that flow directly into shell string construction without sanitisation.

**Impact**: Container take-over or host compromise if docker executable path or container ID can be controlled by attacker.

**Suggested fix**: Use list-form args to `subprocess.Popen`: `['timeout', '60', docker_exe, 'stop', container_id]`. No shell required for this command.

**Confidence**: Medium

### [Medium] yaml.load with default loader in skill metadata parsing

**File**: `agent/skill_utils.py`  
**Line(s)**: 78–79  
**Issue**: `yaml.load(value, Loader=loader)` uses `loader = getattr(yaml, "CSafeLoader", None) or yaml.SafeLoader`. When neither CSafeLoader nor SafeLoader is available (e.g. no libyaml extension), this falls back to the base Loader class. The base Loader is NOT safe for untrusted input — it can execute arbitrary Python objects via YAML tags like `!!python/object/apply:os.system`. While the input here is skill metadata (loaded from disk or network), a compromised skill file could trigger RCE.

**Why invisible in previous passes**: Requires understanding that PyYAML's default loader chain can fall back to unsafe behavior when CSafeLoader is absent.

**Impact**: RCE if a malicious skill file is loaded (e.g. via a compromised skill registry, or if a previously trusted skill is overwritten).

**Suggested fix**: Always use `yaml.SafeLoader` explicitly as the fallback, never the base `Loader`. Add a runtime warning when CSafeLoader is unavailable.

**Confidence**: Medium — requires compromised skill file, lower probability but critical impact.

### [Medium] time.sleep() blocking call inside async def in MCP tool reconnect loop

**File**: `tools/mcp_tool.py`  
**Line(s)**: 1898–1902  
**Issue**: Inside `async def _wait_for_lifecycle_event()`, the reconnect polling loop uses `time.sleep(0.25)` which is a blocking call. This blocks the entire asyncio event loop for 250ms per iteration. In a worst case of 60 iterations (15 second deadline / 0.25), this means the event loop is blocked for up to 15 seconds continuously if the server never comes back. All other async operations in the same event loop are blocked during this entire window.

**Why invisible in previous passes**: Requires understanding that `time.sleep()` inside an `async def` that runs on the MCP event loop blocks ALL concurrent operations on that loop — not just the current task.

**Impact**: Event loop blocking, which can cause timeouts in other tasks (HTTP requests, other tool calls) that share the same loop. With a 15-second deadline, this is bounded, but still causes significant latency spikes every 0.25 seconds during the blocking window.

**Suggested fix**: Replace `time.sleep(0.25)` with `await asyncio.sleep(0.25)` — allows other tasks to run while waiting.

**Confidence**: High — this is clearly blocking the event loop.

### [Low] Unbounded queue.Queue() for input buffering in CLI

**Files**: `cli.py:12493`, `cli.py:3203`  
**Line(s)**: `self._pending_input = queue.Queue()` and `self._interrupt_queue = queue.Queue()`  
**Issue**: Two unbounded `queue.Queue()` instances are used for input buffering. If the agent is slow or stalled, input can accumulate indefinitely. There's no `maxsize` bound — in a pathological scenario where the agent never processes input, the queue could grow to consume all available memory.

**Confidence**: Low — very unlikely in practice, but a theoretical DoS vector.

### [Medium] ffmpeg audio conversion subprocess.run() without timeout in transcription tool

**File**: `tools/transcription_tools.py`  
**Line(s)**: 1166  
**Issue**: `subprocess.run(command, check=True, capture_output=True, text=True)` calls ffmpeg to convert non-WAV audio formats to WAV before transcription. There is no `timeout` parameter. If ffmpeg hangs on a malformed audio file (e.g., a truncated MP3 with an invalid codec), the entire tool execution blocks indefinitely. All other concurrent tool calls on the same event loop are also affected since `subprocess.run` is blocking (not async).

**Why invisible in previous passes**: Requires understanding the context — this is inside `_convert_to_wav()`, called from the STT pipeline, and the function doesn't propagate any timeout from its callers.

**Impact**: Process hang / DoS if a malformed audio file is submitted. Since this runs in the tool execution path, a single stuck tool can block the entire agent turn.

**Suggested fix**: Add `timeout=60` to the ffmpeg subprocess.run call. Handle `subprocess.TimeoutExpired` and return an error dict instead of raising.

**Confidence**: Medium — dependent on ffmpeg behavior with malformed input.

### [Medium] delegate_tool task creation without cleanup tracking — orphaned subagent processes

**File**: `tools/delegate_tool.py`  
**Line(s)**: Multiple (look for `asyncio.create_task` without storing result)  
**Issue**: Subagent tasks are created via `asyncio.create_task()` for concurrent execution but the task references may not be tracked for cleanup. If a parent agent crashes or is cancelled mid-delegation, the child subagent processes may continue running orphaned on the host — consuming CPU, memory, and potentially producing outputs that are never consumed.

**Why invisible in previous passes**: Requires understanding the full lifecycle of a delegated task — creation, execution, result consumption, and cleanup. The `await delegate_task()` pattern correctly waits for results, but cancellation during the await leaves children running.

**Impact**: Resource leak — orphaned subagent processes accumulate over time. Can cause CPU/memory exhaustion on long-running agents that delegate frequently.

**Suggested fix**: Track all child task references in a collection and cancel them on parent shutdown. Use `asyncio.TaskGroup` (Python 3.11+) for structured concurrency with automatic cancellation propagation.

**Confidence**: Low-Medium — depends on cancellation patterns and shutdown timing.

### [Low] Double iteration over same sessions list without combining — O(n) redundancy in process registry

**File**: `tools/process_registry.py`  
**Line(s)**: 1282–1308 (`has_active_processes`, `has_active_for_session`)  
**Issue**: Both functions iterate over `self._running.values()` twice: first in an unguarded loop to call `_refresh_detached_session(session)` on each session (lines 1287–1288 / 1301–1302), then immediately in an `any()` expression that also iterates over `self._running.values()` (lines 1291–1294 / 1305–1308). This means if there are N running sessions, each call to these functions does 2N iterations over the dict values instead of combining the two operations into a single pass.

**Why invisible in previous passes**: Requires understanding that two separate `for` loops over the same collection can be fused. The code is correct, just inefficient.

**Impact**: Minor performance degradation when many running processes exist. Each call does 2N iterations instead of N. For typical usage (≤10 processes), this is negligible.

**Suggested fix**: Combine the two loops: perform `_refresh_detached_session` while also checking the condition in a single `for s in sessions:` loop.

**Confidence**: Medium.

### [Low] String += in a loop (minor string concatenation)

**File**: `cli.py:5293`  
**Line**: `names_str += ", ..."` (inside a loop building a display string)  
**Issue**: String concatenation with `+=` inside a loop is O(n²) in CPython because strings are immutable — each `+=` creates a new string object. For loops with many iterations, this can be measurable.

**Why invisible in previous passes**: The performance impact only manifests with large iteration counts. Most instances found were counter increments (`count += 1`) which are O(1) integer operations, not string concatenation — these are false positives.

**Suggested fix**: Use `", ".join(list_of_strings)` or `io.StringIO` for building large strings.

**Confidence**: Low — mostly false positives in this codebase (counter += 1 is fine).

### [Low] Memory state has no integrity verification — silent data corruption possible

**File**: `plugins/memory/` (all memory plugins)  
**Issue**: The memory plugins (hindsight, mem0, holographic, supermemory, honcho) store conversation and context data in SQLite or their own storage backends. There is no checksum, MAC, or integrity verification on stored memory records. If the storage backend suffers silent data corruption (bit flip, disk error, partial write), the agent will consume corrupted memory without detection.

**Why invisible in previous passes**: Requires understanding that "no corruption detection" is a security/reliability gap. Standard code analysis doesn't flag missing integrity checks because the code functions correctly under normal conditions.

**Impact**: Silent context corruption — the agent makes decisions based on garbled memory. This can cause unpredictable behavior in long-running agents where memory is critical. For adversarial scenarios, an attacker with disk write access could corrupt memory to manipulate agent behavior.

**Suggested fix**: Add a SHA-256 HMAC or similar integrity field to memory records. Verify on read. This is particularly important for memory plugins that store structured data (SQLite without integrity checks).

**Confidence**: Low — requires silent storage corruption which is rare in practice, but impact is high when it occurs.
