# Design

## Boundary and portability

- **Claim boundary.** Linux hard containment is available only after the scheduler owns a dedicated cgroup-v2 boundary, or—when a future systemd adapter exists—a systemd transient unit/scope with `KillMode=control-group`. The preserved implementation currently has the cgroup-v2 path only; it makes no claim that systemd transient-unit/scope support is implemented or tested. A generic inherited cgroup, a process group, an observed `/proc` tree, and a PID/start-time snapshot are never sufficient to make the hard-containment claim. macOS and Windows have no equivalent detached-session contract and must return `unsupported`.
- **Capability discovery.** Discovery is runtime-only and affirmative: identify cgroup v2; prove that a delegated parent can create and remove an empty child; prove an immutable identity and expected parent; prove PID assignment plus membership readback while the child is parked; and prove a writable cgroup-targeted kill operation on the actual allocated boundary. A future systemd adapter must prove equivalent transient-unit/control-group facts before it can advertise availability. No host-specific path is hardcoded. A controller, delegation, permission, or kill-capability failure found before boundary allocation is `unavailable`. A failure after an owned boundary exists but before release retains that ownership and is `cleanup_failed` unless safe teardown completes and the implementation can classify it more specifically.
- **Status truthfulness.** `contained` is reported only after the actual allocated boundary has passed the writable kill probe and the parked child has passed assignment, identity, parent, and membership verification. The implementation and documentation do not claim systemd transient-unit/scope support.
- **Launch-before-membership prevention.** The parent starts the child only to its parked handshake, records the process in the parent-owned registry, creates and records the boundary handle, assigns the observed child PID, and revalidates identity, parent, and membership. Only then does it release the start pipe that permits agent/tool execution. Every failure before release terminates the parked child and tears down only the still-owned boundary; it must not fall through to workload execution.
- **Opaque ownership and teardown.** The handle contains the unique cgroup/unit identity, expected parent, and identity evidence (cgroup inode or the systemd `ControlGroup` identity). Teardown revalidates those facts rather than reconstructing a path from a job ID, targets only that cgroup/unit (`cgroup.kill` or systemd stop with `KillMode=control-group`), then proves boundary emptiness and child reaping. If identity changes, cgroup termination fails, or either proof cannot be obtained, ownership remains registered with `cleanup_failed`; the scheduler must not silently substitute process-group cleanup or report success.
- **Fallback semantics.** When hard containment is unavailable or unsupported, the scheduler exposes `unavailable`/`unsupported` before launch and may use process-group cleanup only as explicitly-labelled best effort. It may not claim that a detached descendant is contained. These statuses preserve cron availability without overstating platform guarantees.
- Remove temporary process-global subreaper, parent-death signal, `/proc` descendant harvesting, and destructive PID signalling from the cron boundary. PID/start-time observations may be diagnostic only.

## Verification contract

- The active direct-boundary test exercises allocation, assignment/readback, teardown, detached descendant termination, and unrelated-process preservation. The capability-gated supervisor test additionally exercises parent registration, parked-child assignment, release only after verification, owned-boundary teardown, leader exit with a `start_new_session=True` descendant, and unrelated-process preservation through `_run_isolated_cron_job`.
- Production-path failure tests inject assignment/identity-parent validation failure, allocation cleanup failure, and termination failure. They assert the child remains parked on failed validation and that `cleanup_failed` retains both process and boundary ownership. Discovery failure coverage reports explicit unavailable/unsupported status. Systemd integration coverage belongs to a separately implemented adapter, not this cgroup-v2 test.

- `_run_isolated_cron_job` computes one wall deadline and uses `min(0.25, remaining)` for result polling. Its `finally` force-signals, joins, then removes `_cron_processes` ownership.
- `cron.executions.execution_is_live` reads the durable ledger and proves the recorded PID/process start owner is live.
- `claim_job_for_fire` stores `execution_id` in `fire_claim`; on expiry it consults durable ownership and fails closed when liveness cannot be disproved.
- `CronScheduler.fire_due` creates the durable attempt before the CAS claim, records rejected/vanished attempts as failed, and passes the winning execution through `run_one_job`.
- Existing process-isolation tests cover actual spawned descendants and non-cooperative workers; the repair adds stale-claim ownership coverage.
