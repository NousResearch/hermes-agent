执行前(在循环里):工具名校验与修复

还没进 _execute_tool_calls 之前,循环先防幻觉:模型常把工具名写成变体——大小写不同、连字符/空格、驼峰、甚至给你加个 _tool 后缀(TodoTool_tool、Patch_tool、BrowserClick_tool,见 #14784)。直接判「未知工具」太浪费,所以先尽力纠正,纠不回才回错给模型。

① 修复:归一化 5 连招 + 模糊匹配

先说人话:Hermes 内部真正的工具名是全小写+下划线(todo、read_file、web_search…),这个集合叫 valid_tool_names。但模型爱写花样变体:想调 todo 却写成 Todo / todo-tool / TodoTool / TodoTool_tool。「归一化」就是把这些花样掰回标准写法再去清单里查。

「5 连招」是从最省事到最费劲依次试 5 种改法,任意一招改完能在清单里查到就用它、不再往下试(这就是「逐级尝试,命中即返回」)。以 TodoTool_tool → 目标 todo 为例:

招做什么例子变化
1. 小写直配直接全小写去查Todo → todo ✅ 停
2. 连字符/空格→下划线再把 -、空格换成 _web-search → web_search ✅
3. 驼峰转蛇形大写字母前插下划线再小写TodoTool → todo_tool
4. 剥 _tool/tool 后缀去掉模型爱加的尾巴,最多剥两层TodoTool_tool → TodoTool → Todo → todo ✅
5. 模糊匹配(兜底)前 4 招全不中,用 difflib 算「谁最像」,相似度 ≥ 0.7 取它redfile → read_file

还有个第 0 步:先剪垃圾——个别厂商(VolcEngine)会把 XML 碎片漏进工具名(terminal" parameter="command" ...),所以遇到第一个 " ' < > 就截断,只留 terminal;但故意不按空格切,否则合法的 write file 会被切成 write 丢掉后半截。

对照源码看这 5 连招:

为什么不「对的先跑、错的报错」?① 消息结构硬约束:每个 tool_call 必须一一对应一条 tool 结果(角色交替铁律),对的先跑会让下一轮重发时配对/顺序错位;② 可能有依赖:模型常把一批工具当一个计划(先 A 再 B),其中一个名字写错说明这批意图没成形,贸然执行对的那个可能做错事。所以设计成整批要么全合法一起跑、要么全打回。
「连错 3 次」是要连续的,且 partial 是体面退出。
· 计数会归零:_invalid_tool_retries 在任何一次校验通过后立刻清零,所以是「连续 3 轮都非法」才触发,中间成功一次就重新计——不会因整个会话里零星错几次攒到 3。
· partial=True 不是崩溃:它是诚实的「半途而止」——本轮不再要模型调工具,但已完成的部分(之前成功的工具、已产出文本)全部落库保留;上层(CLI / 网关 / kanban worker)据此能区分「被非法工具卡住没干完」和「正常完成」。具体每个上层怎么处理,见下面的框。
· 为什么要设上限:每打回一轮都是一次真实 API 调用(上下文越来越长、烧钱);更要防某些弱开源模型看到文件里的 <tool_call> XML 就不停模仿发空工具名的 priming 死循环(#47967)——越喂清单越糟,必须硬上限截断。

并发还是串行:判定规则

agent/tool_dispatch_helpers.py · _should_parallelize_tool_batch(tool_calls)

并发不是「多个工具就一定并行」——要确认这批互不干扰才敢并行,否则老老实实串行。判定逐条走:

条件结果
只有 1 个工具→ 串行
批里含 clarify(_NEVER_PARALLEL_TOOLS)→ 串行(要和用户交互,不能并发)
某个工具参数不是合法 JSON / 非 dict→ 串行(保守)
文件工具(read_file/write_file/patch)目标路径重叠→ 串行(防写冲突)
某工具既不在只读白名单、又不是路径独立的文件工具、也不是声明可并行的 MCP 工具→ 串行
以上全过→ 并发
只读白名单 _PARALLEL_SAFE_TOOLS(无共享可变状态,天然可并行):read_file、search_files、web_search、web_extract、vision_analyze、session_search、skills_list/skill_view、ha_get_state 等。
文件工具靠路径判定:read_file/write_file/patch 各自抽出目标 path,只要两两前缀不重叠就能并行(_paths_overlap 比较路径分量);一旦可能指向同一子树,退回串行。MCP 工具默认不并行,除非其服务器显式声明安全。

    tools.tool_search · resolve_underlying_call / _tool_search_scoped_names

    先说最关键的一点:Tool Search 默认不是一直启用的。只有当可延迟工具的 schema 加起来 ≥ 模型上下文窗口的 10% 时才触发;工具不多时它就是个 no-op,模型直接看到完整工具表,下面这套「搜/查/调」根本不会发生。判定在 should_activate:

if deferrable_tokens <= 0:   return False                 # 没有可延迟工具 → 不触发
if config.enabled == "on":   return True                  # 强制开 → 有就触发
# auto(默认):

return deferrable_tokens >= context_length * (threshold_pct/100)   # 默认 10%
    三个要点:①只算「可延迟」工具——核心工具(terminal/read_file)永不计入、本来就一直全发,所以是「非核心/MCP/插件工具多到一定程度」才触发,不是「工具总数多」;②阈值是窗口的百分比(如 200K 窗口 ×10% ≈ 20K token 的 schema),盯的是「每轮重发这些 schema 贵不贵」,同一批工具大窗口可能不触发、小窗口就触发;③可手动改:tools_search.enabled = auto(默认按 10%)/ on(只要有就强制启用)/ off(永不,全部直发),阈值 threshold_pct 也可调。

    背景:为什么要有「桥」?模型每次 API 调用,所有工具的完整 JSON schema 都要随请求发过去。接了上百个 MCP 工具时,这上百份 schema 每次请求都重复发一遍——又贵、又占满窗口、还分散注意力。所以 Hermes 用渐进式工具披露(tool search):不把长尾工具全列出来,只放三个桥接工具,按需暴露。

    桥作用
    tool_search按描述搜工具(「我想放音乐」→ 找到 spotify_play)
    tool_describe查某工具的完整参数 schema
    tool_call调——「帮我调名叫 spotify_play 的工具,参数是…」

    核心工具(terminal/read_file 等)永不藏,一直直接可见;只有长尾(MCP、非核心插件)躲桥后面;若这些可延迟工具本来就小(< 窗口 10%),整个机制直接跳过、工具表原样放行。

    模型一开始并不知道名字——先搜后调的三步舞蹈

    关键前提:spotify_play 不在模型可见工具表里,所以模型一开始根本不知道有这个名字。它是「先搜、后调」逐步发现的:

    1. 搜:tool_search(query="play music") → 桥返回一份 JSON,里面列出匹配的工具名字 + 描述——模型到这里才学到有个工具叫 spotify_play。

    2. 查(可选):tool_describe(name="spotify_play") → 返回它的完整参数 schema,模型才知道 arguments 该怎么填。
    3. 调:tool_call(name="spotify_play", arguments={...}) → 此时 name 是从第 1 步搜索结果里拄回来的。
    所以那个 name 参数不是模型凭空写的——是它把 tool_search 结果里的名字照抄回来。这也是「省 token」的真正实现:不用的工具模型从头到尾不知道它的名字和 schema,自然一个字节都不占;只有真要用时才把那一个工具拉进上下文。

    为什么必须「剥桥」——两个 OpenClaw 教训。OpenClaw 是相关上游项目(openclaw/openclaw),踩过两个坑,Hermes 拿来当硬约束:① 观察者必须看到真实工具名(守卫/显示/轨迹不能只看到通用桥名);② 工具目录必须每轮从实时注册表重建,别按会话缓存(否则和真实工具漂移、静默丢工具)。下面分别展开。
    教训 ①:观察者必须看到真实工具名——不剥桥会坏掉什么

    剥桥发生在 tool_executor 里、在所有下游关卡之前。如果偷懒不剥,下游每一个「观察者」看到的都是同一个通用名 tool_call,一连串东西会失灵:

    观察者不剥桥的后果
    工具守卫(before/after_call)守卫按工具跟踪行为(如「同一工具连调 3 次没进展就拦」)。全叫 tool_call → 所有延迟工具塌成一个桶,守卫分不清 spotify_play 和 delete_everything,策略全废。
    审批 / 策略「某某工具需要人工批准」这类按工具名的规则,名字是通用桥就永远触发不了。
    用户显示 / 活动流CLI 里刷一串 ⚡ tool_call,你根本看不出它到底在干嘛。
    轨迹(trajectory)为回放/训练保存的记录里,每个延迟调用都是 tool_call → 分析和训练时毫无信息量。
    post 钩子 / 插件观察到的是桥名,没法归因到具体工具。

    所以 Hermes 在两处都做还原:执行链在解析阶段就把 function_name 换成真实名;显示层与轨迹记录器也各自 unwrap 一遍——保证无论从哪个角度看,都是真实工具而非桥。

    教训 ②:catalog 每轮重建 vs 「安全的缓存」

    坑:OpenClaw 的 cron 回归(#84141)把可搜索的工具目录按会话缓存。一个长会话里,实时工具注册表会变(MCP 服务器重连换了工具列表、插件动态加载/卸载),而缓存的目录还是旧的 → 两种故障:搜不到其实存在的新工具(静默消失),或搜到已经没了的旧工具、一调就失败。

    Hermes 的解法——目录无状态,每轮从当前工具表重算(tools/tool_search.py):

    # 每次装配工具表都跑一遍,不跨轮缓存:
tokens = estimate_tokens_from_schemas(deferrable)   # 估 schema 占多少 token
if should_activate(config, tokens, context_len):    # 阈值门:≥10% 才启用
    visible, deferrable = classify_tools(tool_defs)  # 从"当前"工具表分类
    catalog = build_catalog(deferrable)              # 现建目录(工具数<500,很便宜)
    但也不是完全不缓存——而是「安全的缓存」。作用域集 _tool_search_scoped_names 确实缓存在 agent 上,但缓存键绑定注册表的「代次」计数 _registry._generation(外加 enabled/disabled 工具集): 
    cache_key = (registry._generation, frozenset(enabled), frozenset(disabled))
if cached and cached_key == cache_key: return cached   # 命中
# 否则重算并存回
    注册表一变动(MCP 重连会递增 _generation)→ 缓存键变了 → 自动失效、下次重算。所以它是「随注册表变化自动作废的记忆化」,不是 OpenClaw 那种「会话生命周期内一直不变」的危险缓存。常态命中就一次字典查,几乎零成本;该刷新时一定刷新。这正是从教训 ② 提炼出的正确姿势:可以缓存,但必须能感知底层变化并作废。
    为什么必须补一道作用域门:解包后是直接分发底层工具,绕过了 handle_function_call 里那道「这工具在本会话授权范围内吗」的检查。若不在这里补判,子 agent / kanban worker / 受限网关会话就能借桥调到根本没授权给它的工具。所以对照 _tool_search_scoped_names(agent)(该会话「延迟加载工具」的授权子集)校验:不在集内直接拦。这个集合缓存在 agent 上,仅当工具注册表「代次」变化(如 MCP 服务器重连)才刷新,常态是一次字典查。
    tool_search 怎么搜:BM25 关键词检索(不是模糊、不是语义)

    搜索不是字符级模糊匹配、也不是语义/向量检索,而是 BM25 关键词相关性排序(搜索引擎/Elasticsearch 那套)。

    • 索引什么:每个可延迟工具拼成一个文本块 = 工具名(下划线/点/连字符拆成词) + 描述 + 顶层参数名(schema 主体故意不索引,只加噪声)。分词是 [A-Za-z0-9]+ 切词 + 小写。
    • 打分 BM25 三因子:TF(查询词在某工具里出现越多→分越高)× IDF(词在所有工具里越罕见→权重越大,常见词几乎不加分)× 长度归一(短文本命中更值钱,防长描述靠字多刷分)。取前 limit(默认 5)。
    • 子串兜底:若查询词在每个工具里都出现(IDF≈0,BM25 全 0,如 catalog 全是 github_* 时搜 "github"),回退到工具名子串匹配保底。
    别和前面的「修复工具名」混了:_repair_tool_call 用 difflib 字符级模糊(redfile→read_file)——名字已知、只是打错了,按「长得像」纠错;tool_search 用 BM25 关键词——工具未知,按「意图/描述的词」找相关工具。用途不同、算法不同。
    会不会漏掉真正想调的工具?

    会,原则上可能漏——这是渐进式披露天生的取舍(用召回率换 token)。但设计把风险压得很小,几道防线都有真实参数:

    1. 阈值门(最关键):默认 enabled="auto" + threshold_pct=10%——只有当可延迟工具会吃掉上下文窗口 ≥10% 时才启用披露,否则整个工具表原样全发、一个都不藏。所以披露只在「工具多到全发很贵」时才登场;工具少 → 全可见 → 不可能漏。
    2. 核心工具永不藏:terminal/read_file 等永远直接可见,从不进搜索。
    3. 能反复搜 + 告知总数:返回带 total_available、桥描述明写「还有 N 个工具」,模型可换关键词重搜或把 limit 从默认 5 调到最高 max_search_limit=20(上限 50)。
    4. BM25 + 子串兜底:尽量别返回空(见上)。
    残留风险(诚实地说)仍在:
    • 词对不上:模型搜「放音乐」,工具描述只有英文「Play a track」——按词切分不跨语言,可能不命中。
    • 排在第 6 名开外:默认只回前 5,真想要的排第 6 就看不到,除非复搜或调大 limit。
    • 模型压根没想到去搜:若它不知道「可能有这么个工具」,可能干脆不搜、直接说做不了。
    结论:能漏,但被框住了——只在工具多到「全发很贵」时才启用、核心永不藏、可反复搜且知道总数,用「小概率召回损失」换「不必每轮为几百个 schema 付费」。这是刻意的成本/召回权衡,不是 bug。
    在意「一个都不能漏」的话,可在 config 把 tools_search.enabled 设 "off"(全部直发),或调低 threshold_pct、调高 search_default_limit 更激进地披露。

  4. 工具请求中间件

    _apply_tool_request_middleware_for_agent → hermes_cli.middleware.apply_tool_request_middleware

    是什么:给插件一个在工具真正执行前改写参数的机会。中间件是插件通过 ctx.register_middleware("tool_request", cb) 注册的回调链;每个回调拿到 (tool_name, args),可以返回 {"args": {...}} 把参数替换掉,替换后的参数会被后面的钩子、守卫、审批、执行全部看到。典型用途:统一注入默认值、脱敏/重写路径、按策略改参数、纯记录。

    Hermes 的中间件是个对称的 2×2 家族([hermes_cli/middleware.py](hermes_cli/middleware.py#L20-L23)):

    request(改输入)execution(包调用)
    工具侧tool_request ← 本步tool_execution(在真正执行那步包一层)
    模型侧llm_request(第 3 篇改 api_kwargs)llm_execution(第 3 篇包 API 调用)

    所以「和第 3 篇同源」是字面意义的对称——把 LLM 那两个换成工具版就是这里。request 型改的是「输入 payload」(声明式:返回新 args);execution 型包的是「调用本身」(拿到一个 next_call,可在其前后计时/观测/干预)。

    request 这一趟怎么跑(apply_tool_request_middleware):

    # ① 没注册任何 tool_request 中间件 → 快路径:原样返回,changed=False,空 trace(零开销)
if not _has_middleware("tool_request"):
    return Result(payload=args, changed=False, trace=[])

# ② 深拷贝一份(不可深拷贝的对象退化成浅拷贝),原件留底
original = _safe_copy(args); current = _safe_copy(original); trace = []

# ③ 逐个跑注册的中间件;谁返回 {"args": {...}} 就替换 current,并记一条 trace
for result in _invoke_middleware("tool_request", tool_name=name,
                                 args=current, original_args=original, **ctx):
    if isinstance(result, dict) and isinstance(result.get("args"), dict):
        current = _safe_copy(result["args"]);  trace.append(_trace_entry(result))

return Result(payload=current, original_payload=original,
              changed=bool(trace), trace=trace)
    几个设计点:①快路径零开销——没插件注册就直接原样返回,常态不花一分钱;②深拷贝防污染——在拷贝上改,原始 args 留底,坏插件改不脏实参、还能对比出改了什么;③整段 try/except 兜底(在 _apply_tool_request_middleware_for_agent 里)——中间件抛错就退回原始 args + 空 trace,绝不因为一个插件出错而卡住工具调用;④trace 留痕——每步改动记进 middleware_trace,一路带到 post 钩子,便于审计「谁在什么时候把参数改成了什么」。

    举个真实例子(官方文档里的 normalize_terminal_workdir):把每个 terminal 调用约束到一个固定工作目录(除非模型自己指定了)——

    def register(ctx):
    ctx.register_middleware("tool_request", normalize_terminal_workdir)

def normalize_terminal_workdir(**kwargs):
    if kwargs.get("tool_name") != "terminal":       # ← 只管 terminal
        return None                                  # 返回 None = 不改,原样放行
    args = dict(kwargs["args"])                       # 拷一份再动
    args.setdefault("workdir", "/tmp/hermes-demo")    # 没指定 workdir 就补默认
    return {"args": args,                             # 替换后的参数
            "source": "middleware-demo",
            "reason": "defaulted terminal workdir"}   # source/reason → 进 trace 供审计

    读懂这个例子的三个点:

    • 自己挑要不要管:回调对每个工具都会被调,但它一看 tool_name != "terminal" 就 return None——返回 None(或不带 {"args"})= 这个工具我不改,原样过。所以一个中间件只对它关心的工具生效。
    • 它是个策略执行点:因为跑在守卫/审批/执行之前,下游看到的都是被改写后的 workdir——于是「所有 terminal 都被关进 /tmp/hermes-demo」成了硬约束,模型绕不过。这就是沙箱收敛的一种落地方式。
    • 额外字段进 trace:返回里除了 args,source/reason 会被记成一条 trace,最后能在审计里看到「是 middleware-demo 把 workdir 默认成了 /tmp/hermes-demo」。

  5. 拦截评估(短路链,任一拦下就不执行)

    这一步在干嘛:在真正执行工具前,连过三道否决闸,任一道说「不行」,就用一个拦截结果 block_result 顶替真实执行——工具根本不跑,那句「为什么被拦」直接作为 tool 结果回给模型,让它换个做法。三道闸短路:前一道拦下就不判后面的。

    if _ts_scope_block is not None:            # ① 越权(上一步作用域门判的)
    block_result = _ts_scope_block                     # error_type="tool_scope_block"
else:
    block_message = get_pre_tool_call_block_message(...)   # ② 插件 pre 钩子
    if block_message is not None:
        block_result = {"error": block_message}            # error_type="plugin_block"
    else:
        decision = agent._tool_guardrails.before_call(name, args)  # ③ 工具守卫
        if not decision.allows_execution:
            block_result = agent._guardrail_block_result(decision) # error_type="guardrail_block"
            blocked_by_guardrail = True

    三道闸各管一层,顺序从「最根本」到「最行为化」:

    闸管什么拦的依据
    ① 越权
    (授权层)    		这工具在本会话授权范围内吗Tool Search 作用域门已判:底层工具不在会话授权子集 → 拦(防受限会话借桥越权)
    ② 插件 pre 钩子
    (第三方策略)    		企业合规 / 限流 / 审批 / 自定义安全插件的 pre_tool_call 回调返回 {"action":"block","message":...};或线程白名单里没这工具 → 拦
    ③ 工具守卫
    (行为策略·防打转)    		模型是不是在原地空转见下(反复失败 / 只读无进展 / 同工具连败)
    第 ③ 道「工具守卫」到底防什么(before_call)

    agent/tool_guardrails.py · before_call

    守卫是防死循环的——它按「工具名 + 参数」算一个签名,盯着模型有没有在同一个动作上反复空转,hard_stop_enabled 开启时才拦。两种拦法:

    • 反复精确失败(repeated_exact_failure_block):完全相同的调用(名字 + 参数一字不差)已经失败了 N 次(exact_failure_block_after)→ 拦,告诉模型「别再原样重试了,换策略或说明卡点」。
    • 只读工具无进展(idempotent_no_progress_block):对幂等/只读工具,同一个调用连续返回同样结果 N 次(no_progress_block_after)→ 拦,「结果都一样,别再重复搜了,用已有结果或换个查法」。

    (配套的 after_call 还会在「同一工具本轮失败 N 次」时升级为 halt,整轮停手——那是执行之后的事,见收尾篇的守卫。)

    怎么实现的(简化代码):守卫维护三本「账」,before_call 查账决定拦不拦,after_call 记账:

    # ── 阈值(默认值)。注意 hard_stop 默认关,平时只警告不拦 ──
hard_stop_enabled        = False   # ← 默认 False!需在 config 显式开
exact_failure_block_after = 5      # 同一调用失败 5 次 → block
no_progress_block_after   = 5      # 只读调用返回同结果 5 次 → block
same_tool_failure_halt_after = 8   # 同一工具本轮失败 8 次 → halt(after_call)

# 签名 = (工具名, sha256(规范化参数))——不可逆,不存原始参数
sig = ToolCallSignature.from_call(name, args)

def before_call(name, args):
    if not hard_stop_enabled: return ALLOW            # 默认直接放行(只在 after_call 发警告)
    if exact_failure_counts[sig] >= exact_failure_block_after:
        return BLOCK("repeated_exact_failure_block")  # 反复精确失败
    if is_idempotent(name):                           # read_file/web_search/... 才查这条
        result_hash, repeat = no_progress[sig]
        if repeat >= no_progress_block_after:
            return BLOCK("idempotent_no_progress_block")  # 只读无进展
    return ALLOW

def after_call(name, args, result, failed):
    if failed:
        exact_failure_counts[sig] += 1                # 失败:精确签名计数 +1
        same_tool_failure_counts[name] += 1           # 同工具计数 +1
        if same_tool_failure_counts[name] >= same_tool_failure_halt_after:
            return HALT("same_tool_failure_halt")     # 同工具连败 → 整轮停
    else:
        exact_failure_counts.pop(sig)                 # 成功:清掉失败账
        same_tool_failure_counts.pop(name)
        if is_idempotent(name):                       # 只读工具:记「同结果连续几次」
            h = result_hash(result)
            prev = no_progress[sig]
            no_progress[sig] = (h, prev[1]+1 if prev and prev[0]==h else 1)
    关键:默认不拦,只警告。hard_stop_enabled 默认 False——交互式 CLI/TUI 下,守卫只在 after_call 发一条温和警告(「你好像在打转,看看错误换个法子」)塞进工具结果里提醒模型,但不阻止执行。真正的 block/halt 是显式 opt-in(在 config.yaml 的 tool_loop_guardrails.hard_stop_enabled 打开断路器)才生效,阈值也都可配。这样默认体验是「点一下、别硬来」,需要强约束(如无人值守的 cron/kanban)时再开硬停。
    被拦≠静默丢弃:三道闸不管哪道拦下,都会发一条终态 post_tool_call 钩子(status="blocked" + 对应 error_type + 原因),这样遥测、网关、轨迹都看得到「这次调用发生过、被拦了、为什么」,不会变成观测黑洞。而 block_result 本身会作为 tool 结果回灌给模型——模型读到「你越权了 / 被策略拦了 / 你在原地打转」,才能自我纠正。

  6. 检查点预检(只对「将会执行」的工具)

    tools/checkpoint_manager.py · ensure_checkpoint / get_working_dir_for_path

    这一步在干嘛:在工具动手改文件系统之前,先把即将被改的目录快照一份存进影子 git 仓(和你项目里真正的 .git 完全隔离,见第 2 篇),这样万一改坏了能一键回滚到「动手前」的样子。「预检」= 必须在改动之前拍快照,拍晚了就没意义了。

    if block_result is None:                    # 只给"会真执行"的工具建点
    if function_name in {"write_file","patch"} and ckpt.enabled:
        ckpt.ensure_checkpoint(work_dir_of(path), f"before {name}")
    if function_name == "terminal" and ckpt.enabled and _is_destructive_command(cmd):
        ckpt.ensure_checkpoint(cwd, f"before terminal: {cmd[:60]}")   # rm/mv/dd/git reset...
    什么工具才建点(不是所有)
    • write_file / patch:一定改文件 → 总是建点。工作目录由目标 path 推出来(见下)。
    • terminal:只有命令命中破坏性正则 _is_destructive_command(rm/rmdir/cp/mv/sed -i/truncate/dd/shred/git reset|clean|checkout)才建点——普通命令(ls/cat/python x.py)不建,免得噪声太大。
    • 其它(如 execute_code、只读工具)不建点。
    工作目录怎么定(get_working_dir_for_path)

    不是「文件所在的那个文件夹」,而是往上找项目根:从文件目录逐级向上,遇到项目标记(.git、pyproject.toml、package.json、Cargo.toml、go.mod、Makefile…)就停,把那个目录当工作目录来快照。找不到就用文件自己的目录。这样快照的粒度是整个项目,回滚时能一起还原相关文件。

    ensure_checkpoint 的几道闸(为什么不会拖慢/搞乱)
    • 每轮每目录只拍一次:_checkpointed_dirs 集合去重——同一轮里 write_file + patch 都改同一个项目,只有第一次拍(「动手前」状态已经存下了),后面直接跳过。该集合每轮清空。
    • 太宽的目录不拍:是 / 或 $HOME → 跳过,绝不把整个家目录快照进去。
    • 太大的目录不拍:文件数超过上限 _MAX_FILES → 跳过(不为一次改动去快照一棵巨树)。
    • 没装 git 就跳过:影子仓靠 git,git 不在 PATH 就静默禁用。
    • 永不抛错:整个过程绝不因失败而中断工具调用——出错只静默记日志。检查点是「锦上添花的安全网」,不能反过来卡住正常工作。
顺序有讲究:拦截评估必须在检查点预检之前——得先知道「这工具到底会不会执行」,才决定要不要为它建检查点。被拦的工具不建点、不占去重槽,省得白白污染检查点历史。

并发执行的机制

解析完,把 block_result is None(即真要跑)的调用交给线程池。这里有几处很讲究的工程细节:

  1. 线程池 + 上限 8

    ThreadPoolExecutor(max_workers=min(n, _MAX_TOOL_WORKERS=8))

    并发度是「工具数」和 8 的较小值。结果放进按原始索引开好的 results[num_tools] 槽位,被拦的槽位提前填好拦截结果。

  2. 线程上下文传播(安全关键)

    tools/thread_context.py · propagate_context_to_thread

    要解决的问题:一个刚起的线程池 worker 是「白纸」——它带着空的 contextvars.Context、也没有线程本地的审批/sudo 回调。工具一旦在这种线程里跑,会悄无声息地丢掉两样东西,各自对应一个真实的安全漏洞:

    丢了什么后果
    审批的会话/平台 ContextVars(tools.approval / gateway.session_context)网关会话被误判成「非交互」→ 落进 check_dangerous_command 的自动批准分支 → 危险命令不弹审批就跑了(#33057 / #30882)
    线程本地的 CLI 审批/sudo 回调(tools.terminal_tool)prompt_dangerous_approval 够不到用户 → 审批框根本弹不出来(GHSA-qg5c-hvr5-hjgr / #15216)

    两者都意味着:worker 线程里的工具可能没经用户同意就执行危险命令。这个 helper 把「父线程捕获 → worker 安装 → 退出清除」的生命周期收成一处:

    def propagate_context_to_thread(target):
    ctx = contextvars.copy_context()          # ① 父线程:快照全部 ContextVars
    parent_approval_cb = get_approval()       #    捕获当前审批回调
    parent_sudo_cb     = get_sudo()           #    捕获当前 sudo 回调

    def _runner(*args, **kwargs):
        def _inner():
            set_approval(parent_approval_cb)  # ② worker:装回审批/sudo 回调
            set_sudo(parent_sudo_cb)
            try:
                return target(*args, **kwargs) #    真正跑工具
            finally:
                set_approval(None)            # ③ 无论如何退出都清除
                set_sudo(None)                #    (线程会被回收复用,不能留残留)
        return ctx.run(_inner)                # 在"复制出的上下文"里跑 → ContextVars 都在
    return _runner

    三个关键动作:

    • ① 在父线程快照:copy_context() 抓下本轮所有 ContextVars(谁在问、哪个会话/平台);同时抓下当前审批/sudo 回调函数。必须在父线程调,这时上下文才是完整的。
    • ② 在 worker 里安装 + ctx.run:worker 通过 ctx.run(_inner) 在复制出的上下文里执行——ContextVars 全都在;_inner 再把审批/sudo 回调装进这条线程的线程本地。两套机制缺一不可:ContextVars 带「会话身份」(审批弹给谁),回调是「真正弹框/入队的函数」。
    • ③ finally 里必清除:线程池会回收复用线程,不清的话下一个无关任务会继承一个指向已销毁 CLI 的陈旧回调 → 弹错会话、甚至崩。所以无论正常/异常退出都 set_approval(None)。
    fail-closed(失败即拒)是这段的安全底线:万一回调安装失败,就让回调保持 None——这恰恰是安全结果:交互场景下没有回调时 prompt_dangerous_approval 直接拒绝危险命令,网关审批队列没有 notify 回调时会阻塞等待。也就是说,传播出 bug 的方向永远是「拒绝/卡住」,绝不会是「静默放行」。安全代码的正确默认就该这样。

  3. worker 登记自己,好被中断

    tools/interrupt.py(按线程记中断) · run_agent.interrupt(扇出) · tool_executor._run_tool(登记)

    背景:中断是「按线程」记的。Hermes 的中断信号存在一个线程 id 集合 _interrupted_threads 里;工具靠 is_interrupted() 查当前线程的 id 在不在里面。问题来了:并发工具跑在线程池 worker 各自的 tid 上,不是 agent 主执行线程。所以 /stop 只把主线程标记为中断,够不到正在 worker 上跑的工具(比如一条卡在网络 I/O 上的 terminal 命令),它只能干等到自己超时。

    解法是「登记 + 扇出」两半配合:

    # 一半:worker 一进来就登记自己的 tid(tool_executor._run_tool)
_worker_tid = threading.current_thread().ident
with agent._tool_worker_threads_lock:
    agent._tool_worker_threads.add(_worker_tid)          # ① 登记
if agent._interrupt_requested:                            # ② 补漏:登记前就被中断了?立刻自我标记
    _set_interrupt(True, _worker_tid)
set_activity_callback(agent._touch_activity)              # ③ 线程本地活动回调(见下)
try:
    ... 真正执行工具 ...
finally:
    with agent._tool_worker_threads_lock:
        agent._tool_worker_threads.discard(_worker_tid)  # ④ 摘除登记
    _set_interrupt(False, _worker_tid)                    # ⑤ 清掉中断位(关键!)

# 另一半:interrupt() 把中断扇出到每个登记在册的 worker(run_agent.interrupt)
for wtid in list(agent._tool_worker_threads):
    _set_interrupt(True, wtid)                            # 每个 worker tid 都置位
    • ① 登记 + ⑥ 扇出:worker 把 tid 记进 _tool_worker_threads;agent.interrupt() 遍历这个集合,给每个在跑的 worker 单独置中断位 → 工具下次 is_interrupted() 就返回 True,当场中止,不用等超时。
    • ② 补漏(竞态):如果中断发生在「扇出已经跑过、但本 worker 还没登记」的空档,扇出会漏掉它。所以 worker 登记后立刻自查 _interrupt_requested,是就给自己补一刀。
    • ⑤ finally 必清中断位(最容易忽略的坑):线程池会回收复用线程。若不清,tid X 上留着中断位,下一个无关任务被调度到 X 上,一进来 is_interrupted() 就是 True → 无缘无故被中止。所以必须在 finally 里 _set_interrupt(False, tid)(且必须是 finally——KeyboardInterrupt/CancelledError 这类会绕过 except Exception)。
    ③ 顺带装了个「活动回调」:worker 里 set_activity_callback(agent._touch_activity) 是线程本地的——主线程的回调 worker 看不见。长跑的 terminal 命令用它持续报活(心跳),让网关的不活动监控知道「我还在干活、别杀我」。不装的话,worker 里一条几分钟的命令不报活,可能被误判成卡死而中止。

  4. 真正执行 + 失败判定

    agent._invoke_tool → model_tools.handle_function_call → tools/registry

    执行:worker 调 _invoke_tool,它进 handle_function_call——这是总分发器:先按 schema 把参数类型归一(coerce_tool_args,如字符串 "42"→数字 42),再路由到工具注册表(终端 / 文件 / 网页 / MCP…),拿回一个 JSON 结果字符串。所有工具的返回都统一成字符串,便于回灌进消息。

    异常怎么兜:

    • KeyboardInterrupt → 触发 agent.interrupt(),把结果记成「已取消」(status="cancelled"),不当失败。
    • 其它任何异常 → 包成 Error executing tool '<name>': <e> 字符串(带 exc_info 落日志),不让一个工具的崩溃掀翻整批。

    失败判定 _detect_tool_failure:拿到结果后判定这次是成功还是失败,返回 (is_failure, 后缀标签)。它不是只看有没有抛异常——很多工具「正常返回」但内容是失败,得按工具语义看:

    工具怎么算失败
    terminalexit_code != 0(非零退出是规范失败信号)→ 标 [exit 1] 或 [错误信息]
    memorysuccess=false 且含「exceed the limit」→ 标 [full](区分「存满了」和真错误)
    任意工具JSON 里有 {"error": ...} / success=false → 失败 + 截断的错误信息
    文件类写入/patch 成功落地的结果 → 明确算成功(即使文本里有 "error" 字样也不误判)

    这个 is_failure 有两个去处:①日志(打印「tool X failed/completed」);②喂给守卫的 after_call——失败就累加「精确失败/同工具失败」计数,正是上面第 ③ 道守卫防打转的数据来源。

  5. 批超时 + 中断兜底

    HERMES_CONCURRENT_TOOL_TIMEOUT_S(默认 420s) · concurrent.futures.wait 轮询

    整批并发不能无限等——主线程用 concurrent.futures.wait(futures, timeout=…) 轮询着等所有 future 完成,每圈最多等 5 秒,好在这些间隙里处理超时、中断、心跳三件事。

    ① 批超时(默认 420 秒)

    截止时间来自 HERMES_CONCURRENT_TOOL_TIMEOUT_S(默认 420s;设 ≤0 关闭,永不超时)。这个值刻意高于单个工具自己的超时(如 web_extract 360s),免得批级看门狗误杀一个「慢但正常」的工具。到点仍没跑完:

    if time.monotonic() >= deadline:
    for f in not_done: f.cancel()          # 取消还没开始的 future
    for tid in worker_tids:
        _set_interrupt(True, tid)           # 给还在跑的 worker 发中断信号
    abandon_executor = True; break          # 放弃线程池,记下哪些超时了
    ② 用户 /stop(执行中中断)

    轮询间隙若发现 agent._interrupt_requested:取消尚未启动的 future,再给已在跑的工具 3 秒宽限——concurrent.futures.wait(not_done, timeout=3.0),让它们察觉中断位(上一步的登记/扇出机制)后优雅退出,然后 break。有中断检查的工具(terminal/execute_code)会当场停;没有的(read_file/web_search)跑完这最后一下。

    ③ 每 ~30 秒报活(心跳)

    长批可能跑好几分钟。等待循环里每约 30 秒(_conc_elapsed % 30 < 6,6=5s 轮询的 6 圈)_touch_activity 一次,告诉网关的不活动监控「这批工具还在跑,别当我卡死」。

结果处理与回灌

不管并发还是串行,拿到每个工具的结果后,都要经过同样的「回灌」处理。其中最讲究的是预算裁剪 + 溢出落盘——单独拆出来讲。

三层工具输出预算

tools/budget_config.py · tools/tool_result_storage.py

工具结果可能极大(一个 terminal / web_search 能吐几十万字符),直接塞进上下文会撑爆窗口、烧钱、还挤掉真正有用的历史。Hermes 用三层把它压住:

层粒度做什么
第 1 层注册表每个工具注册时声明 max_result_size_chars(web/terminal 等注册 100K)
第 2 层单条结果maybe_persist_tool_result:单条超阈值 → 全文落盘,消息只留「预览+文件路径」
第 3 层整轮汇总enforce_turn_budget:一轮所有工具结果总和超预算 → 从最大的开始落盘,直到降到预算内

第 2 层(单条落盘):

def maybe_persist_tool_result(content, tool_name, tool_use_id, env, config, threshold=None):
    thr = threshold if threshold is not None else config.resolve_threshold(tool_name)
    if thr == inf or len(content) <= thr:
        return content                       # 不超阈值,原样返回
    preview, has_more = generate_preview(content, max_chars=config.preview_size)
    # 通过 env.execute() 写文件 —— 任何后端都行(本地/Docker/SSH/Modal/Daytona)
    if env and _write_to_sandbox(content, remote_path, env):
        return _build_persisted_message(preview, has_more, len(content), remote_path)
    # 写不了沙箱 → 兜底:内联截断,只留预览
    return f"{preview}\n\n[Truncated: 原始 {len(content):,} 字符,未能落盘]"

第 3 层(整轮控总量):把一轮里还没落盘的结果按大小排序,从最大的开始落盘,边落边扣总量,直到 total ≤ turn_budget 为止(已落盘的跳过,不重复处理)。

「落盘」不是不给模型返回——是返回「预览 + 全文路径 + 怎么读」。超阈值时,全文写进沙箱文件,返回给模型的是一个 <persisted-output> 说明块: 
<persisted-output>
This tool result was too large (348,201 characters, 340.0 KB).
Full output saved to: /sandbox/.hermes/tool_results/call_abc123.txt
Use the read_file tool with offset and limit to access specific sections.

Preview (first 2000 chars):
<开头一段预览……>
...
</persisted-output>
所以模型这轮拿到的是开头预览 + 一个「全文在这里,自己按需去读」的指针,而不是把几百 KB 硬灌进上下文。它要细节就用 read_file(path, offset, limit) 精准取那一段——既不撑爆窗口、也不丢数据(全文一直在磁盘上)。这也是为什么落盘排在「内联截断」之前:截断会真丢掉后半,落盘是「预览 + 可回取」,只有连沙箱都写不了时才退回纯截断兜底。

预览怎么生成(generate_preview)

那段预览就是「取结果开头一段,再回退到最近的整行」——不是摘要、不是首尾抽样,纯粹是开头一瞥:

def generate_preview(content, max_chars):
    if len(content) <= max_chars: return content, False   # 没超 → 整个就是预览
    truncated = content[:max_chars]        # ① 先硬取开头 max_chars 个字符(=preview_size)
    last_nl = truncated.rfind("\n")        # ② 找这段里最后一个换行
    if last_nl > max_chars // 2:           # ③ 只要它不太靠前(过了一半)
        truncated = truncated[:last_nl+1]  #    就切到整行结束,避免半句截断
    return truncated, True                 # has_more=True → 块末尾那个 "..." 的来源
  • 取头:预览是结果最前面那段,给模型一个「大概长啥样」的一瞥。
  • 整行回退:直接在第 max_chars 处切多半会把一行砍成半句(JSON/日志断行),所以回退到最近的换行、以完整一行结尾,读着干净。
  • 半行保护:只有换行过了一半才这么切;若开头是个超长无换行的单行,rfind 找到的换行太靠前,照切会把预览砍到只剩一点——这时宁可保留硬截断的 max_chars。
  • 两处共用:落盘和内联截断兜底都用它,保证「无论落盘还是截断,模型看到的开头一致」。

预算随模型上下文窗口缩放

budget_for_context_window(context_length) · #23767

阈值不是死数。大模型(200K+ token)沿用固定默认(单条 100K 字符 / 整轮 200K 字符);但中途切到小窗口模型(如 65K token)时,一条 100K 字符的结果、或 200K 字符的整轮预算(≈50K token)可能单独就把窗口撑爆。所以按窗口按比例缩:

window_chars = context_length × 4          # ~4 字符/token
per_result   = window_chars × 0.15         # 单条最多占窗口 15%
per_turn     = window_chars × 0.30         # 整轮最多占窗口 30%
# 上钳到历史默认(大模型完全不变)、下floor(小模型也留得住可用预览)
per_result = clamp(per_result, 8_000,  100_000)
per_turn   = clamp(per_turn,  16_000,  200_000)
为什么分子/分母都留余量(15% / 30% 而非 100%):工具输出不是窗口里唯一的东西——system prompt、工具 schema、对话历史、模型自己的回复都在抢空间,所以刻意压在远低于 1.0 的比例。_budget_for_agent 在每轮开头算一次这个预算,避免每条结果都重算。

其余回灌步骤

  • 多模态结果:像 computer_use 截图返回的是带 _multimodal=True 的信封(content parts + 文本摘要),按提供商能力决定发图还是发文本摘要。
  • 按原序 append:make_tool_result_message(name, result, tool_call_id) 包成 {"role":"tool", "tool_call_id":..., "content":...},严格按原始 tool_calls 顺序写回 messages。
  • post_tool_call 钩子:发后置钩子给插件观察/记录,网关据此推送「工具完成」事件。
  • 增量落库:_flush_session_db_after_tool_progress 每完成一个工具就刷一次 SQLite——因为工具可能有副作用(重启/杀进程),得保证长工具链中途崩溃也能恢复到最近进度。

回灌后:带着结果再问一次

所有工具结果 append 进 messages 后,执行函数返回,主循环 continue 回到顶部。下一圈第 3 篇的清洗会把这些新增的 tool 消息一起发给模型——于是模型「看到」了工具结果,继续推理。这就是多轮工具调用能连续进行的根本。

同一轮的多个工具:一起回,不分批

同一轮里发的 N 个工具,会全部执行完、按原序把 N 条 tool 结果一起塞进下一次 API 调用——模型一次看全。「分批」只发生在跨轮(模型看完这批再发新一批),不在同一轮内部。

sequenceDiagram

两个层面别混:①消息结构——每个 tool_call 必须有一条 tool_call_id 对应的结果,所以 3 个工具 = 3 条独立 tool 消息(不合并);②网络层面——这 3 条在同一次 create() 请求里一起发。所以并发省的是「3 个工具的执行时间」,不是「3 次 API 往返」——API 往返永远每轮一次。这也是为什么哪怕并发,也要等整批齐了才发下一次请求:API 契约要求 assistant 的 tool_calls 后面必须紧跟它所有结果,不能只回一部分。

  • 解析与所有关卡串行且确定性,只有真正执行才并发(线程池 ≤8);同一轮多个工具等齐后一次请求一起回(N 条 tool 消息),分批只发生在跨轮。
  • 关卡链:Tool Search 解包+作用域门 → 中间件 → 拦截评估(越权/插件/守卫) → 改文件系统前打检查点。
  • Tool Search 默认不启用——只有可延迟工具 schema ≥ 窗口 10% 才触发;核心工具永不藏。用 BM25 关键词搜、模型「先搜后调」三次往返发现工具。
  • 并发的安全命脉是 propagate_context_to_thread——把审批回调 + ContextVars 带进 worker(GHSA-qg5c-hvr5-hjgr),失败即拒(fail-closed);worker 登记 tid 才能被中断扇出。
  • 工具守卫默认只警告不拦(hard_stop_enabled opt-in),按「名字+参数签名」防反复失败/只读无进展的死循环。
  • 结果按原序回灌守角色交替;超大结果三层预算 + 落盘(返回「预览+全文路径+用 read_file 读」,非截断、可回取);每个工具后增量落库可崩溃恢复。
