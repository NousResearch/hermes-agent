收尾阶段

循环跳出后(拿到答复、被中断、或预算耗尽),还要做一连串收尾:落库、转换输出、组装结果字典、安排后台复盘。这些全集中在 finalize_turn。

先分清粒度:是「一轮」的收尾,不是会话结束

「收尾」收的是一轮(turn),不是把整个会话关掉。两个粒度别混:

概念指什么结束标志
一轮(turn)用户发一句话 → agent 循环调工具 → 直到吐出最终文本回复就是本篇的 finalize_turn
一次会话(session)很多轮累积起来的整段对话用户关掉/切会话才算完

所以 finalize_turn 是在这一轮的工具循环跑完、拿到最终回复之后触发的扫尾,对应「一问一答(含中间多次工具调用)这一个回合」的结束。收尾完,会话照样活着——用户再发一句就开启下一轮,又走一遍循环 + 收尾。会话状态、记忆、技能都留着给下一轮用。

一个易忽略的点:最后那步「记忆/技能复盘」是异步的。_spawn_background_review 起的是后台线程去「回头总结这轮的经验」,本轮该返回就返回、不等复盘做完——所以你不会因为「它在总结经验」而卡住下一句输入。这正是第 1 篇讲的「机制 B 后台复盘」。

收尾的入口

循环结束后,run_conversation 把一切交给一个函数,它的返回值就是整个 run_conversation 的返回值:

# agent/conversation_loop.py(循环之后)
result = finalize_turn(agent, final_response, messages, ...,
                       _should_review_memory=..., interrupted=..., ...)
return result
agent/conversation_loop.py → agent/turn_finalizer.py · finalize_turn()
finalize_turn 是同步、单一返回的(没有 await、没有提前 return),刻意镜像它替换掉的那段内联代码。

收尾都做了什么(按真实顺序)

  1. 没答复?先处理「迭代耗尽」

    if final_response is None and 预算/迭代耗尽: _handle_max_iterations(...)

    如果循环跑完了却没拿到 final_response(被工具循环占满、迭代/预算耗尽),不返回空,而是剥掉工具、再发一次请求让模型总结:

    if final_response is None and (api_call_count >= max_iterations
                              or iteration_budget.remaining <= 0):
    _turn_exit_reason = f"max_iterations_reached({api_call_count}/{max_iterations})"
    agent._emit_status("⚠️ 迭代预算耗尽 — 请模型总结")
    final_response = agent._handle_max_iterations(messages, api_call_count)  # 无工具的单次总结调用
    if os.environ.get("HERMES_KANBAN_TASK"):        # 若是 kanban worker:
        _kb._record_task_failure(..., outcome="timed_out")  # 记一次"超时失败",计入熔断计数
    kanban 那段是诚实上报:worker 因预算耗尽没干完,得让调度器知道(outcome="timed_out"),而且要计入 consecutive_failures → 触发 failure_limit 熔断。否则这种任务会每轮静默 block、被操作员自动放行、再 block,陷入无信号的死循环(#29747)。注意:模型这时已被剥掉工具,没法自己调 kanban_block,只能由收尾代替它上报。

  2. 算 completed(这轮到底算不算成功)

    这一步在给这轮盖章:「自然答完」还是「被逼收场」?关键背景在上一步——如果模型一直调工具、从没给文本答复,收尾会强行剥工具让它总结,于是 final_response 也有值了。所以「有答复」有两种来源:

    • A 自己答完的(正常):_turn_exit_reason = "text_response(finish_reason=stop)"。
    • B 被逼总结的(异常):_turn_exit_reason = "max_iterations_reached(90/90)"。

    两种都有答复,但 A 是「干完了」、B 是「没干完硬收场」。completed 就是把它俩分开:

    normal_text_response = _turn_exit_reason.startswith("text_response(")   # 是不是"正常文本收尾"
completed = (final_response is not None      # ① 有答复
             and not failed                  # ② 没被标失败
             and (api_call_count < max_iterations   # ③a 没到迭代上限
                  or normal_text_response))         # ③b 或 是正常文本结束

    ①②好懂,难点是 ③ 那个 or——它专为处理一个边界。逐种情形看:

    情形api_calls < maxnormal_textcompleted
    正常答完(没到上限)✅✅✅ 成功
    正好在最后一圈答完❌(=上限)✅✅ 成功(or 救的就是它)
    烧光迭代、被逼总结❌❌(是 max_iterations_reached)❌ 不算成功

    第 2 行是 or normal_text_response 的意义:模型恰好在允许的最后一圈给出正常文本答复,虽然 api_calls 已等于上限,但它真答完了,不该因「用满额度」被判失败。第 3 行才是真正的「被迫收场」——到了上限、又不是正常文本结束 → 两个条件都 False → completed=False。

    为什么会「api_calls == 上限 且 正常答完」同时成立?

    关键在计数时机:api_call_count 是每圈开头、发请求之前就 +1 的([第 2 篇的主循环](02-main-loop.html)):

    while (api_call_count < max_iterations and ...):   # 顶部判条件
    api_call_count += 1        # ← 进来就 +1,在发请求之前
    ... 发这一次 API 调用 ...

    以 max_iterations=90 走最后一圈:进循环时 api_call_count=89 → 判 89<90 ✅ 进得去 → 开头 +1 变 90 → 发第 90 次请求。这第 90 次的返回决定命运:

    • 返回文本(答完了) → _turn_exit_reason=text_response(stop),break。此刻 api_call_count=90,90<90 是 False,但 normal_text 是 True → or 救场 → completed ✅。这就是那个边界情形。
    • 又调工具 → 执行后回顶部,90<90 False → 退出循环,final_response=None → 被逼总结 → max_iterations_reached → completed ❌。

    所以「api_calls == 上限 且正常答完」= 模型把额度用到最后一格(第 90 次),而这最后一下正好正常答完。没有那个 or,它会因 90<90 为假被冤枉成没完成——明明答完了,只因用满额度。

    那为什么不直接写成 api_calls <= max、省掉 or?不行——<= 会把「被逼收场」也误判成成功。因为边界上那两种情况计数一模一样(都=90),区别只在 normal_text:
    第 90 次干了啥api_calls<=max现写法(< or normal_text)
    正常答完(文本)✅ True(对)✅ True(对)
    还在调工具→被逼总结✅ True(错!冤枉成成功)❌ False(对)
    <= 只看数字,而这两种数字相同、光靠比大小分不开——必须靠 normal_text_response 这个「是不是正常文本结束」的额外信号才能区分,所以不能塌成一个比较。(顺带:< 也更稳——万一将来 grace 之类让计数冲到 max+1,< 不会放行超额的,<= 边界更松。)
    谁在乎 completed:就是第 4 篇讲的——cron / kanban 拿 completed=False 判断「这任务没真干完」→ 重跑/改期/交人;CLI/网关一般照样把答复显示出来,但能据此知道「这是被截断的、不是圆满结束」。一句话:有答复 ≠ 成功;烧光迭代靠强制总结拿到的答复,completed=False。

  3. 容错收尾区(关键设计):三个易错操作各自兜底

    _cleanup_errors[] · #8049

    核心思想:答复已经算出来了,收尾只是「打扫卫生」——打扫时摔一跤,绝不能连答复一起丢。接下来三件事都可能抛异常,而它们对「答复本身」都不是必需的:

    操作干什么为什么会失败
    _save_trajectory把这轮完整过程存成轨迹文件(供回放/训练)文件 I/O 错、JSON 序列化碰到怪对象
    _cleanup_task_resources拆掉本任务的 VM + 浏览器(cleanup_vm + cleanup_browser;持久化沙箱跳过,交给空闲回收器)跟远程沙箱(Morph/Modal/Daytona)走网络超时/断连、浏览器进程拆除失败
    _persist_session把这轮消息写进 JSON 日志 + SQLiteSQLite 锁/磁盘满、序列化错

    它修的 bug(#8049):以前这三个里任一抛错,会直接冲出 run_conversation,把用户已经等着的 final_response 一起丢掉——子进程包装层只看到空 stdout、连 traceback 都没有,是一种静默数据丢失。现在每步独立 try,错误收集进 _cleanup_errors、绝不打断返回:

    _cleanup_errors = []
try: agent._save_trajectory(...)            # 存轨迹
except Exception as e: _cleanup_errors.append(f"save_trajectory: {e}")
try: agent._cleanup_task_resources(task_id) # 拆 VM/浏览器
except Exception as e: _cleanup_errors.append(f"cleanup_task_resources: {e}")
try: ...落库(见下)...
except Exception as e: _cleanup_errors.append(f"persist_session: {e}")

    三个 try 各自独立还有一层意思:一个失败不能连累后两个——存轨迹炸了,拆资源和落库照样要跑(否则会漏拆远程 VM、漏存会话)。

    要是三个全失败呢?

    照样不崩、答复照返回,三条错误全记进 cleanup_errors。但副作用轻重不同:

    哪个失败后果严重度
    _save_trajectory这轮轨迹没存(回放/训练数据缺一条)轻 —— 不影响答复与会话
    _cleanup_task_resourcesVM/浏览器没拆 → 资源泄漏中,但能自愈 —— 空闲回收器(terminal.lifetime_seconds 到点)会兜底拆
    _persist_session这轮没写进 SQLite/JSON 持久层重 —— 见下

    最该在意 _persist_session 也失败:答复本身没丢——完整 messages 是在返回的 result 字典里带回上层的,当前进程/CLI/网关手里有全套,照样能显示交付。但持久层缺了这一轮:之后若进程重启或异地 resume,durable 存储里就没有这轮记录、历史出现缺口。即「答案送到用户面前了,但没存档」。

    这正是设计的取舍哲学:三者都失败时,宁可「把答案交付出去 + 丢掉存档/清理」,也不要「答案和存档一起丢」。答复是用户真正要、且已花钱算出来的,必须优先保住;存档、清理、轨迹都是可事后补救/自愈的次要项。而且「都失败」是可见的——cleanup_errors 同时列出三条,上层看到 persist_session 在其中就能大声告警,而不是像修复前那样两眼一抹黑。
    铁律 + 可观测:无论收尾出什么错,用户等待的答复一定能返回;错误不是吞掉,而是挂到 result 的 cleanup_errors 上——上层据此能区分「干净完成」和「答复给了但收尾出过错」(比如轨迹没存上、VM 没拆掉),做监控告警或补救,但用户体验不受影响。这就是「非核心收尾降级、核心交付不受影响」的典型写法。

  4. 落库前的三处修补(保证存下的历史合法)

    agent._persist_session 之前

    直接把 messages 存下去可能留下不合法的消息序列,严格提供商(Gemini/Claude)下次会因此「幻觉续写」。所以落库前先修三处:

    • ① 删空响应脚手架(_drop_trailing_empty_response_scaffolding):把私有重试时垫的 assistant("(empty)")/恢复提醒剥掉,免得下一轮「continue」又掉进同一个空响应循环。
    • ② 中断时补收尾(close_interrupted_tool_sequence):被 /stop 打断、末尾停在 tool 结果时,补一条合成 assistant 消息闭合工具序列,避免持久化出 tool→user 这种严格提供商会拒的交替(#48879)。
    • ③ 兑现「交付即有 assistant 行」不变量(#43849/#44100):有些恢复/回退路径已经把 final_response 交付给了用户/网关,却没在 transcript 里留 assistant 消息。若 final_response 非空、未中断、且末尾不是 assistant → 补一条 {"role":"assistant","content":final_response},否则下一轮会把「未回答的 user 积压」重答一遍。
    agent._drop_trailing_empty_response_scaffolding(messages)          # ①
if interrupted: close_interrupted_tool_sequence(messages, final_response)  # ②
if final_response and not interrupted and messages[-1]["role"] != "assistant":
    messages.append({"role":"assistant","content":final_response})  # ③
agent._persist_session(messages, conversation_history)             # 落库(JSON + SQLite)
    共同主题:三处修补都是为了让「存下去的历史」是一段合法、可重放的序列。API 要求严格的角色交替;若存下一个「坏尾巴」(孤儿 tool 结果、有 tool_call 没答案、缺 assistant 回复),下次这段历史被加载重发时就不合法——严格提供商(Gemini/Claude)要么拒绝(返回空内容)、要么幻觉续写。所以落库前必须把尾巴「收干净」。
    ① 删空响应脚手架(_drop_trailing_empty_response_scaffolding)

    本轮模型若返回空内容,循环会临时垫一对私有重试脚手架——一条 assistant("(empty)") 占位 + 一条合成的 user「催一下」消息(都打了 _empty_recovery_synthetic 标记;还有恢复耗尽时的 _empty_terminal_sentinel):

    # 空响应后注入这一对(故意配成 assistant→user,让"本轮进行中"的序列合法)
messages.append(assistant("(empty)", _empty_recovery_synthetic=True))
messages.append(user("你刚调完工具却返回空,请处理上面的结果继续", _empty_recovery_synthetic=True))

    这对在同一轮内没问题(tool → assistant("(empty)") → user(催) → 继续 是合法的)。但它是内部重试机器、不是真对话,若这轮就此结束、没删就落库,会害下一轮。_drop_trailing_empty_response_scaffolding 分两趟剥:

    • Pass 1:把尾部带标记的消息 pop 掉(先弹合成 user,再弹 assistant("(empty)"))。
    • Pass 2(仅当 Pass 1 真删了才跑):再往回倒——删掉尾部的 tool 结果、以及产生它们的那条 assistant(tool_calls)(失败那圈留的「有调用没答案」悬空对)。正常进行中的工具循环不受影响(没脚手架标记就直接返回)。

    不删会怎样——「两个 user」从哪来:不删就落库,尾巴停在那条合成的 user(催一下)上;等下一轮真实用户发消息接上去,就成了:

    … tool, assistant("(empty)"), user(上一轮遗留的"催一下"), user(下一轮真实消息)
                                └────────── 两个 user 连排,非法 ──────────┘

    所以那两个 user 是① 上一轮遗留的合成催促 user + ② 下一轮真实 user(前面的 tool 是模型返回空之前那次工具结果)。严格提供商看到 user, user 静默拒绝、返回空 → 又触发空响应恢复 → 又补一对 → 又是 user, user → 每轮都空、永远好不了。所以必须在落库前把这对(及前面悬空的 tool / assistant(tool_calls))倒着剥干净。

    ② 中断时补收尾(close_interrupted_tool_sequence)

    被 /stop 打断的一轮,可能停在一条裸 tool 消息上(工具跑完了、或执行被取消,但模型没来得及吐出闭合的 assistant 轮)。这个 helper:若末尾是 tool → 补一条合成 assistant:

    messages.append({"role":"assistant",
                 "content": final_response.strip() or "Operation interrupted."})

    中断时 final_response 通常是空的,所以用占位串「Operation interrupted.」而不是空内容。不补会怎样:存下 …tool 尾巴,下一轮变 tool → user —— 角色交替违规,Gemini/Claude 会幻觉续写用户那句、无视之前的上下文,用户体感就是「上下文丢了」(#48879)。

    finalize_turn 在「正常中断路径」上调它;但重试/退避/错误型中断是在 conversation_loop 里提前 return、根本到不了收尾——所以它做成共享 helper,那些提前返回的地方也各自调一遍,保证所有中断路径都把工具序列收好。
    ③ 兑现「交付即有 assistant 行」不变量(#43849/#44100)

    有些恢复/回退路径(如部分流恢复、回退到上一轮内容的 break)已经把 final_response 交付给了用户/网关,却没在 transcript 里留下 assistant 消息。要维护的不变量是:「交付了 final_response ⇒ transcript 里必有对应的 assistant 行」。

    所以:final_response 非空、未中断、且末尾不是 assistant → 补一条 {"role":"assistant","content":final_response}。不补会怎样:durable 会话结尾停在 tool/user 上,可调用方和网关其实已经把这条 assistant 答复显示出去了。下一轮模型看到的是一堆「没被回答的 user 积压」→ 把每一条都重答一遍。在收尾这个所有恢复 break 都会汇流的单一咽喉处补上,就能不管答复由哪条路产生,都守住这个不变量。

  5. 轮次退出诊断日志(为什么这轮结束)

    logger.info / logger.warning

    每轮都在 agent.log 记一条结构化诊断,把「这轮为什么结束、干了多少活」一次性记全,便于事后回溯(这条是给运维/排查看的,不是给用户看的):

    Turn ended: reason=text_response(finish_reason=stop) model=... api_calls=12/90
            budget=12/90 tool_turns=8 last_msg_role=assistant response_len=642 session=abc
    字段含义
    reason_turn_exit_reason——这轮到底怎么结束的(正常文本/迭代耗尽/被中断/守卫拦停/空响应耗尽…,见第 2 篇)
    api_calls本轮发了几次 API 调用 / 上限
    budget迭代预算用了多少 / 总额
    tool_turns这轮有几条 assistant 带了 tool_calls(调了几"批"工具)
    last_msg_role历史最后一条是谁(assistant=正常收尾;tool=停在工具结果上,可疑)
    response_len最终答复多少字符(0=没答复)
    session会话 id,便于串起同一会话的多轮
    关键:末尾是 tool 且没中断 → 升 WARNING

    大多数情况这条记在 INFO。但有一种要升级到 WARNING——last_msg_role == "tool" 且 not interrupted:

    if last_msg_role == "tool" and not interrupted:
    # 干到一半、停在工具结果上,却没人中断它 —— 这就是用户常报的"agent 莫名不动了"
    logger.warning("Turn ended with pending tool result (agent may appear stuck)..."
                   + diag + " last_tool=%s", *args, last_tool_name)
else:
    logger.info(diag, *args)

    为什么单挑这种:正常结束末尾应是 assistant 文本答复;如果末尾停在 tool 结果、又不是用户主动中断,说明模型「拿到工具结果后没继续、就这么停了」——这正是用户最常报的「agent 干到一半莫名不动」。升 WARNING 让它在日志里跳出来,还会往回走找出最后调的工具名(last_tool),直接指向「卡在哪个工具后面」,省去翻日志定位的功夫。

  6. 文件改动校验页脚(防模型「谎报」)

    _record_file_mutation_result(执行时记账) · _format_file_mutation_failure_footer(收尾时渲染) · #15524-adjacent

    要治的病:模型并发发一批 patch,其中一半以「Could not find old_string」失败了,但模型在总结里却声称「所有文件都改好了」——用户得自己 git status 才能拆穿这个谎。

    怎么实现:执行时记账,收尾时渲染

    ① 每轮开头清账:_turn_failed_file_mutations = {}(turn_context 重置)。

    ② 每次 write_file/patch 后记账(_record_file_mutation_result,在工具执行那步调):

    landed = file_mutation_result_landed(tool, result)   # 这次写到底落地没
if is_error and not landed:                          # 失败:记下 {路径: {tool, 错误预览}}
    for path in targets:
        if path not in state:                        # 保留"第一次"的错(别被后续不同报错覆盖)
            state[path] = {"tool": tool, "error_preview": preview}
else:                                                 # 成功(landed):
    for path in targets: state.pop(path, None)        # ← 把该路径之前的失败记录弹掉!
    关键在那句 state.pop:「失败且之后再没成功写过同一路径」这个条件就是这么实现的——同一路径若模型本轮内重试并成功了,成功那次会把它的失败记录删掉。所以到收尾时,state 里剩下的只有「到这轮结束仍是失败」的路径,不会冤枉「先失败后修好」的情况。

    ③ 收尾时渲染页脚(_format_file_mutation_failure_footer):若 state 非空且开关开着,拼一段真相页脚追加到答复末尾:

    ⚠️ File-mutation verifier: N file(s) were NOT modified this turn despite any
   wording above that may suggest otherwise. Run `git status` or `read_file` to confirm.
  • `/path/a.py` — [patch] Could not find old_string...
  • `/path/b.py` — [patch] ...
  • … and 3 more            # 最多列 10 条,其余折叠计数

    这样真相每轮都被摆在脸上,模型「说过头」在结构上就不可能了。三道闸门:仅当 ① final_response 有真实文本 ② 未中断(空/中断轮已有别的提示,不叠) ③ display.file_mutation_verifier 开着(默认 True,可 env 覆盖)时才追加。

    关键:页脚只给用户看,不入历史、不回喂模型。看追加时机——_persist_session 在前(存的是没页脚的答复),追加页脚在后、只改返回给上层的 final_response:
    去哪了有页脚?
    持久化历史 / messages❌ 无(落库在追加之前)
    返回给 CLI/网关 → 显示给用户✅ 有
    下一轮再发给模型的上下文❌ 无(读的是历史,历史里没有)
    所以它既不是额外的 API 调用,也不进模型后续上下文。而且模型本轮内早就知道每个 patch 失败了(每条失败的工具结果都回灌给它看过);页脚不是「再告诉模型一次」,而是针对「模型看到了失败却在总结里谎报成功」——在模型下游给用户一层模型改不掉的兜底更正(源码注释:真相 surfaced「past the model」=绕过模型、直达用户)。
    顺带一个安全细节(#35584):页脚里每个文件路径都被 _neutralize_footer_paths 用反引号包起来。因为网关有个「裸路径自动上传」的媒体提取器——会把答复文本里的裸绝对路径当附件传到消息平台;它跳过反引号里的路径。所以万一某条错误信息里回显了敏感路径(如 ~/.hermes/config.yaml),反引号包住后就永远不会被当成可投递路径悄悄上传——纵深防御。

  7. transform_llm_output 插件钩子

    hermes_cli.plugins · transform_llm_output

    工具循环结束、模型给出最终答复之后、交付给用户之前触发一次,让插件用普通程序(而非再问一次模型)改写这段最终文本——不额外烧一分钱 token。回调签名:

    def cb(response_text, session_id, model, platform, **kwargs) -> str | None:
    ...
    return 新文本   # 非空字符串 → 替换;返回 None/空 → 保持原文

    官方例子(海绵宝宝腔,靠环境变量开关):

    import os, re
def spongebob(response_text, **kwargs):
    if os.environ.get("SPONGEBOB_MODE") != "on":
        return None                                  # 不改,原样放行
    return re.sub(r"!", "!! Tartar sauce!", response_text)

def register(ctx):
    ctx.register_hook("transform_llm_output", spongebob)

    更实用的用途:脱敏(把最终文本里的用户标识/内网路径正则替换掉)、署名/水印页脚、统一风格、翻译——这些若用「技能/提示词」去做会白烧推理 token,用这个钩子是零推理成本的后处理。

    • 第一个返回非空字符串的钩子生效(置 response_transformed=True);返回 None/空 → 保持原文,让下一个插件继续。
    • 只在有答复且未中断时触发(/stop 中断轮、空轮不触发)。
    • 钩子抛错只记 warning、不影响返回——坏插件不会拖垮交付。
    和第 4 篇的中间件别混:中间件改的是「发出去的请求/参数」(在调模型之前);这个钩子改的是「回给用户的最终文本」(在模型答完之后)。方向正好相反,各占一头。

  8. post_llm_call 插件钩子

    hermes_cli.plugins · post_llm_call

    同样在有答复且未中断时触发一次,但它是个纯观察者——返回值被忽略,不改答复(和上一个 transform_llm_output 恰好互补)。它把这轮的全貌交给插件做副作用:

    def cb(session_id, user_message, assistant_response,
       conversation_history, model, platform, **kwargs):
    ...   # 返回值忽略,纯做副作用

    典型用途:把对话同步到外部记忆系统、算响应质量指标、记轮次摘要、触发后续动作。官方例子——同步到外部记忆:

    import httpx
def sync_memory(session_id, user_message, assistant_response, **kwargs):
    try:
        httpx.post(f"{MEMORY_API}/store", json={
            "session_id": session_id,
            "user": user_message, "assistant": assistant_response,
        }, timeout=5)
    except Exception:
        pass   # 尽力而为,失败不影响主流程

def register(ctx):
    ctx.register_hook("post_llm_call", sync_memory)
    和 transform_llm_output 的分工:两者都在「答完之后、交付之前」触发,但——transform_llm_output 是变换器(返回新文本会替换答复);post_llm_call 是观察者(返回值忽略,只做记录/同步/触发)。要改文本用前者,要看这轮做点别的事用后者。

  9. 提取本轮 reasoning(不跨轮)

    reasoning 是模型通过 API 字段吐出的思考过程/思维链(Claude thinking、DeepSeek 的 reasoning_content、Codex 的 reasoning…)。这步挑出本轮的思考,放进 result["last_reasoning"] → 给 CLI/TUI 的「推理框」显示。做法:从后往前找最近一条非空 reasoning,遇到本轮起始的 user 消息就停:

    for msg in reversed(messages):
    if msg["role"] == "user": break              # 到本轮边界,停
    if msg["role"] == "assistant" and msg.get("reasoning"):
        last_reasoning = msg["reasoning"]; break  # 本轮最近的非空思考

    两个「为什么」:

    • 为什么遇到 user 就停(不跨轮):若往回一直找,会捞到上一轮的思考,推理框里显示的就是陈旧内容(这轮明明没思考,却显示上轮的)——#17055 就是这个 bug。碰到本轮起始的 user 消息就是「本轮边界」,越过它的都是历史,必须停。
    • 为什么取「最近的非空」而不是「最后一条 assistant」:很多提供商(Claude thinking / DeepSeek v4 / Codex)把 reasoning 放在调工具那一步的 assistant 消息上,而最终答复那步 reasoning=None。若只取最后一条 assistant,拿到的是 None、漏掉本轮真实思考。所以要往回找到本轮内最近一条真有思考的。

  10. 组装结果字典 + 条件字段

    result 是整个 run_conversation 的返回值。固定字段(总在)分四组:

    组字段
    答复final_response、last_reasoning、messages
    状态标志completed、failed、partial、interrupted、turn_exit_reason、api_calls、response_transformed、response_previewed
    路由model、provider、base_url、session_id
    计量(整段会话累计)input/output/cache_read/cache_write/reasoning/prompt/completion/total_tokens、estimated_cost_usd、cost_status、cost_source

    条件字段(有才挂)——直接看代码里的四个 if:

    if agent._tool_guardrail_halt_decision is not None:      # 守卫拦停了这轮
    result["guardrail"] = decision.to_metadata()         #   → 挂拦停元数据

if _cleanup_errors:                                      # 收尾三件事有出错(#8049)
    result["cleanup_errors"] = _cleanup_errors           #   → 挂错误清单

leftover = agent._drain_pending_steer()                  # 答复后才到、没处塞的 /steer
if leftover:
    result["pending_steer"] = leftover                   #   → 交上层当下一轮

if interrupted and agent._interrupt_message:             # 中断且带了消息
    result["interrupt_message"] = agent._interrupt_message
    pending_steer 是什么:/steer 是轮中途注入指引的机制(边跑边引导 agent)。正常它会被塞进「下一批工具调用」里生效;但如果它在最终 assistant 答复之后才到——这轮已经没有工具批次可塞了——就不能应用在本轮。于是把它原样交回上层(pending_steer),让上层作为下一轮的用户输入投递,而不是静默丢掉。

  11. 清状态:中断、流回调、preview 标志

    agent.clear_interrupt() · agent._stream_callback = None

    为什么要专门清:AIAgent 是个长寿、被反复复用的对象——同一个实例处理很多轮(每条用户消息调一次 run_conversation)。所以任何「本轮专用」的状态都必须在轮边界重置,否则泄漏进下一轮。这里清三样:

    ① clear_interrupt() 清得比名字看起来多:

    _interrupt_requested = False; _interrupt_message = None          # 中断标志/消息
_set_interrupt(False, _execution_thread_id)                      # 主执行线程的中断位
for wtid in _tool_worker_threads: _set_interrupt(False, wtid)    # 每个并发 worker 的中断位
_pending_steer = None                                            # 硬中断作废未决的 /steer

    后两条尤其关键(呼应第 4 篇):线程池会回收复用 worker 线程,若某个中断位没清,下一个无关任务被调度到那个 tid 上会「一进来就 is_interrupted()==True」→ 无缘无故被中止;这里再兜底清一遍。而硬中断作废未决的 /steer——steer 本是给「下一次工具迭代」的,既然被中断打断、那次迭代不会发生了,留着只会在中断后那轮突然冒出来吓用户。

    ② _stream_callback = None:流式回调是把 token 逐块喂给消费者(显示 / TTS 语音管线)的函数。不清的话,下一次调用的流可能触发旧回调 → 往一个已关闭/已销毁的消费者吐字(旧 TTS 管道、旧显示,甚至串到别的会话)。③ _response_was_previewed = False:「本轮答复已通过预览显示过」的标志,归零好让下一轮重新判断。

  12. 判断是否该复盘技能

    turn_finalizer · _iters_since_skill >= _skill_nudge_interval

    「技能复盘」是什么:Hermes 会跨会话学习——回头看 agent 这段时间干了啥,把反复出现的套路固化成可复用的「技能」(skill)。这一步只判定要不要触发一次技能复盘,三个条件全满足才置位:

    if (agent._skill_nudge_interval > 0                       # ① 功能开着(0=关)
        and agent._iters_since_skill >= agent._skill_nudge_interval  # ② 攒够了"工具迭代"(默认 10)
        and "skill_manage" in agent.valid_tool_names):    # ③ 本会话真能管技能
    _should_review_skills = True
    agent._iters_since_skill = 0                           # 触发后清零,重新攒
    • 计的是「工具迭代」不是「轮」:_iters_since_skill 数的是工具调用圈数(一轮可能用很多圈,也可能一圈)——这比「过了几轮」更能反映「干了多少活、值不值得沉淀成技能」。它跨轮累计、不在轮开头重置(轮开头只重置别的);只有两处清零:调了 skill_manage(第 4 篇的 nudge 重置)或这里触发复盘。
    • 为什么要 skill_manage 可用:复盘的产物是「建/改技能」,得靠 skill_manage 工具落地;这工具都没授权给本会话,复盘也没意义。
    这步只做决定、不干活。_should_review_skills 会和(更早算好的)_should_review_memory(记忆复盘)一起,在最后一步汇入 _spawn_background_review——真正的复盘是后台异步跑的(见最后一步),不阻塞本轮返回。

  13. 同步外部记忆 + 排下次预取

    agent._sync_external_memory_for_turn(...)

    把完成的这一轮镜像进外部记忆提供商(honcho / mem0 / supermemory…),一次做两件事:

    memory_manager.sync_all(user_text, response_text, ...)      # ① 存:把这轮问答写进外部记忆
memory_manager.queue_prefetch_all(user_text, session_id)   # ② 预热:据本轮问题,提前把
                                                           #    下一轮可能用到的记忆拉起来

    为什么要预取(prefetch):外部记忆的「召回」发生在每轮开头(拼系统提示时,第 1 篇的机制 A)。这轮结束就顺手把下一轮的记忆预热好,下一轮开头直接拿现成的、不用冷启动等一次记忆查询。

    几个关键细节:

    • 用 original_user_message,不是 user_message:后者可能被注入了技能内容(skill 斜杠命令会往用户消息里塞东西),拿去查记忆会撑大或搞坏提供商的查询。所以用干净的原始用户消息。
    • 中断的轮整个跳过(#15218):半截的答复、被打断的工具链、流中途重置——都不是「可信的对话事实」,镜像进记忆会污染将来的召回(存了用户根本没看到完成的状态)。预取也一并跳过:用户下一句多半是重试同一意图,基于中断轮的预取会打在陈旧上下文上。
    • 多模态先拍平:多模态轮的 content 是「分块列表」,提供商只吃纯字符串,所以先按换行拼成文本。
    • 严格尽力而为:整段包在 try/except 里——外部记忆后端配错/离线绝不能挡住用户看到答复。

  14. 安排后台复盘(不阻塞返回)

    agent._spawn_background_review(...)

    仅当有答复、未中断、且该复盘记忆或技能时,另起后台守护线程去「回头总结这轮的经验」——在答复交付之后才跑,绝不与用户任务争抢模型注意力。尽力而为,抛错也不影响主流程(except: pass)。

    if final_response and not interrupted and (_should_review_memory or _should_review_skills):
    try:
        agent._spawn_background_review(messages_snapshot=list(messages),
                                       review_memory=_should_review_memory,
                                       review_skills=_should_review_skills)
    except Exception: pass   # 后台复盘尽力而为
    三个基本设计点:①在答复交付之后才跑——复盘本身是另一个完整的 agent 子任务(要调模型、可能写记忆/改技能),放前台会和用户任务争抢模型注意力、拖慢响应;②传快照 list(messages)——后台线程拿当下历史的副本,不被下一轮改动干扰;③尽力而为——except: pass。这就是第 1 篇「机制 B 后台复盘」的落地点。
    它和前面「判断该不该复盘技能」的关系:决定 vs 执行。别把两步当成两件独立的事——技能复盘只是这一步要干的活之一: 
    决定阶段(只置标志,不干活):
    _should_review_memory  ← 看"过了几轮"(_turns_since_memory,更早算好传进来)
    _should_review_skills  ← 看"过了几圈工具"(_iters_since_skill,上一步算)
                          │
执行阶段(这段):  _spawn_background_review(review_memory=…, review_skills=…)
                    └─ 一个后台线程 + 一个 fork agent
                         └─ 把被标 True 的复盘一起做掉(记忆 + 技能,共用同一个 fork)
    所以:「技能复盘」「记忆复盘」是两种复盘,各有自己的决定门(_should_review_skills / _should_review_memory);而后台复盘是唯一的执行者,起一个后台 fork agent,把标记为 True 的那些一起做完——不会为技能和记忆各起一个线程。前面那步只决定、不干活,活都在这里。
    复盘线程里到底怎么跑(background_review)

    守护线程里会另建一个独立的「复盘 fork agent」(max_iterations=16、quiet_mode=True),用复盘提示词把这轮对话回放给它,让它决定要不要写记忆/改技能。有四处很讲究:

    • 复用父 agent 的缓存前缀(省钱):fork 共享父的 session_id、钉住父缓存的系统提示、并让 tools[] 与父逐字节一致——这样它发出的请求命中同一个提供商缓存前缀(Anthropic 缓存键含 tools),复盘的输入成本几乎白嫖缓存,不用重付。为此还 _skip_mcp_refresh(免得晚连的 MCP 工具破坏 tools 一致性)。
    • 持久化隔离(防「curator 夺舍」):fork 既然共享父的 session_id,若不隔离,它的复盘指令(「回顾上面对话、更新技能库…」)+ 自己的回答会写进用户的真实会话 DB;用户下一轮就把这条注入的指令当成常驻命令、于是 agent「变成了 curator」、拒绝真正的任务。所以 _persist_disabled=True 硬停所有 DB 写——复盘只通过工具写技能/记忆库,不碰会话历史。
    • 只给两把工具:复盘 agent 的可用工具被限制到 {memory, skill_manage}——它只需要写记忆/改技能,不该(也不能)去跑终端等。
    • 线程级静音:用 thread_scoped_silence() 把这一条线程的 stdout/stderr 引到 devnull(不是进程全局——否则会把网关事件循环线程的输出也吞了,#55769),再加 suppress_status_output,让用户只看到最终「做了什么」的摘要,不被复盘中途的预算/限流/压缩告警刷屏。

注意:这里不结束会话

收尾不调用记忆提供商的 on_session_end() / shutdown_all()。因为多轮会话里 run_conversation() 每条用户消息都会被调一次,每轮都关闭会杀死提供商,导致第二条消息没法用。真正的会话结束清理交给 CLI(atexit / /reset)和网关(会话过期 / _reset_session)。
同名陷阱:别把两个 on_session_end 混了。
名字是什么收尾时
记忆提供商的 on_session_end()MemoryProvider ABC 上的方法❌ 不调(上面说的就是它)
on_session_end 插件钩子插件注册的 hook✅ 每次 run_conversation 结尾都触发
所以同一个 finalize_turn 里:记忆提供商的 session-end 清理不做(留到真·会话结束);但插件钩子 on_session_end 每轮都 fire(尽力而为、抛错只 warning)。这个钩子名字有点误导——它叫「session_end」,实际是每轮 turn 结束都触发,供插件做每轮的清理/刷缓冲。

本篇要点

  • 收尾全在 finalize_turn(同步、单一返回):迭代耗尽兜底 → 算 completed → 容错落库 → 诊断日志 → 页脚/钩子/reasoning → 组装结果 → 清状态 → 安排复盘。
  • completed 只认「自然答完」:烧光迭代靠强制总结拿到的答复不算成功(< max or normal_text,不能写 <=);cron/kanban 据此重派。
  • 收尾区容错:存轨迹/拆资源/落库三件事各自 try,异常进 cleanup_errors,绝不吞掉用户的答复(#8049)。三个全失败也照返回,只是「答案送达但没存档」。
  • 落库前先修历史尾巴:删空响应脚手架 / 中断补 assistant / 「交付即有 assistant 行」不变量——否则下一轮 user,user 非法、空响应死循环。
  • 诊断日志每轮记「为什么结束」;末尾停在 tool 且没中断 → WARNING(「莫名停住」场景)。
  • 文件改动校验页脚防模型谎报:失败自愈(成功会弹掉失败记录),只追加给用户、不入历史、不回喂模型(落库在前)。
  • 返回富结果字典:答复 + 推理 + 消息 + token 计量 + 成本 + 退出原因,外加条件字段(guardrail/cleanup_errors/pending_steer/interrupt_message)。
  • 两个插件钩子:transform_llm_output 是变换器(改答复文本),post_llm_call 是观察者(返回值忽略,同步/记录)。
  • 后台记忆/技能复盘在答复之后才跑(机制 B):独立 fork agent、复用父缓存前缀省钱、_persist_disabled 防「curator 夺舍」。技能/记忆是两种复盘、共用一个后台线程。
  • 这里不结束会话(记忆提供商 session-end 留给 CLI/网关);别和每轮都触发的同名插件钩子 on_session_end 混了。
