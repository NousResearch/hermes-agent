这是什么项目

Hermes 是一个个人 AI agent:同一套 agent 内核同时运行在 CLI、消息网关(Telegram、Discord、Slack 等约 20 个平台)、TUI 和 Electron 桌面应用之上。它能跨会话学习(记忆 + 技能)、把任务委派给子 agent、运行定时任务,并驱动真实的终端和浏览器。能力主要通过插件和技能扩展,而非不断膨胀内核。

有两条设计铁律贯穿全局,也是看懂所有取舍的钥匙:

铁律含义
每会话 prompt 缓存神圣不可破长会话每轮复用缓存前缀;中途改上下文 / 换工具集 / 重建 system prompt 都会让缓存失效、成本翻倍。唯一例外是上下文压缩。
内核是「窄腰」,能力在边缘每个工具都在每次 API 调用时下发,新增核心工具门槛极高。优先用 CLI 命令+技能、服务门控工具、插件来扩展。

请求全生命周期

从用户敲下一句话,到拿回最终回答,中间经过了哪些环节?这是理解整个内核的主线。用户发一句话后,Hermes 不是「一问一答」就结束,而是进入一个多轮工具调用循环:模型可能先调几个工具(读文件、跑命令、搜网页),拿到结果后再继续思考,直到它认为可以给出最终答复为止。整段过程算一个请求 / 一轮对话(turn)。

逐步详解

  1. 前端接收输入 → 调用内核

    cli.py / gateway/run.py / tui_gateway → run_agent.py

    无论来自哪个前端,最终都汇聚到同一个入口:AIAgent.chat() 或 run_conversation()。这就是「一套内核、多个前端」的落点——前端只负责收消息、显示结果。

  2. 恢复或构建 System Prompt(缓存关键)

    agent/conversation_loop.py · _restore_or_build_system_prompt()

    同一会话的 system prompt 在第一轮构建后被缓存并持久化到 SessionDB,后续每轮原样复用。这正是 prompt 缓存铁律的实现点——绝不中途重建。

  3. 预检压缩 + 持久化用户消息

    agent/conversation_compression.py · hermes_state.py

    若历史已接近上下文上限,先做一次压缩(唯一允许改动历史的时机)。然后把用户这条消息写入会话存储,保证崩溃可恢复。

  4. 进入工具调用循环

    while (api_call_count < max_iterations and budget.remaining > 0)

    循环开始前先检查中断(用户是否按了 /stop),再把调用计数 +1。预算或迭代上限用尽则退出,防止无限循环烧钱。

  5. 记忆预取(可选)

    agent/memory_manager.py · build_memory_context_block()

    若启用记忆,这里取回相关历史片段,作为临时上下文块拼进本次 API 调用——注意它只在这次调用存在,不写回持久化历史,以免破坏缓存。

  6. 调用模型

    client.chat.completions.create(model, messages, tools)

    把消息历史 + 工具 schema 发给模型。返回要么是纯文本(模型想直接回答),要么带tool_calls(模型想先调工具)。

  7. 分支 A:执行工具调用

    model_tools.py · handle_function_call() → tools/registry.py

    若有 tool_calls,逐个(或并发)经 handle_function_call() 分发到注册表,执行真实工具(终端、文件、网页…),结果以 JSON 字符串返回,append 进 messages,然后回到第 4 步再问一次模型。

  8. 分支 B:得到最终答复

    final_response = response.content

    若模型返回纯文本(不再需要工具),把它记为 final_response 并跳出循环。

  9. 收尾:持久化与后台复盘

    agent/turn_finalizer.py · 后置钩子

    持久化本轮结果,触发后置钩子(插件的 post_llm_call 等),并安排后台的记忆 / 技能复盘(不阻塞返回)。

  10. 返回前端 → 显示

    return {final_response, messages, api_calls}

    结果字典回到 chat(),取出 final_response 交给前端渲染。一个请求至此结束。

为什么是「循环」而不是一问一答

关键就在第 4–8 步那个循环。模型一轮里可以连续推理:

第 1 次调用模型 → 模型说"我要读 a.py"  → 执行 read_file → 结果回灌
第 2 次调用模型 → 模型说"我要跑测试"   → 执行 terminal  → 结果回灌
第 3 次调用模型 → 模型说"修好了,这是答复" → 纯文本 → 跳出循环

每次工具结果都会 append 进同一个 messages 列表,再发给模型——所以模型是「带着记忆」连续推理的。max_iterations(默认 90)和迭代预算是安全阀,防止模型陷入死循环。

流程中的两条铁律

铁律在流程哪一步体现
prompt 缓存不可破第 2 步:system prompt 一次构建、全程复用;第 5 步:记忆块是临时拼接,不写回历史。中途绝不重建上下文。
消息角色严格交替第 6–7 步:user → assistant(tool_calls)→ tool(结果)→ assistant…… 永不连续两条同角色,也不在循环中途塞合成 user 消息。
唯一允许改动历史上下文的地方是第 3 步的压缩。其余任何环节改历史,都会让缓存失效、成本翻倍。
