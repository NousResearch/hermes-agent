注册表:工具的中央户口

所有工具的信息(schema、handler、属于哪个工具集、可用性怎么查)都汇总在一个地方——tools/registry.py 里的 ToolRegistry 单例。这一篇讲清:一个工具文件里那行 registry.register(...) 到底做了什么,注册表替我们挡掉了哪些坑。

为什么要有一个中央注册表

Hermes 有几十个工具,散落在 tools/*.py 几十个文件里。如果没有注册表,model_tools.py 就得维护一堆平行的字典:一个存 schema、一个存 handler、一个存「哪个工具属于哪个工具集」、一个存「怎么判断它可用」……每加一个工具都要改四五处,极易漏改、对不上。

注册表把这些收成一处:每个工具文件在被 import 时,自己调一次 registry.register(...) 把「我叫什么、schema 长啥样、谁来执行我、我属于哪个工具集、怎么查我可不可用」一次性登记好。model_tools.py 之后只管查注册表,不再自己维护任何平行结构。

一句话:注册表是所有工具的中央户口本。工具文件负责「上户口」,其余所有子系统(暴露、分发、可用性、banner、doctor)都只读这一本。

一行 register() 长什么样

先看一个真实工具文件的收尾——这是每个内置工具的固定套路(以 example_tool 为例):

# tools/your_tool.py
import json, os
from tools.registry import registry            # ← 导入那个全局单例

def check_requirements() -> bool:               # 可选:这个工具现在可不可用?
    return bool(os.getenv("EXAMPLE_API_KEY"))

def example_tool(param: str, task_id: str = None) -> str:
    return json.dumps({"success": True, "data": "..."})   # 必须返回 JSON 字符串

registry.register(                              # ← 模块顶层调用:import 时就执行
    name="example_tool",
    toolset="example",
    schema={"name": "example_tool", "description": "...", "parameters": {...}},
    handler=lambda args, **kw: example_tool(param=args.get("param",""), task_id=kw.get("task_id")),
    check_fn=check_requirements,
    requires_env=["EXAMPLE_API_KEY"],
)
tools/registry.py · registry.register()

关键:registry.register(...) 是写在模块顶层的(不在任何函数里)。所以只要这个 .py 文件被 import 一次,登记就自动发生了——这也是下一篇「自动发现」能成立的前提。

register() 的完整参数

ToolRegistry.register() 的签名(tools/registry.py):

参数必填作用
name✅工具名,模型点名时用的就是它;注册表里以它为主键(唯一)
toolset✅归到哪个工具集(分组标签)。决定了它能不能被暴露——见第 3 篇
schema✅OpenAI 格式的工具定义(name/description/parameters),发给模型的就是它
handler✅真正干活的可调用对象,签名 (args: dict, **kwargs) -> str,必须返回 JSON 字符串
check_fn—零参可调用,返回 bool。判断「此刻这工具可不可用」(有没有配 API key、Docker 在不在)。不传=永远可用
requires_env—这工具需要的环境变量名列表(供 hermes tools/setup 展示与提示,不影响运行)
is_async—handler 是不是协程。为 True 时 dispatch() 会自动用 _run_async() 桥接(见第 5 篇)
description—人读描述;不传则回退到 schema["description"]
emoji—CLI/TUI 活动区显示这工具时的图标
max_result_size_chars—该工具结果的字符上限(超了截断/落盘),不传用全局默认
dynamic_schema_overrides—零参可调用,返回一个 dict,在生成 schema 时浅合并覆盖到基础 schema 上。用于「描述要反映当前配置」的场景(见第 4 篇)
override—显式允许覆盖已存在的同名工具。默认 False 会拒绝覆盖,防误伤

登记后存进 ToolEntry

每次 register() 会把这些参数打包成一个 ToolEntry,塞进注册表内部的 self._tools[name] 字典。ToolEntry 用 __slots__ 固定了字段(省内存 + 防止手滑加属性):

class ToolEntry:
    """单个已注册工具的元数据。"""
    __slots__ = (
        "name", "toolset", "schema", "handler", "check_fn",
        "requires_env", "is_async", "description", "emoji",
        "max_result_size_chars", "dynamic_schema_overrides",
    )
tools/registry.py · class ToolEntry

字段和 register() 的参数一一对应。之后所有子系统都通过 registry.get_entry(name) 拿到这个 ToolEntry,再取它的 schema(暴露)、handler(分发)、check_fn(可用性)、toolset(分组)。

插一句:__slots__ 是干嘛的

普通 Python 类,每个实例背后都藏着一个 __dict__(字典)存它的属性——灵活(能随时挂任意新属性),但每个实例都为此多占一份字典内存:

class Normal:
    def __init__(self): self.name = "x"
n = Normal()
n.__dict__          # {'name': 'x'}   ← 每个实例都带一个 dict
n.whatever = 123    # 随便加新属性也行,悄悄存进那个 dict

__slots__ 等于告诉 Python:「这个类的实例只会有列出的这几个属性,别再给我建 __dict__ 了」。于是改用更紧凑的固定槽位(类似 C 结构体):

class Slotted:
    __slots__ = ("name",)
    def __init__(self): self.name = "x"
s = Slotted()
s.name              # x        ✅ 正常读写
s.whatever = 123    # ❌ AttributeError: 没有 'whatever' 这个 slot
好处原因
省内存不给每个实例建 __dict__,改用固定槽位。ToolEntry 一个进程里可能有上百个实例(每工具一个,还可能多 agent),省下的字典累积可观
防手滑加属性写 entry.chek_fn = ...(打错字漏个 c)会当场 AttributeError,而不是普通类那样默默新建一个错属性、让你 entry.check_fn 读到旧值、bug 极难查
ToolEntry 是个纯数据容器——字段就是固定的那 11 个(即 register() 的参数),不需要运行时动态挂属性。这正是 __slots__ 的理想场景:字段集合已知且固定、实例多、只当结构体用。反过来,若一个类需要动态挂属性,就不该用 __slots__。

单例:全进程只有一本户口

注册表是模块级单例——文件末尾实例化一次,整个进程共用:

# tools/registry.py 末尾
registry = ToolRegistry()

所以无论哪个工具文件 from tools.registry import registry,拿到的都是同一个对象。几十个工具文件各自 register(),最后都堆进这一个 registry._tools 里。

循环导入安全是这套设计的隐藏前提。tools/registry.py 不 import model_tools 或任何工具文件,所以它处在导入链的最底层:
registry.py(谁都不依赖)← tools/*.py(只从 registry 导入)← model_tools.py(导入 registry + 所有工具)← run_agent.py / cli.py。
方向永远单向朝上,不会成环。

override 防覆盖闸:不许悄悄顶掉别人

register() 最要紧的一段逻辑,是当同名工具已存在、且来自不同工具集时的处理。默认它拒绝覆盖——防止一个插件(或手滑)悄悄顶掉某个内置工具。判定链:

flowchart TD
    A[register name, toolset, ...] --> B{同名工具已存在
且 toolset 不同?}
    B -- 否 --> Z[直接写入 _tools]
    B -- 是 --> C{两个都是 mcp-* ?}
    C -- 是 --> Z2[允许覆盖
MCP 刷新/重名是合法的]
    C -- 否 --> D{override=True ?}
    D -- 否 --> R[❌ 拒绝并 return
记 ERROR 日志]
    D -- 是 --> E{是插件 handler
且没拿到操作员授权?}
    E -- 是 --> P[❌ 抛 PermissionError
需 allow_tool_override]
    E -- 否 --> Z3[✅ 覆盖
记 INFO 便于审计]

existing = self._tools.get(name)
if existing and existing.toolset != toolset:
    both_mcp = existing.toolset.startswith("mcp-") and toolset.startswith("mcp-")
    if both_mcp:
        ...  # MCP↔MCP:server 刷新或两个 server 重名,合法,放行
    elif override:
        _owner = self._plugin_owner_of(handler)          # 这 handler 是哪个插件写的?
        if _owner is not None and not self._plugin_override_policy.get(_owner, False):
            raise PermissionError(...)                    # 插件想覆盖内置,但没授权 → 拒
        logger.info("... override=True opt-in")           # 授权了(或非插件调用)→ 覆盖
    else:
        logger.error("... would shadow existing tool ...") # 没 override → 拒绝,直接 return
        return

情形结果为什么
同名同 toolset(纯重注册)覆盖(不触发本闸)同一工具重新登记是正常的(如重载)
两个都是 mcp-*允许覆盖MCP server 刷新、或两个 server 有重名工具,都是合法的动态行为
override=False(默认)顶掉别人❌ 拒绝 + ERROR防止插件/手滑悄悄替换内置工具
override=True 但插件没授权❌ 抛 PermissionError覆盖内置工具需操作员在 config 里 allow_tool_override: true
override=True 且已授权(或非插件)✅ 覆盖 + INFO 审计合法的替换(如把默认浏览器工具换成 headed-Chrome CDP 后端)
「谁写的 handler」绑定在定义处,骗不过去。_plugin_owner_of(handler) 看的是 handler.__globals__["__name__"]——即这个函数定义在哪个模块,这是在定义时就固定的,不随调用点/线程/时机漂移。lambda 和嵌套函数继承定义模块的 globals,所以插件没法用回调「洗白」一次越权覆盖。授权策略(register_plugin_override_policy)是持久的、从不清除,即便插件事后用延迟/线程回调再 register(),也照样被同一条策略拦。
展开:__globals__["__name__"] 为什么骗不过去

这段是防覆盖闸里最精妙、也最值得单独拆开看的一处。它回答一个安全问题:怎么保证没授权的插件,没法用小花招偷偷覆盖内置工具?先看那两段真实代码:

def _plugin_owner_of(self, handler):
    mod = handler.__globals__.get("__name__", "")     # ← 关键:handler 定义在哪个模块
    if mod in self._plugin_override_policy: return mod
    if mod.startswith("hermes_plugins."):    return mod
    return None

def register_plugin_override_policy(self, module_namespace, allowed):
    self._plugin_override_policy[module_namespace] = bool(allowed)   # 加载时记下,从不删

什么是 handler.__globals__["__name__"]:Python 里每个函数对象都随身带着 __globals__——它指向这个函数被 def 定义时所在模块的全局命名空间;而每个模块的全局里都有个 __name__,值就是模块名。

# 文件 hermes_plugins/evil/__init__.py 里:
def my_handler(args, **kw): ...
my_handler.__globals__["__name__"]      # → "hermes_plugins.evil"

要害:__globals__ 记的是「函数在哪儿被定义」,不是「在哪儿被调用」。这个绑定在 def 执行那一刻就定死了——你把函数传给别人、塞进变量、跨线程调、延迟几秒再调,它都不变;因为它记的是「出生地」,不是「此刻在谁手里」。

于是插件想把「我这段代码干的越权」洗白成「不是我干的」的各种花招,全部失效:

插件的花招为什么失败
用 lambda args: ... 包一层再 registerlambda 继承定义它的模块的 __globals__ → 还是 hermes_plugins.evil
定义嵌套函数(闭包)去调嵌套函数同样继承外层模块的 globals → 还是插件模块
起后台线程,在线程里 register线程只改「调用点」,handler 的定义处没变 → 还是插件模块
延迟到加载完之后再 register见下「持久策略」——照样拦

对比反例:如果改用「调用栈」判断就能被骗。假如不看 __globals__,而是问「现在是谁在调 register?」,插件把 handler 交给一个「中立」辅助函数去注册,调用栈顶就成了那个中立函数、不是插件——越权得逞。正因为调用栈会随「谁来调」而变、可被操纵,而 __globals__ 是定义时固定的指纹、操纵不了,所以判定必须绑在定义处。

为什么授权策略要「持久、从不清除」:插件的 register() 不一定在加载那一瞬间就调完——它可能挂个回调、开个线程,几秒后才 register。若授权策略「用完就清」,那个延迟发生的 register 查策略时就查不到授权记录了,要么误放行(危险)、要么误拦合法授权。做成持久的,就意味着不管这次 register 是同步立即发生还是延迟/线程回调里发生,查到的都是加载时就定好的同一条策略。

一句话:授权判定问的不是「现在谁在调用」(可伪装),而是「这个 handler 是哪个模块 def 出来的」(定义时固定、不可伪装);再配上授权策略持久保存——于是无论插件用 lambda 包、闭包套、开线程、还是延迟回调,都改不了那段代码的「出生地指纹」,也躲不过加载时登记好的那条策略。越权覆盖内置工具这条路被从根上堵死。

_generation:一个会「涨」的版本号

注册表内部有个单调递增的代际计数器 _generation,每次注册表发生变动(register / deregister / 注册别名 / MCP 刷新)就 +1:

self._tools[name] = ToolEntry(...)
...
self._generation += 1        # 每次 mutation 都涨一格

它的用途:让下游的缓存能安全 memo。第 4 篇会看到 get_tool_definitions() 把结果缓存起来,缓存键里就带着 registry._generation——只要注册表没变(代际号没涨),缓存就一直有效;一旦有工具增删(比如 MCP 动态刷新),代际号一变,旧缓存自动失效。这样既省了重复计算,又不会用到过时的工具列表。

缓存键到底怎么工作(具体走一遍)

「生成发给模型的工具 schema 列表」这件事不便宜(要解析工具集、跑一堆 check_fn、做后处理)。所以下游用一个字典把算好的结果缓存起来,键里带上 _generation:

# 极简版示意(真实版本第 4 篇细讲)
key = (enabled_toolsets, disabled_toolsets, registry._generation)   # ← 代际号是键的一部分
if key in cache:
    return cache[key]          # 命中:直接返回,跳过整条流水线
result = 真正去算一遍(...)      # 未命中:算,然后存
cache[key] = result

关键在于:代际号一变,键就变;键一变,就必然「未命中」→ 强制重算。走一遍:

第 1 轮:_generation=7 → key=(..,7) 未命中 → 算 → 存 cache[(..,7)]
第 2 轮:注册表没变,_generation 还是 7 → key=(..,7) 命中 → 直接用缓存 ✅ 省了
        ── 这时 MCP 刷新了一批工具,_generation: 7 → 8 ──
第 3 轮:_generation=8 → key=(..,8) 是【新键】,cache 里没有 → 未命中 → 重算 ✅ 不会用到旧的

所以旧缓存 cache[(..,7)] 不用谁去主动「删」——它只是再也不会被查到了(没人会再拿 7 当键)。这就叫「精确失效」:只有真正发生过变动(代际号涨了),缓存才失效;没变动就一直命中。

为什么不用「每次都重算」或「时间戳」
做法问题
每次都重算(干脆不缓存)结果永远对,但白白重复算——热路径(网关、agent 初始化)每轮都重跑整条流水线,浪费
用时间戳(记「上次改于几点」,比时间新旧)① 依赖时钟精度——同一毫秒内改两次可能分不出;② 系统时钟被回拨/校时会乱套(“变新”变成“变旧”);③ 时间戳只告诉你「变过」,不天然当键用
代际计数器(本方案)只有真正变动才 +1,单调递增、绝不回头;直接塞进缓存键即可,零时钟依赖、零猜测
再加一层并发安全:所有会改注册表的操作(register/deregister/别名/MCP 刷新)都在同一把锁 self._lock(RLock)里做,_generation += 1 也在锁内——保证「改内容」和「涨代际号」是一起发生的,不会出现「工具已经改了、代际号还没涨」的中间态。读方则先在锁内拿一份稳定快照(_snapshot_entries())再慢慢用,配合这个单调号,就能放心缓存、不怕读到改一半的状态。
别混:_generation 是整个注册表一个,不是每个工具一个。它挂在 ToolRegistry 单例上(self._generation),不在 ToolEntry 里——回看上面 ToolEntry.__slots__ 那 11 个字段,并没有 _generation。
东西挂在哪有几个
ToolEntry(name/schema/handler/check_fn…)每个工具一个几十上百个
_generationToolRegistry 单例本身全进程 1 个
它记的是「整张户口本改过几次」,不是「某个工具改过几次」——任何一个工具的增删都让这同一个计数器 +1(注册 A → 1、注册 B → 2、删掉 A → 3、MCP 刷一批 → 4)。因为下游那个缓存缓的是「一整份工具 schema 列表」,只需要「注册表整体变没变过」这一个信号。像图书馆整馆的馆藏版本号:不管今天进哪本、退哪本,版本号都 +1;它不是贴在每本书上的「这本第几版」(那才是各 ToolEntry 自己的事)。

还顺手记了两样:toolset check 与别名

① toolset 的 check_fn 备案。注册时若带了 check_fn,注册表会把「这个工具集第一次出现的 check_fn」记进 _toolset_checks[toolset]:

if check_fn and toolset not in self._toolset_checks:
    self._toolset_checks[toolset] = check_fn

注意:可用性现在是逐工具判定的(_toolset_has_exposable_tools 会遍历该工具集里每个工具各自的 check_fn),所以这个 _toolset_checks 不再用来「一刀切」地关掉整个工具集。它只剩一个用途:给 banner.py 分类用——判断一个已不可用的工具集是「懒加载(有 check_fn,只是还没满足)」还是「彻底禁用」,只看有没有、从不真调。

为什么逐工具而非按工具集:有些工具集是混合的。比如 terminal 工具集里既有普通 terminal,又有只在桌面 GUI 才可用的 read_terminal(check_fn 探测 HERMES_DESKTOP)。若用「工具集里第一个 check_fn」一刀切,就会因为 read_terminal 不可用而误伤整个终端工具集。所以必须逐个工具查。

② 工具集别名。register_toolset_alias(alias, toolset) 给一个规范工具集名登记一个别名(比如给某个 MCP server 的工具集起个用户友好的名字)。别名冲突会记 warning,并同样让 _generation += 1。第 3 篇讲工具集解析时会用到。

本篇要点

  • 注册表是所有工具的中央户口本:工具文件在模块顶层调一次 registry.register() 上户口,其余子系统只读这一本。
  • register() 把参数打包成 ToolEntry(__slots__ 固定字段)存进单例 registry;四个必填参数是 name / toolset / schema / handler(其余都有默认值),handler 必须返回 JSON 字符串。
  • 注册表处在导入链最底层(不 import 任何工具或 model_tools),所以循环导入安全。
  • override 防覆盖闸:默认拒绝顶掉别人的同名工具;MCP↔MCP 例外;插件想覆盖内置需操作员 allow_tool_override 授权——授权绑定在 handler 的定义模块上,骗不过去。
  • _generation 代际计数器每次变动 +1,让下游缓存(第 4 篇)能精确失效。
  • 划重点(下一篇继续):「注册进 _tools」只是上了户口,还没暴露给模型——那要等它的工具集被启用(第 3 篇)。
