自动发现:tools/*.py 如何自己上户口

第 1 篇说「工具文件被 import 时会自动 register()」。可它们是谁 import 的?几十个工具文件,没有哪张手写清单去逐个 import。这一篇讲 discover_builtin_tools()——用 AST 扫描把「凡是顶层调了 registry.register 的文件」自动找出来并 import。

要解决的问题:别维护 import 清单

先想清楚这套加载路径里的一个 Python 事实:工具模块必须先被 import(直接导入,或作为别的模块的依赖被导入),它的顶层代码才会执行。第 1 篇说过,每个自注册的内置工具文件在模块顶层写了一句 registry.register(...);可这句要真的跑起来,前提是 Python 加载了这个文件。没人 import,文件就静静躺在磁盘上,那句 register 永远不执行,工具也就从没进过注册表。

所以问题变成:几十个 tools/*.py,由谁来负责把它们 import 一遍?朴素做法是在 model_tools.py 里手写一长串:

# model_tools.py —— 朴素做法(Hermes 没这么干)
import tools.terminal_tool     # 触发 terminal_tool 顶层的 register()
import tools.file_tools        # 触发 file_tools 顶层的 register()
import tools.web_tools
import tools.browser_tool
# ... 再手写几十行,每个工具文件一行 ...

这份「import 清单」能用,但很脆:

毛病后果
每加一个工具文件都要回来补一行新写了 tools/my_tool.py,却忘了在 model_tools.py 补 import tools.my_tool → 这个文件没人 import → 顶层的 register 不执行 → 工具压根没进注册表
漏补了也不报错不是抛异常、不是红字警告,而是静默地什么都没发生——工具「查无此人」,模型只会说 Tool xxx does not exist,你得反查半天才想到「哦我忘了加那行 import」
两处要手动保持同步「工具文件本身」和「import 清单」是两个地方,靠人肉记得同时改。清单删漏了(删了工具文件却留着 import)会 ImportError;加漏了(加了文件没加 import)会静默失踪。顺序、重复也都要自己管

Hermes 干脆取消这份手写清单,让发现自动化:程序第一次导入 model_tools 时扫一遍 tools/ 目录,看谁在模块体里以独立语句直接调用 registry.register(),就自动 import 谁。这样「加了新工具文件」和「它被发现」之间不再有人肉维护清单的一步——把文件丢进 tools/、按约定写好 register、并确保模块能成功导入,它就会在下一次进程启动时自动就位。这就是下面 discover_builtin_tools() 做的事。

一句话对比:手写清单是「白名单」——只有列进清单的才会被 import,清单是唯一事实源、必须人肉维护;自动发现是「按约定扫描」——文件放在 tools/、模块体里直接写一条独立的 registry.register(...)、且模块能够成功导入,就自动被收。约定本身就是事实源,不用再单独维护一份名单。

discover_builtin_tools():扫目录、按需 import

def discover_builtin_tools(tools_dir=None) -> List[str]:
    """Import 所有自注册的内置工具模块,返回它们的模块名。"""
    tools_path = Path(tools_dir) if tools_dir is not None else Path(__file__).resolve().parent
    module_names = [
        f"tools.{path.stem}"
        for path in sorted(tools_path.glob("*.py"))            # 扫 tools/*.py
        if path.name not in {"__init__.py", "registry.py", "mcp_tool.py"}  # 排除三个
        and _module_registers_tools(path)                      # ← 顶层真的调了 register 吗?
    ]
    imported = []
    for mod_name in module_names:
        try:
            importlib.import_module(mod_name)                  # import 触发它的 register()
            imported.append(mod_name)
        except Exception as e:
            logger.warning("Could not import tool module %s: %s", mod_name, e)
    return imported
tools/registry.py · discover_builtin_tools()

三步——注意只有第 ③ 步才真正「加载」文件,前两步都只是在挑名单:

步做什么加载(import/执行)了吗
① glob("*.py")列出 tools/ 目录里所有 Python 文件(只拿到文件名)❌ 只是列目录
② 排除三文件 + _module_registers_tools用 AST 静态检查「顶层有没有 registry.register」,筛出合格文件名❌ 只读文本 + 解析(见下一节)
③ importlib.import_module(mod_name)真正 import 每个合格模块——这一下才把文件加载进来、执行它的顶层代码,于是那句 registry.register() 跑起来、工具上户口✅ 加载就发生在这里

换句话说:①② 一路都在跟字符串(文件名/源码文本)打交道、决定「该 import 谁」;直到 ③ 的 import_module 才把这些文件真的搬进 Python 运行时。也是在这一步,顶层的 register() 才被触发(第 1 篇的「上户口」发生在此)。单个模块 import 失败只记 warning、不连累其余——某个工具的可选依赖没装,不该拖垮整个工具系统。

一句话:发现 = 「挑名单(①②,静态)」+「按名单加载(③,import_module)」。真正让工具代码跑起来、进注册表的,是且仅是第 ③ 步那句 importlib.import_module。

为什么用 AST 检查,而不是「import 了再说」

关键在 _module_registers_tools()——它先读源码、解析成 AST、只看模块顶层语句里有没有 registry.register(...) 调用,而不是「先 import 进来看看」:

def _module_registers_tools(module_path: Path) -> bool:
    try:
        source = module_path.read_text(encoding="utf-8")
        tree = ast.parse(source, filename=str(module_path))
    except (OSError, SyntaxError):
        return False
    return any(_is_registry_register_call(stmt) for stmt in tree.body)   # 只看 tree.body(顶层)

def _is_registry_register_call(node) -> bool:
    """判断 node 是不是一句 registry.register(...) 调用表达式。"""
    if not isinstance(node, ast.Expr) or not isinstance(node.value, ast.Call):
        return False
    func = node.value.func
    return (isinstance(func, ast.Attribute) and func.attr == "register"
            and isinstance(func.value, ast.Name) and func.value.id == "registry")
tools/registry.py · _module_registers_tools / _is_registry_register_call

两个「为什么」:

  • 为什么先 AST 筛、而不是 import 全部:很多 tools/ 下的文件是辅助模块(工具函数、解析器、helper),它们自己不是工具,但可能在某个函数体内部调 registry.register()(比如动态注册)。如果无脑 import 所有 .py,会把这些纯 helper 也拖进来、甚至误触发它们的副作用。AST 检查只认「顶层直接写着 registry.register(...)」的文件——any(... for stmt in tree.body) 里的 tree.body 就是模块顶层,函数体内部的调用不算。
  • 为什么解析失败就跳过:ast.parse 遇到 OSError(读不了)或 SyntaxError(语法错)直接返回 False——一个语法坏掉的文件不会中断整个发现流程,只是被安静略过。
AST 检查是「静态」的——它不执行代码,只读文本、看结构。所以它能在「决定要不要 import」之前就判断出「这文件是不是一个工具」,避免了「为了知道该不该导入,只好先导入一遍」的鸡生蛋问题。
这里的「顶层」是一条严格的语法契约,不只是视觉上没缩进。检测器只接受模块体中的独立调用表达式 registry.register(...)。下面只有第一种会被发现: 
registry.register(...)                 # ✅ 模块体中的独立调用表达式

if ENABLED:
    registry.register(...)             # ❌ 包在顶层 if 里面,tree.body 看到的是 ast.If

entry = registry.register(...)         # ❌ 顶层赋值,tree.body 看到的是 ast.Assign
tool_registry.register(...)            # ❌ 接收者不是名为 registry 的 ast.Name

def setup():
    registry.register(...)             # ❌ 在函数体里
这份严格约定来自历史修复 fc6cb5b97:早期用 ast.walk() 扫整棵树,会把函数体里的动态注册也误认成内置工具模块;现在刻意只认这一种直接写法。
划清楚:_module_registers_tools 是「门卫」,不是「搬运工」

容易误会的一点:_module_registers_tools 并不加载(不 import、不执行)工具文件。它只是个静态检查器 / 门卫,回答一个是非题——「这个 .py 顶层有没有写 registry.register(...)?」返回 True/False,仅此而已:

source = module_path.read_text(...)   # ① 把文件当【纯文本】读进来(像用记事本打开看一眼)
tree   = ast.parse(source, ...)       # ② 解析成语法树,【一行都不运行】
return any(... for stmt in tree.body) # ③ 只看顶层有没有一句是 registry.register(...)

真正「加载」工具的是另一步。放到整个流程里,分工是:

方法干什么会执行工具代码吗
_module_registers_tools(门卫)静态检查:这文件顶层有没有 register❌ 只读文本 + 解析 AST
_is_registry_register_call(帮手)帮门卫判断「某一句是不是 registry.register(...)」❌
discover_builtin_tools(搬运工)先用门卫筛出合格文件,再 importlib.import_module 真正加载✅ 这一步才 import、才触发顶层 register
module_names = [ ... for path in ...
                 if ... and _module_registers_tools(path) ]  # ← 先【检查】,筛掉不合格的
for mod_name in module_names:
    importlib.import_module(mod_name)                         # ← 这里才【加载】,工具真正上户口

所以顺序是「先门卫查资格 → 再搬运工搬进门」:门卫（AST 检查）不放行的文件（顶层没有 register 的 helper），搬运工根本不会去 import。

三个被排除的文件

文件为什么排除
__init__.py包初始化文件,不是工具
registry.py就是注册表自己。它定义了 register,但不调用 registry.register(而且它处在导入链最底层,不能反过来依赖工具)
mcp_tool.pyMCP 工具是运行时动态注册的,不走「模块顶层 register」这条路。显式按文件名排除是一道生命周期护栏:即使将来文件里出现顶层 register,自动发现也不会误把它当成静态内置工具(见第 5 篇)
mcp_tool.py 的排除是理解 MCP 的钥匙。内置工具采用「import 时完成初始登记、不会由自动发现器周期性刷新」的生命周期;而 MCP 工具是「运行时连上外部 server、按 server 返回的清单动态 register/deregister、还能中途刷新」。注意「初始登记是静态的」不等于注册表从此不可变化:获得授权的插件仍可覆盖/注销内置条目,dynamic_schema_overrides 也能在暴露阶段动态改 schema。这里区分的是发现与登记机制,不是宣称整个进程内的工具元数据永远不变。

何时运行:import model_tools 的副作用

前面讲清了「发现怎么做」,这一节回答「发现什么时候被触发」。

先弄懂「import 的副作用」是什么意思

「副作用(side effect)」= 一个模块被 import 时,顺带执行的、import 本身之外的动作。回顾一个 Python 事实(第 1 篇也提过):import 一个模块 = 从头到尾执行它的顶层代码一次。所以模块顶层写的不只是 def/class 定义,还可以写会「干活」的语句——那些语句会在 import 的一瞬间跑起来。举个最小例子:

# mymod.py
print("我被 import 了")       # ← 顶层的"干活"语句
def foo(): ...

# 别处:
import mymod                  # 这一行会打印"我被 import 了" —— 这就是 import 的【副作用】

你本意只是「拿到 mymod.foo」,但 import 顺带把顶层那句 print 也执行了。这个「顺带执行的动作」就是副作用。

放到这里:发现是「import model_tools 的副作用」

discover_builtin_tools() 不是被谁在某个函数里「主动调用」的,而是被写在 model_tools.py 的模块顶层——所以只要有人 import model_tools,这句就作为副作用自动跑一遍:

# model_tools.py 顶层(不在任何函数里)
from tools.registry import discover_builtin_tools, registry
...
# ===== Tool Discovery =====
discover_builtin_tools()      # ← 顶层语句:import model_tools 时,这句自动执行
model_tools.py · 模块顶层

于是形成一条连锁:

有人 import model_tools
   └─(副作用)─> 执行顶层的 discover_builtin_tools()
                   └─ 扫 tools/ 目录 → 逐个 import 合格的工具文件
                        └─(又一层副作用)─> 成功加载的工具文件执行顶层 registry.register()
                             └─ 成功导入并完成 register 的工具上户口 ✅

谁会 import model_tools?run_agent.py / cli.py / batch_runner.py 等路径最终都会。所以不管用户从 CLI、网关还是批处理哪个前端进来,只要程序第一次用到工具系统(第一次 import 到 model_tools),内置工具的发现扫描就自动跑一遍——没有哪段生产入口需要显式喊一声「现在开始发现内置工具」。某个模块若导入失败,本轮只记 warning 并继续,因此「扫描完成」不代表每个候选模块都成功登记。

为什么用「副作用」而不是找个地方显式调一次:因为「用工具系统」的入口有好几个(CLI/网关/批处理…),要是靠「显式调用」,就得在每个入口都记得调一次 discover_builtin_tools()——又回到了「靠人肉记得」的老问题。把它挂在 model_tools 的顶层当副作用,等于「谁用到工具系统,谁就自动触发了发现」,一处搞定、天然不会漏。代价是:import 有了副作用、且发现是同步阻塞的——所以耗时/会阻塞的东西(如连外部 MCP server)不能放这儿,见下。
一个历史教训:MCP 发现没放在这里。源码注释记着——外部 MCP server 的发现曾经也放在这个模块顶层当副作用,但 discover_mcp_tools() 内部有个阻塞的 future.result(timeout=120),而网关是在 asyncio 事件循环里、收到第一条用户消息时才懒加载 model_tools 的——结果只要有个配置的 MCP server 慢或连不上,这个「import 副作用」就会把 Discord/Telegram 的心跳冻住最多 120 秒(#16856)。所以 MCP 发现被挪走了,只留内置工具的同步发现在这。这正说明:把慢操作塞进「import 副作用」是危险的——import 谁都以为是瞬间的,没人料到它会卡两分钟。

回顾:模块初始化阶段为什么不成环

把第 1 篇的单例、这一篇的发现串起来,真正需要守住的是初始化主链:registry.py 在模块顶层不反向 import model_tools 或具体工具模块,所以注册表单例会先完整建立;工具模块随后才能安全地回来取这个单例。工具模块之间并非绝对互不依赖——它们可以 import helper,个别工具也会 import 另一个工具模块——因此下面画的是发现相关的主干,不是整个 tools/ 目录的完整依赖图:

flowchart BT
    R["tools/registry.py
(先定义 registry 单例 + discover_builtin_tools)
顶层不反向导入 model_tools / 工具模块"]
    H["tools 下的 helper
以及少量其他工具模块"]
    T["自注册 tools/*.py
(顶层 registry.register 上户口)"]
    M["model_tools.py
(顶层调 discover_builtin_tools + 查注册表)"]
    U["run_agent.py / cli.py / batch_runner.py"]
    T --> R
    T -- "可能依赖" --> H
    M --> R
    M -.import 触发.-> T
    U --> M

model_tools 先 import 已完成初始化的 registry,再由顶层的 discover_builtin_tools() 动态 import 工具文件(虚线),触发它们回到同一个单例执行 register()。这条动态 import 当然仍是一条运行时依赖、也会影响实际加载顺序;它之所以安全,不是因为它「不算依赖」,而是因为被工具模块依赖的注册表已经初始化完毕。registry.dispatch() 等方法里虽然还有对 model_tools 的函数内延迟 import,但那发生在工具真正执行时,不参与注册表模块的启动初始化,因此不会形成这里讨论的初始化环。

发现了 ≠ 暴露了(重要伏笔)

到这里,成功导入并完成顶层 register 的内置工具已经登记进注册表。但请记住第 1 篇末尾那句:登记进 _tools 只是上了户口,模型还看不到它们。自动发现只保证「成功加载的工具模块已把 schema/handler 交给注册表」;至于哪些工具会被真正下发给模型,取决于工具集——它得先属于一个被当前平台启用的工具集,还要通过 check_fn。这正是下一篇的主题。

这是全模块最容易踩的坑:你新写了一个工具文件、register() 也调了、get_errors 也没报错,但模型死活不认它。九成是——你忘了把工具名加进 toolsets.py 的某个工具集。自动发现只做了一半,「接进工具集」是另一半、且是手动的(见第 3、5 篇)。

本篇要点

  • discover_builtin_tools() 扫 tools/*.py,用 AST 只挑出「模块体里以独立表达式直接调用 registry.register」的文件并 import;包在 if/函数里、赋值或别名调用都不算。
  • 用 AST 静态检查(不执行代码)是为了放过纯 helper 模块——只有顶层直接写着 register 的才算工具;函数体内部的 register 不算。
  • 排除 __init__.py / registry.py / mcp_tool.py;MCP 因为是运行时动态注册,不走这条静态发现路。
  • 发现是 model_tools.py 模块顶层的一次性副作用——谁 import model_tools,谁就触发全套发现。
  • 发现相关的初始化主链循环安全:注册表先完成初始化,model_tools 再动态 import 工具;这不代表所有工具模块之间都只有单向依赖。
  • 发现 ≠ 暴露:登记进注册表后,还要「被启用的工具集收录 + 通过 check_fn」才会下发给模型——下一篇。
