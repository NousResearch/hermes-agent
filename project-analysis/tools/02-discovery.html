自动发现:tools/*.py 如何自己上户口

第 1 篇说「工具文件被 import 时会自动 register()」。可它们是谁 import 的?几十个工具文件,没有哪张手写清单去逐个 import。这一篇讲 discover_builtin_tools()——用 AST 扫描把「凡是顶层调了 registry.register 的文件」自动找出来并 import。

要解决的问题:别维护 import 清单

朴素做法是在 model_tools.py 里写一长串:

import tools.terminal_tool
import tools.file_tools
import tools.web_tools
# ... 再手写几十行 ...

问题显而易见:每加一个工具文件都要回来补一行,漏补就等于这工具不存在,而且没有任何报错——它只是静默地没被发现。Hermes 的做法是让发现自动化:扫 tools/ 目录,谁在顶层调了 registry.register(),就 import 谁。

discover_builtin_tools():扫目录、按需 import

def discover_builtin_tools(tools_dir=None) -> List[str]:
    """Import 所有自注册的内置工具模块,返回它们的模块名。"""
    tools_path = Path(tools_dir) if tools_dir else Path(__file__).resolve().parent
    module_names = [
        f"tools.{path.stem}"
        for path in sorted(tools_path.glob("*.py"))            # 扫 tools/*.py
        if path.name not in {"__init__.py", "registry.py", "mcp_tool.py"}  # 排除三个
        and _module_registers_tools(path)                      # ← 顶层真的调了 register 吗?
    ]
    imported = []
    for mod_name in module_names:
        try:
            importlib.import_module(mod_name)                  # import 触发它的 register()
            imported.append(mod_name)
        except Exception as e:
            logger.warning("Could not import tool module %s: %s", mod_name, e)
    return imported
tools/registry.py · discover_builtin_tools()

三步:① glob("*.py") 列出目录里所有 Python 文件;② 排除三个特殊文件 + 用 AST 判断这文件顶层是否真的调了 registry.register;③ 对筛出来的模块逐个 import_module——import 的动作就触发了那个文件顶层的 register(),工具自动上户口。单个模块 import 失败只记 warning、不连累其余(某个工具的可选依赖没装,不该拖垮整个工具系统)。

为什么用 AST 检查,而不是「import 了再说」

关键在 _module_registers_tools()——它先读源码、解析成 AST、只看模块顶层语句里有没有 registry.register(...) 调用,而不是「先 import 进来看看」:

def _module_registers_tools(module_path: Path) -> bool:
    try:
        source = module_path.read_text(encoding="utf-8")
        tree = ast.parse(source, filename=str(module_path))
    except (OSError, SyntaxError):
        return False
    return any(_is_registry_register_call(stmt) for stmt in tree.body)   # 只看 tree.body(顶层)

def _is_registry_register_call(node) -> bool:
    """判断 node 是不是一句 registry.register(...) 调用表达式。"""
    if not isinstance(node, ast.Expr) or not isinstance(node.value, ast.Call):
        return False
    func = node.value.func
    return (isinstance(func, ast.Attribute) and func.attr == "register"
            and isinstance(func.value, ast.Name) and func.value.id == "registry")
tools/registry.py · _module_registers_tools / _is_registry_register_call

两个「为什么」:

  • 为什么先 AST 筛、而不是 import 全部:很多 tools/ 下的文件是辅助模块(工具函数、解析器、helper),它们自己不是工具,但可能在某个函数体内部调 registry.register()(比如动态注册)。如果无脑 import 所有 .py,会把这些纯 helper 也拖进来、甚至误触发它们的副作用。AST 检查只认「顶层直接写着 registry.register(...)」的文件——any(... for stmt in tree.body) 里的 tree.body 就是模块顶层,函数体内部的调用不算。
  • 为什么解析失败就跳过:ast.parse 遇到 OSError(读不了)或 SyntaxError(语法错)直接返回 False——一个语法坏掉的文件不会中断整个发现流程,只是被安静略过。
AST 检查是「静态」的——它不执行代码,只读文本、看结构。所以它能在「决定要不要 import」之前就判断出「这文件是不是一个工具」,避免了「为了知道该不该导入,只好先导入一遍」的鸡生蛋问题。

三个被排除的文件

文件为什么排除
__init__.py包初始化文件,不是工具
registry.py就是注册表自己。它定义了 register,但不调用 registry.register(而且它处在导入链最底层,不能反过来依赖工具)
mcp_tool.pyMCP 工具是运行时动态注册的,不走「模块顶层 register」这条路。放进自动发现会和它的动态刷新机制打架(见第 5 篇)
mcp_tool.py 的排除是理解 MCP 的钥匙。内置工具是「import 一次、静态登记、整个进程不变」;而 MCP 工具是「运行时连上外部 server、按 server 返回的清单动态 register/deregister、还能中途刷新」。两种生命周期完全不同,所以 MCP 不参与这套「开机自动发现」,由它自己的客户端在运行时往同一个注册表里增删(第 5 篇细讲)。

何时运行:import model_tools 的副作用

发现不是被谁「主动调用」的,而是 model_tools.py 在模块顶层就调了一次——只要有人 import 了 model_tools,发现就跑:

# model_tools.py 顶层
from tools.registry import discover_builtin_tools, registry
...
# ===== Tool Discovery(import 每个模块即触发它的 register 调用)=====
discover_builtin_tools()
model_tools.py · 模块顶层

所以「谁 import model_tools,谁就触发了全套工具发现」。而 run_agent.py / cli.py / batch_runner.py 都会 import model_tools——于是不管从哪个前端进来,工具都在第一次用到工具系统时就全部登记就位了。

一个历史教训:MCP 发现没放在这里。源码注释记着——外部 MCP server 的发现曾经也放在这个模块顶层当副作用,但 discover_mcp_tools() 内部有个阻塞的 future.result(timeout=120),而网关是在 asyncio 事件循环里、收到第一条用户消息时才懒加载 model_tools 的——结果只要有个配置的 MCP server 慢或连不上,就会把 Discord/Telegram 的心跳冻住最多 120 秒(#16856)。所以 MCP 发现被挪走了,只留内置工具的同步发现在这。

回顾:循环导入安全的 import 链

把第 1 篇的单例、这一篇的发现串起来,整条依赖链是单向的:

flowchart BT
    R["tools/registry.py
(定义 registry 单例 + discover_builtin_tools)
谁都不依赖"]
    T["tools/*.py
(顶层 registry.register 上户口)
只从 registry 导入"]
    M["model_tools.py
(顶层调 discover_builtin_tools + 查注册表)"]
    U["run_agent.py / cli.py / batch_runner.py"]
    T --> R
    M --> R
    M -.import 触发.-> T
    U --> M

因为 registry.py 不 import 任何工具、也不 import model_tools,它永远在最底层,导入不会成环。model_tools 顶层的 discover_builtin_tools() 反过来动态 import 那些工具文件(虚线),触发它们的 register()——这条「动态触发」不算静态依赖,不影响导入顺序。

发现了 ≠ 暴露了(重要伏笔)

到这里,所有内置工具都已经登记进注册表了。但请记住第 1 篇末尾那句:登记进 _tools 只是上了户口,模型还看不到它们。自动发现只保证「工具的 schema/handler 都在注册表里备好了」;至于哪些工具会被真正下发给模型,取决于工具集——它得先属于一个被当前平台启用的工具集,还要通过 check_fn。这正是下一篇的主题。

这是全模块最容易踩的坑:你新写了一个工具文件、register() 也调了、get_errors 也没报错,但模型死活不认它。九成是——你忘了把工具名加进 toolsets.py 的某个工具集。自动发现只做了一半,「接进工具集」是另一半、且是手动的(见第 3、5 篇)。

本篇要点

  • discover_builtin_tools() 扫 tools/*.py,用 AST 只挑出「顶层调了 registry.register」的文件并 import,触发它们上户口。
  • 用 AST 静态检查(不执行代码)是为了放过纯 helper 模块——只有顶层直接写着 register 的才算工具;函数体内部的 register 不算。
  • 排除 __init__.py / registry.py / mcp_tool.py;MCP 因为是运行时动态注册,不走这条静态发现路。
  • 发现是 model_tools.py 模块顶层的一次性副作用——谁 import model_tools,谁就触发全套发现。
  • 整条 import 链单向朝上,循环导入安全。
  • 发现 ≠ 暴露:登记进注册表后,还要「被启用的工具集收录 + 通过 check_fn」才会下发给模型——下一篇。
