工具集与「注册 ≠ 暴露」

前两篇里,几十个工具都登记进了注册表。但不是每个前端都该拿到全部工具——CLI 要终端和文件,Telegram 网关就未必。toolsets.py 把工具分组打包成「哪个平台用哪一套」,并落实全模块最重要的一条铁律:注册 ≠ 暴露。

工具集是什么:一个分组标签

「工具集(toolset)」就是一组工具的命名集合。定义全在 toolsets.py 的一个大字典 TOOLSETS 里,每个条目三个字段:

TOOLSETS = {
    "web": {
        "description": "Web research and content extraction tools",
        "tools": ["web_search", "web_extract"],   # 直接包含的工具名
        "includes": []                             # 组合进来的其它工具集
    },
    "browser": {
        "description": "Browser automation...",
        "tools": ["browser_navigate", "browser_snapshot", ...],
        "includes": []
    },
    # ...
}
toolsets.py · TOOLSETS
字段作用
description人读说明(hermes tools 界面、doctor 会显示)
tools本工具集直接包含的工具名列表
includes把别的工具集整个并进来(组合),实现「大套装 = 小套装拼起来」

当前的工具集键(节选):web、search、vision、video、image_gen、terminal、file、browser、skills、memory、todo、session_search、clarify、code_execution、delegation、cronjob、homeassistant、kanban、discord / discord_admin、messaging,以及一大批平台套装 hermes-cli / hermes-telegram / hermes-discord / hermes-slack / hermes-gateway ……

_HERMES_CORE_TOOLS:每个平台共享的底座

toolsets.py 顶部有一个列表 _HERMES_CORE_TOOLS——它是「几乎每个平台都该有的核心工具」的单一清单:web 搜索、终端、文件读写、技能、浏览器、todo、memory、委派、定时……改一处,所有平台同步更新。

_HERMES_CORE_TOOLS = [
    "web_search", "web_extract",

toolsets.py · _HERMES_CORE_TOOLS

平台套装就是「核心 + 该平台特有的少量工具」。比如 hermes-cli ≈ _HERMES_CORE_TOOLS 原样;messaging 平台在核心之上加 send_message 之类。这样每个平台的工具表都是「共享底座 + 各自的小 delta」。

_HERMES_CORE_TOOLS 不是死代码——它是每个平台基础工具集继承的默认包。第 5 篇讲「新增内置工具」时会看到:光 register() 还不够,得手动把工具名加进 _HERMES_CORE_TOOLS(或某个具体工具集),它才会被平台收录。

resolve_toolset:把套装递归展开成工具名

工具集之间能靠 includes 层层组合(套装含子套装),所以取「一个工具集到底有哪些工具」时要递归展开。这就是 resolve_toolset():

def resolve_toolset(name, visited=None) -> List[str]:
    if visited is None: visited = set()
    if name in {"all", "*"}:                       # 特殊别名:所有工具集的并集
        ...
        return sorted(all_tools)
    if name in visited: return []                  # 环 / 菱形依赖:静默返回,避免重复/死循环
    visited.add(name)
    toolset = get_toolset(name)
    if not toolset:
        if name.startswith("hermes-"):             # 找不到定义,但形如 hermes-<平台>
            ... 自动生成:_HERMES_CORE_TOOLS + 该插件平台注册的工具 ...
        return []
    tools = set(toolset.get("tools", []))          # 直接工具
    for included in toolset.get("includes", []):   # 递归并入 includes
        tools.update(resolve_toolset(included, visited))
    return sorted(tools)
toolsets.py · resolve_toolset()
  • "all" / "*":返回所有工具集的并集——将来新增工具集会自动包含,不用改这里。
  • visited 集合:处理「环」和「菱形依赖」(A 含 B、A 含 C、B 和 C 都含 D)。已访问过就返回 [],不重复展开、也不会无限递归。
  • 找不到定义、但名字是 hermes-<平台>:自动生成一个工具集——给它 _HERMES_CORE_TOOLS + 该插件平台注册进同名工具集的工具。这样插件平台无需在 TOOLSETS 里预留条目,也能拿到核心工具。

前端如何选工具集 + 启用/禁用

每个平台的适配器会挑一个基础工具集(如 Telegram 用 messaging、CLI 用 hermes-cli)。用户还能在 config.yaml 里用 tools.<platform>.enabled / tools.<platform>.disabled(或 hermes tools 那个 curses 界面)微调。最终这些汇成两个参数传给 get_tool_definitions(enabled_toolsets=..., disabled_toolsets=...)。解析规则(model_tools._compute_tool_definitions):

# ① enabled 给了 → 只收这些工具集(逐个 validate + resolve,累加)
if enabled_toolsets is not None:
    for ts in effective_enabled_toolsets:
        if validate_toolset(ts):
            tools_to_include.update(resolve_toolset(ts))
else:
    # ② 没给 enabled → 默认收全部工具集
    for ts in get_all_toolsets():
        tools_to_include.update(resolve_toolset(ts))

# ③ disabled 永远在最后做减法(即使某个大套装被 enable,disabled 里的照样剔除)
if disabled_toolsets:
    for ts in disabled_toolsets:
        tools_to_include.difference_update(resolve_toolset(ts))
model_tools.py · _compute_tool_definitions()
禁用「平台套装」的陷阱(#33924)——为什么不能直接减整包。平台套装(hermes-*)里含 _HERMES_CORE_TOOLS。如果用户把某个 hermes-xxx 塞进 disabled_toolsets,直接减掉整包会连 terminal / read_file 这些核心工具一起剃光,把模型的工具列表清空。所以对 hermes-* 的禁用改用 bundle_non_core_tools()——只减该套装「非核心的那部分 delta」,保留核心。禁用一个平台套装 = 去掉它的平台特有工具,核心不动。
kanban worker 的强制收录:被调度器 spawn 的看板 worker(设了 HERMES_KANBAN_TASK 环境变量)即使 profile 出于省 token 缩了工具集,也会被强行补上 kanban 工具集——否则它就没法 kanban_complete/block/heartbeat 上报进度了。

铁律:注册 ≠ 暴露

现在可以把这条铁律讲透了。一个工具要真正被模型看到,必须同时满足:

flowchart TD
    A["① register 进注册表
(第 1、2 篇:自动发现 + 上户口)"] --> B{② 它的工具名
在某个被启用的工具集里吗?}
    B -- 否 --> X1["❌ 模型看不到
(登记了也白搭)"]
    B -- 是 --> C{③ 没落进
disabled_toolsets?}
    C -- 落进了 --> X2[❌ 被减掉]
    C -- 没有 --> D{④ check_fn 通过?
(第 4 篇)}
    D -- 否 --> X3["❌ 静默不可用
(缺凭证/依赖没装)"]
    D -- 是 --> Y[✅ schema 下发给模型]

注意 register() 时那个 toolset="xxx" 参数——它只是给工具贴了个组别标签,并不自动把工具塞进 TOOLSETS["xxx"]。要暴露,必须让这个工具名出现在某个会被 resolve_toolset 展开、且被平台启用的工具集的 tools 列表里。二者靠工具名字符串对上,不是自动关联的。

你做了什么结果
只 register(name="foo", toolset="bar")foo 在注册表里,但模型看不到(除非恰好有个被启用的工具集列了 "foo")
再把 "foo" 加进 _HERMES_CORE_TOOLS 或某工具集的 tools✅ 该平台启用该工具集时,foo 被暴露
foo 有 check_fn 且返回 False即便被收录,也静默不下发(如缺 API key)
这就是新手最常见的「我明明写了工具,模型却说 Tool foo does not exist」的根因。九成是漏了第 ② 步——没把工具名接进任何被启用的工具集。register() 成功、get_errors 无报错,都不代表模型能看到它。这是内核「窄腰」设计的直接后果:每个工具都要在每次 API 调用时下发,所以默认不暴露,必须显式接进工具集。

为什么要这么克制

因为「暴露给模型」是有成本的:每个工具的 schema 都会在每一次 API 调用里发给模型——占 prompt token(花钱)、占模型注意力(工具越多越容易点错)。所以工具系统的默认姿态是「按需暴露」:

  • 平台只拿它用得上的工具集(网关不需要桌面 GUI 的 project 工具);
  • 凭证没配的工具自动隐身(check_fn,下一篇);
  • 大量 MCP/插件工具还能延迟到 tool_search 背后,不占常驻 schema(下一篇)。
「注册 ≠ 暴露」不是麻烦,而是成本纪律的落点。注册表尽管把所有工具都备齐;真正下发给模型的,永远只是「这个平台、这个会话、此刻可用」的那一小撮。

本篇要点

  • 工具集是工具名的命名分组,定义在 TOOLSETS(tools 直接列 + includes 组合);_HERMES_CORE_TOOLS 是各平台共享的底座。
  • resolve_toolset() 递归展开 includes,处理环/菱形依赖,支持 all/*,并为 hermes-<平台> 自动生成套装。
  • 暴露 = enabled_toolsets 累加 → disabled_toolsets 最后统一减;禁用 hermes-* 套装只减非核心 delta,保住核心工具(#33924)。
  • 铁律:注册 ≠ 暴露。register(toolset=...) 只贴标签,必须把工具名接进某个被启用的工具集,再通过 check_fn,模型才看得到。
  • 克制暴露是成本纪律:每个工具 schema 每次调用都下发,占 token 占注意力,所以默认按需暴露。
