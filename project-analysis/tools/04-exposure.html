暴露给模型:get_tool_definitions

上一篇把「哪些工具名该收录」算清楚了。这一篇讲最后一公里:get_tool_definitions() 如何把这些名字变成真正发给模型的 schema 列表——中间夹着 check_fn 可用性过滤、动态 schema、结果缓存、几处后处理重写,以及把海量 MCP/插件工具藏到 tool_search 背后的渐进式暴露。

全景:一条流水线

每轮调模型前,conversation_loop 都会拿到一份工具 schema 列表(见 Agent 模块第 3 篇)。这份列表就是 get_tool_definitions() 产出的。它是一条流水线:

缓存命中?}
    MEMO -- 命中 --> RET[直接返回缓存副本]
    MEMO -- 未命中 --> B["解析工具集 → 工具名集合
(第 3 篇)"]
    B --> C["registry.get_definitions(names)
① check_fn 过滤
② dynamic_schema_overrides"]

execute_code / discord / browser_navigate"]

(llama.cpp 兼容)"]
    E --> F["Tool Search 渐进式暴露
(MCP/插件超阈值→藏到桥后)"]

(进程全局)"]

下面逐段拆。

① check_fn 可用性过滤 + 30 秒 TTL 缓存

工具名集合交给 registry.get_definitions(names)。它对每个工具:若登记了 check_fn,就调一次看返回真假——假就跳过(不下发),这就是「缺 API key 的工具自动隐身」的实现:

for name in sorted(tool_names):
    entry = entries_by_name.get(name)
    if not entry: continue
    if entry.check_fn:
        if entry.check_fn not in check_results:
            check_results[entry.check_fn] = _check_fn_cached(entry.check_fn)  # 30s TTL 缓存
        if not check_results[entry.check_fn]:
            continue                       # check 没过 → 静默不下发
    schema_with_name = {**entry.schema, "name": entry.name}
    ... 应用 dynamic_schema_overrides ...
    result.append({"type": "function", "function": schema_with_name})   # ← OpenAI 格式包装
tools/registry.py · get_definitions()

注意最后那层 {"type": "function", "function": {...}} 包装——这是 OpenAI 工具 schema 的标准外壳,模型 API 认的就是这个形状。

为什么 check_fn 要缓存:探测很贵

很多 check_fn 要探测外部状态:check_terminal_requirements 会 subprocess.run(["docker","version"])、浏览器工具探测 playwright 装没装、Modal SDK 在不在。对一个长期运行的 CLI/网关进程,每次生成 schema 都重探一遍纯属浪费——外部状态是按「人的时间尺度」变的。所以结果缓存约 30 秒:

_CHECK_FN_TTL_SECONDS = 30.0            # 缓存有效期
_CHECK_FN_FAILURE_GRACE_SECONDS = 60.0 # 失败宽限窗口(见下)
_check_fn_cache: Dict[Callable, tuple[float, bool]] = {}
_check_fn_last_good: Dict[Callable, float] = {}   # 每个 check_fn 上次返回 True 的时刻

30 秒也够短:用户用 hermes tools enable foo 翻开关、或改了凭证文件,一两轮内就会生效,不用手动刷缓存。

关键设计:抖动宽限——别因一次超时就剃掉工具(#21658 / #5304)

这些探测会偶发抖动:负载高时一次 docker version 超时,check_fn 就返回一次 False。如果直接把这个 False 缓存下来,会在那一瞬间把整个 terminal+file 工具集从正在构建的 agent 上剃掉——最典型的受害者是刚 spawn 的 delegate_task 子 agent,它于是报「Tool read_file does not exist」。为此加了宽限逻辑:

if value:                       # 探测成功
    _check_fn_last_good[fn] = now
    _check_fn_cache[fn] = (now, True); return True
# 探测失败:
last_good = _check_fn_last_good.get(fn)
if last_good is not None and now - last_good < _CHECK_FN_FAILURE_GRACE_SECONDS:
    # 距上次成功不到 60s → 判定为抖动:返回上次的 True,且【不缓存】这次失败
    logger.warning("check_fn ... transient ... keeping tool(s) available")
    return True
# 没有近期成功(或宽限已过)→ 认这次失败,缓存 False
_check_fn_cache[fn] = (now, False); return False
tools/registry.py · _check_fn_cached()
情形处理
探测成功缓存 True(30s),并记下「上次成功时刻」
失败,但距上次成功 < 60s判为抖动:返回上次的 True,不缓存失败(下次会重探)
失败,且距上次成功 ≥ 60s(或从没成功过)认账:缓存 False,工具本轮不可用
拿捏的平衡:既要吸收瞬时抖动(不因一次超时就误伤工具),又不能钉死一个过时的「可用」(真的挂了得反映出来)。所以宽限只在「最近成功过」时生效、且失败不入缓存(下次立刻重探);后端要是真的宕了,超过宽限窗口就如实标不可用。改了配置想立即生效,还有 invalidate_check_fn_cache() 一把清空(hermes tools 改动后会调)。

② dynamic_schema_overrides:描述随配置变

有些工具的 schema 该反映当前配置。典型是 delegate_task:它的描述里要告诉模型「你最多能并发几个子任务、能嵌套几层」,而这些数字来自用户的 delegation.max_concurrent_children / max_spawn_depth。如果写死在静态 schema 里,用户改了配置、模型却被告知错的上限。

所以 register() 支持传一个零参回调 dynamic_schema_overrides,在生成 schema 时调用、把返回的 dict 浅合并覆盖上去:

if entry.dynamic_schema_overrides is not None:
    try:
        overrides = entry.dynamic_schema_overrides()   # 读当前配置,返回 {描述/参数...}
        if isinstance(overrides, dict):
            schema_with_name.update(overrides)          # 浅合并覆盖
    except Exception as exc:
        logger.warning("... using static schema")       # fail-open:出错就用静态 schema
它每次生成 schema 都会被调,所以能实时反映配置。而外层 get_tool_definitions 的缓存键里带了 config.yaml 的 mtime+大小指纹(见下),所以改了 delegation.*,缓存会自动失效、重算——动态描述和缓存不打架。

③ 记忆化:代际号 + 配置指纹当缓存键

生成 schema 不便宜(要解析工具集、跑一堆 check_fn、做后处理)。而网关/agent 初始化是热路径。所以 get_tool_definitions 在 quiet_mode=True 时memo结果:

cache_key = (
    frozenset(enabled_toolsets) if enabled_toolsets is not None else None,
    frozenset(disabled_toolsets) if disabled_toolsets else None,
    registry._generation,          # ← 注册表一变(MCP刷新/插件加载)就失效(第1篇)
    cfg_fp,                        # ← config.yaml 的 (mtime_ns, size),配置一改就失效
    bool(os.environ.get("HERMES_KANBAN_TASK")),  # kanban worker 用不同缓存条目
    bool(skip_tool_search_assembly),
)
model_tools.py · get_tool_definitions()
  • 只在 quiet_mode 缓存:非静默路径会打印工具选择日志,不走缓存。
  • registry._generation(第 1 篇那个代际号):任何工具增删都让它 +1,旧缓存自动作废——这就是代际计数器的用武之地。
  • 配置指纹 cfg_fp:不用给每个改配置的地方都挂失效钩子,靠文件 mtime+size 就能感知「影响动态 schema 的配置改了」。
  • LRU 上限 8(#19251):长寿网关一辈子会见到很多不同的「工具集×配置」指纹,不设上限会无限膨胀。超过 8 条就淘汰最老的。
  • 返回浅拷贝(#17335):命中也返回 list(cached) 的浅拷贝——因为上层(run_agent)会往列表里 append 记忆/LCM 工具 schema,若直接返回缓存本体会被就地污染,导致长寿网关跨多次 agent 初始化累积重名工具,而 DeepSeek/Kimi 等强制工具名唯一的提供商会因此返回 HTTP 400。

④ 后处理:三处按运行时重写 schema

拿到 registry.get_definitions 的结果后,还有三处工具的 schema 要按此刻真实可用的工具重写。它们的共同动机:schema 里不能写死对「别的工具」的引用——那些工具可能此刻不可用,模型看到就会幻觉去调不存在的工具。

available_tool_names = {t["function"]["name"] for t in filtered_tools}  # 真正过了 check 的
工具按什么重写不重写会怎样
execute_code用 SANDBOX_ALLOWED_TOOLS & available 重建 schema,只列沙箱里此刻真能用的工具模型以为沙箱里能 web_search,但 key 没配/工具集禁了 → 白调
discord / discord_admin据 bot 的特权 intents(查 /applications/@me)+ 用户 allowlist 重建,隐藏 bot 不支持的动作模型尝试 bot 权限不允许的操作 → 必失败;拿不到动态 schema 时干脆整个工具剔除
browser_navigate当 web_search/web_extract 不可用时,删掉描述里「优先用 web_search/web_extract」那句描述引导模型去调不存在的 web 工具 → 幻觉调用
# browser_navigate 的例子:web 工具都没有时,剥掉那句交叉引用
if "browser_navigate" in available_tool_names:
    if not ({"web_search","web_extract"} & available_tool_names):
        desc = desc.replace(" For simple information retrieval, prefer web_search or web_extract...", "")
这就是 AGENTS.md 里那条铁规的落点:「工具 schema 描述里不要硬编码别的工具集里的工具名」。静态 schema 若写死「优先用 web_search」,一旦 web 工具集被禁/缺 key,模型就会点名一个不存在的工具。需要交叉引用时,不写进静态 schema,而是在 get_tool_definitions 这里根据「此刻真可用的工具」动态加——正是这三个后处理块的模式。

⑤ schema 清洗:兼容各种后端

后处理完,过一遍 sanitize_tool_schemas()。不同推理后端对 schema 形状的容忍度不同:云厂商悄悄接受的一些形状(空 {"type":"object"} 无 properties、坏 MCP server 吐的字符串型 schema 节点),会被 llama.cpp 的 json-schema→GBNF 转换器拒绝(它要用 schema 生成工具调用的语法解析器)。清洗把这些形状规整成人人都吃的样子;已经规范的 schema 则原样通过(no-op)。出错也只 warning、不挡工具加载。

⑥ Tool Search:把海量 MCP/插件工具藏到桥后

流水线最后一步。装了一堆 MCP server / 插件后,可延迟暴露的工具 schema 可能吃掉大量上下文。Tool Search 做渐进式暴露:当「可延迟的工具」总量超过阈值(默认上下文窗口的 10%)时,把这些 MCP + 插件(非核心)工具替换成三个桥工具——tool_search / tool_describe / tool_call,模型需要时先「搜工具→看详情→调用」,而不是把几百个工具 schema 常驻在每次请求里。

if not skip_tool_search_assembly and ts_cfg.enabled != "off":
    assembly = assemble_tool_defs(filtered_tools, context_length=..., config=ts_cfg)
    filtered_tools = assembly.tool_defs   # MCP/插件被折叠成三个桥工具
model_tools.py · tools/tool_search.py
  • 核心工具永不延迟:_HERMES_CORE_TOOLS 里的工具从不被藏到桥后——它们太常用,常驻更划算。只延迟 MCP/插件这些「多而不一定用」的。
  • 幂等 + 最后做:放在清洗之后、返回之前,且装配是幂等的(即使有人重复调 get_tool_definitions 也不会叠加)。
  • skip_tool_search_assembly:三个桥工具的 handler 内部要读真实的完整目录(不是已折叠的),就靠这个参数拿到「装配前」的原始列表。外部调用方保持默认 False。
这是「窄腰」思想在暴露层的延伸:即使工具很多,常驻 schema 只放核心 + 三个桥,其余按需检索。既省 token,又不让模型在几百个工具里挑花眼。设计细节见 tools/tool_search.py。

顺带:_last_resolved_tool_names(进程全局)

流水线结尾,把最终工具名写进一个进程全局 _last_resolved_tool_names(缓存命中路径也会更新,保持一致):

global _last_resolved_tool_names
_last_resolved_tool_names = [t["function"]["name"] for t in filtered_tools]

execute_code 靠它知道「沙箱里此刻能用哪些工具」。它是进程全局、会被最近一次 get_tool_definitions 覆盖——之所以安全,是因为 agent 循环是同步顺序执行的。Agent 模块第 4 篇提过一个坑:delegate_tool 里 _run_single_child() 会在跑子 agent 前后保存并恢复这个全局,因为子 agent 运行期间它会被临时改写。

本篇要点

  • get_tool_definitions() 是条流水线:解析工具集 → registry.get_definitions(check_fn 过滤 + 动态 schema)→ 后处理重写 → schema 清洗 → Tool Search 装配 → 返回。
  • check_fn 决定「此刻可不可用」,结果 30s TTL 缓存;60s 抖动宽限吸收探测偶发失败,避免误剃工具(delegate 子 agent「read_file 不存在」#21658/#5304)。
  • dynamic_schema_overrides 让 schema 随配置变(如 delegate_task 的并发/嵌套上限);fail-open。
  • 结果记忆化:键含 registry._generation + config 指纹 + kanban env,LRU 上限 8(#19251),返回浅拷贝防污染(#17335)。
  • 三处后处理(execute_code / discord / browser_navigate)按运行时可用工具重写 schema——禁止在静态 schema 里硬编码跨工具引用,否则模型幻觉调不存在的工具。
  • Tool Search 把超阈值的 MCP/插件工具折叠成三个桥工具;核心工具永不延迟。
