分发与扩展:点名之后 + 新增工具

模型看到了工具、并在回复里点名要调其中一个。这一篇讲「点名之后」:请求怎么落到对应 handler、为什么有些工具走的是另一条路(agent 级拦截)、handler 的 JSON 契约与错误清洗;最后讲如何往这套系统里加东西——MCP 动态刷新、插件工具,以及给内核加一个内置工具的正确姿势。

点名之后:两条分发路

模型返回的 tool_calls 里每个都带 name + arguments。Agent 模块第 4 篇讲了并发/串行怎么调度;这一篇聚焦「单个工具调用落到哪」。关键:不是所有工具都走 registry.dispatch——有一批「agent 级工具」被抢先拦截:

路 A:agent 级工具的抢先拦截

有些工具离不开 agent 实例本身的状态——todo 列表存在 agent._todo_store、memory 存在 agent._memory_store、clarify 要回调前端、delegate_task 要 spawn 子 agent。这些没法用「无状态的 registry.dispatch」跑,所以在 invoke_tool() 里按名字分流、就地执行:

# agent/agent_runtime_helpers.py · invoke_tool()
if function_name == "todo":
    def _execute(next_args):  # 用 agent._todo_store
        from tools.todo_tool import todo_tool
        return _finish_agent_tool(todo_tool(todos=..., store=agent._todo_store), next_args)
elif function_name == "session_search":
    ...  # 用 agent._get_session_db_for_recall()
elif function_name == "memory":
    def _execute(next_args):
        result = memory_tool(action=..., store=agent._memory_store)
        if agent._memory_manager:               # 成功写入还要镜像到外部记忆提供商
            agent._memory_manager.notify_memory_tool_write(result, ...)
        return _finish_agent_tool(result, next_args)
elif agent._memory_manager and agent._memory_manager.has_tool(function_name):
    ...  # 记忆提供商注册的工具
elif function_name == "clarify":
    ...  # 用 agent.clarify_callback 回调前端
elif function_name == "read_terminal":
    ...  # 用 agent.read_terminal_callback
elif function_name == "delegate_task":
    def _execute(next_args):
        return _finish_agent_tool(agent._dispatch_delegate_task(next_args), next_args)  # spawn 子 agent
else:
    def _execute(next_args):                    # ← 其余全部工具:落到 model_tools
        return handle_function_call(function_name, next_args, effective_task_id, ...)
agent/agent_runtime_helpers.py · invoke_tool()

换句话说:agent 级工具在到达 model_tools.handle_function_call 之前就被截胡了;只有 else 分支(绝大多数普通工具:终端、文件、网页、浏览器…)才继续往下走。

双保险:model_tools.py 里有个 _AGENT_LOOP_TOOLS = {"todo","memory","session_search","delegate_task"},并在 handle_function_call 开头兜底:万一这些名字真漏到了这里,直接返回 {"error": "... must be handled by the agent loop"}——它们本就不该在这层被执行,这是一道防止「走错路还静默出错」的护栏。

路 B:handle_function_call → registry.dispatch

普通工具进 handle_function_call() 后,并不是立刻执行,中间还夹着好几道关(细节见第 4 篇与 Agent 模块第 4 篇):工具请求中间件(改参数)、pre_tool_call 插件钩子(可拦停)、ACP 编辑审批、越权/守卫评估……全过了,才用一个 _dispatch 闭包真正调注册表:

def _dispatch(next_args):
    return registry.dispatch(function_name, next_args,
                             task_id=task_id, session_id=session_id, user_task=user_task)

result = run_tool_execution_middleware(function_name, function_args, _dispatch, ...)
model_tools.py · handle_function_call()

(execute_code 是特例:它的 _dispatch 会把「沙箱里可用的工具列表」也传进去,并优先用调用方给的列表,防止子 agent 通过进程全局 _last_resolved_tool_names 篡改父的工具集。)

registry.dispatch:JSON 契约 + 异步桥接 + 错误清洗

最终执行落在注册表的 dispatch()。它很短,但把三件事定死了:

def dispatch(self, name, args, **kwargs) -> str:
    entry = self.get_entry(name)
    if not entry:
        return json.dumps({"error": f"Unknown tool: {name}"})   # 未知工具:统一错误形状
    try:
        if entry.is_async:
            from model_tools import _run_async
            return _run_async(entry.handler(args, **kwargs))      # 异步 handler 自动桥接
        return entry.handler(args, **kwargs)                      # 同步直接调
    except Exception as e:
        logger.exception("Tool %s dispatch error: %s", name, e)
        raw = f"Tool execution failed: {type(e).__name__}: {e}"
        sanitized = _sanitize_tool_error(raw)                     # 清洗错误文本
        return json.dumps({"error": sanitized})
tools/registry.py · dispatch()
保证说明
永远返回 JSON 字符串成功由 handler 自己返回 JSON;未知工具/异常由 dispatch 包成 {"error": ...}。上层永远拿到可解析的字符串,不会因工具崩溃拿到裸异常
异步自动桥接is_async=True 的 handler 返回协程,_run_async() 在共享事件循环里跑完再返回结果——工具作者写 async 也能被同步的分发调用
错误文本清洗异常字符串里可能混着 XML 标签 / 代码围栏 / CDATA 之类结构性噪声,直接喂给模型会被当成结构指令。_sanitize_tool_error 把这些框架 token 去掉,免得污染模型上下文
handler 的铁律:必须返回 JSON 字符串。这是整个工具系统的统一契约——第 1 篇的 example_tool 结尾 return json.dumps({...}) 就是它。返回别的类型(dict、None、裸字符串)会在回灌给模型时出问题。注册表只在出错时替你兜底成 JSON,正常路径的 JSON 化是 handler 自己的责任。

扩展一:MCP 工具的运行时动态刷新

第 2 篇说过 mcp_tool.py 被排除在自动发现之外——因为 MCP 工具不是开机静态登记,而是运行时连上外部 MCP server、按它返回的清单动态注册,还能中途刷新。流程:

  • MCP 客户端收到 server 的 notifications/tools/list_changed;
  • 对旧工具 registry.deregister(),对新工具 registry.register(..., toolset="mcp-<server>", override=True, is_async=True);
  • MCP↔MCP 覆盖是被允许的(第 1 篇的防覆盖闸对 mcp-* 前缀放行);
  • 每次增删都让 _generation += 1 → 第 4 篇的 get_tool_definitions 缓存自动失效,下次重算就带上新工具。
为什么这套能安全并发:MCP 刷新会在一个线程里增删注册表,而别的线程可能正在读工具元数据生成 schema。注册表用一把 RLock 把所有 mutation 串行化,读方则通过 _snapshot_entries() 在锁内拿一份稳定快照再慢慢用。配合单调递增的 _generation,读方还能安全地 memo。这就是第 1 篇「代际号 + 锁 + 快照」三件套的完整用途。

扩展二:插件工具

插件不改核心代码,靠 register(ctx) 里的 ctx.register_tool(...) 把工具送进同一个注册表(底层还是 registry.register)。它们和内置工具走完全一样的暴露/分发路,只是多了一层越权保护(第 1 篇的 override 策略):

  • 插件工具默认不能覆盖内置工具。想覆盖(比如把默认浏览器换成自家 CDP 后端),要 override=True 且操作员在 config 里 plugins.entries.<id>.allow_tool_override: true;
  • 授权靠 _plugin_owner_of(handler) 认定——绑定在 handler 的定义模块(__globals__["__name__"])上,而非调用点,所以插件没法用回调洗白一次越权;
  • 授权策略 register_plugin_override_policy() 是持久的,即便插件事后用线程/延迟回调再注册,也受同一条策略约束。
规矩(AGENTS.md):插件不得改核心文件。插件要的能力若框架没暴露,应该去拓宽通用插件面(加个 hook、加个 ctx 方法),而不是把插件专属逻辑硬塞进 run_agent.py / model_tools.py / 注册表。工具系统给插件留的口子就是 ctx.register_tool + 那套 override 策略——够用且受控。

如何新增一个内置工具

如果你是在给内核贡献一个新的核心工具(注意:AGENTS.md 的「Footprint Ladder」说大多数能力不该做成核心工具,优先 CLI+技能 / 服务门控 / 插件 / MCP),正确姿势是改两个文件:

  1. 建 tools/your_tool.py

    写好 handler(返回 JSON 字符串)+ 可选 check_requirements,模块顶层调一次 registry.register(...)(见第 1 篇范式)。放进 tools/ 目录即被自动发现(第 2 篇),无需手写 import。

  2. 把工具名加进 toolsets.py

    加到 _HERMES_CORE_TOOLS(所有平台可见)或某个具体工具集的 tools 列表。这一步是必须的、且是手动的——自动发现只负责登记 schema;工具要暴露给 agent,名字必须出现在某个被启用的工具集里(第 3 篇的「注册 ≠ 暴露」)。

最常踩的坑,再强调一次:只做了第 1 步、忘了第 2 步 —— 工具在注册表里、get_errors 无报错,但模型报「Tool xxx does not exist」。_HERMES_CORE_TOOLS 不是死代码,它是每个平台基础工具集继承的默认包;不把工具名加进去(或加进某个启用的工具集),它就永远隐身。
几个进阶点:schema 描述里若提到文件路径,用 display_hermes_home() 让它随 profile 变;工具要存持久状态,用 get_hermes_home() 而非硬编码 ~/.hermes;agent 级工具(要用 agent 的 store/回调)不走这条路——它们在 run_agent.py / invoke_tool() 里被拦截(本篇路 A),参考 tools/todo_tool.py 的模式。

工具系统就是内核那条「窄腰」的具体形态:注册表把所有工具备齐,但每一层都在克制暴露——按工具集收、按 check_fn 过、超量的藏到 tool_search 后。真正下发给模型的,永远只是「此时此地该有」的那一小撮。想加能力,先爬一遍 Footprint Ladder,大多数时候答案不是「加个核心工具」。

本篇要点

  • 点名后有两条路:todo/memory/session_search/clarify/read_terminal/delegate_task 等 agent 级工具在 invoke_tool 里被抢先拦截(要用 agent 的 store/回调/委派);其余走 handle_function_call → registry.dispatch。
  • handle_function_call 有个 _AGENT_LOOP_TOOLS 兜底:agent 级工具漏到这层直接报错(双保险)。
  • registry.dispatch 定死三条:永远返回 JSON 字符串、异步 handler 自动桥接、异常文本清洗后包成 {"error":...}。handler 必须自己返回 JSON。
  • MCP 运行时动态 register/deregister(mcp-* 可互相覆盖),靠 RLock + 快照 + 代际号并发安全、缓存自动失效。
  • 插件用 ctx.register_tool 进同一注册表,覆盖内置需操作员授权;插件不得改核心文件。
  • 新增内置工具改两个文件:tools/your_tool.py(自动发现)+ toolsets.py(手动接进工具集,否则不暴露)。加核心工具前先掂量 Footprint Ladder。
