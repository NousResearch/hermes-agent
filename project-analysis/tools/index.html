工具系统

模型靠「调用工具」跟真实世界打交道:读文件、跑命令、搜网页、开浏览器。本模块讲清一件事——一个工具从 tools/*.py 里的一行 registry.register() 出发,如何一路被发现、分组、暴露给模型,再被点名执行。拆成 5 篇,建议按顺序读。

这块代码在哪

工具系统的代码是「一个中央注册表 + 一层暴露/过滤 + 一个分发入口 + 一堆各自登记的工具文件」:

文件角色
tools/registry.py中央注册表:ToolRegistry 单例、register()、自动发现、check_fn 缓存、get_definitions()、dispatch()——本模块的主角
toolsets.py工具集:TOOLSETS 字典 + _HERMES_CORE_TOOLS,把零散工具打包成「每个平台用哪一套」
model_tools.py暴露 + 分发:get_tool_definitions() 生成发给模型的 schema 列表,handle_function_call() 接住模型的点名
tools/*.py每个工具文件,模块顶层调一次 registry.register() 给自己上户口(终端、文件、网页、浏览器…)
阅读策略:model_tools.py 很大,别从头读。本模块只追两条主线——暴露(get_tool_definitions)和分发(handle_function_call);其余细节都挂在 tools/registry.py 这一个文件上。

一句话全貌

每个工具文件在被 import 时调 registry.register() 上户口 → discover_builtin_tools() 用 AST 扫描把它们自动发现并 import → toolsets.py 把工具分组成「哪个平台用哪套」→ get_tool_definitions() 按工具集 + check_fn 过滤,生成发给模型的 schema 列表 → 模型点名后 handle_function_call() 分发到对应 handler 执行,结果以 JSON 字符串回灌。
全模块最重要的一条铁律:「注册」不等于「暴露」。一个工具就算成功 register() 进了注册表,只要它的 toolset 没被列进当前启用的工具集(或 check_fn 没通过),模型就永远看不到它。这是内核「窄腰」设计的落点——每个工具都要在每次 API 调用时下发,所以默认只暴露该暴露的。
