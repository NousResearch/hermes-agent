# Repomix Context Packaging

Use this skill when a coding agent needs a compact, portable repository bundle:
large code review, architecture reconnaissance, handoff to Codex, or sending a
slice of a repo to an external model without dumping ad hoc files into chat.

Repomix is useful because it is git-aware, counts tokens, can compress code with
Tree-sitter, and runs Secretlint checks to reduce the chance of packaging
secrets.

## When to use

- Before launching Codex on a broad codebase where context budget matters.
- To hand a bounded repo snapshot to a reviewer or planning agent.
- To compare architecture across repositories without cloning them into the
  active workspace.
- To package a curated file set for debugging, refactoring, or documentation.

Skip it for small, local edits where direct `read_file`/`search_files` is
clearer and cheaper.

## Safety rules

1. **Do not package secrets deliberately.** Repomix includes Secretlint checks,
   but still inspect the target and respect `.gitignore`, `.ignore`, and
   `.repomixignore`.
2. **Prefer tracked source files.** Avoid build outputs, logs, caches, database
   dumps, exported chats, screenshots, and `.env`-like files.
3. **Write bundles outside the repo by default** so generated context files do
   not become accidental project artifacts.
4. **Use narrow includes before broad bundles.** A precise bundle beats a giant
   XML monument to uncertainty.
5. **Do not paste huge bundles into Hermes chat unless needed.** Save the file
   and pass the path to Codex or a subagent.

## Quick commands

Run without installing globally:

```bash
npx --yes repomix@latest --version
```

Pack a repository into a temporary Markdown bundle:

```bash
OUT="/tmp/repomix-$(basename "$PWD")-$(date +%Y%m%d-%H%M%S).md"
npx --yes repomix@latest --style markdown --output "$OUT" .
```

Pack a compressed source-oriented bundle for Codex planning:

```bash
OUT="/tmp/repomix-$(basename "$PWD")-compressed.md"
npx --yes repomix@latest --style markdown --compress --output "$OUT" .
```

Pack a deliberately narrow slice:

```bash
OUT="/tmp/repomix-auth-slice.md"
npx --yes repomix@latest \
  --style markdown \
  --compress \
  --include "src/auth/**,tests/**,README.md,pyproject.toml" \
  --output "$OUT" \
  .
```

Package only a file list produced by git:

```bash
git ls-files "*.py" "*.md" | npx --yes repomix@latest --stdin --style markdown --output /tmp/repomix-python.md
```

Include recent git context when reviewing project evolution:

```bash
npx --yes repomix@latest --include-logs --include-logs-count 10 --include-diffs --output /tmp/repomix-with-git.xml .
```

## Codex handoff pattern

1. Create the smallest useful bundle.
2. Launch Codex with the bundle path and explicit repo path.
3. Tell Codex the bundle is context, not the write target.
4. Still require Codex to inspect live files before editing.

Example:

```bash
BUNDLE=/tmp/repomix-hermes-auth.md
npx --yes repomix@latest --style markdown --compress --include "agent/**,tools/**,tests/**" --output "$BUNDLE" .

codex exec --cd "$PWD" --sandbox workspace-write "
You are working in this git repo: $PWD.
Use this Repomix context bundle for orientation only: $BUNDLE.
Before editing, inspect the live files you touch.
Implement the smallest safe fix, add/update tests, run targeted tests, and report exact output.
"
```

## Verification checklist

- [ ] Bundle was generated by a real command and saved to a known path.
- [ ] Output path is outside the repository unless intentionally committed.
- [ ] Include/ignore patterns exclude secrets and generated noise.
- [ ] For Codex work, live files were inspected before edits.
- [ ] Targeted tests ran after implementation.

## Pitfalls

- `repomix-output.xml` in the repo root is easy to commit accidentally. Prefer
  `/tmp/...` output paths.
- `--compress` is excellent for orientation, but not a substitute for reading
  the exact live file before editing.
- Remote packaging is convenient, but local clones are better when you need
  branch-specific dirty state or private repo context.
