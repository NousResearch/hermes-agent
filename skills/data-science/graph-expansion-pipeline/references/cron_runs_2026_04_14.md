## Edge Rebuild Strategy

When correcting edge types on a known set of targets, it is simpler and more reliable to **DELETE ALL existing edges between the target set, then recreate them with correct types**, than to attempt in-place upgrades. In-place upgrades using MATCH+SET fail when the target edge type doesn't exist yet (e.g., upgrading `colleague` to `coauthor` when the old edge was `colleague_at_parent_company`).

### Steps:
1. Delete all `Knows` edges between target pairs (both directions)
2. Compute correct edge types using the employer map and coauthor pairs
3. Create fresh bidirectional edges with correct `rel_type`, `context`, `confidence`, `source_ref`, and `record_time`
4. Verify total edge count matches expected math

This avoids:
- "Found duplicated primary key value" error from MERGE
- Cascading errors from delete-then-recreate when recreation fails
- Stale `colleague_at_parent_company` and other spurious edge types lingering

## Spurious Edge Types

`colleague_at_parent_company` is NOT a valid edge type. The valid set is: `colleague`, `former_colleague`, `coauthor`, `shared_field`, `friend`, `family`, `acquaintance`. Any other `rel_type` on `Knows` edges between expansion targets should be deleted during the cleanup phase before synthesis.

## Google OAuth Token Expiry in Cron Jobs

The OAuth refresh token can expire (`invalid_grant` error on refresh attempt). When this happens, `documents.create` returns 401 (UNAUTHENTICATED). Do NOT attempt to create Google Docs when the token is expired. Instead:
1. Save the report as a local markdown file at `/root/.hermes/commons/data/ocas-expansion/graph_expansion_report_YYYYMMDD.md`
2. Write the local path to `last_run_report.txt`
3. A user must re-authorize the Google OAuth token interactively before Doc creation will work again

## DBLP API Returning 0 Results

In cron/sandbox environments, DBLP's `author:{name}` search endpoint may return 0 publications for all targets, even targets with known publications. This appears to be an API or network issue in sandbox environments. When DBLP returns 0, use cached results from previous interactive runs (`dblp_results_*.json` in the expansion data directory).

## Execution Code Variable Scope

Variables defined in one `execute_code` block are NOT available in subsequent blocks (each runs in a separate Python process). Common mistakes:
- `run_id` defined in block 1 → `NameError` when used in block 2
- `prev_enriched` dict comprehension failing because `prev_enriched` was loaded as a list, not keyed by ID

Always re-define or re-load needed variables at the top of each `execute_code` block.
