# Codex Sidecar

Codex is a bounded sidecar capability for Hermes. It may be used as an
audit-readonly sidecar or, after explicit policy, as an optional Codex runtime
worker lane. Hermes remains the PM, architect, task classifier, supervisor,
verifier, local commit authority, and Operator-facing reporter.

Source of truth for this skill is `/Users/preston/.hermes/hermes-agent/skills/development/codex-sidecar/SKILL.md`.
The final branch-local audit prompt must be generated by
`/Users/preston/.hermes/hermes-agent/tools/render_crypto_bot_sidecar_audit_prompt.py`, and the
result must be verified by
`/Users/preston/.hermes/hermes-agent/tools/crypto_bot_completion_gate.py`.
Remote/PR/merge lifecycle evidence is governed by
`/Users/preston/.hermes/hermes-agent/docs/autonomy/crypto_bot_remote_lifecycle_contract.md`.
Codex sidecar evidence may support a PR evidence packet only after the local
completion gate has already passed.

Bundled Codex skills, GitHub examples, and generic PR workflows do not override
the crypto_bot Gitea policy. For crypto_bot, do not use `gh pr create`; PR
creation remains gated through the Hermes-owned Gitea adapter only after future
explicit policy enables it.

## Sidecar Rules

- Codex may inspect only safe local repo state needed for the audit prompt.
- Codex must not modify files, commit, push, deploy, start services, run
  workflows, start runners, mutate Gitea, inspect secrets, or touch
  broker/trading/runtime/financial surfaces.
- The final audit prompt must not prefill `clean`, `pass`, `blocked surfaces:
  none`, or any other successful conclusion. It may state the expected branch,
  base ref, and final HEAD/ref, but Codex must independently verify them.
- The final audit prompt must require `git status --short --branch`,
  `git rev-parse --abbrev-ref HEAD`, `git rev-parse HEAD`,
  `git diff --name-only <base>..<head>`, `git diff --check <base>..<head>`,
  command exit codes, changed files, branch, full HEAD, and blocked-surface
  path scan evidence.
- The final audit prompt must use the canonical full-SHA base/head range and
  stay bounded to the listed read-only commands. It must not ask Codex to run
  full verification, broad repository scans, services, workflows, runners, or
  test suites.
- A docs-only contract path may mention daemon/service concepts only when the
  prompt lists an explicit safe docs allowlist basis. Runtime code,
  service-start scripts/configs, workflow/runner, deploy, secret, trading,
  broker, and financial paths remain blocked.
- A failed, missing, timed-out, stale, or empty Codex result is a blocker for
  branch-local completion when sidecar audit is required.
- Hermes must not replace failed Codex evidence with Hermes self-audit, PM
  status, strategic-plan summaries, checkpoint docs, capability maps, or old
  diagnostic files.
- Hermes must independently verify successful Codex claims with read-only local
  evidence before reporting completion.
- Codex must not create remote branches, create PRs, update PR metadata, read
  secrets for Gitea authentication, run CI, start runners, or merge.
- Codex must not run `gh pr create` for crypto_bot.
- Codex may help audit proposed PR evidence packets in read-only mode, but PR
  creation and merge are separate future policy gates.
- For branch-local completion under `crypto_bot_target_loop_v2.md`, the final
  sidecar audit must run after the local commit on a clean worktree and must
  report the same branch and commit hash as the completion report.
- A dirty-worktree, pre-commit, smoke-test, stale, empty, timed-out, or
  different-HEAD result is not final completion evidence. A newer independent
  audit may help diagnose the state, but it does not retroactively make an
  earlier Hermes completion claim valid.
- A sidecar result is not final completion evidence unless the completion gate
  verifies that it names the correct branch, full HEAD, base/head range,
  `git diff --check` command, `git diff --check` exit code, and blocked-surface
  scan. If local gate validators fail, the sidecar result cannot override them.
- The sidecar final conclusion must parse as exactly `PASS`; `FAIL`,
  `BLOCKED`, missing, or ambiguous conclusions block completion.
- Machine fields must be parser-compatible, not merely semantically clear. In
  particular, `Blocked-surface scan` should be emitted as `PASS with basis: ...`
  or a standalone `PASS` with details elsewhere for maximum compatibility. If a
  semantically valid `PASS, ...` or similar machine field is rejected, use TDD to
  patch the completion-gate parser rather than rerunning broad audits. Do not
  treat a file as blocked merely because it is absent from a task allowlist; the
  allowlist is for otherwise sensitive surfaces, while ordinary docs, JSON
  evidence, discovery artifacts, and validation scripts need their own surface
  classification. See `references/blocked-surface-pass-format.md` for the S006
  parser pitfall, allowlist nuance, and regression-test pattern.

## Required Final Audit Shape

The result must include:

- `Schema: hermes.autonomy.crypto_bot_sidecar_audit.v1`
- commands run, working directories, and exit codes
- `Branch observed`
- `Full HEAD observed`
- `Base/head range audited`
- `Changed files`
- `Worktree status`
- `git diff --check exit code`
- `Blocked-surface scan`
- `Final conclusion`: exactly one of `PASS`, `FAIL`, or `BLOCKED`

Do not infer pass from clean worktree. Do not accept prefilled conclusions. If a
validator fails, report `FAIL`.

## Escalation

Escalate instead of using Codex when the requested work requires secrets, live
broker/trading/financial access, runtime service starts, deployment, protected
branch merges, workflow/runner execution, Gitea write expansion, policy changes,
new tool permissions, or `ruff format`.
