# 研究员读论文框架（工人必读）

目标不是「摘要翻译」，而是产出**可决策**的材料：能否信、能否用、下一步做什么。

## 30 秒：先回答四个问题

| 问题 | 产出位置 |
|------|----------|
| 一句话论点（Thesis） | T0 `thesis_one_liner` |
| 新在哪里（Novelty） | T1 主张表 |
| 证据是否够（Evidence） | T1 + T4 |
| 公式是否站得住（Math） | T2 方法节 + T5 核心公式 |
| 我该怎么用（So what） | T5 参考方向 |

## CEL 表（Claims–Evidence–Limits）— T1 核心产物

每个**可检验主张**一行，禁止泛泛形容词（「显著提升」要有数字或 ablation）。

| ID | 主张 Claim (ZH) | 证据 Evidence | 强度 | 局限 Limit |
|----|-----------------|---------------|------|------------|
| C1 | … | Fig.2 / Table.1 / §3.2 + 指标名 | 强/中/弱 | 数据集偏差、无开源等 |

**强度定义：**
- **强**：主文实验 + 多设定/消融支撑
- **中**：仅主表一行或间接对比
- **弱**：仅作者陈述、无对照、demo 级

## 阅读地图（T0）— 先扫再深读

```markdown
1. Abstract + Conclusion → 论点是否成立？
2. Figure 1 / 系统图 → 数据流与模块边界
3. Method § → 训练/推理/损失，记符号表与 1–3 条核心公式
4. Experiments § → 只读主表 + 最强 baseline
5. Appendix → 复现细节、超参、失败案例
```

记录 `pdf_sections_read: ["abstract","fig1","§3",...]` 到 handoff，避免 QA 时发现「只读了摘要」。

## 实验审计（T4 / kanban-coder）— 五问

1. **Baseline 公平吗？** 同算力/同数据量/同 prompt？
2. **指标对口吗？** 训练目标 vs 报告指标是否一致？
3. **方差报告了吗？** 单次跑分 → 标为弱证据
4. **消融支持因果吗？** 去掉模块 X 性能掉多少？
5. **外部可复现吗？** 代码/数据/权重链接是否有效？

## 方法精读（T2）— 研究员增强项

1. **公式最小集**：目标函数 / 奖励 / 损失 / 推理决策式，至少抽 1 条，最多 3 条
2. **符号释义**：每条公式要解释变量、优化方向、近似假设
3. **运行时链路**：输入表征 → 候选生成 → 搜索/解码 → 执行动作
4. **部署坑点**：列出最影响 sim-to-real / latency / reliability 的工程细节
5. **Delta 视角**：论文设定 vs 我们当前软硬件，写成迁移差异表

## 写作原则（T5 飞书文档）

- **中文先行**：决策写在中文 bullet；English 是 mirror，不是主叙事
- **数字要带单位与设定**（「51.7%」→「MATH pass@1, 7B, no tool」）
- **区分「论文说」与「我们判断」**：用「作者声称 / 我们认为」
- **参考方向必须可执行**：复现 / 产品 / 对标 三条，每条 ≤ 2 行
- **公式必须可解释**：不要只贴公式；必须补“它控制了什么行为”
- **边界分析必须现代化**：若今天已有更强范式（如端侧 VLM / VLA），要点出可替换环节

## 常见误读（禁止）

- 把 Related Work 当创新点
- 只复述 Abstract 不写 CEL
- 无证据强度的形容词堆砌
- 把附录实验当主结论
- 多篇论文写入同一飞书 doc
