# 系统调试

## 概述

随机修复浪费时间并创建新的错误。快速补丁掩盖了潜在问题。

**核心原则：** 总是在尝试修复之前找到根本原因。症状修复是失败的。

**违反此过程的字面意思就是违反调试的精神。**

## 铁律

```
没有根因调查，就没有修复
```

如果你还没有完成第1阶段，你不能提出修复方案。

## 何时使用

用于任何技术问题：
- 测试失败
- 生产环境中的错误
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别在以下情况使用：**
- 时间紧迫（紧急情况使猜测变得诱人）
- "只需一个快速修复"似乎很明显
- 你已经尝试了多种修复
- 之前的修复不起作用
- 你不完全理解问题

**不要跳过的情况：**
- 问题看起来简单（简单的错误也有根因）
- 你很匆忙（匆忙保证需要返工）
- 有人希望立即修复（系统方法比混乱更快）

## 四个阶段

你必须完成每个阶段才能进入下一个阶段。

---

## 阶段 1：根因调查

**在尝试任何修复之前：**

### 1. 仔细阅读错误信息

- 不要跳过错误或警告
- 它们通常包含确切的解决方案
- 完整阅读堆栈跟踪
- 记录行号、文件路径、错误代码

**行动：** 对相关源文件使用 `read_file`。使用 `search_files` 在代码库中查找错误字符串。

### 2. 一致地重现

- 你能可靠地触发它吗？
- 确切的步骤是什么？
- 它每次都会发生吗？
- 如果无法重现 → 收集更多数据，不要猜测

**行动：** 使用 `terminal` 工具运行失败的测试或触发错误：

```bash
# 运行特定的失败测试
pytest tests/test_module.py::test_name -v

# 运行详细输出
pytest tests/test_module.py -v --tb=long
```

### 3. 检查最近的更改

- 什么变化可能导致这个问题？
- Git diff，最近的提交
- 新的依赖项，配置更改

**行动：**

```bash
# 最近的提交
git log --oneline -10

# 未提交的更改
git diff

# 特定文件的更改
git log -p --follow src/problematic_file.py | head -100
```

### 4. 在多组件系统中收集证据

**当系统有多个组件时（API → 服务 → 数据库，CI → 构建 → 部署）：**

**在提出修复之前，添加诊断工具：**

对于每个组件边界：
- 记录进入组件的数据
- 记录离开组件的数据
- 验证环境/配置传播
- 检查每层的状态

运行一次以收集证据，显示它在哪里中断。
然后分析证据以识别失败的组件。
然后调查该特定组件。

### 5. 追踪数据流

**当错误在调用栈深处时：**

- 错误值起源于哪里？
- 什么用错误值调用了这个函数？
- 继续向上游追踪直到找到源头
- 在源头修复，而不是在症状处

**行动：** 使用 `search_files` 追踪引用：

```python
# 找到函数被调用的地方
search_files("function_name(", path="src/", file_glob="*.py")

# 找到变量被设置的地方
search_files("variable_name\s*=", path="src/", file_glob="*.py")
```

### 阶段 1 完成清单

- [ ] 错误信息已完全阅读和理解
- [ ] 问题已一致重现
- [ ] 最近的更改已识别和审查
- [ ] 证据已收集（日志、状态、数据流）
- [ ] 问题已隔离到特定组件/代码
- [ ] 形成了根因假设

**停止：** 在理解它为什么发生之前，不要进入第 2 阶段。

---

## 阶段 2：模式分析

**在修复之前找到模式：**

### 1. 找到工作示例

- 在同一代码库中定位类似的工作代码
- 什么是有效的，类似于什么是坏的？

**行动：** 使用 `search_files` 找到可比较的模式：

```python
search_files("similar_pattern", path="src/", file_glob="*.py")
```

### 2. 与参考进行比较

- 如果实现模式，请完全阅读参考实现
- 不要浏览 — 阅读每一行
- 在应用之前完全理解模式

### 3. 识别差异

- 工作和损坏之间有什么不同？
- 列出每一个差异，无论多么小
- 不要假设"那不重要"

### 4. 理解依赖关系

- 这需要其他哪些组件？
- 什么设置、配置、环境？
- 它做了什么假设？

---

## 阶段 3：假设和测试

**科学方法：**

### 1. 形成单一假设

- 明确说明："我认为 X 是根因，因为 Y"
- 写下来
- 具体，不要模糊

### 2. 最小化测试

- 做出最小的可能改变来测试假设
- 一次一个变量
- 不要一次修复多个问题

### 3. 继续之前验证

- 它有效吗？ → 阶段 4
- 无效？ → 形成新假设
- 不要在上面添加更多修复

### 4. 当你不知道时

- 说"我不理解 X"
- 不要假装知道
- 向用户寻求帮助
- 研究更多

---

## 阶段 4：实现

**修复根因，而不是症状：**

### 1. 创建失败测试用例

- 最简单的可能重现
- 尽可能自动化测试
- 修复前必须有
- 使用 `test-driven-development` 技能

### 2. 实现单一修复

- 解决已识别的根因
- 一次一个更改
- 没有"顺便"的改进
- 没有捆绑的重构

### 3. 验证修复

```bash
# 运行特定的回归测试
pytest tests/test_module.py::test_regression -v

# 运行完整套件 — 无回归
pytest tests/ -q
```

### 4. 如果修复不起作用 — 三法则

- **停止。**
- 计数：你尝试了多少修复？
- 如果 < 3：返回阶段 1，用新信息重新分析
- **如果 ≥ 3：停止并质疑架构（下面的步骤 5）**
- 没有架构讨论，不要尝试修复 #4

### 5. 如果 3+ 修复失败：质疑架构

**表明架构问题的模式：**
- 每个修复都在不同的地方揭示新的共享状态/耦合
- 修复需要"大规模重构"来实现
- 每个修复在其他地方创建新的症状

**停止并质疑基本原理：**
- 这个模式从根本上是合理的吗？
- 我们是否"仅仅因为惯性而坚持它"？
- 我们应该重构架构还是继续修复症状？

**在尝试更多修复之前与用户讨论。**

这不是一个失败的假设 — 这是一个错误的架构。

---

## 红旗 — 停止并遵循流程

如果你发现自己在想：
- "现在快速修复，以后调查"
- "只需尝试更改 X 看看是否有效"
- "添加多个更改，运行测试"
- "跳过测试，我会手动验证"
- "可能是 X，让我修复它"
- "我不完全理解，但这可能有效"
- "模式说 X 但我会以不同方式适应它"
- "以下是主要问题：[列出没有调查的修复]"
- 在追踪数据流之前提出解决方案
- **"再一次修复尝试"（已经尝试了 2+）**
- **每个修复都在不同的地方揭示新问题**

**所有这些都意味着：停止。返回阶段 1。**

**如果 3+ 修复失败：** 质疑架构（阶段 4 步骤 5）。

## 常见的合理化

| 借口 | 现实 |
|--------|---------|
| "问题简单，不需要流程" | 简单问题也有根因。流程对简单错误很快。 |
| "紧急情况，没有时间流程" | 系统调试比猜测和检查的混乱更快。 |
| "先试试这个，然后调查" | 第一次修复设置模式。从一开始就做对。 |
| "我会在确认修复有效后编写测试" | 未经测试的修复不会持久。测试首先证明它。 |
| "一次多个修复节省时间" | 无法隔离什么有效。导致新错误。 |
| "参考太长，我会适应模式" | 部分理解保证错误。完全阅读它。 |
| "我看到了问题，让我修复它" | 看到症状 ≠ 理解根因。 |
| "再一次修复尝试"（2+ 失败后） | 3+ 失败 = 架构问题。质疑模式，不要再修复。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. 根因** | 阅读错误，重现，检查更改，收集证据，追踪数据流 | 理解什么和为什么 |
| **2. 模式** | 找到工作示例，比较，识别差异 | 知道什么不同 |
| **3. 假设** | 形成理论，最小化测试，一次一个变量 | 确认或新假设 |
| **4. 实现** | 创建回归测试，修复根因，验证 | 错误已解决，所有测试通过 |

## Hermes Agent 集成

### 调查工具

在阶段 1 中使用这些 Hermes 工具：

- **`search_files`** — 查找错误字符串，追踪函数调用，定位模式
- **`read_file`** — 读取带有行号的源代码以进行精确分析
- **`terminal`** — 运行测试，检查 git 历史，重现错误
- **`web_search`/`web_extract`** — 研究错误消息，库文档

### 与 delegate_task 一起使用

对于复杂的多组件调试，分派调查子代理：

```python
delegate_task(
    goal="调查为什么 [特定测试/行为] 失败",
    context="""
    遵循系统调试技能：
    1. 仔细阅读错误消息
    2. 重现问题
    3. 追踪数据流以找到根因
    4. 报告发现 — 尚未修复

    错误：[粘贴完整错误]
    文件：[失败代码的路径]
    测试命令：[确切命令]
    """,
    toolsets=['terminal', 'file']
)
```

### 与 test-driven-development 一起使用

修复错误时：
1. 编写重现错误的测试（红）
2. 系统调试以找到根因
3. 修复根因（绿）
4. 测试证明修复并防止回归

## 现实世界影响

来自调试会话：
- 系统方法：15-30 分钟修复
- 随机修复方法：2-3 小时混乱
- 首次修复率：95% 对 40%
- 引入的新错误：几乎为零对常见

**没有捷径。没有猜测。系统总是赢。**
