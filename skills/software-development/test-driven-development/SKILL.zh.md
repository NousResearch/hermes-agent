# 测试驱动开发 (TDD)

## 概述

先编写测试。看着它失败。编写最少的代码使其通过。

**核心原则：** 如果你没有看到测试失败，你就不知道它是否测试了正确的东西。

**违反规则的字面意思就是违反规则的精神。**

## 何时使用

**总是：**
- 新功能
- 错误修复
- 重构
- 行为变更

**例外情况（请先询问用户）：**
- 一次性原型
- 生成的代码
- 配置文件

想“就这一次跳过 TDD”？停下来。那是合理化。

## 铁律

```
没有失败的测试，就没有生产代码
```

在测试之前写代码？删除它。重新开始。

**没有例外：**
- 不要将其保留为“参考”
- 不要在编写测试时“改编”它
- 不要看它
- 删除意味着删除

根据测试重新实现。就这样。

## 红-绿-重构循环

### 红 — 编写失败的测试

编写一个最小的测试，展示应该发生什么。

**好的测试：**
```python
def test_retries_failed_operations_3_times():
    attempts = 0
    def operation():
        nonlocal attempts
        attempts += 1
        if attempts < 3:
            raise Exception('fail')
        return 'success'

    result = retry_operation(operation)

    assert result == 'success'
    assert attempts == 3
```
清晰的名称，测试真实行为，只测试一件事。

**坏的测试：**
```python
def test_retry_works():
    mock = MagicMock()
    mock.side_effect = [Exception(), Exception(), 'success']
    result = retry_operation(mock)
    assert result == 'success'  # 重试次数呢？时间呢？
```
模糊的名称，测试模拟而不是真实代码。

**要求：**
- 每个测试一个行为
- 清晰的描述性名称（名称中有“和”？拆分它）
- 真实代码，不是模拟（除非真正不可避免）
- 名称描述行为，而不是实现

### 验证红 — 看着它失败

**必须。永远不要跳过。**

```bash
# 使用终端工具运行特定测试
pytest tests/test_feature.py::test_specific_behavior -v
```

确认：
- 测试失败（不是拼写错误导致的错误）
- 失败消息符合预期
- 失败是因为功能缺失

**测试立即通过？** 你在测试现有行为。修复测试。

**测试出错？** 修复错误，重新运行直到正确失败。

### 绿 — 最小代码

编写最简单的代码来通过测试。仅此而已。

**好的：**
```python
def add(a, b):
    return a + b  # 没有多余的东西
```

**坏的：**
```python
def add(a, b):
    result = a + b
    logging.info(f"Adding {a} + {b} = {result}")  # 多余！
    return result
```

不要添加功能、重构其他代码或超出测试范围的“改进”。

**在绿阶段作弊是可以的：**
- 硬编码返回值
- 复制粘贴
- 重复代码
- 跳过边缘情况

我们会在重构阶段修复它。

### 验证绿 — 看着它通过

**必须。**

```bash
# 运行特定测试
pytest tests/test_feature.py::test_specific_behavior -v

# 然后运行所有测试以检查是否有回归
pytest tests/ -q
```

确认：
- 测试通过
- 其他测试仍然通过
- 输出干净（无错误、警告）

**测试失败？** 修复代码，不是测试。

**其他测试失败？** 立即修复回归。

### 重构 — 清理

仅在绿阶段之后：
- 移除重复
- 改进名称
- 提取助手
- 简化表达式

在整个过程中保持测试为绿色。不要添加行为。

**如果重构期间测试失败：** 立即撤销。采取更小的步骤。

### 重复

下一个失败的测试对应下一个行为。一次一个循环。

## 为什么顺序很重要

**“我会在之后写测试来验证它是否工作”**

在代码之后编写的测试会立即通过。立即通过证明不了什么：
- 可能测试了错误的东西
- 可能测试了实现，而不是行为
- 可能遗漏了你忘记的边缘情况
- 你从未见过它捕获错误

测试优先迫使你看到测试失败，证明它实际上测试了一些东西。

**“我已经手动测试了所有边缘情况”**

手动测试是临时的。你认为你测试了一切，但是：
- 没有你测试内容的记录
- 代码更改时无法重新运行
- 在压力下容易忘记情况
- “我尝试时它工作了”≠ 全面

自动化测试是系统性的。它们每次都以相同的方式运行。

**“删除 X 小时的工作是浪费的”**

沉没成本谬误。时间已经过去了。你现在的选择：
- 删除并使用 TDD 重写（高信心）
- 保留它并在之后添加测试（低信心，可能有错误）

“浪费”是保留你不能信任的代码。

**“TDD 是教条的，务实意味着适应”**

TDD 是务实的：
- 在提交前发现错误（比之后调试更快）
- 防止回归（测试立即捕获中断）
- 记录行为（测试显示如何使用代码）
- 启用重构（自由更改，测试捕获中断）

“务实”的捷径 = 在生产中调试 = 更慢。

**“测试之后达到相同的目标 — 这是精神而不是仪式”**

不。测试之后回答“这是做什么的？”测试之前回答“这应该做什么？”

测试之后受到你的实现的偏见。你测试你构建的东西，而不是需要的东西。测试之前迫使你在实现之前发现边缘情况。

## 常见的合理化

| 借口 | 现实 |
|--------|---------|
| "太简单了，不需要测试" | 简单的代码会中断。测试需要 30 秒。 |
| "我会之后测试" | 测试立即通过证明不了什么。 |
| "测试之后达到相同的目标" | 测试之后 = "这是做什么的？" 测试之前 = "这应该做什么？" |
| "已经手动测试过了" | 临时 ≠ 系统。没有记录，无法重新运行。 |
| "删除 X 小时是浪费的" | 沉没成本谬误。保留未验证的代码是技术债务。 |
| "保留为参考，先写测试" | 你会改编它。那是测试之后。删除意味着删除。 |
| "需要先探索" | 可以。扔掉探索，从 TDD 开始。 |
| "测试困难 = 设计不清晰" | 听测试的。难以测试 = 难以使用。 |
| "TDD 会减慢我" | TDD 比调试更快。务实 = 测试优先。 |
| "手动测试更快" | 手动不能证明边缘情况。你会在每次更改时重新测试。 |
| "现有代码没有测试" | 你正在改进它。为你触及的代码添加测试。 |

## 红旗 — 停止并重新开始

如果你发现自己做了以下任何一件事，删除代码并使用 TDD 重新开始：

- 测试之前的代码
- 实现之后的测试
- 测试在第一次运行时立即通过
- 无法解释为什么测试失败
- "后来"添加的测试
- 合理化"就这一次"
- "我已经手动测试过了"
- "测试之后达到相同的目的"
- "保留为参考"或"改编现有代码"
- "已经花了 X 小时，删除是浪费的"
- "TDD 是教条的，我是务实的"
- "这是不同的，因为..."

**所有这些都意味着：删除代码。使用 TDD 重新开始。**

## 验证清单

在标记工作完成之前：

- [ ] 每个新函数/方法都有测试
- [ ] 在实现之前观看每个测试失败
- [ ] 每个测试都因为预期的原因失败（功能缺失，不是拼写错误）
- [ ] 为通过每个测试编写了最小的代码
- [ ] 所有测试都通过
- [ ] 输出干净（无错误、警告）
- [ ] 测试使用真实代码（只有在不可避免时才使用模拟）
- [ ] 覆盖了边缘情况和错误

不能勾选所有框？你跳过了 TDD。重新开始。

## 卡住时

| 问题 | 解决方案 |
|---------|----------|
| 不知道如何测试 | 编写期望的 API。先写断言。询问用户。 |
| 测试太复杂 | 设计太复杂。简化接口。 |
| 必须模拟一切 | 代码耦合度过高。使用依赖注入。 |
| 测试设置巨大 | 提取助手。仍然复杂？简化设计。 |

## Hermes Agent 集成

### 运行测试

在每个步骤使用 `terminal` 工具运行测试：

```python
# 红 — 验证失败
terminal("pytest tests/test_feature.py::test_name -v")

# 绿 — 验证通过
terminal("pytest tests/test_feature.py::test_name -v")

# 完整套件 — 验证无回归
terminal("pytest tests/ -q")
```

### 与 delegate_task 一起使用

当分派子代理进行实现时，在目标中强制执行 TDD：

```python
delegate_task(
    goal="使用严格的 TDD 实现 [功能]",
    context="""
    遵循测试驱动开发技能：
    1. 首先编写失败的测试
    2. 运行测试以验证它失败
    3. 编写最小的代码以通过
    4. 运行测试以验证它通过
    5. 必要时重构
    6. 提交

    项目测试命令：pytest tests/ -q
    项目结构：[描述相关文件]
    """,
    toolsets=['terminal', 'file']
)
```

### 与 systematic-debugging 一起使用

发现错误？编写重现它的失败测试。遵循 TDD 循环。测试证明修复并防止回归。

永远不要在没有测试的情况下修复错误。

## 测试反模式

- **测试模拟行为而不是真实行为** — 模拟应该验证交互，而不是替换被测系统
- **测试实现细节** — 测试行为/结果，而不是内部方法调用
- **只测试 happy path** — 总是测试边缘情况、错误和边界
- **脆弱的测试** — 测试应该验证行为，而不是结构；重构不应该破坏它们

## 最终规则

```
生产代码 → 测试存在且首先失败
否则 → 不是 TDD
```

未经用户明确许可，没有例外。
