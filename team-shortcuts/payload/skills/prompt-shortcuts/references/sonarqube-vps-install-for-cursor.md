# Prompt ติดตั้ง SonarQube บน VPS สำหรับ Cursor

คัดลอกข้อความตั้งแต่หัวข้อ `เริ่ม Prompt` ถึง `จบ Prompt` ไปให้ Cursor ทำงานบนเครื่องที่เชื่อมต่อ VPS ได้

## เริ่ม Prompt

คุณเป็นวิศวกรระบบที่ได้รับมอบหมายให้ติดตั้ง SonarQube Community Build บน VPS ที่มีระบบอื่นใช้งานอยู่แล้ว งานนี้เป็นการติดตั้งครั้งเดียวเพื่อรองรับประมาณ 30 โปรเจกต์ โดยเจ้าของใช้คนเดียว แบ่งเป็นประมาณ 20 โปรเจกต์บน VPS และ 10 โปรเจกต์บนเครื่องส่วนตัว

เป้าหมายคือได้ SonarQube Community Build รุ่นฟรี ใช้ PostgreSQL เก็บข้อมูล รันด้วย Docker Compose และไม่เปิดพอร์ต 9000 สู่อินเทอร์เน็ต ผู้ใช้เข้าหน้าเว็บและส่งผลสแกนผ่าน SSH tunnel เท่านั้น

### กฎความปลอดภัย

1. เริ่มด้วยการตรวจแบบอ่านอย่างเดียว ห้ามติดตั้ง ห้ามหยุดบริการ ห้ามแก้ firewall และห้ามสร้างไฟล์ก่อนแสดงรายงานก่อนติดตั้ง
2. ตรวจ `AGENTS.md`, runbook, กฎ VPS และทะเบียนโปรเจกต์ที่มีอยู่ก่อนเสมอ
3. ถ้าใช้ `/srv/projects/` ให้ตรวจชื่อใกล้เคียงก่อน ห้ามสร้างโฟลเดอร์ซ้ำ หนึ่งระบบต้องมีโฟลเดอร์เดียว แนะนำ `/srv/projects/sonarqube` เมื่อไม่ชนของเดิม
4. ห้ามอ่านหรือแสดงค่าใน `.env` ห้ามพิมพ์รหัสผ่าน token หรือ secret ลงรายงาน แชต log หรือประวัติ shell
5. ห้ามใช้ฐานข้อมูล H2 สำหรับระบบจริง ต้องใช้ PostgreSQL
6. ห้ามใช้ image tag แบบลอย เช่น `latest` ให้ตรวจรุ่น Community Build ปัจจุบันจากเอกสารและ registry ทางการ แล้วตรึงรุ่นที่แน่นอน พร้อมบันทึกลิงก์แหล่งที่มา
7. ห้ามเปิด `0.0.0.0:9000` หรือพอร์ต 9000 ต่อสาธารณะ ให้ bind เฉพาะ `127.0.0.1`
8. ห้ามเปลี่ยน reverse proxy, Docker network, PostgreSQL, firewall หรือบริการเดิมถ้ายังไม่ทราบผลกระทบ
9. ถ้าทรัพยากรไม่พอ พอร์ตชน ดิสก์เสี่ยงเต็ม หรือมี SonarQube/PostgreSQL เดิม ให้หยุดและรายงาน ห้ามแก้ด้วยการลบของเดิม
10. หลังรายงานก่อนติดตั้ง ให้ขออนุมัติเจ้าของหนึ่งครั้งก่อนเริ่มเขียนหรือเปลี่ยนระบบ

### เฟส 1 — ตรวจ VPS แบบอ่านอย่างเดียว

ตรวจและรายงานค่าจริง โดยไม่เดา:

- ระบบปฏิบัติการ รุ่น kernel และสถาปัตยกรรม CPU
- จำนวน vCPU, RAM รวม/ว่าง, swap และพื้นที่ดิสก์รวม/ว่าง
- ภาระ CPU และ RAM ของบริการเดิม
- Docker และ Docker Compose พร้อมรุ่น
- container, network, volume และพอร์ตที่ใช้อยู่ โดยไม่แสดงค่า secret
- พอร์ต `9000`, `5432`, `80`, `443` ว่าถูกใช้หรือไม่
- firewall ที่ใช้งานและกฎที่เกี่ยวข้อง
- โฟลเดอร์โปรเจกต์เดิมและชื่อที่ใกล้ `sonar` หรือ `sonarqube`
- PostgreSQL เดิมว่ามีหรือไม่ แต่ห้ามต่อฐานข้อมูลหรือเปลี่ยนข้อมูลโดยไม่จำเป็น
- วิธีสำรองข้อมูลและพื้นที่ปลายทางที่มีอยู่
- ตรวจว่าเครื่องรองรับข้อกำหนด Linux ของ Elasticsearch ที่ SonarQube ใช้ เช่น `vm.max_map_count` และข้อจำกัดเปิดไฟล์

สรุปผลเป็นตาราง `ผ่าน / ไม่ผ่าน / ต้องแก้ก่อน` พร้อมหลักฐานคำสั่งและค่าที่ตรวจพบ ปิดท้ายด้วยคำตัดสิน `INSTALL_APPROVAL_REQUIRED` และรายการสิ่งที่จะสร้างหรือเปลี่ยน

### เฟส 2 — ติดตั้งหลังได้รับอนุมัติ

เมื่อเจ้าของอนุมัติแล้ว ให้ทำดังนี้:

1. สร้างโฟลเดอร์ระบบเพียงแห่งเดียวตามกฎ VPS
2. สร้าง `compose.yaml` สำหรับ:
   - SonarQube Community Build รุ่นที่ตรึงแล้ว
   - PostgreSQL รุ่นรองรับที่ตรึงแล้ว
   - volume ถาวรสำหรับข้อมูล ส่วนขยาย log และฐานข้อมูล
   - healthcheck สำหรับ SonarQube และ PostgreSQL
   - restart policy ที่ทำให้กลับมาหลัง VPS เริ่มใหม่
   - resource limit ที่ไม่แย่งเครื่องจนบริการเดิมเสียหาย
   - SonarQube bind เฉพาะ `127.0.0.1:9000`
3. เก็บรหัสฐานข้อมูลใน `.env` สิทธิ์ไฟล์ `600` และสร้างรหัสแบบสุ่ม ห้ามนำค่าไปแสดง
4. ตั้ง kernel และ file limit เฉพาะค่าที่เอกสาร SonarSource กำหนดจริง พร้อมเก็บไฟล์การตั้งค่าให้ตรวจย้อนหลังได้
5. เริ่ม PostgreSQL ก่อน รอ healthy แล้วจึงเริ่ม SonarQube
6. ห้ามนำ token วิเคราะห์โค้ดใส่ใน compose หรือ repository
7. แสดงให้เจ้าของเปลี่ยนรหัสผ่านผู้ดูแลเริ่มต้นด้วยตนเอง ห้ามบันทึกรหัสผ่านใหม่

### เฟส 3 — ตรวจระบบหลังติดตั้ง

ต้องตรวจตามลำดับและแสดงผลที่ตัดข้อมูลลับออกแล้ว:

1. `docker compose config` ต้องอ่านโครงสร้างได้โดยไม่แสดง secret
2. container ทั้งสองต้องเป็น healthy หรือสถานะที่เอกสารทางการระบุว่าใช้งานได้
3. `curl http://127.0.0.1:9000/api/system/status` ต้องตอบ HTTP 200 และสถานะ `UP`
4. ตรวจจาก compose และ log ว่าใช้ PostgreSQL ไม่ใช่ H2
5. ตรวจว่า `ss` หรือเครื่องมือเทียบเท่าเห็นพอร์ต 9000 เฉพาะ `127.0.0.1`
6. ตรวจจากภายนอกว่า `VPS_PUBLIC_IP:9000` เข้าไม่ได้ โดยไม่เปลี่ยน firewall เพื่อให้ผ่าน
7. ตรวจ volume และสิทธิ์ไฟล์
8. ตรวจ log ว่าไม่มี error ระดับร้ายแรงซ้ำ
9. restart ชุดบริการหนึ่งครั้ง แล้วตรวจข้อ 2–5 ซ้ำ
10. ถ้าปลอดภัยกับบริการเดิม ให้ตรวจการกลับมาหลัง restart Docker; ห้าม reboot VPS โดยไม่ได้รับอนุมัติแยกต่างหาก

ห้ามใช้คำว่าเสร็จหรือใช้งานได้ หากยังไม่มีผล HTTP 200, สถานะ `UP`, container status และหลักฐานว่าพอร์ตไม่เปิดสาธารณะ

### เฟส 4 — การสำรองและกู้คืน

สร้างสคริปต์สำรองที่:

- ใช้ `pg_dump` สำรอง PostgreSQL
- สำรองไฟล์ compose และรายการรุ่น image โดยไม่รวม secret
- ตั้งชื่อไฟล์ด้วยเวลาและลบไฟล์เก่าตามจำนวนที่กำหนดใน config
- เขียน log ที่ไม่เปิดเผยรหัสผ่าน
- คืน exit code ที่ชัดเจนเมื่อสำเร็จหรือผิดพลาด

ทดสอบว่าสร้าง dump ได้และตรวจไฟล์ dump ด้วยเครื่องมือ PostgreSQL หากจะทดสอบกู้คืนจริง ต้องใช้ฐานข้อมูลชั่วคราวแยกจากระบบหลักและขออนุมัติก่อน ห้ามกู้ทับฐานข้อมูลจริง

### เฟส 5 — วิธีเข้าจากเครื่องส่วนตัว

ให้คำสั่งตัวอย่างโดยไม่ฝัง IP ชื่อผู้ใช้ หรือ secret จริง:

```bash
ssh -N -L 9000:127.0.0.1:9000 <VPS_USER>@<VPS_HOST>
```

จากนั้นผู้ใช้เปิด:

```text
http://127.0.0.1:9000
```

สำหรับสแกนโปรเจกต์จากเครื่องส่วนตัว ให้ใช้ tunnel เดียวกัน แล้วตั้ง:

```bash
export SONAR_HOST_URL="http://127.0.0.1:9000"
export SONAR_TOKEN="<TOKEN_FROM_SONARQUBE>"
```

ห้ามเขียน token ลง `sonar-project.properties`, repository หรือไฟล์ที่ Git ติดตาม

### เฟส 6 — ทดลองกับหนึ่งโปรเจกต์

1. ให้เจ้าของเลือกโปรเจกต์ทดลองหนึ่งโปรเจกต์ หรือใช้โปรเจกต์ที่ระบุพร้อมคำสั่งนี้
2. ตรวจภาษาและระบบ build จากไฟล์จริง
3. สร้าง project key ที่ไม่ซ้ำและไม่เปิดเผยข้อมูลลูกค้า
4. ใช้ SonarScanner ที่ตรงกับระบบ build
5. ไม่รวม `.git`, dependency, build output, cache, secret และไฟล์ขนาดใหญ่ที่ไม่ใช่ source
6. รันสแกนและเก็บ exit code
7. เรียก API ของ SonarQube เพื่อตรวจว่าโปรเจกต์ปรากฏและมีผลวิเคราะห์ล่าสุด
8. สรุป Quality Gate, reliability, security, maintainability, duplication และ coverage เป็นภาษาไทย

### รายงานส่งมอบ

ส่งรายงานสุดท้ายประกอบด้วย:

- รุ่น SonarQube, PostgreSQL, Docker และ Compose ที่ติดตั้ง
- ตำแหน่งไฟล์ระบบ โดยไม่เปิดเผย secret
- สถานะการตรวจแต่ละข้อเป็น N/M
- ผล HTTP และสถานะระบบ
- หลักฐานว่าพอร์ต 9000 ไม่เปิดสาธารณะ
- คำสั่ง start, stop, restart, status และดู log
- คำสั่งเปิด SSH tunnel
- วิธีสร้าง project token
- วิธีเพิ่มโปรเจกต์บน VPS และเครื่องส่วนตัว
- วิธีสำรองข้อมูลและตำแหน่งไฟล์สำรอง
- วิธีอัปเกรดแบบสำรองก่อน และห้ามกระโดดข้ามรุ่นที่ผู้ผลิตไม่รองรับ
- ความเสี่ยงที่ยังเหลือ
- ขั้นตอนถัดไปหนึ่งข้อที่แนะนำ
