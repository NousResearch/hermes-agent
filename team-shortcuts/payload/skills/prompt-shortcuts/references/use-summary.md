# Use Summary

## Shortcut

```text
Use Summary
```

## Prompt

```text
Use Summary กับ link + content ที่ผู้ใช้ส่งมา

หน้าที่: สรุป + วิเคราะห์ + เสนอทางเลือก ก่อนบันทึกลงไฟล์/memory/registry รอเจ้าของอนุมัติ

[เลือกโหมดก่อนเสมอ — โหมดเป็นตัวกำหนดว่าตอบกี่ส่วน]
บรรทัดแรกของคำตอบต้องประกาศเสมอ:
"โหมดรอบนี้: <สั้น/ปกติ/ลึก/ติดตั้ง-อัปเกรด> — เหตุผล: <เงื่อนไขที่เข้าเกณฑ์>"

ใช้เกณฑ์ที่วัดได้ตามตารางนี้ ห้ามเลือกจากความรู้สึกล้วน:
| โหมด | ใช้เมื่อ | ต้องตอบส่วน |
|---|---|---|
| สั้น | เนื้อหา <30 บรรทัด และไม่มีคำ install / upgrade / plugin / tool / software / ติดตั้ง / อัปเกรด / ปลั๊กอิน / เครื่องมือ / ซอฟต์แวร์ และไม่กระทบหลายระบบ | 1, 2, 8 |
| ปกติ | ไม่เข้าเงื่อนไขสั้น/ลึก/ติดตั้ง หรือไม่แน่ใจว่าเข้าโหมดไหน | 1, 2, 3, 8 |
| ลึก | กระทบตั้งแต่ 2 ระบบ/โปรเจกต์/ทีมขึ้นไป หรือแตะ security / auth / payment / privacy / infrastructure / policy หรือเนื้อหา >120 บรรทัด | ครบ 8 ส่วน + ตรวจแหล่งภายนอกเพิ่มเมื่อทำได้ |
| ติดตั้ง-อัปเกรด | มีคำ install / upgrade / plugin / tool / software / package / dependency / ติดตั้ง / อัปเกรด / ปลั๊กอิน / เครื่องมือ / ซอฟต์แวร์ / แพ็กเกจ / ไลบรารี | 1, 2, 3, 5, 8 |

ถ้าเข้าได้หลายโหมดพร้อมกัน ให้ใช้โหมดที่เสี่ยงสูงกว่า: ลึก > ติดตั้ง-อัปเกรด > สั้น > ปกติ
ถ้าเป็นงานติดตั้งและกระทบตั้งแต่ 2 ระบบขึ้นไป ให้ใช้โหมดลึก และส่วน 5 ต้องมีรายละเอียดครบ

[2 เฟสแยกชัด]
เฟส 1 ก่อนอนุมัติ = review only ห้ามเขียนไฟล์/memory/registry/ติดตั้งใด ๆ
เฟส 2 หลังอนุมัติ = เขียน/ติดตั้ง/sync/ประเมินผล
ข้ามไปเฟส 2 ได้เฉพาะเมื่อผู้ใช้สั่งชัดทั้ง "ให้เลือก" และ "ให้บันทึก/ติดตั้ง"

คำที่นับเป็นอนุมัติเข้าเฟส 2:
- "บันทึกเลย"
- "จดเลย"
- "เก็บเลย"
- "ติดตั้งเลย"
- "ตามข้อ 8"
- "จัดการเลย"
- "ทำให้จบ"
- "ทำตามที่แนะนำ"
- "เลือกให้แล้วทำเลย"
- "อนุมัติให้บันทึก"
- "อนุมัติให้ติดตั้ง"

"OK" เฉย ๆ = เห็นด้วยกับข้อเสนอล่าสุดเท่านั้น ยังไม่ใช่คำสั่งให้เขียนไฟล์/memory/registry/ติดตั้ง
ถ้าคำตอบผู้ใช้ก้ำกึ่ง ให้ถามยืนยัน 1 ประโยคก่อนเข้าเฟส 2 เช่น:
"ยืนยันให้บันทึก/ติดตั้งตามข้อ 8 เลยไหม?"

กฎหลัก:
- ใช้ภาษาผู้ใช้ + แปลศัพท์เทคนิคทันที
- เปิดลิงก์ไม่ได้หรือเนื้อหาไม่พอ → ค่อยขอเนื้อหาเพิ่ม · มี content แล้ว → ใช้ content เป็นหลัก
- ถ้าเปิดเว็บ/ตรวจแหล่งภายนอกไม่ได้ ให้ติดป้าย "ยังไม่ได้ตรวจภายนอก" ห้ามเดา/เติมผลแทนเด็ดขาด
- เจ้าของอยู่ในแชท = review เลย ห้ามเลื่อนเป็น "คิวงานหลังอนุมัติ"
  (คิวงานหลังอนุมัติใช้ได้เฉพาะงานหลังรีวิวแล้ว เช่น sandbox test, promote)
- ถ้ามีหลายลิงก์ในข้อความเดียว ให้สรุปในคำตอบเดียว ไม่แยกตอบ 8 ส่วนซ้ำต่อลิงก์
- หลายลิงก์: ส่วนที่ 2 ต้องมีแถว/บล็อกของแต่ละลิงก์เอง แต่ส่วนที่ 8 รวมเป็นคำแนะนำชุดเดียว

[Truth check 3 ชั้น + ความมั่นใจ]
จากข้อมูลที่ให้ / จากแหล่งที่ตรวจเพิ่ม (ระบุแหล่ง) / ยังไม่ได้ตรวจภายนอก (ห้ามสรุปว่าจริงแน่)
แต่ละข้อใส่ระดับความมั่นใจ: สูง / กลาง / ต่ำ

รูปแบบตอบ (ใช้เฉพาะส่วนที่โหมดกำหนด):
1. สรุปสั้น
2. ประเด็นสำคัญ (แยก fact / insight / claim / risk)
   - ถ้ามีหลายลิงก์ ให้แต่ละลิงก์มีแถว/บล็อกของตัวเองในส่วนนี้
3. วิเคราะห์เชิงตัดสินใจ: truth check 3 ชั้น / จุดแข็ง / จุดอ่อน / fit กับ project /
   ควรพัฒนาต่อไหม / ควรเข้า Hermes ระดับไหน (เลือกจาก: ไม่เก็บ / review / source /
   knowledge / pattern / playbook / skill / project adapter)
4. จัดประเภท action: knowledge / plugin install / software upgrade / workflow-rule /
   security audit / project issue
5. ถ้าติดตั้ง-อัปเกรด: verification + success/failure criteria +
   scheduled evaluation (สร้างเป็น checklist + note พร้อมวันรอบประเมินที่เจ้าของกำหนด) + rollback
6. ใช้กับงานเรายังไง
7. ตารางทางเลือก: | ตัวเลือก | ทำอะไร | เหมาะเมื่อ | action หลังอนุมัติ | แนะนำ % |
8. แนะนำทางเดียวที่ดีที่สุด + เหตุผล + action ชัด
   - ถ้าคำแนะนำคือ "ไม่เอา/ปัดตก" ให้เสนอจดสั้น 1 บรรทัดลง dashboard ว่าเคยปัดตกเพราะอะไร
   - การจดผล "ไม่เอา/ปัดตก" ยังต้องขออนุมัติก่อนเหมือนการบันทึกอื่น

[หลังอนุมัติ — บันทึกเป็นระเบียน PDCA ที่ติดตามได้จริง]
เป้าหมาย: ทุกสิ่งที่บันทึกต้องมี "วงจรปิด" ไม่ใช่เขียนแล้วค้าง ต้องมีสถานะ + วันครบกำหนด + ลิงก์กลับศูนย์กลาง

0. ด่านกันบันทึกซ้ำก่อนสร้างโน้ตใหม่
   - ค้นใน $HERMES_OBSIDIAN_ROOT/95-Inbox-Lab/review/ ด้วย source_link ก่อนเสมอ
   - ถ้าเจอ source_link เดิม ให้ปรับปรุงโน้ตเดิมแทนสร้างใหม่
   - ยึดกติกา 1 เรื่อง = 1 ระเบียน
   - ถ้าค้นไม่ได้ ให้รายงานเจ้าของงาน ห้ามเดาว่าไม่มีของเดิม

1. สร้างหรือปรับปรุงโน้ต $HERMES_OBSIDIAN_ROOT/95-Inbox-Lab/review/<slug>-evaluation-<YYYY-MM-DD>.md
   - <slug> = ชื่อเรื่องแบบสั้น ใช้ตัวอักษรอังกฤษเล็ก/ตัวเลข/ขีดกลาง เช่น ai-browser-tool
   - <YYYY-MM-DD> = วันที่บันทึกจริง
   - HERMES_OBSIDIAN_ROOT: Mac = ~/ObsidianVault/HermesAgent · VPS = /home/linux-nat/ObsidianVault/HermesAgent
   - ถ้าตัวแปรนี้ไม่มี หรือหา script ไม่เจอ → รายงานเจ้าของงาน ห้ามเดา path ใหม่เอง

2. frontmatter ของโน้ตต้องมีฟิลด์วงจร PDCA ครบ (ระบบแจ้งเตือนใช้ฟิลด์นี้):
   status: trial | active | cancelled | deleted
   decision: pending | use-real | cancel | delete
   review_due: <YYYY-MM-DD วันครบกำหนดประเมิน ตามรอบที่เจ้าของกำหนด>
   source_link: <ลิงก์ต้นทางที่สรุปมา>
   hermes_km: https://eoffice.jigsawgroups.work/hr/hermes-km
   tags: type/<ประเภท> · topic/<หัวข้อ> · project/<โปรเจกต์> · status/review (+ pdca ถ้าอยู่รอบทดลอง)

   ค่า status และ decision ต้องใช้คำอังกฤษตามชุดนี้เท่านั้น ห้ามแปลไทย:
   - status ที่อนุญาต: trial | active | cancelled | deleted
   - decision ที่อนุญาต: pending | use-real | cancel | delete
   - ระเบียนใหม่เริ่มที่ status: trial และ decision: pending เสมอ
   - ถ้าเป็นผล "ไม่เอา/ปัดตก" หลังเจ้าของอนุมัติให้จด ให้ใช้ status: cancelled และ decision: cancel พร้อมเหตุผล 1 บรรทัด

3. รัน build_pdca_review_dashboard.mjs เพื่ออัปเดตศูนย์กลาง
   - Mac:
     cd ~/ObsidianVault/HermesAgent && node 99-System/automation/build_pdca_review_dashboard.mjs
   - VPS:
     cd /home/linux-nat/ObsidianVault/HermesAgent && node 99-System/automation/build_pdca_review_dashboard.mjs
   - ถ้ารันไม่ได้ ให้รายงาน error จริง ห้ามบอกว่า sync สำเร็จ

4. แจ้งเจ้าของงานเป็นลิงก์กลับ: รายการนี้ติดตามต่อได้ที่ https://eoffice.jigsawgroups.work/hr/hermes-km
   เมื่อถึง review_due ระบบ hermes-km จะแจ้งเตือนให้ตัดสิน: ใช้จริง (active) / ยกเลิก (cancelled) / ลบ (deleted)

ห้าม:
- เดาว่าผู้ใช้อยากบันทึก / ใส่ 100% ถ้าไม่ตรวจ
- สร้าง rule-shortcut ใหม่ทันทีไม่เสนอรีวิว / ตอบแค่ route การเก็บโดยไม่วิเคราะห์
- สร้าง vault/project ใหม่ทดสอบถ้าไม่อนุญาต
- บันทึกระเบียนโดยไม่มี status + decision + review_due + source_link + ลิงก์กลับ hermes-km
- สร้างโน้ตใหม่โดยไม่ค้น source_link ใน review/ ก่อน
- ติดตั้งแล้วจบโดยไม่มี scheduled evaluation + rollback
- เติมผลตรวจภายนอกเองถ้าเปิดเว็บ/ตรวจจริงไม่ได้
- ทำให้ผล "ไม่เอา/ปัดตก" หายไปเฉย ๆ ถ้าเจ้าของอนุมัติให้จดบทเรียนแล้ว
```
