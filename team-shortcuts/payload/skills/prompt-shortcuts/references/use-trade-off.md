# Use Trade-off (v1.0 · 2026-07-07)

> ที่มา: ทีมงาน (Kla) เสนอฟอร์แมตนี้ 2026-07-07 เพื่อให้ทุกการตัดสินใจเห็นสองด้านก่อนเลือก — ในงานวิศวกรรมซอฟต์แวร์แทบทุกการออกแบบมี trade-off: เพิ่มคุณสมบัติหนึ่ง มักแลกกับอีกคุณสมบัติหนึ่ง

## Shortcut

```text
Use Trade-off
```

## Prompt

```text
Use Trade-off กับเรื่องนี้

เมื่อฉันถามว่า "Trade-off ของเรื่องนี้คืออะไร" ให้ตอบครบ 8 หัวข้อ ตามลำดับนี้เสมอ:

1. วิธีนี้คืออะไร — อธิบายสั้น ๆ ว่ากำลังเลือกอะไร
2. ได้อะไรจากการเลือกวิธีนี้ — ประโยชน์ที่จับต้องได้
3. ต้องแลกกับอะไร — ต้นทุน เวลา ความซับซ้อน ความเสี่ยง วินัยที่ต้องรักษา (ห้ามข้ามข้อนี้หรือเขียนให้ดูเบากว่าจริง)
4. ทำไมบางองค์กรจึงเลือกวิธีนี้
5. ทำไมบางองค์กรไม่เลือกวิธีนี้
6. ถ้าเปรียบเทียบกับทางเลือกอื่น แตกต่างกันอย่างไร — อย่างน้อย 2 ทางเลือก รวม "อยู่แบบเดิมไม่เปลี่ยน" ด้วยเสมอ
7. เปรียบเทียบด้วยตัวอย่างในชีวิตจริง — เช่น การซื้อรถ การเช่าบ้าน หรือการเดินทาง เลือกตัวอย่างเดียวแล้วเทียบให้สุดทาง
8. สรุปว่าไม่มีวิธีที่ดีที่สุด แต่มีวิธีที่เหมาะกับสถานการณ์ — จบด้วยว่า "สถานการณ์ของเราตอนนี้" ชี้ไปทางไหนเพราะอะไร

กฎบังคับ:
- ภาษาคนทั่วไป อ่านจบใน 2-3 นาที ไม่ต้องมีพื้นเทคนิค · ศัพท์เทคนิคต้องแปลทันที
- ห้ามเชียร์ข้างเดียว — ข้อ 3 กับข้อ 5 ต้องจริงใจเท่าข้อ 2 กับข้อ 4
- ทุกข้อผูกกับบริบทของทีม/โปรเจกต์เราจริง ไม่ใช่คำตอบกลาง ๆ จากตำรา
- ถ้าข้อมูลไม่พอจะเทียบ ให้บอกว่าขาดอะไร ไม่ใช่เดา
- ปิดท้ายด้วยคำแนะนำ 1 บรรทัด + สิ่งที่ต้องตัดสินใจ (ถ้ามี)
- ถ้าเป็นการตัดสินใจสำคัญ ให้เสนอบันทึกลง Decision Log (.project/DECISIONS.md หรือหมวด P ของคู่มือ) ด้วยโครง: เลือกอะไร / แลกอะไร / ทำไม / วันที่ / ใครตัดสิน
```

## เมื่อไหร่ต้องใช้ (บังคับใน workflow)

- AI (Fable/Codex/Grok/Opus) เสนอการเปลี่ยนแปลงใหญ่: เปลี่ยนโครงสร้าง, เพิ่ม/เปลี่ยนเครื่องมือ, ลบข้อมูล, เปลี่ยนวิธีทำงานของทีม → **ต้องแนบ trade-off ครบ 8 ข้อก่อนขอเจ้าของอนุมัติ**
- ทีมถกเถียงกันว่าจะเลือกทางไหน → ใช้ฟอร์แมตนี้ปูข้อมูลก่อนคุย
- ตัดสินใจเสร็จ → เก็บสรุปเข้า Decision Log เสมอ เพื่อคนมาทีหลังไม่ต้องเถียงซ้ำ
