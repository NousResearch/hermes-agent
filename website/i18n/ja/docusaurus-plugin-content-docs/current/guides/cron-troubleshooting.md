# Cron トラブルシューティング

cron ジョブが期待どおりに動作しないときは、これらのチェックを順番に進めてください。ほとんどの問題は、タイミング、配信、権限、スキル読み込みの 4 つのカテゴリのいずれかに当てはまります。

---

## ジョブが実行されない

### チェック 1: ジョブが存在し、アクティブであることを確認

```bash
hermes cron list
```

ジョブを探し、その状態が `[active]`（`[paused]` や `[completed]` ではない）であることを確認します。`[completed]` と表示されている場合、リピート回数が使い果たされている可能性があります — ジョブを編集してリセットしてください。

### チェック 2: スケジュールが正しいことを確認

不正な形式のスケジュールは、静かに one-shot にデフォルトされるか、完全に拒否されます。式をテストしてください:

| あなたの式 | 評価されるべき内容 |
|----------------|-------------------|
| `0 9 * * *` | 毎日午前 9:00 |
| `0 9 * * 1` | 毎週月曜日の午前 9:00 |
| `every 2h` | 今から 2 時間ごと |
| `30m` | 今から 30 分後 |
| `2025-06-01T09:00:00` | 2025 年 6 月 1 日の午前 9:00（UTC） |

ジョブが一度実行された後にリストから消える場合、それは one-shot スケジュール（`30m`、`1d`、または ISO タイムスタンプ）です — これは期待される動作です。

### チェック 3: ゲートウェイは実行中か

cron ジョブは、60 秒ごとにティックするゲートウェイのバックグラウンドティッカースレッドによって発火します。通常の CLI チャットセッションは cron ジョブを自動的に発火**しません**。

ジョブが自動的に発火することを期待している場合は、実行中のゲートウェイが必要です（フォアグラウンドなら `hermes gateway`、インストール済みサービスなら `hermes gateway start`）。一度限りのデバッグには、`hermes cron tick` で手動でティックをトリガーできます。

### チェック 4: システムクロックとタイムゾーンを確認

ジョブはローカルタイムゾーンを使用します。マシンのクロックが間違っていたり、期待と異なるタイムゾーンになっていたりすると、ジョブは間違った時刻に発火します。確認:

```bash
date
hermes cron list   # next_run の時刻をローカル時刻と比較
```

## 配信の失敗

### チェック 1: 配信先（deliver target）が正しいことを確認

配信先は大文字小文字を区別し、正しいプラットフォームが設定されている必要があります。設定が誤った配信先は、応答を静かにドロップします。

| ターゲット | 必要なもの |
|--------|-----------|
| `telegram` | `~/.hermes/.env` 内の `TELEGRAM_BOT_TOKEN` |
| `discord` | `~/.hermes/.env` 内の `DISCORD_BOT_TOKEN` |
| `slack` | `~/.hermes/.env` 内の `SLACK_BOT_TOKEN` |
| `whatsapp` | WhatsApp ゲートウェイの設定 |
| `signal` | Signal ゲートウェイの設定 |
| `matrix` | Matrix ホームサーバーの設定 |
| `email` | `config.yaml` での SMTP 設定 |
| `sms` | SMS プロバイダーの設定 |
| `local` | `~/.hermes/cron/output/` への書き込みアクセス |
| `origin` | ジョブが作成されたチャットに配信 |

その他のサポートされるプラットフォームには、`mattermost`、`homeassistant`、`dingtalk`、`feishu`、`wecom`、`weixin`、`bluebubbles`、`qqbot`、`webhook` があります。`platform:chat_id` 構文（例: `telegram:-1001234567890`）で特定のチャットをターゲットにすることもできます。

配信が失敗しても、ジョブ自体は実行されます — ただどこにも送信されないだけです。`hermes cron list` で更新された `last_error` フィールド（利用可能な場合）を確認してください。

### チェック 2: `[SILENT]` の使用を確認

cron ジョブが何も出力しない、またはエージェントが `[SILENT]` で応答する場合、配信は抑制されます。これは監視ジョブでは意図的な動作ですが、プロンプトが誤ってすべてを抑制していないか確認してください。

「何も変わっていなければ [SILENT] で応答する」と書かれたプロンプトは、空でない応答も静かに飲み込んでしまいます。条件ロジックを確認してください。

### チェック 3: プラットフォームトークンの権限

各メッセージングプラットフォームのボットは、メッセージを受信するために特定の権限が必要です。配信が静かに失敗する場合:

- **Telegram**: ボットがターゲットのグループ/チャンネルの管理者である必要があります
- **Discord**: ボットがターゲットチャンネルへの送信権限を持っている必要があります
- **Slack**: ボットがワークスペースに追加され、`chat:write` スコープを持っている必要があります

### チェック 4: 応答のラッピング

デフォルトでは、cron の応答はヘッダーとフッターでラップされます（`config.yaml` の `cron.wrap_response: true`）。一部のプラットフォームや統合では、これがうまく扱われない場合があります。無効化するには:

```yaml
cron:
  wrap_response: false
```

## スキル読み込みの失敗

### チェック 1: スキルがインストールされていることを確認

```bash
hermes skills list
```

スキルは、cron ジョブにアタッチする前にインストールされている必要があります。スキルが見つからない場合は、まず `hermes skills install <skill-name>` または CLI の `/skills` でインストールしてください。

### チェック 2: スキル名とスキルフォルダ名を確認

スキル名は大文字小文字を区別し、インストールされたスキルのフォルダ名と一致する必要があります。ジョブが `ai-funding-daily-report` を指定しているのにスキルフォルダが `ai-funding-daily-report` の場合は、`hermes skills list` で正確な名前を確認してください。

### チェック 3: インタラクティブなツールを必要とするスキル

cron ジョブは、`cronjob`、`messaging`、`clarify` のツールセットが無効化された状態で実行されます。これは、再帰的な cron 作成、直接的なメッセージ送信（配信はスケジューラが処理します）、およびインタラクティブなプロンプトを防止します。スキルがこれらのツールセットに依存している場合、cron コンテキストでは動作しません。

スキルが非インタラクティブ（ヘッドレス）モードで動作することを、スキルのドキュメントで確認してください。

### チェック 4: 複数スキルの順序

複数のスキルを使用する場合、それらは順番に読み込まれます。スキル A がスキル B のコンテキストに依存する場合は、B が先に読み込まれるようにしてください:

```bash
/cron add "0 9 * * *" "..." --skill context-skill --skill target-skill
```

この例では、`context-skill` が `target-skill` より先に読み込まれます。

## ジョブのエラーと失敗

### チェック 1: 最近のジョブ出力を確認

ジョブが実行されて失敗した場合、次の場所にエラーのコンテキストが表示されることがあります:

1. ジョブが配信されるチャット（配信が成功した場合）
2. スケジューラのメッセージは `~/.hermes/logs/agent.log`（警告は `errors.log`）
3. `hermes cron list` 経由のジョブの `last_run` メタデータ

### チェック 2: よくあるエラーパターン

**スクリプトの「No such file or directory」**
`script` のパスは絶対パス（または Hermes 設定ディレクトリからの相対パス）である必要があります。確認:
```bash
ls ~/.hermes/scripts/your-script.py   # 存在する必要がある
hermes cron edit <job_id> --script ~/.hermes/scripts/your-script.py
```

**ジョブ実行時の「Skill not found」**
スキルは、スケジューラを実行しているマシンにインストールされている必要があります。マシン間を移動する場合、スキルは自動的に同期されません — `hermes skills install <skill-name>` で再インストールしてください。

**ジョブは実行されるが何も配信されない**
配信先の問題（上記の「配信の失敗」を参照）か、静かに抑制された応答（`[SILENT]`）である可能性が高いです。

**ジョブがハングまたはタイムアウトする**
スケジューラは非アクティブベースのタイムアウトを使用します（デフォルト 600 秒、`HERMES_CRON_TIMEOUT` 環境変数で設定可能、`0` で無制限）。エージェントはツールをアクティブに呼び出している限り実行を続けられます — タイマーは持続的な非アクティブの後にのみ発火します。長時間実行のジョブは、データ収集をスクリプトで処理し、結果のみを配信するべきです。

### チェック 3: ロックの競合

スケジューラは、ティックの重複を防ぐためにファイルベースのロックを使用します。2 つのゲートウェイインスタンスが実行されている場合（または CLI セッションがゲートウェイと競合する場合）、ジョブが遅延またはスキップされることがあります。

重複したゲートウェイプロセスを終了します:
```bash
ps aux | grep hermes
# 重複したプロセスを終了し、1 つだけ残す
```

### チェック 4: jobs.json の権限

ジョブは `~/.hermes/cron/jobs.json` に保存されます。このファイルがユーザーによって読み書きできない場合、スケジューラは静かに失敗します:

```bash
ls -la ~/.hermes/cron/jobs.json
chmod 600 ~/.hermes/cron/jobs.json   # ユーザーが所有しているべき
```

## パフォーマンスの問題

### ジョブの起動が遅い

各 cron ジョブは新しい AIAgent セッションを作成します。これにはプロバイダー認証やモデルの読み込みが伴う場合があります。時間に敏感なスケジュールでは、バッファ時間を追加してください（例: `0 9 * * *` の代わりに `0 8 * * *`）。

### 重複するジョブが多すぎる

スケジューラは各ティック内でジョブを順次実行します。複数のジョブが同時刻に期限を迎える場合、それらは次々と実行されます。スケジュールをずらすこと（例: 両方を `0 9 * * *` にする代わりに `0 9 * * *` と `5 9 * * *` にする）を検討して、遅延を回避してください。

### 大きなスクリプト出力

数メガバイトの出力をダンプするスクリプトは、エージェントを遅くし、トークン制限に達する可能性があります。スクリプトレベルでフィルタリング/要約してください — エージェントが推論する必要があるものだけを出力します。

## 診断コマンド

```bash
hermes cron list                    # すべてのジョブ、状態、next_run の時刻を表示
hermes cron run <job_id>            # 次のティックでスケジュール（テスト用）
hermes cron edit <job_id>           # 設定の問題を修正
hermes logs                         # 最近の Hermes ログを表示
hermes skills list                  # インストール済みスキルを確認
```

---

## さらなるヘルプを得る

このガイドを進めても問題が解決しない場合:

1. `hermes cron run <job_id>` でジョブを実行し（次のゲートウェイティックで発火）、チャット出力でエラーを確認します
2. スケジューラのメッセージは `~/.hermes/logs/agent.log`、警告は `~/.hermes/logs/errors.log` を確認します
3. [github.com/NousResearch/hermes-agent](https://github.com/NousResearch/hermes-agent) で以下を添えて issue を開きます:
   - ジョブ ID とスケジュール
   - 配信先
   - 期待した内容と実際に起きた内容
   - ログからの関連するエラーメッセージ
