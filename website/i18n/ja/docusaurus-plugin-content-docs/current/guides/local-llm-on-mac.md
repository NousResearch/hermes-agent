# MacでローカルLLMを実行する

このガイドでは、OpenAI互換APIを備えたローカルLLMサーバーをmacOS上で実行する手順を順を追って説明します。完全なプライバシー、APIコストゼロ、そしてApple Silicon上での驚くほど優れたパフォーマンスが得られます。

ここでは2つのバックエンドを扱います。

| バックエンド | インストール | 得意分野 | フォーマット |
|---------|---------|---------|--------|
| **llama.cpp** | `brew install llama.cpp` | 最速の最初のトークンまでの時間、低メモリ向けの量子化KVキャッシュ | GGUF |
| **omlx** | [omlx.ai](https://omlx.ai) | 最速のトークン生成、ネイティブMetal最適化 | MLX (safetensors) |

どちらもOpenAI互換の `/v1/chat/completions` エンドポイントを公開します。Hermesはどちらとも動作します — `http://localhost:8080` または `http://localhost:8000` を指すだけです。

:::info Apple Silicon専用
このガイドはApple Silicon（M1以降）搭載のMacを対象としています。Intel MacはGPUアクセラレーションなしのllama.cppで動作しますが — パフォーマンスは大幅に遅くなることが予想されます。
:::

## モデルの選択

手始めには、**Qwen3.5-9B** をおすすめします — 量子化により8GB以上の統合メモリに余裕を持って収まる、強力な推論モデルです。

| バリアント | ディスク上のサイズ | 必要なRAM（128Kコンテキスト） | バックエンド |
|---------|-------------|---------------------------|---------|
| Qwen3.5-9B-Q4_K_M (GGUF) | 5.3 GB | 量子化KVキャッシュで約10 GB | llama.cpp |
| Qwen3.5-9B-mlx-lm-mxfp4 (MLX) | 約5 GB | 約12 GB | omlx |

**メモリの目安:** モデルサイズ + KVキャッシュ。9B Q4モデルは約5 GBです。128Kコンテキストでのq4量子化のKVキャッシュは約4〜5 GBを追加します。デフォルト（f16）のKVキャッシュでは、これが約16 GBまで膨れ上がります。llama.cppの量子化KVキャッシュフラグは、メモリに制約のあるシステムにとって鍵となるテクニックです。

より大きなモデル（27B、35B）には、32 GB以上の統合メモリが必要です。9Bは8〜16 GBマシンにとってのスイートスポットです。

## オプションA: llama.cpp

llama.cppは、最も移植性の高いローカルLLMランタイムです。macOSでは、デフォルトでGPUアクセラレーションにMetalを使用します。

### インストール

```bash
brew install llama.cpp
```

これにより、`llama-server` コマンドがグローバルに使えるようになります。

### モデルのダウンロード

GGUF形式のモデルが必要です。最も簡単な入手元は、`huggingface-cli` 経由のHugging Faceです。

```bash
brew install huggingface-cli
```

そしてダウンロードします。

```bash
huggingface-cli download unsloth/Qwen3.5-9B-GGUF Qwen3.5-9B-Q4_K_M.gguf --local-dir ~/models
```

:::tip ゲート付きモデル
Hugging Face上の一部のモデルは認証を必要とします。401や404エラーが出る場合は、先に `huggingface-cli login` を実行してください。
:::

### サーバーの起動

```bash
llama-server -m ~/models/Qwen3.5-9B-Q4_K_M.gguf \
  -ngl 99 \
  -c 131072 \
  -np 1 \
  -fa on \
  --cache-type-k q4_0 \
  --cache-type-v q4_0 \
  --host 0.0.0.0
```

各フラグの役割は次のとおりです。

| フラグ | 目的 |
|------|---------|
| `-ngl 99` | すべてのレイヤーをGPU（Metal）にオフロードします。何もCPUに残らないように高い数値を使います。 |
| `-c 131072` | コンテキストウィンドウのサイズ（128Kトークン）。メモリが不足している場合はこれを減らします。 |
| `-np 1` | 並列スロット数。シングルユーザー用途では1のままにします — スロットを増やすとメモリ予算が分割されます。 |
| `-fa on` | Flash attention。メモリ使用量を削減し、長いコンテキストの推論を高速化します。 |
| `--cache-type-k q4_0` | キーキャッシュを4ビットに量子化します。**これが大きなメモリ節約になります。** |
| `--cache-type-v q4_0` | バリューキャッシュを4ビットに量子化します。上記と合わせて、KVキャッシュのメモリをf16と比べて約75%削減します。 |
| `--host 0.0.0.0` | すべてのインターフェースでリッスンします。ネットワークアクセスが不要な場合は `127.0.0.1` を使います。 |

次の表示が出たらサーバーの準備が整っています。

```
main: server is listening on http://0.0.0.0:8080
srv  update_slots: all slots are idle
```

### 制約のあるシステム向けのメモリ最適化

`--cache-type-k q4_0 --cache-type-v q4_0` フラグは、メモリが限られたシステムにとって最も重要な最適化です。128Kコンテキストでの影響は次のとおりです。

| KVキャッシュタイプ | KVキャッシュメモリ（128Kコンテキスト、9Bモデル） |
|---------------|--------------------------------------|
| f16（デフォルト） | 約16 GB |
| q8_0 | 約8 GB |
| **q4_0** | **約4 GB** |

8 GBのMacでは、`q4_0` のKVキャッシュを使い、コンテキストを `-c 32768`（32K）に減らします。16 GBでは、128Kコンテキストを余裕を持って扱えます。32 GB以上では、より大きなモデルや複数の並列スロットを実行できます。

それでもメモリ不足になる場合は、まずコンテキストサイズ（`-c`）を減らし、次により小さい量子化（Q4_K_Mの代わりにQ3_K_M）を試します。

### 動作確認

```bash
curl -s http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "Qwen3.5-9B-Q4_K_M.gguf",
    "messages": [{"role": "user", "content": "Hello!"}],
    "max_tokens": 50
  }' | jq .choices[0].message.content
```

### モデル名の取得

モデル名を忘れた場合は、modelsエンドポイントにクエリします。

```bash
curl -s http://localhost:8080/v1/models | jq '.data[].id'
```

## オプションB: omlx経由のMLX

[omlx](https://omlx.ai) は、MLXモデルを管理・提供するmacOSネイティブのアプリです。MLXはApple独自の機械学習フレームワークで、Apple Siliconの統合メモリアーキテクチャに特化して最適化されています。

### インストール

[omlx.ai](https://omlx.ai) からダウンロードしてインストールします。モデル管理用のGUIと組み込みサーバーを提供します。

### モデルのダウンロード

omlxアプリを使ってモデルを閲覧・ダウンロードします。`Qwen3.5-9B-mlx-lm-mxfp4` を検索してダウンロードします。モデルはローカル（通常は `~/.omlx/models/`）に保存されます。

### サーバーの起動

omlxはデフォルトで `http://127.0.0.1:8000` でモデルを提供します。アプリのUIから提供を開始するか、利用可能であればCLIを使用します。

### 動作確認

```bash
curl -s http://127.0.0.1:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "Qwen3.5-9B-mlx-lm-mxfp4",
    "messages": [{"role": "user", "content": "Hello!"}],
    "max_tokens": 50
  }' | jq .choices[0].message.content
```

### 利用可能なモデルの一覧表示

omlxは複数のモデルを同時に提供できます。

```bash
curl -s http://127.0.0.1:8000/v1/models | jq '.data[].id'
```

## ベンチマーク: llama.cpp vs MLX

両バックエンドを同じマシン（Apple M5 Max、128 GB統合メモリ）で、同じモデル（Qwen3.5-9B）を同等の量子化レベル（GGUFはQ4_K_M、MLXはmxfp4）で実行してテストしました。多様な5つのプロンプトを各3回実行し、リソースの競合を避けるためにバックエンドは順番にテストしました。

### 結果

| 指標 | llama.cpp (Q4_K_M) | MLX (mxfp4) | 勝者 |
|--------|-------------------|-------------|--------|
| **TTFT（平均）** | **67 ms** | 289 ms | llama.cpp（4.3倍高速） |
| **TTFT（p50）** | **66 ms** | 286 ms | llama.cpp（4.3倍高速） |
| **生成（平均）** | 70 tok/s | **96 tok/s** | MLX（37%高速） |
| **生成（p50）** | 70 tok/s | **96 tok/s** | MLX（37%高速） |
| **合計時間（512トークン）** | 7.3s | **5.5s** | MLX（25%高速） |

### この結果が意味すること

- **llama.cpp** はプロンプト処理に優れています — flash attention + 量子化KVキャッシュのパイプラインにより、最初のトークンを約66msで得られます。知覚される応答性が重要なインタラクティブなアプリケーション（チャットボット、オートコンプリート）を構築している場合、これは意味のある利点です。

- **MLX** は、一度動き出すとトークンを約37%速く生成します。バッチワークロード、長文生成、または初期レイテンシより合計完了時間が重要なタスクでは、MLXのほうが早く終わります。

- 両バックエンドとも**極めて一貫しています** — 実行間のばらつきは無視できる程度でした。これらの数値は信頼できます。

### どちらを選ぶべきか

| ユースケース | 推奨 |
|----------|---------------|
| インタラクティブチャット、低レイテンシのツール | llama.cpp |
| 長文生成、大量処理 | MLX (omlx) |
| メモリ制約あり（8〜16 GB） | llama.cpp（量子化KVキャッシュは他に類を見ません） |
| 複数モデルを同時に提供 | omlx（組み込みのマルチモデルサポート） |
| 最大限の互換性（Linuxも） | llama.cpp |

## Hermesへの接続

ローカルサーバーが起動したら、次を実行します。

```bash
hermes model
```

**Custom endpoint** を選択し、プロンプトに従います。ベースURLとモデル名を尋ねられるので — 上記でセットアップしたいずれかのバックエンドの値を使用します。

## タイムアウト {#timeouts}

Hermesはローカルエンドポイント（localhost、LAN IP）を自動的に検出し、ストリーミングのタイムアウトを緩和します。ほとんどのセットアップでは設定は不要です。

それでもタイムアウトエラーが発生する場合（例: 遅いハードウェア上での非常に大きなコンテキスト）は、ストリーミングの読み取りタイムアウトを上書きできます。

```bash
# .env で — デフォルトの120秒から30分に引き上げる
HERMES_STREAM_READ_TIMEOUT=1800
```

| タイムアウト | デフォルト | ローカル時の自動調整 | 環境変数による上書き |
|---------|---------|----------------------|------------------|
| ストリーム読み取り（ソケットレベル） | 120秒 | 1800秒に引き上げ | `HERMES_STREAM_READ_TIMEOUT` |
| 停滞ストリームの検出 | 180秒 | 完全に無効化 | `HERMES_STREAM_STALE_TIMEOUT` |
| API呼び出し（非ストリーミング） | 1800秒 | 変更不要 | `HERMES_API_TIMEOUT` |

ストリーム読み取りのタイムアウトは、最も問題を引き起こしやすいものです — これは次のデータチャンクを受信するためのソケットレベルの期限です。大きなコンテキストのプリフィル中、ローカルモデルはプロンプトを処理している間、数分にわたって出力を生成しないことがあります。自動検出がこれを透過的に処理します。
