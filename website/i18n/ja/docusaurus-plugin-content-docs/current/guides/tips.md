# ヒントとベストプラクティス

Hermes Agentをすぐにより効果的に使えるようになる、実用的なヒントを集めた「クイックウィン」集です。各セクションは異なる側面を扱っています — 見出しをざっと眺めて、関連するものに飛んでください。

---

## 最良の結果を得る

### 何が欲しいか具体的に伝える

曖昧なプロンプトは曖昧な結果を生みます。「コードを直して」ではなく、「`api/handlers.py` の47行目のTypeErrorを直して — `process_request()` 関数が `parse_body()` から `None` を受け取っている」と言いましょう。与えるコンテキストが多いほど、必要な反復は少なくなります。

### コンテキストを前もって提供する

リクエストの冒頭に関連する詳細を盛り込みましょう: ファイルパス、エラーメッセージ、期待される挙動。よく練られた1つのメッセージは、3往復の確認に勝ります。エラーのトレースバックは直接貼り付けてください — エージェントはそれを解析できます。

### 繰り返しの指示にはコンテキストファイルを使う

同じ指示（「スペースではなくタブを使う」「pytestを使っている」「APIは `/api/v2` にある」）を繰り返していることに気づいたら、それらを `AGENTS.md` ファイルに入れましょう。エージェントは毎セッション自動的にそれを読みます — セットアップ後の手間はゼロです。

### エージェントにツールを使わせる

すべてのステップを手取り足取り教えようとしないでください。「`tests/test_foo.py` を開いて、42行目を見て、それから…」ではなく「失敗しているテストを見つけて直して」と言いましょう。エージェントにはファイル検索、ターミナルアクセス、コード実行があります — 探索と反復を任せてください。

### 複雑なワークフローにはスキルを使う

何かのやり方を説明する長いプロンプトを書く前に、すでにそれ用のスキルがないか確認しましょう。`/skills` と入力して利用可能なスキルを閲覧するか、`/axolotl` や `/github-pr-workflow` のように直接呼び出してください。

## CLIパワーユーザーのヒント

### 複数行入力

**Alt+Enter**、**Ctrl+J**、または **Shift+Enter** を押すと、送信せずに改行を挿入できます。`Shift+Enter` は、ターミナルがそれを独立したキーストロークとして送る場合のみ機能します（デフォルトでKitty / foot / WezTerm / Ghostty、Kittyキーボードプロトコルが有効化されればiTerm2 / Alacritty / VS Codeのターミナル）。他の2つはあらゆるターミナルで機能します。

### ペースト検出

CLIは複数行のペーストを自動検出します。コードブロックやエラーのトレースバックをそのまま貼り付けてください — 各行が別々のメッセージとして送信されることはありません。ペーストはバッファリングされ、1つのメッセージとして送信されます。

### 中断とリダイレクト

**Ctrl+C** を1回押すと、応答の途中でエージェントを中断できます。その後、新しいメッセージを入力してリダイレクトできます。2秒以内にCtrl+Cを2回押すと強制終了します。これは、エージェントが間違った方向に進み始めたときに非常に役立ちます。

### `-c` でセッションを再開する

前回のセッションから何か忘れていましたか？ `hermes -c` を実行すると、完全な会話履歴を復元した状態で、中断したところから正確に再開できます。タイトルで再開することもできます: `hermes -r "my research project"`。

### クリップボード画像の貼り付け

**Ctrl+V** を押すと、クリップボードから画像を直接チャットに貼り付けられます。エージェントはビジョンを使って、スクリーンショット、図、エラーポップアップ、UIモックアップを解析します — 先にファイルに保存する必要はありません。

### スラッシュコマンドのオートコンプリート

`/` と入力して **Tab** を押すと、利用可能なすべてのコマンドが表示されます。これには組み込みコマンド（`/compress`、`/model`、`/title`）とインストール済みのすべてのスキルが含まれます。何も覚える必要はありません — Tab補完がすべて面倒を見てくれます。

:::tip
`/verbose` を使うと、ツール出力の表示モードを切り替えられます: **off → new → all → verbose**。「all」モードはエージェントの動作を見るのに最適で、「off」は単純なQ&Aには最もすっきりしています。
:::

## コンテキストファイル

### AGENTS.md: プロジェクトの頭脳

プロジェクトのルートに `AGENTS.md` を作成し、アーキテクチャの決定事項、コーディング規約、プロジェクト固有の指示を記述しましょう。これは毎セッション自動的に注入されるため、エージェントは常にプロジェクトのルールを把握しています。

```markdown
# Project Context
- This is a FastAPI backend with SQLAlchemy ORM
- Always use async/await for database operations
- Tests go in tests/ and use pytest-asyncio
- Never commit .env files
```

### SOUL.md: パーソナリティのカスタマイズ

Hermesに安定したデフォルトの声色を持たせたいですか？ `~/.hermes/SOUL.md`（カスタムのHermesホームを使っている場合は `$HERMES_HOME/SOUL.md`）を編集してください。Hermesは現在、スターターSOULを自動的にシードし、そのグローバルファイルをインスタンス全体のパーソナリティのソースとして使用します。

詳しい手順については、[HermesでSOUL.mdを使う](/docs/guides/use-soul-with-hermes)を参照してください。

```markdown
# Soul
You are a senior backend engineer. Be terse and direct.
Skip explanations unless asked. Prefer one-liners over verbose solutions.
Always consider error handling and edge cases.
```

永続的なパーソナリティには `SOUL.md` を使います。プロジェクト固有の指示には `AGENTS.md` を使います。

### .cursorrules 互換性

すでに `.cursorrules` または `.cursor/rules/*.mdc` ファイルを持っていますか？ Hermesはそれらも読みます。コーディング規約を複製する必要はありません — 作業ディレクトリから自動的に読み込まれます。

### 検出

Hermesは、セッション開始時に現在の作業ディレクトリからトップレベルの `AGENTS.md` を読み込みます。サブディレクトリの `AGENTS.md` ファイルは、ツール呼び出し中に（`subdirectory_hints.py` を介して）遅延検出され、ツール結果に注入されます — システムプロンプトに前もって読み込まれることはありません。

:::tip
コンテキストファイルは焦点を絞って簡潔に保ちましょう。それらはすべてのメッセージに注入されるため、1文字ごとにトークン予算を消費します。
:::

## メモリとスキル

### メモリ vs. スキル: 何をどこに置くか

**メモリ**は事実のためのものです: あなたの環境、好み、プロジェクトの場所、エージェントがあなたについて学んだこと。**スキル**は手順のためのものです: 複数ステップのワークフロー、ツール固有の指示、再利用可能なレシピ。「何を」にはメモリを、「どうやって」にはスキルを使いましょう。

### スキルを作成すべきとき

5ステップ以上かかり、また繰り返すであろうタスクを見つけたら、エージェントにそのためのスキルを作成するよう頼みましょう。「今やったことを `deploy-staging` というスキルとして保存して」と言ってください。次回からは、`/deploy-staging` と入力するだけで、エージェントが手順全体を読み込みます。

### メモリ容量の管理

メモリは意図的に上限が設けられています（MEMORY.mdは約2,200文字、USER.mdは約1,375文字）。いっぱいになると、エージェントはエントリを統合します。「メモリを整理して」や「古いPython 3.9のメモを置き換えて — 今は3.12だから」と言うことで手助けできます。

### エージェントに記憶させる

実りある作業セッションの後、「次回のためにこれを覚えておいて」と言えば、エージェントが重要なポイントを保存します。具体的に指定することもできます: 「CIがGitHub Actionsで `deploy.yml` ワークフローを使っていることをメモリに保存して」。

:::warning
メモリは凍結されたスナップショットです — セッション中に行われた変更は、次のセッションが始まるまでシステムプロンプトに現れません。エージェントはすぐにディスクに書き込みますが、プロンプトキャッシュはセッション途中で無効化されません。
:::

## パフォーマンスとコスト

### プロンプトキャッシュを壊さない

ほとんどのLLMプロバイダーは、システムプロンプトのプレフィックスをキャッシュします。システムプロンプトを安定させておけば（同じコンテキストファイル、同じメモリ）、セッション内の以降のメッセージは大幅に安価な**キャッシュヒット**を得られます。セッション途中でモデルやシステムプロンプトを変更するのは避けましょう。

### 制限に達する前に /compress を使う

長いセッションはトークンを蓄積します。応答が遅くなったり切り詰められたりし始めたら、`/compress` を実行しましょう。これは会話履歴を要約し、重要なコンテキストを保ちながらトークン数を劇的に削減します。`/usage` を使って現在の状況を確認してください。

### 並列作業には委譲を

3つのトピックを一度にリサーチする必要がありますか？ 並列のサブタスクで `delegate_task` を使うようエージェントに頼みましょう。各サブエージェントは独自のコンテキストで独立して実行され、最終的なサマリーのみが返ってきます — メインの会話のトークン使用量を大幅に削減します。

### バッチ操作には execute_code を使う

ターミナルコマンドを1つずつ実行する代わりに、すべてを一度に行うスクリプトを書くようエージェントに頼みましょう。「すべての `.jpeg` ファイルを `.jpg` にリネームするPythonスクリプトを書いて実行して」は、ファイルを個別にリネームするより安価で高速です。

### 適切なモデルを選ぶ

`/model` を使ってセッション途中でモデルを切り替えられます。複雑な推論やアーキテクチャの決定にはフロンティアモデル（Claude Sonnet/Opus、GPT-4o）を使いましょう。フォーマット、リネーム、ボイラープレート生成のような単純なタスクには、より速いモデルに切り替えましょう。

:::tip
`/usage` を定期的に実行して、トークン消費を確認しましょう。`/insights` を実行すると、過去30日間の使用パターンをより広い視点で見られます。
:::

## メッセージングのヒント

### ホームチャンネルを設定する

お好みのTelegramまたはDiscordのチャットで `/sethome` を使い、それをホームチャンネルに指定しましょう。cronジョブの結果やスケジュールされたタスクの出力はここに配信されます。設定しないと、エージェントは能動的なメッセージを送る先がありません。

### /title でセッションを整理する

`/title auth-refactor` や `/title research-llm-quantization` でセッションに名前を付けましょう。名前付きのセッションは `hermes sessions list` で見つけやすく、`hermes -r "auth-refactor"` で再開できます。名前のないセッションは積み上がり、区別がつかなくなります。

### チームアクセスのためのDMペアリング

許可リスト用にユーザーIDを手動で集める代わりに、DMペアリングを有効にしましょう。チームメイトがボットにDMを送ると、一度限りのペアリングコードを受け取ります。`hermes pairing approve telegram XKGH5N7P` でそれを承認します — シンプルで安全です。

### ツール進行状況の表示モード

`/verbose` を使って、表示されるツールアクティビティの量を制御できます。メッセージングプラットフォームでは、通常は少ないほど良いので — 新しいツール呼び出しだけを見るには「new」のままにしておきましょう。CLIでは、「all」がエージェントの全動作を満足のいくライブビューで見せてくれます。

:::tip
メッセージングプラットフォームでは、アイドル時間（デフォルト: 24時間）の後、または毎日午前4時にセッションが自動リセットされます。より長いセッションが必要な場合は、`~/.hermes/config.yaml` でプラットフォームごとに調整してください。
:::

## セキュリティ

### 信頼できないコードにはDockerを使う

信頼できないリポジトリを扱ったり、馴染みのないコードを実行したりするときは、DockerまたはDaytonaをターミナルバックエンドとして使いましょう。`.env` に `TERMINAL_BACKEND=docker` を設定します。コンテナ内の破壊的なコマンドは、ホストシステムに害を与えられません。

```bash
# .env で:
TERMINAL_BACKEND=docker
TERMINAL_DOCKER_IMAGE=hermes-sandbox:latest
```

### Windowsのエンコーディングの落とし穴を避ける

Windowsでは、一部のデフォルトエンコーディング（`cp125x` など）はすべてのUnicode文字を表現できず、テストやスクリプトでファイルを書き込む際に `UnicodeEncodeError` を引き起こすことがあります。

- ファイルを開くときは明示的なUTF-8エンコーディングを使うのが望ましいです:

```python
with open("results.txt", "w", encoding="utf-8") as f:
    f.write("✓ All good\n")
```

- PowerShellでは、現在のセッションをコンソールとネイティブコマンドの出力についてUTF-8に切り替えることもできます:

```powershell
$OutputEncoding = [Console]::OutputEncoding = [Text.UTF8Encoding]::new($false)
```

これにより、PowerShellと子プロセスがUTF-8に保たれ、Windows固有の失敗を避けるのに役立ちます。

### 「always」を選ぶ前に確認する

エージェントが危険なコマンドの承認（`rm -rf`、`DROP TABLE` など）をトリガーすると、4つの選択肢が表示されます: **once**、**session**、**always**、**deny**。「always」を選ぶ前によく考えてください — それはそのパターンを永久に許可リストに登録します。慣れるまでは「session」から始めましょう。

### コマンド承認はあなたのセーフティネット

Hermesは、実行前にすべてのコマンドを、入念に作られた危険なパターンのリストと照合します。これには再帰的削除、SQLのドロップ、curlをシェルにパイプすることなどが含まれます。本番環境でこれを無効化しないでください — それには正当な理由があります。

:::warning
コンテナバックエンド（Docker、Singularity、Modal、Daytona）で実行している場合、コンテナがセキュリティ境界となるため、危険なコマンドのチェックは**スキップされます**。コンテナイメージが適切にロックダウンされていることを確認してください。
:::

### メッセージングボットには許可リストを使う

ターミナルアクセスを持つボットで `GATEWAY_ALLOW_ALL_USERS=true` を決して設定しないでください。常にプラットフォーム固有の許可リスト（`TELEGRAM_ALLOWED_USERS`、`DISCORD_ALLOWED_USERS`）またはDMペアリングを使って、誰がエージェントと対話できるかを制御してください。

```bash
# 推奨: プラットフォームごとの明示的な許可リスト
TELEGRAM_ALLOWED_USERS=123456789,987654321
DISCORD_ALLOWED_USERS=123456789012345678

# またはクロスプラットフォームの許可リストを使う
GATEWAY_ALLOWED_USERS=123456789,987654321
```
