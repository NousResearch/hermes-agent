# 永続メモリ

Hermes Agent は、セッションをまたいで保持される、上限付きで厳選されたメモリを持っています。これにより、あなたの好み、プロジェクト、環境、そして学習した事柄を記憶できます。

## 仕組み

エージェントのメモリは2つのファイルで構成されています。

| ファイル | 目的 | 文字数上限 |
|------|---------|------------|
| **MEMORY.md** | エージェント自身のメモ — 環境に関する事実、慣習、学習した事柄 | 2,200文字（約800トークン） |
| **USER.md** | ユーザープロファイル — あなたの好み、コミュニケーションスタイル、期待 | 1,375文字（約500トークン） |

どちらも `~/.hermes/memories/` に保存され、セッション開始時に固定されたスナップショットとしてシステムプロンプトに注入されます。エージェントは `memory` ツールを介して自身のメモリを管理します — エントリの追加、置換、削除が可能です。

:::info
文字数上限によってメモリを焦点の絞られた状態に保ちます。メモリが満杯になると、エージェントはエントリを統合または置換して新しい情報のための余地を作ります。
:::

## メモリがシステムプロンプトにどう現れるか

すべてのセッションの開始時に、メモリのエントリがディスクから読み込まれ、固定されたブロックとしてシステムプロンプトにレンダリングされます。

```
══════════════════════════════════════════════
MEMORY (your personal notes) [67% — 1,474/2,200 chars]
══════════════════════════════════════════════
User's project is a Rust web service at ~/code/myapi using Axum + SQLx
§
This machine runs Ubuntu 22.04, has Docker and Podman installed
§
User prefers concise responses, dislikes verbose explanations
```

フォーマットには次の要素が含まれます。
- どのストア（MEMORY か USER PROFILE か）を示すヘッダー
- エージェントが容量を把握できるよう、使用率と文字数
- `§`（セクション記号）区切り文字で分割された個々のエントリ
- エントリは複数行にわたることができます

**固定スナップショットパターン:** システムプロンプトへの注入はセッション開始時に一度だけ取得され、セッションの途中で変化することはありません。これは意図的なものです — LLM のプレフィックスキャッシュを保持してパフォーマンスを維持するためです。エージェントがセッション中にメモリのエントリを追加・削除すると、その変更は即座にディスクに永続化されますが、次のセッションが開始するまではシステムプロンプトには現れません。ツールのレスポンスは常にライブの状態を表示します。

## メモリツールのアクション

エージェントは `memory` ツールを次のアクションで使用します。

- **add** — 新しいメモリエントリを追加する
- **replace** — 既存のエントリを更新後の内容で置換する（`old_text` による部分文字列マッチングを使用）
- **remove** — もはや関連性のないエントリを削除する（`old_text` による部分文字列マッチングを使用）

`read` アクションは存在しません — メモリの内容はセッション開始時に自動的にシステムプロンプトへ注入されます。エージェントは自身のメモリを会話コンテキストの一部として参照します。

### 部分文字列マッチング

`replace` と `remove` のアクションは、短く一意な部分文字列マッチングを使用します — エントリの全文は必要ありません。`old_text` パラメータは、ちょうど1つのエントリを特定する一意な部分文字列であれば十分です。

```python
# メモリに "User prefers dark mode in all editors" が含まれている場合
memory(action="replace", target="memory",
       old_text="dark mode",
       content="User prefers light mode in VS Code, dark mode in terminal")
```

部分文字列が複数のエントリにマッチする場合は、より具体的なマッチを求めるエラーが返されます。

## 2つのターゲットの説明

### `memory` — エージェント自身のメモ

エージェントが環境、ワークフロー、得られた教訓について記憶しておく必要のある情報のためのものです。

- 環境に関する事実（OS、ツール、プロジェクト構成）
- プロジェクトの慣習や設定
- 発見したツールの癖や回避策
- 完了したタスクの日誌エントリ
- 有効だったスキルやテクニック

### `user` — ユーザープロファイル

ユーザーの素性、好み、コミュニケーションスタイルに関する情報のためのものです。

- 名前、役割、タイムゾーン
- コミュニケーションの好み（簡潔か詳細か、フォーマットの好み）
- 嫌がること、避けるべき事柄
- ワークフローの習慣
- 技術的なスキルレベル

## 保存すべきもの vs 省くもの

### これらは保存する（積極的に）

エージェントは自動的に保存します — 依頼する必要はありません。次のような事柄を学習したときに保存します。

- **ユーザーの好み:** 「JavaScript より TypeScript が好き」→ `user` に保存
- **環境に関する事実:** 「このサーバーは Debian 12 と PostgreSQL 16 で動作している」→ `memory` に保存
- **訂正:** 「Docker コマンドに `sudo` を使わないで、ユーザーは docker グループに入っている」→ `memory` に保存
- **慣習:** 「プロジェクトはタブ、120文字の行幅、Google スタイルの docstring を使用」→ `memory` に保存
- **完了した作業:** 「2026-01-15 にデータベースを MySQL から PostgreSQL へ移行した」→ `memory` に保存
- **明示的なリクエスト:** 「私の API キーのローテーションは毎月行われることを覚えておいて」→ `memory` に保存

### これらは省く

- **些細・自明な情報:** 「ユーザーが Python について尋ねた」 — 役に立つには曖昧すぎる
- **簡単に再発見できる事実:** 「Python 3.12 は f-string のネストをサポートしている」 — Web 検索できる
- **生データのダンプ:** 大きなコードブロック、ログファイル、データテーブル — メモリには大きすぎる
- **セッション固有の一時情報:** 一時的なファイルパス、その場限りのデバッグコンテキスト
- **すでにコンテキストファイルにある情報:** SOUL.md や AGENTS.md の内容

## 容量管理

メモリには、システムプロンプトを上限内に保つための厳格な文字数制限があります。

| ストア | 上限 | 一般的なエントリ数 |
|-------|-------|----------------|
| memory | 2,200文字 | 8〜15エントリ |
| user | 1,375文字 | 5〜10エントリ |

### メモリが満杯になったとき

上限を超えるエントリを追加しようとすると、ツールはエラーを返します。

```json
{
  "success": false,
  "error": "Memory at 2,100/2,200 chars. Adding this entry (250 chars) would exceed the limit. Replace or remove existing entries first.",
  "current_entries": ["..."],
  "usage": "2,100/2,200"
}
```

このときエージェントは次のようにすべきです。
1. 現在のエントリを読む（エラーレスポンスに表示されています）
2. 削除または統合できるエントリを特定する
3. `replace` を使って関連するエントリをより短いバージョンに統合する
4. その後、新しいエントリを `add` する

**ベストプラクティス:** メモリが容量の80%を超えたら（システムプロンプトのヘッダーで確認できます）、新しいエントリを追加する前にエントリを統合してください。たとえば、3つの別々の「プロジェクトは X を使用」というエントリを、1つの包括的なプロジェクト説明エントリにまとめます。

### 良いメモリエントリの実例

**簡潔で情報密度の高いエントリが最も効果的です。**

```
# 良い例: 関連する複数の事実を詰め込んでいる
User runs macOS 14 Sonoma, uses Homebrew, has Docker Desktop and Podman. Shell: zsh with oh-my-zsh. Editor: VS Code with Vim keybindings.

# 良い例: 具体的で実行可能な慣習
Project ~/code/api uses Go 1.22, sqlc for DB queries, chi router. Run tests with 'make test'. CI via GitHub Actions.

# 良い例: 文脈つきの教訓
The staging server (10.0.1.50) needs SSH port 2222, not 22. Key is at ~/.ssh/staging_ed25519.

# 悪い例: 曖昧すぎる
User has a project.

# 悪い例: 冗長すぎる
On January 5th, 2026, the user asked me to look at their project which is
located at ~/code/api. I discovered it uses Go version 1.22 and...
```

## 重複防止

メモリシステムは、完全に重複するエントリを自動的に拒否します。すでに存在する内容を追加しようとした場合、「重複は追加されませんでした」というメッセージとともに成功を返します。

## セキュリティスキャン

メモリのエントリはシステムプロンプトに注入されるため、受け入れられる前にインジェクションや情報持ち出しのパターンがスキャンされます。脅威パターン（プロンプトインジェクション、認証情報の持ち出し、SSH バックドア）にマッチする内容や、不可視の Unicode 文字を含む内容はブロックされます。

## セッション検索

MEMORY.md と USER.md を超えて、エージェントは `session_search` ツールを使って過去の会話を検索できます。

- すべての CLI およびメッセージングのセッションは、FTS5 全文検索を備えた SQLite（`~/.hermes/state.db`）に保存されます
- 検索クエリは、Gemini Flash による要約とともに関連する過去の会話を返します
- エージェントは、アクティブなメモリに含まれていなくても、数週間前に議論した事柄を見つけられます

```bash
hermes sessions list    # 過去のセッションを閲覧
```

### session_search vs memory

| 機能 | 永続メモリ | セッション検索 |
|---------|------------------|----------------|
| **容量** | 合計約1,300トークン | 無制限（全セッション） |
| **速度** | 即時（システムプロンプト内） | 検索 + LLM による要約が必要 |
| **ユースケース** | 常に利用可能な重要な事実 | 特定の過去の会話を見つける |
| **管理** | エージェントが手動で厳選 | 自動 — 全セッションを保存 |
| **トークンコスト** | セッションごとに固定（約1,300トークン） | オンデマンド（必要なときに検索） |

**メモリ**は、常にコンテキストにあるべき重要な事実のためのものです。**セッション検索**は、「先週 X について議論したか？」といった、エージェントが過去の会話から具体的な内容をリコールする必要があるクエリのためのものです。

## 設定

```yaml
# ~/.hermes/config.yaml 内
memory:
  memory_enabled: true
  user_profile_enabled: true
  memory_char_limit: 2200   # ~800 tokens
  user_char_limit: 1375     # ~500 tokens
```

## 外部メモリプロバイダー

MEMORY.md と USER.md を超えた、より深く永続的なメモリのために、Hermes は8つの外部メモリプロバイダープラグインを同梱しています — Honcho、OpenViking、Mem0、Hindsight、Holographic、RetainDB、ByteRover、Supermemory です。

外部プロバイダーは組み込みメモリと**並行して**動作し（組み込みメモリを置き換えることは決してありません）、ナレッジグラフ、セマンティック検索、自動的な事実抽出、セッションをまたいだユーザーモデリングといった機能を追加します。

```bash
hermes memory setup      # プロバイダーを選んで設定する
hermes memory status     # 何が有効かを確認する
```

各プロバイダーの詳細、セットアップ手順、比較については、[メモリプロバイダー](./memory-providers.md)ガイドを参照してください。
