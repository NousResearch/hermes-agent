# ビジョンと画像ペースト

Hermes Agent は**マルチモーダルビジョン**をサポートします — クリップボードから画像を CLI に直接貼り付けて、エージェントに分析、説明、または操作を依頼できます。画像は base64 エンコードされたコンテンツブロックとしてモデルに送られるため、ビジョン対応の任意のモデルがそれを処理できます。

## 仕組み

1. 画像をクリップボードにコピーします（スクリーンショット、ブラウザ画像など）
2. 以下のいずれかの方法で添付します
3. 質問を入力して Enter を押します
4. 画像が入力欄の上に `[📎 Image #1]` バッジとして表示されます
5. 送信時、画像はビジョンコンテンツブロックとしてモデルに送られます

送信前に複数の画像を添付できます — それぞれが独自のバッジを取得します。`Ctrl+C` を押すと、添付されたすべての画像をクリアします。

画像はタイムスタンプ付きのファイル名で PNG ファイルとして `~/.hermes/images/` に保存されます。

## ペースト方法

画像を添付する方法は、ターミナル環境に依存します。すべての方法がどこでも動作するわけではありません — ここに完全な内訳を示します:

### `/paste` コマンド

**最も信頼性の高い明示的な画像添付フォールバック。**

```
/paste
```

`/paste` と入力して Enter を押します。Hermes はクリップボードに画像がないか確認し、それを添付します。これは、ターミナルが `Cmd+V`/`Ctrl+V` を書き換える場合や、画像のみをコピーして検査するブラケットペーストのテキストペイロードがない場合に最も安全な選択肢です。

### Ctrl+V / Cmd+V

Hermes はペーストを階層化されたフローとして扱うようになりました:
- まず通常のテキストペースト
- ターミナルがテキストをクリーンに配信しなかった場合は、ネイティブクリップボード / OSC52 テキストフォールバック
- クリップボードまたはペーストされたペイロードが画像または画像パスとして解決された場合は画像添付

これは、ペーストされた macOS スクリーンショットの一時パスや `file://...` の画像 URI が、生のテキストとしてコンポーザーに残るのではなく、即座に添付できることを意味します。

:::warning
クリップボードに**画像のみ**（テキストなし）がある場合、ターミナルは依然としてバイナリの画像バイトを直接送信できません。明示的な画像添付フォールバックとして `/paste` を使用してください。
:::

### VS Code / Cursor / Windsurf 向けの `/terminal-setup`

macOS のローカルな VS Code 系の統合ターミナル内で TUI を実行している場合、Hermes は複数行と undo/redo の互換性を向上させるために推奨される `workbench.action.terminal.sendSequence` バインディングをインストールできます:

```text
/terminal-setup
```

これは、`Cmd+Enter`、`Cmd+Z`、`Shift+Cmd+Z` が IDE によってインターセプトされている場合に特に便利です。ローカルマシンでのみ実行してください — SSH セッション内では実行しないでください。

## プラットフォーム互換性

| 環境 | `/paste` | Cmd/Ctrl+V | `/terminal-setup` | 備考 |
|---|:---:|:---:|:---:|---|
| **macOS Terminal / iTerm2** | ✅ | ✅ | n/a | 最高の体験 — ネイティブクリップボード + スクリーンショットパスの回復 |
| **Apple Terminal** | ✅ | ✅ | n/a | Cmd+←/→/⌫ が書き換えられる場合は、Ctrl+A / Ctrl+E / Ctrl+U のフォールバックを使用 |
| **Linux X11 デスクトップ** | ✅ | ✅ | n/a | `xclip` が必要（`apt install xclip`） |
| **Linux Wayland デスクトップ** | ✅ | ✅ | n/a | `wl-paste` が必要（`apt install wl-clipboard`） |
| **WSL2（Windows Terminal）** | ✅ | ✅ | n/a | `powershell.exe` を使用 — 追加のインストール不要 |
| **VS Code / Cursor / Windsurf（ローカル）** | ✅ | ✅ | ✅ | より良い Cmd+Enter / undo / redo の互換性のために推奨 |
| **VS Code / Cursor / Windsurf（SSH）** | ❌² | ❌² | ❌³ | 代わりにローカルマシンで `/terminal-setup` を実行 |
| **SSH ターミナル（すべて）** | ❌² | ❌² | n/a | リモートクリップボードにアクセス不可 |

² 下記の [SSH とリモートセッション](#ssh--remote-sessions) を参照
³ このコマンドはローカル IDE のキーバインドを書き込むため、リモートホストからは実行しないでください

## プラットフォーム別のセットアップ

### macOS

**セットアップ不要。** Hermes はクリップボードを読み取るために `osascript`（macOS に組み込み）を使用します。より高速なパフォーマンスのために、オプションで `pngpaste` をインストールできます:

```bash
brew install pngpaste
```

### Linux（X11）

`xclip` をインストールします:

```bash
# Ubuntu/Debian
sudo apt install xclip

# Fedora
sudo dnf install xclip

# Arch
sudo pacman -S xclip
```

### Linux（Wayland）

モダンな Linux デスクトップ（Ubuntu 22.04+、Fedora 34+）は、デフォルトで Wayland を使用することがよくあります。`wl-clipboard` をインストールします:

```bash
# Ubuntu/Debian
sudo apt install wl-clipboard

# Fedora
sudo dnf install wl-clipboard

# Arch
sudo pacman -S wl-clipboard
```

:::tip Wayland かどうかを確認する方法
```bash
echo $XDG_SESSION_TYPE
# "wayland" = Wayland、"x11" = X11、"tty" = ディスプレイサーバーなし
```
:::

### WSL2

**追加のセットアップ不要。** Hermes は WSL2 を自動的に検出し（`/proc/version` 経由）、.NET の `System.Windows.Forms.Clipboard` を通じて Windows クリップボードにアクセスするために `powershell.exe` を使用します。これは WSL2 の Windows 相互運用に組み込まれています — `powershell.exe` はデフォルトで利用可能です。

クリップボードのデータは base64 エンコードされた PNG として stdout 経由で転送されるため、ファイルパス変換や一時ファイルは不要です。

:::info WSLg に関する注記
WSLg（GUI サポート付きの WSL2）を実行している場合、Hermes はまず PowerShell パスを試し、その後 `wl-paste` にフォールバックします。WSLg のクリップボードブリッジは画像に対して BMP 形式のみをサポートします — Hermes は Pillow（インストールされている場合）または ImageMagick の `convert` コマンドを使って BMP を PNG に自動変換します。
:::

#### WSL2 クリップボードアクセスの検証

```bash
# 1. WSL 検出を確認
grep -i microsoft /proc/version

# 2. PowerShell がアクセス可能か確認
which powershell.exe

# 3. 画像をコピーしてから確認
powershell.exe -NoProfile -Command "Add-Type -AssemblyName System.Windows.Forms; [System.Windows.Forms.Clipboard]::ContainsImage()"
# "True" を出力するはず
```

## SSH とリモートセッション {#ssh--remote-sessions}

**クリップボードの画像ペーストは SSH 越しでは完全には動作しません。** リモートマシンに SSH すると、Hermes CLI はリモートホストで実行されます。クリップボードツール（`xclip`、`wl-paste`、`powershell.exe`、`osascript`）は、それらが実行されているマシンのクリップボードを読み取ります — それはローカルマシンではなくリモートサーバーです。したがって、ローカルクリップボードの画像はリモート側からアクセスできません。

テキストはターミナルペーストや OSC52 を通じて橋渡しできる場合がありますが、画像クリップボードへのアクセスとローカルのスクリーンショット一時パスは、Hermes を実行しているマシンに紐づいたままです。

### SSH の回避策

1. **画像ファイルをアップロードする** — 画像をローカルに保存し、`scp`、VSCode のファイルエクスプローラ（ドラッグ＆ドロップ）、または任意のファイル転送方法でリモートサーバーにアップロードします。その後、パスで参照します。*（`/attach <filepath>` コマンドは将来のリリースで予定されています。）*

2. **URL を使用する** — 画像がオンラインでアクセス可能であれば、メッセージに URL を貼り付けるだけです。エージェントは `vision_analyze` を使って任意の画像 URL を直接見ることができます。

3. **X11 フォワーディング** — `ssh -X` で接続して X11 をフォワードします。これにより、リモートマシンの `xclip` がローカルの X11 クリップボードにアクセスできます。ローカルで X サーバーが実行されている必要があります（macOS では XQuartz、Linux X11 デスクトップでは組み込み）。大きな画像では遅くなります。

4. **メッセージングプラットフォームを使用する** — Telegram、Discord、Slack、WhatsApp 経由で Hermes に画像を送信します。これらのプラットフォームは画像アップロードをネイティブに処理し、クリップボード/ターミナルの制限の影響を受けません。

## なぜターミナルは画像をペーストできないのか

これはよくある混乱の原因なので、ここに技術的な説明を示します:

ターミナルは**テキストベース**のインターフェースです。Ctrl+V（または Cmd+V）を押すと、ターミナルエミュレーターは:

1. クリップボードから**テキストコンテンツ**を読み取ります
2. それを[ブラケットペースト](https://en.wikipedia.org/wiki/Bracketed-paste)のエスケープシーケンスでラップします
3. ターミナルのテキストストリームを通じてアプリケーションに送信します

クリップボードに画像のみ（テキストなし）が含まれている場合、ターミナルは送信するものがありません。バイナリの画像データのための標準的なターミナルエスケープシーケンスは存在しません。ターミナルは単に何もしません。

これが、Hermes が別個のクリップボードチェックを使用する理由です — ターミナルペーストイベントを通じて画像データを受け取る代わりに、OS レベルのツール（`osascript`、`powershell.exe`、`xclip`、`wl-paste`）をサブプロセス経由で直接呼び出して、クリップボードを独立して読み取ります。

## サポートされるモデル

画像ペーストは、ビジョン対応の任意のモデルで動作します。画像は OpenAI ビジョンコンテンツ形式の base64 エンコードされたデータ URL として送られます:

```json
{
  "type": "image_url",
  "image_url": {
    "url": "data:image/png;base64,..."
  }
}
```

GPT-4 Vision、Claude（ビジョン付き）、Gemini、OpenRouter 経由で提供されるオープンソースのマルチモーダルモデルを含む、ほとんどのモダンなモデルがこの形式をサポートします。

## 画像ルーティング（ビジョン対応 vs テキストのみのモデル）

ユーザーが画像を添付すると — CLI クリップボード、ゲートウェイ（Telegram/Discord の写真）、その他の任意の入口から — Hermes は、現在のモデルが実際にビジョンをサポートしているかどうかに基づいてそれをルーティングします:

| あなたのモデル | 画像に何が起こるか |
|---|---|
| **ビジョン対応**（GPT-4V、ビジョン付き Claude、Gemini、Qwen-VL、MiMo-VL など） | 上記のプロバイダーのネイティブ画像コンテンツ形式を使って**実際のピクセル**として送られます。テキスト要約レイヤーなし。 |
| **テキストのみ**（DeepSeek V3、小型のオープンソースモデル、古いチャット専用エンドポイント） | `vision_analyze` 補助ツールを通じてルーティングされます — 補助のビジョンモデルが画像を説明し、そのテキスト説明が会話に注入されます。 |

これは設定不要です — Hermes はプロバイダーメタデータで現在のモデルの能力を調べ、正しいパスを自動的に選択します。実際の効果として、セッション途中でビジョンモデルと非ビジョンモデルを切り替えても、ワークフローを変えることなく画像処理が「ちゃんと動く」ようになります。テキストのみのモデルは、拒否せざるを得ない壊れたマルチモーダルペイロードではなく、画像についての一貫したコンテキストを得ます。

テキスト説明のパスをどの補助モデルが処理するかは、`auxiliary.vision` の下で設定可能です — [Auxiliary Models](/docs/user-guide/configuration#auxiliary-models) を参照してください。
