# 팁 & 모범 사례 (Tips & Best Practices)

Hermes Agent를 즉시 더 효과적으로 사용할 수 있게 해주는 실용적인 팁 모음입니다. 각 섹션은 서로 다른 측면을 다룹니다 — 헤더를 훑어보고 관련 있는 부분으로 바로 넘어가세요.

:::tip 어떤 모델을 선택할지 고민되시나요?
`hermes setup --portal`을 실행하세요 — Claude, GPT-5, Gemini를 포함한 300개 이상의 모델을 단일 구독으로 사용할 수 있습니다. [Nous Portal](/integrations/nous-portal)을 참조하세요.
:::

## 최상의 결과 얻기

### 원하는 것을 구체적으로 명시하세요

모호한 프롬프트는 모호한 결과를 낳습니다. "코드 수정해줘"라고 하는 대신, "47번째 줄의 `api/handlers.py`에서 발생하는 TypeError를 수정해줘 — `process_request()` 함수가 `parse_body()`로부터 `None`을 받고 있어"라고 말하세요. 컨텍스트를 더 많이 제공할수록 반복 횟수가 줄어듭니다.

### 컨텍스트를 미리 제공하세요

파일 경로, 오류 메시지, 예상되는 동작과 같은 관련 세부 정보를 요청의 앞부분에 배치하세요. 잘 작성된 한 번의 메시지가 세 번의 명확화 과정보다 낫습니다. 오류 트레이스백을 직접 붙여넣으세요 — 에이전트가 이를 구문 분석할 수 있습니다.

### 반복되는 지침에는 컨텍스트 파일을 사용하세요

동일한 지침을 반복하고 있다면("공백 대신 탭을 사용해라", "우리는 pytest를 쓴다", "API는 `/api/v2`에 있다"), 이를 `AGENTS.md` 파일에 넣으세요. 에이전트는 매 세션마다 이 파일을 자동으로 읽으므로 설정 후에는 아무런 노력이 필요하지 않습니다.

### 에이전트가 자체 도구를 사용하게 하세요

모든 단계를 직접 안내하려 하지 마세요. "`tests/test_foo.py`를 열고 42번째 줄을 본 다음..."이라고 하기보다는 "실패하는 테스트를 찾아 수정해줘"라고 말하세요. 에이전트는 파일 검색, 터미널 액세스, 코드 실행 권한을 가지고 있습니다 — 에이전트가 탐색하고 반복할 수 있게 해주세요.

### 복잡한 워크플로우에는 스킬을 사용하세요

어떤 일을 어떻게 하는지 설명하는 긴 프롬프트를 작성하기 전에, 그에 대한 스킬이 이미 존재하는지 확인하세요. 사용 가능한 스킬을 찾아보려면 `/skills`를 입력하거나, `/axolotl` 또는 `/github-pr-workflow`처럼 스킬을 직접 호출하세요.

## CLI 파워 유저 팁

### 다중 줄 입력 (Multi-Line Input)

**Alt+Enter**, **Ctrl+J**, 또는 **Shift+Enter**를 눌러 메시지를 전송하지 않고 새 줄을 삽입하세요. `Shift+Enter`는 터미널이 이를 고유한 키 입력으로 보낼 때만 작동합니다 (Kitty / foot / WezTerm / Ghostty에서는 기본적으로 작동; iTerm2 / Alacritty / VS Code 터미널에서는 Kitty 키보드 프로토콜이 활성화된 경우 작동). 다른 두 가지 단축키는 모든 터미널에서 작동합니다.

### 붙여넣기 감지 (Paste Detection)

CLI는 다중 줄 붙여넣기를 자동으로 감지합니다. 코드 블록이나 오류 트레이스백을 직접 붙여넣기만 하면 됩니다 — 각 줄을 별도의 메시지로 전송하지 않습니다. 붙여넣은 내용은 버퍼링되어 하나의 메시지로 전송됩니다.

### 중단 및 방향 전환 (Interrupt and Redirect)

에이전트의 응답 중간에 **Ctrl+C**를 한 번 눌러 중단할 수 있습니다. 그런 다음 에이전트의 방향을 전환하기 위해 새 메시지를 입력할 수 있습니다. 2초 내에 Ctrl+C를 두 번 누르면 강제로 종료됩니다. 이는 에이전트가 잘못된 경로로 진행하기 시작할 때 매우 유용합니다.

### `-c`로 세션 재개

지난 세션에서 무언가를 잊으셨나요? `hermes -c`를 실행하여 전체 대화 기록이 복원된 상태로 중단했던 위치에서 정확히 다시 시작하세요. `hermes -r "나의 연구 프로젝트"`와 같이 제목으로 재개할 수도 있습니다.

### 클립보드 이미지 붙여넣기

**Ctrl+V**를 눌러 클립보드에 있는 이미지를 채팅에 직접 붙여넣으세요. 에이전트는 비전(vision)을 사용하여 스크린샷, 다이어그램, 오류 팝업, UI 목업을 분석합니다 — 먼저 파일로 저장할 필요가 없습니다.

### 슬래시 명령어 자동 완성

`/`를 입력하고 **Tab**을 눌러 사용 가능한 모든 명령어를 확인하세요. 여기에는 기본 제공 명령어(`/compress`, `/model`, `/title`)와 설치된 모든 스킬이 포함됩니다. 아무것도 외울 필요가 없습니다 — Tab 자동 완성이 다 해결해 줍니다.

:::tip
`/verbose`를 사용하여 도구 출력 표시 모드를 순환하세요: **off → new → all → verbose**. "all" 모드는 에이전트가 무엇을 하는지 관찰하기에 좋고, "off"는 단순한 Q&A에서 가장 깔끔합니다.
:::

## 컨텍스트 파일

### AGENTS.md: 프로젝트의 두뇌

프로젝트 루트에 `AGENTS.md`를 생성하여 아키텍처 결정, 코딩 규칙, 프로젝트별 지침을 작성하세요. 이 파일은 모든 세션에 자동으로 주입되므로 에이전트는 항상 프로젝트의 규칙을 알 수 있습니다.

```markdown
# 프로젝트 컨텍스트
- 이것은 SQLAlchemy ORM을 사용하는 FastAPI 백엔드입니다
- 데이터베이스 작업에는 항상 async/await를 사용하세요
- 테스트는 tests/에 들어가며 pytest-asyncio를 사용합니다
- 절대로 .env 파일을 커밋하지 마세요
```

### SOUL.md: 성격 커스터마이징

Hermes가 일관된 기본 목소리를 가지길 원하시나요? `~/.hermes/SOUL.md` (또는 사용자 지정 Hermes 홈을 사용하는 경우 `$HERMES_HOME/SOUL.md`)를 편집하세요. 이제 Hermes는 자동으로 기본 SOUL을 시딩하고, 이 전역 파일을 인스턴스 전체의 성격 소스로 사용합니다.

전체 안내는 [Use SOUL.md with Hermes](/guides/use-soul-with-hermes)를 참조하세요.

```markdown
# Soul
당신은 시니어 백엔드 엔지니어입니다. 간결하고 직설적으로 말하세요.
요청받지 않는 한 설명을 생략하세요. 장황한 해결책보다 한 줄짜리 해결책을 선호하세요.
항상 오류 처리와 엣지 케이스를 고려하세요.
```

지속적인 성격 설정에는 `SOUL.md`를 사용하세요. 프로젝트별 지침에는 `AGENTS.md`를 사용하세요.

### .cursorrules 호환성

이미 `.cursorrules`나 `.cursor/rules/*.mdc` 파일이 있으신가요? Hermes는 이러한 파일도 읽습니다. 코딩 규칙을 중복해서 작성할 필요가 없습니다 — 작업 디렉토리에서 자동으로 로드됩니다.

### 발견 (Discovery)

Hermes는 세션 시작 시 현재 작업 디렉토리에서 최상위 `AGENTS.md`를 로드합니다. 하위 디렉토리의 `AGENTS.md` 파일은 도구 호출 중(`subdirectory_hints.py`를 통해)에 지연 로드(lazily discovered)되어 도구 결과에 주입됩니다 — 시스템 프롬프트에 미리 로드되지는 않습니다.

:::tip
컨텍스트 파일은 집중적이고 간결하게 유지하세요. 이 파일들은 모든 메시지에 주입되므로 모든 글자가 토큰 예산에 합산됩니다.
:::

## 메모리 & 스킬

### 메모리 vs 스킬: 어디에 무엇이 들어갈까

**메모리**는 사실(facts)을 위한 것입니다: 환경, 선호도, 프로젝트 위치, 에이전트가 사용자에 대해 학습한 내용 등입니다. **스킬**은 절차(procedures)를 위한 것입니다: 다단계 워크플로우, 도구별 지침, 재사용 가능한 레시피 등입니다. 메모리는 "무엇(what)"에 사용하고, 스킬은 "어떻게(how)"에 사용하세요.

### 스킬을 생성해야 할 때

5단계 이상 걸리고 앞으로 다시 수행할 작업을 발견했다면, 에이전트에게 이를 스킬로 생성해 달라고 요청하세요. "지금 방금 한 작업을 `deploy-staging`이라는 스킬로 저장해줘"라고 말해보세요. 다음번에는 `/deploy-staging`을 입력하기만 하면 에이전트가 전체 절차를 로드합니다.

### 메모리 용량 관리

메모리는 의도적으로 제한되어 있습니다 (MEMORY.md의 경우 ~2,200자, USER.md의 경우 ~1,375자). 용량이 가득 차면 에이전트는 항목들을 통합합니다. "메모리를 정리해줘"나 "이전 Python 3.9 메모를 교체해줘 — 이제 우리는 3.12를 사용하고 있어"라고 말함으로써 에이전트를 도울 수 있습니다.

### 에이전트가 기억하게 하세요

생산적인 세션이 끝난 후, "다음번을 위해 이걸 기억해둬"라고 말하면 에이전트가 주요 내용을 저장할 것입니다. 더 구체적으로 말할 수도 있습니다: "우리 CI가 `deploy.yml` 워크플로우를 포함한 GitHub Actions를 사용한다는 사실을 메모리에 저장해줘."

:::warning
메모리는 고정된 스냅샷입니다 — 세션 중에 만들어진 변경 사항은 다음 세션이 시작될 때까지 시스템 프롬프트에 나타나지 않습니다. 에이전트는 디스크에 즉시 기록하지만 세션 중간에 프롬프트 캐시가 무효화되지는 않습니다.
:::

## 성능 & 비용

### 프롬프트 캐시를 깨지 마세요

대부분의 LLM 제공자는 시스템 프롬프트의 접두사를 캐시합니다. 시스템 프롬프트를 안정적으로 유지하면(동일한 컨텍스트 파일, 동일한 메모리), 세션의 후속 메시지들은 훨씬 더 저렴한 **캐시 히트**를 얻습니다. 세션 중간에 모델이나 시스템 프롬프트를 변경하지 마세요.

### 한계에 도달하기 전에 /compress 사용하기

긴 세션은 토큰을 누적시킵니다. 응답이 느려지거나 잘리는 것을 발견하면 `/compress`를 실행하세요. 이 명령어는 주요 컨텍스트를 보존하면서 대화 기록을 요약하여 토큰 수를 극적으로 줄여줍니다. 현재 상태를 확인하려면 `/usage`를 사용하세요.

### 병렬 작업을 위한 위임 (Delegation)

세 가지 주제를 한 번에 조사해야 하나요? 에이전트에게 병렬 하위 작업과 함께 `delegate_task`를 사용하도록 요청하세요. 각 하위 에이전트는 독립적으로 자체 컨텍스트를 가지고 실행되며, 최종 요약본만 반환되므로 메인 대화의 토큰 사용량을 크게 줄일 수 있습니다.

### 일괄 처리에 execute_code 사용하기

터미널 명령어를 한 번에 하나씩 실행하는 대신, 에이전트에게 한 번에 모든 것을 처리하는 스크립트를 작성하라고 요청하세요. "모든 `.jpeg` 파일의 이름을 `.jpg`로 바꾸는 Python 스크립트를 작성해서 실행해줘"라고 하는 것이 파일을 개별적으로 이름 변경하는 것보다 비용도 저렴하고 속도도 빠릅니다.

### 적절한 모델 선택하기

`/model`을 사용하여 세션 중간에 모델을 전환하세요. 복잡한 추론과 아키텍처 결정에는 최상위 모델(Claude Sonnet/Opus, GPT-4o)을 사용하세요. 포맷팅, 이름 변경, 보일러플레이트 생성과 같은 간단한 작업에는 더 빠른 모델로 전환하세요.

:::tip
주기적으로 `/usage`를 실행하여 토큰 소비량을 확인하세요. 지난 30일 동안의 사용 패턴을 더 넓게 보려면 `/insights`를 실행하세요.
:::

## 메시징 팁

### 홈 채널 설정하기

선호하는 Telegram이나 Discord 채팅에서 `/sethome`을 사용하여 해당 채널을 홈 채널로 지정하세요. 크론 작업 결과와 예약된 작업 출력이 여기로 전달됩니다. 홈 채널이 없으면 에이전트가 선제적인 메시지를 보낼 곳이 없습니다.

### 세션을 구성하기 위해 /title 사용하기

`/title auth-refactor` 또는 `/title research-llm-quantization`으로 세션 이름을 지정하세요. 이름이 지정된 세션은 `hermes sessions list`로 찾기 쉽고 `hermes -r "auth-refactor"`로 재개하기 쉽습니다. 이름이 지정되지 않은 세션은 쌓이게 되고 구별하기 불가능해집니다.

### 팀 액세스를 위한 DM 페어링

허용 목록(allowlist)을 위해 사용자 ID를 수동으로 수집하는 대신 DM 페어링을 활성화하세요. 팀원이 봇에게 DM을 보내면 일회용 페어링 코드를 받습니다. 당신은 `hermes pairing approve telegram XKGH5N7P`로 이를 승인합니다 — 간단하고 안전합니다.

### 도구 진행 상황 표시 모드

도구 활동을 얼마나 볼지 제어하려면 `/verbose`를 사용하세요. 메시징 플랫폼에서는 보통 덜 표시하는 것이 더 좋습니다 — 새로운 도구 호출만 보려면 "new"로 유지하세요. CLI에서는 "all"로 설정하면 에이전트가 수행하는 모든 작업을 실시간으로 볼 수 있어 만족스럽습니다.

:::tip
메시징 플랫폼에서는 유휴 시간(기본값: 24시간) 후 또는 매일 오전 4시에 세션이 자동 리셋됩니다. 더 긴 세션이 필요한 경우 `~/.hermes/config.yaml`에서 플랫폼별로 설정하세요.
:::

## 보안

### 신뢰할 수 없는 코드에는 Docker 사용하기

신뢰할 수 없는 리포지토리로 작업하거나 익숙하지 않은 코드를 실행할 때는 Docker나 Daytona를 터미널 백엔드로 사용하세요. `.env`에 `TERMINAL_BACKEND=docker`를 설정하세요. 컨테이너 내부의 파괴적인 명령은 호스트 시스템에 피해를 줄 수 없습니다.

```bash
# .env 파일에서:
TERMINAL_BACKEND=docker
TERMINAL_DOCKER_IMAGE=hermes-sandbox:latest
```

### Windows 인코딩 문제 피하기

Windows에서 일부 기본 인코딩(예: `cp125x`)은 모든 유니코드 문자를 나타낼 수 없어, 테스트나 스크립트에서 파일을 쓸 때 `UnicodeEncodeError`를 일으킬 수 있습니다.

- 명시적인 UTF-8 인코딩으로 파일을 여는 것을 선호하세요:

```python
with open("results.txt", "w", encoding="utf-8") as f:
    f.write("✓ 이상 없음\n")
```

- PowerShell에서는 콘솔 및 네이티브 명령어 출력을 위해 현재 세션을 UTF-8로 전환할 수도 있습니다:

```powershell
$OutputEncoding = [Console]::OutputEncoding = [Text.UTF8Encoding]::new($false)
```

이 방법은 PowerShell과 자식 프로세스를 UTF-8로 유지하여 Windows 전용 실패를 방지하는 데 도움이 됩니다.

### "Always(항상)"를 선택하기 전에 검토하기

에이전트가 위험한 명령어(`rm -rf`, `DROP TABLE` 등) 승인을 유발할 때 네 가지 옵션이 제공됩니다: **once(한 번)**, **session(세션)**, **always(항상)**, **deny(거부)**. "항상"을 선택하기 전에 신중히 생각하세요 — 이는 해당 패턴을 영구적으로 허용 목록에 추가합니다. 익숙해질 때까지는 "세션"으로 시작하세요.

### 명령어 승인은 당신의 안전망입니다

Hermes는 실행 전에 큐레이션된 위험한 패턴 목록과 비교하여 모든 명령어를 확인합니다. 여기에는 재귀적 삭제, SQL 삭제(drop), 셸로 curl 파이핑 등이 포함됩니다. 프로덕션 환경에서 이를 비활성화하지 마세요 — 타당한 이유가 있어 존재하는 기능입니다.

:::warning
컨테이너 백엔드(Docker, Singularity, Modal, Daytona)에서 실행할 때는 컨테이너 자체가 보안 경계가 되기 때문에 위험한 명령어 검사를 **건너뜁니다**. 컨테이너 이미지가 적절히 제한되어 있는지 확인하세요.
:::

### 메시징 봇에 허용 목록(Allowlists) 사용하기

터미널 액세스 권한이 있는 봇에는 절대 `GATEWAY_ALLOW_ALL_USERS=true`를 설정하지 마세요. 에이전트와 상호 작용할 수 있는 사람을 제어하려면 항상 플랫폼별 허용 목록(`TELEGRAM_ALLOWED_USERS`, `DISCORD_ALLOWED_USERS`)이나 DM 페어링을 사용하세요.

```bash
# 권장: 플랫폼별 명시적 허용 목록
TELEGRAM_ALLOWED_USERS=123456789,987654321
DISCORD_ALLOWED_USERS=123456789012345678

# 또는 크로스 플랫폼 허용 목록 사용
GATEWAY_ALLOWED_USERS=123456789,987654321
```
