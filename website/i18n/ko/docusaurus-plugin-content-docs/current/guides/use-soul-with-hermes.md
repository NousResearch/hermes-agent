# Hermes에서 SOUL.md 사용하기

`SOUL.md`는 귀하의 Hermes 인스턴스를 위한 **기본 아이덴티티(primary identity)**입니다. 시스템 프롬프트에서 가장 첫 번째로 위치하며 — 에이전트가 누구인지, 어떻게 말하는지, 무엇을 피해야 하는지를 정의합니다.

Hermes와 대화할 때마다 똑같은 어시스턴트처럼 느껴지게 하고 싶거나, Hermes의 페르소나를 당신만의 것으로 완전히 교체하고 싶다면 이 파일을 사용하세요.

## SOUL.md의 용도

`SOUL.md`는 다음을 위해 사용합니다:
- 어조 (tone)
- 성격 (personality)
- 의사소통 스타일
- Hermes가 얼마나 직설적이어야 하거나 따뜻해야 하는지
- Hermes가 스타일 측면에서 무엇을 피해야 하는지
- Hermes가 불확실성, 의견 불일치 및 모호함에 어떻게 대처해야 하는지

간단히 말해:
- `SOUL.md`는 Hermes가 누구인지, 그리고 어떻게 말하는지에 관한 것입니다.

## SOUL.md의 용도가 아닌 것

다음을 위해서는 사용하지 마세요:
- 리포지토리 관련 코딩 컨벤션
- 파일 경로
- 명령어
- 서비스 포트
- 아키텍처 메모
- 프로젝트 워크플로 지침

위의 항목들은 `AGENTS.md`에 속합니다.

좋은 규칙:
- 모든 곳에 적용되어야 한다면 `SOUL.md`에 넣으세요.
- 하나의 프로젝트에만 속한다면 `AGENTS.md`에 넣으세요.

## 파일 위치

Hermes는 현재 인스턴스에 대해 오직 글로벌 SOUL 파일 하나만을 사용합니다:

```text
~/.hermes/SOUL.md
```

사용자 정의 홈 디렉터리로 Hermes를 실행하는 경우, 다음 위치가 됩니다:

```text
$HERMES_HOME/SOUL.md
```

## 첫 실행 시의 동작

Hermes는 `SOUL.md`가 없는 경우 자동으로 시작(starter) 파일을 생성해 줍니다.

이는 대부분의 사용자가 즉시 읽고 편집할 수 있는 실제 파일에서부터 시작한다는 것을 의미합니다.

중요:
- 이미 `SOUL.md`를 가지고 있다면 Hermes는 이를 덮어쓰지 않습니다.
- 파일이 존재하지만 비어 있다면 Hermes는 해당 파일에서 어떤 내용도 프롬프트에 추가하지 않습니다.

## Hermes의 활용 방식

Hermes가 세션을 시작할 때, `HERMES_HOME`에서 `SOUL.md`를 읽고 프롬프트 인젝션 패턴을 스캔하며 필요시 내용을 자르고, 시스템 프롬프트의 1번 슬롯인 **에이전트 아이덴티티**로 사용합니다. 이는 SOUL.md가 내장된 기본 아이덴티티 텍스트를 완전히 대체함을 의미합니다.

SOUL.md가 없거나, 비어 있거나, 로드할 수 없는 경우 Hermes는 내장된 기본 아이덴티티로 대체(fallback)합니다.

파일을 둘러싸는 래퍼(wrapper) 언어는 추가되지 않습니다. 콘텐츠 자체가 중요합니다 — 에이전트가 생각하고 말하기를 원하는 방식으로 작성하세요.

## 처음 해보면 좋은 수정

아무것도 하지 않더라도 파일을 열어 자신과 잘 맞는 느낌이 들도록 단 몇 줄만 변경해 보세요.

예를 들어:

```markdown
당신은 직설적이고, 차분하며, 기술적으로 정확합니다.
형식적인 예의보다는 실질적인 내용을 선호하세요.
아이디어가 설득력 없을 때는 명확하게 반대 의견을 제시하세요.
깊이 있는 설명이 더 유용한 경우가 아니라면 대답은 짧게 유지하세요.
```

이것만으로도 Hermes가 주는 느낌을 눈에 띄게 바꿀 수 있습니다.

## 예시 스타일

### 1. 실용주의 엔지니어 (Pragmatic engineer)

```markdown
당신은 실용적인 시니어 엔지니어입니다.
훌륭하게 들리는 것보다 정확성과 운영상의 현실성을 더 중요하게 생각합니다.

## Style
- 직설적으로 말하세요.
- 복잡해서 깊이 있는 설명이 필요한 경우가 아니라면 간결하게 답하세요.
- 잘못된 아이디어라면 그렇다고 말하세요.
- 이상적인 추상화보다 현실적인 트레이드오프(tradeoff)를 선호하세요.

## Avoid
- 아부 (Sycophancy)
- 과장된 표현 (Hype language)
- 명백한 내용에 대한 과도한 설명
```

### 2. 연구 파트너 (Research partner)

```markdown
당신은 사려 깊은 연구 협력자입니다.
호기심이 많고, 불확실성에 솔직하며, 색다른 아이디어에 흥미를 느낍니다.

## Style
- 확실한 척하지 않고 다양한 가능성을 탐구하세요.
- 근거와 추측을 명확히 구분하세요.
- 아이디어의 구체성이 부족할 때는 명확한 이해를 위한 질문을 던지세요.
- 피상적인 완벽함보다 개념적인 깊이를 선호하세요.
```

### 3. 교육자 / 해설자 (Teacher / explainer)

```markdown
당신은 인내심 있는 기술 교육자입니다.
성과보다는 이해를 중시합니다.

## Style
- 명확하게 설명하세요.
- 도움이 된다면 예시를 활용하세요.
- 사용자가 명시하지 않는 한 사전 지식이 있다고 가정하지 마세요.
- 직관에서 출발하여 세부 내용으로 나아가세요.
```

### 4. 엄격한 리뷰어 (Tough reviewer)

```markdown
당신은 엄격한 리뷰어입니다.
공정하지만, 중요한 비판을 완화시켜 말하지 않습니다.

## Style
- 취약한 가정을 직접적으로 지적하세요.
- 조화로움보다 정확성을 우선시하세요.
- 위험과 트레이드오프에 대해 명시적으로 설명하세요.
- 모호하고 외교적인 표현보다는 직설적이고 명확한 표현을 선호하세요.
```

## 좋은 SOUL.md의 조건

좋은 `SOUL.md`는 다음과 같습니다:
- 안정적입니다.
- 광범위하게 적용할 수 있습니다.
- 특유의 명확한 목소리가 있습니다.
- 일시적인 지시 사항이 너무 많지 않습니다.

나쁜 `SOUL.md`는 다음과 같습니다:
- 프로젝트 관련 세부 사항으로 가득 차 있습니다.
- 내용에 모순이 있습니다.
- 대답의 모든 형태를 일일이 관리(micro-manage)하려 합니다.
- "도움이 되도록 하라", "명확하게 하라"와 같은 일반적인 내용이 대부분입니다.

Hermes는 이미 도움이 되고 명확해지려고 노력합니다. `SOUL.md`는 뻔한 기본값을 되풀이하는 것이 아니라 진짜 개성과 스타일을 추가해야 합니다.

## 권장 구조

헤딩(Heading)이 반드시 필요한 것은 아니지만 도움이 됩니다.

잘 작동하는 간단한 구조:

```markdown
# Identity
Hermes가 누구인지.

# Style
Hermes가 어떤 어조로 말해야 하는지.

# Avoid
Hermes가 피해야 할 것.

# Defaults
모호한 상황이 발생했을 때 Hermes가 어떻게 행동해야 하는지.
```

## SOUL.md vs /personality

이 둘은 상호 보완적입니다.

`SOUL.md`는 지속적인 기본값으로 사용하세요.
`/personality`는 일시적인 모드 전환용으로 사용하세요.

예시:
- 기본 SOUL은 실용적이고 직설적입니다.
- 단 한 번의 세션에서 `/personality teacher`를 사용합니다.
- 나중에 기본 목소리 파일을 변경하지 않고 다시 원래대로 돌아옵니다.

## SOUL.md vs AGENTS.md

이 부분에서 사람들이 가장 많이 혼동합니다.

### SOUL.md에 포함할 내용
- "직설적으로 말하세요."
- "과장된 표현을 피하세요."
- "깊은 설명이 도움이 되지 않는 한 짧은 대답을 선호하세요."
- "사용자가 틀렸을 때 반박하세요."

### AGENTS.md에 포함할 내용
- "unittest가 아닌 pytest를 사용하세요."
- "프론트엔드는 `frontend/`에 있습니다."
- "마이그레이션(migrations)을 직접 편집하지 마세요."
- "API는 8000 포트에서 실행됩니다."

## 편집 방법

```bash
nano ~/.hermes/SOUL.md
```

또는

```bash
vim ~/.hermes/SOUL.md
```

그 후 Hermes를 다시 시작하거나 새 세션을 시작하세요.

## 실용적인 작업 흐름 (Workflow)

1. 초기 제공된 기본(default) 파일에서 시작합니다.
2. 원하는 목소리 느낌과 맞지 않는 부분은 잘라냅니다.
3. 어조와 기본 행동 양식을 명확하게 정의하는 4~8줄을 추가합니다.
4. 한동안 Hermes와 대화해 봅니다.
5. 여전히 어색한 부분이 있다면 그에 맞춰 조정합니다.

이러한 반복적인 접근 방식이 한 번에 완벽한 페르소나를 디자인하려는 시도보다 더 잘 작동합니다.

## 문제 해결

### SOUL.md를 편집했는데 Hermes의 말투가 그대로입니다

확인 사항:
- 리포지토리 로컬의 `SOUL.md`가 아닌 `~/.hermes/SOUL.md` 또는 `$HERMES_HOME/SOUL.md`를 편집했는지 확인하세요.
- 파일이 비어 있지 않은지 확인하세요.
- 편집 후 세션을 다시 시작했는지 확인하세요.
- `/personality` 오버레이가 결과를 덮어쓰고 있지 않은지 확인하세요.

### Hermes가 내 SOUL.md의 일부를 무시합니다

가능한 원인:
- 더 높은 우선순위의 지시 사항이 이를 덮어쓰고 있습니다.
- 파일에 충돌하는 지침이 포함되어 있습니다.
- 파일이 너무 길어 일부가 잘렸습니다.
- 텍스트 일부가 프롬프트 인젝션 콘텐츠와 유사하여 스캐너에 의해 차단되거나 변경되었을 수 있습니다.

### 내 SOUL.md가 너무 특정 프로젝트에 종속되었습니다

프로젝트 관련 지침은 `AGENTS.md`로 옮기고, `SOUL.md`는 아이덴티티와 스타일에만 집중하도록 유지하세요.
