# 지속적인 목표 (Persistent Goals, `/goal`)

`/goal`은 Hermes에게 여러 턴(turns)에 걸쳐 유지되는 고정된 목표(standing objective)를 부여합니다. 매 턴이 끝난 후 경량의 판사 모델(judge model)이 에이전트의 마지막 응답을 통해 목표가 충족되었는지 확인합니다. 목표가 충족되지 않았다면, Hermes는 자동으로 같은 세션에 계속 진행하라는 프롬프트를 다시 넣고 작업을 계속합니다 — 목표가 달성되거나, 당신이 일시 중지하거나 지울 때까지, 혹은 턴 예산이 소진될 때까지 계속됩니다.

이는 Eric Traut (OpenAI)가 만든 [Codex CLI 0.128.0의 `/goal`](https://github.com/openai/codex)에서 직접적인 영감을 받은 우리의 **랄프 루프(Ralph loop)** 버전입니다. 핵심 아이디어 — 목표를 여러 턴 동안 유지하고 달성될 때까지 멈추지 않는 것 — 는 그들의 것입니다. 여기에 있는 구현은 독립적이며 Hermes의 아키텍처에 맞게 조정되었습니다.

## 언제 사용해야 할까요

`/goal`은 매 턴마다 당신이 프롬프트를 다시 입력하지 않고도 Hermes 스스로 반복 작업하기를 원하는 작업에 사용하세요:

- "`src/`의 모든 린트 에러를 수정하고 `ruff check`가 통과하는지 확인해 줘"
- "Y 레포지토리의 X 기능을 포팅하고, 테스트를 포함해서 CI를 통과하게 만들어"
- "실행 중간에 압축을 할 때 가끔 세션 ID가 어긋나는 이유를 조사하고 보고서를 작성해"
- "EXIF 날짜로 파일 이름을 변경하는 작은 CLI를 만들고, photos/ 폴더를 대상으로 테스트해 봐"

에이전트가 한 턴만 수행하고 멈춰야 하는 작업은 `/goal`이 필요하지 않습니다. *당신이 "계속해"라고 세 번은 말해야 할 것 같은* 작업에서 이 기능이 빛을 발합니다.

## 빠른 시작

```
/goal tests/hermes_cli/에서 실패하는 모든 테스트를 수정하고 해당 디렉토리에 대해 scripts/run_tests.sh가 통과하는지 확인해
```

당신이 보게 될 결과:

1. **목표 수락됨** — `⊙ Goal set (20-turn budget): <your goal>`
2. **턴 1 실행됨** — Hermes는 당신이 목표를 일반 메시지로 보낸 것처럼 작업을 시작합니다.
3. **판사 실행됨** — 턴이 끝나면, 판사 모델이 `done(완료)` 또는 `continue(계속)`을 결정합니다.
4. **필요시 루프 실행** — `continue`인 경우, `↻ Continuing toward goal (1/20): <judge's reason>`이 표시되고 Hermes가 자동으로 다음 단계를 수행합니다.
5. **종료** — 결국 `✓ Goal achieved: <reason>`을 보거나 `⏸ Goal paused — N/20 turns used`를 보게 됩니다.

## 명령어

| 명령어 | 기능 |
|---|---|
| `/goal <text>` | 고정된 목표를 설정(또는 교체)합니다. 첫 번째 턴을 즉시 시작하므로 별도의 메시지를 보낼 필요가 없습니다. |
| `/goal` 또는 `/goal status` | 현재 목표, 상태 및 사용된 턴 수를 표시합니다. |
| `/goal pause` | 목표를 지우지 않고 자동 계속 루프를 중지합니다. |
| `/goal resume` | 루프를 재개합니다 (턴 카운터를 0으로 재설정합니다). |
| `/goal clear` | 목표를 완전히 삭제합니다. |

CLI 및 모든 게이트웨이 플랫폼(Telegram, Discord, Slack, Matrix, Signal, WhatsApp, SMS, iMessage, Webhook, API 서버 및 웹 대시보드)에서 동일하게 작동합니다.

## 목표 진행 중 조건 추가: `/subgoal`

목표가 활성화된 동안 루프를 재설정하지 않고 `/subgoal <text>`를 사용하여 추가적인 수락 기준을 덧붙일 수 있습니다. 호출할 때마다 목표의 하위 목표 목록(subgoal list)에 번호가 매겨진 항목이 하나씩 추가됩니다; 다음 턴에 에이전트가 보게 될 **계속 진행 프롬프트(continuation prompt)**에는 원래 목표와 함께 "루프 도중에 사용자가 추가한 추가 기준" 블록이 포함되며, **판사 프롬프트(judge prompt)**는 모든 하위 목표를 고려하도록 다시 작성됩니다 — 목표는 원래 목표**와** 모든 하위 목표가 충족될 때까지 완료된 것으로 간주되지 않습니다.

| 명령어 | 기능 |
|---|---|
| `/subgoal <text>` | 활성화된 목표에 새 기준을 추가합니다. 활성화된 `/goal`이 필요합니다. |
| `/subgoal` (인자 없음) | 현재 번호가 매겨진 하위 목표 목록을 표시합니다. |
| `/subgoal remove <N>` | N번째 하위 목표를 제거합니다 (1부터 시작). |
| `/subgoal clear` | 원래 목표는 그대로 유지하면서 모든 하위 목표를 삭제합니다. |

하위 목표는 `SessionDB.state_meta`에 목표와 함께 지속적으로 저장되므로 `/resume` 이후에도 살아남습니다. 새로운 `/goal <text>`를 설정하면 기존 목표가 대체되고 하위 목표 목록이 지워지며; `/goal clear`를 해도 동일하게 지워집니다.

이 기능은 루프를 시작하고("실패하는 테스트를 고쳐줘") 중간에 "방금 패치한 버그에 대한 회귀 테스트(regression test) 추가해"라는 내용을 추가하고 싶을 때 사용하세요 — `/subgoal 회귀 테스트 추가해`는 실행 중인 루프를 깨지 않으면서 성공 기준을 강화합니다.

## 동작 세부 정보

### 판사 (The judge)

매 턴이 끝난 후 Hermes는 다음과 같은 정보를 사용하여 보조 모델(auxiliary model)을 호출합니다:

- 고정된 목표 텍스트
- 에이전트의 가장 최근 최종 응답 (마지막 약 4 KB의 텍스트)
- 판사에게 엄격한 JSON 형식으로 답하도록 지시하는 시스템 프롬프트: `{"done": <bool>, "reason": "<한 문장의 이유>"}`

판사는 의도적으로 보수적입니다: 응답이 목표가 완료되었음을 **명시적으로** 확인하거나, 최종 결과물이 명확하게 생성되었거나, 목표를 달성할 수 없거나 막힌 경우(불가능한 작업에 예산을 낭비하지 않도록 차단된 이유와 함께 DONE으로 처리됨)에만 목표를 `done`으로 표시합니다.

### 페일 오픈 (Fail-open) 의미론

판사에게 오류가 발생하면 (네트워크 문제, 형식이 잘못된 응답, 보조 클라이언트 사용 불가), Hermes는 판결을 `continue`로 처리합니다 — 고장난 판사가 진행을 멈추게(wedge) 하지 않습니다. **턴 예산(turn budget)**이 진정한 최후의 방어선(backstop)입니다.

### 턴 예산 (Turn budget)

기본값은 20번의 계속 진행 턴입니다 (`config.yaml`의 `goals.max_turns`). 예산에 도달하면 Hermes는 자동으로 일시 중지하고 어떻게 진행해야 할지 정확하게 알려줍니다:

```
⏸ Goal paused — 20/20 turns used. Use /goal resume to keep going, or /goal clear to stop.
(목표 일시 중지 — 20/20 턴 사용됨. 계속 진행하려면 /goal resume을, 중지하려면 /goal clear를 사용하세요.)
```

`/goal resume`은 카운터를 0으로 재설정하므로, 계획된 분량만큼씩 계속 진행할 수 있습니다.

### 사용자 메시지가 항상 우선합니다

목표가 활성화된 동안 당신이 보내는 모든 실제 메시지는 계속 진행(continuation) 루프보다 우선순위를 가집니다. CLI에서는 당신의 메시지가 대기 중인 계속 진행보다 먼저 `_pending_input`에 도착하며, 게이트웨이에서는 동일한 방식으로 어댑터 FIFO를 통과합니다. 당신의 턴이 끝난 후 판사가 다시 실행되므로 — 만약 당신의 메시지가 목표를 완료하게 만든다면, 판사가 이를 감지하고 멈출 것입니다.

### 중간 실행 안전장치 (Mid-run safety, 게이트웨이)

에이전트가 이미 실행 중일 때, `/goal status`, `/goal pause`, `/goal clear`를 실행하는 것은 안전합니다 — 이 명령어들은 컨트롤 플레인(control-plane) 상태만 건드리고 현재 턴을 방해하지 않습니다. 실행 중간에 **새로운** 목표를 설정하는 것(`/goal <new text>`)은 먼저 `/stop`을 하라는 메시지와 함께 거부되므로, 이전의 계속 진행 명령이 새로운 명령과 충돌(race)하지 않습니다.

### 지속성 (Persistence)

목표 상태는 `SessionDB.state_meta`에 `goal:<session_id>`를 키로 하여 저장됩니다. 이는 `/resume`이 당신이 떠났던 바로 그 지점부터 다시 시작한다는 것을 의미합니다 — 목표를 설정하고, 노트북을 닫고, 내일 다시 돌아와서 `/resume`을 하면 목표는 (활성화, 일시 중지, 또는 완료 상태 그대로) 당신이 남겨둔 모습 그대로 남아있습니다.

### 프롬프트 캐시 (Prompt cache)

계속 진행 프롬프트(continuation prompt)는 기록(history)에 추가되는 단순한 사용자 역할(user-role) 메시지입니다. 시스템 프롬프트를 변경하거나, 도구 세트를 바꾸거나, Hermes의 프롬프트 캐시를 무효화하는 방식으로 대화를 조작하지 **않습니다**. 20턴의 목표를 실행하는 것은 캐시 관점에서 볼 때 20턴의 일반 대화와 비용이 동일합니다.

## 구성 (Configuration)

`~/.hermes/config.yaml`에 다음을 추가하세요:

```yaml
goals:
  # Hermes가 자동으로 일시 중지하고 사용자에게 /goal resume을 
  # 요청하기 전의 최대 계속 진행 턴 수. 기본값은 20입니다. 
  # 더 짧은 루프를 원하면 이 값을 낮추고, 
  # 오래 실행되는 리팩토링의 경우 값을 올리세요.
  max_turns: 20
```

### 판사 모델 선택

판사는 `goal_judge` 보조 작업(auxiliary task)을 사용합니다. 기본적으로는 당신의 메인 모델로 해결됩니다 ([보조 모델(Auxiliary Models)](/user-guide/configuration#auxiliary-models) 참조). 비용을 낮게 유지하기 위해 판사를 저렴하고 빠른 모델로 라우팅하고 싶다면, 재정의(override)를 추가하세요:

```yaml
auxiliary:
  goal_judge:
    provider: openrouter
    model: google/gemini-3-flash-preview
```

판사 호출은 작고 (~200개의 출력 토큰) 턴당 한 번 실행되므로, 저렴하고 빠른 모델을 사용하는 것이 대개 올바른 선택입니다.

## 예시 설명 (Example walkthrough)

```
You: /goal 4개의 파일 /tmp/note_{1..4}.txt를 만들어줘. 턴당 하나씩 만들고, 각 파일에는 해당 번호를 텍스트로 적어줘

  ⊙ Goal set (20-turn budget): 4개의 파일 /tmp/note_{1..4}.txt를 만들어줘. 턴당 하나씩 만들고, 각 파일에는 해당 번호를 텍스트로 적어줘

Hermes: 지금 /tmp/note_1.txt를 생성합니다.
  💻 echo "1" > /tmp/note_1.txt   (0.1s)
  "1"이라는 내용으로 /tmp/note_1.txt를 생성했습니다. 지정하신 대로 다음 턴에서 남은 파일들을 계속 생성하겠습니다.

  ↻ Continuing toward goal (1/20): 4개의 파일 중 1개만 생성되었습니다; 3개의 파일이 남았습니다.

Hermes: [계속해서 고정된 목표를 향해 진행 중]
  💻 echo "2" > /tmp/note_2.txt   (0.1s)
  /tmp/note_2.txt를 생성했습니다. 두 개 남았습니다.

  ↻ Continuing toward goal (2/20): 4개의 파일 중 2개가 생성되었습니다; 2개가 남았습니다.

Hermes: [계속해서 고정된 목표를 향해 진행 중]
  💻 echo "3" > /tmp/note_3.txt   (0.1s)
  /tmp/note_3.txt를 생성했습니다.

  ↻ Continuing toward goal (3/20): 4개의 파일 중 3개가 생성되었습니다; 1개가 남았습니다.

Hermes: [계속해서 고정된 목표를 향해 진행 중]
  💻 echo "4" > /tmp/note_4.txt   (0.1s)
  4개의 파일 모두 생성되었습니다: /tmp/note_1.txt 부터 /tmp/note_4.txt까지, 각각 해당 번호를 포함하고 있습니다.

  ✓ Goal achieved: 지정된 내용으로 4개의 파일이 모두 생성되어 목표가 완료되었습니다.

You: _
```

4개의 턴, 한 번의 `/goal` 호출, 그리고 당신이 "계속해"라고 프롬프트를 입력할 일은 0번이었습니다.

## 판사가 틀렸을 때

완벽한 판사는 없습니다. 주의해야 할 두 가지 실패 모드:

**거짓 음성 (False negative) — 목표가 실제로 완료되었는데 판사가 계속 진행하라고 말할 때.** 턴 예산이 이를 잡아냅니다. `⏸ Goal paused`를 보게 될 것이며, `/goal clear`를 하거나 그냥 새로운 메시지를 보낼 수 있습니다.

**거짓 양성 (False positive) — 작업이 남았는데 판사가 완료되었다고 말할 때.** 당신은 `✓ Goal achieved`를 보게 되겠지만 실제로는 완료되지 않은 것을 압니다. 계속 진행하도록 후속 메시지를 보내거나, 목표를 더 구체적으로 다시 설정하세요: `/goal <더 구체적인 텍스트>`. 거짓 양성을 거짓 음성보다 드물게 만들기 위해 판사의 시스템 프롬프트는 의도적으로 보수적으로 설정되어 있습니다.

만약 판사의 판결이 납득되지 않는다면, `↻ Continuing toward goal`이나 `✓ Goal achieved` 줄에 있는 이유 텍스트를 통해 판사가 무엇을 근거로 판단했는지 정확히 알 수 있습니다. 이는 목표 텍스트가 모호했는지, 아니면 모델의 응답이 모호했는지 진단하기에 대개 충분합니다.
