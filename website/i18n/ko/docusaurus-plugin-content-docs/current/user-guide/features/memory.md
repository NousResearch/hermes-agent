# 영구 메모리 (Persistent Memory)

Hermes 에이전트는 세션 전반에 걸쳐 지속되는, 제한되고 큐레이션된 메모리를 갖추고 있습니다. 이를 통해 사용자의 기본 설정, 프로젝트, 환경 및 학습한 내용을 기억할 수 있습니다.

## 작동 방식 (How It Works)

에이전트의 메모리는 두 개의 파일로 구성됩니다:

| 파일 | 목적 | 글자 수 제한 |
|------|---------|------------|
| **MEMORY.md** | 에이전트의 개인 노트 — 환경에 대한 사실, 규약(conventions), 학습한 것들 | 2,200자 (~800토큰) |
| **USER.md** | 사용자 프로필 — 선호도, 커뮤니케이션 스타일, 기대치 | 1,375자 (~500토큰) |

둘 다 `~/.hermes/memories/`에 저장되며 세션 시작 시 시스템 프롬프트에 고정된(frozen) 스냅샷으로 주입됩니다. 에이전트는 `memory` 도구를 통해 자신의 메모리를 관리하며 항목을 추가, 바꾸기 또는 제거할 수 있습니다.

:::info
문자 수 제한은 메모리를 집중된 상태로 유지합니다. 메모리가 가득 차면 에이전트는 새로운 정보를 위한 공간을 만들기 위해 항목을 통합하거나 교체합니다.
:::

## 시스템 프롬프트에서 메모리가 표시되는 방법

모든 세션의 시작 부분에서 메모리 항목이 디스크에서 로드되어 시스템 프롬프트에 고정된 블록으로 렌더링됩니다:

```
══════════════════════════════════════════════
MEMORY (your personal notes) [67% — 1,474/2,200 chars]
══════════════════════════════════════════════
사용자의 프로젝트는 ~/code/myapi에 있는 Rust 웹 서비스이며 Axum + SQLx를 사용함
§
이 머신은 Ubuntu 22.04를 실행하며 Docker 및 Podman이 설치되어 있음
§
사용자는 간결한 응답을 선호하며 장황한 설명을 싫어함
```

형식에는 다음이 포함됩니다:
- 어느 저장소(MEMORY 또는 USER PROFILE)인지 보여주는 헤더
- 에이전트가 용량을 알 수 있도록 하는 사용률 퍼센트와 글자 수
- `§` (섹션 기호) 구분 기호로 분리된 개별 항목
- 항목은 여러 줄일 수 있음

**고정된 스냅샷 패턴:** 시스템 프롬프트 주입은 세션 시작 시 한 번 캡처되며 세션 중간에 절대 변경되지 않습니다. 이는 의도적인 것으로, 성능 향상을 위해 LLM의 접두사 캐시(prefix cache)를 보존합니다. 세션 중에 에이전트가 메모리 항목을 추가/제거하면 변경 사항은 디스크에 즉시 지속되지만, 다음 세션이 시작될 때까지 시스템 프롬프트에는 나타나지 않습니다. 도구 응답은 항상 라이브 상태를 표시합니다.

## 메모리 도구 작업 (Memory Tool Actions)

에이전트는 다음 작업들과 함께 `memory` 도구를 사용합니다:

- **add** — 새 메모리 항목 추가
- **replace** — 기존 항목을 업데이트된 콘텐츠로 교체 (`old_text`를 통한 하위 문자열 일치 사용)
- **remove** — 더 이상 관련이 없는 항목 제거 (`old_text`를 통한 하위 문자열 일치 사용)

`read` 작업은 없습니다 — 메모리 콘텐츠는 세션 시작 시 시스템 프롬프트에 자동으로 주입됩니다. 에이전트는 자신의 메모리를 대화 컨텍스트의 일부로 봅니다.

### 하위 문자열 일치 (Substring Matching)

`replace` 및 `remove` 작업은 짧고 고유한 하위 문자열 일치를 사용합니다. 항목의 전체 텍스트가 필요하지 않습니다. `old_text` 매개변수는 정확히 하나의 항목을 식별하는 고유한 하위 문자열이기만 하면 됩니다:

```python
# 메모리에 "사용자는 모든 편집기에서 다크 모드를 선호함"이 포함된 경우
memory(action="replace", target="memory",
       old_text="다크 모드",
       content="사용자는 VS Code에서는 라이트 모드, 터미널에서는 다크 모드를 선호함")
```

하위 문자열이 여러 항목과 일치하는 경우, 더 구체적인 일치를 요청하는 오류가 반환됩니다.

## 두 개의 대상 (Target) 설명

### `memory` — 에이전트의 개인 노트

환경, 워크플로우 및 학습한 교훈에 대해 에이전트가 기억해야 할 정보:

- 환경 사실 (OS, 도구, 프로젝트 구조)
- 프로젝트 규약 및 구성
- 발견된 도구의 특성 및 문제 해결 방법(workarounds)
- 완료된 작업 일지 항목
- 효과가 있었던 스킬 및 기술

### `user` — 사용자 프로필

사용자의 정체성, 선호도 및 커뮤니케이션 스타일에 대한 정보:

- 이름, 역할, 시간대
- 커뮤니케이션 기본 설정 (간결함 대 상세함, 형식 기본 설정)
- 짜증을 유발하는 것(Pet peeves) 및 피해야 할 것
- 워크플로우 습관
- 기술적 숙련도

## 저장해야 할 것 vs 건너뛸 것 (What to Save vs Skip)

### 저장 대상 (주도적으로 수행)

에이전트가 자동으로 저장합니다 — 요청할 필요가 없습니다. 무언가 배웠을 때 저장합니다:

- **사용자 선호도:** "나는 JavaScript보다 TypeScript를 선호해" → `user`에 저장
- **환경 사실:** "이 서버는 PostgreSQL 16이 설치된 Debian 12를 실행해" → `memory`에 저장
- **수정 사항:** "Docker 명령에 `sudo`를 쓰지 마, 사용자는 docker 그룹에 속해있어" → `memory`에 저장
- **규약:** "프로젝트는 탭, 120자 줄 너비, Google 스타일의 독스트링(docstrings)을 사용함" → `memory`에 저장
- **완료된 작업:** "2026-01-15에 MySQL에서 PostgreSQL로 데이터베이스 마이그레이션 수행" → `memory`에 저장
- **명시적 요청:** "내 API 키 순환이 매월 발생한다는 것을 기억해줘" → `memory`에 저장

### 건너뛸 것

- **사소하거나 명백한 정보:** "사용자가 Python에 대해 물어봤음" — 너무 모호해서 유용하지 않음
- **쉽게 다시 발견할 수 있는 사실:** "Python 3.12는 f-문자열 중첩을 지원함" — 웹 검색으로 찾을 수 있음
- **원시 데이터 덤프:** 큰 코드 블록, 로그 파일, 데이터 테이블 — 메모리에 담기에 너무 큼
- **세션별 일회성 내용:** 임시 파일 경로, 일회성 디버깅 컨텍스트
- **이미 컨텍스트 파일에 있는 정보:** SOUL.md 및 AGENTS.md 콘텐츠

## 용량 관리 (Capacity Management)

시스템 프롬프트를 제한된 상태로 유지하기 위해 메모리에는 엄격한 글자 수 제한이 있습니다:

| 저장소 | 제한 | 일반적인 항목 수 |
|-------|-------|----------------|
| memory | 2,200자 | 8-15개 항목 |
| user | 1,375자 | 5-10개 항목 |

### 메모리가 가득 찼을 때 일어나는 일

제한을 초과하는 항목을 추가하려고 하면 도구는 오류를 반환합니다:

```json
{
  "success": false,
  "error": "Memory at 2,100/2,200 chars. Adding this entry (250 chars) would exceed the limit. Replace or remove existing entries first.",
  "current_entries": ["..."],
  "usage": "2,100/2,200"
}
```

그러면 에이전트는 다음과 같이 해야 합니다:
1. 현재 항목을 읽습니다 (오류 응답에 표시됨)
2. 제거하거나 통합할 수 있는 항목을 식별합니다
3. `replace`를 사용하여 관련 항목을 더 짧은 버전으로 병합합니다
4. 그런 다음 새 항목을 `add`합니다

**모범 사례:** 메모리가 80% 용량을 넘으면(시스템 프롬프트 헤더에 표시됨) 새 항목을 추가하기 전에 항목을 통합하세요. 예를 들어 세 개의 개별 "프로젝트는 X를 사용함" 항목을 하나의 종합적인 프로젝트 설명 항목으로 병합합니다.

### 좋은 메모리 항목의 실제 예

**정보가 압축된 조밀한 항목이 가장 효과적입니다:**

```
# 좋음: 관련된 여러 사실을 압축
사용자는 macOS 14 Sonoma를 실행하고, Homebrew를 쓰며, Docker Desktop과 Podman을 보유. 셸: zsh 및 oh-my-zsh. 편집기: Vim 키바인딩이 있는 VS Code.

# 좋음: 구체적이고 조치를 취할 수 있는 규약
프로젝트 ~/code/api는 Go 1.22, DB 쿼리용 sqlc, chi 라우터를 사용. 'make test'로 테스트 실행. GitHub Actions를 통한 CI.

# 좋음: 컨텍스트와 함께 배운 교훈
스테이징 서버(10.0.1.50)는 22번이 아닌 2222번 SSH 포트가 필요함. 키는 ~/.ssh/staging_ed25519에 있음.

# 나쁨: 너무 모호함
사용자가 프로젝트를 가지고 있음.

# 나쁨: 너무 장황함
2026년 1월 5일에 사용자가 ~/code/api에 있는 프로젝트를 봐 달라고 요청함. Go 버전 1.22를 사용한다는 것을 알게 되었고...
```

## 중복 방지 (Duplicate Prevention)

메모리 시스템은 완전히 중복된 항목을 자동으로 거부합니다. 이미 존재하는 콘텐츠를 추가하려고 시도하면 "중복 추가되지 않음" 메시지와 함께 성공을 반환합니다.

## 보안 스캐닝 (Security Scanning)

메모리 항목은 시스템 프롬프트에 주입되므로 수락되기 전에 주입(injection) 및 유출(exfiltration) 패턴을 스캔합니다. 위협 패턴(프롬프트 주입, 자격 증명 유출, SSH 백도어)과 일치하거나 보이지 않는 유니코드 문자가 포함된 콘텐츠는 차단됩니다.

## 세션 검색 (Session Search)

MEMORY.md 및 USER.md 이외에도 에이전트는 `session_search` 도구를 사용하여 지난 대화를 검색할 수 있습니다.

- 모든 CLI 및 메시징 세션은 FTS5 전체 텍스트 검색을 사용하여 SQLite(`~/.hermes/state.db`)에 저장됩니다.
- 검색 쿼리는 LLM 요약이나 잘림 없이 DB의 실제 메시지를 반환합니다.
- 에이전트는 활성 메모리에 없는 내용이라도 몇 주 전에 논의한 내용을 찾을 수 있습니다.
- 에이전트는 찾은 세션 내에서 앞뒤로 스크롤할 수도 있습니다.

```bash
hermes sessions list    # 과거 세션 찾아보기
```

세 가지 호출 형태(검색 / 스크롤 / 탐색) 및 응답 형식은 [세션 검색 도구(Session Search Tool)](/user-guide/sessions#session-search-tool)를 참조하세요.

### session_search vs memory

| 특징 | 영구 메모리 (Persistent Memory) | 세션 검색 (Session Search) |
|---------|------------------|----------------|
| **용량** | 총 ~1,300 토큰 | 무제한 (모든 세션) |
| **속도** | 즉시 (시스템 프롬프트 내) | ~20ms FTS5 쿼리, ~1ms 스크롤 |
| **비용** | 모든 프롬프트에서의 토큰 비용 | 무료 — LLM 호출 없음 |
| **사용 사례** | 항상 사용할 수 있는 주요 사실 | 특정한 과거 대화 찾기 |
| **관리** | 에이전트에 의해 수동으로 큐레이팅 | 자동 — 모든 세션이 저장됨 |
| **토큰 비용** | 세션당 고정 (~1,300 토큰) | 온디맨드 (필요할 때 검색됨) |

**메모리(Memory)** 는 항상 컨텍스트에 있어야 하는 중요한 사실을 위한 것입니다. **세션 검색(Session search)** 은 에이전트가 과거 대화에서 구체적인 내용을 회상해야 하는 "우리가 지난주에 X를 논의했나요?"와 같은 쿼리를 위한 것입니다.

## 설정 (Configuration)

```yaml
# ~/.hermes/config.yaml 내
memory:
  memory_enabled: true
  user_profile_enabled: true
  memory_char_limit: 2200   # ~800 토큰
  user_char_limit: 1375     # ~500 토큰
```

## 외부 메모리 공급자 (External Memory Providers)

MEMORY.md 및 USER.md를 뛰어넘는 더 깊고 영구적인 메모리를 위해 Hermes는 Honcho, OpenViking, Mem0, Hindsight, Holographic, RetainDB, ByteRover 및 Supermemory를 포함한 8개의 외부 메모리 공급자 플러그인과 함께 제공됩니다.

외부 공급자는 내장 메모리와 **나란히** 실행되며(결코 교체하지 않음), 지식 그래프, 시맨틱 검색, 자동 사실 추출 및 세션 간 사용자 모델링과 같은 기능을 추가합니다.

```bash
hermes memory setup      # 공급자를 선택하고 구성
hermes memory status     # 무엇이 활성화되어 있는지 확인
```

각 제공업체에 대한 전체 내용, 설정 지침 및 비교는 [메모리 공급자 (Memory Providers)](./memory-providers.md) 가이드를 참조하세요.
