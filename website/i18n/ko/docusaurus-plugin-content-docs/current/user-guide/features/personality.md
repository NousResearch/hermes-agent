# 성격 & SOUL.md

Hermes Agent의 성격은 완벽하게 사용자 지정할 수 있습니다. `SOUL.md`는 **주요 정체성(primary identity)**입니다 — 시스템 프롬프트에서 가장 먼저 나타나며 에이전트가 누구인지 정의합니다.

- `SOUL.md` — `HERMES_HOME`에 존재하는 영구적인 페르소나 파일로, 에이전트의 정체성 역할을 합니다 (시스템 프롬프트의 슬롯 #1)
- 기본 제공 또는 사용자 지정 `/personality` 프리셋 — 세션 수준의 시스템 프롬프트 오버레이

Hermes가 누구인지 변경하거나 — 완전히 다른 에이전트 페르소나로 교체하고 싶다면 — `SOUL.md`를 편집하세요.

## 현재 SOUL.md의 작동 방식

이제 Hermes는 기본 `SOUL.md`를 다음 경로에 자동으로 생성합니다:

```text
~/.hermes/SOUL.md
```

더 정확히 말하면, 현재 인스턴스의 `HERMES_HOME`을 사용하므로, 사용자 지정 홈 디렉토리로 Hermes를 실행하면 다음 경로를 사용합니다:

```text
$HERMES_HOME/SOUL.md
```

### 주요 동작

- **SOUL.md는 에이전트의 주요 정체성입니다.** 시스템 프롬프트의 슬롯 #1을 차지하며, 하드코딩된 기본 정체성을 대체합니다.
- Hermes는 `SOUL.md`가 아직 존재하지 않는 경우 시작용 `SOUL.md`를 자동으로 생성합니다.
- 기존 사용자의 `SOUL.md` 파일은 절대 덮어쓰지 않습니다.
- Hermes는 오직 `HERMES_HOME`에서만 `SOUL.md`를 로드합니다.
- Hermes는 현재 작업 디렉토리에서 `SOUL.md`를 찾지 않습니다.
- `SOUL.md`가 존재하지만 비어 있거나 로드할 수 없는 경우, Hermes는 내장된 기본 정체성으로 폴백(fallback)합니다.
- `SOUL.md`에 내용이 있으면, 그 내용은 보안 스캔 및 자르기(truncation)를 거친 후 그대로 주입됩니다.
- SOUL.md는 컨텍스트 파일 섹션에 **중복되지 않습니다** — 정체성으로서 단 한 번만 나타납니다.

이러한 방식은 `SOUL.md`를 단순한 추가 레이어가 아닌, 진정한 사용자별 또는 인스턴스별 정체성으로 만듭니다.

## 왜 이렇게 디자인되었는가

이것은 성격을 예측 가능하게 유지합니다.

만약 Hermes가 현재 실행 중인 임의의 디렉토리에서 `SOUL.md`를 로드한다면, 프로젝트 간에 성격이 예기치 않게 변경될 수 있습니다. 오직 `HERMES_HOME`에서만 로드함으로써, 성격은 Hermes 인스턴스 자체에 속하게 됩니다.

이는 사용자에게 설명하기도 더 쉽습니다:
- "Hermes의 기본 성격을 변경하려면 `~/.hermes/SOUL.md`를 편집하세요."

## 어디서 편집해야 하는가

대부분의 사용자의 경우:

```bash
~/.hermes/SOUL.md
```

사용자 지정 홈을 사용하는 경우:

```bash
$HERMES_HOME/SOUL.md
```

## SOUL.md에는 무엇이 들어가야 하는가?

다음과 같은 영구적인 음성(voice) 및 성격 지침에 사용하세요:
- 어조 (tone)
- 의사소통 스타일
- 직설성의 수준
- 기본 상호작용 스타일
- 문체적으로 피해야 할 것
- 불확실성, 의견 불일치 또는 모호성을 처리하는 방법

다음과 같은 용도로는 사용을 자제하세요:
- 일회성 프로젝트 지침
- 파일 경로
- 레포지토리 규칙
- 일시적인 워크플로우 세부 정보

이러한 내용은 `SOUL.md`가 아닌 `AGENTS.md`에 속합니다.

## 좋은 SOUL.md 콘텐츠

좋은 SOUL 파일은 다음과 같습니다:
- 다양한 맥락(context)에서 안정적입니다
- 여러 대화에 적용될 수 있을 만큼 포괄적입니다
- 어조를 실질적으로 형성할 수 있을 만큼 구체적입니다
- 작업별 지침이 아닌, 의사소통과 정체성에 초점을 맞춥니다

### 예시

```markdown
# 성격

당신은 확고한 취향을 가진 실용적인 시니어 엔지니어입니다.
당신은 예의 바른 태도보다는 진실성, 명확성, 그리고 유용성을 최우선으로 생각합니다.

## 스타일
- 차갑지 않으면서도 직설적으로 말하세요
- 장황한 말보다는 내용의 본질을 선호하세요
- 나쁜 아이디어라고 생각되면 반대 의견을 제시하세요
- 불확실성은 숨기지 말고 분명히 인정하세요
- 깊이 있는 설명이 필요한 경우가 아니라면, 설명은 간결하게 유지하세요

## 피해야 할 것
- 아부하는 태도
- 과장된 언어
- 사용자의 프레이밍이 틀렸을 때 그것을 그대로 반복하는 것
- 뻔한 내용을 지나치게 길게 설명하는 것

## 기술적 태도
- 기발한 시스템보다 단순한 시스템을 선호하세요
- 이상적인 아키텍처보다는 실제 운영 환경의 현실을 중요하게 생각하세요
- 엣지 케이스(edge cases)를 사후 처리가 아닌 설계의 일부로 다루세요
```

## Hermes가 프롬프트에 주입하는 것

`SOUL.md`의 내용은 시스템 프롬프트의 슬롯 #1 — 에이전트 정체성 위치 — 로 직접 들어갑니다. 그 주위에 어떤 래퍼(wrapper) 언어도 추가되지 않습니다.

콘텐츠는 다음 과정을 거칩니다:
- 프롬프트 인젝션 스캔
- 용량이 너무 클 경우 자르기(truncation)

파일이 비어 있거나, 공백만 있거나, 읽을 수 없는 경우, Hermes는 내장된 기본 정체성("당신은 Nous Research가 만든 지능형 AI 어시스턴트 Hermes Agent입니다...")으로 폴백합니다. 이 폴백은 `skip_context_files`가 설정된 경우(예: 하위 에이전트/위임 컨텍스트)에도 적용됩니다.

## 보안 스캔

`SOUL.md`는 포함되기 전에 다른 컨텍스트 포함 파일들과 마찬가지로 프롬프트 인젝션 패턴에 대해 스캔됩니다.

즉, 숨겨진 메타 지침을 넣으려 하지 말고 페르소나/어조에 집중해야 한다는 것을 의미합니다.

## SOUL.md vs AGENTS.md

이것은 가장 중요한 차이점입니다.

### SOUL.md
다음을 위해 사용하세요:
- 정체성
- 어조
- 스타일
- 의사소통 기본값
- 성격 수준의 행동

### AGENTS.md
다음을 위해 사용하세요:
- 프로젝트 아키텍처
- 코딩 규칙 (conventions)
- 도구 선호도
- 레포지토리별 워크플로우
- 명령어, 포트, 경로, 배포 메모

유용한 규칙:
- 만약 당신이 어디를 가든 따라와야 하는 내용이라면, `SOUL.md`에 속합니다.
- 만약 프로젝트에 속하는 내용이라면, `AGENTS.md`에 속합니다.

## SOUL.md vs `/personality`

`SOUL.md`는 영구적인 기본 성격입니다.

`/personality`는 현재 시스템 프롬프트를 변경하거나 보완하는 세션 수준의 오버레이입니다.

따라서:
- `SOUL.md` = 기본 어조
- `/personality` = 일시적인 모드 전환

예시:
- 실용적인 기본 SOUL을 유지하다가, 튜터링 대화에서는 `/personality teacher`를 사용
- 간결한 SOUL을 유지하다가, 브레인스토밍을 할 때는 `/personality creative`를 사용

## 내장된 성격들

Hermes에는 `/personality` 명령어로 전환할 수 있는 내장된 성격들이 제공됩니다.

| 이름 | 설명 |
|------|-------------|
| **helpful** | 친절하고 다목적인 어시스턴트 |
| **concise** | 간결하고 요점만 말하는 답변 |
| **technical** | 상세하고 정확한 기술 전문가 |
| **creative** | 혁신적이고 틀에서 벗어난 사고 |
| **teacher** | 명확한 예시를 들어 설명하는 인내심 있는 교육자 |
| **kawaii** | 귀여운 표현, 반짝임, 그리고 열정 ★ |
| **catgirl** | 고양이 같은 표현을 쓰는 네코쨩, 냥~ |
| **pirate** | 캡틴 헤르메스, 기술에 정통한 해적 |
| **shakespeare** | 극적인 감각을 지닌 음유시인 산문 |
| **surfer** | 완전 쿨한 브로 바이브 |
| **noir** | 하드보일드 탐정 내레이션 |
| **uwu** | uwu 화법을 쓰는 최고의 귀요미 |
| **philosopher** | 모든 질문에 대한 깊은 고찰 |
| **hype** | 최고조의 에너지와 열정!!! |

## 명령어로 성격 전환하기

### CLI

```text
/personality
/personality concise
/personality technical
```

### 메시징 플랫폼

```text
/personality teacher
```

이러한 오버레이는 편리하지만, 오버레이가 의미 있는 변경을 하지 않는 한 전역 `SOUL.md`가 여전히 Hermes에게 지속적인 기본 성격을 부여합니다.

## 구성에서 사용자 지정 성격 정의하기

`~/.hermes/config.yaml` 파일의 `agent.personalities` 아래에 이름이 지정된 사용자 지정 성격을 정의할 수도 있습니다.

```yaml
agent:
  personalities:
    codereviewer: >
      당신은 꼼꼼한 코드 리뷰어입니다. 버그, 보안 문제,
      성능 문제, 불분명한 설계 선택을 식별하세요. 정확하고 건설적으로 피드백하세요.
```

그런 다음 다음 명령어로 전환할 수 있습니다:

```text
/personality codereviewer
```

## 권장 워크플로우

강력한 기본 설정은 다음과 같습니다:

1. `~/.hermes/SOUL.md`에 신중하게 작성된 전역 `SOUL.md`를 유지하세요.
2. 프로젝트 지침은 `AGENTS.md`에 넣으세요.
3. 일시적인 모드 전환을 원할 때만 `/personality`를 사용하세요.

이렇게 하면 다음과 같은 이점이 있습니다:
- 안정적인 어조
- 제자리에 있는 프로젝트별 행동
- 필요할 때의 일시적인 통제력

## 성격이 전체 프롬프트와 상호작용하는 방식

높은 수준에서 프롬프트 스택에는 다음이 포함됩니다:
1. **SOUL.md** (에이전트 정체성 — 또는 SOUL.md를 사용할 수 없는 경우 내장된 폴백)
2. 도구 인식 행동 지침
3. 메모리/사용자 컨텍스트
4. 기술(skills) 지침
5. 컨텍스트 파일 (`AGENTS.md`, `.cursorrules`)
6. 타임스탬프
7. 플랫폼별 포매팅 힌트
8. 선택적 시스템 프롬프트 오버레이 (예: `/personality`)

`SOUL.md`는 기반이며 — 다른 모든 것은 그 위에 구축됩니다.

## CLI 외관 vs 대화형 성격

대화형 성격과 CLI 외관은 별개입니다:

- `SOUL.md`, `agent.system_prompt`, `/personality`는 Hermes가 말하는 방식에 영향을 줍니다.
- `display.skin`과 `/skin`은 터미널에서 Hermes가 보이는 모습에 영향을 줍니다.

터미널 외관에 대해서는 [스킨 및 테마](./skins.md)를 참조하세요.
