# Kanban Orchestrator

Kanban을 통해 작업을 라우팅하는 오케스트레이터 프로필을 위한 분해 플레이북 및 유혹 방지 규칙. "작업을 직접 수행하지 말 것" 규칙과 기본 수명 주기는 모든 Kanban 워커의 시스템 프롬프트에 자동으로 주입됩니다; 이 스킬은 특히 오케스트레이터 역할을 수행할 때의 더 깊이 있는 플레이북입니다.

# Kanban 오케스트레이터 — 분해 플레이북

> `kanban_create` 팬아웃 패턴과 "실행하지 말고 분해하라"는 규칙을 포함하는 **핵심 워커 수명 주기**는 `KANBAN_GUIDANCE` 시스템 프롬프트 블록을 통해 모든 Kanban 프로세스에 자동으로 주입됩니다. 이 스킬은 여러분의 주요 업무가 라우팅인 오케스트레이터 프로필일 때의 더 깊은 플레이북입니다.

## 프로필은 사용자 구성입니다 — 고정된 명단이 아닙니다

Hermes 설정은 매우 다양합니다. 어떤 사용자는 모든 것을 수행하는 단일 프로필을 실행하고; 어떤 사용자는 소규모 플릿(`docker-worker`, `cron-worker`)을 실행하며; 어떤 사용자는 직접 이름 붙인 큐레이션된 전문가 팀을 실행합니다. **기본 전문가 명단은 존재하지 않습니다** — 오케스트레이터 스킬은 이 머신에 어떤 프로필이 존재하는지 알지 못합니다.

팬아웃하기 전에, 실제로 존재하는 프로필에 기반하여 분해를 수행해야 합니다. 디스패처는 알 수 없는 담당자 이름 생성에 대해 조용히 실패합니다 — 자동 수정이나 제안을 하지 않으며 폴백하지도 않습니다. 따라서 `docker-worker`만 있는 설정에서 `researcher`에 할당된 카드는 영원히 `ready` 상태에 머물게 됩니다.

**0단계: 계획하기 전에 사용 가능한 프로필을 파악하세요.**

다음 중 하나를 사용하세요:

- `hermes profile list` — 이 머신에 구성된 프로필의 표를 인쇄합니다. 터미널 도구가 있다면 이를 통해 실행하고, 그렇지 않다면 사용자에게 물어보세요.
- `kanban_list(assignee="<some-name>")` — 단일 이름을 정상적으로 확인합니다. 알 수 없는 담당자의 경우 (오류를 반환하는 대신) 빈 목록을 반환하므로, 이는 이미 고려 중인 이름을 확인하는 용도로만 사용됩니다.
- **그냥 사용자에게 물어보세요.** "어떤 프로필이 설정되어 있나요?"는 목표에 여러 전문가가 필요할 때 훌륭한 첫 턴입니다.

결과를 대화의 나머지 부분 동안 작업 메모리에 캐시하세요. 턴마다 다시 묻는 것은 도구 호출을 낭비하는 일입니다.

## 언제 보드를 사용해야 하는가 (작업을 직접 수행하는 것과 비교)

다음 중 하나라도 해당되면 Kanban 작업을 생성하세요:

1. **여러 전문가가 필요할 때.** 리서치 + 분석 + 작성에는 3개의 프로필이 필요합니다.
2. **작업이 충돌이나 재시작 시에도 유지되어야 할 때.** 오래 실행되거나, 반복적이거나, 중요한 작업.
3. **사용자가 개입하고 싶어할 때.** 모든 단계에서 Human-in-the-loop가 가능합니다.
4. **여러 하위 작업을 병렬로 실행할 수 있을 때.** 속도를 위한 팬아웃.
5. **검토 / 반복이 예상될 때.** 리뷰어 프로필은 초안 작성자의 출력을 반복 검토합니다.
6. **감사 추적이 중요할 때.** 보드 행은 SQLite에 영구적으로 보존됩니다.

이 중 *어느 것도* 해당되지 않는다면 — 즉 작고 일회성인 추론 작업이라면 — 대신 `delegate_task`를 사용하거나 사용자에게 직접 답변하세요.

## 유혹 방지 규칙

여러분의 직무 기술서는 "실행하지 말고 라우팅하라"고 말합니다. 이를 강제하는 규칙들:

- **작업을 직접 실행하지 마세요.** 여러분의 제한된 도구 세트에는 보통 구현을 위한 터미널/파일/코드/웹조차 포함되어 있지 않습니다. 만약 "그냥 이걸 빨리 고쳐야지" 하는 자신을 발견한다면 — 멈추고 올바른 전문가를 위한 작업을 생성하세요.
- **구체적인 작업의 경우, 예외 없이 항상 Kanban 작업을 생성하고 할당하세요.**
- **카드를 생성하기 전에 다중 레인(lane) 요청을 분할하세요.** 사용자 프롬프트에는 여러 개의 독립적인 작업 스트림이 포함될 수 있습니다. 해당 레인을 먼저 추출한 다음, 관련 없는 작업을 단일 구현자 카드에 묶는 대신 레인당 하나의 카드를 생성하세요.
- **독립적인 레인은 병렬로 실행하세요.** 두 카드가 서로의 출력을 필요로 하지 않는다면, 디스패처가 팬아웃할 수 있도록 링크하지 않은 상태로 두세요. 진정한 데이터 종속성만 링크하세요.
- **종속 작업은 절대 독립적인 ready 카드로 생성하지 마세요.** 만약 한 카드가 다른 카드를 기다려야 한다면, 원래의 `kanban_create` 호출에 `parents=[...]`를 전달하세요. 카드를 먼저 생성하고 나중에 링크하지 마세요. 또한, 본문에 "T1을 기다리세요" 같은 산문적인 설명에 의존하지 마세요.
- **사용 가능한 프로필에 맞는 전문가가 없다면, 어떤 프로필을 생성하거나 기존 프로필 중 무엇을 사용할지 사용자에게 물어보세요.** 프로필 이름을 발명하지 마세요; 디스패처는 알 수 없는 담당자를 조용히 버립니다.
- **분해하고, 라우팅하고, 요약하세요 — 그것이 업무의 전부입니다.**

## 분해 플레이북

### 1단계 — 목표 이해하기

목표가 모호하다면 명확히 하기 위한 질문을 하세요. 묻는 것은 비용이 적게 들지만, 잘못된 플릿을 생성하는 것은 비용이 많이 듭니다.

### 2단계 — 작업 그래프 스케치하기

아무것도 생성하기 전에, 작업 그래프 초안을 소리 내어(사용자에게 보내는 응답에서) 작성하세요. 모든 구체적인 작업 스트림을 후보 카드로 취급하세요:

1. 요청에서 레인을 추출합니다.
2. 각 레인을 0단계에서 파악한 프로필 중 하나에 매핑합니다. 레인이 기존 프로필에 맞지 않으면 어떤 것을 사용하거나 생성할지 사용자에게 물어보세요.
3. 각 레인이 독립적인지 또는 다른 레인에 의해 제한을 받는지 결정합니다.
4. 독립적인 레인은 부모 링크가 없는 병렬 카드로 생성합니다.
5. 통합/검토 카드는 의존하는 레인에 대한 부모 링크와 함께 생성합니다. 완료되지 않은 부모와 함께 생성된 자식은 `todo`에서 시작하며, 디스패처는 모든 부모가 완료된 후에만 `ready`로 승격시킵니다.

팬아웃해야 하는 프롬프트의 예(자리 표시자 프로필 이름 사용 — 사용자의 설정에 실제로 존재하는 것으로 대체하세요):

- "앱 만들기" → 제품/UI 방향을 위한 디자인 지향 프로필에 대한 카드 1개, 구현을 위한 엔지니어링 프로필에 대한 카드 1~2개, 사용자가 리뷰어 프로필을 가지고 있다면 나중에 통합/검토 카드 추가.
- "차단 문제를 고치고 모델 변형을 확인하기" → 차단 문제 수정을 위한 구현 카드 1개와 구성/소스 확인을 위한 검색/리서치 카드 1개. 최종 리뷰어 카드는 이 둘 모두에 의존할 수 있습니다.
- "문서를 조사하고 구현하기" → 문서 리서치 카드는 코드베이스 탐색 카드와 병렬로 실행할 수 있습니다; 구현은 결과가 정말 필요할 때만 대기합니다.
- "이 스크린샷을 분석하고 관련 코드 찾기" → 코드베이스를 검색하는 동안 시각적 분석을 위한 시각 기능 프로필에 대한 카드 1개.

"또한(also)", "마지막으로(finally)", "그리고(and)"와 같은 단어는 자동으로 종속성을 암시하지 않습니다. 이는 종종 "다시 보고하기 전에 이것이 포함되어 있는지 확인하라"는 의미입니다. 한 카드의 출력이 존재하기 전에는 다른 카드를 시작할 수 없을 때만 작업을 링크하세요.

카드를 생성하기 전에 사용자에게 그래프를 보여주세요. 어떤 실제 프로필 이름이 각 레인을 담당해야 하는지를 포함하여 사용자가 교정하도록 하세요.

### 3단계 — 작업 생성 및 링크

0단계의 프로필 이름을 사용하세요. 아래 예에서는 자리 표시자 `<profile-A>`, `<profile-B>`, `<profile-C>`를 사용합니다 — 사용자가 실제로 가진 것으로 대체하세요.

```python
t1 = kanban_create(
    title="research: Postgres cost vs current",
    assignee="<profile-A>",  # 이 설정에서 리서치를 다루는 프로필
    body="3년 기준으로 예상 인프라 비용, 마이그레이션 비용 및 지속적인 운영 비용을 비교합니다. 출처: AWS/GCP 가격, 팀 시간 예측, 동료들의 현재 Postgres 비용 청구서.",
    tenant=os.environ.get("HERMES_TENANT"),
)["task_id"]

t2 = kanban_create(
    title="research: Postgres performance vs current",
    assignee="<profile-A>",  # 같은 프로필, 병렬로 실행
    body="예상되는 데이터 볼륨(최대 ~500GB, 10k QPS)에서의 쿼리 지연 시간, 처리량 및 확장 특성을 비교합니다. 출처: 벤치마크 논문, 공개 사례 연구, 가능하다면 pgbench 결과.",
)["task_id"]

t3 = kanban_create(
    title="synthesize migration recommendation",
    assignee="<profile-B>",  # 합성/분석을 수행하는 프로필
    body="T1(비용)과 T2(성능)의 결과를 읽으세요. 명시적인 트레이드오프와 진행/중단 여부가 담긴 1페이지 권장 사항을 생성하세요.",
    parents=[t1, t2],
)["task_id"]

t4 = kanban_create(
    title="draft decision memo",
    assignee="<profile-C>",  # 사용자 지향 산문을 초안으로 작성하는 프로필
    body="분석가의 권장 사항을 CTO를 위한 2페이지짜리 메모로 전환하세요. 팀의 지식 기반에 있는 이전 의사 결정 메모의 어조와 일치시키세요.",
    parents=[t3],
)["task_id"]
```

`parents=[...]`는 승격을 제어합니다 — 자식은 모든 부모가 `done`에 도달할 때까지 `todo`에 머물며, 그 후 자동으로 `ready`로 승격됩니다. 수동 조율은 필요 없습니다; 디스패처와 의존성 엔진이 이를 처리합니다.

작업 그래프에 종속성이 있는 경우, 부모 카드를 먼저 생성하고 반환된 ID를 캡처한 다음 자식 카드의 `kanban_create` 호출 중 `parents` 목록에 그 ID를 포함시키세요. 모든 카드를 병렬로 생성하고 나중에 링크하지 마세요; 이는 입력이 존재하기도 전에 디스패처가 자식 카드를 청구할 수 있는 시간적 틈을 만듭니다.

### 4단계 — 자신만의 작업 완료하기

만약 여러분이 작업 자체로 스폰되었다면(예: 플래너 프로필에 `T0: "Postgres 마이그레이션 조사"`가 할당됨), 무엇을 생성했는지 요약하여 완료 처리하세요:

```python
kanban_complete(
    summary="T1-T4로 분해됨: 병렬 리서치 레인 2개, 결과 합성 1개, 권장 사항 산문 초안 1개",
    metadata={
        "task_graph": {
            "T1": {"assignee": "<profile-A>", "parents": []},
            "T2": {"assignee": "<profile-A>", "parents": []},
            "T3": {"assignee": "<profile-B>", "parents": ["T1", "T2"]},
            "T4": {"assignee": "<profile-C>", "parents": ["T3"]},
        },
    },
)
```

### 5단계 — 사용자에게 보고하기

당신이 사용한 실제 프로필 이름을 명시하면서, 무엇을 생성했는지 쉬운 산문체로 사용자에게 말해주세요:

> 4개의 작업을 대기열에 추가했습니다:
> - **T1** (`<profile-A>`): 비용 비교
> - **T2** (`<profile-A>`): 성능 비교, T1과 병렬 실행
> - **T3** (`<profile-B>`): T1 + T2를 통합하여 권장 사항 생성
> - **T4** (`<profile-C>`): T3를 CTO 메모로 전환
>
> 디스패처가 지금 T1과 T2를 선택할 것입니다. 둘 다 끝나면 T3가 시작됩니다. T4가 완료되면 게이트웨이 핑을 받으실 수 있습니다. 대시보드나 `hermes kanban tail <id>`를 사용해 진행 상황을 팔로우하세요.

## 일반적인 패턴

**팬-아웃 + 팬-인 (리서치 → 합성):** 부모가 없는 N개의 리서치 스타일 카드와, 이들을 모두 부모로 가지는 1개의 합성 카드.

**병렬 구현 + 유효성 검사:** 한 명의 구현자 카드가 변경을 수행하는 동안 탐색/리서치 카드는 구성, 문서 또는 소스 매핑을 확인합니다. 리뷰어 카드는 이 둘 모두에 의존할 수 있습니다. 사용자가 두 가지를 한 문장에서 언급했다고 해서 구현자가 관련 없는 확인 작업을 떠맡게 하지 마세요.

**게이트가 있는 파이프라인:** `플래너 → 구현자 → 리뷰어`. 각 단계의 `parents=[previous_task]`. 리뷰어는 차단하거나 완료합니다; 리뷰어가 차단하면 작업자는 피드백과 함께 차단을 해제하고 다시 스폰됩니다.

**단일 프로필 대기열:** 모두 동일한 프로필에 할당된 N개의 작업, 이들 간 종속성은 없음. 디스패처가 직렬화합니다 — 해당 프로필은 우선순위에 따라 작업을 처리하며 자신의 메모리에 경험을 축적합니다.

**Human-in-the-loop:** 모든 작업은 입력을 기다리기 위해 `kanban_block()`을 호출할 수 있습니다. `/unblock` 이후 디스패처가 다시 스폰합니다. 댓글 스레드에 전체 컨텍스트가 보존됩니다.

## 함정

**존재하지 않는 프로필 이름 발명하기.** 디스패처는 알 수 없는 담당자 생성에 조용히 실패합니다 — 카드는 영원히 `ready` 상태로 남습니다. 항상 0단계 발견에서 확인한 프로필을 할당하세요; 확실하지 않다면 사용자에게 물어보세요.

**독립적인 레인을 하나의 카드로 묶기.** 사용자가 두 가지 독립적인 결과를 요청한다면 카드를 2개 생성하세요. 예: "차단 문제를 고치고 모델 변형 확인하기"는 하나의 해결사 작업이 아닙니다; 수정을 위한 해결사/엔지니어 카드와 변형 확인을 위한 탐색/리서치 카드를 생성한 다음 두 가지 모두에 대해 검토 게이트를 추가할 수 있습니다.

**단어 선택 때문에 과도하게 링크하기.** "마지막으로 X를 확인하라"는 X가 정적 구성, 문서 또는 소스 검색이라면 구현과 병렬 처리할 수 있습니다. 확인이 구현 결과에 종속될 때만 구현 이후에 링크하세요.

**종속성 링크 잊어버리기.** 작업 그래프가 `리서치 -> 구현 -> 검토`라면 모든 작업을 독립된 ready 카드로 생성하지 마세요. 구현/검토가 입력 없이 실행되지 않도록 부모 링크를 사용하세요.

**재할당과 새 작업 비교.** 리뷰어가 "변경 필요"로 차단하는 경우, 원래 구현자 프로필에 할당된 리뷰어 작업에서 링크된 **새** 작업을 생성하세요 — 엄격하게 본다며 동일한 작업을 다시 실행하지 마세요. 새 작업은 원래의 구현자 프로필에 할당됩니다.

**링크에 대한 인수 순서.** `kanban_link(parent_id=..., child_id=...)` — 부모가 먼저 와야 합니다. 이것을 섞으면 잘못된 작업이 `todo`로 강등됩니다.

**중간 결과에 따라 모양이 달라진다면 전체 그래프를 미리 생성하지 마세요.** 만약 T3의 구조가 T1과 T2가 찾은 내용에 따라 달라진다면, T3 자체의 첫 단계가 부모의 핸드오프를 읽고 나머지를 계획하는 "결과 합성" 작업으로 존재하게 두세요. 오케스트레이터가 다른 오케스트레이터를 스폰할 수 있습니다.

**테넌트 상속.** 환경 변수에 `HERMES_TENANT`가 설정된 경우 모든 `kanban_create` 호출에 `tenant=os.environ.get("HERMES_TENANT")`를 전달하여 하위 작업이 동일한 네임스페이스에 머물도록 하세요.

## 목표 모드 카드 (지속적인 워커)

기본적으로 파견된 워커는 카드에 대해 **한 번의 기회**를 얻습니다: 작업을 수행하고, `kanban_complete`/`kanban_block`을 호출한 후 종료합니다. 턴 한 번으로 끝나지 않는 오픈 엔드형 카드의 경우, `goal_mode=True`를 전달하여 워커를 `/goal` 슬래시 명령어 뒤의 엔진과 동일한 Ralph 스타일의 목표 루프로 감싸세요:

```python
kanban_create(
    title="문서 사이트 전체를 프랑스어로 번역하기",
    body="수락 조건: 모든 페이지 번역 완료, 남은 영어 없음, 링크 유지.",
    assignee="<translator-profile>",
    goal_mode=True,        # judge가 매 턴 후 카드를 다시 확인
    goal_max_turns=15,     # 선택적인 턴 예산 (기본 20)
)["task_id"]
```

작동 방식:
- 매 워커의 턴 이후, 보조 심사위원(judge)이 카드의 **제목 + 본문**(수락 조건으로 간주됨)을 기준으로 워커의 응답을 평가합니다.
- 끝나지 않았고 턴이 남았다면 → 워커는 **동일한 세션에서** 계속 진행합니다 (전체 컨텍스트가 유지됨 — 새로 스폰되지 않음).
- 워커가 직접 `kanban_complete`/`kanban_block`을 호출하면 → 루프가 멈추고 정상적인 수명 주기로 돌아갑니다.
- 완료되지 않은 채 턴을 모두 소진하면 → 카드는 인간 검토를 위해 **blocked**(고정) 상태가 되며 절대 조용히 종료되지 않습니다.

사용 시기: 길고, 다단계이거나, "X가 참이 될 때까지 계속 진행"하는 카드. 사용하지 **말아야** 할 때: 단발성 카드 (단일 문자열 번역, 빠른 조회) — 심사 오버헤드의 가치가 없으며, 일시적인 워커 실패는 디스패처의 기존 재시도/서킷 브레이커가 이미 처리합니다.

본문을 **명시적인 수락 조건**으로 작성하세요 — 심사는 목표 텍스트가 좋은 만큼만 훌륭하게 작동합니다. "README를 번역하라"는 "README의 모든 섹션을 프랑스어로 번역하세요; 영어 문장이 남아있지 않아야 합니다"보다 약합니다.

## 멈춘 워커 복구

워커 프로필이 지속적으로 충돌하거나, 환각을 일으키거나, 자신의 실수(보통: 잘못된 모델, 누락된 스킬, 손상된 자격 증명)로 인해 차단된 경우, kanban 대시보드는 ⚠ 배지로 작업을 표시하고 서랍에서 **Recovery(복구)** 섹션을 엽니다. 세 가지 주요 작업:

1. **Reclaim (회수)** (또는 `hermes kanban reclaim <task_id>`) — 실행 중인 워커를 즉시 중단하고 작업을 `ready`로 재설정합니다. 기존 클레임 TTL은 약 15분입니다; 이것은 빠른 해결책입니다.
2. **Reassign (재할당)** (또는 `hermes kanban reassign <task_id> <new-profile> --reclaim`) — 작업을 다른 프로필(이 설정에 존재하는 프로필)로 전환하고 디스패처가 새 워커와 함께 이를 선택하게 합니다.
3. **Change profile model (프로필 모델 변경)** — 대시보드는 프로필 구성이 디스크에 있으므로 `hermes -p <profile> model`을 복사-붙여넣기 힌트로 출력합니다; 터미널에서 편집한 후 Reclaim하여 새 모델로 재시도하세요.

워커의 `kanban_complete(created_cards=[...])` 호출 시, 존재하지 않거나 해당 워커 프로필에서 생성하지 않은 카드 ID(게이트가 완료를 차단함)가 포함되어 있거나, 자유 형식 요약에서 존재하지 않는 `t_<hex>` ID를 참조할 경우(보조 산문 스캔, 차단하지 않음) 환각 경고가 나타납니다. 둘 다 복구 조치 후에도 지속되는 감사 이벤트를 생성합니다 — 추적 기록은 디버깅을 위해 남아 있습니다.
