# 체계적 디버깅 (Systematic Debugging)

## 개요

무작위로 시도하는 수정은 시간을 낭비하고 새로운 버그를 만듭니다. 빠른 땜질은 근본적인 문제를 감춥니다.

**핵심 원칙:** 수정을 시도하기 전에 항상 근본 원인을 찾으세요. 증상만을 해결하는 것은 실패입니다.

**이 프로세스의 절차를 어기는 것은 디버깅의 정신을 위반하는 것입니다.**

## 절대 규칙 (The Iron Law)

```
근본 원인 조사 없이 수정을 시도하지 말 것
```

1단계를 완료하지 않았다면 어떠한 수정 제안도 할 수 없습니다.

## 사용 시기

모든 기술적 문제에 대해 사용하세요:
- 테스트 실패
- 프로덕션의 버그
- 예상치 못한 동작
- 성능 문제
- 빌드 실패
- 통합 문제

**다음에 해당할 때 특히 사용하세요:**
- 시간 압박을 받을 때 (긴급 상황은 추측을 부추깁니다)
- "빠르게 고칠 방법 하나"가 명백해 보일 때
- 이미 여러 수정을 시도해 보았을 때
- 이전 수정이 작동하지 않았을 때
- 문제를 완전히 이해하지 못했을 때

**다음에 해당하더라도 건너뛰지 마세요:**
- 문제가 간단해 보일 때 (단순한 버그에도 근본 원인이 있습니다)
- 서두르고 있을 때 (서두르면 무조건 재작업이 보장됩니다)
- 누군가 지금 당장 고치길 원할 때 (체계적으로 접근하는 것이 무작위 시도보다 빠릅니다)

## 4가지 단계

다음 단계로 넘어가기 전에 각 단계를 반드시 완료해야 합니다.

---

## 1단계: 근본 원인 조사 (Root Cause Investigation)

**어떤 수정이라도 시도하기 전에:**

### 1. 에러 메시지 주의 깊게 읽기

- 에러나 경고를 무시하고 넘기지 마세요
- 종종 정확한 해결책이 포함되어 있습니다
- 스택 트레이스(stack traces)를 끝까지 읽으세요
- 줄 번호, 파일 경로, 오류 코드를 기록하세요

**실행:** 관련 소스 파일에 `read_file`을 사용하세요. 코드베이스에서 해당 에러 문자열을 찾으려면 `search_files`를 사용하세요.

### 2. 일관되게 재현하기

- 신뢰할 수 있게 오류를 발생시킬 수 있습니까?
- 정확한 단계가 무엇입니까?
- 매번 발생합니까?
- 재현할 수 없다면 → 추측하지 말고 더 많은 데이터를 수집하세요

**실행:** `terminal` 도구를 사용하여 실패하는 테스트를 실행하거나 버그를 트리거하세요:

```bash
# 특정 실패 테스트 실행
pytest tests/test_module.py::test_name -v

# 자세한 출력(verbose)으로 실행
pytest tests/test_module.py -v --tb=long
```

### 3. 최근 변경 사항 확인

- 이것을 유발할 만한 어떤 변경이 있었습니까?
- Git diff, 최근 커밋들
- 새로운 종속성(dependencies), 구성(config) 변경 사항

**실행:**

```bash
# 최근 커밋 확인
git log --oneline -10

# 커밋되지 않은 변경 사항
git diff

# 특정 파일의 변경 사항
git log -p --follow src/problematic_file.py | head -100
```

### 4. 다중 컴포넌트 시스템에서 증거 수집

**시스템에 여러 컴포넌트(API → 서비스 → 데이터베이스, CI → 빌드 → 배포)가 있는 경우:**

**수정을 제안하기 전에 진단용 계측(instrumentation)을 추가하세요:**

각 컴포넌트 경계에 대해:
- 컴포넌트로 들어가는 데이터 로깅
- 컴포넌트에서 나가는 데이터 로깅
- 환경/구성 전파 확인
- 각 계층에서의 상태(state) 확인

증거를 수집하기 위해 한 번 실행하여 어디에서 문제가 발생하는지 확인하세요.
그다음 증거를 분석하여 실패하는 컴포넌트를 식별하세요.
그런 다음 해당 특정 컴포넌트를 조사하세요.

### 5. 데이터 흐름 추적

**에러가 콜 스택 깊은 곳에 있는 경우:**

- 잘못된 값이 어디서 생성되었습니까?
- 이 함수를 잘못된 값으로 호출한 것은 무엇입니까?
- 근원을 찾을 때까지 상위(upstream)로 계속 추적하세요
- 증상이 아닌 원천(source)에서 수정하세요

**실행:** `search_files`를 사용하여 참조를 추적하세요:

```python
# 함수가 어디서 호출되는지 찾기
search_files("function_name(", path="src/", file_glob="*.py")

# 변수가 할당되는 곳 찾기
search_files("variable_name\\s*=", path="src/", file_glob="*.py")
```

### 1단계 완료 체크리스트

- [ ] 에러 메시지를 완전히 읽고 이해했음
- [ ] 문제를 일관되게 재현했음
- [ ] 최근 변경 사항을 식별하고 검토했음
- [ ] 증거를 수집했음 (로그, 상태, 데이터 흐름)
- [ ] 문제가 특정 컴포넌트/코드로 격리되었음
- [ ] 근본 원인 가설을 수립했음

**정지 (STOP):** 왜 발생했는지 이해할 때까지 2단계로 넘어가지 마세요.

---

## 2단계: 패턴 분석 (Pattern Analysis)

**수정하기 전에 패턴을 찾으세요:**

### 1. 작동하는 예시 찾기

- 동일한 코드베이스에서 작동하는 유사한 코드를 찾으세요
- 깨진 것과 비슷하지만 잘 작동하는 것은 무엇입니까?

**실행:** `search_files`를 사용하여 비교 가능한 패턴을 찾으세요:

```python
search_files("similar_pattern", path="src/", file_glob="*.py")
```

### 2. 참조 대상(References)과 비교하기

- 패턴을 구현하는 경우, 참조할 구현 코드를 끝까지 완벽하게 읽으세요
- 훑어보지 마세요 — 한 줄 한 줄 읽으세요
- 적용하기 전에 패턴을 완전히 이해하세요

### 3. 차이점 식별하기

- 작동하는 것과 깨진 것의 차이는 무엇입니까?
- 작든 크든 모든 차이점을 나열하세요
- "그건 중요하지 않을 거야"라고 속단하지 마세요

### 4. 종속성(Dependencies) 이해하기

- 이것이 작동하려면 어떤 다른 컴포넌트들이 필요합니까?
- 어떤 설정, 구성, 환경이 필요합니까?
- 어떤 가정(assumptions)을 하고 있습니까?

---

## 3단계: 가설 및 테스트 (Hypothesis and Testing)

**과학적 방법 (Scientific method):**

### 1. 단일 가설 수립

- 명확하게 명시하세요: "Y 때문에 X가 근본 원인이라고 생각합니다"
- 기록해두세요
- 모호하지 않게 구체적으로 작성하세요

### 2. 최소한으로 테스트하기

- 가설을 테스트하기 위해 가능한 가장 작은 변경을 하세요
- 한 번에 하나의 변수만 변경하세요
- 여러 가지를 동시에 고치지 마세요

### 3. 진행하기 전에 검증하기

- 작동했습니까? → 4단계로 진행
- 작동하지 않았습니까? → 새로운 가설 수립
- 해결되지 않았는데 그 위에 추가 수정 사항을 더하지 마세요

### 4. 모를 때 대처법

- "저는 X를 이해하지 못합니다"라고 말하세요
- 아는 척하지 마세요
- 사용자에게 도움을 요청하세요
- 더 많은 조사를 하세요

---

## 4단계: 구현 (Implementation)

**증상이 아닌 근본 원인을 수정하세요:**

### 1. 실패하는 테스트 케이스 작성하기

- 가능한 한 가장 단순한 재현
- 가능하면 자동화된 테스트로 작성
- 수정하기 전에 반드시 가지고 있어야 함
- `test-driven-development` 스킬을 사용하세요

### 2. 단일 수정 구현하기

- 식별된 근본 원인 해결
- 한 번에 하나의 변경
- "하는 김에 같이" 하는 개선 작업 금지
- 리팩토링을 묶어서 함께 하지 마세요

### 3. 수정 사항 검증하기

```bash
# 특정 회귀(regression) 테스트 실행
pytest tests/test_module.py::test_regression -v

# 전체 테스트 실행 — 회귀 없음 확인
pytest tests/ -q
```

### 4. 수정이 작동하지 않을 경우 — '3의 법칙' (The Rule of Three)

- **정지 (STOP).**
- 세어보세요: 얼마나 많은 수정을 시도했습니까?
- 만약 3회 미만이라면: 1단계로 돌아가서 새로운 정보로 재분석하세요
- **만약 3회 이상이라면: 정지하고 아키텍처에 의문을 제기하세요 (아래 5단계)**
- 아키텍처에 대한 논의 없이 4번째 수정을 시도하지 마세요

### 5. 3번 이상의 수정이 실패한 경우: 아키텍처에 의문 제기하기

**아키텍처적 문제가 있음을 나타내는 패턴:**
- 수정을 시도할 때마다 계속 다른 곳에서 새로운 공유 상태/결합성(coupling)이 드러남
- 수정을 구현하기 위해 "대규모 리팩토링"이 필요함
- 각 수정이 다른 곳에서 새로운 증상을 만들어 냄

**정지하고 근본적인 문제에 의문을 제기하세요:**
- 이 패턴이 근본적으로 건전한가?
- 그저 관성만으로 이것을 고집하고 있는가?
- 계속해서 증상을 고치기보단 아키텍처를 리팩토링해야 하지 않을까?

**더 많은 수정을 시도하기 전에 사용자와 상의하세요.**

이것은 실패한 가설이 아닙니다 — 이것은 잘못된 아키텍처입니다.

---

## 적신호 (Red Flags) — 정지하고 프로세스를 따를 것

다음과 같은 생각이 들기 시작한다면:
- "일단 빨리 고치고, 조사는 나중에 하자"
- "그냥 X를 바꿔보고 작동하는지 보자"
- "여러 가지를 한꺼번에 변경하고 테스트 돌려보자"
- "테스트 코드는 건너뛰고 내가 직접 확인해볼게"
- "아마도 X일 테니까 그것부터 고쳐볼게"
- "완전히 이해하진 못했지만 이렇게 하면 작동할지도 몰라"
- "패턴은 X라고 하지만, 난 다르게 맞춰서 적용할게"
- "주요 문제는 다음과 같습니다: [조사 없이 수정 제안 나열]"
- 데이터 흐름 추적 없이 해결책 제안하기
- **"한 번만 더 고쳐보자" (이미 2번 이상 시도했는데도 실패했을 때)**
- **고칠 때마다 계속해서 다른 곳에서 새로운 문제가 나타날 때**

**이 모든 것은 다음을 의미합니다: 정지 (STOP). 1단계로 돌아가세요.**

**만약 3번 이상의 수정 시도가 실패했다면:** 아키텍처에 의문을 제기하세요 (4단계의 5번째 항목).

## 흔한 합리화

| 변명 | 현실 |
|--------|---------|
| "문제는 간단해, 프로세스는 필요 없어" | 단순한 문제에도 근본 원인이 있습니다. 단순한 버그는 프로세스를 따르면 금방 해결됩니다. |
| "긴급 상황이야, 프로세스를 따를 시간 없어" | 체계적인 디버깅이 추측 기반의 무작위 시도보다 빠릅니다. |
| "일단 이거 먼저 시도해보고 조사하자" | 첫 번째 수정이 이후 패턴을 결정합니다. 처음부터 올바르게 하세요. |
| "수정 사항이 작동하는지 확인한 후 나중에 테스트를 쓸게" | 검증되지 않은 수정은 지속되지 않습니다. 테스트를 먼저 쓰는 것이 이를 증명합니다. |
| "한꺼번에 여러 개를 고치면 시간 절약됨" | 무엇이 해결책이었는지 분리할 수 없습니다. 새로운 버그의 원인이 됩니다. |
| "참고용 코드가 너무 길어, 내 맘대로 맞춰 쓸게" | 부분적인 이해는 버그를 보장합니다. 끝까지 완벽하게 읽으세요. |
| "문제가 보이니까 바로 수정할게" | 증상을 보는 것과 근본 원인을 이해하는 것은 다릅니다. |
| "한 번만 더 고쳐보자" (2번 실패 후) | 3번 이상의 실패는 구조적 문제입니다. 또 고치려고 하지 말고 패턴 자체에 의문을 가지세요. |

## 빠른 참조 가이드

| 단계 | 주요 활동 | 성공 기준 |
|-------|---------------|------------------|
| **1. 근본 원인** | 에러 읽기, 재현하기, 변경 사항 확인, 증거 수집, 데이터 흐름 추적 | 무언지, 왜 일어나는지 이해함 |
| **2. 패턴** | 작동하는 예시 찾기, 비교하기, 차이점 식별하기 | 무엇이 다른지 파악함 |
| **3. 가설** | 이론 수립, 최소한으로 테스트, 한 번에 변수 하나씩 | 가설이 확인되거나 새로운 가설 수립됨 |
| **4. 구현** | 회귀 테스트 생성, 근본 원인 수정, 검증하기 | 버그가 해결되고 모든 테스트 통과함 |

## Hermes 에이전트 통합

### 조사 도구

1단계에서 다음 Hermes 도구들을 사용하세요:

- **`search_files`** — 오류 문자열 찾기, 함수 호출 추적, 패턴 찾기
- **`read_file`** — 정밀한 분석을 위해 줄 번호가 포함된 소스 코드 읽기
- **`terminal`** — 테스트 실행, Git 히스토리 확인, 버그 재현
- **`web_search`/`web_extract`** — 에러 메시지, 라이브러리 문서 조사

### delegate_task와 함께 사용

다중 컴포넌트 복합 디버깅의 경우, 조사 서브에이전트를 파견하세요:

```python
delegate_task(
    goal="[특정 테스트/기능]이 왜 실패하는지 조사하세요",
    context="""
    systematic-debugging 스킬의 과정을 따르세요:
    1. 에러 메시지를 주의 깊게 읽습니다.
    2. 문제를 재현합니다.
    3. 데이터 흐름을 추적하여 근본 원인을 찾습니다.
    4. 결과를 보고합니다 — 아직 수정하지 마세요.

    오류: [전체 오류 붙여넣기]
    파일: [실패하는 코드의 경로]
    테스트 명령: [정확한 명령]
    """,
    toolsets=['terminal', 'file']
)
```

### test-driven-development와 함께 사용

버그를 수정할 때:
1. 버그를 재현하는 테스트를 작성합니다 (RED)
2. 체계적으로 디버깅하여 근본 원인을 찾습니다
3. 근본 원인을 수정합니다 (GREEN)
4. 이 테스트가 해결책을 증명하고 회귀(regression)를 방지합니다

## 실제 도입 효과

실제 디버깅 세션 결과:
- 체계적 접근 방식: 해결하는 데 15~30분 소요
- 무작위 수정 방식: 2~3시간의 방황
- 첫 시도 성공률: 95% vs 40%
- 새롭게 만들어지는 버그: 0건에 가까움 vs 흔히 발생
