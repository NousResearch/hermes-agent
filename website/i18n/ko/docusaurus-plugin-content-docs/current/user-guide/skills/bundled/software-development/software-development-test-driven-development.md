# 테스트 주도 개발 (Test-Driven Development, TDD)

## 개요

먼저 테스트를 작성하세요. 실패하는 것을 지켜보세요. 그런 다음 통과할 수 있는 최소한의 코드를 작성하세요.

**핵심 원칙:** 테스트가 실패하는 것을 눈으로 직접 보지 않았다면, 올바른 것을 테스트하고 있는지 알 수 없습니다.

**규칙의 문구를 어기는 것은 규칙의 정신을 위반하는 것입니다.**

## 사용 시기

**항상:**
- 새로운 기능
- 버그 수정
- 리팩토링
- 동작 변경

**예외 (사용자에게 먼저 물어보세요):**
- 버려질 프로토타입
- 생성된 코드
- 구성 파일(Configuration files)

"이번 한 번만 TDD를 건너뛸까?"라고 생각하시나요? 멈추세요. 그것은 합리화입니다.

## 절대 법칙 (The Iron Law)

```
실패하는 테스트를 먼저 작성하지 않고는 프로덕션 코드를 작성하지 마세요
```

테스트보다 코드를 먼저 작성했나요? 삭제하세요. 처음부터 다시 시작하세요.

**예외 없음:**
- "참고용"으로 보관하지 마세요
- 테스트를 작성하면서 "수정"하지 마세요
- 쳐다보지도 마세요
- 삭제한다는 것은 정말로 삭제한다는 뜻입니다

테스트를 기반으로 처음부터 다시 구현하세요. 예외는 없습니다.

## 레드-그린-리팩터(Red-Green-Refactor) 사이클

### RED — 실패하는 테스트 작성

어떤 일이 일어나야 하는지 보여주는 최소한의 테스트 하나를 작성하세요.

**좋은 테스트:**
```python
def test_retries_failed_operations_3_times():
    attempts = 0
    def operation():
        nonlocal attempts
        attempts += 1
        if attempts < 3:
            raise Exception('fail')
        return 'success'

    result = retry_operation(operation)

    assert result == 'success'
    assert attempts == 3
```
명확한 이름, 실제 동작을 테스트, 하나의 책임.

**나쁜 테스트:**
```python
def test_retry_works():
    mock = MagicMock()
    mock.side_effect = [Exception(), Exception(), 'success']
    result = retry_operation(mock)
    assert result == 'success'  # 재시도 횟수는? 타이밍은?
```
모호한 이름, 실제 코드가 아닌 모의 객체(mock) 테스트.

**요구 사항:**
- 테스트당 하나의 동작
- 명확하고 설명적인 이름 (이름에 "and"가 있나요? 분리하세요)
- 정말 피할 수 없는 경우를 제외하고는 모의 객체가 아닌 실제 코드 사용
- 구현이 아닌 동작을 설명하는 이름 지정

### Verify RED — 실패하는 것을 확인

**필수. 절대 건너뛰지 마세요.**

```bash
# 터미널 도구를 사용하여 특정 테스트 실행
pytest tests/test_feature.py::test_specific_behavior -v
```

다음을 확인하세요:
- 오타로 인한 오류가 아니라, 테스트가 정말로 실패함
- 실패 메시지가 예상과 일치함
- 기능이 누락되었기 때문에 실패함

**테스트가 바로 통과하나요?** 기존 동작을 테스트하고 있는 것입니다. 테스트를 고치세요.

**테스트에서 에러가 발생하나요?** 에러를 고치고, 제대로 실패할 때까지 다시 실행하세요.

### GREEN — 최소한의 코드

테스트를 통과하기 위한 가장 단순한 코드를 작성하세요. 그 이상은 작성하지 마세요.

**좋음:**
```python
def add(a, b):
    return a + b  # 그 어떤 추가 내용도 없음
```

**나쁨:**
```python
def add(a, b):
    result = a + b
    logging.info(f"Adding {a} + {b} = {result}")  # 추가적인 내용!
    return result
```

기능을 추가하거나, 다른 코드를 리팩토링하거나, 테스트 이상의 것을 "개선"하지 마세요.

**GREEN 단계에서는 편법도 허용됩니다:**
- 반환 값 하드코딩
- 복사 및 붙여넣기
- 코드 중복
- 예외 사례(edge cases) 무시

이는 REFACTOR 단계에서 고칠 것입니다.

### Verify GREEN — 통과하는 것을 확인

**필수.**

```bash
# 특정 테스트 실행
pytest tests/test_feature.py::test_specific_behavior -v

# 그런 다음 모든 테스트를 실행하여 회귀(regression) 확인
pytest tests/ -q
```

다음을 확인하세요:
- 해당 테스트가 통과함
- 다른 테스트들도 여전히 통과함
- 출력이 깔끔함 (에러나 경고 없음)

**테스트가 실패하나요?** 테스트가 아니라 코드를 고치세요.

**다른 테스트들이 실패하나요?** 발생한 회귀를 지금 고치세요.

### REFACTOR — 정리하기

테스트가 통과한 후에만 수행합니다:
- 중복 제거
- 이름 개선
- 헬퍼 함수 추출
- 복잡한 표현식 단순화

리팩토링 과정 내내 테스트가 통과(Green)하도록 유지하세요. 동작을 추가하지 마세요.

**리팩토링 중에 테스트가 실패한다면:** 즉시 실행을 취소(Undo)하세요. 더 작은 단계로 진행하세요.

### 반복 (Repeat)

다음 동작을 위한 다음 실패 테스트를 작성하세요. 한 번에 하나의 사이클만 진행합니다.

## 왜 순서가 중요한가

**"이게 잘 작동하는지 확인하기 위해 나중에 테스트를 작성할게"**

코드 작성 후에 만들어진 테스트는 즉시 통과합니다. 즉시 통과하는 테스트는 아무것도 증명하지 못합니다:
- 잘못된 것을 테스트하고 있을 수 있습니다.
- 동작이 아닌 구현 자체를 테스트하고 있을 수 있습니다.
- 당신이 잊어버린 예외 사례(edge cases)를 놓칠 수 있습니다.
- 이 테스트가 실제로 버그를 잡는 것을 본 적이 없습니다.

테스트를 먼저 작성하면 테스트가 실패하는 것을 강제로 보게 되며, 이는 테스트가 실제로 어떤 것을 검증하고 있음을 증명합니다.

**"난 이미 예외 사례들을 전부 수동으로 테스트했어"**

수동 테스트는 임시방편입니다. 모든 것을 테스트했다고 생각하겠지만:
- 무엇을 테스트했는지 기록이 없습니다.
- 코드가 변경될 때 다시 실행할 수 없습니다.
- 압박감 속에서는 케이스를 잊어버리기 쉽습니다.
- "내가 해봤을 땐 됐어" ≠ 포괄적임

자동화된 테스트는 체계적입니다. 매번 동일한 방식으로 실행됩니다.

**"X 시간 들여 만든 작업을 지우는 건 낭비야"**

매몰 비용의 오류입니다. 그 시간은 이미 지나갔습니다. 이제 당신의 선택은:
- 삭제하고 TDD로 다시 작성하기 (높은 신뢰도)
- 코드를 유지하고 나중에 테스트 추가하기 (낮은 신뢰도, 버그 가능성 높음)

진짜 "낭비"는 믿을 수 없는 코드를 유지하는 것입니다.

**"TDD는 교조적이야, 실용적이려면 상황에 맞춰 적응해야 해"**

TDD가 바로 실용적인 방법입니다:
- 커밋 전에 버그를 찾아냅니다 (나중에 디버깅하는 것보다 빠름).
- 회귀(regression)를 방지합니다 (테스트가 즉시 문제를 잡아냄).
- 동작을 문서화합니다 (테스트가 코드 사용법을 보여줌).
- 리팩토링을 가능하게 합니다 (자유롭게 변경해도 테스트가 문제를 잡아줌).

"실용적"이라는 핑계의 지름길은 결국 프로덕션에서의 디버깅을 뜻하며, 더 느려지게 만듭니다.

**"코딩 후 테스트도 같은 목표를 달성할 수 있어 — 중요한 건 의식(ritual)이 아니라 정신(spirit)이야"**

아닙니다. 나중에 하는 테스트는 "이게 무엇을 하는가?"에 답합니다. 테스트 주도는 "이게 무엇을 해야 하는가?"에 답합니다.

나중에 하는 테스트는 당신의 구현 방식에 편향되어 있습니다. 요구사항이 아니라 당신이 이미 만들어놓은 것을 테스트하게 됩니다. 테스트를 먼저 작성하면 구현하기 전에 강제로 예외 사례들을 발견하게 됩니다.

## 흔한 합리화

| 핑계 | 현실 |
|--------|---------|
| "테스트하기엔 너무 단순해" | 단순한 코드도 깨집니다. 테스트 작성은 30초면 됩니다. |
| "나중에 테스트할게" | 즉시 통과하는 테스트는 아무것도 증명하지 못합니다. |
| "코딩 후 테스트도 목표는 같아" | 코딩 후 테스트 = "이게 뭘 하는가?", 테스트 먼저 = "이게 뭘 해야 하는가?" |
| "이미 수동으로 다 해봤어" | 임시방편은 체계적인 테스트가 아닙니다. 기록도 없고 다시 실행할 수도 없습니다. |
| "X시간의 작업을 지우는 건 낭비야" | 매몰 비용의 오류. 검증되지 않은 코드를 유지하는 것이 기술 부채입니다. |
| "참고용으로 보관하고 테스트 먼저 작성할게" | 결국 기존 코드에 맞춰서 테스트를 짜게 됩니다. 삭제하라는 건 정말 삭제하라는 뜻입니다. |
| "일단 탐색이 필요해" | 좋습니다. 탐색 코드를 버리고 TDD로 다시 시작하세요. |
| "테스트하기가 너무 어려워 = 설계가 불명확해" | 테스트에 귀를 기울이세요. 테스트하기 어렵다는 건 사용하기도 어렵다는 뜻입니다. |
| "TDD는 속도를 늦출 거야" | 디버깅하는 것보다 TDD가 빠릅니다. 진정한 실용주의 = 테스트 먼저. |
| "수동 테스트가 더 빨라" | 수동 테스트는 예외 사례를 증명하지 못합니다. 변경될 때마다 다시 테스트해야 할 것입니다. |
| "기존 코드에 테스트가 없어" | 당신이 그것을 개선하고 있습니다. 당신이 손대는 코드에 대한 테스트를 추가하세요. |

## 적신호 (Red Flags) — 멈추고 다시 시작하세요

자신이 다음 중 하나를 하고 있다면 코드를 삭제하고 TDD로 다시 시작하세요:

- 테스트보다 먼저 코드 작성
- 구현 후 테스트 작성
- 테스트를 처음 실행하자마자 통과
- 테스트가 왜 실패했는지 설명할 수 없음
- "나중에" 추가되는 테스트
- "이번 한 번만"이라며 합리화
- "내가 이미 수동으로 테스트했어"
- "나중에 작성하는 테스트도 목적은 같아"
- "참고용으로 남겨두기" 또는 "기존 코드를 응용하기"
- "이미 X시간을 보냈으니 삭제하는 건 낭비야"
- "TDD는 교조적이야, 난 실용적으로 하는 중"
- "이 경우는 좀 다르기 때문에..."

**이 모든 것은 다음을 의미합니다: 코드를 삭제하세요. TDD로 다시 시작하세요.**

## 검증 체크리스트

작업을 완료로 표시하기 전에:

- [ ] 모든 새로운 함수/메서드에 테스트가 있습니다.
- [ ] 코드를 구현하기 전에 각 테스트가 실패하는 것을 눈으로 확인했습니다.
- [ ] 각 테스트가 오타가 아닌, 예상된 이유(기능 누락)로 실패했습니다.
- [ ] 테스트를 통과하기 위한 최소한의 코드만 작성했습니다.
- [ ] 모든 테스트가 통과합니다.
- [ ] 출력이 깨끗합니다 (에러나 경고 없음).
- [ ] 모의 객체(mock)가 불가피한 경우를 제외하고는 테스트가 실제 코드를 사용합니다.
- [ ] 예외 사례(edge cases)와 오류들이 테스트에 포함되었습니다.

모든 항목에 체크할 수 없나요? 당신은 TDD를 건너뛰었습니다. 다시 시작하세요.

## 막혔을 때

| 문제 | 해결책 |
|---------|----------|
| 테스트 방법을 모르겠음 | 당신이 원했던 API를 그대로 작성하세요. 단언(assertion)부터 작성하세요. 사용자에게 물어보세요. |
| 테스트가 너무 복잡함 | 설계가 너무 복잡한 것입니다. 인터페이스를 단순화하세요. |
| 모든 것을 모의 객체(mock)로 만들어야 함 | 코드가 너무 강하게 결합되어 있습니다. 의존성 주입(dependency injection)을 사용하세요. |
| 테스트 설정이 너무 긺 | 헬퍼(helper)를 추출하세요. 그래도 복잡하다고요? 설계를 단순화하세요. |

## Hermes 에이전트 통합

### 테스트 실행

각 단계에서 `terminal` 도구를 사용하여 테스트를 실행하세요:

```python
# RED — 실패 검증
terminal("pytest tests/test_feature.py::test_name -v")

# GREEN — 통과 검증
terminal("pytest tests/test_feature.py::test_name -v")

# 전체 스위트 — 회귀(regression) 없음을 검증
terminal("pytest tests/ -q")
```

### delegate_task와 함께 사용

구현을 위해 서브에이전트를 파견할 때 목표(goal)에 TDD를 강제하세요:

```python
delegate_task(
    goal="엄격한 TDD를 사용하여 [기능]을 구현하세요",
    context="""
    test-driven-development 스킬을 따르세요:
    1. 실패하는 테스트를 가장 먼저 작성
    2. 테스트를 실행하여 실패하는지 확인
    3. 통과를 위한 최소한의 코드 작성
    4. 테스트를 실행하여 통과하는지 확인
    5. 필요한 경우 리팩토링
    6. 커밋

    프로젝트 테스트 명령어: pytest tests/ -q
    프로젝트 구조: [관련 파일들 설명]
    """,
    toolsets=['terminal', 'file']
)
```

### systematic-debugging과 함께 사용

버그를 찾았나요? 버그를 재현하는 실패하는 테스트를 작성하세요. 그 후 TDD 사이클을 따르세요. 테스트는 수정 사항을 증명하고 회귀(regression)를 방지합니다.

절대 테스트 없이 버그를 수정하지 마세요.

## 테스트 안티 패턴

- **실제 동작 대신 모의 객체(mock) 동작 테스트** — 모의 객체는 시스템을 대체하는 것이 아니라 상호작용을 검증해야 합니다.
- **구현 세부 정보 테스트** — 내부 메서드 호출 여부가 아니라 동작/결과를 테스트하세요.
- **정상적인 흐름(Happy path)만 테스트** — 항상 예외 사례, 오류 및 경계값을 테스트하세요.
- **깨지기 쉬운(Brittle) 테스트** — 테스트는 구조가 아니라 동작을 검증해야 합니다. 리팩토링으로 인해 테스트가 깨져서는 안 됩니다.

## 최종 규칙

```
프로덕션 코드 → 테스트가 존재하며 가장 먼저 실패했음
그렇지 않은 경우 → TDD가 아님
```

사용자의 명시적인 허가 없이는 절대 예외가 없습니다.
