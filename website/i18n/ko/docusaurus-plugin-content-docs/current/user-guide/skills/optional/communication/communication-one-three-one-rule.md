# 1-3-1 규칙 (One Three One Rule)

기술적 제안 및 트레이드오프 분석을 위한 구조화된 의사결정 프레임워크입니다. 사용자가 여러 접근 방식(아키텍처 결정, 도구 선택, 리팩토링 전략, 마이그레이션 경로 등) 중에서 선택해야 할 때, 이 스킬은 하나의 명확한 문제 설명, 장단점이 있는 세 가지 구별되는 옵션, 그리고 완료 조건(Definition of Done) 및 구현 계획이 포함된 하나의 구체적인 권장 사항을 포함하는 1-3-1 형식을 생성합니다. 사용자가 "1-3-1"을 요청하거나, "옵션을 줘"라고 말하거나, 경쟁하는 접근 방식 중에서 선택하는 데 도움이 필요할 때 사용하세요.

# 1-3-1 커뮤니케이션 규칙 (1-3-1 Communication Rule)

작업에 실현 가능한 접근 방식이 여러 개 있고 사용자에게 명확한 권장 사항이 필요할 때 사용하는 구조화된 의사결정 형식입니다. 간결한 문제 정의, 트레이드오프가 있는 세 가지 옵션, 그리고 권장하는 경로에 대한 실행 가능한 계획을 제시합니다.

## 사용 시기

- 사용자가 명시적으로 "1-3-1" 형식의 응답을 요청할 때.
- 사용자가 기술적 결정에 대해 "옵션을 줘" 또는 "어떤 선택지가 있나"라고 물어볼 때.
- 작업에 의미 있는 트레이드오프가 있는 여러 실현 가능한 접근 방식(아키텍처, 도구, 마이그레이션 전략 등)이 존재할 때.
- 사용자가 팀이나 이해관계자에게 전달할 수 있는 제안서가 필요할 때.

명백한 답이 하나인 간단한 질문, 디버깅 세션, 또는 사용자가 이미 접근 방식을 결정한 작업에는 사용하지 마십시오.

## 절차

1. **문제 (하나의 문장)**
   - 핵심적인 결정 사항이나 원하는 결과를 단일하고 간결한 문장으로 명시합니다.
   - 구체적인 구현 세부 정보, 도구 이름 또는 기술 등 *어떻게(how)*가 아닌 *무엇(what)*에 초점을 맞춥니다.
   - 간결하게 유지하세요. "그리고(and)"가 필요하다면 두 가지 문제를 설명하고 있는 것입니다.

2. **옵션 (정확히 세 가지)**
   - A, B, C로 레이블이 지정된 세 가지의 구별되고 실현 가능한 접근 방식을 제시합니다.
   - 각 옵션은 간단한 설명, 장점(Pros), 단점(Cons)을 포함해야 합니다.
   - 옵션은 같은 접근 방식의 사소한 변형이 아닌, 진정으로 다른 전략을 나타내야 합니다.

3. **권장 사항 (하나의 옵션)**
   - 사용자의 맥락과 우선순위에 기반하여 어떤 옵션을 추천하는지와 그 이유를 명시합니다.
   - 단도직입적으로 말하세요 — 이것은 피하는 것이 아니라 여러분의 전문적인 판단입니다.

4. **완료 조건 (Definition of Done)**
   - 권장 옵션에 대한 구체적인 성공 기준을 나열합니다.
   - 막연한 포부가 아닌 구체적이고 검증 가능한 결과여야 합니다.
   - 사용자가 다른 옵션을 선택하면 해당 옵션에 맞게 이 섹션을 수정합니다.

5. **구현 계획 (Implementation Plan)**
   - 권장 옵션을 실행하기 위한 구체적인 단계입니다.
   - 해당되는 경우 특정 명령어, 도구 또는 작업을 포함합니다.
   - 사용자가 다른 옵션을 선택하면 해당 옵션에 맞게 이 섹션을 수정합니다.

## 검증 (Verification)

- 응답에 정확히 하나의 문제(Problem) 문장이 있습니다.
- 응답에 장단점이 포함된 정확히 세 가지 옵션(Options) (A, B, C)이 있습니다.
- 응답에 이유와 함께 하나의 옵션을 선택하는 단일 권장 사항(Recommendation)이 있습니다.
- 완료 조건(Definition of Done) 및 구현 계획(Implementation Plan)이 권장 옵션과 일치합니다.
- 사용자가 다른 옵션을 선택하는 경우 권장 사항, 완료 조건, 구현 계획이 그에 따라 업데이트됩니다.

## 예시

사용자: "우리 API 클라이언트에 재시도 로직을 추가하기 위한 1-3-1을 줘."

에이전트:

**문제 (Problem):** 외부 서비스에 대한 API 호출이 부하가 걸릴 때 간헐적으로 실패하므로 원활한 재시도 처리가 필요합니다.

**옵션 (Options):**

- **옵션 A: 공유 유틸리티에서 지터(jitter)가 포함된 지수 백오프(Exponential backoff) 사용.**
  장점: 단순하고 잘 알려진 패턴; 모든 호출자가 공유하는 단일 구현.
  단점: 서킷 브레이킹 없음; 재시도 중에도 호출자는 여전히 차단됨.

- **옵션 B: 각 엔드포인트 호출에 데코레이터로 재시도 라이브러리(tenacity) 사용.**
  장점: 엔드포인트별로 구성 가능; 서킷 브레이커 및 속도 제한에 대한 내장 지원.
  단점: 종속성 추가; 데코레이터 패턴이 제어 흐름을 모호하게 만들 수 있음.

- **옵션 C: 데드 레터(dead letter) 처리가 포함된 큐(Queue) 기반 비동기 재시도.**
  장점: 차단되지 않음(Non-blocking); 실패한 호출이 추적되고 재생 가능; 확장에 유리함.
  단점: 복잡도 크게 증가; 큐 백엔드 필요; 호출량이 적은 서비스에는 과잉(overkill).

**권장 사항 (Recommendation):** 옵션 A. 호출량이 큐 인프라를 구축할 만큼 많지 않으며, 지수 백오프와 지터가 포함된 공유 유틸리티는 새로운 종속성 없이 실패 모드를 깔끔하게 커버합니다.

**완료 조건 (Definition of Done):**
- `api_retry()` 유틸리티가 일시적인 HTTP 오류(429, 502, 503, 504)를 처리합니다.
- 지수 백오프와 지터를 사용하여 최대 3회 재시도합니다.
- 모든 외부 API 호출이 재시도 유틸리티를 통해 라우팅됩니다.
- 각 재시도 시도에 대해 엔드포인트와 시도 횟수를 로깅합니다.
- 테스트가 재시도 소진, 즉각적인 성공, 그리고 일시적 오류 후 성공 시나리오를 커버합니다.

**구현 계획 (Implementation Plan):**
1. 구성 가능한 최대 재시도 횟수, 기본 지연 시간, 재시도 가능한 상태 코드를 포함하는 `utils/api_retry.py`를 생성합니다.
2. 쏠림 현상(thundering herd)을 방지하기 위해 `random.uniform(0, base_delay)`를 사용하여 지터를 추가합니다.
3. `api_client.py`의 기존 API 호출을 재시도 유틸리티로 래핑합니다.
4. 각 재시도 시나리오에 대한 HTTP 응답을 모의(mocking)하는 단위 테스트를 추가합니다.
5. 불안정한 엔드포인트 모의 객체를 대상으로 간단한 스트레스 테스트를 통해 부하 상태에서 검증합니다.
