# Adversarial Ux Test (적대적 UX 테스트)

제품에 대해 가장 까다롭고 기술을 기피하는 사용자의 역할극을 수행합니다. 해당 페르소나로 앱을 탐색하며 모든 UX 불편 사항을 찾아낸 다음, 실용주의(pragmatism) 필터를 통해 불만을 걸러내어 단순한 불평과 진짜 문제를 분리합니다. 오직 진짜 문제에서만 조치 가능한 티켓을 생성합니다.

# Adversarial UX Test

제품에 대한 최악의 사용자 — 기술을 싫어하고, 소프트웨어를 원하지 않으며, 불평할 이유를 어떻게든 찾아내는 사람 — 의 역할극을 수행합니다. 그런 다음 실용주의 필터를 통해 그들의 피드백을 걸러내어, "난 컴퓨터가 싫어" 같은 단순한 노이즈와 진짜 UX 문제를 분리합니다.

분노한 버전의 자동화된 "맘 테스트(mom test)"라고 생각하시면 됩니다.

## 이것이 효과적인 이유

대부분의 QA는 버그를 찾지만, 이 테스트는 **마찰(friction)**을 찾습니다. 기술적으로 올바른 앱이라도 실제 사람들에게는 사용하기 어려울 수 있습니다. 적대적인 페르소나는 다음을 포착합니다:
- 개발자에게는 의미가 있지만 사용자에게는 혼란스러운 용어
- 기본적인 작업을 수행하는 데 너무 많은 단계
- 온보딩이나 "아하 모먼트(aha moments)"의 부재
- 접근성 문제 (글꼴 크기, 대비, 클릭 영역)
- 콜드 스타트 문제 (빈 화면, 데모 콘텐츠 부재)
- 전환율을 떨어뜨리는 결제창/가입 마찰

**실용주의 필터** (3단계)는 이것을 단순히 재미있는 것을 넘어 유용하게 만드는 요소입니다. 이 필터가 없다면, PDF를 이해하지 못하는 할아버지를 위해 모든 화면에 "이 페이지 인쇄하기" 버튼을 추가하게 될 것입니다.

## 사용 방법

에이전트에게 다음과 같이 말하세요:
```
"[URL]에서 적대적 UX 테스트를 실행해줘"
"까칠한 [페르소나 유형]이 되어서 [앱 이름]을 테스트해봐"
"내 스테이징 사이트에서 진상 사용자 테스트를 해줘"
```

페르소나를 직접 제공하거나, 제품의 타겟 고객을 기반으로 에이전트가 하나를 생성하도록 할 수 있습니다.

## 1단계: 페르소나 정의하기

제공된 페르소나가 없다면, 다음 질문에 답하여 페르소나를 생성합니다:

1. **이 제품에서 가장 까다로운 사용자는 누구인가?** (50세 이상, 비기술적 직무, "옛날 방식"으로 일한 수십 년의 경험)
2. **기술에 대한 숙련도는 어느 정도인가?** (낮을수록 좋습니다 — WhatsApp만 사용, 종이 노트 사용, 아내가 이메일 설정을 해줌)
3. **그들이 달성해야 하는 단 한 가지는 무엇인가?** (기능 목록이 아닌 그들의 핵심 업무)
4. **무엇이 그들을 포기하게 만들 것인가?** (너무 많은 클릭, 전문 용어, 느림, 혼란스러움)
5. **좌절했을 때 그들은 어떻게 말하는가?** (직설적, 욕설, 무시, 한숨)

### 좋은 페르소나 예시
> **"빅 믹(Big Mick)" 맥알리스터** — 58세의 근력 및 컨디셔닝(S&C) 코치. WhatsApp만 사용합니다. 그의 "스프레드시트"는 종이 노트입니다. "10초 안에 이해하지 못하면 내 노트로 돌아갈 거야." 25명 선수의 훈련 결과를 기록해야 합니다. 작은 글씨, 전문 용어, 비밀번호를 싫어합니다.

### 나쁜 페르소나 예시
> "앱을 좋아하지 않는 사용자" — 너무 모호하고, 제약이 없으며, 특색이 없습니다.

페르소나는 20분의 테스트 동안 **캐릭터를 유지할 수 있을 만큼 충분히 구체적이어야** 합니다.

## 2단계: 진상 되기 (페르소나로 브라우징하기)

1. 앱 컨텍스트와 URL에 대한 사용 가능한 프로젝트 문서를 읽습니다.
2. **페르소나에 완전히 몰입합니다** — 그들의 좌절, 한계, 목표.
3. 브라우저 도구를 사용하여 앱으로 이동합니다.
4. **페르소나의 실제 작업을 시도합니다** (기능 투어가 아님):
   - 그들이 하려던 것을 할 수 있습니까?
   - 그것을 달성하기 위해 몇 번의 클릭/화면이 필요합니까?
   - 무엇이 그들을 혼란스럽게 합니까?
   - 무엇이 그들을 화나게 합니까?
   - 어디서 길을 잃습니까?
   - 무엇이 그들을 포기하고 예전 방식으로 돌아가게 만들겠습니까?

5. 다음 마찰 범주를 테스트합니다:
   - **첫인상** — 랜딩 페이지를 넘어 애를 쓸 가치가 있겠습니까?
   - **핵심 워크플로우** — 그들이 가장 자주 해야 하는 단 한 가지
   - **오류 복구** — 뭔가 잘못했을 때 어떻게 됩니까?
   - **가독성** — 텍스트 크기, 대비, 정보 밀도
   - **속도** — 현재 방법보다 빠르다고 느껴집니까?
   - **용어** — 그들이 이해하지 못할 전문 용어가 있습니까?
   - **내비게이션** — 뒤로 돌아가는 길을 찾을 수 있습니까? 자신이 어디에 있는지 압니까?

6. 모든 불편 사항(pain point)의 스크린샷을 찍습니다.
7. 모든 페이지에서 브라우저 콘솔의 JS 오류를 확인합니다.

## 3단계: 불평 (캐릭터에 맞춰 피드백 작성)

피드백을 페르소나의 입장에서 — 그들의 목소리로, 그들의 좌절을 담아 작성합니다. 이것은 버그 리포트가 아닙니다. 실제 인간의 하소연입니다.

```
[제품]에 대한 [페르소나 이름]의 리뷰

총평: [계속 사용할 것인가? 예/아니오/조건부 예]

좋은 점 (마지못해 인정):
- [그들조차도 효과가 있다고 인정할 수밖에 없는 것들]

나쁜 점 (합당한 UX 문제):
- [그들이 제품 사용을 중단하게 만들 실제 문제들]

최악인 점 (쇼스토퍼):
- [즉시 삭제/취소하게 만들 것들]

구체적인 불만 사항:
1. [페이지/기능]: "[페르소나 목소리의 인용구]" — [발생한 일, 예상했던 일]
2. ...

판결: "[그들의 경험을 요약하는 페르소나의 한 줄 평]"
```

## 4단계: 실용주의 필터 (중요 — 건너뛰지 마세요)

페르소나에서 빠져나옵니다. 프로덕트 담당자로서 각 불만을 평가합니다:

- **빨간색(RED): 진짜 UX 버그** — 까칠한 사용자뿐만 아니라 모든 사용자가 겪을 문제입니다. 고쳐야 합니다.
- **노란색(YELLOW): 타당하지만 낮은 우선순위** — 실제 문제지만 극단적인 사용자에게만 해당합니다. 기록해 둡니다.
- **흰색(WHITE): 페르소나 노이즈** — "난 컴퓨터가 싫어" 식의 불만이며 제품 문제가 아닙니다. 무시합니다.
- **초록색(GREEN): 기능 요청** — 불만 속에 숨겨진 좋은 아이디어입니다. 고려해 봅니다.

### 필터 기준
1. 바쁘지만 유능한 35세 사용자도 같은 불만을 가질 것인가? → 빨간색
2. 이것이 진짜 접근성 문제(글꼴 크기, 대비, 클릭 영역)인가? → 빨간색
3. 디지털에 대한 "종이처럼 작동했으면 좋겠어" 식의 저항인가? → 흰색
4. 페르소나가 우연히 발견한 실제 워크플로우의 비효율성인가? → 노란색 또는 빨간색
5. 이것을 고치면 잘 쓰고 있는 80%의 사용자에게 복잡성을 더할 것인가? → 흰색
6. 불만이 온보딩 순간의 누락을 보여주는가? → 초록색

**이 필터는 필수적입니다.** 페르소나의 날것 그대로의 불만을 절대 티켓으로 바로 넘기지 마세요.

## 5단계: 티켓 생성하기

**빨간색** 및 **초록색** 항목에 대해서만:
- 명확하고 조치 가능한 제목
- 페르소나의 원문 인용구 포함 (재미있고 기억에 남음)
- 그 이면의 실제 UX 문제 (객관적)
- 제안하는 해결책 (조치 가능)
- 태그/라벨: "ux-review"

**노란색** 항목의 경우: 모든 노트를 포함하는 포괄적인 티켓 1개 생성.

**흰색** 항목은 리포트에만 나타나며, 티켓으로 만들지 않습니다.

**세션당 최대 10개의 티켓** — 최악의 문제에 집중하세요.

## 6단계: 리포트

다음을 전달합니다:
1. 페르소나의 불평 (3단계) — 재미있고 본능적임
2. 필터링된 평가 (4단계) — 실용적이고 조치 가능함
3. 생성된 티켓 (5단계) — 링크 포함
4. 주요 문제의 스크린샷

## 팁

- **세션당 하나의 페르소나.** 관점을 섞지 마세요.
- **2-3단계 동안 캐릭터를 유지하세요.** 4단계에서만 캐릭터에서 벗어납니다.
- **핵심 워크플로우를 먼저 테스트하세요.** 설정 페이지에 정신을 빼앗기지 마세요.
- **빈 화면(Empty states)은 금광입니다.** 신규 사용자 경험에서 가장 많은 마찰이 드러납니다.
- **가장 좋은 발견은 페르소나가 다른 것을 하려다 우연히 발견한 빨간색 항목입니다.**
- **페르소나에게 불만이 전혀 없다면, 페르소나가 너무 기술에 능숙한 것입니다.** 더 나이가 많고, 참을성이 없고, 자기 방식에 더 얽매인 사람으로 만드세요.
- **데모, 출시 전, 또는 기능을 배포한 후에 이 테스트를 실행하세요.**
- **가능하다면 신규 사용자로 등록하세요.** 미리 설정된 관리자 계정을 사용하지 마세요 — 대부분의 마찰은 콜드 스타트 경험에 존재합니다.
- **흰색 항목이 0개라는 것은 실패가 아니라 신호입니다.** 실용주의 필터가 노이즈를 전혀 찾지 못했다면, 제품에 단순히 까칠한 페르소나가 아닌 실제 UX 문제가 있다는 뜻입니다.
- **테스트 후에 프로젝트 문서의 알려진 문제를 확인하세요.** 페르소나가 찾은 버그가 이미 알려진 문제 목록에 있다면, 그것은 팀이 버그를 알고 있었지만 사용자의 고통을 전혀 공감하지 못했다는 가장 치명적인 결과입니다.
- **구독/결제창 테스트는 매우 중요합니다.** 활성 계정뿐만 아니라 만료된 계정으로도 테스트하세요. "결제할 수 없을 때 어떤 일이 일어나는가" 경험은 제품이 사용자를 존중하는지 아니면 데이터를 볼모로 잡는지 보여줍니다.
- **페르소나의 단 한 가지 작업을 달성하기 위해 클릭 수를 세어보세요.** 5번이 넘는다면, 페르소나의 기술 수준과 상관없이 거의 항상 빨간색 항목입니다.

## 산업별 페르소나 예시

이것들은 시작점이며, 특정 제품에 맞게 사용자 정의하세요:

| 제품 유형 | 페르소나 | 나이 | 핵심 특징 |
|-------------|---------|-----|-----------|
| CRM | 양로원 원장 | 68 | 파일 캐비닛이 현재의 CRM임 |
| 사진 SaaS | 시골 웨딩 사진작가 | 62 | 전화로 예약을 받고, 종이에 청구서를 씀 |
| AI/ML 도구 | 백화점 바이어 | 55 | 3번의 실패한 기술 스타트업 때문에 데인 적이 있음 |
| 피트니스 앱 | 올드스쿨 체육관 코치 | 58 | 종이 노트, 굵은 손가락, 나쁜 시력 |
| 회계 | 가족 빵집 사장 | 64 | 영수증 상자를 모아둠, 구독을 싫어함 |
| 이커머스 | 시장 가판대 상인 | 60 | 현금만 사용, 스마트폰은 전화용임 |
| 헬스케어 | 고참 일반의 | 63 | 노트를 구술하고, 간호사가 컴퓨터를 다룸 |
| 교육 | 베테랑 교사 | 57 | 분필과 판서, 바인더에 있는 학습지 |

## 규칙

- 2-3단계 동안 캐릭터를 유지할 것
- 진심으로 까칠하되 공정하게 — 꾸며낸 문제가 아닌 진짜 문제를 찾을 것
- 실용주의 필터 (4단계)는 **필수 사항**임
