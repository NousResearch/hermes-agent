# Советы и лучшие практики

Подборка коротких практических советов, которые сразу делают работу с Hermes Agent заметно эффективнее. Просто просмотрите заголовки и откройте те разделы, которые нужны вам сейчас.

---

## Как получать лучшие результаты

### Формулируйте запросы конкретно

Размытые запросы дают размытый результат. Вместо "исправь код" лучше сказать: "исправь TypeError в `api/handlers.py` на строке 47 - функция `process_request()` получает `None` из `parse_body()`". Чем больше контекста вы даёте, тем меньше итераций потребуется.

### Сразу давайте контекст

Начинайте запрос с нужных подробностей: пути к файлам, сообщений об ошибке, ожидаемого поведения. Один хорошо собранный запрос лучше, чем три раунда уточнений. Ошибки и трассировки можно вставлять прямо - агент умеет их разбирать.

### Используйте файлы контекста для повторяющихся инструкций

Если вы постоянно повторяете одни и те же указания ("используй табы, а не пробелы", "у нас pytest", "API доступен на `/api/v2`"), вынесите их в `AGENTS.md`. Агент будет читать его автоматически в каждом сеансе - после настройки ничего больше делать не нужно.

### Позвольте агенту использовать свои инструменты

Не пытайтесь водить его за руку на каждом шаге. Лучше сказать: "найди и исправь падающий тест", а не "открой `tests/test_foo.py`, посмотри на строку 42, потом..." У агента есть поиск по файлам, доступ к терминалу и выполнение кода - дайте ему возможность исследовать и уточнять решение.

### Используйте навыки для сложных сценариев

Прежде чем писать длинный запрос с инструкцией "как это делать", проверьте, нет ли уже готового навыка. Введите `/skills`, чтобы просмотреть доступные навыки, или вызовите нужный напрямую, например `/axolotl` или `/github-pr-workflow`.

## Советы для продвинутой работы в CLI

### Многострочный ввод

Нажмите **Alt+Enter**, **Ctrl+J** или **Shift+Enter**, чтобы вставить новую строку без отправки сообщения. `Shift+Enter` работает только в терминалах, которые передают его как отдельное нажатие клавиши (по умолчанию Kitty / foot / WezTerm / Ghostty; в iTerm2 / Alacritty / VS Code terminal - после включения Kitty keyboard protocol). Остальные два варианта работают в любом терминале.

### Определение вставки

CLI автоматически распознаёт многострочные вставки. Просто вставьте блок кода или трассировку ошибки прямо в чат - каждая строка не будет отправляться отдельно. Вставка буферизуется и уходит как одно сообщение.

### Прервать и перенаправить

Нажмите **Ctrl+C** один раз, чтобы прервать ответ агента на середине. После этого можно ввести новое сообщение и перенаправить его. Двойное нажатие **Ctrl+C** в течение 2 секунд принудительно завершает работу. Это особенно полезно, когда агент явно уходит не туда.

### Возобновляйте сеансы через `-c`

Забыли что-то из прошлого сеанса? Запустите `hermes -c`, чтобы продолжить именно с того места, где остановились, с полной историей разговора. Можно также возобновить по названию: `hermes -r "my research project"`.

### Вставка изображений из буфера

Нажмите **Ctrl+V**, чтобы вставить изображение из буфера обмена прямо в чат. Агент использует vision, чтобы анализировать скриншоты, схемы, всплывающие окна с ошибками или макеты интерфейсов - не нужно сначала сохранять файл.

### Автодополнение слэш-команд

Введите `/` и нажмите **Tab**, чтобы увидеть все доступные команды. В список входят встроенные команды (`/compress`, `/model`, `/title`) и все установленные навыки. Ничего запоминать не нужно - автодополнение по Tab уже всё покажет.

:::tip
Используйте `/verbose`, чтобы переключать режимы вывода активности инструментов: **off → new → all → verbose**. Режим `all` удобен, когда хочется видеть, что именно делает агент; `off` даёт самый чистый интерфейс для простых вопросов и ответов.
:::

## Файлы контекста

### AGENTS.md: центр правил вашего проекта

Создайте `AGENTS.md` в корне проекта и зафиксируйте там архитектурные решения, стиль кодирования и проектные инструкции. Этот файл автоматически подмешивается в каждый сеанс, поэтому агент всегда знает правила вашего проекта.

```markdown
# Контекст проекта
- Это бэкенд на FastAPI с ORM SQLAlchemy
- Для операций с базой данных всегда используй async/await
- Тесты лежат в tests/ и используют pytest-asyncio
- Никогда не коммить файлы `.env`
```

### SOUL.md: настройка личности

Хотите, чтобы у Hermes был стабильный голос по умолчанию? Отредактируйте `~/.hermes/SOUL.md` (или `$HERMES_HOME/SOUL.md`, если используете собственный Hermes home). Hermes автоматически создаёт начальный `SOUL.md` и использует этот глобальный файл как источник личности.

Подробный разбор смотрите в [Использовании SOUL.md с Hermes](/docs/guides/use-soul-with-hermes).

```markdown
# Личность
Вы опытный бэкенд-инженер. Пишите кратко и по делу.
Пропускайте объяснения, если их не просят. Отдавайте предпочтение коротким ответам вместо пространных решений.
Всегда учитывайте обработку ошибок и крайние случаи.
```

Используйте `SOUL.md` для постоянной личности. Используйте `AGENTS.md` для проектных инструкций.

### Совместимость с .cursorrules

Уже есть `.cursorrules` или `.cursor/rules/*.mdc`? Hermes читает и их тоже. Никакого дублирования правил кодирования не нужно - они автоматически подхватываются из рабочей директории.

### Обнаружение

Hermes загружает верхнеуровневый `AGENTS.md` из текущей рабочей директории в начале сеанса. Файлы `AGENTS.md` в подкаталогах обнаруживаются по мере необходимости во время вызовов инструментов (через `subdirectory_hints.py`) и автоматически подхватываются в результаты инструментов — заранее в системную инструкцию они не загружаются.

:::tip
Держите файлы контекста короткими и сфокусированными. Каждый символ влияет на токенный бюджет, потому что эти файлы попадают в каждое сообщение.
:::

## Память и навыки

### Память и навыки: что куда класть

**Память** - для фактов: ваше окружение, предпочтения, расположение проектов и то, что агент уже узнал о вас. **Навыки** - для процедур: многошаговых рабочих процессов, инструкций для конкретных инструментов и переиспользуемых рецептов. Используйте память для "что", а навыки для "как".

### Когда создавать навыки

Если вы понимаете, что задача состоит из 5+ шагов и будет повторяться, попросите агента сохранить её как навык. Скажите: "сохрани то, что ты только что сделал, как навык `deploy-staging`". В следующий раз достаточно будет ввести `/deploy-staging`, и агент загрузит весь процесс целиком.

### Управление ёмкостью памяти

Память намеренно ограничена (~2,200 символов для `MEMORY.md`, ~1,375 символов для `USER.md`). Когда она заполняется, агент консолидирует записи. Вы можете помочь ему, сказав "почисти память" или "замени старую заметку про Python 3.9 - теперь у нас 3.12".

### Позвольте агенту помнить

После удачного сеанса можно сказать "запомни это на будущее" — и агент сохранит ключевые выводы. Можно быть и конкретнее: "сохрани в память, что наш CI использует рабочий процесс GitHub Actions `deploy.yml`".

:::warning
Память — это замороженный снимок. Изменения, сделанные во время сеанса, не попадают в системную инструкцию до следующего запуска. Агент пишет на диск сразу, но кэш промпта не обновляется посреди сеанса.
:::

## Производительность и стоимость

### Не ломайте кэш промпта

Большинство LLM-провайдеров кэшируют префикс системной инструкции. Если держать системную инструкцию стабильной (те же файлы контекста, та же память), последующие сообщения в сеансе получают **попадания в кэш**, которые заметно дешевле. Избегайте смены модели или системной инструкции посреди сеанса.

### Используйте /compress до упора

Длинные сеансы накапливают токены. Когда ответы начинают замедляться или обрезаться, запустите `/compress`. Команда суммирует историю разговора, сохраняя ключевой контекст и резко сокращая объём токенов. Используйте `/usage`, чтобы посмотреть текущее состояние.

### Делегируйте параллельную работу

Нужно одновременно исследовать три темы? Попросите агента использовать `delegate_task` с параллельными подзадачами. Каждый субагент работает независимо и имеет собственный контекст, а обратно возвращаются только итоговые сводки — это резко экономит токены в основном диалоге.

### Используйте execute_code для пакетных операций

Вместо того чтобы по одному запускать команды в терминале, попросите агента написать скрипт, который сделает всё сразу. "Напиши Python-скрипт, который переименует все `.jpeg` в `.jpg`, и запусти его" - дешевле и быстрее, чем переименовывать файлы по одному.

### Выбирайте подходящую модель

Используйте `/model`, чтобы переключать модели прямо в сеансе. Для сложных задач рассуждения и архитектурных решений выбирайте фронтирную модель (Claude Sonnet/Opus, GPT-4o). Для простых задач вроде форматирования, переименований или генерации шаблонного кода переключайтесь на более быструю модель.

:::tip
Периодически запускайте `/usage`, чтобы видеть расход токенов. А `/insights` даст более полную картину использования за последние 30 дней.
:::

## Советы для мессенджеров

### Назначьте домашний канал

Используйте `/sethome` в выбранном чате Telegram или Discord, чтобы назначить его домашним каналом. Именно сюда будут приходить результаты cron-задач и других запланированных действий. Без этого агенту просто некуда слать проактивные сообщения.

### Используйте /title для организации сеансов

Даже короткие названия вроде `/title auth-refactor` или `/title research-llm-quantization` сильно упрощают жизнь. Такие сеансы легко найти через `hermes sessions list` и возобновить через `hermes -r "auth-refactor"`. Безымянные сеансы быстро сливаются в неразличимую массу.

### DM pairing для командного доступа

Вместо ручного сбора ID пользователя для списка разрешений включите DM pairing. Когда коллега пишет боту в личку, он получает одноразовый код pairing. Вы подтверждаете его через `hermes pairing approve telegram XKGH5N7P` - просто и безопасно.

### Режимы отображения прогресса инструментов

Используйте `/verbose`, чтобы контролировать, сколько активности инструментов вы видите. В мессенджерах обычно лучше меньше, чем больше - оставляйте режим `new`, чтобы видеть только новые вызовы инструментов. В CLI режим `all` даёт приятный живой просмотр всего, что делает агент.

:::tip
В мессенджерах сеансы автоматически сбрасываются после простоя (по умолчанию 24 часа) или ежедневно в 4:00 утра. При необходимости более длинных сеансов настройте это по платформам в `~/.hermes/config.yaml`.
:::

## Безопасность

### Используйте Docker для недоверенного кода

Если вы работаете с недоверенным репозиторием или запускаете незнакомый код, используйте Docker или Daytona в качестве терминального бэкенда. Установите `TERMINAL_BACKEND=docker` в `.env`. Деструктивные команды внутри контейнера не смогут повредить хост-систему.

```bash
# В вашем .env:
TERMINAL_BACKEND=docker
TERMINAL_DOCKER_IMAGE=hermes-sandbox:latest
```

### Избегайте ловушек кодировки Windows

В Windows некоторые кодировки по умолчанию (например, `cp125x`) не умеют представлять все Unicode-символы, из-за чего при записи файлов в тестах или скриптах может возникать `UnicodeEncodeError`.

- Открывайте файлы с явной UTF-8 encoding:

```python
with open("results.txt", "w", encoding="utf-8") as f:
    f.write("✓ All good\n")
```

- В PowerShell можно также переключить текущий сеанс на UTF-8 для консоли и вывода нативных команд:

```powershell
$OutputEncoding = [Console]::OutputEncoding = [Text.UTF8Encoding]::new($false)
```

Так PowerShell и дочерние процессы остаются на UTF-8, и меньше шансов нарваться на Windows-only сбой.

### Подумайте, прежде чем выбирать "always"

Когда агент показывает запрос подтверждения опасной команды (`rm -rf`, `DROP TABLE` и т. п.), у вас есть четыре варианта: **once**, **session**, **always**, **deny**. С осторожностью выбирайте `always` - это навсегда добавляет шаблон в список разрешений. Пока не уверены, начинайте с `session`.

### Подтверждение команд - ваша страховка

Hermes перед выполнением проверяет каждую команду по куратору опасных шаблонов. Это включает рекурсивные удаления, SQL DROP, передачу `curl` в shell и многое другое. Не отключайте это в боевом окружении — защита нужна не просто так.

:::warning
Если вы запускаете контейнерный бэкенд (Docker, Singularity, Modal, Daytona), проверки опасных команд **пропускаются**, потому что именно контейнер считается границей безопасности. Убедитесь, что контейнерные образы у вас действительно хорошо защищены.
:::

### Используйте списки разрешений для мессенджер-ботов

Никогда не ставьте `GATEWAY_ALLOW_ALL_USERS=true` на боте с доступом к терминалу. Всегда используйте либо платформенные списки разрешений (`TELEGRAM_ALLOWED_USERS`, `DISCORD_ALLOWED_USERS`), либо DM pairing, чтобы контролировать, кто может взаимодействовать с агентом.

```bash
# Рекомендуется: явные списки разрешений по платформам
TELEGRAM_ALLOWED_USERS=123456789,987654321
DISCORD_ALLOWED_USERS=123456789012345678

# Или общий список разрешений
GATEWAY_ALLOWED_USERS=123456789,987654321
```
