# Справочник CLI-команд

На этой странице собраны **терминальные команды**, которые вы запускаете из shell.

Для слэш-команд внутри чата см. [Справочник слэш-команд](./slash-commands.md).

## Глобальная точка входа

```bash
hermes [global-options] <command> [subcommand/options]
```

### Глобальные параметры

| Параметр | Описание |
|--------|-------------|
| `--version`, `-V` | Показать версию и выйти. |
| `--profile <name>`, `-p <name>` | Выбрать, какой профиль Hermes использовать для этого запуска. Переопределяет закреплённое значение по умолчанию, заданное через `hermes profile use`. |
| `--resume <session>`, `-r <session>` | Возобновить предыдущий сеанс по ID или заголовку. |
| `--continue [name]`, `-c [name]` | Возобновить самый последний сеанс или самый последний сеанс с совпадающим заголовком. |
| `--worktree`, `-w` | Начать работу в изолированном git worktree для сценариев с параллельными агентами. |
| `--yolo` | Пропустить подтверждения опасных команд. |
| `--pass-session-id` | Добавить ID сеанса в системную инструкцию агента. |
| `--ignore-user-config` | Игнорировать `~/.hermes/config.yaml` и использовать встроенные значения по умолчанию. Учётные данные из `.env` при этом всё равно загружаются. |
| `--ignore-rules` | Не подгружать `AGENTS.md`, `SOUL.md`, `.cursorrules`, память и предзагруженные навыки. |
| `--tui` | Запустить [TUI](/docs/user-guide/tui) вместо классического CLI. Эквивалентно `HERMES_TUI=1`. |
| `--dev` | Вместе с `--tui`: запускать TypeScript-исходники напрямую через `tsx`, а не собранный bundle (для тех, кто разрабатывает TUI). |

## `hermes chat`

```bash
hermes chat [options]
```

Часто используемые параметры:

| Параметр | Описание |
|--------|-------------|
| `-q`, `--query "..."` | Одноразовый неинтерактивный запрос. |
| `-m`, `--model <model>` | Переопределить модель для этого запуска. |
| `-t`, `--toolsets <csv>` | Включить наборы инструментов через запятую. |
| `--provider <provider>` | Принудительно выбрать провайдера: `auto`, `openrouter`, `nous`, `openai-codex`, `copilot-acp`, `copilot`, `anthropic`, `gemini`, `google-gemini-cli`, `huggingface`, `novita`, `zai`, `kimi-coding`, `kimi-coding-cn`, `minimax`, `minimax-cn`, `minimax-oauth`, `kilocode`, `xiaomi`, `arcee`, `gmi`, `alibaba`, `alibaba-coding-plan` (alias `alibaba_coding`), `deepseek`, `nvidia`, `ollama-cloud`, `xai` (alias `grok`), `xai-oauth` (alias `grok-oauth`), `qwen-oauth`, `bedrock`, `opencode-zen`, `opencode-go`, `ai-gateway`, `azure-foundry`, `lmstudio`, `stepfun`, `tencent-tokenhub` (alias `tencent`, `tokenhub`). |
| `-s`, `--skills <name>` | Предзагрузить один или несколько навыков на сеанс (можно повторять или указывать через запятую). |
| `-v`, `--verbose` | Подробный вывод. |
| `-Q`, `--quiet` | Режим для программного использования: убрать баннер, спиннер и предпросмотр инструментов. |
| `--image <path>` | Прикрепить локальное изображение к одному запросу. |
| `--resume <session>` / `--continue [name]` | Возобновить сеанс прямо из `chat`. |
| `--worktree` | Создать для этого запуска изолированный git worktree. |
| `--checkpoints` | Включить файловые checkpoints перед разрушительными изменениями файлов. |
| `--yolo` | Пропустить запросы на подтверждение. |
| `--pass-session-id` | Передать ID сеанса в системную инструкцию. |
| `--ignore-user-config` | Игнорировать `~/.hermes/config.yaml` и использовать встроенные значения по умолчанию. Учётные данные из `.env` при этом всё равно загружаются. Полезно для изолированных CI-запусков, воспроизводимых баг-репортов и сторонних интеграций. |
| `--ignore-rules` | Не подгружать `AGENTS.md`, `SOUL.md`, `.cursorrules`, постоянную память и предзагруженные навыки. Вместе с `--ignore-user-config` даёт полностью изолированный запуск. |
| `--source <tag>` | Тег источника сеанса для фильтрации (по умолчанию: `cli`). Используйте `tool` для сторонних интеграций, которые не должны попадать в списки пользовательских сеансов. |
| `--max-turns <N>` | Максимальное число итераций вызова инструментов на один ход разговора (по умолчанию: 90, либо `agent.max_turns` в конфиге). |

Примеры:

```bash
hermes
hermes chat -q "Summarize the latest PRs"
hermes chat --provider openrouter --model anthropic/claude-sonnet-4.6
hermes chat --toolsets web,terminal,skills
hermes chat --quiet -q "Return only JSON"
hermes chat --worktree -q "Review this repo and open a PR"
hermes chat --ignore-user-config --ignore-rules -q "Repro without my personal setup"
```

### `hermes -z <prompt>` — сценарный одноразовый режим

Для программных вызовов (shell-скриптов, CI, cron, родительских процессов, которые передают запрос во входной поток) `hermes -z` — это самая «чистая» точка входа для одноразового запроса: **один запрос на входе, только финальный ответ на выходе, ничего лишнего в stdout или stderr.** Никакого баннера, никаких спиннеров, никакого предпросмотра инструментов, никакой строки `Session:` — только финальный ответ агента в виде обычного текста.

```bash
hermes -z "What's the capital of France?"
# → Paris.

# Родительский скрипт может надёжно захватить ответ:
answer=$(hermes -z "summarize this" < /path/to/file.txt)
```

Переопределения на один запуск (без изменения `~/.hermes/config.yaml`):

| Параметр | Эквивалентная переменная env | Назначение |
|---|---|---|
| `-m` / `--model <model>` | `HERMES_INFERENCE_MODEL` | Переопределить модель для этого запуска |
| `--provider <provider>` | `HERMES_INFERENCE_PROVIDER` | Переопределить провайдера для этого запуска |

```bash
hermes -z "…" --provider openrouter --model openai/gpt-5.5
# или:
HERMES_INFERENCE_MODEL=anthropic/claude-sonnet-4.6 hermes -z "…"
```

Это тот же агент, те же инструменты и те же навыки — только без интерактивного и визуального слоя. Если в транскрипте нужен ещё и вывод инструментов, используйте `hermes chat -q`; `-z` специально предназначен для сценария «мне нужен только финальный ответ».

## `hermes model`

Интерактивный выбор провайдера и модели. **Это команда для добавления новых провайдеров, настройки API-ключей и запуска OAuth-потоков.** Запускайте её из терминала, а не внутри активного сеанса Hermes.

```bash
hermes model
```

Используйте эту команду, когда хотите:
- **добавить нового провайдера** (OpenRouter, Anthropic, Copilot, DeepSeek, custom и т. д.)
- войти в OAuth-провайдеры (Anthropic, Copilot, Codex, Nous Portal)
- ввести или обновить API-ключи
- выбрать модель из списка конкретного провайдера
- настроить собственную или самостоятельно размещённую конечную точку
- сохранить новое значение по умолчанию в конфиг

:::warning `hermes model` vs `/model` — важно не перепутать
**`hermes model`** (запускается из терминала, вне активного сеанса Hermes) — это **полный мастер настройки провайдера**. Он может добавить нового провайдера, запустить OAuth-поток, запросить API-ключи и настроить конечную точку.

**`/model`** (вводится внутри активного чата Hermes) может только **переключаться между уже настроенными провайдерами и моделями**. Он не умеет добавлять новых провайдеров, запускать OAuth или запрашивать API-ключи.

**Если нужно добавить нового провайдера:** сначала выйдите из сеанса Hermes (`Ctrl+C` или `/quit`), а затем запустите `hermes model` из интерактивной оболочки.
:::

### слэш-команда `/model` в середине сеанса

Переключение между уже настроенными моделями без выхода из сеанса:

```
/model                              # Показать текущую модель и доступные варианты
/model claude-sonnet-4              # Переключить модель (провайдер определяется автоматически)
/model zai:glm-5                    # Переключить и провайдера, и модель
/model custom:qwen-2.5              # Использовать модель на собственной конечной точке
/model custom                       # Определить модель автоматически по собственной конечной точке
/model custom:local:qwen-2.5        # Использовать именованного пользовательского провайдера
/model openrouter:anthropic/claude-sonnet-4  # Вернуться к облачному провайдеру
```

По умолчанию изменения через `/model` действуют **только на текущий сеанс**. Добавьте `--global`, чтобы сохранить изменение в `config.yaml`:

```
/model claude-sonnet-4 --global     # Переключить и сохранить как новую настройку по умолчанию
```

:::info Что делать, если я вижу только модели OpenRouter?
Если вы настроили только OpenRouter, `/model` будет показывать только его модели. Чтобы добавить другого провайдера (Anthropic, DeepSeek, Copilot и т. д.), выйдите из сеанса и запустите `hermes model` из терминала.
:::

Изменения провайдера и базовый URL сохраняются в `config.yaml` автоматически. Когда вы уходите с собственной конечной точки, устаревший базовый URL очищается, чтобы он не «протекал» в других провайдеров.

## `hermes gateway`

```bash
hermes gateway <subcommand>
```

Подкоманды:

| Подкоманда | Описание |
|------------|-------------|
| `run` | Запустить шлюз в foreground. Рекомендуется для WSL, Docker и Termux. |
| `start` | Запустить установленную фоновую службу systemd/launchd. |
| `stop` | Остановить службу (или foreground-процесс). |
| `restart` | Перезапустить службу. |
| `status` | Показать состояние службы. |
| `list` | Показать **все профили** и то, запущен ли у каждого шлюз (с PID, если доступен). Удобно, когда вы работаете сразу с несколькими профилями и хотите один общий обзор. |
| `install` | Установить как фоновую службу systemd (Linux) или launchd (macOS). |
| `uninstall` | Удалить установленную службу. |
| `setup` | Интерактивная настройка платформ обмена сообщениями. |

Параметры:

| Параметр | Описание |
|--------|-------------|
| `--all` | Для `start` / `restart` / `stop`: применить действие **ко всем** шлюзам профилей, а не только к активному `HERMES_HOME`. Полезно, если вы ведёте несколько профилей параллельно и хотите перезапустить их все после `hermes update`. |

:::tip Пользователям WSL
Используйте `hermes gateway run` вместо `hermes gateway start` — поддержка systemd в WSL ненадёжна. Для устойчивой работы оберните команду в tmux: `tmux new -s hermes 'hermes gateway run'`. Подробности см. в [WSL FAQ](/docs/reference/faq#wsl-gateway-keeps-disconnecting-or-hermes-gateway-start-fails).
:::

## `hermes lsp`

```bash
hermes lsp <subcommand>
```

Управление интеграцией Language Server Protocol. LSP запускает настоящие
языковые серверы (pyright, gopls, rust-analyzer, …) в фоне и передаёт их
диагностику в post-write-проверку, которую используют `write_file` и `patch`.
Работает только внутри git-worktree — LSP запускается лишь тогда, когда cwd
или редактируемый файл находится внутри git worktree.

Подкоманды:

| Подкоманда | Описание |
|------------|-------------|
| `status` | Показать состояние службы, настроенные серверы и статус установки. |
| `list` | Вывести реестр поддерживаемых серверов. Добавьте `--installed-only`, чтобы пропустить отсутствующие. |
| `install <id>` | Заблаговременно установить бинарник одного сервера. |
| `install-all` | Установить каждый сервер, для которого есть известный рецепт автоустановки. |
| `restart` | Отключить работающие клиенты, чтобы при следующем изменении они поднялись заново. |
| `which <id>` | Показать путь к бинарнику для одного сервера. |

Полное руководство, поддерживаемые языки и параметры конфигурации см. в разделе [LSP — Semantic Diagnostics](/docs/user-guide/features/lsp).

## `hermes setup`

```bash
hermes setup [model|tts|terminal|gateway|tools|agent] [--non-interactive] [--reset] [--quick] [--reconfigure]
```

**Первый запуск:** открывает мастер первичной настройки.

**Возвращающийся пользователь (уже настроено):** сразу открывает полный мастер перенастройки — каждый шаг показывает текущее значение как значение по умолчанию; нажмите Enter, чтобы оставить его, или введите новое. Никакого меню.

Переход сразу в нужный раздел вместо полного мастера:

| Раздел | Описание |
|---------|-------------|
| `model` | Настройка провайдера и модели. |
| `terminal` | Настройка терминального бэкенда и песочницы. |
| `gateway` | Настройка платформы обмена сообщениями. |
| `tools` | Включение и отключение инструментов по платформам. |
| `agent` | Настройки поведения агента. |

Параметры:

| Параметр | Описание |
|--------|-------------|
| `--quick` | Для повторных запусков: спрашивать только о том, что ещё не задано или не настроено. Уже настроенные пункты пропускать. |
| `--non-interactive` | Использовать значения по умолчанию / значения из env без дополнительных вопросов. |
| `--reset` | Сбросить конфигурацию до значений по умолчанию перед настройкой. |
| `--reconfigure` | Обратносуместимый псевдоним: на уже установленной системе обычный `hermes setup` теперь делает именно это. |

## `hermes slack`

```bash
hermes slack manifest              # Вывести manifest в stdout
hermes slack manifest --write      # Записать в ~/.hermes/slack-manifest.json
hermes slack manifest --slashes-only  # Только массив features.slash_commands
```

Генерирует манифест Slack-приложения, который регистрирует каждую команду gateway из
`COMMAND_REGISTRY` (`/btw`, `/stop`, `/model`, …) как полноценную
Slack слэш-команду — с паритетом для Discord и Telegram. Вставьте вывод в
конфигурацию Slack-приложения по адресу
[https://api.slack.com/apps](https://api.slack.com/apps) → ваше приложение →
**Features → App Manifest → Edit**, затем **Save**. Если набор scopes или
слэш-команд изменился, Slack предложит переустановить приложение.

| Параметр | По умолчанию | Назначение |
|------|---------|---------|
| `--write [PATH]` | stdout | Записать в файл вместо stdout. Если `--write` указан без пути, файл пишется в `$HERMES_HOME/slack-manifest.json`. |
| `--name NAME` | `Hermes` | Отображаемое имя бота в Slack. |
| `--description DESC` | стандартное описание | Описание бота, показываемое в каталоге Slack-приложений. |
| `--slashes-only` | off | Выводить только `features.slash_commands` для слияния в манифест, который вы ведёте вручную. |

После `hermes update` запустите `hermes slack manifest --write` ещё раз, чтобы
подхватить новые команды.

## `hermes kanban`

```bash
hermes kanban [--board <slug>] <action> [options]
```

Доска совместной работы для нескольких профилей и нескольких проектов. Каждая установка может хранить много досок (по одной на проект, репозиторий или домен); каждая доска — это самостоятельная очередь со своей SQLite-базой и своей областью для диспетчера. После новой установки создаётся одна доска `default`, а её база хранится в `~/.hermes/kanban.db` ради обратной совместимости; дополнительные доски лежат в `~/.hermes/kanban/boards/<slug>/kanban.db`. Встроенный в gateway диспетчер обходит все доски на каждом тике.

**Глобальные флаги (применяются ко всем действиям ниже):**

| Флаг | Назначение |
|------|---------|
| `--board <slug>` | Работать с конкретной доской. По умолчанию используется текущая доска (её можно задать через `hermes kanban boards switch`, переменную `HERMES_KANBAN_BOARD` или `default`). |

**Это интерфейс для ручной и сценарной работы.** Агентные исполнители, которых запускает диспетчер, работают с доской через отдельный `kanban_*` [набор инструментов](/docs/user-guide/features/kanban#how-workers-interact-with-the-board) (`kanban_show`, `kanban_complete`, `kanban_block`, `kanban_create`, `kanban_link`, `kanban_comment`, `kanban_heartbeat`; профили с ролью `orchestrator` дополнительно получают `kanban_list` и `kanban_unblock`) вместо прямого вызова `hermes kanban`. У этих исполнителей в переменных окружения закреплён `HERMES_KANBAN_BOARD`, поэтому они физически не могут видеть другие доски.

| Действие | Назначение |
|--------|---------|
| `init` | Создать `kanban.db`, если его нет. Идемпотентно. |
| `boards list` / `boards ls` | Показать все доски и количество задач в каждой. Поддерживает `--json`, `--all` (включая архивные). |
| `boards create <slug>` | Создать новую доску. Флаги: `--name`, `--description`, `--icon`, `--color`, `--switch` (сделать активной). Slug — в kebab-case, автоматически приводится к нижнему регистру. |
| `boards switch <slug>` / `boards use` | Сохранить `<slug>` как активную доску (записывает `~/.hermes/kanban/current`). |
| `boards show` / `boards current` | Вывести имя текущей активной доски, путь к БД и количество задач. |
| `boards rename <slug> "<name>"` | Изменить отображаемое имя доски. Slug неизменяем. |
| `boards rm <slug>` | Архивировать (по умолчанию) или удалить доску окончательно. `--delete` пропускает этап архивации. Архивные доски перемещаются в `boards/_archived/<slug>-<ts>/`. Для `default` запрещено. |
| `create "<title>"` | Создать новую задачу на активной доске. Флаги: `--body`, `--assignee`, `--parent` (можно повторять), `--workspace scratch\|worktree\|dir:<path>`, `--tenant`, `--priority`, `--triage`, `--idempotency-key`, `--max-runtime`, `--max-retries`, `--skill` (можно повторять). |
| `list` / `ls` | Показать задачи на активной доске. Фильтры: `--mine`, `--assignee`, `--status`, `--tenant`, `--archived`, `--json`. |
| `show <id>` | Показать задачу вместе с комментариями и событиями. `--json` — для машинного вывода. |
| `assign <id> <profile>` | Назначить или переназначить. Используйте `none`, чтобы снять назначение. Пока задача выполняется, команда отклоняется. |
| `link <parent> <child>` | Добавить зависимость. Проверяется на циклы. Обе задачи должны находиться на одной доске. |
| `unlink <parent> <child>` | Удалить зависимость. |
| `claim <id>` | Атомарно взять готовую задачу в работу. Выводит путь к рабочему пространству. |
| `comment <id> "<text>"` | Добавить комментарий. Следующий исполнитель, который возьмёт задачу, увидит его в ответе `kanban_show()`. |
| `complete <id>` | Пометить задачу выполненной. Флаги: `--result`, `--summary`, `--metadata`. |
| `block <id> "<reason>"` | Пометить задачу заблокированной до вмешательства человека. Также добавляет причину как комментарий. |
| `schedule <id> "<reason>"` | Отложить работу с таймером / follow-up в состояние `scheduled`, чтобы это не воспринималось как ручная блокировка. |
| `unblock <id>` | Вернуть заблокированную или отложенную задачу в `ready` (или `todo`, если зависимости ещё не закрыты). |
| `archive <id>` | Скрыть из списка по умолчанию. `gc` удалит scratch-workspaces. |
| `tail <id>` | Следить за потоком событий задачи. |
| `dispatch` | Один проход диспетчера по активной доске. Флаги: `--dry-run`, `--max N`, `--failure-limit N`, `--json`. |
| `context <id>` | Показать полный контекст задачи, который увидит исполнитель (title + body + результаты родительских задач + комментарии). |
| `specify <id>` / `specify --all` | Превратить задачу первичного разбора в конкретную спецификацию (title + body с целью, подходом и критериями приёмки) через вспомогательную модель, а затем перевести её в `todo`. Флаги: `--tenant` (ограничить `--all` одним тенантом), `--author`, `--json`. Модель настраивается через `auxiliary.triage_specifier` в `config.yaml`. |
| `decompose <id>` / `decompose --all` | Разложить задачу первичного разбора на граф дочерних задач, распределённых по профилям специалистов на основе описания (по схеме оркестрации). Если модель решит, что разветвление не нужно, задача переводится по схеме `specify` в одиночную задачу. Те же флаги, что у `specify`. Модель настраивается через `auxiliary.kanban_decomposer` в `config.yaml`. Также запускается автоматически на каждом такте диспетчера, если включён `kanban.auto_decompose: true` (по умолчанию). См. [Автоматическая и ручная оркестрация](/docs/user-guide/features/kanban#auto-vs-manual-orchestration). |
| `gc` | Удалить scratch-workspaces для архивных задач. |

Примеры:

```bash
# Создать вторую доску и добавить туда задачу, не переключаясь на неё.
hermes kanban boards create atm10-server --name "ATM10 Server" --icon 🎮
hermes kanban --board atm10-server create "Restart server" --assignee ops

# Переключить активную доску для последующих вызовов.
hermes kanban boards switch atm10-server
hermes kanban list                  # Показать задачи доски atm10-server

# Архивировать доску (её можно восстановить) или удалить окончательно.
hermes kanban boards rm atm10-server
hermes kanban boards rm atm10-server --delete
```

Порядок выбора доски (от наивысшего приоритета к наименьшему): флаг `--board <slug>` → env `HERMES_KANBAN_BOARD` → файл `~/.hermes/kanban/current` → `default`.

Все действия доступны и как слэш-команда в gateway (`/kanban …`) с тем же набором аргументов — включая подкоманды `boards` и флаг `--board`.

Полное описание — сравнение с Cline Kanban / Paperclip / NanoClaw / Gemini Enterprise, восемь паттернов совместной работы, четыре пользовательских сценария, доказательство корректности при конкуренции — см. в `docs/hermes-kanban-v1-spec.pdf` в репозитории или в [Kanban user guide](/docs/user-guide/features/kanban).

## `hermes debug`

Загружает отчет для отладки (системную информацию и свежие логи) в paste-сервис и выдаёт ссылку, которой можно поделиться. Удобно для быстрых запросов в поддержку — внутри есть всё, что нужно для диагностики.

## `hermes backup`

```bash
hermes backup [options]
```

Создаёт zip-архив с вашей конфигурацией Hermes, навыками, сеансами и данными. Сам кодовый репозиторий `hermes-agent` в резервную копию не входит.

| Параметр | Описание |
|--------|-------------|
| `-o`, `--output <path>` | Путь вывода для zip-файла (по умолчанию: `~/hermes-backup-<timestamp>.zip`). |
| `-q`, `--quick` | Быстрый снимок: только критичные файлы состояния (`config.yaml`, `state.db`, `.env`, auth, cron-задачи). Значительно быстрее полного бэкапа. |
| `-l`, `--label <name>` | Метка для снимка (используется только вместе с `--quick`). |

Для безопасного копирования резервная копия использует API `backup()` из SQLite, поэтому корректно работает даже тогда, когда Hermes уже запущен (совместимо с WAL-mode).

**Что исключается из zip:**

- `*.db-wal`, `*.db-shm`, `*.db-journal` — sidecar-файлы WAL / shared-memory / journal SQLite. Сам файл `*.db` уже получил согласованный снимок через `sqlite3.backup()`; включение живых sidecar-файлов вместе с ним может привести к восстановлению полусохранённого состояния.
- `checkpoints/` — кэши траекторий по сеансам. Они хэшируются по ключу и пересоздаются для каждого сеанса; на другой установке всё равно не переносятся корректно.
- Сам код `hermes-agent` (это резервная копия пользовательских данных, а не снимок репозитория).

### Примеры

```bash
hermes backup                           # Полная копия в ~/hermes-backup-*.zip
hermes backup -o /tmp/hermes.zip        # Полная копия в указанный путь
hermes backup --quick                   # Быстрый снимок только состояния
hermes backup --quick --label "pre-upgrade"  # Быстрый снимок с меткой
```

## `hermes checkpoints`

```bash
hermes checkpoints [COMMAND]
```

Просмотр и управление теневым git-хранилищем `~/.hermes/checkpoints/` — слоем хранения, который стоит за командой `/rollback` в сеансе. Безопасно запускать в любой момент; агент при этом не должен быть запущен.

| Подкоманда | Описание |
|------------|-------------|
| `status` (по умолчанию) | Показать общий размер, число проектов и разбивку по каждому проекту. Просто `hermes checkpoints` — это то же самое. |
| `list` | Alias для `status`. |
| `prune` | Принудительно выполнить очистку — удалить осиротевшие и устаревшие проекты, запустить GC хранилища и соблюсти ограничение по размеру. Игнорирует 24-часовой маркер идемпотентности. |
| `clear` | Удалить всю базу checkpoints. Необратимо; запросит подтверждение, если не указан `-f`. |
| `clear-legacy` | Удалить только архивы `legacy-<timestamp>/`, которые были созданы миграцией v1 → v2. |

### Параметры

| Параметр | Подкоманда | Описание |
|--------|------------|-------------|
| `--limit N` | `status`, `list` | Максимум проектов в списке (по умолчанию 20). |
| `--retention-days N` | `prune` | Удалять проекты, у которых `last_touch` старше N дней (по умолчанию 7). |
| `--max-size-mb N` | `prune` | После прохода по orphan/stale удалять самый старый commit в каждом проекте, пока общий размер хранилища не станет ≤ N МБ (по умолчанию 500). |
| `--keep-orphans` | `prune` | Не удалять проекты, у которых уже нет рабочей директории. |
| `-f`, `--force` | `clear`, `clear-legacy` | Пропустить запрос на подтверждение. |

### Примеры

```bash
hermes checkpoints                                  # Краткая сводка состояния
hermes checkpoints prune --retention-days 3         # Агрессивная очистка
hermes checkpoints prune --max-size-mb 200          # Один раз ужать лимит размера
hermes checkpoints clear-legacy -f                  # Удалить архивы v1
hermes checkpoints clear -f                         # Очистить всё
```

Полную архитектуру и команды в сеансе см. в разделе [Контрольные точки и `/rollback`](/docs/user-guide/checkpoints-and-rollback).

## `hermes logs`

```bash
hermes logs [log_name] [options]
```

Просмотр, отслеживание и фильтрация файлов логов Hermes. Все логи хранятся в `~/.hermes/logs/` (или в `<profile>/logs/` для нестандартных профилей).

### Файлы логов

| Имя | Файл | Что записывается |
|------|------|-----------------|
| `agent` (по умолчанию) | `agent.log` | Вся активность агента: вызовы API, dispatch инструментов, жизненный цикл сеанса (INFO и выше) |
| `errors` | `errors.log` | Только предупреждения и ошибки — отфильтрованная выборка из `agent.log` |
| `gateway` | `gateway.log` | Активность шлюза сообщений: подключения к платформам, отправка сообщений, webhook-события |

### Параметры

| Параметр | Описание |
|--------|-------------|
| `log_name` | Какой лог смотреть: `agent` (по умолчанию), `errors`, `gateway` или `list`, чтобы показать доступные файлы с размерами. |
| `-n`, `--lines <N>` | Сколько строк показать (по умолчанию: 50). |
| `-f`, `--follow` | Следить за логом в реальном времени, как `tail -f`. Для выхода нажмите Ctrl+C. |
| `--level <LEVEL>` | Минимальный уровень логов: `DEBUG`, `INFO`, `WARNING`, `ERROR`, `CRITICAL`. |
| `--session <ID>` | Фильтровать строки по подстроке ID сеанса. |
| `--since <TIME>` | Показать строки за время относительно сейчас: `30m`, `1h`, `2d` и т. п. Поддерживаются `s` (секунды), `m` (минуты), `h` (часы), `d` (дни). |
| `--component <NAME>` | Фильтр по компоненту: `gateway`, `agent`, `tools`, `cli`, `cron`. |

### Примеры

```bash
# Показать последние 50 строк agent.log (по умолчанию)
hermes logs

# Следить за agent.log в реальном времени
hermes logs -f

# Показать последние 100 строк gateway.log
hermes logs gateway -n 100

# Показать только предупреждения и ошибки за последний час
hermes logs --level WARNING --since 1h

# Отфильтровать по конкретному сеансу
hermes logs --session abc123

# Следить за errors.log, начиная с 30 минут назад
hermes logs errors --since 30m -f

# Показать все файлы логов и их размеры
hermes logs list
```

### Фильтрация

Фильтры можно комбинировать. Когда включено несколько фильтров, строка лога должна пройти **через все** из них, чтобы быть показанной:

```bash
# Строки WARNING+ за последние 2 часа, содержащие сеанс "tg-12345"
hermes logs --level WARNING --since 2h --session tg-12345
```

Строки без разбираемой временной метки включаются и при активном `--since` (это могут быть продолжения многострочной записи). Строки без распознаваемого уровня тоже включаются при активном `--level`.

### Ротация логов

Hermes использует `RotatingFileHandler` из Python. Старые логи ротируются автоматически — ищите `agent.log.1`, `agent.log.2` и т. д. Подкоманда `hermes logs list` показывает все файлы логов, включая ротированные.

```bash

hermes skills inspect skills-sh/vercel-labs/json-render/json-render-react
hermes skills install official/migration/openclaw-migration
```

Примечания:
- `--force` может переопределять неопасные ограничения политик для сторонних и community-навыков.

## `hermes curator`

```bash
hermes curator <subcommand>
```

Curator — это фоновая задача на вспомогательной модели, которая периодически просматривает созданные агентом навыки, очищает устаревшие, объединяет пересечения и архивирует ненужные. Встроенные навыки и навыки, установленные из Hub, она никогда не трогает. Архивы можно восстановить; автоматического удаления нет.

| Подкоманда | Описание |
|------------|-------------|
| `status` | Показать состояние curator и статистику по навыкам |
| `run` | Запустить проверку curator сейчас (блокирует до завершения прохода LLM) |
| `run --background` | Запустить проход LLM в фоновом потоке и сразу вернуть управление |
| `run --dry-run` | Только предпросмотр — сгенерировать отчёт без изменений |
| `backup` | Создать вручную tar.gz-снимок `~/.hermes/skills/` (curator также делает снимок автоматически перед каждым реальным запуском) |
| `rollback` | Восстановить `~/.hermes/skills/` из снимка (по умолчанию самый свежий) |
| `rollback --list` | Показать доступные снимки |
| `rollback --id <ts>` | Восстановить конкретный снимок по ID |
| `rollback -y` | Пропустить запрос на подтверждение |
| `pause` | Приостановить curator до возобновления |
| `resume` | Возобновить приостановленный curator |
| `pin <skill>` | Закрепить навык, чтобы curator никогда не переводил его автоматически |
| `unpin <skill>` | Снять закрепление навыка |
| `restore <skill>` | Восстановить архивный навык |
| `archive <skill>` | Архивировать навык вручную |
| `prune` | Вручную удалить навыки, которые curator обычно чистит |
| `list-archived` | Показать архивные навыки (их можно восстановить через `restore`) |

На свежей установке первый плановый проход откладывается на полный `interval_hours` (по умолчанию 7 дней) — gateway не начнёт курировать сразу на первом тике после `hermes update`. Используйте `hermes curator run --dry-run`, чтобы посмотреть отчёт до того, как это произойдёт.

Поведение и конфигурация — в разделе [Curator](/docs/user-guide/features/curator).

Без `--summary` команда открывает интерактивный UI настройки инструментов по платформам.

## `hermes computer-use`

```bash
hermes computer-use <subcommand>
```

Подкоманды:

| Подкоманда | Описание |
|------------|-------------|
| `install` | Запустить upstream-установщик cua-driver (только macOS). |
| `install --upgrade` | Повторно запустить установщик, даже если cua-driver уже есть в PATH. Upstream-скрипт всегда подтягивает последний релиз, так что это выполняет обновление на месте. |
| `status` | Показать, есть ли `cua-driver` в `$PATH` и какая версия установлена. |

`hermes computer-use install` — стабильная точка входа для установки бинарника
[cua-driver](https://github.com/trycua/cua), который используется набором инструментов
`computer_use`. Он запускает тот же upstream-установщик, что и `hermes tools`
при первом включении Computer Use, поэтому его безопасно использовать повторно,
если переключатель набора инструментов не запустил установку автоматически (например, на
возвращающихся пользователях).

`hermes update` автоматически повторно запускает upstream-установщик в конце
обновления, если cua-driver уже есть в PATH, так что большинству пользователей
не нужно вызывать `--upgrade` вручную. Используйте его, когда upstream
выпускает исправление, которое нужно получить сразу, не дожидаясь следующего
обновления Hermes.

## `hermes claw`

```bash
hermes claw migrate [options]
```

Переносит вашу настройку OpenClaw в Hermes. Читает данные из `~/.openclaw` (или из пользовательского пути) и записывает их в `~/.hermes`. Автоматически определяет старые имена каталогов (`~/.clawdbot`, `~/.moltbot`) и имена файлов конфигурации (`clawdbot.json`, `moltbot.json`).

| Параметр | Описание |
|--------|-------------|
| `--dry-run` | Посмотреть, что было бы перенесено, не записывая ничего. |
| `--preset <name>` | Пресет миграции: `full` (все совместимые настройки) или `user-data` (без инфраструктурной конфигурации). Ни один пресет не импортирует секреты — для этого нужно явно указать `--migrate-secrets`. |
| `--overwrite` | Перезаписывать существующие файлы Hermes при конфликтах (по умолчанию: при наличии конфликтов применение отклоняется). |
| `--migrate-secrets` | Включить API-ключи в миграцию. Требуется даже при `--preset full`. |
| `--no-backup` | Пропустить zip-снимок `~/.hermes/` перед миграцией (по умолчанию перед применением пишется один архив-точка-восстановления в `~/.hermes/backups/pre-migration-*.zip`; его можно восстановить через `hermes import`). |
| `--source <path>` | Пользовательский каталог OpenClaw (по умолчанию: `~/.openclaw`). |
| `--workspace-target <path>` | Целевой каталог для инструкций рабочего пространства (`AGENTS.md`). |
| `--skill-conflict <mode>` | Как обрабатывать совпадения имён навыков: `skip` (по умолчанию), `overwrite` или `rename`. |
| `--yes` | Пропустить запрос на подтверждение. |

### Что переносится

Миграция охватывает 30+ категорий: persona, memory, skills, модели-провайдеры, платформы обмена сообщениями, поведение агента, политики сеансов, MCP-серверы, TTS и многое другое. Элементы либо **напрямую импортируются** в эквиваленты Hermes, либо **архивируются** для ручного просмотра.

**Импортируется напрямую:** SOUL.md, MEMORY.md, USER.md, AGENTS.md, навыки (4 исходных каталога), модель по умолчанию, пользовательские провайдеры, MCP-серверы, токены и списки разрешённых платформ обмена сообщениями (Telegram, Discord, Slack, WhatsApp, Signal, Matrix, Mattermost), параметры агента по умолчанию (уровень рассуждений, сжатие, человеческая задержка, часовой пояс, песочница), политики сброса сеанса, правила подтверждения, конфигурация TTS, настройки браузера, настройки инструментов, таймаут выполнения, списки разрешённых команд, конфигурация шлюза и API-ключи из 3 источников.

**Архивируется для ручного просмотра:** cron-задачи, плагины, hooks/webhooks, бэкенд памяти (QMD), конфиг реестра навыков, UI/identity, журналы, мультиагентная настройка, привязки каналов, IDENTITY.md, TOOLS.md, HEARTBEAT.md, BOOTSTRAP.md.

**Разрешение API-ключей** проверяет три источника в таком порядке: значения конфигурации → `~/.openclaw/.env` → `auth-profiles.json`. Все token-поля понимают обычные строки, env-шаблоны (`${VAR}`) и объекты SecretRef.

Полную карту ключей конфигурации, детали обработки SecretRef и post-migration checklist см. в [полном руководстве по миграции](/docs/guides/migrate-from-openclaw).

### Примеры

```bash
# Посмотреть, что было бы перенесено
hermes claw migrate --dry-run

# Полная миграция (все совместимые настройки, без секретов)
hermes claw migrate --preset full

# Полная миграция вместе с API-ключами
hermes claw migrate --preset full --migrate-secrets

# Перенести только пользовательские данные (без секретов), перезаписывая конфликты
hermes claw migrate --preset user-data --overwrite

# Перенести из пользовательского пути OpenClaw
hermes claw migrate --source /home/user/old-openclaw
```

## `hermes dashboard`

```bash
hermes dashboard [options]
```

Запускает веб-панель — веб-интерфейс для управления конфигурацией, API-ключами и мониторинга сеансов. Требует `pip install hermes-agent[web]` (FastAPI + Uvicorn). Встроенная вкладка Chat в браузере требует `--tui` и дополнительный `pty`. Полная документация — в разделе [«Веб-панель»](/docs/user-guide/features/web-dashboard).

| Параметр | По умолчанию | Описание |
|--------|---------|-------------|
| `--port` | `9119` | Порт, на котором будет работать web-сервер |
| `--host` | `127.0.0.1` | Адрес привязки |
| `--no-open` | — | Не открывать браузер автоматически |
| `--tui` | off | Включить вкладку Chat в браузере, запустив `hermes --tui` через мост PTY/WebSocket. Требует `pip install 'hermes-agent[web,pty]'` и POSIX PTY-окружение, например Linux, macOS или WSL2. |
| `--insecure` | off | Разрешить привязку к хостам, отличным от localhost. Раскрывает учётные данные веб-панели в сети; используйте только за доверенными средствами сетевой защиты. |
| `--stop` | — | Остановить запущенные процессы `hermes dashboard` и выйти. |
| `--status` | — | Показать запущенные процессы `hermes dashboard` и выйти. |

```bash
# По умолчанию — открывает браузер на http://127.0.0.1:9119
hermes dashboard

# Пользовательский порт, без открытия браузера
hermes dashboard --port 8080 --no-open

# Включить вкладку Chat в браузере
hermes dashboard --tui
```

## `hermes profile`

```bash
hermes profile <subcommand>
```

Управление профилями — несколькими изолированными инстансами Hermes, у каждого из которых есть собственная конфигурация, сеансы, навыки и домашний каталог.

| Подкоманда | Описание |
|------------|-------------|
| `list` | Показать все профили. |
| `use <name>` | Задать закреплённый профиль по умолчанию. |
| `create <name> [--clone] [--clone-all] [--clone-from <source>] [--no-alias]` | Создать новый профиль. `--clone` копирует конфиг, `.env` и `SOUL.md` из активного профиля. `--clone-all` копирует все состояние. `--clone-from` задаёт исходный профиль. |
| `delete <name> [-y]` | Удалить профиль. |
| `show <name>` | Показать сведения о профиле (домашний каталог, конфиг и т. п.). |
| `alias <name> [--remove] [--name NAME]` | Управлять wrapper-скриптами для быстрого доступа к профилю. |
| `rename <old> <new>` | Переименовать профиль. |
| `export <name> [-o FILE]` | Экспортировать профиль в архив `.tar.gz` (локальный backup). |
| `import <archive> [--name NAME]` | Импортировать профиль из архива `.tar.gz` (локальное восстановление). |
| `install <source> [--name N] [--alias] [--force] [-y]` | Установить дистрибутив профиля из git URL или локального каталога. |
| `update <name> [--force-config] [-y]` | Повторно подтянуть дистрибутив; пользовательские данные (memories, sessions, auth) сохраняются. |
| `info <name>` | Показать manifest дистрибутива профиля (версия, требования, источник). |

Примеры:

```bash
hermes profile list
hermes profile create work --clone
hermes profile use work
hermes profile alias work --name h-work
hermes profile export work -o work-backup.tar.gz
hermes profile import work-backup.tar.gz --name restored
hermes profile install github.com/user/my-distro --alias
hermes profile update work
hermes -p work chat -q "Hello from work profile"
```

## `hermes update`

```bash
hermes update [--check] [--backup] [--restart-gateway]
```

Подтягивает свежий код `hermes-agent` и переустанавливает зависимости в вашем venv, а затем повторно запускает post-install hooks (MCP servers, skills sync, install completion). Безопасно запускать на уже работающей установке.

**pip installs:** `hermes update` автоматически распознаёт pip-установки — он запрашивает PyPI на предмет последнего релиза и вместо `git pull` выполняет `pip install --upgrade hermes-agent`. Релизы PyPI следуют за тегированными версиями (major/minor releases), а не за каждым коммитом в `main`. Используйте `--check`, чтобы посмотреть, доступен ли более новый релиз PyPI, без установки.

| Параметр | Описание |
|--------|-------------|
| `--check` | Показать текущий commit и самый свежий commit `origin/main` рядом и выйти с кодом 0, если они совпадают, или 1, если вы отстали. Ничего не тянет, не устанавливает и не перезапускает. |
| `--backup` | Создать помеченный pre-update снимок `HERMES_HOME` (config, auth, sessions, skills, pairing data) перед pull. По умолчанию **выключено** — прежнее поведение со всегда включённым backup добавляло минуты к каждому обновлению на больших home-ах. Включить постоянно можно через `update.backup: true` в `config.yaml`. |
| `--restart-gateway` | После успешного обновления перезапустить работающую службу gateway. Если установлено несколько профилей, подразумевает поведение `--all`. |

Дополнительное поведение:

- **Снимок pairing data.** Даже если `--backup` выключен, `hermes update` перед `git pull` делает лёгкий снимок `~/.hermes/pairing/` и правил комментариев Feishu. Откатить это можно через `hermes backup restore --state pre-update`, если pull перезаписал файл, который вы редактировали.
- **Предупреждение о старом `hermes.service`.** Если Hermes обнаруживает старый systemd unit `hermes.service` (до переименования), он выводит однократную подсказку о миграции, чтобы вы избежали flap-loop проблем.
- **Коды выхода.** `0` — успех, `1` — ошибки pull/install/post-install, `2` — неожиданные изменения в рабочем дереве, которые блокируют `git pull`.
