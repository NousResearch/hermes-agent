# Справочник команд профилей

На этой странице собраны все команды, связанные с [профилями Hermes](../user-guide/profiles.md). Общий справочник CLI-команд см. в [Справочнике CLI-команд](./cli-commands.md).

## `hermes profile`

```bash
hermes profile <subcommand>
```

Основная команда для управления профилями. Если вызвать `hermes profile` без подкоманды, Hermes покажет справку.

| Подкоманда | Описание |
|------------|-------------|
| `list` | Показать все профили. |
| `use` | Назначить активный профиль по умолчанию. |
| `create` | Создать новый профиль. |
| `delete` | Удалить профиль. |
| `show` | Показать сведения о профиле. |
| `alias` | Пересоздать alias-обёртку для профиля. |
| `rename` | Переименовать профиль. |
| `export` | Экспортировать профиль в архив tar.gz. |
| `import` | Импортировать профиль из архива tar.gz. |
| `install` | Установить дистрибутив профиля из Git URL или локального каталога. См. [Дистрибутивы профилей](../user-guide/profile-distributions.md). |
| `update` | Повторно загрузить дистрибутив профиля и применить его обновление. |
| `info` | Показать метаданные дистрибутива профиля (исходный URL, commit, время последнего обновления). |

## `hermes profile list`

```bash
hermes profile list
```

Показывает все профили. Текущий активный профиль отмечается `*`.

**Пример:**

```bash
$ hermes profile list
  default
* work
  dev
  personal
```

Без параметров.

## `hermes profile use`

```bash
hermes profile use <name>
```

Назначает `<name>` активным профилем. Все последующие команды `hermes` (без `-p`) будут использовать именно его.

| Аргумент | Описание |
|----------|-------------|
| `<name>` | Имя профиля, который нужно активировать. Используйте `default`, чтобы вернуться к базовому профилю. |

**Пример:**

```bash
hermes profile use work
hermes profile use default
```

## `hermes profile create`

```bash
hermes profile create <name> [options]
```

Создаёт новый профиль.

| Аргумент / опция | Описание |
|------------------|-------------|
| `<name>` | Имя нового профиля. Должно быть допустимым именем каталога (буквы, цифры, дефисы и подчёркивания). |
| `--clone` | Скопировать `config.yaml`, `.env` и `SOUL.md` из текущего профиля. |
| `--clone-all` | Скопировать всё (конфигурацию, память, навыки, сеансы, состояние) из текущего профиля. |
| `--clone-from <profile>` | Клонировать из конкретного профиля, а не из текущего. Используется вместе с `--clone` или `--clone-all`. |
| `--no-alias` | Не создавать alias-обёртку. |
| `--description "<text>"` | Краткое описание в одно-два предложения о том, для чего этот профиль лучше всего подходит. Его использует оркестратор Kanban, чтобы распределять задачи не только по имени профиля, но и по роли. Можно пропустить и добавить позже через `hermes profile describe`. Сохраняется в `<profile_dir>/profile.yaml`. |
| `--no-skills` | Создать **пустой** профиль без включённых встроенных навыков. Записывает в профиль маркер `.no-skills`, чтобы будущие `hermes update` не подмешивали встроенный набор заново, и не позволяет сочетать его с `--clone` / `--clone-all` (которые всё равно скопировали бы навыки). Полезно для узких профилей-исполнителей или песочниц, которым не нужен полный каталог навыков. |

Создание профиля **не** делает каталог этого профиля каталогом проекта/рабочего пространства по умолчанию для терминальных команд. Если нужно, чтобы профиль сразу стартовал в конкретном проекте, задайте `terminal.cwd` в его `config.yaml`.

**Примеры:**

```bash
# Пустой профиль — потребует полной настройки
hermes profile create mybot

# Скопировать только config из текущего профиля
hermes profile create work --clone

# Скопировать всё из текущего профиля
hermes profile create backup --clone-all

# Скопировать config из конкретного профиля
hermes profile create work2 --clone --clone-from work
```

## `hermes profile describe`

```bash
hermes profile describe [<name>] [options]
```

Прочитать или задать описание профиля. Описание помогает оркестратору Kanban распределять задачи по назначению профиля, а не угадывать по его имени. Оно сохраняется в `<profile_dir>/profile.yaml`, поэтому переживает перезапуски и доступно шлюзу.

Если не передавать флаги, команда покажет текущее описание (или `(no description set for '<name>')`, если описание ещё не задано).

| Аргумент / опция | Описание |
|------------------|-------------|
| `<name>` | Профиль, который нужно описать. Обязателен, если не используется `--all --auto`. |
| `--text "<text>"` | Задать точное описание (вручную написанный текст). Перезаписывает любое существующее описание. |
| `--auto` | Автоматически сгенерировать описание из 1–2 предложений с помощью вспомогательной модели на основе установленных навыков профиля, настроенной модели и имени. Модель настраивается в `auxiliary.profile_describer` в `config.yaml`. Сгенерированные описания помечаются как `description_auto: true`, чтобы панель управления могла подсветить их на проверку. |
| `--overwrite` | Вместе с `--auto` перезаписывать и пользовательские описания (по умолчанию профили с явно заданным описанием пропускаются). |
| `--all` | Вместе с `--auto` пройтись по всем профилям без описания. |

**Примеры:**

```bash
# Прочитать текущее описание
hermes profile describe researcher

# Задать описание вручную
hermes profile describe researcher --text "Читает исходный код и оформляет выводы."

# Сгенерировать описание с помощью LLM
hermes profile describe researcher --auto

# Заполнить описания для всех профилей, у которых их ещё нет
hermes profile describe --all --auto
```

## `hermes profile delete`

```bash
hermes profile delete <name> [options]
```

Удаляет профиль и убирает его alias-обёртку.

| Аргумент / опция | Описание |
|------------------|-------------|
| `<name>` | Профиль, который нужно удалить. |
| `--yes`, `-y` | Пропустить запрос на подтверждение. |

**Пример:**

```bash
hermes profile delete mybot
hermes profile delete mybot --yes
```

:::warning
Команда безвозвратно удаляет весь каталог профиля, включая всю конфигурацию, память, сеансы и навыки. Текущий активный профиль удалить нельзя.
:::

## `hermes profile show`

```bash
hermes profile show <name>
```

Показывает сведения о профиле: домашний каталог, настроенную модель, статус шлюза, количество навыков и состояние файла конфигурации.

Здесь выводится именно домашний каталог Hermes для профиля, а не рабочий каталог терминала. Терминальные команды стартуют из `terminal.cwd` (или из каталога запуска на локальном бэкенде, если `cwd: "."`).

| Аргумент | Описание |
|----------|-------------|
| `<name>` | Профиль, который нужно посмотреть. |

**Пример:**

```bash
$ hermes profile show work
Profile: work
Path:    ~/.hermes/profiles/work
Model:   anthropic/claude-sonnet-4 (anthropic)
Шлюз: остановлен
Skills:  12
.env:    exists
SOUL.md: exists
Alias:   ~/.local/bin/work
```

## `hermes profile alias`

```bash
hermes profile alias <name> [options]
```

Пересоздаёт alias-обёртку в `~/.local/bin/<name>`. Полезно, если alias случайно удалили или если его нужно обновить после переноса установки Hermes.

| Аргумент / опция | Описание |
|------------------|-------------|
| `<name>` | Профиль, для которого нужно создать или обновить alias. |
| `--remove` | Удалить alias-обёртку вместо создания. |
| `--name <alias>` | Задать имя alias вручную (по умолчанию используется имя профиля). |

**Пример:**

```bash
hermes profile alias work
# Создаёт или обновляет ~/.local/bin/work

hermes profile alias work --name mywork
# Создаёт ~/.local/bin/mywork

hermes profile alias work --remove
# Удаляет alias-обёртку
```

## `hermes profile rename`

```bash
hermes profile rename <old-name> <new-name>
```

Переименовывает профиль. Обновляет каталог и alias-обёртку.

| Аргумент | Описание |
|----------|-------------|
| `<old-name>` | Текущее имя профиля. |
| `<new-name>` | Новое имя профиля. |

**Пример:**

```bash
hermes profile rename mybot assistant
# ~/.hermes/profiles/mybot → ~/.hermes/profiles/assistant
# ~/.local/bin/mybot → ~/.local/bin/assistant
```

## `hermes profile export`

```bash
hermes profile export <name> [options]
```

Экспортирует профиль в сжатый архив tar.gz.

| Аргумент / опция | Описание |
|------------------|-------------|
| `<name>` | Профиль, который нужно экспортировать. |
| `-o`, `--output <path>` | Путь к выходному файлу (по умолчанию: `<name>.tar.gz`). |

**Пример:**

```bash
hermes profile export work
# Создаёт work.tar.gz в текущем каталоге

hermes profile export work -o ./work-2026-03-29.tar.gz
```

## `hermes profile import`

```bash
hermes profile import <archive> [options]
```

Импортирует профиль из архива tar.gz.

| Аргумент / опция | Описание |
|------------------|-------------|
| `<archive>` | Путь к архиву tar.gz, который нужно импортировать. |
| `--name <name>` | Имя импортируемого профиля (по умолчанию: выводится из архива). |

**Пример:**

```bash
hermes profile import ./work-2026-03-29.tar.gz
# Имя профиля выводится из архива

hermes profile import ./work-2026-03-29.tar.gz --name work-restored
```

## Команды дистрибутива

:::tip
**Впервые работаете с дистрибутивами?** Начните с [руководства пользователя по дистрибутивам профилей](../user-guide/profile-distributions.md) — там с примерами разобрано, зачем это нужно, когда это полезно и как этим пользоваться. Раздел ниже — сухой CLI-справочник на случай, если вы уже знаете, что хотите сделать.
:::

Дистрибутив превращает профиль в общий, версионируемый артефакт, опубликованный
как **git-репозиторий**. Получатель устанавливает его одной командой и потом
может обновлять его на месте, не затрагивая локальную память, сеансы и учётные данные.

`auth.json` и `.env` никогда не входят в дистрибутив — они остаются на машине устанавливающего пользователя.

Пользовательские данные получателя (память, сеансы, учётные данные, собственные
правки в `.env`) всегда сохраняются и при первой установке, и при последующих
обновлениях.

:::info
`hermes profile export` / `import` по-прежнему нужны для **локального резервного копирования и восстановления** профиля на вашей машине. Distribution (`install` / `update` / `info`) — это отдельный сценарий: вы публикуете профиль через git, а кто-то другой потом ставит его у себя.
:::

### `hermes profile install`

```bash
hermes profile install <source> [--name <name>] [--alias] [--force] [--yes]
```

Устанавливает дистрибутив профиля из git URL или локального каталога.

| Опция | Описание |
|-------|-------------|
| `<source>` | Git URL (`github.com/user/repo`, `https://...`, `git@...`, `ssh://`, `git://`) или локальный каталог, в корне которого лежит `distribution.yaml`. |
| `--name NAME` | Переопределить имя профиля из манифеста. |
| `--alias` | Также создать alias-обёртку (например, `telemetry` → `hermes -p telemetry`). |
| `--force` | Перезаписать уже существующий профиль с тем же именем. Пользовательские данные при этом всё равно сохраняются. |
| `-y`, `--yes` | Пропустить подтверждение с предпросмотром манифеста. |

Установщик показывает манифест, перечисляет требуемые переменные окружения и предупреждает
о cron-задачах перед запросом подтверждения. Требуемые переменные окружения попадают в файл
`.env.EXAMPLE`, который затем нужно скопировать в `.env` и заполнить.

**Примеры:**

```bash
# Установить из GitHub-репозитория (сокращённая запись)
hermes profile install github.com/kyle/telemetry-distribution --alias

# Установить по полному HTTPS git URL
hermes profile install https://github.com/kyle/telemetry-distribution.git

# Установить по SSH
hermes profile install git@github.com:kyle/telemetry-distribution.git

# Установить из локального каталога во время разработки
hermes profile install ./telemetry/
```

### `hermes profile update`

```bash
hermes profile update <name> [--force-config] [--yes]
```

Повторно клонирует дистрибутив из записанного источника и применяет обновления.
Файлы, принадлежащие дистрибутиву (SOUL.md, skills/, cron/, mcp.json),
перезаписываются; пользовательские данные (память, сеансы, учётные данные, `.env`) никогда не трогаются.

`config.yaml` по умолчанию сохраняется, чтобы не затирать локальные переопределения.
Если нужно сбросить его к версии, поставляемой дистрибутивом, используйте `--force-config`.

### `hermes profile info`

```bash
hermes profile info <name>
```

Печатает манифест дистрибутива профиля: имя, версию, требуемую версию Hermes, автора, требования к переменным окружения, URL/путь источника и временную метку `Installed:`, которую Hermes записал при последней операции `install` или `update`. Это полезно, чтобы понять, что нужно конкретному общему профилю ещё до установки, и чтобы заметить, что профиль был установлен полгода назад и с тех пор не обновлялся.

`hermes profile list` также показывает имя и версию дистрибутива в колонке `Distribution`, а `hermes profile show <name>` / `delete <name>` выводят URL источника, чтобы сразу было видно, пришёл ли профиль из git-репозитория или был создан локально.

### Приватные дистрибутивы

Приватный git-репозиторий можно использовать как источник дистрибутива без
дополнительной настройки — установка вызывает ваш обычный `git`, поэтому
используется та аутентификация, которая уже настроена в shell (SSH key,
git credential helper, сохранённые HTTPS-учётные данные GitHub CLI и т. п.).

```bash
# Использует ваш SSH key, как и любой другой `git clone`
hermes profile install git@github.com:your-org/internal-assistant.git

# Использует ваш git credential helper
hermes profile install https://github.com/your-org/internal-assistant.git
```

Если во время установки clone запрашивает учётные данные интерактивно в терминале,
Hermes просто пропускает этот запрос наружу. Сначала настройте аутентификацию так, как
вы обычно настраиваете `git clone` для того же репозитория, а уже потом ставьте
дистрибутив.

### Манифест дистрибутива (`distribution.yaml`)

У каждого дистрибутива в корне репозитория есть файл `distribution.yaml`:

```yaml
name: telemetry
version: 0.1.0
description: "Compliance monitoring harness"
hermes_requires: ">=0.12.0"
author: "Your Name"
license: "MIT"
env_requires:
  - name: OPENAI_API_KEY
    description: "API-ключ OpenAI"
    required: true
  - name: GRAPHITI_MCP_URL
    description: "URL графа памяти"
    required: false
    default: "http://127.0.0.1:8000/sse"
distribution_owned:   # необязательно; по умолчанию: SOUL.md, config.yaml,
                      #   mcp.json, skills/, cron/, distribution.yaml
  - SOUL.md
  - skills/compliance/
  - cron/
```

`hermes_requires` поддерживает `>=`, `<=`, `==`, `!=`, `>`, `<` или просто
версию без префикса (это означает `>=`). Установка завершается с понятной
ошибкой, если текущая версия Hermes не соответствует ограничению.

`distribution_owned` — необязательный параметр. Если он задан, при обновлении
перезаписываются только перечисленные пути; всё остальное в профиле остаётся
пользовательским. Если его нет, применяются значения по умолчанию, указанные выше.

### Публикация дистрибутива

Создать дистрибутив — значит просто отправить git push:

1. В каталоге профиля создайте `distribution.yaml` как минимум с полями `name`
   и `version`.
2. Инициализируйте git repo (или используйте уже существующий) и отправьте его
   на GitHub / GitLab / любой другой хост, откуда Hermes умеет клонировать.
3. Скажите получателям запускать `hermes profile install <your-repo-url>`.

Для версионированных релизов используйте git tags — те, кто клонирует `HEAD`,
получат актуальное состояние, а версию в манифесте всегда можно поднять вручную.

## `hermes -p` / `hermes --profile`

```bash
hermes -p <name> <command> [options]
hermes --profile <name> <command> [options]
```

Глобальный флаг, который запускает любую команду Hermes под конкретным профилем,
не меняя sticky default. Он переопределяет активный профиль только на время
текущего запуска.

| Опция | Описание |
|-------|-------------|
| `-p <name>`, `--profile <name>` | Профиль, который нужно использовать для этой команды. |

**Примеры:**

```bash
hermes -p work chat -q "Check the server status"
hermes --profile dev gateway start
hermes -p personal skills list
hermes -p work config edit
```

## `hermes completion`

```bash
hermes completion <shell>
```

Генерирует скрипты автодополнения оболочки. В них входят и автодополнения для имён профилей, и автодополнения для подкоманд профилей.

| Аргумент | Описание |
|----------|-------------|
| `<shell>` | Shell, для которого нужно сгенерировать completion: `bash`, `zsh` или `fish`. |

**Примеры:**

```bash
# Установить completion
hermes completion bash >> ~/.bashrc
hermes completion zsh >> ~/.zshrc
hermes completion fish > ~/.config/fish/completions/hermes.fish

# Перезагрузить shell
source ~/.bashrc
```

После установки автодополнение по Tab работает для:
- `hermes profile <TAB>` — подкоманды (list, use, create и т. д.)
- `hermes profile use <TAB>` — имена профилей
- `hermes -p <TAB>` — имена профилей
