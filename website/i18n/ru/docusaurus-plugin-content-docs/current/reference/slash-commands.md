# Справочник слэш-команд

У Hermes есть два интерфейса слэш-команд, и оба работают через центральный `COMMAND_REGISTRY` в `hermes_cli/commands.py`:

- **Интерактивные CLI слэш-команды** — обрабатываются `cli.py`, автодополнение берётся из реестра
- **Слэш-команды платформ обмена сообщениями** — обрабатываются `gateway/run.py`, текст помощи и меню платформ генерируются из реестра

Установленные навыки тоже доступны как динамические слэш-команды в обоих интерфейсах. Сюда входят встроенные навыки вроде `/plan`, который открывает режим планирования и сохраняет планы в Markdown в `.hermes/plans/` внутри активной рабочей директории `workspace/backend`.

## Права доступа и разделение admin/user {#permissions-and-adminuser-split}

Для платформ, где список разрешённых пользователей настраивается отдельно для каждой области доступа (Telegram, Discord, Slack, Matrix, Mattermost, Signal и т. д.), предусмотрено двухуровневое разделение слэш-команд: **администраторы** видят все зарегистрированные команды, а **обычные пользователи** — только команды из `user_allowed_commands` плюс всегда доступные `/help` и `/whoami`. Настройте `allow_admin_from` и `user_allowed_commands` (а для групп — `group_allow_admin_from` и `group_user_allowed_commands`) в блоке `extra:` конфигурации платформы в `~/.hermes/gateway-config.yaml`.

Смотрите примеры в документации каждой платформы — структура везде одинаковая:

- [Telegram](/docs/user-guide/messaging/telegram#slash-command-access-control)
- [Discord](/docs/user-guide/messaging/discord)
- [Slack](/docs/user-guide/messaging/slack)
- [Matrix](/docs/user-guide/messaging/matrix)
- [Mattermost](/docs/user-guide/messaging/mattermost)
- [Signal](/docs/user-guide/messaging/signal)

Если `allow_admin_from` не задан для конкретного scope, этот scope остаётся в режиме обратной совместимости без разделения уровней — любой разрешённый пользователь может запускать любую команду.

## Interactive CLI слэш-команды

Введите `/` в CLI, чтобы открыть меню автодополнения. Встроенные команды не чувствительны к регистру.

### Сеанс

| Команда | Описание |
|---------|-------------|
| `/new [name]` (alias: `/reset`) | Начать новый сеанс (новый ID сеанса + пустая история). Необязательный `[name]` задаёт начальный заголовок сеанса - например, `/new my-experiment` откроет свежий сеанс уже с названием `my-experiment`, и потом его будет проще найти через `/resume` или `/sessions`. |
| `/clear` | Очистить экран и начать новый сеанс |
| `/history` | Показать историю разговора |
| `/save` | Сохранить текущий разговор |
| `/retry` | Повторить последнее сообщение (отправить агенту ещё раз) |
| `/undo` | Удалить последнее сообщение пользователя/ответ ассистента |
| `/title` | Задать заголовок текущего сеанса (usage: /title My Session Name) |
| `/compress [focus topic]` | Вручную сжать контекст разговора (flush memories + summary). Необязательная тема фокуса сужает то, что сохранит краткое изложение. |
| `/rollback` | Показать или восстановить контрольные точки файловой системы (usage: /rollback [number]) |
| `/snapshot [create\|restore <id>\|prune]` (alias: `/snap`) | Создать или восстановить снимки состояния Hermes config/state. `create [label]` сохраняет snapshot, `restore <id>` откатывает к нему, `prune [N]` удаляет старые снимки, а без аргументов выводится список. |
| `/stop` | Завершить все запущенные фоновые процессы |
| `/queue <prompt>` (alias: `/q`) | Поставить запрос в очередь на следующий ход (не прерывает текущий ответ агента). |
| `/steer <prompt>` | Вставить заметку посреди выполнения - она попадёт к агенту **после следующего вызова инструмента**, без прерывания и без нового пользовательского хода. Текст добавляется к содержимому последнего tool result после завершения текущего инструмента, чтобы дать агенту новый контекст без разрыва текущего цикла вызовов инструментов. Используйте это, чтобы мягко скорректировать направление посреди задачи (например, "сфокусируйся на auth module", пока агент гоняет тесты). |
| `/goal <text>` | Задать постоянную цель, к которой Hermes будет идти между ходами. После каждого хода проверяющая модель определяет, достигнута ли цель; если нет, Hermes продолжает автоматически. Это наш вариант цикла Ralph. Подкоманды: `/goal status`, `/goal pause`, `/goal resume`, `/goal clear`. По умолчанию доступно до 20 ходов (`goals.max_turns`); любое реальное сообщение пользователя прерывает цикл продолжения, а состояние сохраняется через `/resume`. См. [Persistent Goals](/docs/user-guide/features/goals) для полного разбора. |
| `/subgoal <text>` | Добавить пользовательский критерий к активной цели посреди цикла. Промпт продолжения показывает все подцели агенту дословно, а judge учитывает их при вердикте DONE/CONTINUE - то есть цель не считается выполненной, пока не выполнены и исходная цель, и каждая подцель. Подкоманды: `/subgoal` (список), `/subgoal remove <N>`, `/subgoal clear`. Требуется активный `/goal`. |
| `/resume [name]` | Возобновить ранее названный сеанс |
| `/sessions` | Просмотреть и возобновить прошлые сеансы через интерактивный picker |
| `/redraw` | Принудительно перерисовать весь интерфейс (помогает при дрейфе терминала после resize в tmux, артефактах mouse selection и т. п.) |
| `/status` | Показать сведения о сеансе — модель, провайдер, профиль, ID сеанса, рабочий каталог, заголовок, временные метки создания и обновления, суммарные токены, состояние работы агента — а затем локальный блок **Краткая сводка сеанса** (свежие счётчики пользовательских и ассистентских ходов, число результатов инструментов, самые часто используемые инструменты, последние несколько затронутых файлов, последний запрос пользователя и последний ответ ассистента). Сводка строится локально из памяти о разговоре, без вызова LLM и без влияния на кэш системной инструкции. |
| `/agents` (alias: `/tasks`) | Показать активных агентов и выполняющиеся задачи в текущем сеансе. |
| `/background <prompt>` (alias: `/bg`, `/btw`) | Запустить запрос в отдельном фоновом сеансе. Агент обрабатывает запрос независимо - текущий сеанс остаётся свободным для другой работы. Результаты появляются в панели, когда задача завершается. См. [CLI Background Sessions](/docs/user-guide/cli#background-sessions). |
| `/branch [name]` (alias: `/fork`) | Разветвить текущий сеанс (исследовать другой путь) |
| `/handoff <platform>` | **Только CLI.** Передать текущий сеанс на платформу обмена сообщениями (Telegram, Discord, Slack, WhatsApp, Signal, Matrix). Шлюз сразу подхватывает его, создаёт новый тред там, где это поддерживается (Telegram topics, Discord text-channel threads, Slack message-anchored threads), перепривязывает чат назначения к вашему `session_id`, чтобы полностью восстановить транскрипт с учётом ролей, и создаёт синтетический пользовательский ход, чтобы агент подтвердил перенос. После успешной передачи CLI завершается корректно и показывает подсказку `/resume`; вернуться локально можно в любой момент через `/resume <title>`. Команда отклоняется, если выполнять её в середине хода. Требуется запущенный шлюз и настроенный домашний канал для целевой платформы (`/sethome` из чата назначения). См. [Кроссплатформенная передача](/docs/user-guide/sessions#cross-platform-handoff). |

### Конфигурация

| Команда | Описание |
|---------|-------------|
| `/config` | Показать текущую конфигурацию |
| `/model [model-name]` | Показать или изменить текущую модель. Действует со следующего сеанса. |
| `/personality` | Задать предустановленную личность |
| `/verbose` | Переключать отображение прогресса инструментов: off → new → all → verbose. Можно [включить для messaging](#примечания) через config. |

| `/skin` | Показать или изменить skin/theme интерфейса |
| `/statusbar` (alias: `/sb`) | Включить или выключить строку состояния context/model |
| `/voice [on\|off\|tts\|status]` | Переключить CLI voice mode и озвучивание ответа. Запись использует `voice.record_key` (по умолчанию: `Ctrl+B`). |
| `/yolo` | Переключить YOLO mode - пропустить все опасные запросы на подтверждение команд. |
| `/footer [on\|off\|status]` | Переключить runtime-metadata footer в финальных ответах (показывает model, число инструментов, timing). |
| `/busy [queue\|steer\|interrupt\|status]` | Только CLI: управлять тем, что делает Enter, пока Hermes работает, — поставить новый запрос в очередь, передать его в текущий ход или прервать сразу. |
| `/indicator [kaomoji\|emoji\|unicode\|ascii]` | Только CLI: выбрать стиль busy-indicator в TUI. |

### Быстрые команды

Пользовательские быстрые команды связывают короткую слэш-команду либо с shell-командой, либо с другой слэш-командой. Настройте их в `~/.hermes/config.yaml`:

```yaml
quick_commands:
  status:
    type: exec
    command: systemctl status hermes-agent
  deploy:
    type: exec
    command: scripts/deploy.sh
  inbox:
    type: alias
    target: /gmail unread
```

Затем вводите `/status`, `/deploy` или `/inbox` в CLI или на платформе обмена сообщениями. Быстрые команды разрешаются в момент выполнения и могут не отображаться в каждой встроенной таблице автодополнения/помощи.

Команды-ярлыки, состоящие только из одной строки запроса, не поддерживаются как быстрые команды. Если сценарий нужно переиспользовать, храните его в навыке или используйте `type: alias`, чтобы сослаться на уже существующую слэш-команду.

### Пользовательские алиасы моделей

Задайте свои короткие имена для моделей, которыми вы пользуетесь часто, и вызывайте их через `/model <alias>` в CLI или на любой платформе обмена сообщениями. Алиасы работают одинаково и там, и там, как для настроек только текущего сеанса (по умолчанию), так и для переключателей `--global`.

Поддерживаются два формата config:

**Полная форма** — жёстко задаёт точную модель, провайдера и, при необходимости, base URL. Поместите это в `~/.hermes/config.yaml`:

```yaml
model_aliases:
  fav:
    model: claude-sonnet-4.6
    provider: anthropic
  grok:
    model: grok-4
    provider: x-ai
  ollama-qwen:
    model: qwen3-coder:30b
    provider: custom
    base_url: http://localhost:11434/v1
```

**Короткая форма** — `provider/model` в одной строке. Задавайте её из shell без редактирования YAML:

```bash
hermes config set model.aliases.fav anthropic/claude-opus-4.6
hermes config set model.aliases.grok x-ai/grok-4
```

Потом в чате:

```
/model fav            # только для текущего сеанса
/model grok --global  # также сохраняет change current-model в config.yaml
```

Пользовательские алиасы имеют приоритет над встроенными короткими именами, поэтому алиас вроде `sonnet`, `kimi`, `opus` и т. п. затмит встроенный. Имена алиасов не чувствительны к регистру.

### Разрешение алиасов

Команды поддерживают поиск по префиксу: ввод `/h` разрешается в `/help`, `/mod` - в `/model`. Когда префикс неоднозначен (совпадает с несколькими командами), побеждает первое совпадение в порядке реестра. Полные имена команд и зарегистрированные алиасы всегда важнее совпадений по префиксу.

## Слэш-команды платформ обмена сообщениями

Шлюз поддерживает следующие встроенные команды в чатах Telegram, Discord, Slack, WhatsApp, Signal, Email, Home Assistant и Teams:

| Команда | Описание |
|---------|-------------|
| `/new` | Начать новый разговор. |
| `/reset` | Сбросить историю разговора. |
| `/status` | Показать сведения о сеансе, а затем локальный блок **Краткая сводка сеанса** (свежие счётчики ходов, самые часто используемые инструменты, затронутые файлы, последний запрос и последний ответ). |
| `/stop` | Завершить все запущенные фоновые процессы и прервать работающего агента. |
| `/model [provider:model]` | Показать или изменить модель. Поддерживает переключение провайдера (`/model zai:glm-5`), собственные конечные точки (`/model custom:model`), именованных пользовательских провайдеров (`/model custom:local:qwen`), auto-detect (`/model custom`) и пользовательские алиасы (`/model fav`, `/model grok` — см. ## Примечания

- `/skin`, `/snapshot`, `/gquota`, `/reload`, `/tools`, `/toolsets`, `/browser`, `/config`, `/cron`, `/skills`, `/platforms`, `/paste`, `/image`, `/statusbar`, `/plugins`, `/busy`, `/indicator`, `/redraw`, `/clear`, `/history`, `/save`, `/copy`, `/handoff` и `/quit` - это **только CLI** команды.
- `/verbose` по умолчанию **только CLI**, но его можно включить и для платформ обмена сообщениями через `display.tool_progress_command: true` в `config.yaml`. Когда он включён, он переключает режим `display.tool_progress` и сохраняет изменение в config.
- `/sethome`, `/update`, `/restart`, `/approve`, `/deny`, `/topic` и `/commands` - это **только messaging** команды.
- `/status`, `/background`, `/queue`, `/steer`, `/voice`, `/reload-mcp`, `/reload-skills`, `/rollback`, `/debug`, `/fast`, `/footer`, `/curator`, `/kanban`, `/sessions` и `/yolo` работают и в **CLI**, и в шлюзе обмена сообщениями.
- `/voice join`, `/voice channel` и `/voice leave` имеют смысл только в Discord.

## Подтверждение перед разрушительными командами

CLI запрашивает подтверждение перед запуском слэш-команд, которые выбрасывают несохранённое состояние сеанса. Сейчас к разрушительным относятся:

| Команда | Что она уничтожает |
|---------|------------------|
| `/clear` | Очищает экран и начинает новый сеанс - текущий ID сеанса и история в памяти исчезают. |
| `/new` / `/reset` | Начинает свежий сеанс (новый ID сеанса + пустая история). |
| `/undo` | Удаляет последний обмен user/assistant из истории. |
| `/exit --delete` / `/quit --delete` | Выходит **и** навсегда удаляет SQLite-историю текущего сеанса и transcripts на диске. |

Для каждой из этих команд CLI открывает модальное окно с тремя вариантами: **Approve Once** (продолжить один раз), **Always Approve** (продолжить и сохранить `approvals.destructive_slash_confirm: false`, чтобы в будущем разрушительные команды запускались без подтверждения) или **Cancel**.

Установите `approvals.destructive_slash_confirm: false` в `~/.hermes/config.yaml`, чтобы отключить подтверждения глобально; верните значение `true`, чтобы снова их включить. За контекстом см. [Подтверждение опасных слэш-команд](/docs/user-guide/security#dangerous-command-approval).
