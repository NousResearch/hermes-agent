# Конфигурация

Все настройки хранятся в каталоге `~/.hermes/`, чтобы к ним было удобно обращаться.

## Структура каталогов

```text
~/.hermes/
├── config.yaml     # Настройки (модель, терминал, TTS, сжатие и т. д.)
├── .env            # API-ключи и секреты
├── auth.json       # Учётные данные OAuth-провайдеров (Nous Portal и др.)
├── SOUL.md         # Основная идентичность агента (слот #1 в системной инструкции)
├── memories/       # Постоянная память (MEMORY.md, USER.md)
├── skills/         # Навыки, созданные агентом (управляются через skill_manage)
├── cron/           # Запланированные задачи
├── sessions/       # Сеансы шлюза
└── logs/           # Журналы (errors.log, gateway.log — секреты автоматически маскируются)
```

## Управление конфигурацией

```bash
hermes config              # Показать текущую конфигурацию
hermes config edit         # Открыть config.yaml в редакторе
hermes config set KEY VAL  # Задать конкретное значение
hermes config check        # Проверить, не пропущены ли какие-то параметры (после обновлений)
hermes config migrate      # Интерактивно добавить недостающие параметры

# Примеры:
hermes config set model anthropic/claude-opus-4
hermes config set terminal.backend docker
hermes config set OPENROUTER_API_KEY sk-or-...  # Сохраняет в `.env`
```

:::tip
Команда `hermes config set` автоматически отправляет значения в нужный файл — API-ключи сохраняются в `.env`, всё остальное попадает в `config.yaml`.
:::

## Приоритет конфигурации

Параметры применяются в таком порядке (от наивысшего приоритета к наименьшему):

1. **Аргументы CLI** — например, `hermes chat --model anthropic/claude-sonnet-4` (переопределение только на один запуск)
2. **`~/.hermes/config.yaml`** — основной файл конфигурации для всех не секретных настроек
3. **`~/.hermes/.env`** — резервный источник для переменных окружения; **обязателен** для секретов (API-ключей, токенов, паролей)
4. **Встроенные значения по умолчанию** — безопасные жёстко заданные значения, если больше ничего не указано

:::info На практике
Секреты (API-ключи, токены ботов, пароли) кладите в `.env`. Всё остальное (модель, бэкенд терминала, параметры сжатия, лимиты памяти, наборы инструментов) храните в `config.yaml`. Если заданы оба варианта, для не секретных настроек приоритет остаётся за `config.yaml`.
:::

## Подстановка переменных окружения

В `config.yaml` можно ссылаться на переменные окружения через синтаксис `${VAR_NAME}`:

```yaml
auxiliary:
  vision:
    api_key: ${GOOGLE_API_KEY}
    base_url: ${CUSTOM_VISION_URL}

delegation:
  api_key: ${DELEGATION_KEY}
```

Можно использовать несколько подстановок в одном значении: `url: "${HOST}:${PORT}"`. Если указанная переменная не задана, значение-заглушка остаётся как есть (`${UNDEFINED_VAR}` не разворачивается). Поддерживается только синтаксис `${VAR}` — голый `$VAR` не подставляется.

Для настройки провайдеров ИИ (OpenRouter, Anthropic, Copilot, собственные конечные точки, самостоятельно размещаемые LLM, fallback-модели и т. д.) см. [Провайдеры ИИ](/integrations/providers).

### Таймауты провайдеров

Можно задать `providers.<id>.request_timeout_seconds` как общий таймаут запроса для провайдера, а также `providers.<id>.models.<model>.timeout_seconds` как переопределение для конкретной модели. Это применяется к основному клиенту запросов на любом транспортном слое (OpenAI-wire, native Anthropic, Anthropic-compatible), к цепочке fallback, к повторной сборке после ротации учётных данных и, для OpenAI-wire, к параметру `timeout` на каждый запрос — так что значение из конфигурации имеет приоритет над устаревшей переменной окружения `HERMES_API_TIMEOUT`.

Также можно задать `providers.<id>.stale_timeout_seconds` для обнаружения зависших вызовов без потоковой передачи и `providers.<id>.models.<model>.stale_timeout_seconds` для переопределения на уровне модели. Это значение имеет приоритет над устаревшей переменной окружения `HERMES_API_CALL_STALE_TIMEOUT`.

Если оставить эти параметры пустыми, будут использованы устаревшие значения по умолчанию (`HERMES_API_TIMEOUT=1800`s, `HERMES_API_CALL_STALE_TIMEOUT=300`s, native Anthropic 900s). Для AWS Bedrock это пока не подключено (и `bedrock_converse`, и пути AnthropicBedrock SDK используют boto3 со своей конфигурацией таймаутов). См. закомментированный пример в [`cli-config.yaml.example`](https://github.com/NousResearch/hermes-agent/blob/main/cli-config.yaml.example).

## Конфигурация терминального бэкенда

Hermes поддерживает семь терминальных бэкендов. Каждый из них определяет, где выполняются shell-команды агента: на вашей локальной машине, в контейнере Docker, на удалённом сервере через SSH, в облачной песочнице Modal напрямую или через шлюз, управляемый Nous, в рабочем пространстве Daytona, в Vercel Sandbox или в контейнере Singularity/Apptainer.

```yaml
terminal:
  backend: local    # local | docker | ssh | modal | daytona | vercel_sandbox | singularity
  cwd: "."          # Рабочий каталог gateway/cron (CLI всегда использует каталог запуска)
  timeout: 180      # Таймаут одной команды в секундах
  env_passthrough: []  # Имена переменных окружения, передаваемых в sandboxed execution (terminal + execute_code)
  singularity_image: "docker://nikolaik/python-nodejs:python3.11-nodejs20"  # Образ контейнера для бэкенда Singularity
  modal_image: "nikolaik/python-nodejs:python3.11-nodejs20"                 # Образ контейнера для бэкенда Modal
  daytona_image: "nikolaik/python-nodejs:python3.11-nodejs20"               # Образ контейнера для бэкенда Daytona
```

Для облачных песочниц, таких как Modal, Daytona и Vercel Sandbox, `container_persistent: true` означает, что Hermes будет стараться сохранить состояние файловой системы при повторном создании песочницы. Это не означает, что будет восстановлена та же самая песочница, пространство PID или фоновые процессы.

### Обзор бэкендов

| Бэкенд | Где выполняются команды | Изоляция | Лучше всего подходит для |
|---------|------------------------|-----------|--------------------------|
| **local** | Непосредственно на вашей машине | Нет | Разработки и личного использования |
| **docker** | Один постоянный контейнер Docker (общий для сеанса, `/new` и субагентов) | Полная (пространства имён, отключение capabilities) | Безопасной изоляции, CI/CD |
| **ssh** | На удалённом сервере через SSH | Сетевая граница | Удалённой разработки, мощного железа |
| **modal** | Облачная песочница Modal | Полная (облачная VM) | Временных облачных вычислений, оценочных запусков |
| **daytona** | Рабочее пространство Daytona | Полная (облачный контейнер) | Управляемых облачных сред разработки |
| **vercel_sandbox** | Vercel Sandbox | Полная (облачная microVM) | Облачного выполнения с сохранением файловой системы через snapshots |
| **singularity** | Контейнер Singularity/Apptainer | Пространства имён (`--containall`) | HPC-кластеров и общих машин |

### Локальный бэкенд

Вариант по умолчанию. Команды выполняются прямо на вашей машине без изоляции. Дополнительная настройка не нужна.

```yaml
terminal:
  backend: local
```

:::warning
Агент получает тот же доступ к файловой системе, что и ваша учётная запись. Используйте `hermes tools`, чтобы отключить ненужные инструменты, или переключитесь на Docker, если нужна изоляция.
:::

### Бэкенд Docker {#docker-backend}

Запускает команды внутри контейнера Docker с усиленной защитой (все capabilities отключены, эскалации привилегий нет, PID ограничены).

**Один постоянный контейнер, а не контейнер на каждую команду.** Hermes при первом использовании поднимает ОДИН долгоживущий контейнер и направляет через `docker exec` в него все вызовы terminal, file и `execute_code` — в рамках сеансов, `/new`, `/reset` и субагентов через `delegate_task` — на всё время жизни процесса Hermes. Смена рабочего каталога, установленные пакеты и файлы в `/workspace` сохраняются от одного вызова инструмента к другому, как в локальной оболочке. При завершении работы контейнер останавливается и удаляется. Подробности см. в разделе **Жизненный цикл контейнера** ниже.

```yaml
terminal:
  backend: docker
  docker_image: "nikolaik/python-nodejs:python3.11-nodejs20"
  docker_mount_cwd_to_workspace: false  # Примонтировать каталог запуска в /workspace
  docker_run_as_host_user: false   # См. ниже раздел "Запуск контейнера от имени пользователя хоста"
  docker_forward_env:              # Переменные окружения, которые нужно передать в контейнер
    - "GITHUB_TOKEN"
  docker_volumes:                  # Примонтированные каталоги хоста
    - "/home/user/projects:/workspace/projects"
    - "/home/user/data:/data:ro"   # Только чтение
  docker_extra_args:               # Дополнительные флаги, добавляемые к `docker run` без изменений
    - "--gpus=all"
    - "--network=host"

  # Ограничения ресурсов
  container_cpu: 1                 # Ядра CPU (0 = без ограничений)
  container_memory: 5120           # МБ (0 = без ограничений)
  container_disk: 51200            # МБ (требуется overlay2 на XFS+pquota)
  container_persistent: true       # Сохранять /workspace и /root между сеансами
```

**`terminal.docker_extra_args`** (также можно переопределить через `TERMINAL_DOCKER_EXTRA_ARGS='["--gpus=all"]'`) позволяет передавать произвольные флаги `docker run`, которые Hermes не выводит как отдельные ключи, например `--gpus`, `--network`, `--add-host`, альтернативные переопределения `--security-opt` и т. п. Каждый элемент должен быть строкой; список добавляется в конец итогового вызова `docker run`, так что при необходимости может переопределить значения по умолчанию Hermes. Используйте это осторожно: флаги, которые обходят усиление безопасности песочницы (отключение capabilities, `--user`, bind mount рабочего каталога), незаметно ослабят изоляцию.

**Требования:** установленный и запущенный Docker Desktop или Docker Engine. Hermes проверяет `$PATH`, а также типичные macOS-пути установки (`/usr/local/bin/docker`, `/opt/homebrew/bin/docker`, bundle приложения Docker Desktop). Поддерживается и Podman: задайте `HERMES_DOCKER_BINARY=podman` (или полный путь), чтобы принудительно использовать его, если установлены оба варианта.

**Жизненный цикл контейнера:** Hermes переиспользует один долгоживущий контейнер (`docker run -d ... sleep 2h`) для всех вызовов terminal и file-tool, в том числе внутри сеансов, `/new`, `/reset` и субагентов через `delegate_task`, на всё время жизни процесса Hermes. Команды запускаются через `docker exec` с login shell, поэтому смена рабочего каталога, установленные пакеты и файлы в `/workspace` сохраняются от одного вызова к другому. Контейнер останавливается и удаляется при завершении Hermes (или когда idle-sweep его забирает).

Параллельные субагенты, запущенные через `delegate_task(tasks=[...])`, используют один и тот же контейнер — одновременные `cd`, изменения окружения и записи в один и тот же путь будут конфликтовать. Если субагенту нужна изолированная песочница, он должен переопределить Docker-образ для этой задачи через `register_task_env_overrides()`. RL- и benchmark-окружения (TerminalBench2, HermesSweEnv и т. п.) делают это автоматически, поднимая отдельный Docker-образ на каждую задачу.

**Усиление безопасности:**
- `--cap-drop ALL` с возвратом только `DAC_OVERRIDE`, `CHOWN`, `FOWNER`
- `--security-opt no-new-privileges`
- `--pids-limit 256`
- tmpfs с ограничением размера для `/tmp` (512MB), `/var/tmp` (256MB), `/run` (64MB)

**Передача учётных данных:** переменные окружения, перечисленные в `docker_forward_env`, сначала берутся из вашего shell-окружения, а затем из `~/.hermes/.env`. Skills могут также объявлять `required_environment_variables`, которые подхватываются автоматически.

### Бэкенд SSH

Запускает команды на удалённом сервере по SSH. Для повторного использования соединения применяется ControlMaster; в простое параметр keepalive составляет 5 минут. Постоянная оболочка включена по умолчанию — состояние (cwd, переменные окружения) сохраняется между командами.

```yaml
terminal:
  backend: ssh
  persistent_shell: true           # Держать долгоживущую bash-сессию (по умолчанию: true)
```

**Обязательные переменные окружения:**

```bash
TERMINAL_SSH_HOST=my-server.example.com
TERMINAL_SSH_USER=ubuntu
```

**Необязательно:**

| Переменная | По умолчанию | Описание |
|----------|--------------|----------|
| `TERMINAL_SSH_PORT` | `22` | Порт SSH |
| `TERMINAL_SSH_KEY` | (системное значение по умолчанию) | Путь к приватному ключу SSH |
| `TERMINAL_SSH_PERSISTENT` | `true` | Включить постоянную оболочку |

**Как это работает:** при инициализации подключение устанавливается с `BatchMode=yes` и `StrictHostKeyChecking=accept-new`. Постоянная оболочка удерживает на удалённом хосте один процесс `bash -l`, обмениваясь данными через временные файлы. Команды, которым нужен `stdin_data` или `sudo`, автоматически переходят в одноразовый режим.

### Бэкенд Modal

Запускает команды в облачной песочнице [Modal](https://modal.com). Каждая задача получает изолированную VM с настраиваемыми CPU, памятью и диском. Файловую систему можно сохранять в snapshot и восстанавливать между сеансами.

```yaml
terminal:
  backend: modal
  container_cpu: 1                 # Ядра CPU
  container_memory: 5120           # МБ (5GB)
  container_disk: 51200            # МБ (50GB)
  container_persistent: true       # Создавать и восстанавливать снимок файловой системы
```

**Требуется:** либо переменные окружения `MODAL_TOKEN_ID` + `MODAL_TOKEN_SECRET`, либо файл `~/.modal.toml`.

**Сохранение состояния:** если опция включена, файловая система песочницы сохраняется в snapshot при завершении и восстанавливается в следующем сеансе. Снимки хранятся в `~/.hermes/modal_snapshots.json`. Это сохраняет состояние файловой системы, но не живые процессы, пространство PID или фоновые задания.

**Файлы учётных данных:** автоматически монтируются из `~/.hermes/` (OAuth-токены и т. п.) и синхронизируются перед каждой командой.

### Бэкенд Daytona

Запускает команды в управляемом рабочем пространстве Daytona. Поддерживает stop/resume для сохранения состояния.

```yaml
terminal:
  backend: daytona
  container_cpu: 1                 # Ядра CPU
  container_memory: 5120           # МБ → преобразуется в GiB
  container_disk: 10240            # МБ → преобразуется в GiB (макс. 10 GiB)
  container_persistent: true       # Останавливать и возобновлять вместо удаления
```

**Требуется:** переменная окружения `DAYTONA_API_KEY`.

**Сохранение состояния:** если опция включена, песочницы при завершении не удаляются, а останавливаются и возобновляются в следующем сеансе. Имена песочниц следуют шаблону `hermes-{task_id}`.

**Лимит диска:** Daytona ограничивает размер 10 GiB. Запросы сверх этого значения обрезаются с предупреждением.

### Бэкенд Vercel Sandbox

Запускает команды в облачной microVM [Vercel Sandbox](https://vercel.com/docs/vercel-sandbox). Hermes использует обычные инструменты `terminal` и `file tool`; специальных инструментов Vercel, доступных модели, нет.

```yaml
terminal:
  backend: vercel_sandbox
  vercel_runtime: node24          # node24 | node22 | python3.13
  cwd: /vercel/sandbox            # Корень рабочего пространства по умолчанию
  container_persistent: true      # Создавать и восстанавливать снимок файловой системы
  container_disk: 51200           # Только общее значение по умолчанию; пользовательский диск не поддерживается
```

**Требуется установка:** установите опциональный SDK extra:

```bash
pip install 'hermes-agent[vercel]'
```

**Требуется аутентификация:** настройте access-token auth, указав одновременно `VERCEL_TOKEN`, `VERCEL_PROJECT_ID` и `VERCEL_TEAM_ID`. Это поддерживаемая схема для деплоев и обычных долгоживущих процессов Hermes на Render, Railway, Docker и похожих хостингах.

Для разовой локальной разработки Hermes также принимает короткоживущие Vercel OIDC-токены:

```bash
VERCEL_OIDC_TOKEN="$(vc project token <project-name>)" hermes chat
```

Если вы находитесь в каталоге связанного Vercel-проекта, имя проекта можно не указывать:

```bash
VERCEL_OIDC_TOKEN="$(vc project token)" hermes chat
```

OIDC-токены живут недолго, и их не стоит использовать как рекомендуемый способ деплоя.

**Runtime:** `terminal.vercel_runtime` поддерживает `node24`, `node22` и `python3.13`. Если значение не задано, Hermes по умолчанию использует `node24`.

**Сохранение состояния:** если `container_persistent: true`, Hermes создаёт снимок файловой системы песочницы при завершении и при следующем запуске той же задачи восстанавливает песочницу из этого снимка. В снимок могут попасть синхронизированные Hermes учётные данные, навыки и кэш-файлы, которые были скопированы в песочницу. Это сохраняет только состояние файловой системы; живую идентичность песочницы, пространство PID, состояние shell и фоновые процессы оно не сохраняет.

**Фоновые команды:** `terminal(background=true)` использует общий нелокальный поток фоновых процессов Hermes. Пока песочница жива, вы можете запускать процессы, опрашивать их, ждать завершения, смотреть логи и завершать их через обычный инструмент процессов. Нативного восстановления отсоединённых процессов Vercel после очистки или перезапуска Hermes не предоставляет.

**Размер диска:** Vercel Sandbox пока не поддерживает ресурсный параметр `container_disk` Hermes. Оставьте `container_disk` пустым или равным общему значению по умолчанию `51200`; нестандартные значения приводят к ошибкам диагностики и создания бэкенда вместо тихого игнорирования.

### Бэкенд Singularity/Apptainer

Запускает команды в контейнере [Singularity/Apptainer](https://apptainer.org). Рассчитан на HPC-кластеры и общие машины, где Docker недоступен.

```yaml
terminal:
  backend: singularity
  singularity_image: "docker://nikolaik/python-nodejs:python3.11-nodejs20"
  container_cpu: 1                 # Ядра CPU
  container_memory: 5120           # МБ
  container_persistent: true       # Записываемый overlay сохраняется между сеансами
```

**Требования:** бинарник `apptainer` или `singularity` в `$PATH`.

**Обработка образов:** Docker URL (`docker://...`) автоматически конвертируются в SIF-файлы и кэшируются. Уже существующие `.sif`-файлы используются напрямую.

**Каталог scratch:** определяется по порядку: `TERMINAL_SCRATCH_DIR` → `TERMINAL_SANDBOX_DIR/singularity` → `/scratch/$USER/hermes-agent` (HPC-конвенция) → `~/.hermes/sandboxes/singularity`.

**Изоляция:** используется `--containall --no-home` для полной namespace-изоляции без монтирования домашнего каталога хоста.

### Частые проблемы терминальных бэкендов

Если команды терминала сразу падают или инструмент `terminal` помечается как отключённый:

- **Local** — специальных требований нет. Это самый безопасный вариант по умолчанию, когда вы только начинаете.
- **Docker** — запустите `docker version`, чтобы проверить, что Docker работает. Если команда падает, исправьте Docker или выполните `hermes config set terminal.backend local`.
- **SSH** — должны быть заданы и `TERMINAL_SSH_HOST`, и `TERMINAL_SSH_USER`. Hermes пишет понятную ошибку, если чего-то не хватает.
- **Modal** — нужен либо `MODAL_TOKEN_ID`, либо `~/.modal.toml`. Для проверки запустите `hermes doctor`.
- **Daytona** — нужен `DAYTONA_API_KEY`. Конфигурацией URL сервера занимается Daytona SDK.
- **Singularity** — нужен `apptainer` или `singularity` в `$PATH`. Такое часто встречается на HPC-кластерах.

Если сомневаетесь, верните `terminal.backend` в `local` и сначала проверьте, что команды работают именно там.

### Синхронизация файлов с удалённого хоста обратно при завершении

Для бэкендов **SSH**, **Modal** и **Daytona** (то есть там, где рабочее дерево агента находится не на том же хосте, на котором запущен Hermes), Hermes отслеживает файлы, которые агент изменил внутри удалённой песочницы, и при завершении сеанса или очистке песочницы **синхронизирует изменённые файлы обратно на хост** в `~/.hermes/cache/remote-syncs/<session-id>/`.

- Срабатывает при: закрытии сеанса, `/new`, `/reset`, таймауте сообщения gateway, завершении субагента через `delegate_task`, если дочерний агент работал на удалённом бэкенде.
- Охватывает всё дерево, которое агент изменил, а не только те файлы, которые он явно открывал. Добавления, изменения и удаления фиксируются одинаково.
- К моменту, когда вы заглянете в него, удалённая песочница уже может быть уничтожена; локальная копия в `~/.hermes/cache/remote-syncs/…` — это и есть единственный достоверный источник о том, что изменил агент.
- Крупные бинарные результаты (model checkpoints, сырые датасеты) ограничиваются по размеру — синхронизация пропускает файлы больше `file_sync_max_mb` (по умолчанию `100`). Увеличьте значение, если ожидаете вернуть более крупные артефакты.

```yaml
terminal:
  file_sync_max_mb: 100     # по умолчанию — синхронизировать файлы размером до 100 МБ каждый
  file_sync_enabled: true   # по умолчанию — false отключает синхронизацию целиком
```

Так можно вернуть результаты из одноразовых облачных песочниц, которые уничтожаются после завершения сеанса, не заставляя агента вручную делать `scp` или `modal volume put` для каждого артефакта.

### Подключение томов Docker

При использовании Docker-бэкенда `docker_volumes` позволяет делиться с контейнером каталогами хоста. Каждый элемент использует стандартный Docker-синтаксис `-v`: `host_path:container_path[:options]`.

```yaml
terminal:
  backend: docker
  docker_volumes:
    - "/home/user/projects:/workspace/projects"   # Чтение и запись (по умолчанию)
    - "/home/user/datasets:/data:ro"              # Только чтение
    - "/home/user/.hermes/cache/documents:/output" # Экспорты, видимые gateway
```

Это полезно для:
- **Передачи файлов** агенту (датасетов, конфигов, эталонного кода)
- **Получения файлов** от агента (сгенерированного кода, отчётов, экспортов)
- **Общих рабочих пространств**, где и вы, и агент работают с одними и теми же файлами

Если вы используете шлюз обмена сообщениями и хотите, чтобы агент отправлял сгенерированные файлы через `MEDIA:/...`, лучше выделить отдельный export mount, видимый на хосте, например `/home/user/.hermes/cache/documents:/output`.

- Записывайте файлы внутри Docker в `/output/...`
- В `MEDIA:` указывайте **путь хоста**, например:
  `MEDIA:/home/user/.hermes/cache/documents/report.txt`
- **Не** указывайте `/workspace/...` или `/output/...`, если только такой путь не существует и для процесса gateway на хосте

:::warning
Дублирующиеся ключи YAML тихо перезаписывают предыдущие значения. Если у вас уже есть блок `docker_volumes:`, добавляйте новые точки монтирования в тот же список, а не создавайте позже ещё один ключ `docker_volumes:`.
:::

Также можно задать через переменную окружения: `TERMINAL_DOCKER_VOLUMES='["/host:/container"]'` (JSON-массив).

### Передача учётных данных в Docker

По умолчанию Docker-сеансы терминала не наследуют произвольные учётные данные хоста. Если внутри контейнера нужен конкретный токен, добавьте его в `terminal.docker_forward_env`.

```yaml
terminal:
  backend: docker
  docker_forward_env:
    - "GITHUB_TOKEN"
    - "NPM_TOKEN"
```

Hermes сначала берёт каждую указанную переменную из текущего shell, а если она была сохранена через `hermes config set`, то переходит к `~/.hermes/.env`.

:::warning
Всё, что перечислено в `docker_forward_env`, становится видимым для команд, запущенных внутри контейнера. Передавайте только те учётные данные, которые не жалко показать terminal-сеансу.
:::

### Запуск контейнера от имени пользователя хоста

По умолчанию контейнеры Docker запускаются как `root` (UID 0). Файлы, созданные внутри `/workspace` или других примонтированных каталогов, на хосте оказываются принадлежащими root, так что после сеанса их приходится `sudo chown`-ить, прежде чем редактировать их из хостового редактора. Флаг `terminal.docker_run_as_host_user` решает эту проблему:

```yaml
terminal:
  backend: docker
  docker_run_as_host_user: true   # по умолчанию: false
```

Когда этот режим включён, Hermes добавляет `--user $(id -u):$(id -g)` к команде `docker run`, так что файлы, записанные в примонтированные каталоги (`/workspace`, `/root`, всё из `docker_volumes`), принадлежат вашему пользователю хоста, а не root. Компромисс в том, что контейнер больше не сможет выполнять `apt install` или писать в принадлежащие root пути вроде `/root/.npm`. Если нужны и права пользователя хоста, и root-owned пути, используйте базовый образ, у которого `HOME` принадлежит не-root пользователю, либо добавьте нужные инструменты на этапе сборки образа.

Оставляйте этот флаг `false` (значение по умолчанию), если вам важна обратная совместимость. Включайте его, когда ваш сценарий в основном сводится к редактированию примонтированных файлов хоста и вы устали от `sudo chown -R`.

### Дополнительно: монтирование каталога запуска в `/workspace`

По умолчанию Docker-песочницы остаются изолированными. Hermes **не** передаёт текущий рабочий каталог хоста в контейнер, если вы явно не разрешите это.

Включите это в `config.yaml`:

```yaml
terminal:
  backend: docker
  docker_mount_cwd_to_workspace: true
```

Когда этот режим включён:
- если вы запускаете Hermes из `~/projects/my-app`, этот каталог хоста монтируется в `/workspace`
- Docker-бэкенд стартует в `/workspace`
- file tools и terminal-команды видят один и тот же примонтированный проект

Когда режим выключен, `/workspace` остаётся принадлежащим песочнице, пока вы явно не примонтируете что-нибудь через `docker_volumes`.

Компромисс по безопасности:
- `false` сохраняет границу песочницы
- `true` даёт песочнице прямой доступ к каталогу, из которого вы запустили Hermes

Включайте эту опцию только тогда, когда контейнер действительно должен работать с живыми файлами хоста.

### Постоянная оболочка

По умолчанию каждая терминальная команда запускается в своём подпроцессе — рабочий каталог, переменные окружения и shell-переменные между командами сбрасываются. Когда **постоянная оболочка** включена, один долгоживущий bash-процесс остаётся активным между вызовами `execute()`, так что состояние сохраняется от команды к команде.

Особенно полезно это для **SSH-бэкенда**, где ещё и убирает накладные расходы на подключение для каждой команды. Для SSH постоянная оболочка **включена по умолчанию**, а для локального бэкенда выключена.

```yaml
terminal:
  persistent_shell: true   # по умолчанию — включает постоянную оболочку для SSH
```

Чтобы отключить:

```bash
hermes config set terminal.persistent_shell false
```

**Что сохраняется между командами:**
- Рабочий каталог (`cd /tmp` сохраняется для следующей команды)
- Экспортированные переменные окружения (`export FOO=bar`)
- Переменные shell (`MY_VAR=hello`)

**Приоритеты:**

| Уровень | Переменная | Значение по умолчанию |
|-------|----------|----------------------|
| Конфиг | `terminal.persistent_shell` | `true` |
| Переопределение SSH | `TERMINAL_SSH_PERSISTENT` | следует конфигу |
| Локальное переопределение | `TERMINAL_LOCAL_PERSISTENT` | `false` |

Переменные окружения для конкретного бэкенда имеют наивысший приоритет. Если хотите включить постоянную оболочку и для локального бэкенда:

```bash
export TERMINAL_LOCAL_PERSISTENT=true
```

:::note
Команды, которым нужен `stdin_data` или `sudo`, автоматически переходят в одноразовый режим, потому что `stdin` постоянной оболочки уже занят IPC-протоколом.
:::

Подробности по каждому бэкенду см. в [Выполнении кода](/user-guide/features/code-execution) и в [разделе Terminal README](/user-guide/features/tools).

## Настройки навыков

Навыки могут объявлять собственные настройки конфигурации через frontmatter в `SKILL.md`. Это не секретные значения (пути, предпочтения, параметры доменов), которые хранятся в namespace `skills.config` внутри `config.yaml`.

```yaml
skills:
  config:
    myplugin:
      path: ~/myplugin-data   # Пример — каждый навык определяет собственные ключи
```

**Как работают настройки навыков:**

- `hermes config migrate` сканирует все включённые навыки, находит ещё не настроенные параметры и предлагает пройти их интерактивно
- `hermes config show` показывает все настройки навыков в разделе "Skill Settings" вместе с тем навыком, которому они принадлежат
- Когда навык загружается, уже разрешённые значения конфигурации автоматически подставляются в его контекст

**Ручная установка значений:**

```bash
hermes config set skills.config.myplugin.path ~/myplugin-data
```

Подробности о том, как объявлять настройки в собственных навыках, см. в [Creating Skills — Config Settings](/developer-guide/creating-skills#config-settings-configyaml).

### Защита от записей навыков, созданных агентом

Когда агент использует `skill_manage`, чтобы создать, отредактировать, исправить или удалить навык, Hermes может дополнительно проверять новый или обновлённый контент на опасные ключевые паттерны (сбор учётных данных, очевидные попытки prompt-инъекции, инструкции по эксфильтрации). Эта проверка **выключена по умолчанию** — реальные рабочие процессы агента, которые легитимно трогают `~/.ssh/` или упоминают `$OPENAI_API_KEY`, слишком часто срабатывали на эвристике. Включите её обратно, если хотите, чтобы сканер спрашивал подтверждение до того, как записи навыков будут приняты:

```yaml
skills:
  guard_agent_created: true   # по умолчанию: false
```

Когда опция включена, любая помеченная запись `skill_manage` показывается как запрос на подтверждение с пояснением сканера. Принятые записи сохраняются; отклонённые возвращают агенту поясняющую ошибку.

## Безопасное чтение файлов

Определяет, сколько содержимого может вернуть один вызов `read_file`. Запросы, превышающие лимит, отклоняются с ошибкой, которая предлагает агенту использовать `offset` и `limit` для меньшего диапазона. Это защищает окно контекста от единственного чтения минифицированного JS-бандла или большого data-файла.

```yaml
file_read_max_chars: 100000  # по умолчанию — ~25–35K токенов
```

Увеличьте это значение, если вы работаете на модели с большим окном контекста и часто читаете большие файлы. Для моделей с небольшим контекстом, наоборот, уменьшите его, чтобы чтения оставались эффективными:

```yaml
# Модель с большим контекстом (200K+)
file_read_max_chars: 200000

# Небольшая локальная модель (контекст 16K)
file_read_max_chars: 30000
```

Агент также автоматически дедуплицирует чтения файлов: если один и тот же участок файла читается дважды и сам файл не менялся, вместо повторной отправки содержимого возвращается лёгкий stub. Это сбрасывается при сжатии контекста, чтобы агент мог перечитать файлы после того, как их содержимое уже было суммировано.

## Ограничения на усечение вывода инструментов

Три связанных лимита определяют, сколько сырого вывода может вернуть инструмент до того, как Hermes его усечёт:

```yaml
tool_output:
  max_bytes: 50000        # лимит вывода terminal (символы)
  max_lines: 2000         # лимит постраничного вывода read_file
  max_line_length: 2000   # лимит длины строки в режиме с номерами строк у read_file
```

- **`max_bytes`** — Когда команда `terminal` выдаёт больше этого числа символов в сумме stdout/stderr, Hermes оставляет первые 40% и последние 60% и вставляет между ними заметку `[OUTPUT TRUNCATED]`. По умолчанию `50000` (≈12-15K токенов у типичных токенизаторов).
- **`max_lines`** — Верхняя граница параметра `limit` для одного вызова `read_file`. Запросы выше этого значения обрезаются, чтобы одно чтение не могло залить весь контекст. По умолчанию `2000`.
- **`max_line_length`** — Лимит на длину строки, который применяется, когда `read_file` возвращает вид с нумерацией строк. Более длинные строки обрезаются до этого числа символов, после чего добавляется `... [truncated]`. По умолчанию `2000`.

Увеличивайте эти лимиты на моделях с большим окном контекста, которым по карману более сырой вывод за один вызов. Для моделей с небольшим контекстом лучше, наоборот, уменьшать их, чтобы результаты инструментов оставались компактными:

```yaml
# Модель с большим контекстом (200K+)
tool_output:
  max_bytes: 150000
  max_lines: 5000

# Небольшая локальная модель (контекст 16K)
tool_output:
  max_bytes: 20000
  max_lines: 500
```

## Глобальное отключение наборов инструментов

Чтобы отключить определённые toolsets сразу во всём CLI и на всех платформах gateway, перечислите их имена в `agent.disabled_toolsets`:

```yaml
agent:
  disabled_toolsets:
    - memory       # скрыть инструменты памяти + внедрение MEMORY_GUIDANCE
    - web          # нигде не показывать web_search / web_extract
```

Это применяется **после** per-platform tool config (`platform_toolsets`, которые пишет `hermes tools`), так что toolset, указанный здесь, всегда будет удалён — даже если в сохранённой конфигурации платформы он всё ещё присутствует. Используйте это, когда нужен один общий переключатель “выключить X везде”, а не ручное редактирование 15+ строк платформ в UI `hermes tools`.

Пустой список или отсутствие ключа ничего не меняют.

## Изоляция Git worktree

Включите изолированные git worktree, если хотите запускать несколько агентов параллельно в одном и том же репозитории:

```yaml
worktree: true    # Всегда создавать worktree (то же, что `hermes -w`)
# worktree: false # По умолчанию — только когда передан флаг `-w`
```

Когда опция включена, каждый CLI-сеанс создаёт новый worktree в `.worktrees/` со своей веткой. Агенты могут редактировать файлы, делать commit, push и создавать PR, не мешая друг другу. Чистые worktree удаляются при выходе; dirty-ветки сохраняются для ручного восстановления.

Можно также перечислить файлы из `.gitignore`, которые нужно копировать в worktree, через `.worktreeinclude` в корне репозитория:

```
# .worktreeinclude
.env
.venv/
node_modules/
```

## Сжатие контекста

Hermes автоматически сжимает длинные разговоры, чтобы укладываться в окно контекста модели. Суммаризатор сжатия — это отдельный LLM-вызов; его можно направить на любого провайдера или конечную точку.

Все настройки сжатия живут в `config.yaml` (переменные окружения не используются).

### Полная справка

```yaml
compression:
  enabled: true                                     # Включать или выключать сжатие
  threshold: 0.50                                   # Сжимать при этом % от лимита контекста
  target_ratio: 0.20                                # Доля порога, которую сохраняем как свежий хвост
  protect_last_n: 20                                # Минимум последних сообщений, которые остаются без сжатия
  hygiene_hard_message_limit: 400                   # Предохранитель gateway — см. ниже

# Модель и провайдер суммаризации настраиваются через auxiliary:
auxiliary:
  compression:
    model: ""                                       # Пусто = использовать основную чат-модель. Для более дешёвого и быстрого сжатия задайте, например, "google/gemini-3-flash-preview".
    provider: "auto"                                # Провайдер: "auto", "openrouter", "nous", "codex", "main" и т. д.
    base_url: null                                  # Собственная конечная точка, совместимая с OpenAI (переопределяет провайдера)
```

:::info Миграция старой конфигурации
Старые конфиги с `compression.summary_model`, `compression.summary_provider` и `compression.summary_base_url` при первой загрузке автоматически мигрируют в `auxiliary.compression.*` (config version 17). Никаких ручных действий не требуется.
:::

`hygiene_hard_message_limit` — это только для gateway **pre-compression safety valve**. Разросшиеся сеансы с тысячами сообщений могут упереться в лимит контекста модели ещё до того, как сработает обычный порог по проценту от окна; когда число сообщений пересекает этот рубеж, Hermes принудительно включает сжатие независимо от использования токенов. Значение по умолчанию — `400`. Увеличивайте его на платформах, где очень длинные сеансы нормальны, и уменьшайте, если хотите более агрессивное сжатие. Изменение этого параметра на работающем gateway начинает действовать со следующего сообщения (см. ниже).

:::tip Горячая перезагрузка сжатия и длины контекста на gateway
Начиная с недавних релизов, изменение `model.context_length` или любого ключа `compression.*` в `config.yaml` на работающем gateway начинает действовать со следующего сообщения — без перезапуска gateway, без `/reset` и без смены сеанса. Эти ключи входят в сигнатуру кэшированного агента, поэтому gateway прозрачно пересобирает agent, когда видит изменение. API-ключи и конфигурация инструментов и навыков по-прежнему требуют обычной перезагрузки.
:::

### Типовые сценарии

**По умолчанию (автоопределение) — настройка не нужна:**
```yaml
compression:
  enabled: true
  threshold: 0.50
```
Использует ваш основной провайдер и основную модель. Переопределяйте это для конкретной задачи (например, `auxiliary.compression.provider: openrouter` + `model: google/gemini-2.5-flash`), если хотите сжимать контекст на более дешёвой модели, чем основная чат-модель.

**Принудительно задать конкретного провайдера** (через OAuth или API-ключ):
```yaml
auxiliary:
  compression:
    provider: nous
    model: gemini-3-flash
```
Работает с любым провайдером: `nous`, `openrouter`, `codex`, `anthropic`, `main` и т. д.

**Собственная конечная точка** (самостоятельно размещаемая, Ollama, zai, DeepSeek и т. д.):
```yaml
auxiliary:
  compression:
    model: glm-4.7
    base_url: https://api.z.ai/api/coding/paas/v4
```
Указывает на собственную конечную точку, совместимую с OpenAI. Для аутентификации используется `OPENAI_API_KEY`.

### Как взаимодействуют три параметра

| `auxiliary.compression.provider` | `auxiliary.compression.base_url` | Результат |
|---------------------|---------------------|-----------|
| `auto` (по умолчанию) | не задан | Автоматически выбирается лучший доступный провайдер |
| `nous` / `openrouter` / и т. п. | не задан | Принудительно используется этот провайдер, со своей аутентификацией |
| любой | задан | Используется напрямую указанная собственная конечная точка (provider игнорируется) |

:::warning Требование к длине контекста summary-модели
У summary-модели **обязательно** должно быть окно контекста не меньше, чем у основной модели агента. Компрессор отправляет в summary-модель весь средний кусок разговора — если окно контекста этой модели меньше, чем у основной, вызов суммаризации завершится ошибкой длины контекста. Когда это происходит, средние ходы **отбрасываются без summary**, и контекст разговора теряется тихо. Если вы переопределяете модель, проверьте, что её длина контекста равна или больше, чем у основной модели.
:::

## Движок контекста

Движок контекста определяет, как управляются разговоры, когда они приближаются к токен-лимиту модели. Встроенный движок `compressor` использует сжатие с потерями (см. [Сжатие контекста](/developer-guide/context-compression-and-caching)). Плагин-движки могут заменять его альтернативными стратегиями.

```yaml
context:
  engine: "compressor"    # по умолчанию — встроенное сжатие с потерями
```

Чтобы использовать плагин-движок (например, LCM для управления контекстом без потерь):

```yaml
context:
  engine: "lcm"          # должно совпадать с именем плагина
```

Плагин-движки **никогда не активируются автоматически** — нужно явно указать `context.engine` с именем плагина. Доступные движки можно просматривать и выбирать через `hermes plugins` → Provider Plugins → Context Engine.

См. [Провайдеры памяти](/user-guide/features/memory-providers) для аналогичной схемы выбора одного варианта среди плагинов памяти.

## Давление на бюджет итераций

Когда агент работает над сложной задачей с большим количеством вызовов инструментов, он может незаметно израсходовать свой бюджет итераций (по умолчанию 90 ходов). Давление бюджета автоматически предупреждает модель по мере приближения к лимиту:

| Порог | Уровень | Что видит модель |
|-----------|-------|---------------------|
| **70%** | Осторожность | `[BUDGET: 63/90. 27 iterations left. Start consolidating.]` |
| **90%** | Предупреждение | `[BUDGET WARNING: 81/90. Only 9 left. Respond NOW.]` |

Предупреждения внедряются в JSON последнего результата инструмента (как поле `_budget_warning`), а не отправляются отдельными сообщениями — так сохраняется кэширование системной инструкции и не ломается структура разговора.

```yaml
agent:
  max_turns: 90                # Максимум итераций на один ход разговора (по умолчанию: 90)
  api_max_retries: 3           # Повторы для каждого провайдера до переключения на резервный (по умолчанию: 3)
```

Давление на бюджет итераций включено по умолчанию. Агент видит предупреждения прямо в результатах инструментов, что подталкивает его сводить работу воедино и выдавать ответ до того, как закончится бюджет.

Когда бюджет итераций полностью исчерпан, CLI показывает пользователю уведомление: `⚠ Iteration budget reached (90/90) — response may be incomplete`. Если бюджет заканчивается во время активной работы, агент перед остановкой генерирует сводку того, что успел сделать.

`agent.api_max_retries` определяет, сколько раз Hermes повторяет вызов API провайдера при временных ошибках (rate limit, обрыв соединения, 5xx) **до** того, как включится переключение на резервного провайдера. По умолчанию это `3`, то есть всего четыре попытки. Если у вас настроены [резервные провайдеры](/user-guide/features/fallback-providers) и вы хотите быстрее уходить на резерв, поставьте `0`, чтобы первая же временная ошибка на основном провайдере сразу передавала запрос резервному провайдеру вместо того, чтобы снова и снова ретраить нестабильную конечную точку.

### Таймауты API

У Hermes есть отдельные уровни таймаутов для потоковой передачи, а также детектор зависших вызовов без потоковой передачи. Он автоматически подстраивается под локальные провайдеры только если оставить значения неявными по умолчанию.

| Таймаут | По умолчанию | Локальные провайдеры | Конфиг / env |
|---------|--------------|----------------|--------------|
| Таймаут чтения socket | 120s | Автоматически повышается до 1800s | `HERMES_STREAM_READ_TIMEOUT` |
| Детектор зависших потоковых вызовов | 180s | Автоматически отключается | `HERMES_STREAM_STALE_TIMEOUT` |
| Детектор зависших вызовов без потоковой передачи | 300s | Автоматически отключается, если значение оставлено неявным | `providers.<id>.stale_timeout_seconds` или `HERMES_API_CALL_STALE_TIMEOUT` |
| Вызов API без потоковой передачи | 1800s | Без изменений | `providers.<id>.request_timeout_seconds` / `timeout_seconds` или `HERMES_API_TIMEOUT` |

**Таймаут чтения socket** определяет, как долго httpx ждёт следующую порцию данных от провайдера. Локальные LLM могут тратить минуты на prefill в больших контекстах до появления первого токена, поэтому Hermes поднимает это значение до 30 минут, когда обнаруживает локальную конечную точку. Если вы явно зададите `HERMES_STREAM_READ_TIMEOUT`, именно это значение будет использоваться всегда, независимо от того, обнаружена конечная точка или нет.

**Детектор зависших потоковых вызовов** завершает соединения, которые получают SSE-сигналы поддержания соединения, но не получают фактического содержимого. Для локальных провайдеров эта проверка полностью отключается, потому что во время prefill они такие сигналы не отправляют.

**Детектор зависших вызовов без потоковой передачи** завершает такие вызовы, если они слишком долго не возвращают ответ. По умолчанию Hermes отключает его на локальных конечных точках, чтобы избежать ложных срабатываний во время длинного prefill. Если вы явно зададите `providers.<id>.stale_timeout_seconds`, `providers.<id>.models.<model>.stale_timeout_seconds` или `HERMES_API_CALL_STALE_TIMEOUT`, это значение будет применяться даже на локальных конечных точках.

## Предупреждения о давлении на контекст

В отличие от давления на бюджет итераций, давление на контекст отслеживает, насколько разговор приблизился к **compaction threshold** — точке, где сжатие контекста срабатывает и сворачивает старые сообщения в сводку. Это помогает и вам, и агенту понимать, когда разговор становится длинным.

| Прогресс | Уровень | Что происходит |
|----------|-------|-------------|
| **≥ 60%** до порога | Info | CLI показывает голубую полосу прогресса; gateway отправляет информационное уведомление |
| **≥ 85%** до порога | Warning | CLI показывает жирную жёлтую полосу; gateway предупреждает, что compaction уже близко |

В CLI давление на контекст отображается полосой прогресса в потоке вывода инструментов:

```
  ◐ context ████████████░░░░░░░░ 62% to compaction  48k threshold (50%) · approaching compaction
```

На платформах обмена сообщениями отправляется текстовое уведомление:

```
◐ Context: ████████████░░░░░░░░ 62% to compaction (threshold: 50% of window).
```

Если auto-compression отключено, предупреждение сообщает, что контекст может быть усечён.

Давление на контекст срабатывает автоматически — настраивать его не нужно. Оно лишь показывает уведомление пользователю и не меняет поток сообщений и не внедряет ничего в контекст модели.

## Стратегии пула учётных данных

Когда у одного и того же провайдера есть несколько API-ключей или OAuth-токенов, настройте стратегию ротации:

```yaml
credential_pool_strategies:
  openrouter: round_robin    # равномерно чередовать ключи
  anthropic: least_used      # всегда брать наименее использованный ключ
```

Варианты: `fill_first` (по умолчанию), `round_robin`, `least_used`, `random`. Полная документация — в [Credential Pools](/user-guide/features/credential-pools).

## Кэширование промпта

Hermes автоматически включает кэширование промптов между сеансами, когда активный провайдер это поддерживает, — никакой пользовательской настройки не требуется.

Для Claude на **native Anthropic**, **OpenRouter** и **Nous Portal** Hermes ставит `cache_control` breakpoints с TTL на 1 час (`ttl: "1h"`) на системную инструкцию и блоки навыков. Первый запрос в новом часу оплачивается по полной ставке входных токенов; последующие запросы из любого сеанса в тот же час берутся из кэша по сниженной ставке чтения из кэша. Это означает, что системная инструкция, загруженный контент навыков и начало любого include с большим контекстом переиспользуются между сеансами `hermes` и между ответвлёнными субагентами в течение первого часа.

У Qwen Cloud (Alibaba DashScope) upstream ограничивает TTL кэша 5 минутами, поэтому Hermes использует там пятиминутную точку разрыва TTL. Другие пути Claude через сторонние сервисы (AWS Bedrock, Azure Foundry) откатываются к собственным настройкам кэширования провайдера. У xAI Grok используется отдельный механизм conversation-id, привязанный к сеансу — см. [xAI prompt caching](/integrations/providers#xai-grok--responses-api--prompt-caching).

Отключить это нельзя: кэширование всегда включено и экономит деньги даже на однократных разговорах, потому что одна только системная инструкция уже составляет заметную долю входных токенов.

## Вспомогательные модели

Hermes использует "auxiliary" модели для вспомогательных задач вроде анализа изображений, суммирования веб-страниц, анализа скриншотов браузера, генерации заголовка сеанса и сжатия контекста. По умолчанию (`auxiliary.*.provider: "auto"`) Hermes направляет каждую такую задачу на вашу **основную чат-модель** — тот же провайдер и ту же модель, которые вы выбрали через `hermes model`. Для старта не нужно ничего настраивать, но имейте в виду: на дорогих reasoning-моделях (Opus, MiniMax M2.7 и т. п.) вспомогательные задачи заметно увеличивают стоимость. Если хотите дешёвые и быстрые вспомогательные задачи независимо от основной модели, задайте `auxiliary.<task>.provider` и `auxiliary.<task>.model` явно (например, Gemini Flash на OpenRouter для vision и web extraction).

:::note Почему "auto" использует основную модель
Раньше для пользователей агрегаторов (OpenRouter, Nous Portal) использовалось дешёвое значение по умолчанию на стороне провайдера. Это было неожиданно: пользователи, которые платили за подписку агрегатора, видели, что их auxiliary-трафик обслуживает другая модель. Теперь `auto` использует основную модель для всех, а переопределения на уровне задач в `config.yaml` по-прежнему имеют приоритет (см. ниже [Полную справку по auxiliary-конфигу](#полная-справка-по-auxiliary-конфигу)).
:::

### Настройка вспомогательных моделей в интерактивном режиме

Вместо ручного редактирования YAML запустите `hermes model` и выберите в меню **"Настроить вспомогательные модели"**. Появится интерактивный выбор по задачам:

```
$ hermes model
→ Настроить вспомогательные модели

[ ] vision               currently: auto / main model
[ ] web_extract          currently: auto / main model
[ ] title_generation     currently: openrouter / google/gemini-3-flash-preview
[ ] compression          currently: auto / main model
[ ] approval             currently: auto / main model
[ ] triage_specifier     currently: auto / main model
[ ] kanban_decomposer    currently: auto / main model

```

Выберите задачу, затем провайдера (для OAuth откроется браузер; провайдеры с API-ключом попросят ввести ключ), затем модель. Изменение сохранится в `auxiliary.<task>.*` внутри `config.yaml`. Это та же механика, что и у выбора основной модели — никакого дополнительного синтаксиса учить не нужно.

### Универсальный шаблон конфигурации

Каждый слот модели в Hermes — вспомогательные задачи, сжатие и резервные модели — использует одни и те же три параметра:

| Ключ | Что делает | По умолчанию |
|-----|-------------|--------------|
| `provider` | Какого провайдера использовать для аутентификации и маршрутизации | `"auto"` |
| `model` | Какую модель запрашивать | значение по умолчанию провайдера |
| `base_url` | Собственная конечная точка, совместимая с OpenAI (переопределяет провайдера) | не задан |

Когда задан `base_url`, Hermes игнорирует `provider` и напрямую вызывает эту конечную точку (используя `api_key` или `OPENAI_API_KEY` для аутентификации). Если задан только `provider`, Hermes использует встроенные данные аутентификации и базовый URL этого провайдера.

Доступные провайдеры для вспомогательных задач: `auto`, `main`, а также любой провайдер из [реестра провайдеров](/reference/environment-variables) — `openrouter`, `nous`, `openai-codex`, `copilot`, `copilot-acp`, `anthropic`, `gemini`, `google-gemini-cli`, `qwen-oauth`, `zai`, `kimi-coding`, `kimi-coding-cn`, `minimax`, `minimax-cn`, `minimax-oauth`, `deepseek`, `nvidia`, `xai`, `xai-oauth`, `ollama-cloud`, `alibaba`, `bedrock`, `huggingface`, `arcee`, `xiaomi`, `kilocode`, `opencode-zen`, `opencode-go`, `ai-gateway`, `azure-foundry` — а также любой именованный пользовательский провайдер из списка `custom_providers` (например, `provider: "beans"`).

:::tip MiniMax OAuth
`minimax-oauth` входит через OAuth в браузере (API-ключ не нужен). Запустите `hermes model` и выберите **MiniMax (OAuth)**, чтобы пройти аутентификацию. Для вспомогательных задач автоматически используется `MiniMax-M2.7-highspeed`. См. [руководство по MiniMax OAuth](/guides/minimax-oauth).
:::

:::tip xAI Grok OAuth
`xai-oauth` входит через OAuth в браузере для подписчиков SuperGrok (API-ключ не нужен). Запустите `hermes model` и выберите **xAI Grok OAuth (SuperGrok Subscription)**, чтобы пройти аутентификацию. Тот же OAuth-токен используется для всех direct-to-xAI поверхностей (чат, вспомогательные задачи, TTS, image gen, video gen, transcription). См. [руководство по xAI Grok OAuth](/guides/xai-grok-oauth), а если Hermes работает на удалённом хосте, см. [OAuth over SSH / Remote Hosts](/guides/oauth-over-ssh).
:::

:::warning `"main"` только для вспомогательных задач
Опция провайдера `"main"` означает «используй того же провайдера, что и моя основная модель» — она допустима только внутри конфигураций `auxiliary:`, `compression:` и `fallback_model:`. Это **не** допустимое значение для верхнеуровневого `model.provider`. Если вы используете собственную конечную точку, совместимую с OpenAI, укажите `provider: custom` в секции `model:`. Все варианты провайдера для основной модели см. в [Провайдерах ИИ](/integrations/providers).
:::

### Полная справка по auxiliary-конфигу

```yaml
auxiliary:
  # Анализ изображений (tool vision_analyze + скриншоты браузера)
  vision:
    provider: "auto"           # "auto", "openrouter", "nous", "codex", "main" и т. д.
    model: ""                  # Например, "openai/gpt-4o", "google/gemini-2.5-flash"
    base_url: ""               # Собственная конечная точка, совместимая с OpenAI (переопределяет провайдера)
    api_key: ""                # API-ключ для base_url (иначе берётся OPENAI_API_KEY)
    timeout: 120               # секунды — таймаут вызова LLM API; для vision обычно нужен щедрый запас
    download_timeout: 30       # секунды — HTTP-загрузка изображения; увеличьте для медленных соединений

  # Суммирование веб-страниц + извлечение текста со страниц браузера
  web_extract:
    provider: "auto"
    model: ""                  # Например, "google/gemini-2.5-flash"
    base_url: ""
    api_key: ""
    timeout: 360               # секунд (6 мин) — суммаризация LLM на каждую попытку

  # Классификатор опасных команд для подтверждения
  approval:
    provider: "auto"
    model: ""
    base_url: ""
    api_key: ""
    timeout: 30                # секунд

  # Таймаут сжатия контекста (отдельно от конфигурации compression.*)
  compression:
    timeout: 120               # секунд — сжатие суммаризирует длинные разговоры и требует больше времени

  # Центр навыков — сопоставление и поиск навыков
  skills_hub:
    provider: "auto"
    model: ""
    base_url: ""
    api_key: ""
    timeout: 30

  # Диспетчеризация MCP tool
  mcp:
    provider: "auto"
    model: ""
    base_url: ""
    api_key: ""
    timeout: 30

  # Уточнитель triage для Kanban — `hermes kanban specify <id>` (или кнопка
  # ✨ Specify на карточках в колонке Triage) использует этот
  # слот, чтобы превратить короткую запись в конкретную спецификацию и
  # перевести задачу в `todo`. Здесь хорошо работают дешёвые и быстрые модели;
  # уточнение короткое и не требует глубокой рассуждающей модели.
  triage_specifier:
    provider: "auto"
    model: ""
    base_url: ""
    api_key: ""
    timeout: 120
```

:::tip
У каждой вспомогательной задачи есть настраиваемый `timeout` (в секундах). Значения по умолчанию: vision — 120 с, web_extract — 360 с, approval — 30 с, compression — 120 с. Увеличьте их, если используете медленные локальные модели для вспомогательных задач. У vision есть ещё отдельный `download_timeout` (по умолчанию 30 с) для загрузки изображения по HTTP — увеличьте его при медленном соединении или если у вас собственный сервер изображений.
:::

:::info
У сжатия контекста есть собственный блок `compression:` для порогов и отдельный блок `auxiliary.compression:` для настроек модели и провайдера — см. [Сжатие контекста](#сжатие-контекста) выше. Резервная модель использует блок `fallback_model:` — см. [Резервные провайдеры](/docs/integrations/providers#fallback-providers). Во всех трёх случаях используется одна и та же схема `provider` / `model` / `base_url`.
:::

### Маршрутизация OpenRouter и Pareto Code для вспомогательных задач

Когда вспомогательная задача разрешается через OpenRouter (либо явно, либо через `provider: "main"`, если основная модель тоже работает через OpenRouter), настройки `provider_routing` и `openrouter.min_coding_score` основной модели **не наследуются** — по замыслу каждая вспомогательная задача независима. Чтобы задать настройки провайдера OpenRouter или использовать [Pareto Code router](/integrations/providers#openrouter-pareto-code-router) для конкретной вспомогательной задачи, задайте их на уровне задачи через `extra_body`:

```yaml
auxiliary:
  compression:
    provider: openrouter
    model: openrouter/pareto-code         # Использовать маршрут Pareto Code для этой задачи
    extra_body:
      provider:                            # Параметры маршрутизации провайдера OpenRouter
        order: [anthropic, google]         # Попробовать этих провайдеров по порядку
        sort: throughput                   # или "price" | "latency"
        # only: [anthropic]                # Ограничить конкретным провайдером
        # ignore: [deepinfra]              # Исключить конкретных провайдеров
      plugins:                             # Плагин маршрутизации OpenRouter Pareto Code
        - id: pareto-router
          min_coding_score: 0.5            # 0.0–1.0; выше = сильнее coders
```

Формат повторяет то, что OpenRouter принимает в body запроса chat completions. Hermes передаёт весь `extra_body` без изменений, поэтому любой другой параметр request body OpenRouter, описанный в [openrouter.ai/docs](https://openrouter.ai/docs), работает так же.

### Смена vision-модели

Чтобы использовать GPT-4o вместо Gemini Flash для анализа изображений:

```yaml
auxiliary:
  vision:
    model: "openai/gpt-4o"
```

Или через переменную окружения (в `~/.hermes/.env`):

```bash
AUXILIARY_VISION_MODEL=openai/gpt-4o
```

### Варианты провайдеров

Эти параметры относятся к **настройкам вспомогательных задач** (`auxiliary:`, `compression:`, `fallback_model:`), а не к вашему основному `model.provider`.

| Провайдер | Описание | Требования |
|----------|-------------|------------|
| `"auto"` | Лучший доступный вариант (по умолчанию). Vision пробует OpenRouter → Nous → Codex. | — |
| `"openrouter"` | Принудительно OpenRouter — маршрутизация к любой модели (Gemini, GPT-4o, Claude и т. д.) | `OPENROUTER_API_KEY` |
| `"nous"` | Принудительно Nous Portal | `hermes auth` |
| `"codex"` | Принудительно Codex OAuth (учётная запись ChatGPT). Поддерживает vision (gpt-5.3-codex). | `hermes model` → Codex |
| `"minimax-oauth"` | Принудительно MiniMax OAuth (вход через браузер, API-ключ не нужен). Для вспомогательных задач использует MiniMax-M2.7-highspeed. | `hermes model` → MiniMax (OAuth) |
| `"xai-oauth"` | Принудительно xAI Grok OAuth (вход через браузер для подписчиков SuperGrok, API-ключ не нужен). Один и тот же OAuth-токен покрывает чат, TTS, image, video и transcription. | `hermes model` → xAI Grok OAuth (SuperGrok Subscription) |
| `"main"` | Использует вашу активную основную или собственную конечную точку. Это может быть `OPENAI_BASE_URL` + `OPENAI_API_KEY` или собственная конечная точка, сохранённая через `hermes model` / `config.yaml`. Работает с OpenAI, локальными моделями и любым API, совместимым с OpenAI. **Только для вспомогательных задач — не подходит для `model.provider`.** | Учётные данные конечной точки + базовый URL |

Провайдеры с API-ключом из основного каталога провайдеров тоже работают здесь, если вы хотите, чтобы вспомогательные задачи обходили ваш обычный маршрутизатор. `gmi` допустим, если настроен `GMI_API_KEY`:

```yaml
auxiliary:
  compression:
    provider: "gmi"
    model: "anthropic/claude-opus-4.6"
```

Для маршрутизации вспомогательных задач через GMI используйте точный model ID, который возвращает конечная точка `/v1/models` у GMI.

### Типовые сценарии

**Использование собственной конечной точки** (понятнее, чем `provider: "main"` для локальных или самостоятельно размещённых API):
```yaml
auxiliary:
  vision:
    base_url: "http://localhost:1234/v1"
    api_key: "local-key"
    model: "qwen2.5-vl"
```

`base_url` имеет приоритет над `provider`, так что это самый явный способ направить вспомогательную задачу на конкретную конечную точку. Для прямых переопределений конечной точки Hermes использует заданный `api_key` или, если его нет, `OPENAI_API_KEY`; `OPENROUTER_API_KEY` для такой собственной конечной точки не переиспользуется.

**Использование API-ключа OpenAI для vision:**
```yaml
# В `~/.hermes/.env`:
# OPENAI_BASE_URL=https://api.openai.com/v1
# OPENAI_API_KEY=sk-...

auxiliary:
  vision:
    provider: "main"
    model: "gpt-4o"       # или "gpt-4o-mini", если нужно дешевле
```

**Использование OpenRouter для vision** (маршрутизация к любой модели):
```yaml
auxiliary:
  vision:
    provider: "openrouter"
    model: "openai/gpt-4o"      # или "google/gemini-2.5-flash" и т. д.
```

**Использование Codex OAuth** (аккаунт ChatGPT Pro/Plus — API-ключ не нужен):
```yaml
auxiliary:
  vision:
    provider: "codex"     # использует ваш OAuth-токен ChatGPT
    # модель по умолчанию — gpt-5.3-codex (поддерживает vision)
```

**Использование MiniMax OAuth** (вход через браузер, API-ключ не нужен):
```yaml
model:
  default: MiniMax-M2.7
  provider: minimax-oauth
  base_url: https://api.minimax.io/anthropic
```
Запустите `hermes model` и выберите **MiniMax (OAuth)**, чтобы войти и настроить это автоматически. Для региона China базовый URL будет `https://api.minimaxi.com/anthropic`. Подробное руководство см. в [руководстве по MiniMax OAuth](/guides/minimax-oauth).

**Использование локальной или самостоятельно размещённой модели:**
```yaml
auxiliary:
  vision:
    provider: "main"      # использует вашу активную собственную конечную точку
    model: "my-local-model"
```

`provider: "main"` использует тот провайдер, который Hermes применяет для обычного чата, будь то именованный пользовательский провайдер (например, `beans`), встроенный провайдер вроде `openrouter` или устаревшая конечная точка `OPENAI_BASE_URL`.

:::tip
Если вы используете Codex OAuth как основной провайдер модели, vision работает автоматически — дополнительная настройка не нужна. Codex входит в цепочку автоопределения для vision.
:::

:::warning
**Vision требует мультимодальную модель.** Если вы задаёте `provider: "main"`, убедитесь, что ваша конечная точка поддерживает multimodal/vision, иначе анализ изображений не сработает.
:::

### Переменные окружения (legacy)

Вспомогательные модели можно настраивать и через переменные окружения. Однако `config.yaml` остаётся предпочтительным способом — им проще управлять, и он поддерживает все параметры, включая `base_url` и `api_key`.

| Параметр | Переменная окружения |
|---------|----------------------|
| Провайдер vision | `AUXILIARY_VISION_PROVIDER` |
| Модель vision | `AUXILIARY_VISION_MODEL` |
| Конечная точка vision | `AUXILIARY_VISION_BASE_URL` |
| API-ключ для vision | `AUXILIARY_VISION_API_KEY` |
| Провайдер web extract | `AUXILIARY_WEB_EXTRACT_PROVIDER` |
| Модель web extract | `AUXILIARY_WEB_EXTRACT_MODEL` |
| Конечная точка web extract | `AUXILIARY_WEB_EXTRACT_BASE_URL` |
| API-ключ для web extract | `AUXILIARY_WEB_EXTRACT_API_KEY` |

Настройки `compression` и резервной модели доступны только через `config.yaml`.

:::tip
Запустите `hermes config`, чтобы посмотреть текущие настройки auxiliary-моделей. Переопределения отображаются только тогда, когда они отличаются от значений по умолчанию.
:::

## Уровень рассуждений

Настройте, сколько размышлений модель выполняет перед ответом:

```yaml
agent:
  reasoning_effort: ""   # пусто = medium (по умолчанию). Варианты: none, minimal, low, medium, high, xhigh (max)
```

Если значение не задано (по умолчанию), уровень рассуждений равен "medium" — это сбалансированный режим, который хорошо подходит для большинства задач. Если задать значение явно, оно его переопределит: более высокий reasoning effort даёт лучшие результаты на сложных задачах, но требует больше токенов и даёт большую задержку.

Изменить уровень рассуждений можно и во время работы с помощью команды `/reasoning`:

```
/reasoning           # Показать текущий уровень усилий и состояние отображения
/reasoning high      # Установить уровень рассуждений в high
/reasoning none      # Отключить рассуждения
/reasoning show      # Показывать ход рассуждений модели над каждым ответом
/reasoning hide      # Скрыть ход рассуждений модели
```

## Принудительное использование инструментов

Некоторые модели иногда описывают предполагаемые действия словами вместо того, чтобы реально вызывать инструменты (`I would run the tests...` вместо фактического вызова `terminal`). Принудительное использование инструментов добавляет в системную инструкцию подсказку, которая возвращает модель к реальному вызову инструментов.

```yaml
agent:
  tool_use_enforcement: "auto"   # "auto" | true | false | ["model-substring", ...]
```

| Значение | Поведение |
|----------|-----------|
| `"auto"` (по умолчанию) | Включено для моделей, которые соответствуют: `gpt`, `codex`, `gemini`, `gemma`, `grok`. Для всех остальных (Claude, DeepSeek, Qwen и т. д.) выключено. |
| `true` | Всегда включено, независимо от модели. Полезно, если текущая модель часто описывает действия вместо того, чтобы их выполнять. |
| `false` | Всегда выключено, независимо от модели. |
| `["gpt", "codex", "qwen", "llama"]` | Включается только тогда, когда имя модели содержит одну из перечисленных подстрок (без учёта регистра). |

### Что именно внедряется

Когда опция включена, в системную инструкцию могут добавляться три слоя подсказок:

1. **Общее принуждение к использованию инструментов** (для всех совпавших моделей) — требует немедленно вызывать инструменты вместо описания намерений, работать до завершения задачи и никогда не заканчивать ход обещанием что-то сделать позже.

2. **Дисциплина выполнения OpenAI** (только для моделей GPT и Codex) — дополнительная подсказка, которая закрывает характерные для GPT ошибки: отказ от работы после частичного результата, пропуск необходимых предварительных проверок, галлюцинации вместо реальных вызовов инструментов и объявление "done" без проверки.

3. **Операционные подсказки Google** (только для моделей Gemini и Gemma) — краткость, абсолютные пути, параллельные вызовы инструментов и шаблон verify-before-edit.

Для пользователя это прозрачно и влияет только на системную инструкцию. Моделям, которые и так надёжно работают с инструментами (например, Claude), такая подсказка не нужна, поэтому `"auto"` их и исключает.

### Когда включать

Если вы используете модель, которой нет в стандартном auto-списке, и замечаете, что она часто описывает то, что *собирается* сделать, вместо того чтобы реально это сделать, установите `tool_use_enforcement: true` или добавьте подстроку модели в список:

```yaml
agent:
  tool_use_enforcement: ["gpt", "codex", "gemini", "grok", "my-custom-model"]
```

### Проверка изменения файлов

Когда `display.file_mutation_verifier` равен `true` (по умолчанию), Hermes добавляет к финальному ответу ассистента одну строку с предупреждением, если во время хода вызов `write_file` или `patch` не удался и не был позже успешно заменён записью в тот же путь. Это помогает ловить класс ошибок вида "пачка параллельных патчей, половина молча падает, модель рапортует успех", не заставляя вас вручную запускать `git status` после каждого изменения.

Пример footer:

```
⚠️ File-mutation verifier: 3 file(s) were NOT modified this turn despite any wording above that may suggest otherwise. Run `git status` or `read_file` to confirm.
  • concepts/automatic-organization.md — [patch] Could not find match for old_string
  • concepts/lora.md — [patch] Could not find match for old_string
  • concepts/rag-pipeline.md — [patch] Could not find match for old_string
```

Чтобы скрыть footer, установите `file_mutation_verifier: false` (или `HERMES_FILE_MUTATION_VERIFIER=0`). Проверка срабатывает только тогда, когда к концу хода остаются реальные ошибки: если модель повторила неудавшийся patch и успешно завершила его в том же ходу, footer для этого файла не появится.

### Язык UI для статических сообщений

Параметр `display.language` переводит небольшой набор статических сообщений для пользователя: запросы подтверждения в CLI и несколько ответов шлюза на слэш-команды, например уведомления о `restart-drain`, `"approval expired"` и `"goal cleared"`. Он **не** переводит ответы агента, строки логов, вывод инструментов, traceback'и ошибок и описания слэш-команд — всё это остаётся на английском. Если вы хотите, чтобы сам агент отвечал на другом языке, просто укажите это прямо в запросе или в системной инструкции.

Поддерживаемые значения: `en` (по умолчанию), `zh` (упрощённый китайский), `ja` (японский), `de` (немецкий), `es` (испанский), `fr` (французский), `tr` (турецкий), `uk` (украинский). Неизвестные значения откатываются к английскому.

Это можно также задать отдельно для сеанса через переменную `HERMES_LANGUAGE`, которая перекрывает значение из конфига.

```yaml
display:
  language: zh   # Запросы подтверждения в CLI выводятся на китайском
```

| Режим | Что увидите |
|------|-------------|
| `off` | Тихий режим — только финальный ответ |
| `new` | Индикатор инструмента только при его смене |
| `all` | Каждый вызов инструмента с коротким предпросмотром (по умолчанию) |
| `verbose` | Полные аргументы, результаты и debug-логи |

В CLI переключайтесь между этими режимами через `/verbose`. Чтобы использовать `/verbose` в месседжинг-платформах (Telegram, Discord, Slack и т. д.), задайте `tool_progress_command: true` в секции `display` выше. Команда будет переключать режим и сохранять его в конфиг.

### Footer с метаданными выполнения (только gateway)

Когда `display.runtime_footer.enabled: true`, Hermes добавляет к **финальному** сообщению каждого хода gateway небольшой footer со служебным контекстом выполнения — теми же данными, которые CLI показывает в строке состояния (модель, процент контекста, cwd, длительность сеанса, токены, стоимость). По умолчанию выключено; включайте для каждого gateway отдельно, если вашей команде нужно, чтобы каждый ответ содержал сведения о происхождении.

```yaml
display:
  runtime_footer:
    enabled: true
    fields: ["model", "context_pct", "cwd"]   # любые из: model, context_pct, cwd, duration, tokens, cost
```

Слэш-команда `/footer` переключает это во время работы в любом сеансе.

Пример footer, который добавляется к ответу в Telegram/Discord/Slack:

```
— claude-opus-4.7 · 12 tool calls · 2m 14s · $0.042
```

Только **финальное** сообщение хода получает этот footer; промежуточные обновления остаются чистыми.

### Переопределения прогресса для платформ

Разные платформы требуют разной степени многословности. Например, Signal не умеет редактировать сообщения, поэтому каждое обновление прогресса превращается в отдельное сообщение — это шумно. Используйте `display.platforms`, чтобы задавать режимы вывода отдельно для каждой платформы:

```yaml
display:
  tool_progress: all          # глобальное значение по умолчанию
  platforms:
    signal:
      tool_progress: 'off'    # скрывать прогресс в Signal
    telegram:
      tool_progress: verbose  # подробный прогресс в Telegram
    slack:
      tool_progress: 'off'    # тихо в общем рабочем пространстве Slack
```

Платформы без собственных переопределений используют глобальное значение `tool_progress`. Допустимые ключи платформ: `telegram`, `discord`, `slack`, `signal`, `whatsapp`, `matrix`, `mattermost`, `email`, `sms`, `homeassistant`, `dingtalk`, `feishu`, `wecom`, `weixin`, `bluebubbles`, `qqbot`. Устаревший ключ `display.tool_progress_overrides` по-прежнему загружается для обратной совместимости, но считается deprecated и при первой загрузке мигрируется в `display.platforms`.

`interim_assistant_messages` работает только на gateway. Когда он включён, Hermes отправляет завершённые промежуточные обновления ассистента как отдельные сообщения чата. Это независимо от `tool_progress` и не требует потоковой передачи через шлюз.

## Конфиденциальность

```yaml
privacy:
  redact_pii: false  # Удалять PII из контекста LLM (только gateway)
```

Когда `redact_pii` равен `true`, gateway удаляет персонально идентифицируемую информацию из системной инструкции перед отправкой в LLM на поддерживаемых платформах:

| Поле | Обработка |
|------|-----------|
| Номера телефонов (user ID в WhatsApp/Signal) | Хешируются в `user_<12-char-sha256>` |
| User ID | Хешируются в `user_<12-char-sha256>` |
| Chat ID | Хешируется только числовая часть, префикс платформы сохраняется (`telegram:<hash>`) |
| ID домашнего канала | Хешируется только числовая часть |
| Имена пользователей / usernames | **Не затрагиваются** (это пользовательский выбор, публично видимый текст) |

**Поддержка платформ:** маскирование применяется к WhatsApp, Signal и Telegram. Discord и Slack исключены, потому что их системы упоминаний (`<@user_id>`) требуют реальный ID в контексте LLM.

Хеши детерминированы: один и тот же пользователь всегда получает один и тот же хеш, поэтому модель по-прежнему может различать участников групповых чатов. Маршрутизация и доставка внутри используют оригинальные значения.

## Распознавание речи (STT)

```yaml
stt:
  provider: "local"            # "local" | "groq" | "openai" | "mistral"
  local:
    model: "base"              # tiny, base, small, medium, large-v3
  openai:
    model: "whisper-1"         # whisper-1 | gpt-4o-mini-transcribe | gpt-4o-transcribe
  # model: "whisper-1"         # Устаревший fallback-ключ по-прежнему поддерживается
```

Поведение провайдеров:

- `local` использует `faster-whisper`, который работает на вашей машине. Установите его отдельно через `pip install faster-whisper`.
- `groq` использует совместимую с Whisper конечную точку Groq и читает `GROQ_API_KEY`.
- `openai` использует OpenAI speech API и читает `VOICE_TOOLS_OPENAI_KEY`.

Если нужный провайдер недоступен, Hermes автоматически откатывается в таком порядке: `local` → `groq` → `openai`.

Переопределения моделей для Groq и OpenAI задаются через переменные окружения:

```bash
STT_GROQ_MODEL=whisper-large-v3-turbo
STT_OPENAI_MODEL=whisper-1
GROQ_BASE_URL=https://api.groq.com/openai/v1
STT_OPENAI_BASE_URL=https://api.openai.com/v1
```

## Потоковая передача

Потоково передавайте токены в терминал или на платформы обмена сообщениями по мере их поступления, а не ждите полный ответ.

### Поток CLI

```yaml
display:
  streaming: true         # Потоково выводить токены в terminal
  show_reasoning: true    # Дополнительно выводить reasoning/thinking tokens (необязательно)
```

Когда режим включён, ответы появляются токен за токеном внутри блока потоковой передачи. Вызовы инструментов при этом по-прежнему скрываются. Если провайдер не поддерживает потоковую передачу, Hermes автоматически возвращается к обычному выводу.

### Потоковая передача в gateway (Telegram, Discord, Slack)

```yaml
streaming:
  enabled: true           # Включить постепенное редактирование сообщений
  transport: edit         # "edit" (постепенное редактирование сообщений) или "off"
  edit_interval: 0.3      # Секунды между редактированиями сообщения
  buffer_threshold: 40    # Количество символов до принудительного flush редактирования
  cursor: " ▉"            # Курсор, показываемый во время streaming
  fresh_final_after_seconds: 60   # Отправлять свежий финал (Telegram), если предпросмотр старше этого значения; 0 = всегда редактировать на месте
```

Когда режим включён, бот отправляет сообщение на первом токене, а затем постепенно редактирует его по мере поступления новых токенов. Платформы, которые не умеют редактировать сообщения (Signal, Email, Home Assistant), определяются автоматически уже на первой попытке — для такого сеанса потоковый режим мягко отключается, без потока лишних сообщений.

Если нужны отдельные естественные промежуточные обновления ассистента без прогрессивного редактирования токенов, установите `display.interim_assistant_messages: true`.

**Обработка переполнения:** если потоковый текст превышает лимит длины сообщения платформы (примерно 4096 символов), текущее сообщение завершается, и автоматически начинается новое.

**Свежий финал (Telegram):** `editMessageText` в Telegram сохраняет исходную временную метку сообщения, так что у долгого потокового ответа метка первого токена сохранялась бы даже после завершения. Когда `fresh_final_after_seconds > 0` (по умолчанию `60`), завершённый ответ отправляется как совершенно новое сообщение (а устаревший предпросмотр по возможности удаляется), чтобы видимая временная метка в Telegram соответствовала времени завершения. Короткие предпросмотры по-прежнему завершаются на месте. Установите `0`, чтобы всегда редактировать сообщение на месте.

:::note
Потоковая передача выключена по умолчанию. Включите её в `~/.hermes/config.yaml`, чтобы попробовать этот режим.
:::

## Изоляция сеансов в групповых чатах

Управляйте тем, сохраняет ли общий чат один разговор на комнату или один разговор на каждого участника:

```yaml
group_sessions_per_user: true  # true = изоляция по пользователям в группах/каналах, false = один общий сеанс на чат
```

- `true` — значение по умолчанию и рекомендуемая настройка. В каналах Discord, группах Telegram, каналах Slack и похожих общих контекстах каждый отправитель получает свой собственный сеанс, если платформа предоставляет user ID.
- `false` возвращает старый режим общего чата. Это удобно, если вы явно хотите, чтобы Hermes воспринимал канал как один совместный разговор, но тогда пользователи начнут делить контекст, стоимость токенов и состояние прерывания.
- Это не затрагивает личные сообщения. Hermes по-прежнему использует chat/DM ID как обычно.
- Треды в любом случае остаются изолированными от родительского канала; при `true` каждый участник внутри треда тоже получает свой отдельный сеанс.

Подробности и примеры поведения см. в [Сеансах](/user-guide/sessions) и в [руководстве по Discord](/user-guide/messaging/discord).

## Поведение при неавторизованных DM

Задайте, что Hermes делает, когда неизвестный пользователь отправляет direct message:

```yaml
unauthorized_dm_behavior: pair

whatsapp:
  unauthorized_dm_behavior: ignore
```

- `pair` — значение по умолчанию. Hermes отказывает в доступе, но отвечает в DM одноразовым кодом сопряжения.
- `ignore` — молча отбрасывает неавторизованные DM.
- Разделы платформ переопределяют глобальное значение, так что можно оставить pairing включённым в целом, но сделать одну платформу тише.

## Быстрые команды

Определите собственные команды, которые либо выполняют shell-команды без вызова LLM, либо делают alias одной слэш-команды на другую. Exec quick commands не расходуют токены и удобны на платформах обмена сообщениями (Telegram, Discord и т. д.) для быстрых проверок сервера или служебных скриптов.

```yaml
quick_commands:
  status:
    type: exec
    command: systemctl status hermes-agent
  disk:
    type: exec
    command: df -h /
  update:
    type: exec
    command: cd ~/.hermes/hermes-agent && git pull && pip install -e .
  gpu:
    type: exec
    command: nvidia-smi --query-gpu=name,utilization.gpu,memory.used,memory.total --format=csv,noheader
  restart:
    type: alias
    target: /gateway restart
```

Использование: введите `/status`, `/disk`, `/update`, `/gpu` или `/restart` в CLI или на любой платформе обмена сообщениями. Команды `exec` выполняются локально на хосте и возвращают вывод напрямую — без вызова LLM и без расхода токенов. Команды `alias` переписываются в целевую слэш-команду из конфигурации.

- **Таймаут 30 секунд** — долго выполняющиеся команды завершаются с сообщением об ошибке
- **Приоритет** — быстрые команды проверяются раньше skill-команд, так что вы можете переопределять имена навыков
- **Автодополнение** — быстрые команды разрешаются в момент dispatch и не показываются во встроенных таблицах автодополнения слэш-команд
- **Тип** — поддерживаются `exec` и `alias`; для других типов показывается ошибка
- **Работает везде** — CLI, Telegram, Discord, Slack, WhatsApp, Signal, Email, Home Assistant

Команды-ярлыки, состоящие только из одной строки запроса, не являются допустимыми quick commands. Если нужен переиспользуемый сценарий, создайте навык или alias на существующую слэш-команду.

## Выполнение кода

Настройте инструмент `execute_code`:

```yaml
code_execution:
  mode: project                # project (по умолчанию) | strict
  timeout: 300                 # Максимальное время выполнения в секундах
  max_tool_calls: 50           # Максимум вызовов инструментов внутри выполнения кода
```

**`mode`** управляет рабочим каталогом и Python-интерпретатором для скриптов:

- **`project`** (по умолчанию) — скрипты выполняются в рабочем каталоге сеанса с python из активного virtualenv/conda env. Зависимости проекта (`pandas`, `torch`, пакеты проекта) и относительные пути (`.env`, `./data.csv`) разрешаются естественно, так же как их видит `terminal()`.
- **`strict`** — скрипты выполняются в временном staging-каталоге с `sys.executable` (собственным python Hermes). Максимальная воспроизводимость, но project deps и относительные пути работать не будут.

Sanitizing окружения (удаление `*_API_KEY`, `*_TOKEN`, `*_SECRET`, `*_PASSWORD`, `*_CREDENTIAL`, `*_PASSWD`, `*_AUTH`) и белый список инструментов работают одинаково в обоих режимах — переключение режима не меняет уровень безопасности.

## Бэкенды веб-поиска

Инструменты `web_search`, `web_extract` и `web_crawl` поддерживают пять провайдеров бэкенда. Настройте бэкенд в `config.yaml` или через `hermes tools`:

```yaml
web:
  backend: firecrawl    # firecrawl | searxng | parallel | tavily | exa

  # Или используйте ключи на уровне capability, чтобы смешивать провайдеров (например, бесплатный поиск + платное извлечение):
  search_backend: "searxng"
  extract_backend: "firecrawl"
```

| Бэкенд | Переменная окружения | Поиск | Извлечение | Обход |
|---------|----------------------|--------|------------|-------|
| **Firecrawl** (по умолчанию) | `FIRECRAWL_API_KEY` | ✔ | ✔ | ✔ |
| **SearXNG** | `SEARXNG_URL` | ✔ | — | — |
| **Parallel** | `PARALLEL_API_KEY` | ✔ | ✔ | — |
| **Tavily** | `TAVILY_API_KEY` | ✔ | ✔ | ✔ |
| **Exa** | `EXA_API_KEY` | ✔ | ✔ | — |

**Выбор бэкенда:** если `web.backend` не задан, бэкенд автоматически определяется по доступным API-ключам. Если задан только `SEARXNG_URL`, используется SearXNG. Если задан только `EXA_API_KEY`, используется Exa. Если задан только `TAVILY_API_KEY`, используется Tavily. Если задан только `PARALLEL_API_KEY`, используется Parallel. Иначе по умолчанию выбирается Firecrawl.

**SearXNG** — это бесплатный самостоятельно размещаемый метапоисковый движок с упором на приватность, который опрашивает 70+ поисковых систем. API-ключ не нужен — просто укажите `SEARXNG_URL` на ваш инстанс (например, `http://localhost:8080`). SearXNG умеет только искать; для `web_extract` и `web_crawl` нужен отдельный extract provider (задайте `web.extract_backend`). Инструкции по настройке Docker см. в [руководстве по веб-поиску](/user-guide/features/web-search).

**Self-hosted Firecrawl:** укажите `FIRECRAWL_API_URL`, чтобы направить его на ваш собственный инстанс. Когда задан custom URL, API-ключ становится необязательным (на сервере можно отключить auth через `USE_DB_AUTHENTICATION=***`).

**Режимы Parallel search:** задайте `PARALLEL_SEARCH_MODE`, чтобы управлять поведением поиска — `fast`, `one-shot` или `agentic` (по умолчанию `agentic`).

**Exa:** задайте `EXA_API_KEY` в `~/.hermes/.env`. Поддерживаются фильтры `category` (`company`, `research paper`, `news`, `people`, `personal site`, `pdf`) и фильтры по домену/дате.

## Браузер

Настройте поведение автоматизации браузера:

```yaml
browser:
  inactivity_timeout: 120        # Секунды до автоматического закрытия неактивных сеансов
  command_timeout: 30             # Таймаут в секундах для браузерных команд (screenshot, navigate и т. д.)
  record_sessions: false         # Автоматически записывать сеансы браузера в WebM-видео в ~/.hermes/browser_recordings/
  # Необязательное переопределение CDP — если оно задано, Hermes подключается напрямую к вашему
  # Chrome (через /browser connect), а не запускает браузер без интерфейса.
  cdp_url: ""
  # Диспетчер диалогов — управляет нативными JS-диалогами (alert / confirm / prompt)
  # при подключённом CDP-бэкенде (Browserbase, локальный Chrome через
  # /browser connect). В Camofox и в стандартном локальном режиме agent-browser игнорируется.
  dialog_policy: must_respond    # must_respond | auto_dismiss | auto_accept
  dialog_timeout_s: 300          # Автоотмена безопасности при `must_respond` (секунды)
  camofox:
    managed_persistence: false   # Если true, сеансы Camofox сохраняют cookies и логины между перезапусками
    user_id: ""                  # Необязательный внешне управляемый user ID Camofox
    session_key: ""              # Необязательный ключ сеанса, отправляемый, когда Hermes создаёт вкладку
    adopt_existing_tab: false    # Сначала повторно использовать существующую вкладку для этой личности
```

**Политики диалогов:**

- `must_respond` (по умолчанию) — перехватывает диалог, показывает его в `browser_snapshot.pending_dialogs` и ждёт, пока агент вызовет `browser_dialog(action=...)`. Если в течение `dialog_timeout_s` секунд ответа нет, диалог автоматически закрывается, чтобы поток JavaScript страницы не завис навсегда.
- `auto_dismiss` — перехватывает и сразу закрывает. Агент всё равно видит запись о диалоге в `browser_snapshot.recent_dialogs` с `closed_by="auto_policy"` постфактум.
- `auto_accept` — перехватывает и сразу принимает. Полезно на страницах с агрессивными `beforeunload`-диалогами.

Полный сценарий диалогов см. на [странице browser](/user-guide/features/browser#browser_dialog).

Набор browser-инструментов поддерживает несколько провайдеров. Подробности о Browserbase, Browser Use и локальной настройке Chrome CDP см. на [странице browser](/user-guide/features/browser).

## Часовой пояс

Переопределите серверный локальный часовой пояс строкой IANA. Это влияет на временные метки в логах, расписание cron и подстановку времени в системную инструкцию.

```yaml
timezone: "America/New_York"   # IANA timezone (по умолчанию: "" = server-local time)
```

Поддерживаются любые IANA timezone identifiers (например, `America/New_York`, `Europe/London`, `Asia/Kolkata`, `UTC`). Оставьте поле пустым или не указывайте его, если нужен серверный локальный часовой пояс.

## Discord

Настройте поведение, специфичное для Discord, для шлюза обмена сообщениями:

```yaml
discord:
  require_mention: true          # Требовать упоминания, чтобы отвечать в каналах сервера
  free_response_channels: ""     # ID каналов через запятую, где бот отвечает без упоминания
  auto_thread: true              # Автоматически создавать треды при упоминании в каналах
```

- `require_mention` — когда `true` (по умолчанию), бот отвечает в серверных каналах только тогда, когда его упомянули через `@`. DMs всегда работают без упоминания.
- `free_response_channels` — список channel ID через запятую, где бот отвечает на каждое сообщение без требования упоминания.
- `auto_thread` — когда `true` (по умолчанию), упоминания в каналах автоматически создают thread для разговора, сохраняя каналы чистыми (как Slack threading).

## Безопасность

Проверка безопасности до выполнения и скрытие секретов:

```yaml
security:
  redact_secrets: false          # Маскировать шаблоны API-ключ в выводе инструментов и логах (по умолчанию выключено)
  tirith_enabled: true           # Включить проверку безопасности Tirith для терминальных команд
  tirith_path: "tirith"          # Путь к бинарнику tirith (по умолчанию: "tirith" из $PATH)
  tirith_timeout: 5              # Сколько секунд ждать завершения проверки Tirith
  tirith_fail_open: true         # Разрешать выполнение команды, если tirith недоступен
  website_blocklist:             # См. раздел Website Blocklist ниже
    enabled: false
    domains: []
    shared_files: []
```

- `redact_secrets` — когда `true`, автоматически обнаруживает и скрывает шаблоны, похожие на API-ключи, токены и пароли, в выводе инструментов до того, как они попадут в контекст разговора и логи. **Выключено по умолчанию** — включайте, если вы часто работаете с реальными учётными данными в выводе инструментов и хотите подстраховку. Чтобы включить, задайте явно `true`.
- `tirith_enabled` — когда `true`, terminal commands перед выполнением проверяются [Tirith](https://github.com/sheeki03/tirith), чтобы обнаружить потенциально опасные операции.
- `tirith_path` — путь к бинарнику Tirith. Укажите его, если Tirith установлен в нестандартном месте.
- `tirith_timeout` — максимальное число секунд ожидания проверки Tirith. Если проверка не успевает, команды продолжают выполняться.
- `tirith_fail_open` — когда `true` (по умолчанию), команды разрешается выполнять, если Tirith недоступен или завершился с ошибкой. Поставьте `false`, если хотите блокировать команды, когда Tirith не может их проверить.

## Список блокировки сайтов

Запретите доступ к определённым доменам через инструменты web и browser агента:

```yaml
security:
  website_blocklist:
    enabled: false               # Включить блокировку URL (по умолчанию: false)
    domains:                     # Список шаблонов заблокированных доменов
      - "*.internal.company.com"
      - "admin.example.com"
      - "*.local"
    shared_files:                # Загружать дополнительные правила из внешних файлов
      - "/etc/hermes/blocked-sites.txt"
```

Когда опция включена, любой URL, который совпадает с шаблоном заблокированного домена, отклоняется ещё до запуска веб- или browser-инструмента. Это относится к `web_search`, `web_extract`, `browser_navigate` и любым другим инструментам, которые обращаются к URL.

Правила доменов поддерживают:
- Точные домены: `admin.example.com`
- Поддомены с wildcard: `*.internal.company.com` (блокирует все поддомены)
- TLD-маски: `*.local`

В shared files каждая строка — одно правило домена (пустые строки и комментарии `#` игнорируются). Если файл отсутствует или его нельзя прочитать, в лог пишется предупреждение, но остальные web tools не отключаются.

Политика кэшируется на 30 секунд, так что изменения конфига вступают в силу быстро и без перезапуска.

## Умные подтверждения

Настройте, как Hermes обращается с потенциально опасными командами:

```yaml
approvals:
  mode: manual   # manual | smart | off
```

| Режим | Поведение |
|------|----------|
| `manual` (по умолчанию) | Перед выполнением любой помеченной команды спрашивает пользователя. В CLI показывает интерактивный диалог подтверждения. На платформах обмена сообщениями ставит запрос на подтверждение в очередь. |
| `smart` | Использует вспомогательную LLM, чтобы оценить, действительно ли помеченная команда опасна. Команды с низким риском автоматически подтверждаются с сохранением на уровне сеанса. Действительно рискованные команды передаются пользователю. |
| `off` | Пропускает все проверки подтверждения. Эквивалентно `HERMES_YOLO_MODE=true`. **Используйте осторожно.** |

Режим smart особенно полезен, когда хочется меньше уставать от подтверждений: он даёт агенту больше автономии на безопасных операциях, но по-прежнему ловит по-настоящему разрушительные команды.

:::warning
Установка `approvals.mode: off` отключает все проверки безопасности для terminal-команд. Используйте это только в доверенных песочницах.
:::

## Делегирование

Настройте поведение субагентов для инструмента delegate:

```yaml
delegation:
  # model: "google/gemini-3-flash-preview"  # Переопределить модель (пустое значение = наследовать родителя)
  # provider: "openrouter"                  # Переопределить провайдера (пустое значение = наследовать родителя)
  # base_url: "http://localhost:1234/v1"    # Прямая конечная точка, совместимая с OpenAI (имеет приоритет над provider)
  # api_key: "local-key"                    # API-ключ для base_url (иначе берётся OPENAI_API_KEY)
  # api_mode: ""                            # Протокол взаимодействия для base_url: "chat_completions", "codex_responses" или "anthropic_messages". Пустое значение = автоопределение по URL (например, суффикс /anthropic → anthropic_messages). Укажите явно для нестандартных конечных точек, которые эвристика не распознаёт.
  max_concurrent_children: 3                # Параллельно запускаемые дочерние агенты в одной пачке (минимум 1, без потолка). Также через env var `DELEGATION_MAX_CONCURRENT_CHILDREN`.
  max_spawn_depth: 1                        # Ограничение глубины дерева делегирования (1-3). 1 = плоская структура (по умолчанию): родитель порождает только агентов с ролью leaf, которые не могут делегировать дальше. 2 = дочерние агенты с ролью orchestrator могут порождать внуков с ролью leaf. 3 = три уровня.
  orchestrator_enabled: true                # Глобальный выключатель. Если false, `role="orchestrator"` игнорируется, и каждый дочерний агент принудительно получает роль leaf независимо от `max_spawn_depth`.
```

**Переопределение provider:model для субагентов:** по умолчанию субагенты наследуют провайдера и модель родительского агента. Задайте `delegation.provider` и `delegation.model`, чтобы направить субагентов на другую пару provider:model — например, использовать дешёвую или быструю модель для узких подзадач, пока основной агент работает на дорогой reasoning-модели.

**Прямое переопределение конечной точки:** если нужен явный путь с собственной конечной точкой, задайте `delegation.base_url`, `delegation.api_key` и `delegation.model`. Тогда субагенты будут направляться прямо на эту конечную точку, совместимую с OpenAI, и это имеет приоритет над `delegation.provider`. Если `delegation.api_key` не указан, Hermes использует только `OPENAI_API_KEY`.

**Протокол взаимодействия (`api_mode`):** Hermes автоматически определяет протокол взаимодействия по `delegation.base_url` (например, пути, оканчивающиеся на `/anthropic`, → `anthropic_messages`; hostnames Codex / native Anthropic / Kimi-coding продолжают распознаваться как раньше). Для конечных точек, которые эвристика не может классифицировать — например Azure AI Foundry, MiniMax, Zhipu GLM или LiteLLM-прокси перед бэкендом в стиле Anthropic — задайте `delegation.api_mode` явно как `chat_completions`, `codex_responses` или `anthropic_messages`. Оставьте поле пустым (значение по умолчанию), чтобы сохранить автоопределение.

Провайдер делегирования использует ту же схему разрешения учётных данных, что и запуск CLI/шлюза. Поддерживаются все настроенные провайдеры: `openrouter`, `nous`, `copilot`, `zai`, `kimi-coding`, `minimax`, `minimax-cn`. Когда провайдер задан, система автоматически определяет правильные базовые URL, API-ключ и режим API — вручную настраивать учётные данные не нужно.

**Приоритеты:** `delegation.base_url` в конфиге → `delegation.provider` в конфиге → родительский провайдер (наследуется). `delegation.model` в конфиге → родительская модель (наследуется). Если задать только `model` без `provider`, изменится только имя модели, а учётные данные родителя сохранятся (полезно для переключения моделей внутри одного и того же провайдера, например OpenRouter).

**Ширина и глубина:** `max_concurrent_children` ограничивает, сколько субагентов может работать параллельно в одной пачке (по умолчанию `3`, минимум `1`, без верхнего потолка). Его можно также задать через переменную `DELEGATION_MAX_CONCURRENT_CHILDREN`. Когда модель передаёт массив `tasks` длиннее лимита, `delegate_task` возвращает ошибку инструмента с объяснением ограничения, а не молча обрезает список. `max_spawn_depth` управляет глубиной дерева делегирования (ограничено 1-3). При значении по умолчанию `1` делегирование плоское: дочерние агенты не могут порождать внуков, а передача `role="orchestrator"` молча деградирует в `leaf`. Поднимите значение до `2`, чтобы дочерние агенты с ролью orchestrator могли порождать внуков с ролью leaf; до `3` — для трёхуровневых деревьев. Агент включает оркестрацию точечно через `role="orchestrator"`; `orchestrator_enabled: false` принудительно возвращает каждого дочернего агента в `leaf` независимо от прочих настроек. Стоимость растёт мультипликативно: при `max_spawn_depth: 3` и `max_concurrent_children: 3` дерево может дойти до 3×3×3 = 27 параллельных leaf-агентов. См. [Делегирование субагентам → Ограничение глубины и вложенная оркестрация](/user-guide/features/delegation#depth-limit-and-nested-orchestration) для примеров использования.

## Контекстные файлы (SOUL.md, AGENTS.md)

Hermes использует два разных контекста:

| Файл | Назначение | Область действия |
|------|------------|------------------|
| `SOUL.md` | **Основная идентичность агента** — определяет, кем является агент (слот #1 в системной инструкции) | `~/.hermes/SOUL.md` или `$HERMES_HOME/SOUL.md` |
| `.hermes.md` / `HERMES.md` | Инструкции для конкретного проекта (наивысший приоритет) | Поднимаются до git root |
| `AGENTS.md` | Инструкции для конкретного проекта, соглашения по коду | Рекурсивный обход каталогов |
| `CLAUDE.md` | Контекстные файлы Claude Code (тоже обнаруживаются) | Только рабочий каталог |
| `.cursorrules` | Правила Cursor IDE (тоже обнаруживаются) | Только рабочий каталог |
| `.cursor/rules/*.mdc` | Файлы правил Cursor (тоже обнаруживаются) | Только рабочий каталог |

- **SOUL.md** — основная идентичность агента. Он занимает слот #1 в системной инструкции и полностью заменяет встроенную стандартную идентичность. Отредактируйте его, чтобы точно задать роль агента.
- Если SOUL.md отсутствует, пуст или не может быть загружен, Hermes откатывается к встроенной стандартной идентичности.
- **Проектные контекстные файлы подчиняются системе приоритетов** — загружается только один тип (первое совпадение побеждает): `.hermes.md` → `AGENTS.md` → `CLAUDE.md` → `.cursorrules`. SOUL.md всегда загружается отдельно.
- **AGENTS.md** иерархичен: если в подкаталогах тоже есть AGENTS.md, они объединяются.
- Hermes автоматически создаёт стандартный `SOUL.md`, если его ещё нет.
- Все загруженные context files ограничены 20 000 символов и обрезаются интеллектуальным усечением.

См. также:
- [Personality & SOUL.md](/user-guide/features/personality)
- [Файлы контекста](/user-guide/features/context-files)

## Рабочий каталог

| Контекст | По умолчанию |
|---------|--------------|
| **CLI (`hermes`)** | Текущий каталог, из которого вы запускаете команду |

```bash
# В `~/.hermes/.env` или `~/.hermes/config.yaml`:

TERMINAL_CWD=/workspace                # Все терминальные сеансы
```
