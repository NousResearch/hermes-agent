# Файлы контекста

Hermes Agent автоматически находит и загружает файлы контекста, которые определяют его поведение. Часть из них привязана к проекту и обнаруживается в рабочем каталоге. `SOUL.md` теперь глобален для экземпляра Hermes и загружается только из `HERMES_HOME`.

## Поддерживаемые файлы контекста

| Файл | Назначение | Обнаружение |
|------|------------|-------------|
| **.hermes.md** / **HERMES.md** | Инструкции проекта (наивысший приоритет) | Поднимается до корня git-репозитория |
| **AGENTS.md** | Инструкции проекта, соглашения, архитектура | В каталоге запуска и во вложенных каталогах по мере продвижения |
| **CLAUDE.md** | Файлы контекста Claude Code (тоже поддерживаются) | В каталоге запуска и во вложенных каталогах по мере продвижения |
| **SOUL.md** | Глобальная настройка личности и тона для этого экземпляра Hermes | Только `HERMES_HOME/SOUL.md` |
| **.cursorrules** | Соглашения для Cursor IDE | Только текущий каталог |
| **.cursor/rules/*.mdc** | Модули правил Cursor IDE | Только текущий каталог |

:::info Система приоритетов
За один сеанс загружается только **один** тип проектного контекста (первое совпадение выигрывает): `.hermes.md` → `AGENTS.md` → `CLAUDE.md` → `.cursorrules`. **SOUL.md** всегда загружается отдельно как идентичность агента (слот #1).
:::

## AGENTS.md

`AGENTS.md` — основной файл контекста проекта. В нём описывается структура проекта, соглашения, которые нужно соблюдать, и специальные указания.

### Прогрессивное обнаружение во вложенных каталогах

В начале сеанса Hermes загружает `AGENTS.md` из рабочего каталога в системную инструкцию. По мере того как агент перемещается во вложенные каталоги во время работы (`read_file`, `terminal`, `search_files` и т. п.), он **по мере надобности** находит соответствующие файлы контекста в этих каталогах и подставляет их в разговор в тот момент, когда они становятся актуальны.

```
my-project/
├── AGENTS.md              ← Загружается при запуске (в системную инструкцию)
├── frontend/
│   └── AGENTS.md          ← Находится, когда агент читает файлы в frontend/
├── backend/
│   └── AGENTS.md          ← Находится, когда агент читает файлы в backend/
└── shared/
    └── AGENTS.md          ← Находится, когда агент читает файлы в shared/
```

У такого подхода есть два преимущества по сравнению с загрузкой всего сразу:
- **Нет распухания системной инструкции** — подсказки из подкаталогов появляются только тогда, когда нужны
- **Сохраняется кэш промпта** — системная инструкция остаётся стабильной между ходами

Каждый подкаталог проверяется не более одного раза за сеанс. При этом поиск идёт и вверх по родительским каталогам, так что чтение `backend/src/main.py` обнаружит `backend/AGENTS.md`, даже если в `backend/src/` собственного файла контекста нет.

:::info
Файлы контекста из подкаталогов проходят ту же [проверку на безопасность](#безопасность-защита-от-инъекций-промпта), что и файлы, загруженные при запуске. Вредоносные файлы блокируются.
:::

### Пример AGENTS.md

```markdown
# Контекст проекта

Это веб-приложение на Next.js 14 с бэкендом на Python FastAPI.

## Архитектура
- Фронтенд: Next.js 14 с App Router в `/frontend`
- Бэкенд: FastAPI в `/backend`, использует ORM SQLAlchemy
- База данных: PostgreSQL 16
- Развёртывание: Docker Compose на Hetzner VPS

## Соглашения
- Для всего frontend-кода используйте TypeScript в строгом режиме
- Python-код следует PEP 8, повсюду используйте аннотации типов
- Все API-эндпоинты возвращают JSON со схемой `{data, error, meta}`
- Тесты размещайте в каталогах `__tests__/` (frontend) или `tests/` (backend)

## Важные примечания
- Никогда не изменяйте файлы миграций напрямую — используйте команды Alembic
- В `.env.local` лежат реальные API-ключи, не коммитьте этот файл
- Порт фронтенда — 3000, бэкенда — 8000, базы данных — 5432
```

## SOUL.md

`SOUL.md` управляет личностью, тоном и стилем общения агента. Полное описание см. на странице [Личность](/docs/user-guide/features/personality).

**Расположение:**

- `~/.hermes/SOUL.md`
- или `$HERMES_HOME/SOUL.md`, если Hermes запущен с собственным домашним каталогом

Важные детали:

- Hermes автоматически создаёт базовый `SOUL.md`, если его ещё нет
- Hermes загружает `SOUL.md` только из `HERMES_HOME`
- Hermes не ищет `SOUL.md` в рабочем каталоге
- если файл пустой, ничего из `SOUL.md` в промпт не добавляется
- если файл не пустой, его содержимое подставляется дословно после проверки и усечения

## .cursorrules

Hermes совместим с файлом `.cursorrules` и модулями правил `.cursor/rules/*.mdc` из Cursor IDE. Если эти файлы есть в корне проекта и файл более высокого приоритета (`.hermes.md`, `AGENTS.md` или `CLAUDE.md`) не найден, именно они используются как проектный контекст.

Иными словами, ваши существующие соглашения Cursor автоматически применяются и в Hermes.

## Как загружаются файлы контекста

### При запуске (системная инструкция)

Файлы контекста загружаются функцией `build_context_files_prompt()` в `agent/prompt_builder.py`:

1. **Сканирование рабочего каталога** — проверяются `.hermes.md` → `AGENTS.md` → `CLAUDE.md` → `.cursorrules` (первое совпадение выигрывает)
2. **Чтение содержимого** — каждый файл читается как текст UTF-8
3. **Проверка безопасности** — содержимое проверяется на признаки инъекции промпта
4. **Усечение** — файлы длиннее 20 000 символов усекaются с сохранением начала и конца (70% начала, 20% конца, с маркером посередине)
5. **Сборка** — все секции объединяются под заголовком `# Project Context`
6. **Подстановка** — собранное содержимое добавляется в системную инструкцию

### Во время сеанса (прогрессивное обнаружение)

`SubdirectoryHintTracker` в `agent/subdirectory_hints.py` следит за аргументами вызовов инструментов и вытаскивает из них пути к файлам:

1. **Извлечение путей** — после каждого вызова инструмента извлекаются пути из аргументов (`path`, `workdir`, строки shell-команд)
2. **Обход предков** — каталог и до 5 родительских каталогов проверяются по очереди (до уже посещённых каталогов)
3. **Загрузка подсказки** — если найден `AGENTS.md`, `CLAUDE.md` или `.cursorrules`, он загружается (по одному первому совпадению на каталог)
4. **Проверка безопасности** — та же проверка на инъекцию, что и для файлов, загружаемых при старте
5. **Усечение** — не более 8 000 символов на файл
6. **Подстановка** — содержимое добавляется к результату инструмента, чтобы модель видела его естественным образом в контексте

Итоговый фрагмент промпта выглядит примерно так:

```text
# Project Context

The following project context files have been loaded and should be followed:

## AGENTS.md

[Your AGENTS.md content here]

## .cursorrules

[Your .cursorrules content here]

[Your SOUL.md content here]
```

Обратите внимание: содержимое SOUL подставляется напрямую, без дополнительной обёртки.

## Безопасность: защита от инъекций промпта

Все файлы контекста перед включением в промпт проверяются на потенциальную инъекцию. Сканер ищет:

- **Попытки переопределить инструкции**: «ignore previous instructions», «disregard your rules»
- **Шаблоны обмана**: «do not tell the user»
- **Попытки переопределить системную инструкцию**: «system prompt override»
- **Скрытые HTML-комментарии**: `<!-- ignore instructions -->`
- **Скрытые элементы div**: `<div style="display:none">`
- **Эксфильтрацию секретов**: `curl ... $API_KEY`
- **Доступ к секретным файлам**: `cat .env`, `cat credentials`
- **Невидимые символы**: zero-width spaces, bidi-override, word joiners

Если обнаруживается угроза, файл блокируется:

```text
[BLOCKED: AGENTS.md содержит потенциальную инъекцию промпта (prompt_injection). Содержимое не загружено.]
```

:::warning
Этот сканер защищает от распространённых шаблонов инъекций, но не заменяет ручную проверку файлов контекста в общих репозиториях. Всегда проверяйте содержимое `AGENTS.md` в проектах, которые вы не писали сами.
:::

## Ограничения по размеру

| Ограничение | Значение |
|-------------|----------|
| Максимум символов на файл | 20 000 (~7 000 токенов) |
| Доля усечения начала | 70% |
| Доля усечения конца | 20% |
| Маркер усечения | 10% (показывает количество символов и предлагает открыть файл через инструменты) |

Если файл превышает 20 000 символов, сообщение об усечении выглядит так:

```text
[...truncated AGENTS.md: kept 14000+4000 of 25000 chars. Use file tools to read the full file.]
```

## Советы по контекстным файлам

:::tip Лучшие практики для AGENTS.md
1. **Держите файл кратким** — сильно ниже 20K символов; агент читает его на каждом ходе
2. **Стройте структуру заголовками** — используйте `##` для архитектуры, соглашений и важных замечаний
3. **Даёте конкретные примеры** — показывайте предпочтительные шаблоны кода, формы API и соглашения по именованию
4. **Пишите, чего делать не следует** — например: «никогда не изменяйте файлы миграций напрямую»
5. **Указывайте ключевые пути и порты** — агент использует их при работе в терминале
6. **Обновляйте файл по мере развития проекта** — устаревший контекст хуже, чем его отсутствие
:::

### Контекст на уровне подкаталогов

Для монорепозиториев помещайте инструкции, специфичные для подкаталога, во вложенные файлы `AGENTS.md`:

```markdown
<!-- frontend/AGENTS.md -->
# Контекст фронтенда

- Используйте `pnpm`, а не `npm`, для управления пакетами
- Компоненты хранятся в `src/components/`, страницы — в `src/app/`
- Используйте Tailwind CSS, никогда не применяйте inline-стили
- Тесты запускайте через `pnpm test`
```

```markdown
<!-- backend/AGENTS.md -->
# Контекст бэкенда

- Используйте `poetry` для управления зависимостями
- Запускайте dev-сервер через `poetry run uvicorn main:app --reload`
- Для всех эндпоинтов нужны OpenAPI docstrings
- Модели базы данных лежат в `models/`, схемы — в `schemas/`
```
