# Память Honcho

[Honcho](https://github.com/plastic-labs/honcho) — это ИИ-ориентированный бэкенд памяти, который добавляет диалектическое рассуждение и глубокое моделирование пользователя поверх встроенной системы памяти Hermes. Вместо простого key-value-хранилища Honcho поддерживает непрерывную модель того, кто перед вами: предпочтения, стиль общения, цели и поведенческие паттерны пользователя. Для этого он анализирует разговоры уже после того, как они завершились.

:::info Honcho — это плагин-провайдер памяти
Honcho подключается через систему [провайдеров памяти](./memory-providers.md). Все возможности ниже доступны через единый интерфейс провайдера памяти.
:::

## Что добавляет Honcho

| Возможность | Встроенная память | Honcho |
|-----------|-------------------|--------|
| Межсеансовая устойчивость | ✔ `MEMORY.md` / `USER.md` на файловой основе | ✔ на стороне сервера через API |
| Профиль пользователя | ✔ Ручная курация агентом | ✔ Автоматическое диалектическое рассуждение |
| Сводка сеанса | — | ✔ Контекст, привязанный к текущему сеансу |
| Изоляция между агентами | — | ✔ Раздельные профили для каждого участника |
| Режимы наблюдения | — | ✔ `directional` или `unified` |
| Выводы | — | ✔ Серверное рассуждение о паттернах |
| Поиск по истории | ✔ FTS5-поиск по сеансам | ✔ Семантический поиск по выводам |

**Диалектическое рассуждение.** После каждого хода разговора, если сработал `dialecticCadence`, Honcho анализирует обмен репликами и извлекает выводы о предпочтениях, привычках и целях пользователя. Эти выводы накапливаются со временем и дают агенту более глубокое понимание, чем то, что пользователь успел прямо сформулировать. Диалектика поддерживает многошаговую глубину (1–3 прохода) и автоматически выбирает промпты для холодного старта и прогретого сеанса: холодные запросы фокусируются на общих фактах о пользователе, а прогретые — на контексте текущего сеанса.

**Контекст, привязанный к сеансу.** Базовый контекст теперь включает сводку сеанса вместе с представлением пользователя и карточкой участника. Благодаря этому агент понимает, что уже обсуждалось в текущем сеансе, меньше повторяется и лучше удерживает нить разговора.

**Многопользовательские профили.** Когда несколько экземпляров Hermes общаются с одним и тем же пользователем, Honcho хранит отдельные профили участников. Каждый участник видит только свои наблюдения и выводы, поэтому контекст не смешивается между профилями.

## Настройка

```bash
hermes memory setup    # выбрать "honcho" в списке провайдеров
```

Или задайте вручную:

```yaml
# ~/.hermes/config.yaml
memory:
  provider: honcho
```

```bash
echo 'HONCHO_API_KEY=***' >> ~/.hermes/.env
```

Получить API-ключ можно на [honcho.dev](https://honcho.dev).

## Архитектура

### Двухслойная подстановка контекста

На каждом ходе (в режимах `hybrid` или `context`) Honcho собирает два слоя контекста и подставляет их в системную инструкцию:

1. **Базовый контекст** — сводка сеанса, представление пользователя, карточка пользовательского участника, самопредставление ИИ и карточка ИИ-участника. Обновляется по `contextCadence`. Это слой «кто этот пользователь».
2. **Диалектическая надстройка** — сгенерированное LLM рассуждение о текущем состоянии и потребностях пользователя. Обновляется по `dialecticCadence`. Это слой «что важно прямо сейчас».

Оба слоя объединяются и обрезаются по бюджету `contextTokens`, если он задан.

### Выбор промптов для холодного старта и прогретого сеанса

Диалектика автоматически выбирает одну из двух стратегий:

- **Холодный старт** — если базового контекста ещё нет. Запрос общего вида: «Кто этот человек? Каковы его предпочтения, цели и стиль работы?»
- **Прогретый сеанс** — если базовый контекст уже есть. Запрос с привязкой к текущему сеансу: «Что из обсуждённого в этом сеансе сейчас наиболее релевантно?»

### Три независимые ручки конфигурации

Стоимость и глубина контролируются тремя независимыми параметрами:

| Параметр | Что управляет | Значение по умолчанию |
|----------|---------------|-----------------------|
| `contextCadence` | Сколько ходов проходит между вызовами `context()` API | `1` |
| `dialecticCadence` | Сколько ходов проходит между вызовами `peer.chat()` | `2` (рекомендуется 1–5) |
| `dialecticDepth` | Сколько проходов `.chat()` выполняется за один вызов диалектики | `1` |

Эти параметры независимы: можно часто обновлять базовый контекст и редко запускать диалектику, либо наоборот. Например, `contextCadence: 1, dialecticCadence: 5, dialecticDepth: 2` обновляет базовый слой на каждом ходе, запускает диалектику раз в пять ходов и делает по два прохода на каждый запуск.

### Глубина диалектики

Если `dialecticDepth > 1`, каждый запуск диалектики проходит несколько итераций `.chat()`:

- **Проход 0**: холодный/тёплый промпт
- **Проход 1**: самопроверка — выявление пробелов и сбор свидетельств из недавних сеансов
- **Проход 2**: согласование — проверка противоречий между предыдущими проходами и финальная сводка

Каждый проход использует пропорциональный уровень рассуждения: более лёгкий на ранних проходах и базовый на основном. При необходимости можно переопределить уровни через `dialecticDepthLevels`, например `["minimal", "medium", "high"]` для глубины 3.

Если предыдущий проход уже дал сильный сигнал и развёрнутый ответ, последующие проходы могут завершиться раньше — глубина 3 не всегда означает три вызова LLM.

### Предзагрузка на старте сеанса

При инициализации сеанса Honcho запускает диалектический вызов в фоне на полной глубине `dialecticDepth` и передаёт результат прямо в сборку контекста для первого хода. Один проход на холодном участнике часто даёт слишком скромный результат, поэтому многошаговая глубина сначала выполняет цикл audit/reconcile ещё до первого сообщения пользователя. Если к первому ходу предзагрузка ещё не завершилась, система переходит на синхронный вызов с ограниченным таймаутом.

### Адаптивный уровень рассуждения

Автоматически подставляемая диалектика масштабирует `dialecticReasoningLevel` по длине запроса: +1 уровень при ≥120 символах, +2 при ≥400, с ограничением `reasoningLevelCap` (по умолчанию `"high"`). Отключите это через `reasoningHeuristic: false`, если хотите жёстко зафиксировать `dialecticReasoningLevel` для каждого автоматического вызова.

## Параметры конфигурации

Honcho настраивается в `~/.honcho/config.json` (глобально) или в `$HERMES_HOME/honcho.json` (локально для профиля). Мастер настройки делает это за вас.

### Полный справочник

| Ключ | По умолчанию | Описание |
|-----|--------------|----------|
| `contextTokens` | `null` (без лимита) | Бюджет токенов для автоматически подставляемого контекста на ход. Усечение идёт по границам слов |
| `contextCadence` | `1` | Минимум ходов между вызовами `context()` API |
| `dialecticCadence` | `2` | Минимум ходов между LLM-вызовами `peer.chat()`. Применяется к режимам `hybrid` и `context` |
| `dialecticDepth` | `1` | Количество проходов `.chat()` за один запуск диалектики. Диапазон 1–3 |
| `dialecticDepthLevels` | `null` | Необязательный массив уровней reasoning на каждый проход, например `["minimal", "low", "medium"]` |
| `dialecticReasoningLevel` | `'low'` | Базовый уровень reasoning: `minimal`, `low`, `medium`, `high`, `max` |
| `dialecticDynamic` | `true` | Позволяет модели переопределять уровень reasoning для каждого вызова через параметр инструмента |
| `dialecticMaxChars` | `600` | Максимум символов диалектического результата, подставляемых в системную инструкцию |
| `recallMode` | `'hybrid'` | `hybrid` (автоподстановка + инструменты), `context` (только подстановка), `tools` (только инструменты) |
| `writeFrequency` | `'async'` | Когда сбрасывать сообщения: `async` (фон), `turn` (синхронно), `session` (пакетом в конце) или число N |
| `saveMessages` | `true` | Сохранять ли сообщения в API Honcho |
| `observationMode` | `'directional'` | `directional` (всё включено) или `unified` (общий пул). Может быть переопределён объектом `observation` |
| `messageMaxChars` | `25000` | Максимум символов на сообщение; при превышении сообщение режется на части |
| `dialecticMaxInputChars` | `10000` | Максимум символов входа для запроса `peer.chat()` |
| `sessionStrategy` | `'per-directory'` | `per-directory`, `per-repo`, `per-session`, `global` |

**Стратегия сеансов** определяет, как Honcho сопоставляет сеансы с вашей работой:

- `per-session` — каждый запуск `hermes` создаёт новый сеанс. Чистый старт, память живёт через инструменты.
- `per-directory` — один сеанс Honcho на рабочий каталог.
- `per-repo` — один сеанс на git-репозиторий.
- `global` — один сеанс на все каталоги.

**Режим извлечения** управляет тем, как память попадает в разговор:

- `hybrid` — контекст автоматически подставляется в системную инструкцию, и при этом доступны инструменты
- `context` — только автоподстановка, инструменты скрыты
- `tools` — только инструменты, без автоподстановки

## Наблюдение (Directional vs. Unified) {#observation-directional-vs-unified}

Honcho моделирует разговор как обмен сообщениями между участниками. У каждого участника есть два переключателя наблюдения, которые соответствуют `SessionPeerConfig`:

| Переключатель | Эффект |
|--------------|--------|
| `observeMe` | Honcho строит представление об этом участнике на основе его собственных сообщений |
| `observeOthers` | Этот участник наблюдает сообщения другого участника и питает cross-peer reasoning |

Два участника × два переключателя = четыре флага. `observationMode` — это удобный пресет:

| Пресет | Пользователь | AI | Смысл |
|--------|-------------|----|-------|
  | `"directional"` (по умолчанию) | me: on, others: on | me: on, others: on | Полное взаимное наблюдение. Включает межучастническую диалектику |
  | `"unified"` | me: on, others: off | me: off, others: on | Общий пул: ИИ наблюдает только сообщения пользователя, а пользовательский участник моделирует только себя |

При необходимости пресет можно переопределить явным блоком `observation`:

```json
"observation": {
  "user": { "observeMe": true, "observeOthers": true },
  "ai":   { "observeMe": true, "observeOthers": false }
}
```

Частые сценарии:

| Цель | Конфигурация |
|------|--------------|
| Полное наблюдение для большинства пользователей | `"observationMode": "directional"` |
| ИИ не должен пере-моделировать пользователя из собственных ответов | `"ai": {"observeMe": true, "observeOthers": false}` |
| Жёсткий режим персоны, где ИИ не должен обновляться из самонаблюдения | `"ai": {"observeMe": false, "observeOthers": true}` |

Серверные переключатели, заданные в [панели Honcho](https://app.honcho.dev), имеют приоритет над локальными значениями по умолчанию — при старте сеанса Hermes синхронизирует их обратно.

## Инструменты

Когда Honcho активен как провайдер памяти, становятся доступны пять инструментов:

| Инструмент | Назначение |
|------------|------------|
| `honcho_profile` | Читать или обновлять карточку участника — передайте `card`, чтобы записать данные, или ничего, чтобы прочитать |
| `honcho_search` | Семантический поиск по контексту — выдаёт сырой фрагмент без LLM-синтеза |
| `honcho_context` | Полный контекст сеанса — сводка, представление, карточка и недавние сообщения |
| `honcho_reasoning` | Синтезированный ответ от LLM Honcho — передайте `reasoning_level` (`minimal/low/medium/high/max`) |
| `honcho_conclude` | Создавать или удалять выводы — передайте `conclusion` для создания или `delete_id` для удаления |

## Команды CLI

Подкоманда `hermes honcho` регистрируется **только когда Honcho выбран активным провайдером памяти** (`memory.provider: honcho` в `config.yaml`). Сначала выполните `hermes memory setup` и выберите Honcho — подкоманда появится при следующем запуске.

```bash
hermes honcho status          # статус подключения, конфиг и ключевые параметры
hermes honcho setup           # перенаправление на `hermes memory setup`
hermes honcho strategy        # показать или задать session strategy
hermes honcho peer            # показать или обновить имена участников и уровень reasoning
hermes honcho mode            # показать или задать recall mode
hermes honcho tokens          # показать или задать token budget
hermes honcho identity        # задать или показать identity ИИ-участника
hermes honcho sync            # синхронизировать конфиг Honcho со всеми профилями
hermes honcho peers           # показать идентичности участников по всем профилям
hermes honcho sessions        # показать сопоставления сеансов Honcho
hermes honcho map             # сопоставить текущий каталог с именем сеанса Honcho
hermes honcho enable          # включить Honcho для активного профиля
hermes honcho disable         # отключить Honcho для активного профиля
hermes honcho migrate         # пошаговая миграция со старого openclaw-honcho
```

## Переход с `hermes honcho`

Если вы раньше использовали отдельный `hermes honcho setup`:

1. Текущая конфигурация (`honcho.json` или `~/.honcho/config.json`) сохраняется
2. Серверные данные — память, выводы и профили пользователя — остаются на месте
3. Чтобы снова включить систему, просто задайте `memory.provider: honcho` в `config.yaml`

Повторная регистрация или повторная настройка не нужны. Запустите `hermes memory setup` и выберите `honcho` — мастер увидит уже существующую конфигурацию.
