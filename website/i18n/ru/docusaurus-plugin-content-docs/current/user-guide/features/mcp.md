# MCP (Model Context Protocol)

MCP позволяет Hermes Agent подключаться к внешним серверам инструментов, чтобы агент мог использовать инструменты, которые живут вне самого Hermes — GitHub, базы данных, файловые системы, браузерные стеки, внутренние API и многое другое.

Если вы когда-нибудь хотели, чтобы Hermes воспользовался уже существующим инструментом, MCP обычно даёт самый чистый способ это сделать.

## Что даёт MCP

- Доступ к внешним экосистемам инструментов без написания собственного нативного инструмента Hermes
- Локальные stdio-серверы и удалённые HTTP MCP-серверы в одной конфигурации
- Автоматическое обнаружение и регистрация инструментов при запуске
- Утилитарные обёртки для MCP-ресурсов и prompt-ов, если сервер это поддерживает
- Фильтрация на уровне каждого сервера, чтобы показывать Hermes только те MCP-инструменты, которые вам действительно нужны

## Быстрый старт

1. Установите поддержку MCP (она уже включена, если вы использовали стандартный скрипт установки):

```bash
cd ~/.hermes/hermes-agent
uv pip install -e ".[mcp]"
```

2. Добавьте MCP-сервер в `~/.hermes/config.yaml`:

```yaml
mcp_servers:
  filesystem:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-filesystem", "/home/user/projects"]
```

3. Запустите Hermes:

```bash
hermes chat
```

4. Попросите Hermes воспользоваться возможностью, предоставленной через MCP.

Например:

```text
Покажи файлы в /home/user/projects и кратко опиши структуру репозитория.
```

Hermes обнаружит инструменты MCP-сервера и будет использовать их как любые другие инструменты.

## Два типа MCP-серверов

### stdio-серверы

stdio-серверы работают как локальные подпроцессы и общаются через stdin/stdout.

```yaml
mcp_servers:
  github:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "***"
```

Используйте stdio-серверы, если:
- сервер установлен локально
- вам нужен минимальный латентный доступ к локальным ресурсам
- в документации MCP-сервера показаны поля `command`, `args` и `env`

### HTTP-серверы

HTTP MCP-серверы — это удалённые конечные точки, к которым Hermes подключается напрямую.

```yaml
mcp_servers:
  remote_api:
    url: "https://mcp.example.com/mcp"
    headers:
      Authorization: "Bearer ***"
```

Используйте HTTP-серверы, если:
- MCP-сервер размещён где-то ещё
- ваша организация публикует внутренние MCP-эндпоинты
- вы не хотите, чтобы Hermes запускал локальный подпроцесс ради этой интеграции

## Базовый справочник по конфигурации

Hermes читает MCP-конфигурацию из `~/.hermes/config.yaml` в разделе `mcp_servers`.

### Общие ключи

| Ключ | Тип | Значение |
|---|---|---|
| `command` | string | Исполняемый файл для stdio MCP-сервера |
| `args` | list | Аргументы для stdio-сервера |
| `env` | mapping | Переменные окружения, передаваемые stdio-серверу |
| `url` | string | HTTP MCP-эндпоинт |
| `headers` | mapping | HTTP-заголовки для удалённых серверов |
| `timeout` | number | Таймаут вызова инструмента |
| `connect_timeout` | number | Таймаут первоначального подключения |
| `enabled` | bool | Если `false`, Hermes полностью пропускает сервер |
| `supports_parallel_tool_calls` | bool | Если `true`, инструменты этого сервера могут запускаться параллельно |
| `tools` | mapping | Фильтрация инструментов и политика утилит для конкретного сервера |

### Минимальный пример для stdio

```yaml
mcp_servers:
  filesystem:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"]
```

### Минимальный пример для HTTP

```yaml
mcp_servers:
  company_api:
    url: "https://mcp.internal.example.com"
    headers:
      Authorization: "Bearer ***"
```

## Встроенные пресеты

Для известных MCP-серверов команда `hermes mcp add` принимает флаг `--preset`, который подставляет транспортные детали, чтобы вам не пришлось вручную искать `command` и `args`. Пресет задаёт только значения по умолчанию — всё остальное (env vars, headers, фильтрация), переданное в той же команде, имеет приоритет.

| Пресет | Что настраивает |
|---|---|
| `codex` | MCP-сервер Codex CLI (`codex mcp-server` через stdio). Требует наличия `codex` в PATH. |

```bash
# Добавить Codex CLI как MCP-сервер одной командой
hermes mcp add codex --preset codex
```

Это эквивалентно:

```yaml
mcp_servers:
  codex:
    command: "codex"
    args: ["mcp-server"]
```

Вы можете выбрать любое локальное имя (`hermes mcp add my-codex --preset codex` тоже подходит); пресет лишь подставляет значения `command`/`args` по умолчанию.

## Как Hermes регистрирует MCP-инструменты

Hermes добавляет к MCP-инструментам префикс, чтобы они не конфликтовали со встроенными именами:

```text
mcp_<server_name>_<tool_name>
```

Примеры:

| Сервер | MCP-инструмент | Зарегистрированное имя |
|---|---|---|
| `filesystem` | `read_file` | `mcp_filesystem_read_file` |
| `github` | `create-issue` | `mcp_github_create_issue` |
| `my-api` | `query.data` | `mcp_my_api_query_data` |

На практике вам обычно не нужно вызывать префиксованное имя вручную — Hermes видит инструмент и сам выбирает его в процессе обычного рассуждения.

## Утилитарные MCP-инструменты

Если сервер поддерживает эти возможности, Hermes также регистрирует вспомогательные инструменты вокруг MCP-ресурсов и prompt-ов:

- `list_resources`
- `read_resource`
- `list_prompts`
- `get_prompt`

Они регистрируются отдельно для каждого сервера с тем же шаблоном префикса, например:

- `mcp_github_list_resources`
- `mcp_github_get_prompt`

### Важно

Эти утилитарные инструменты теперь учитывают поддерживаемые возможности:
- Hermes регистрирует обёртки для ресурсов только если MCP-сеанс действительно поддерживает операции с ресурсами
- Hermes регистрирует обёртки для prompt-ов только если MCP-сеанс действительно поддерживает операции с prompt-ами

Так что сервер, который предоставляет вызываемые инструменты, но не поддерживает ресурсы или prompt-ы, не получит эти дополнительные обёртки.

## Фильтрация на уровне сервера

Вы можете управлять тем, какие инструменты каждый MCP-сервер добавляет в Hermes, и таким образом тонко контролировать пространство инструментов.

### Отключить сервер целиком

```yaml
mcp_servers:
  legacy:
    url: "https://mcp.legacy.internal"
    enabled: false
```

Если `enabled: false`, Hermes полностью пропускает сервер и даже не пытается подключиться.

### Белый список инструментов сервера

```yaml
mcp_servers:
  github:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "***"
    tools:
      include: [create_issue, list_issues]
```

Будут зарегистрированы только эти MCP-инструменты.

### Чёрный список инструментов сервера

```yaml
mcp_servers:
  stripe:
    url: "https://mcp.stripe.com"
    tools:
      exclude: [delete_customer]
```

Будут зарегистрированы все инструменты сервера, кроме исключённых.

### Правило приоритета

Если присутствуют оба списка:

```yaml
tools:
  include: [create_issue]
  exclude: [create_issue, delete_issue]
```

`include` выигрывает.

### Можно фильтровать и утилитарные инструменты

Можно отдельно отключить и обёртки, которые добавляет сам Hermes:

```yaml
mcp_servers:
  docs:
    url: "https://mcp.docs.example.com"
    tools:
      prompts: false
      resources: false
```

Это означает:
- `tools.resources: false` отключает `list_resources` и `read_resource`
- `tools.prompts: false` отключает `list_prompts` и `get_prompt`

### Полный пример

```yaml
mcp_servers:
  github:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "***"
    tools:
      include: [create_issue, list_issues, search_code]
      prompts: false

  stripe:
    url: "https://mcp.stripe.com"
    headers:
      Authorization: "Bearer ***"
    tools:
      exclude: [delete_customer]
      resources: false

  legacy:
    url: "https://mcp.legacy.internal"
    enabled: false
```

## Что происходит, если всё отфильтровано

Если ваша конфигурация отфильтровывает все вызываемые инструменты и отключает либо не включает все поддерживаемые утилиты, Hermes не создаёт для этого сервера пустой набор инструментов MCP во время выполнения.

Так список инструментов остаётся чистым.

## Поведение во время работы

### Обнаружение при старте

Hermes обнаруживает MCP-серверы при запуске и регистрирует их инструменты в обычном реестре инструментов.

### Динамическое обнаружение инструментов

MCP-серверы могут уведомлять Hermes об изменениях доступных инструментов во время работы, отправляя уведомление `notifications/tools/list_changed`. Когда Hermes получает такое уведомление, он автоматически повторно запрашивает список инструментов сервера и обновляет реестр — без ручного `/reload-mcp`.

Это полезно для MCP-серверов, возможности которых меняются динамически (например, сервер добавляет инструменты после загрузки новой схемы базы данных или убирает их, когда служба уходит в офлайн).

Обновление защищено блокировкой, чтобы частые уведомления от одного и того же сервера не запускали перекрывающиеся обновления. Уведомления об изменении prompt-ов и ресурсов (`prompts/list_changed`, `resources/list_changed`) принимаются, но пока не обрабатываются.

### Перезагрузка

Если вы изменили MCP-конфигурацию, используйте:

```text
/reload-mcp
```

Это перечитывает MCP-серверы из конфигурации и обновляет список доступных инструментов. Для изменений инструментов, которые сервер сам передаёт во время работы, см. раздел [Динамическое обнаружение инструментов](#динамическое-обнаружение-инструментов) выше.
