# Постоянная память

Hermes Agent использует ограниченную, тщательно отобранную память, которая сохраняется между сеансами. Благодаря этому он помнит ваши предпочтения, проекты, окружение и усвоенные сведения.

## Как это работает

Память агента состоит из двух файлов:

| Файл | Назначение | Лимит символов |
|------|------------|----------------|
| **MEMORY.md** | Личные заметки агента — факты об окружении, соглашения, усвоенные сведения | 2 200 символов (~800 токенов) |
| **USER.md** | Профиль пользователя — ваши предпочтения, стиль общения, ожидания | 1 375 символов (~500 токенов) |

Оба файла хранятся в `~/.hermes/memories/` и при запуске сеанса подставляются в системную инструкцию как замороженный снимок. Агент управляет собственной памятью через инструмент `memory` — он может добавлять, заменять и удалять записи.

:::info
Ограничения по размеру помогают держать память сфокусированной. Когда память заполняется, агент сворачивает или заменяет записи, чтобы освободить место под новые сведения.
:::

## Как память выглядит в системной инструкции

В начале каждого сеанса память загружается с диска и отображается в системной инструкции как замороженный блок:

```
══════════════════════════════════════════════
MEMORY (your personal notes) [67% — 1,474/2,200 chars]
══════════════════════════════════════════════
User's project is a Rust web service at ~/code/myapi using Axum + SQLx
§
This machine runs Ubuntu 22.04, has Docker and Podman installed
§
User prefers concise responses, dislikes verbose explanations
```

Формат включает:
- заголовок, показывающий, какое хранилище используется (`MEMORY` или `USER PROFILE`)
- процент использования и количество символов, чтобы агент видел остаток места
- отдельные записи, разделённые символом `§`
- многострочные записи

**Паттерн замороженного снимка:** подстановка памяти в системную инструкцию делается один раз в начале сеанса и больше не меняется до следующего сеанса. Это сделано специально: так сохраняется cache prefix для производительности. Когда во время сеанса агент добавляет или удаляет записи, изменения сразу сохраняются на диск, но в системной инструкции они появятся только в следующем сеансе. Ответы инструментов всегда показывают актуальное состояние.

## Действия с памятью

Агент использует инструмент `memory` с такими действиями:

- **add** — добавить новую запись
- **replace** — заменить существующую запись обновлённым содержимым (поиск ведётся по подстроке через `old_text`)
- **remove** — удалить запись, которая больше не нужна (тоже по подстроке через `old_text`)

Действия `read` не существует — содержимое памяти автоматически подставляется в системную инструкцию при старте сеанса. Агент видит свои воспоминания как часть контекста беседы.

### Сопоставление по подстроке

Для `replace` и `remove` используется короткое уникальное совпадение по подстроке — полный текст записи указывать не нужно. Параметр `old_text` должен содержать достаточно уникальный фрагмент, чтобы однозначно найти одну запись:

```python
# Если в памяти есть "User prefers dark mode in all editors"
memory(action="replace", target="memory",
       old_text="dark mode",
       content="User prefers light mode in VS Code, dark mode in terminal")
```

Если подстрока совпадает с несколькими записями, возвращается ошибка с просьбой указать более точное совпадение.

## Два хранилища

### `memory` — личные заметки агента

Используется для информации, которую агенту нужно помнить об окружении, рабочих процессах и усвоенных уроках:

- факты об окружении (ОС, инструменты, структура проекта)
- соглашения проекта и конфигурация
- особенности инструментов и обходные пути, которые пришлось найти
- записи о завершённых задачах
- навыки и приёмы, которые сработали

### `user` — профиль пользователя

Используется для информации об идентичности пользователя, предпочтениях и стиле общения:

- имя, роль, часовой пояс
- предпочтения по общению (кратко или подробно, формат ответа)
- вещи, которые не нравятся
- привычки рабочего процесса
- уровень технической подготовки

## Что сохранять, а что пропускать

### Сохранять стоит автоматически

Агент сохраняет сведения сам — спрашивать не нужно. Сохраняйте, если он узнал:

- **Предпочтения пользователя:** «Я предпочитаю TypeScript вместо JavaScript» → сохранить в `user`
- **Факты об окружении:** «Этот сервер работает на Debian 12 с PostgreSQL 16» → сохранить в `memory`
- **Поправки:** «Не используй `sudo` для команд Docker, пользователь состоит в группе docker» → сохранить в `memory`
- **Соглашения:** «В проекте используются табы, ширина строки 120 символов, docstring в стиле Google» → сохранить в `memory`
- **Выполненную работу:** «Перенёс базу с MySQL на PostgreSQL 2026-01-15» → сохранить в `memory`
- **Явные запросы:** «Запомни, что ротация API-ключей у меня раз в месяц» → сохранить в `memory`

### Пропускать стоит

- **Тривиальные или очевидные сведения:** «Пользователь спрашивал про Python» — слишком расплывчато
- **Факты, которые легко перепроверить:** «Python 3.12 поддерживает вложенные f-string» — проще найти поиском
- **Сырые дампы данных:** большие блоки кода, журналы, таблицы — слишком объёмно для памяти
- **Одноразовый контекст сессии:** временные пути к файлам, разовый отладочный контекст
- **Сведения, уже описанные в контекстных файлах:** содержимое SOUL.md и AGENTS.md

## Управление ёмкостью

У памяти жёсткие лимиты по символам, чтобы системная инструкция не разрасталась:

| Хранилище | Лимит | Типичные записи |
|----------|-------|----------------|
| memory | 2 200 символов | 8–15 записей |
| user | 1 375 символов | 5–10 записей |

### Что происходит, когда память заполнена

Когда вы пытаетесь добавить запись, которая превысит лимит, инструмент вернёт ошибку:

```json
{
  "success": false,
  "error": "Memory at 2,100/2,200 chars. Adding this entry (250 chars) would exceed the limit. Replace or remove existing entries first.",
  "current_entries": ["..."],
  "usage": "2,100/2,200"
}
```

В этом случае агент должен:
1. Посмотреть текущие записи, которые вернул инструмент
2. Определить, какие записи можно убрать или объединить
3. Использовать `replace`, чтобы слить похожие записи в одну, более компактную
4. После этого добавить новую запись через `add`

**Лучший подход:** когда память заполнена более чем на 80% (это видно в заголовке системной инструкции), перед добавлением новых данных стоит выполнить консолидацию записей. Например, объединить три отдельных заметки о проекте X в одну более ёмкую запись.

### Практические примеры хороших записей в память

**Компактные, насыщенные информацией записи работают лучше всего:**

```
# Хорошо: несколько связанных фактов в одной записи
Пользователь работает на macOS 14 Sonoma, использует Homebrew, у него установлены Docker Desktop и Podman. Shell: zsh с oh-my-zsh. Редактор: VS Code с привязками клавиш Vim.

# Хорошо: конкретное и применимое соглашение
Проект ~/code/api использует Go 1.22, sqlc для запросов к БД и chi router. Тесты запускать через 'make test'. CI — через GitHub Actions.

# Хорошо: полезный урок с контекстом
Стадировочный сервер (10.0.1.50) требует SSH-порт 2222, а не 22. Ключ лежит в ~/.ssh/staging_ed25519.

# Плохо: слишком расплывчато
У пользователя есть проект.

# Плохо: слишком многословно
5 января 2026 года пользователь попросил меня посмотреть его проект,
расположенный в ~/code/api. Я обнаружил, что он использует Go 1.22 и...
```

## Защита от дубликатов

Система памяти автоматически отклоняет точные дубликаты записей. Если вы попытаетесь добавить содержимое, которое уже существует, она вернёт успех с сообщением вроде «no duplicate added».

## Проверка безопасности

Перед тем как принять запись, система памяти проверяет её на шаблоны инъекций и эксфильтрации, поскольку эти данные подставляются в системную инструкцию. Содержимое, совпадающее с шаблонами угроз (инъекция промпта, эксфильтрация учётных данных, SSH backdoor) или содержащее невидимые символы Unicode, блокируется.

## Поиск по сеансам

Помимо `MEMORY.md` и `USER.md`, агент может искать по прошлым разговорам с помощью инструмента `session_search`:

- Все CLI- и messaging-сеансы хранятся в SQLite (`~/.hermes/state.db`) и доступны через полнотекстовый поиск FTS5
- Запросы поиска возвращают реальные сообщения из базы — без LLM-суммаризации и без усечения
- Агент может находить то, что обсуждал даже несколько недель назад, если этого нет в активной памяти
- Агент также может прокручивать найденный сеанс вперёд и назад

```bash
hermes sessions list    # Просмотреть прошлые сеансы
```

См. [Инструмент поиска по сеансам](/docs/user-guide/sessions#session-search-tool) — там описаны три формы вызова (обнаружение / прокрутка / просмотр) и формат ответа.

### `session_search` против памяти

| Возможность | Постоянная память | Поиск по сеансам |
|-------------|------------------|------------------|
| **Ёмкость** | ~1 300 токенов суммарно | Не ограничена (все сеансы) |
| **Скорость** | Мгновенно (в системной инструкции) | ~20 мс на FTS5-запрос, ~1 мс на прокрутку |
| **Стоимость** | Стоимость токенов в каждом промпте | Бесплатно — без LLM-вызовов |
| **Сценарий использования** | Ключевые факты, которые должны быть всегда под рукой | Поиск конкретного разговора из прошлого |
| **Управление** | Редактируется вручную агентом | Автоматически — хранятся все сеансы |
| **Стоимость токенов** | Фиксированная на сеанс (~1 300 токенов) | По запросу (только когда ищем) |

**Память** нужна для критически важных фактов, которые всегда должны быть в контексте. **Поиск по сеансам** нужен для вопросов вроде «мы обсуждали это на прошлой неделе?» — то есть когда требуется найти конкретику в старых разговорах.

## Конфигурация

```yaml
# В ~/.hermes/config.yaml
memory:
  memory_enabled: true
  user_profile_enabled: true
  memory_char_limit: 2200   # ~800 токенов
  user_char_limit: 1375     # ~500 токенов
```

## Внешние провайдеры памяти

Для более глубокой, постоянной памяти, выходящей за рамки `MEMORY.md` и `USER.md`, Hermes поставляется с 8 плагинами внешних провайдеров памяти — включая Honcho, OpenViking, Mem0, Hindsight, Holographic, RetainDB, ByteRover и Supermemory.

Внешние провайдеры работают **параллельно** со встроенной памятью (а не вместо неё) и добавляют такие возможности, как графы знаний, семантический поиск, автоматическое извлечение фактов и межсеансовое моделирование пользователя.

```bash
hermes memory setup      # выбрать провайдера и настроить его
hermes memory status     # проверить, что активно
```

Полные сведения по каждому провайдеру, инструкции по настройке и сравнение см. в [руководстве по провайдерам памяти](./memory-providers.md).
