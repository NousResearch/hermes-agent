# Настройка Discord

Hermes Agent интегрируется с Discord как бот, так что вы можете общаться с ИИ-ассистентом через личные сообщения или каналы сервера. Бот принимает сообщения, прогоняет их через весь пайплайн Hermes Agent, включая инструменты, память и рассуждение, и отвечает в реальном времени. Поддерживаются текст, голосовые сообщения, файловые вложения и слэш-команды.

Перед настройкой полезно понять главное: как Hermes ведёт себя уже после добавления на сервер.

## Как ведёт себя Hermes

| Контекст | Поведение |
|---------|----------|
| **Личные сообщения** | Hermes отвечает на каждое сообщение. `@mention` не нужен. У каждого DM свой собственный сеанс. |
| **Каналы сервера** | По умолчанию Hermes отвечает только при `@mention`. Если отправить сообщение без упоминания, Hermes его игнорирует. |
| **Каналы свободного ответа** | Можно сделать отдельные каналы без упоминания через `DISCORD_FREE_RESPONSE_CHANNELS` или отключить обязательное упоминание глобально через `DISCORD_REQUIRE_MENTION=false`. В этих каналах ответы идут прямо в поток, без автосоздания треда, так что канал остаётся лёгким чатом. |
| **Треды** | Hermes отвечает в том же треде. Правила упоминания по-прежнему действуют, если только сам тред или его родительский канал не помечен как free-response. История сеанса в треде изолирована от родительского канала. |
| **Общие каналы с несколькими пользователями** | По умолчанию Hermes разделяет историю сеанса по пользователям внутри канала, чтобы было безопаснее и понятнее. Два человека в одном канале не делят один и тот же транскрипт, если вы явно не отключите это поведение. |
| **Сообщения с упоминанием других пользователей** | Когда `DISCORD_IGNORE_NO_MENTION` равен `true` (значение по умолчанию), Hermes молчит, если сообщение упоминает других пользователей, но **не** упоминает самого бота. Это защищает бота от вмешательства в разговоры, адресованные другим людям. Если хотите, чтобы бот отвечал на все сообщения независимо от того, кого ещё упомянули, поставьте `false`. Это правило действует только в каналах сервера, не в DM. |

:::tip
Если вам нужен обычный канал поддержки, где люди могут писать Hermes без постоянного `@mention`, добавьте этот канал в `DISCORD_FREE_RESPONSE_CHANNELS`.
:::

### Модель шлюза Discord

Hermes в Discord не работает как stateless webhook. Он проходит через полноценный messaging gateway, то есть каждое входящее сообщение идёт через:

1. авторизацию (`DISCORD_ALLOWED_USERS`)
2. проверки `@mention` / free-response
3. поиск сеанса
4. загрузку транскрипта сеанса
5. обычное выполнение агента Hermes, включая инструменты, память и слэш-команды
6. доставку ответа обратно в Discord

Это важно, потому что поведение в загруженном сервере зависит и от маршрутизации Discord, и от политики сеансов Hermes.

### Модель сеансов в Discord {#session-model-in-discord}

По умолчанию:

- каждый DM получает свой отдельный сеанс
- каждый серверный тред получает своё пространство сеанса
- каждый пользователь в общем канале получает свой собственный сеанс внутри этого канала

То есть если Алиса и Боб одновременно пишут Hermes в `#research`, Hermes по умолчанию считает это двумя отдельными разговорами, хотя визуально они находятся в одном канале Discord.

Это управляется через `config.yaml`:

```yaml
group_sessions_per_user: true
```

Поставьте `false` только если вам действительно нужен один общий разговор на весь чат:

```yaml
group_sessions_per_user: false
```

Общие сеансы могут быть полезны для совместной работы, но у них есть и цена:

- пользователи делят рост контекста и расход токенов
- длинная инструментальная задача одного человека может раздувать контекст для всех
- незавершённый запуск одного участника может мешать ответам другого

### Прерывания и параллельность

Hermes отслеживает запущенных агентов по ключу сеанса.

При значении по умолчанию `group_sessions_per_user: true`:

- если Алиса прерывает свой собственный незавершённый запрос, это влияет только на её сеанс в этом канале
- Боб может продолжать писать в том же канале, не наследуя историю Алисы и не прерывая её запуск

При `group_sessions_per_user: false`:

- весь чат делит один слот запущенного агента для этого канала или треда
- сообщения-уточнения от разных людей могут прерывать друг друга или ставиться в очередь друг за другом

Это руководство проведёт вас через полный процесс настройки — от создания бота в Discord Developer Portal до отправки первого сообщения.

## Шаг 1: Создайте приложение Discord

1. Откройте [Discord Developer Portal](https://discord.com/developers/applications) и войдите в свою учётную запись Discord.
2. Нажмите **New Application** в правом верхнем углу.
3. Укажите имя приложения, например `"Hermes Agent"`, и примите Developer Terms of Service.
4. Нажмите **Create**.

Откроется страница **General Information**. Запомните **Application ID** — он понадобится позже для формирования invite URL.

## Шаг 2: Создайте бота

1. В левом меню нажмите **Bot**.
2. Discord автоматически создаст bot user для вашего приложения. Вы увидите username бота, который можно изменить.
3. В разделе **Authorization Flow**:
   - установите **Public Bot** в **ON** — это требуется для использования invite-ссылки, которую выдаёт Discord (рекомендуется). Так вкладка Installation сможет сгенерировать стандартный authorization URL.
   - оставьте **Require OAuth2 Code Grant** в положении **OFF**.

:::tip
На этой странице можно задать боту собственный avatar и banner. Именно их увидят пользователи в Discord.
:::

:::info[Альтернатива для private bot]
Если вы хотите оставить бота приватным (`Public Bot = OFF`), вы **обязательно** должны использовать ручной метод URL из Шага 5, а не вкладку Installation. Ссылка, которую выдаёт Discord, требует включённого Public Bot.
:::

## Шаг 3: Включите Privileged Gateway Intents

Это самый важный шаг всей настройки. Без нужных intents бот подключится к Discord, но **не сможет читать содержимое сообщений**.

На странице **Bot** прокрутите до **Privileged Gateway Intents**. Там будет три переключателя:

| Intent | Назначение | Обязательно? |
|--------|------------|-------------|
| **Presence Intent** | Видеть статус пользователя online/offline | Необязательно |
| **Server Members Intent** | Получать список участников, сопоставлять имена пользователей | **Обязательно** |
| **Message Content Intent** | Читать текст сообщений | **Обязательно** |

**Включите и Server Members Intent, и Message Content Intent**, переведя их в положение **ON**.

- Без **Message Content Intent** бот будет получать события о сообщениях, но их текст окажется пустым — бот буквально не увидит, что вы написали.
- Без **Server Members Intent** бот не сможет сопоставлять имена пользователей со списком разрешённых и может не определить, кто именно написал сообщение.

:::warning[Это причина №1, почему Discord-боты не работают]
Если бот онлайн, но никогда не отвечает на сообщения, почти наверняка отключён **Message Content Intent**. Вернитесь в [Developer Portal](https://discord.com/developers/applications), выберите ваше приложение → Bot → Privileged Gateway Intents и убедитесь, что **Message Content Intent** включён. Затем нажмите **Save Changes**.
:::

**Про количество серверов:**
- Если бот добавлен **менее чем на 100 серверов**, intents можно просто включать и выключать.
- Если бот находится **на 100 и более серверах**, Discord требует пройти verification application, чтобы использовать privileged intents. Для личного использования это обычно неактуально.

Нажмите **Save Changes** внизу страницы.

## Шаг 4: Получите токен бота

Токен бота — это учётные данные, с помощью которых Hermes Agent входит в Discord как ваш bot. Находясь всё ещё на странице **Bot**:

1. В разделе **Token** нажмите **Reset Token**.
2. Если у вашей Discord-учётной записи включена двухфакторная аутентификация, введите код 2FA.
3. Discord покажет новый токен. **Сразу скопируйте его.**

:::warning[Токен показывается только один раз]
Токен отображается только один раз. Если вы его потеряете, придётся сбросить его и сгенерировать новый. Никогда не публикуйте токен и не коммитьте его в Git — любой, у кого есть этот токен, получает полный контроль над ботом.
:::

Храните токен в безопасном месте, например в менеджере паролей. Он понадобится на Шаге 8.

## Шаг 5: Сгенерируйте invite URL

Чтобы пригласить бота на сервер, нужен OAuth2 URL. Есть два способа его получить:

### Вариант A: Через вкладку Installation (рекомендуется)

:::note[Требуется Public Bot]
Для этого метода на Шаге 2 должен быть включён **Public Bot**. Если вы выключили Public Bot, используйте ручной URL ниже.
:::

1. В левом меню нажмите **Installation**.
2. В разделе **Installation Contexts** включите **Guild Install**.
3. В поле **Install Link** выберите **Discord Provided Link**.
4. В **Default Install Settings** для Guild Install:
   - **Scopes**: выберите `bot` и `applications.commands`
   - **Permissions**: выберите разрешения, перечисленные ниже

### Вариант B: Ручной URL

Ссылку приглашения можно собрать вручную в таком формате:

```text
https://discord.com/oauth2/authorize?client_id=YOUR_APP_ID&scope=bot+applications.commands&permissions=274878286912
```

Замените `YOUR_APP_ID` на Application ID из Шага 1.

### Обязательные разрешения

Минимальный набор разрешений для бота:

- **View Channels** — видеть доступные ему каналы
- **Send Messages** — отвечать на ваши сообщения
- **Embed Links** — оформлять rich-ответы
- **Attach Files** — отправлять изображения, аудио и другие файлы
- **Read Message History** — сохранять контекст разговора

### Рекомендуемые дополнительные разрешения

- **Send Messages in Threads** — отвечать в тредах
- **Add Reactions** — ставить реакции для подтверждения получения сообщения

### Числовые значения permissions

| Уровень | Число permissions | Что включено |
|-------|-------------------|-------------|
| Минимум | `117760` | View Channels, Send Messages, Read Message History, Attach Files |
| Рекомендуется | `274878286912` | Всё выше плюс Embed Links, Send Messages in Threads, Add Reactions |

## Шаг 6: Пригласите бота на свой сервер

1. Откройте invite URL в браузере (из вкладки Installation или из ручного URL).
2. В выпадающем списке **Add to Server** выберите свой сервер.
3. Нажмите **Continue**, затем **Authorize**.
4. Если появится CAPTCHA, пройдите её.

:::info
Чтобы приглашать ботов на сервер, вам нужна роль с разрешением **Manage Server**. Если ваш сервер не отображается в списке, попросите администратора сервера открыть invite-ссылку у себя.
:::

После авторизации бот появится в списке участников сервера и будет отображаться offline, пока вы не запустите Hermes gateway.

## Шаг 7: Найдите свой Discord User ID

Hermes Agent использует ваш Discord User ID, чтобы контролировать, кто может взаимодействовать с ботом. Чтобы его найти:

1. Откройте Discord, desktop или web.
2. Перейдите в **Settings** → **Advanced** → включите **Developer Mode**.
3. Закройте настройки.
4. Щёлкните правой кнопкой по своему username в сообщении, списке участников или профиле → **Copy User ID**.

Ваш User ID — это длинное число, например `284102345871466496`.

:::tip
Developer Mode также позволяет так же копировать **Channel IDs** и **Server IDs** — щёлкните правой кнопкой по имени канала или сервера и выберите Copy ID. Channel ID нужен, если вы хотите вручную задать домашний канал.
:::

## Шаг 8: Настройте Hermes Agent

### Вариант A: Интерактивная настройка (рекомендуется)

Запустите мастер настройки:

```bash
hermes gateway setup
```

Выберите **Discord**, когда появится запрос, а затем вставьте токен бота и ваш user ID.

### Вариант B: Ручная настройка

Добавьте следующее в файл `~/.hermes/.env`:

```bash
# Required
DISCORD_BOT_TOKEN=your-bot-token
DISCORD_ALLOWED_USERS=284102345871466496

# Multiple allowed users (comma-separated)
# DISCORD_ALLOWED_USERS=284102345871466496,198765432109876543
```

Затем запустите gateway:

```bash
hermes gateway
```

Через несколько секунд бот должен появиться онлайн в Discord. Отправьте ему сообщение — в DM или в канале, который ему доступен, — чтобы проверить настройку.

:::tip
`hermes gateway` можно запускать в фоне или как systemd-службу для постоянной работы. Подробности см. в документации по развёртыванию.
:::

## Справочник конфигурации

Поведение Discord управляется двумя файлами: **`~/.hermes/.env`** для учётных данных и env-переключателей, а **`~/.hermes/config.yaml`** для структурированных настроек. Если один и тот же параметр задан и там и там, переменная окружения имеет приоритет.

### Переменные окружения (`.env`)

| Переменная | Обязательно | Значение по умолчанию | Описание |
|----------|-------------|-----------------------|----------|
| `DISCORD_BOT_TOKEN` | **Да** | — | Токен бота из [Discord Developer Portal](https://discord.com/developers/applications). |
| `DISCORD_ALLOWED_USERS` | **Да** | — | Разделённый запятыми список Discord user ID, которым разрешено взаимодействовать с ботом. Без этого параметра **или** `DISCORD_ALLOWED_ROLES` шлюз отклоняет всех пользователей. |
| `DISCORD_ALLOWED_ROLES` | Нет | — | Разделённый запятыми список Discord role ID. Любой участник с одной из этих ролей авторизован. Работает по логике `OR` вместе с `DISCORD_ALLOWED_USERS`. Автоматически включает **Server Members Intent** при подключении. Полезно, когда состав модераторов часто меняется: новый модератор получает доступ сразу после выдачи роли, без обновления конфигурации. |
| `DISCORD_HOME_CHANNEL` | Нет | — | Channel ID, куда бот отправляет инициативные сообщения: результаты cron, напоминания, уведомления. |
| `DISCORD_HOME_CHANNEL_NAME` | Нет | `"Home"` | Отображаемое имя домашнего канала в логах и статусе. |
| `DISCORD_COMMAND_SYNC_POLICY` | Нет | `"safe"` | Управляет синхронизацией native slash-команд при запуске. `"safe"` сравнивает глобальные команды и обновляет только изменившиеся, пересоздавая команды, когда метаданные Discord нельзя применить патчем. `"bulk"` сохраняет старое поведение `tree.sync()`. `"off"` полностью пропускает синхронизацию при старте. |
| `DISCORD_REQUIRE_MENTION` | Нет | `true` | Если `true`, бот отвечает в каналах сервера только при `@mention`. Поставьте `false`, чтобы бот отвечал на все сообщения во всех каналах. |
| `DISCORD_THREAD_REQUIRE_MENTION` | Нет | `false` | Если `true`, отключает shortcut с неявным упоминанием внутри треда: треды требуют `@mention` так же, как и каналы, даже если бот уже участвовал в разговоре. Полезно, когда в одном треде работают несколько ботов и каждому нужно явно адресованное сообщение. |
| `DISCORD_FREE_RESPONSE_CHANNELS` | Нет | — | Разделённый запятыми список Channel ID, где бот отвечает без `@mention`, даже если `DISCORD_REQUIRE_MENTION` равен `true`. |
| `DISCORD_IGNORE_NO_MENTION` | Нет | `true` | Если `true`, бот молчит, когда сообщение упоминает других пользователей, но не упоминает самого бота. Это не даёт боту вмешиваться в разговоры, адресованные другим людям. Действует только в каналах сервера, не в DM. |
| `DISCORD_AUTO_THREAD` | Нет | `true` | Если `true`, любое `@mention` в обычном текстовом канале автоматически создаёт новый тред для разговора, чтобы основной канал оставался чистым. Сообщения внутри уже существующих тредов или DM не затрагиваются. |
| `DISCORD_ALLOW_BOTS` | Нет | `"none"` | Определяет, как бот обрабатывает сообщения от других Discord-ботов. `"none"` — игнорировать всех остальных ботов. `"mentions"` — принимать только сообщения ботов, которые `@mention` Hermes. `"all"` — принимать все сообщения ботов. |
| `DISCORD_REACTIONS` | Нет | `true` | Если `true`, бот ставит emoji-реакции на сообщения в ходе обработки (`👀` в начале, `✅` при успехе, `❌` при ошибке). Поставьте `false`, чтобы полностью отключить реакции. |
| `DISCORD_IGNORED_CHANNELS` | Нет | — | Разделённый запятыми список Channel ID, где бот **никогда** не отвечает, даже при `@mention`. Имеет приоритет над всеми остальными настройками каналов. |
| `DISCORD_ALLOWED_CHANNELS` | Нет | — | Разделённый запятыми список Channel ID. Если задан, бот отвечает **только** в этих каналах, плюс в DM, если это разрешено. Переопределяет `config.yaml` `discord.allowed_channels`. Можно сочетать с `DISCORD_IGNORED_CHANNELS` для явных правил allow/deny. |
| `DISCORD_NO_THREAD_CHANNELS` | Нет | — | Разделённый запятыми список Channel ID, где бот отвечает прямо в канале, а не создаёт тред. Имеет смысл только при `DISCORD_AUTO_THREAD=true`. |
| `DISCORD_HISTORY_BACKFILL` | Нет | `true` | Если `true`, бот при каждом `@mention` подтягивает недостающую историю канала. Когда `require_mention: true`, бот видит только сообщения, которые адресованы ему напрямую, а весь остальной канал в транскрипт не попадает. History backfill просматривает недавнюю историю канала от последнего ответа бота до текущего упоминания и добавляет её как контекст. |
| `DISCORD_HISTORY_BACKFILL_LIMIT` | Нет | `50` | Максимальное число сообщений, которое бот просматривает назад при восстановлении контекста канала. На практике просмотр обычно заканчивается раньше — на последнем сообщении самого бота в канале. |
| `DISCORD_REPLY_TO_MODE` | Нет | `"first"` | Управляет поведением reply-reference: `"off"` — никогда не отвечать ссылкой на исходное сообщение, `"first"` — добавлять reply-reference только к первому чанку ответа (по умолчанию), `"all"` — добавлять reply-reference к каждому чанку. |
| `DISCORD_ALLOW_MENTION_EVERYONE` | Нет | `false` | Если `false` (по умолчанию), бот не может пинговать `@everyone` или `@here`, даже если такие токены есть в ответе. Поставьте `true`, чтобы разрешить это. См. [Управление упоминаниями](#mention-control) ниже. |
| `DISCORD_ALLOW_MENTION_ROLES` | Нет | `false` | Если `false` (по умолчанию), бот не может пинговать `@role`. Поставьте `true`, чтобы разрешить. |
| `DISCORD_ALLOW_MENTION_USERS` | Нет | `true` | Если `true` (по умолчанию), бот может пинговать отдельных пользователей по ID. |
| `DISCORD_ALLOW_MENTION_REPLIED_USER` | Нет | `true` | Если `true` (по умолчанию), ответ на сообщение пингует его автора. |
| `DISCORD_PROXY` | Нет | — | Proxy URL для соединений Discord (HTTP, WebSocket, REST). Переопределяет `HTTPS_PROXY`/`ALL_PROXY`. Поддерживаются схемы `http://`, `https://` и `socks5://`. |
| `DISCORD_ALLOW_ANY_ATTACHMENT` | Нет | `false` | Если `true`, бот принимает вложения любого типа файлов, а не только встроенный список разрешённых типов для PDF/text/zip/office. Неизвестные типы кэшируются на диск и передаются агенту как локальный путь с MIME `application/octet-stream`, чтобы он мог осмотреть их через `terminal`, `read_file`, `ffprobe` и т. п. |
| `DISCORD_MAX_ATTACHMENT_BYTES` | Нет | `33554432` | Максимальный размер одного вложения, который gateway загрузит и закэширует. По умолчанию 32 MiB. Поставьте `0`, чтобы снять лимит (но вложения будут лежать в памяти во время записи, так что у бесконечного лимита есть реальная цена по RAM). |
| `HERMES_DISCORD_TEXT_BATCH_DELAY_SECONDS` | Нет | `0.6` | Время ожидания перед сбросом очередного текстового чанка. Полезно для сглаживания потокового вывода. |
| `HERMES_DISCORD_TEXT_BATCH_SPLIT_DELAY_SECONDS` | Нет | `2.0` | Задержка между частями, если одно сообщение превышает лимит длины Discord. |

### Файл конфигурации (`config.yaml`)

Раздел `discord` в `~/.hermes/config.yaml` повторяет env-переменные выше. Значения из config.yaml используются как defaults: если соответствующая переменная окружения уже задана, побеждает env.

```yaml
# Настройки, специфичные для Discord
discord:
  require_mention: true           # Требовать @mention в каналах сервера
  thread_require_mention: false   # Если true, требовать @mention и в тредах (multi-bot)
  free_response_channels: ""      # Разделённый запятыми список Channel ID (или YAML list)
  auto_thread: true               # Автоматически создавать тред при @mention
  reactions: true                 # Ставить emoji-реакции во время обработки
  ignored_channels: []            # Channel ID, где бот никогда не отвечает
  no_thread_channels: []          # Channel ID, где бот отвечает без треда
  history_backfill: true          # Добавлять недавнюю историю канала при упоминании
  history_backfill_limit: 50      # Максимум сообщений для просмотра назад
  channel_prompts: {}             # Эфемерные системные промпты для отдельных каналов
  allow_mentions:                 # Что бот может пинговать (безопасные defaults)
    everyone: false               # Пинги @everyone / @here (default: false)
    roles: false                  # Пинги @role (default: false)
    users: true                   # Пинги отдельных @user (default: true)
    replied_user: true            # reply-reference пингует автора (default: true)

# Изоляция сеансов (действует для всех платформ gateway, не только для Discord)
group_sessions_per_user: true     # Изолировать сеансы по пользователям в общих каналах
```

#### `discord.require_mention`

**Тип:** boolean — **По умолчанию:** `true`

Когда параметр включён, бот отвечает в каналах сервера только при прямом `@mention`. В DM бот отвечает всегда, независимо от этого параметра.

#### `discord.thread_require_mention` {#discordthread_require_mention}

**Тип:** boolean — **По умолчанию:** `false`

По умолчанию, как только бот поучаствовал в треде, созданном автоматически по `@mention`, или ответил там один раз, он продолжает отвечать на все последующие сообщения в этом треде без повторного `@mention`. Для разговоров один на один это удобное поведение по умолчанию.

В **multi-bot тредах**, где пользователи обращаются к разным ботам по очереди, это становится ловушкой: каждый бот начинает срабатывать на каждое сообщение, сжигать токены и спамить канал. Поставьте `thread_require_mention: true`, чтобы отключить этот shortcut внутри треда и сделать правила для тредов такими же, как для каналов. Явные `@mention` по-прежнему работают как раньше.

```yaml
discord:
  require_mention: true
  thread_require_mention: true    # multi-bot setup
```

#### `discord.free_response_channels`

**Тип:** string или list — **По умолчанию:** `""`

Channel ID, где бот отвечает на все сообщения без `@mention`. Поддерживается либо строка через запятую, либо YAML list:

```yaml
# Формат строки
discord:
  free_response_channels: "1234567890,9876543210"

# Формат списка
discord:
  free_response_channels:
    - 1234567890
    - 9876543210
```

Если родительский канал треда находится в этом списке, сам тред тоже становится free-response.

Каналы свободного ответа также **пропускают автосоздание треда** — бот отвечает прямо в канал, а не создаёт новый тред на каждое сообщение. Это делает канал удобным как лёгкую chat surface. Если вам нужен тредовый режим, не добавляйте канал в free-response: используйте обычный поток `@mention`.

#### `discord.auto_thread` {#discordauto_thread}

**Тип:** boolean — **По умолчанию:** `true`

Когда параметр включён, каждое `@mention` в обычном текстовом канале автоматически создаёт новый тред для разговора. Так основной канал остаётся чистым, а у каждого диалога есть собственная изолированная история сеанса. После создания треда последующие сообщения в нём уже не требуют `@mention` — бот понимает, что он уже участвует. Чтобы отключить этот in-thread shortcut для multi-bot сценариев, установите [`thread_require_mention`](#discordthread_require_mention) в `true`.

Сообщения в уже существующих тредах и DM этот параметр не затрагивает. Каналы из `discord.free_response_channels` и `discord.no_thread_channels` тоже обходят автосоздание тредов и получают ответы прямо в канал.

#### `discord.reactions`

**Тип:** boolean — **По умолчанию:** `true`

Определяет, будет ли бот добавлять emoji-реакции к сообщениям как визуальную обратную связь:
- 👀 — когда бот начинает обрабатывать сообщение
- ✅ — когда ответ успешно доставлен
- ❌ — если во время обработки произошла ошибка

Отключите это, если реакции мешают, или если роли бота не хватает разрешения **Add Reactions**.

#### `discord.ignored_channels`

**Тип:** string или list — **По умолчанию:** `[]`

Channel ID, где бот **никогда** не отвечает, даже при прямом `@mention`. Это правило имеет наивысший приоритет: если канал есть в этом списке, бот молча игнорирует все сообщения там, независимо от `require_mention`, `free_response_channels` и любых других настроек.

```yaml
# Формат строки
discord:
  ignored_channels: "1234567890,9876543210"

# Формат списка
discord:
  ignored_channels:
    - 1234567890
    - 9876543210
```

Если родительский канал треда находится в этом списке, сообщения в самом треде тоже игнорируются.

#### `discord.no_thread_channels`

**Тип:** string или list — **По умолчанию:** `[]`

Channel ID, где бот отвечает прямо в канале, а не создаёт тред автоматически. Это работает только когда `auto_thread` включён (по умолчанию). В таких каналах бот отвечает inline, как обычным сообщением, вместо создания нового треда.

```yaml
discord:
  no_thread_channels:
    - 1234567890  # Бот отвечает здесь прямо в канале
```

Полезно для каналов, посвящённых взаимодействию с ботом, где дополнительные треды только создают шум.

#### `discord.channel_prompts`

**Тип:** mapping — **По умолчанию:** `{}`

Эфемерные системные промпты для отдельных каналов. Они подставляются на каждом ходе в соответствующем Discord-канале или треде, но не сохраняются в историю сеанса.

```yaml
discord:
  channel_prompts:
    "1234567890": |
      This channel is for research tasks. Prefer deep comparisons,
      citations, and concise synthesis.
    "9876543210": |
      This forum is for therapy-style support. Be warm, grounded,
      and non-judgmental.
```

Поведение:
- Точные совпадения по thread/channel ID имеют приоритет.
- Если сообщение приходит в треде или forum post, а для этого треда нет явной записи, Hermes берёт родительский channel/forum ID.
- Промпты применяются эфемерно во время выполнения, поэтому их изменение сразу влияет на будущие ходы, не переписывая прошлую историю сеанса.

#### `discord.history_backfill`

**Тип:** boolean — **По умолчанию:** `true`

Когда параметр включён, бот восстанавливает пропущенные сообщения канала при каждом `@mention`. При `require_mention: true` бот обрабатывает только сообщения, которые напрямую его тегают, а всё остальное в канале остаётся вне транскрипта сеанса. History backfill просматривает недавнюю историю канала в момент триггера, собирает сообщения между последним ответом бота и текущим упоминанием и добавляет их как контекст.

Поведение зависит от поверхности:

- **Каналы сервера** (`require_mention: true`): backfill просматривает канал с момента последнего ответа бота. Это полезно, когда другие участники писали, пока бот не был упомянут.
- **Треды**: backfill просматривает только сам тред — `channel.history()` для треда возвращает только сообщения этого треда, а не родительского канала. Это правильный охват, потому что треды обычно самодостаточны.
- **DM**: пропускаются. Каждое DM-сообщение уже триггерит бота, так что история сеанса и так полная, и заполнять разрыв не нужно.
- **Каналы свободного ответа** и **автосозданные треды самого бота**: пропускаются по той же причине — без gating по `@mention` нет и разрыва.

Сеансы по пользователям (`group_sessions_per_user: true`, значение по умолчанию) тоже выигрывают: в сеансе пользователя не хватает контекста, который отправляли другие участники канала, а также его собственных сообщений до упоминания бота. Backfill закрывает оба разрыва.

```yaml
discord:
  history_backfill: true   # default
```

Чтобы отключить:

```yaml
discord:
  history_backfill: false
```

> **Примечание:** Сообщения, которые приходят *во время* обработки ботом (между триггером и ответом), не захватываются. Это осознанное упрощение: пользователь может отправить сообщение ещё раз или повторно упомянуть бота.

#### `discord.history_backfill_limit`

**Тип:** integer — **По умолчанию:** `50`

Максимальное число сообщений, которое бот просматривает назад при восстановлении контекста канала. На практике просмотр обычно заканчивается раньше — на последнем сообщении самого бота в канале, которое и является естественной границей между ходами. Этот лимит нужен как safety cap на холодный старт и длинные паузы, когда свежего сообщения бота ещё нет.

```yaml
discord:
  history_backfill: true
  history_backfill_limit: 50
```

#### `group_sessions_per_user`

**Тип:** boolean — **По умолчанию:** `true`

Это глобальная настройка gateway, не только для Discord. Она определяет, будут ли пользователи в одном канале получать изолированные истории сеансов.

Когда `true`: Алиса и Боб, пишущие в `#research`, получают по отдельному разговору с Hermes. Когда `false`: весь канал делит один транскрипт разговора и один слот запущенного агента.

```yaml
group_sessions_per_user: true
```

См. раздел [Модель сеансов в Discord](#session-model-in-discord) выше, чтобы увидеть все последствия каждого режима.

#### `display.tool_progress`

**Тип:** string — **По умолчанию:** `"all"` — **Значения:** `off`, `new`, `all`, `verbose`

Определяет, будет ли бот показывать в чате сообщения о ходе выполнения во время обработки, например `"Reading file..."` или `"Running terminal command..."`. Это глобальная настройка gateway, которая действует на все платформы.

```yaml
display:
  tool_progress: "all"    # off | new | all | verbose
```

- `off` — сообщений о ходе выполнения нет
- `new` — показывается только первый вызов инструмента в ходе одного хода
- `all` — показываются все вызовы инструментов (обрезаются до 40 символов в gateway-сообщениях)
- `verbose` — показываются полные детали вызова инструментов (может порождать длинные сообщения)

#### `display.tool_progress_command`

**Тип:** boolean — **По умолчанию:** `false`

Если включить, в gateway появится слэш-команда `/verbose`, с помощью которой можно переключать режимы tool progress (`off → new → all → verbose → off`) без правки config.yaml.

```yaml
display:
  tool_progress_command: true
```

## Управление доступом к слэш-командам {#slash-command-access-control}

По умолчанию любой разрешённый пользователь может запускать любые слэш-команды. Если хотите разделить список разрешённых пользователей на **администраторов** с полным доступом и **обычных пользователей**, которые могут вызывать только явно разрешённые команды, добавьте `allow_admin_from` и `user_allowed_commands` в блок `extra` платформы Discord:

```yaml
gateway:
  platforms:
    discord:
      extra:
        # Существующий список разрешённых пользователей (без изменений)
        allow_from:
          - "123456789012345678"  # admin user ID
          - "999888777666555444"  # regular user ID

        # НОВОЕ — администраторы получают все слэш-команды (встроенные + плагинные)
        allow_admin_from:
          - "123456789012345678"

        # НОВОЕ — обычные разрешённые пользователи могут запускать только эти слэш-команды.
        # /help и /whoami всегда разрешены, чтобы пользователь видел свои права.
        user_allowed_commands:
          - status
          - model
          - history

        # Необязательно: отдельные списки администраторов и команд для каналов сервера
        group_allow_admin_from:
          - "123456789012345678"
        group_user_allowed_commands:
          - status
```

**Поведение:**

- Пользователь в `allow_admin_from` для конкретной области (DM или канала сервера) может запускать **каждую** зарегистрированную слэш-команду — встроенную и зарегистрированную плагинами — через живой registry команд.
- Пользователь вне `allow_admin_from` может запускать только команды из `user_allowed_commands`, плюс неизменно разрешённые `/help` и `/whoami`.
- Обычный чат, не-slash сообщения, не затрагивается. Пользователи без admin-статуса по-прежнему могут говорить с агентом как обычно; просто они не смогут запускать произвольные команды.
- **Обратная совместимость:** если `allow_admin_from` не задан для области, gating слэш-команд для этой области отключён. Существующие установки продолжают работать без изменений.
- Admin-статус в DM не означает admin-статус в канале сервера. У каждой области свой список администраторов.

Используйте `/whoami`, чтобы увидеть активную область, ваш уровень (`admin` / `user` / `unrestricted`) и список слэш-команд, которые вам доступны.

## Интерактивный выбор модели

Отправьте `/model` без аргументов в Discord-канале, чтобы открыть dropdown-пикер моделей:

1. **Выбор провайдера** — Select dropdown со списком доступных провайдеров (до 25).
2. **Выбор модели** — второй dropdown с моделями для выбранного провайдера (до 25).

Пикер истекает через 120 секунд. Взаимодействовать с ним могут только авторизованные пользователи (те, кто есть в `DISCORD_ALLOWED_USERS`). Если знаете имя модели, можете сразу написать `/model <name>`.

## Нативные слэш-команды для skills

Hermes автоматически регистрирует установленные skills как **нативные команды приложений Discord**. Это значит, что skills появляются в автодополнении Discord `/` рядом со встроенными командами.

- Каждый skill превращается в слэш-команду Discord, например `/code-review` или `/ascii-art`
- Skills принимают необязательный строковый параметр `args`
- В Discord есть лимит в 100 команд приложений на одного бота — если skills больше, чем доступных слотов, лишние пропускаются с предупреждением в логах
- Skills регистрируются при старте бота вместе со встроенными командами вроде `/model`, `/reset` и `/background`

Дополнительной настройки не нужно — любой skill, установленный через `hermes skills install`, автоматически регистрируется как слэш-команда Discord после следующего перезапуска gateway.

### Отключение регистрации slash-команд

Если вы запускаете несколько Hermes gateway против одного и того же Discord application, например staging и production, только один из них должен владеть глобальной регистрацией slash-команд — иначе победит последний старт и регистрация будет мигать. На "follower" gateway отключите регистрацию:

```yaml
gateway:
  platforms:
    discord:
      extra:
        slash_commands: false   # default: true
```

Если оставить `true` на "primary" gateway, сохранится обычное поведение: глобальные команды в меню `/` для встроенных функций и установленных skills.

## Отправка медиа (`send_message` + теги `MEDIA:`)

Discord adapter поддерживает нативную загрузку файлов для всех распространённых медиа-типов через инструмент `send_message` и встроенные теги `MEDIA:/path/to/file`, которые может сгенерировать агент:

| Тип | Как доставляется |
|---|---|
| Изображения (PNG/JPG/WebP) | Нативное вложение Discord-изображения с предпросмотром |
| Анимированные GIF | `send_animation` загружает файл как `animation.gif`, чтобы Discord проигрывал его inline, а не как статичный thumbnail |
| Видео (MP4/MOV) | `send_video` — нативный видеоплеер |
| Аудио / голос | `send_voice` — нативное voice message, если возможно, иначе вложение-файл |
| Документы (PDF/ZIP/docx и т. п.) | `send_document` — нативное вложение с кнопкой download |

Лимит размера одной загрузки в Discord зависит от уровня boost на сервере (25 MB бесплатно, до 500 MB). Если Hermes получает HTTP 413, adapter не падает молча, а возвращает ссылку на локальный cache path.

## Приём любых типов файлов

По умолчанию бот кэширует только те загрузки, которые входят во встроенный список разрешённых типов: изображения, аудио, видео, PDF, text/markdown/csv/log, JSON/XML/YAML/TOML, zip, docx/xlsx/pptx. Всё остальное — например `.wav`, `.bin` или дамп с нестандартным расширением — логируется как `Unsupported document type` и отбрасывается до того, как агент его увидит.

Чтобы принимать файлы любых типов, включите `discord.allow_any_attachment`:

```yaml
discord:
  allow_any_attachment: true
  # Необязательно — увеличить или отключить лимит на размер одного файла.
  # По умолчанию 32 MiB. Весь файл хранится в памяти во время кэширования,
  # поэтому безлимитные загрузки реально расходуют RAM.
  max_attachment_bytes: 33554432   # bytes; 0 = unlimited
```

Когда флаг включён, любой загруженный файл скачивается, кэшируется в `~/.hermes/cache/documents/` и передаётся агенту как сообщение типа `DOCUMENT` с MIME `application/octet-stream`. Агент получает контекстную подсказку с локальным путём (автоматически преобразованным для Docker/Modal изолированных терминалов через `to_agent_visible_cache_path`) и может изучить файл через `terminal` (`ffprobe`, `unzip`, `file`, `strings` и т. п.) или `read_file`. Тело файла **не** вставляется целиком в промпт — передаётся только путь, так что бинарные загрузки не раздувают контекстное окно.

Известные текстовые форматы, уже попадающие в список разрешённых типов (`.txt`, `.md`, `.log`), по-прежнему автоматически подставляются в промпт до 100 KiB; это поведение не меняется при включении флага.

Эквивалентные env vars: `DISCORD_ALLOW_ANY_ATTACHMENT=true` и `DISCORD_MAX_ATTACHMENT_BYTES=33554432` (или `0` без лимита).

:::warning Memory cost of unlimited
Если снять ограничение на размер (`max_attachment_bytes: 0`), пользователь сможет прислать боту файл на несколько гигабайт, и gateway будет честно буферизовать его в памяти во время кэширования на диск. Включайте это только в доверенных одно-пользовательских установках. Для общих ботов лучше оставить 32 MiB по умолчанию или увеличить лимит очень осторожно.
:::

## Интерактивные уточнения (clarify)

Когда агент вызывает инструмент `clarify`, чтобы спросить, какой подход вы предпочитаете, получить feedback после задачи или уточнить неочевидное решение, Discord показывает вопрос с **одной кнопкой на каждый вариант**:

> Какой фреймворк мне выбрать для панели управления?
>
> [1. Next.js] [2. Remix] [3. Astro] [4. Другой вариант (введите ответ)]

Нажмите кнопку с номером, чтобы ответить, или нажмите **Другой вариант**, чтобы ввести свободный текст — следующее сообщение, которое вы отправите в этом канале, станет ответом. Вызовы `clarify` без готовых вариантов (open-ended) не показывают кнопки и просто считают вашим ответом следующее сообщение.

Кнопки отключаются после выбора, чтобы повторный клик не запускал один и тот же запрос дважды. Таймаут ответа настраивается через `agent.clarify_timeout` в `~/.hermes/config.yaml` (по умолчанию `600` секунд). Если вы не ответите вовремя, агент завершит запрос служебным сообщением и подстроится, а не зависнет.

## Домашний канал

Можно назначить домашний канал, куда бот будет отправлять инициативные сообщения, например результаты cron-задач, напоминания и уведомления. Есть два способа его задать:

### Через слэш-команду

Введите `/sethome` в любом Discord-канале, где присутствует бот. Этот канал станет домашним каналом.

### Ручная настройка

Добавьте в `~/.hermes/.env`:

```bash
DISCORD_HOME_CHANNEL=123456789012345678
DISCORD_HOME_CHANNEL_NAME="#bot-updates"
```

Замените ID на реальный Channel ID (щелчок правой кнопкой → Copy Channel ID при включённом Developer Mode).

## Голосовые сообщения

Hermes Agent поддерживает голосовые сообщения Discord:

- **Входящие голосовые сообщения** автоматически транскрибируются через настроенный STT provider: локальный `faster-whisper` (без ключа), Groq Whisper (`GROQ_API_KEY`) или OpenAI Whisper (`VOICE_TOOLS_OPENAI_KEY`).

- **Голосовые каналы Discord**: Hermes может подключаться к голосовому каналу, слушать речь пользователей и отвечать прямо в канал.

## Каналы-форумы

Discord forum channels (type 15) не принимают прямые сообщения — каждый post в форуме должен быть тредом. Hermes автоматически распознаёт forum channels и создаёт новый thread post всякий раз, когда ему нужно что-то туда отправить, так что `send_message`, TTS, изображения, голосовые сообщения и файловые вложения работают без специальной логики со стороны агента.

- **Имя треда** берётся из первой строки сообщения (markdown heading prefix убирается, длина ограничивается 100 символами). Если сообщение состоит только из вложения, вместо него используется имя файла.
- **Вложения** отправляются вместе со стартовым сообщением нового треда — без отдельного шага загрузки и без частичных отправок.
- **Один вызов — один тред**: каждый send в forum создаёт новый тред. Поэтому последующие отправки в тот же форум будут создавать отдельные треды.
- **Определение работает в три слоя**: сначала cache каталога каналов, затем локальный process cache, и только потом live `GET /channels/{id}` probe, результат которого memoize-ится на весь срок жизни процесса.

Обновление каталога (`/channels refresh` на платформах, где это доступно, или перезапуск gateway) заполнит cache всеми forum channels, созданными после запуска бота.

## Диагностика

### Бот online, но не отвечает на сообщения

**Причина**: отключён Message Content Intent.

**Решение**: откройте [Developer Portal](https://discord.com/developers/applications) → ваше приложение → Bot → Privileged Gateway Intents → включите **Message Content Intent** → Save Changes. Затем перезапустите gateway.

### Ошибка `"Disallowed Intents"` при запуске

**Причина**: код запрашивает intents, которые не включены в Developer Portal.

**Решение**: включите все три Privileged Gateway Intents (Presence, Server Members, Message Content) в настройках Bot и перезапустите.

### Бот не видит сообщения в конкретном канале

**Причина**: у роли бота нет права просматривать этот канал.

**Решение**: в Discord откройте настройки канала → Permissions → добавьте роль бота с включёнными **View Channel** и **Read Message History**.

### Ошибки `403 Forbidden`

**Причина**: у бота не хватает нужных разрешений.

**Решение**: пригласите бота заново с правильными permissions по ссылке из Шага 5 или вручную поправьте права роли бота в Server Settings → Roles.

### Бот offline

**Причина**: Hermes gateway не запущен или токен неверный.

**Решение**: проверьте, что `hermes gateway` действительно работает. Проверьте `DISCORD_BOT_TOKEN` в `.env`. Если вы недавно сбрасывали токен, обновите его.

### `"User not allowed"` / бот игнорирует вас

**Причина**: вашего User ID нет в `DISCORD_ALLOWED_USERS`.

**Решение**: добавьте свой User ID в `DISCORD_ALLOWED_USERS` в `~/.hermes/.env` и перезапустите gateway.

### Пользователи в одном канале неожиданно делят контекст

**Причина**: `group_sessions_per_user` отключён или платформа не может получить user ID для сообщений в этом контексте.

**Решение**: задайте это в `~/.hermes/config.yaml` и перезапустите gateway:

```yaml
group_sessions_per_user: true
```

Если вы специально хотите общий чат на всех, оставьте значение выключенным — просто помните, что тогда у всех будет общий transcript и общее поведение при прерываниях.

## Безопасность

:::warning
Всегда задавайте `DISCORD_ALLOWED_USERS` или `DISCORD_ALLOWED_ROLES`, чтобы ограничить круг тех, кто может взаимодействовать с ботом. Если не задано ни то, ни другое, gateway по умолчанию отклоняет всех пользователей как меру безопасности. Разрешайте доступ только тем, кому доверяете: авторизованные пользователи получают полный доступ к возможностям агента, включая использование инструментов и системный доступ.
:::

### Доступ по ролям

Для серверов, где доступ управляется ролями, а не списком отдельных пользователей, например для команд модераторов, support staff или внутреннего tooling, используйте `DISCORD_ALLOWED_ROLES` — список role ID через запятую. Любой участник с одной из этих ролей авторизован.

```bash
# ~/.hermes/.env — работает вместе с DISCORD_ALLOWED_USERS или вместо него
DISCORD_ALLOWED_ROLES=987654321098765432,876543210987654321
```

Смысл:

- **OR со списком разрешённых пользователей.** Пользователь авторизован, если его ID есть в `DISCORD_ALLOWED_USERS` **или** у него есть любая роль из `DISCORD_ALLOWED_ROLES`.
- **Server Members Intent включается автоматически.** Когда задан `DISCORD_ALLOWED_ROLES`, бот включает Members intent при подключении — он нужен, чтобы Discord передавал информацию о ролях вместе с записью участника.
- **Нужны role ID, а не имена.** Их можно взять в Discord: **User Settings → Advanced → Developer Mode ON**, затем щёлкните правой кнопкой любую роль → **Copy Role ID**.
- **Fallback для DM.** В DM проверка ролей идёт по общим guilds; пользователь с разрешённой ролью в любом общем сервере авторизуется и в DM.

Это предпочтительный вариант, когда состав модераторов часто меняется: новые модераторы получают доступ сразу после выдачи роли, без правки `.env` и без перезапуска gateway.

### Управление упоминаниями {#mention-control}

По умолчанию Hermes запрещает боту пинговать `@everyone`, `@here` и упоминания ролей, даже если такие токены есть в ответе. Это защищает сервер от случайного спама, если LLM сгенерирует неудачный текст или повторит пользовательский ввод. Индивидуальные `@user`-пинги и reply-reference пинги — маленький chip `"replying to..."` — остаются включёнными, чтобы обычный диалог работал как надо.

Ослабить эти defaults можно через env vars или `config.yaml`:

```yaml
# ~/.hermes/config.yaml
discord:
  allow_mentions:
    everyone: false      # разрешить боту пинговать @everyone / @here
    roles: false         # разрешить боту пинговать @role
    users: true          # разрешить боту пинговать отдельных @users
    replied_user: true   # reply-reference пингует автора
```

```bash
# ~/.hermes/.env — env vars win over config.yaml
DISCORD_ALLOW_MENTION_EVERYONE=false
DISCORD_ALLOW_MENTION_ROLES=false
DISCORD_ALLOW_MENTION_USERS=true
DISCORD_ALLOW_MENTION_REPLIED_USER=true
```

:::tip
Оставляйте `everyone` и `roles` в значении `false`, если не уверены, зачем вам их включать. LLM очень легко может вставить строку `@everyone` в обычный на вид ответ; без этой защиты она уведомит весь сервер.
:::

Подробнее о защите развёртывания Hermes Agent см. в [Security Guide](../security.md).
