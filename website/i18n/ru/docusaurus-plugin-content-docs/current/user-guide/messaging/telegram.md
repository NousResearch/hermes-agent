# Настройка Telegram {#telegram-setup}

Hermes Agent умеет работать с Telegram как полноценный чат-бот. После подключения вы сможете общаться с агентом с любого устройства, отправлять голосовые заметки с автоматической транскрибацией, получать результаты запланированных задач и использовать агента в групповых чатах. Интеграция построена на [python-telegram-bot](https://python-telegram-bot.org/) и поддерживает текст, голос, изображения и файловые вложения.

## Шаг 1. Создайте бота через BotFather {#step-1-create-a-bot-via-botfather}

Каждому Telegram-боту нужен API token, выданный [@BotFather](https://t.me/BotFather), официальным инструментом Telegram для управления ботами.

1. Откройте Telegram и найдите **@BotFather** или перейдите на [t.me/BotFather](https://t.me/BotFather)
2. Отправьте `/newbot`
3. Выберите **отображаемое имя** - например, `"Hermes Agent"` - оно может быть любым
4. Выберите **username** - он должен быть уникальным и оканчиваться на `bot` - например, `my_hermes_bot`
5. BotFather пришлёт ваш **API token**. Он выглядит так:

```
123456789:ABCdefGHIjklMNOpqrSTUvwxYZ
```

:::warning
Храните токен бота в секрете. Любой, у кого есть этот токен, сможет управлять вашим ботом. Если токен утечёт, немедленно отзовите его через `/revoke` в BotFather.
:::

## Шаг 2. Настройте бота по желанию {#step-2-customize-your-bot-optional}

Эти команды BotFather улучшают пользовательский опыт. Напишите @BotFather и используйте:

| Команда | Назначение |
|---------|------------|
| `/setdescription` | Текст `What can this bot do?` — описание, которое показывается до начала чата |
| `/setabouttext` | Короткий текст на странице профиля бота |
| `/setuserpic` | Загрузить аватар для бота |
| `/setcommands` | Задать меню команд (кнопка `/` в чате) |
| `/setprivacy` | Управлять тем, видит ли бот все сообщения в группах (см. шаг 3) |

:::tip
Для `/setcommands` хороший стартовый набор такой:

```
help - Показать справку
new - Начать новый разговор
sethome - Назначить этот чат домашним каналом
```
:::

## Шаг 3. Режим privacy (критично для групп) {#step-3-privacy-mode-critical-for-groups}

У Telegram-ботов есть **режим privacy**, который **включён по умолчанию**. Это самый частый источник путаницы при работе ботов в группах.

**Когда privacy mode включён**, бот видит только:
- Сообщения, которые начинаются с `/`-команды
- Ответы напрямую на собственные сообщения бота
- Служебные сообщения (вход/выход участников, закреплённые сообщения и т. п.)
- Сообщения в каналах, где бот является администратором

**Когда privacy mode выключён**, бот получает все сообщения в группе.

### Как отключить privacy mode

1. Напишите **@BotFather**
2. Отправьте `/mybots`
3. Выберите своего бота
4. Перейдите в **Bot Settings → Group Privacy → Turn off**

:::warning
**После смены настройки privacy нужно удалить бота из группы и добавить его снова.** Telegram кэширует состояние privacy в момент входа бота в группу, и оно не обновится, пока бота не удалят и не добавят заново.
:::

:::tip
Альтернатива отключению privacy mode - назначить бота **group admin**. Боты-администраторы всегда получают все сообщения независимо от настройки privacy, и тогда не нужно переключать глобальный режим privacy.
:::

## Шаг 4. Найдите свой User ID {#step-4-find-your-user-id}

Hermes Agent использует числовые Telegram user ID для контроля доступа. Ваш user ID - это **не** username, а число вроде `123456789`.

**Способ 1 (рекомендуется):** напишите [@userinfobot](https://t.me/userinfobot) - он сразу ответит вашим user ID.

**Способ 2:** напишите [@get_id_bot](https://t.me/get_id_bot) - ещё один надёжный вариант.

Сохраните этот номер, он понадобится на следующем шаге.

## Шаг 5. Настройте Hermes {#step-5-configure-hermes}

### Вариант A: интерактивная настройка (рекомендуется)

```bash
hermes gateway setup
```

Выберите **Telegram** при появлении запроса. Мастер попросит ваш bot token и разрешённые user ID, а затем сам запишет конфигурацию.

### Вариант B: ручная конфигурация

Добавьте следующее в `~/.hermes/.env`:

```bash
TELEGRAM_BOT_TOKEN=123456789:ABCdefGHIjklMNOpqrSTUvwxYZ
TELEGRAM_ALLOWED_USERS=123456789    # Для нескольких пользователей через запятую
```

### Запустите шлюз

```bash
hermes gateway
```

Бот должен появиться в сети через несколько секунд. Отправьте ему сообщение в Telegram, чтобы проверить.

## Отправка сгенерированных файлов из Docker-терминалов

Если ваш терминальный бэкенд — `docker`, помните, что вложения Telegram отправляет **процесс шлюза**, а не контейнер. Значит, итоговый путь `MEDIA:/...` должен быть доступен на хосте, где запущен шлюз.

Типичный сбой выглядит так:

- агент записывает файл внутри Docker в `/workspace/report.txt`
- модель выводит `MEDIA:/workspace/report.txt`
- доставка в Telegram не срабатывает, потому что `/workspace/report.txt` существует только внутри контейнера, а не на хосте

Рекомендуемый шаблон:

```yaml
terminal:
  backend: docker
  docker_volumes:
    - "/home/user/.hermes/cache/documents:/output"
```

Затем:

- записывайте файлы внутри Docker в `/output/...`
- указывайте в `MEDIA:` **видимый хосту** путь, например:
  `MEDIA:/home/user/.hermes/cache/documents/report.txt`

Если у вас уже есть секция `docker_volumes:`, добавьте новый mount в тот же список. Дублирующиеся ключи YAML тихо перезаписывают предыдущие значения.

### Поддерживаемые расширения файлов `MEDIA:`

Шлюз извлекает теги `MEDIA:/path/to/file` из ответов агента и отправляет указанный файл как нативное вложение платформы. Поддерживаются такие расширения:

| Категория | Расширения |
|---|---|
| Изображения | `png`, `jpg`, `jpeg`, `gif`, `webp`, `bmp`, `tiff`, `svg` |
| Аудио | `mp3`, `wav`, `ogg`, `m4a`, `opus`, `flac`, `aac` |
| Видео | `mp4`, `mov`, `webm`, `mkv`, `avi` |
| **Документы** | `pdf`, `txt`, `md`, `csv`, `json`, `xml`, `html`, `yaml`, `yml`, `log` |
| **Офисные файлы** | `docx`, `xlsx`, `pptx`, `odt`, `ods`, `odp` |
| **Архивы** | `zip`, `rar`, `7z`, `tar`, `gz`, `bz2` |
| **Книги / пакеты** | `epub`, `apk`, `ipa` |

Всё из этого списка доставляется как нативное вложение на платформах, где это поддерживается (Telegram, Discord, Signal, Slack, WhatsApp, Feishu, Matrix и т. п.); на платформах без нативной поддержки оно откатывается к ссылке или текстовой пометке. **Жирные** категории добавлены в последних релизах - если раньше вы полагались на формат вида `here is the file: /path/to/report.docx`, замените его на `MEDIA:/path/to/report.docx`, чтобы получить нативную доставку.

## Режим webhook {#webhook-mode}

По умолчанию Hermes подключается к Telegram в режиме **long polling** - шлюз отправляет исходящие запросы на серверы Telegram, чтобы забирать новые обновления. Это хорошо работает для локальных и постоянно включённых развёртываний.

Для **cloud deployments** (Fly.io, Railway, Render и т. п.) **режим webhook** обычно выгоднее. Такие платформы могут будить спящие машины по входящему HTTP-трафику, но не по исходящим соединениям. Поскольку polling - это исходящий трафик, polling-бот не сможет заснуть. Webhook меняет направление - Telegram присылает обновления на ваш HTTPS-URL, и это позволяет запускать бота только по необходимости.

| | Polling (по умолчанию) | Webhook |
|---|---|---|
| Направление | Шлюз → Telegram (исходящее) | Telegram → Шлюз (входящее) |
| Лучше всего подходит для | Локальных, всегда включённых серверов | Cloud-платформ с авто-пробуждением |
| Настройка | Без дополнительной конфигурации | Задайте `TELEGRAM_WEBHOOK_URL` |
| Цена простоя | Машина должна быть включена постоянно | Машина может спать между сообщениями |

### Конфигурация

Добавьте следующее в `~/.hermes/.env`:

```bash
TELEGRAM_WEBHOOK_URL=https://my-app.fly.dev/telegram
TELEGRAM_WEBHOOK_SECRET="$(openssl rand -hex 32)"  # required
# TELEGRAM_WEBHOOK_PORT=8443        # необязательно, по умолчанию 8443
```

| Переменная | Требуется | Описание |
|----------|----------|-------------|
| `TELEGRAM_WEBHOOK_URL` | Yes | Публичный HTTPS-URL, куда Telegram будет отправлять обновления. Путь URL извлекается автоматически, например `/telegram` из примера выше. |
| `TELEGRAM_WEBHOOK_SECRET` | **Yes** (when `TELEGRAM_WEBHOOK_URL` is set) | Секретный токен, который Telegram возвращает в каждом webhook-запросе для проверки. Шлюз не запустится без него - см. [GHSA-3vpc-7q5r-276h](https://github.com/NousResearch/hermes-agent/security/advisories/GHSA-3vpc-7q5r-276h). Сгенерируйте его через `openssl rand -hex 32`. |
| `TELEGRAM_WEBHOOK_PORT` | No | Локальный порт, на котором слушает webhook-сервер (по умолчанию `8443`). |

Когда задан `TELEGRAM_WEBHOOK_URL`, шлюз запускает HTTP webhook-сервер вместо polling. Если переменная не задана, используется polling-режим - поведение не меняется по сравнению с предыдущими версиями.

### Пример облачного развёртывания (Fly.io)

1. Добавьте переменные окружения в secrets вашего Fly.io-приложения:

```bash
fly secrets set TELEGRAM_WEBHOOK_URL=https://my-app.fly.dev/telegram
fly secrets set TELEGRAM_WEBHOOK_SECRET=$(openssl rand -hex 32)
```

2. Откройте webhook-порт в `fly.toml`:

```toml
[[services]]
  internal_port = 8443
  protocol = "tcp"

  [[services.ports]]
    handlers = ["tls", "http"]
    port = 443
```

3. Разверните приложение:

```bash
fly deploy
```

В логах шлюза должно появиться: `[telegram] Connected to Telegram (webhook mode)`.

## Поддержка прокси для Telegram {#telegram-proxy-support}

Если API Telegram заблокирован или вам нужно направить трафик через proxy, задайте отдельный Telegram proxy URL. Он имеет приоритет над общими переменными окружения `HTTPS_PROXY` / `HTTP_PROXY`.

**Вариант 1: config.yaml (рекомендуется)**

```yaml
telegram:
  proxy_url: "socks5://127.0.0.1:1080"
```

**Вариант 2: переменная окружения**

```bash
TELEGRAM_PROXY=socks5://127.0.0.1:1080
```

Поддерживаемые схемы: `http://`, `https://`, `socks5://`.

Прокси применяется и к основному соединению с Telegram, и к запасному IP-transport. Если Telegram-специфичный proxy не задан, шлюз переходит к `HTTPS_PROXY` / `HTTP_PROXY` / `ALL_PROXY` (или к автоопределению системного proxy в macOS).

## Домашний канал {#home-channel}

Используйте команду `/sethome` в любом Telegram-чате, личном или групповом, чтобы назначить его **домашним каналом**. Результаты запланированных задач (cron jobs) будут доставляться именно в этот канал.

Его также можно задать вручную в `~/.hermes/.env`:

```bash
TELEGRAM_HOME_CHANNEL=-1001234567890
TELEGRAM_HOME_CHANNEL_NAME="My Notes"
```

:::tip
ID группового чата - отрицательные числа, например `-1001234567890`. ID вашего личного DM-чата совпадает с вашим user ID.
:::

### Доставка cron в режиме topics

Если в вашем DM у бота включён topic mode, сообщения cron, доставленные в корневой чат, попадают в служебное лобби - ответить там нельзя, сеанс не открывается, и вы увидите уведомление `main chat is reserved for system commands`. Создайте отдельный forum topic, например `Cron`, и задайте:

```bash
TELEGRAM_CRON_THREAD_ID=<topic_thread_id>
```

`TELEGRAM_CRON_THREAD_ID` переопределяет `TELEGRAM_HOME_CHANNEL_THREAD_ID` только для доставок cron. Ответы в этом topic продолжают существующий сеанс этого topic.

## Голосовые сообщения

### Входящий голос (Speech-to-Text)

Голосовые сообщения, которые вы отправляете в Telegram, автоматически транскрибируются через настроенный STT-провайдер Hermes и подставляются в разговор как текст.

- `local` использует `faster-whisper` на машине, где запущен Hermes - API key не нужен
- `groq` использует Groq Whisper и требует `GROQ_API_KEY`
- `openai` использует OpenAI Whisper и требует `VOICE_TOOLS_OPENAI_KEY`

#### Пропустить STT: передать агенту исходный аудиофайл

Если вы хотите, чтобы **сам агент** работал с аудио - для диаризации, собственной транскрипции или просто архивирования записи - задайте `stt.enabled: false` в `~/.hermes/config.yaml`:

```yaml
stt:
  enabled: false
```

При отключённом STT шлюз всё равно скачивает голосовое или аудиовложение в аудиокэш Hermes, но **не транскрибирует** его. Агент получает сообщение с маркером вроде:

```
[The user sent a voice message: /home/<user>/.hermes/cache/audio/<hash>.ogg]
```

Дальше ваши инструменты или навыки могут прочитать этот путь напрямую - например, передать его в локальный pipeline диаризации, более мощную модель транскрибации или загрузить в долговременное хранилище. Расширение файла отражает исходный формат, который выдал Telegram (`.ogg` для voice notes, `.mp3`/`.m4a`/и т. д. для аудиовложений).

Это хорошо сочетается с разделом про [локальный сервер Bot API](#local-bot-api-server) ниже, который поднимает предел Telegram `getFile` с 20 MB до 2 GB - полезно, когда записи, которые вы хотите обработать, длиннее пары минут.

### Исходящий голос (Text-to-Speech)

Когда агент генерирует аудио через TTS, оно отправляется как нативные Telegram **голосовые сообщения** — круглые встроенные сообщения, которые можно воспроизводить прямо в чате.

- **OpenAI и ElevenLabs** нативно отдают Opus - дополнительная настройка не нужна
- **Edge TTS** (бесплатный провайдер по умолчанию) выдаёт MP3 и требует **ffmpeg** для конвертации в Opus:

```bash
# Ubuntu/Debian
sudo apt install ffmpeg

# macOS
brew install ffmpeg
```

Без `ffmpeg` аудио Edge TTS отправляется как обычный аудиофайл — его всё ещё можно воспроизвести, но вместо голосового сообщения будет прямоугольный плеер.

Настройте TTS-провайдер в `config.yaml` через ключ `tts.provider`.

## Большие файлы (>20 MB) через локальный Bot API server {#local-bot-api-server}

**Публичный** Bot API Telegram ограничивает загрузки `getFile` величиной **20 MB**, поэтому любая voice note, аудиофайл, видео или документ больше этого лимита будет молча отклонён Hermes с ответом `too large`. Документированный способ обойти это ограничение - поднять **локальный** демон [telegram-bot-api](https://github.com/tdlib/telegram-bot-api), то есть тот же серверный софт, который использует Telegram, но у вас в сети. Локальный сервер поднимает потолок файлов до **2 GB**, а Hermes автоматически снимает собственный внутренний лимит, когда видит настроенный `base_url`.

Это открывает такие сценарии:

- Отправка длинных голосовых заметок, например 45-минутных встреч или подкастов
- Загрузка больших видео для обработки vision-инструментами
- Архивирование сырого аудио для офлайн-пайплайнов, например диаризации, выравнивания или датасетов для обучения

### Шаг 1. Получите Telegram API credentials

Локальный сервер обращается напрямую к MTProto-слою Telegram, а не к публичному Bot API, поэтому ему нужны **MTProto credentials**:

1. Перейдите на [my.telegram.org/apps](https://my.telegram.org/apps) и войдите под своим аккаунтом Telegram.
2. Создайте новое приложение - подойдут любое имя и краткое описание.
3. Скопируйте `api_id` и `api_hash` - нужны оба значения.

### Шаг 2. Запустите сервер telegram-bot-api

Проще всего использовать поддерживаемый сообществом Docker-образ [`aiogram/telegram-bot-api`](https://hub.docker.com/r/aiogram/telegram-bot-api). Минимальный `docker-compose.yaml` (используйте режим `--local`, чтобы включить более высокий лимит):

```yaml
services:
  tg-bot-api:
    image: aiogram/telegram-bot-api:latest
    container_name: tg-bot-api
    restart: unless-stopped
    ports:
      - "127.0.0.1:8081:8081"   # bind to loopback only; see security note
    environment:
      TELEGRAM_API_ID: "12345"           # your api_id from Step 1
      TELEGRAM_API_HASH: "abcdef..."     # your api_hash from Step 1
      TELEGRAM_LOCAL: "1"                # enable --local mode (raises 20MB → 2GB)
    volumes:
      - ./tg-bot-api-data:/var/lib/telegram-bot-api
```

Поднимите его:

```bash
docker compose up -d tg-bot-api
docker logs --tail 20 tg-bot-api
```

:::warning Security
Локальный сервер Bot API принимает bot token прямо в пути URL, например `/bot<TOKEN>/getMe`, и при этом **не требует дополнительной аутентификации**. Любой, кто сможет достучаться до порта, получит полный контроль над вашим ботом - сможет читать все сообщения, которые бот видит, отправлять сообщения от его имени и т. д. Привяжите контейнер к `127.0.0.1` и/или заверните его в reverse proxy во внутренней сети. **Никогда не выставляйте порт 8081 в публичный интернет.**
:::

### Шаг 3. Выйдите из публичного API один раз

Бот может быть активен только на **одном** Bot API server одновременно. Если ваш бот уже работал через `api.telegram.org` (а скорее всего так и было), сначала нужно явно выполнить logout там, прежде чем локальный сервер начнёт принимать его:

```bash
curl "https://api.telegram.org/bot<YOUR_BOT_TOKEN>/logOut"
# expected response: {"ok":true,"result":true}
```

Это одноразовая миграция - повторять её при каждом рестарте не нужно. Telegram после `logOut` будет доставлять новые сообщения через новый сервер.

Проверьте, что локальный сервер может общаться с Telegram от имени бота:

```bash
curl "http://127.0.0.1:8081/bot<YOUR_BOT_TOKEN>/getMe"
# expected response: {"ok":true,"result":{"id":...,"is_bot":true,...}}
```

### Шаг 4. Укажите локальный сервер в Hermes

Добавьте URL под `platforms.telegram.extra` в `~/.hermes/config.yaml`:

```yaml
platforms:
  telegram:
    extra:
      base_url: "http://127.0.0.1:8081/bot"
      base_file_url: "http://127.0.0.1:8081/file/bot"
      local_mode: true        # see Step 5 below — only set this if the bot's data
                              # directory is readable by the Hermes process
```

:::caution Используйте `platforms.telegram.extra`, а не `telegram.extra`
Сейчас только форма `platforms.<name>.extra` deep-merge'ится в конфигурацию платформы. Ключи, помещённые напрямую в верхнеуровневый блок `telegram.extra`, молча отбрасываются.
:::

Когда задан `base_url`, Hermes:

- Строит клиент `python-telegram-bot` поверх локального сервера
- Автоматически поднимает внутренний лимит на размер документов и аудио с 20 MB до 2 GB
- Показывает активный лимит в сообщении `too large` (`Maximum: 2048 MB.`), чтобы сразу было видно, в каком режиме вы работаете

Перезапустите шлюз и найдите строку подтверждения в логах:

```bash
hermes gateway restart
grep -E "Using custom Telegram base_url|Using Telegram local_mode" ~/.hermes/logs/gateway.log | tail
```

### Шаг 5. `local_mode` - доступ к файлам на диске

У локального сервера есть **два способа** отдавать файлы:

1. **Без `--local`** (по умолчанию): файлы отдаются по HTTP через `/file/bot<TOKEN>/<path>`, как и в публичном Bot API. Предел 20 MB остаётся. Это полезно только как способ обойти сетевые проблемы, например когда `api.telegram.org` недоступен, но вы можете поднять свой сервер. Для снятия лимита это не подходит.
2. **С `--local`** (задаётся через `TELEGRAM_LOCAL=1` выше): файлы записываются в файловую систему сервера, а ответ `getFile` возвращает **абсолютный путь** вместо HTTP URL. Предел 20 MB снимается. Hermes тогда должен читать байты **с диска**, а не по HTTP.

Чтобы путь чтения с диска заработал, задайте `local_mode: true` в конфигурации выше **и** убедитесь, что процесс Hermes может прочитать путь, который вернёт сервер. Здесь есть два сценария:

- **Одна машина** - telegram-bot-api и Hermes работают на одном хосте. Примонтируйте volume данных в каталог, который Hermes может читать, например `/var/lib/telegram-bot-api`, и убедитесь, что владелец файла совпадает. Контейнер сбрасывает привилегии до внутреннего пользователя `telegram-bot-api` (uid зависит от образа); самый простой способ - добавить `user: "<UID>:<GID>"` в compose-сервис, чтобы файлы принадлежали uid, под которым Hermes уже работает.
- **Разные машины** - сервер бота запущен на одном хосте, например NAS или отдельной VM, а Hermes - на другом. Каталог данных сервера должен быть общим с машиной Hermes по **тому же абсолютному пути**, который возвращает сервер, обычно `/var/lib/telegram-bot-api`. Для этого хорошо подходит NFS; CIFS/SMB с remapping через `uid=` удобнее, если вы не хотите разбираться с несоответствием uid на уровне файловой системы.

Если `local_mode: true` включён, но Hermes не может `stat`-нуть возвращённый путь к файлу из-за прав доступа или неправильного mount, `python-telegram-bot` молча откатывается к HTTP `getFile` в локальный сервер - а в режиме `--local` тот отвечает `404 Not Found`. В логе `gateway.log` это выглядит так:

```
[Telegram] Failed to cache voice: Not Found
telegram.error.InvalidToken: Not Found
```

Если вы видите это, значит снятие лимита работает, но общий файловый доступ - нет. Проверьте `ls -la /var/lib/telegram-bot-api/<TOKEN>/voice/` с машины Hermes от имени пользователя, под которым запущен шлюз, и убедитесь, что один конкретный файл можно прочитать через `cat` без ошибки прав доступа.

### Шаг 6. Проверьте работу

Отправьте боту voice note или аудиофайл больше 20 MB. Следите за логом шлюза:

```bash
tail -f ~/.hermes/logs/gateway.log | grep -iE "telegram|cache"
```

Вы должны увидеть строку `[Telegram] Cached user voice at /home/<user>/.hermes/cache/audio/...` и **не** увидеть отказ `too large`. Если вместе с этим включён `stt.enabled: false` (см. выше), путь к исходному аудиофайлу попадёт во входное сообщение агента для дальнейшей обработки.

## Использование в групповых чатах {#group-chat-usage}

Hermes Agent работает в Telegram group chats с несколькими важными оговорками:

- **Privacy mode** определяет, какие сообщения бот вообще видит (см. [Шаг 3](#step-3-privacy-mode-critical-for-groups))
- `TELEGRAM_ALLOWED_USERS` по-прежнему действует — только авторизованные пользователи могут запускать бота, даже в группах
- Вы можете не давать боту отвечать на обычную групповую переписку с помощью `telegram.require_mention: true`
- При `telegram.require_mention: true` сообщения группы принимаются, если они:
  - являются ответом на одно из сообщений бота
  - содержат явное упоминание `@botusername`
  - используют форму `/command@botusername` (формат команды меню Telegram с включённым именем бота)
  - совпадают с одним из настроенных regex wake words в `telegram.mention_patterns`
- В группах с несколькими Hermes-ботами `telegram.exclusive_bot_mentions` делает маршрутизацию детерминированной. Когда сообщение явно упоминает одного или нескольких Telegram bot usernames, его обрабатывают только упомянутые профили бота; остальные Hermes-боты игнорируют сообщение ещё до обработки ответа и резервных срабатываний по ключевым словам. По умолчанию это включено.
- Используйте `telegram.ignored_threads`, чтобы Hermes молчал в определённых темах Telegram-форума, даже если в остальной группе ответы разрешены или срабатывают через упоминание
- Если `telegram.require_mention` не задан или равен `false`, Hermes сохраняет прежнее поведение открытой группы и отвечает на обычные групповые сообщения, которые он видит

### Несколько Hermes-ботов в одной группе

Если вы запускаете несколько профилей Hermes в одной Telegram-группе, создайте отдельный Telegram bot token для каждого профиля и запустите по одному gateway на профиль. Не используйте один и тот же bot token в нескольких работающих шлюзах одновременно - Telegram отклонит конкурентный polling для одного токена.

Рекомендуемая конфигурация для группы:

```yaml
telegram:
  require_mention: true
  exclusive_bot_mentions: true
  mention_patterns: []
```

С такой настройкой сообщение вроде `@research_bot @ops_bot summarize this` будет обработано только `research_bot` и `ops_bot`. Остальные Hermes-боты в группе промолчат, даже если сообщение является ответом на одно из их предыдущих сообщений или совпадает с общим wake word.

Ставьте `exclusive_bot_mentions: false` только для старых групп, где явные упоминания не должны перекрывать ответы и срабатывания по ключевым словам.

Чтобы управлять несколькими профилями, запускайте команду gateway по одному разу на профиль. Например:

```bash
# default profile
hermes gateway start
hermes gateway status
hermes gateway stop

# named profiles
hermes -p research gateway start
hermes -p research gateway status
hermes -p research gateway stop
```

Для небольшого фиксированного набора профилей используйте shell-loop или скрипт, который вызывает `hermes gateway <action>` для профиля по умолчанию и `hermes -p <profile> gateway <action>` для каждого именованного профиля. Это надёжнее, чем предполагать, что одна команда на уровне процесса управляет всеми именованными профилями через любой service manager.

### Диагностика: в личке работает, в группах нет

Если бот отвечает в личном чате, но молчит в группе, проверьте эти точки по порядку:

1. **Доставка Telegram:** выключите privacy mode в BotFather, сделайте бота администратором группы или упомяните бота напрямую. Hermes не может отвечать на сообщения группы, которые Telegram вообще не доставляет боту.
2. **Повторный вход после смены privacy:** удалите бота из группы и добавьте его снова после изменения настроек privacy в BotFather. Telegram может сохранять старое поведение доставки для уже существующих участий.
3. **Авторизация Hermes:** убедитесь, что отправитель есть в `TELEGRAM_ALLOWED_USERS` или `TELEGRAM_GROUP_ALLOWED_USERS`, либо разрешите сам групповой чат через `TELEGRAM_GROUP_ALLOWED_CHATS`.
4. **Фильтры упоминаний:** если задано `telegram.require_mention: true`, обычная переписка группы игнорируется, пока сообщение не является слэш-командой, ответом боту, упоминанием `@botusername` или совпадением с `mention_patterns`.
5. **Маршрутизация между ботами:** если в группе несколько ботов, убедитесь, что каждый профиль Hermes использует уникальный bot token, и оставьте `exclusive_bot_mentions` включённым, если вы специально не хотите старое поведение общих триггеров.

Отрицательные chat ID для Telegram-групп и supergroups — это нормально. Если вы используете авторизацию, привязанную к чату, кладите эти ID в `TELEGRAM_GROUP_ALLOWED_CHATS`, а не в список разрешений пользовательских ID отправителя.

### Пример конфигурации триггера для группы

Добавьте это в `~/.hermes/config.yaml`:

```yaml
telegram:
  require_mention: true
  exclusive_bot_mentions: true
  mention_patterns:
    - "^\\s*chompy\\b"
  ignored_threads:
    - 31
    - "42"
```

Этот пример разрешает все обычные прямые триггеры плюс сообщения, которые начинаются с `chompy`, даже если там нет явного упоминания.
Сообщения в Telegram topics `31` и `42` всегда игнорируются ещё до проверки упоминания и свободного ответа.

### Примечания к `mention_patterns`

- Шаблоны используют Python regular expressions
- Сопоставление не чувствительно к регистру
- Шаблоны проверяются и по текстовым сообщениям, и по подписям к медиа
- Невалидные regex-паттерны игнорируются с предупреждением в логах шлюза, а не валят бота
- Если шаблон должен срабатывать только в начале сообщения, заякорьте его с помощью `^`

## Личные чат-топики (Bot API 9.4) {#private-chat-topics-bot-api-94}

Telegram Bot API 9.4 (февраль 2026) представил **Private Chat Topics** - бот может создавать forum-style topic threads прямо в DM-чате один на один, без супергруппы. Это позволяет запускать несколько изолированных рабочих пространств внутри уже существующей лички с Hermes.

### Сценарий использования

Если вы ведёте несколько долгих проектов, topics помогают держать контекст отдельно:

- **Тема "Website"** - работа над продакшн web-сервисом
- **Тема "Research"** - обзор литературы и изучение статей
- **Тема "General"** - разовые задачи и короткие вопросы

У каждой темы свой разговорный сеанс, история и контекст - полностью изолированные от остальных.

### Конфигурация

:::caution Предварительные условия
Перед тем как добавлять topics в конфигурацию, пользователь должен **включить Topics mode** в DM-чате с ботом:

1. Откройте личный чат с Hermes bot в Telegram
2. Нажмите на имя бота вверху, чтобы открыть информацию о чате
3. Включите **Topics** (переключатель, который превращает чат в forum)

Без этого Hermes при старте запишет `The chat is not a forum` в лог и пропустит создание topics. Это настройка на стороне Telegram-клиента - бот не может включить её программно.
:::

Добавьте topics в `platforms.telegram.extra.dm_topics` в `~/.hermes/config.yaml`:

```yaml
platforms:
  telegram:
    extra:
      dm_topics:
      - chat_id: 123456789        # Your Telegram user ID
        topics:
        - name: General
          icon_color: 7322096
        - name: Website
          icon_color: 9367192
        - name: Research
          icon_color: 16766590
          skill: arxiv              # Автоматически загружать навык в этой теме
```

**Поля:**

| Поле | Обязательно | Описание |
|------|-------------|----------|
| `name` | Yes | Отображаемое имя темы |
| `icon_color` | No | Числовой код цвета иконки Telegram |
| `icon_custom_emoji_id` | No | ID custom emoji для иконки темы |
| `skill` | No | Навык, который нужно автоматически загрузить в новых сеансах этой темы |
| `thread_id` | No | Автоматически заполняется после создания темы - вручную задавать не нужно |

### Как это работает

1. При старте шлюза Hermes вызывает `createForumTopic` для каждой темы, у которой ещё нет `thread_id`
2. `thread_id` автоматически записывается обратно в `config.yaml` - последующие перезапуски уже не делают API-вызов
3. Каждая тема получает изолированный ключ сеанса: `agent:main:telegram:dm:{chat_id}:{thread_id}`
4. У каждой темы своя история разговоров, flush памяти и окно контекста

### Обработка корневого DM

По умолчанию сообщения, отправленные в корневой DM (вне какой-либо темы), обрабатываются как обычно. Установите `ignore_root_dm: true`, чтобы превратить корневой DM в лобби - обычные сообщения для пользователей с настроенными DM topics будут молча игнорироваться, а системные команды (`/start`, `/help`, `/status` и т. п.) продолжат работать.

```yaml
platforms:
  telegram:
    extra:
      ignore_root_dm: true
      dm_topics:
        - chat_id: 123456789
          topics:
            - name: General
```

Проверка действует **на уровне чата**: только у пользователей, у которых есть хотя бы одна запись в `dm_topics`, корневой DM будет переведён в этот режим. Пользователей без настроенных topics это не затрагивает.

### Привязка навыка

Темы с полем `skill` автоматически загружают этот навык, когда в теме стартует новый сеанс. Это работает ровно так же, как если бы вы написали `/skill-name` в начале разговора: содержимое навыка вставляется в первое сообщение, а последующие сообщения видят его уже в истории разговора.

Например, тема с `skill: arxiv` будет подгружать навык arxiv каждый раз, когда её сеанс сбрасывается, например по idle timeout, по ежедневному сбросу или через ручной `/reset`.

:::tip
Темы, созданные вне конфигурации, например через ручной вызов Telegram API, обнаруживаются автоматически, когда приходит служебное сообщение `forum_topic_created`. Вы также можете добавить темы в конфигурацию, пока шлюз уже работает - они подхватятся при следующем cache miss.
:::

## Многосессионный DM-режим (`/topic`) {#multi-session-dm-mode-topic}

Многосессионный DM в стиле ChatGPT - один бот, много параллельных разговоров. В отличие от управляемых оператором `extra.dm_topics` выше, этот режим **инициирует сам пользователь**: без конфигурации, без заранее объявленных названий тем. Конечный пользователь включает его через `/topic`, затем нажимает кнопку Telegram **+** и создаёт столько topics, сколько нужно, - каждая из них становится полностью независимым сеансом Hermes.

### Подкоманды `/topic`

| Форма | Контекст | Эффект |
|------|----------|--------|
| `/topic` | Root DM, ещё не включён | Проверить возможности BotFather, включить multi-session mode, создать закреплённую System topic |
| `/topic` | Root DM, уже включён | Показать статус: доступные для восстановления непривязанные сеансы |
| `/topic` | Внутри topic | Показать привязку текущей topic к сеансу |
| `/topic help` | Любой | Inline usage |
| `/topic off` | Root DM | Отключить multi-session mode и очистить все привязки topics для этого чата |
| `/topic <session-id>` | Внутри topic | Восстановить предыдущий Telegram-сеанс в текущую topic |

Запускать `/topic` могут только авторизованные пользователи (список разрешений через `TELEGRAM_ALLOWED_USERS` / platform auth config). Неавторизованный отправитель получит отказ вместо активации.

### DM Topics и многосеансовый DM-режим

| | `extra.dm_topics` (из конфигурации) | `/topic` (по инициативе пользователя) |
|---|---|---|
| Кто включает | Оператор в `config.yaml` | Конечный пользователь, отправив `/topic` |
| Список тем | Фиксированный набор из конфигурации | Пользователь свободно создаёт и удаляет темы |
| Имена тем | Выбирает оператор | Выбирает пользователь; автоматически переименовываются под заголовок сеанса Hermes |
| Поведение корневого DM | Обычный чат или лобби при `ignore_root_dm: true` | Становится system lobby, обычные сообщения отклоняются |
| Основной сценарий | Постоянные рабочие пространства с опциональной привязкой навыков | Параллельные разовые сеансы |
| Хранение | `extra.dm_topics` в config | Таблицы `telegram_dm_topic_mode` + `telegram_dm_topic_bindings` SQLite |

Оба механизма могут жить на одном боте: вы запускаете `/topic` из DM пользователя, а `extra.dm_topics` по-прежнему управляет темами, которые объявил оператор, для других чатов.

### Предварительные условия

В **@BotFather** откройте вашего бота → **Bot Settings → Threads Settings**:

1. Включите **Threaded Mode** (это активирует `has_topics_enabled`)
2. **Не** отключайте возможность пользователей создавать topics (`allows_users_to_create_topics` должна остаться включённой)

Когда пользователь впервые запускает `/topic`, Hermes вызывает `getMe`, чтобы проверить оба флага. Если хотя бы один выключен, Hermes отправит скриншот страницы BotFather Threads Settings и объяснит, что нужно переключить - активация не произойдёт, пока условия не выполнены.

### Поток активации

Из корневого DM отправьте:

```
/topic
```

Hermes:

1. Проверит `getMe().has_topics_enabled` и `allows_users_to_create_topics`
2. Если оба значения `true`, включит multi-session topic mode для этого DM
3. Создаст и закрепит topic **System** для статуса и команд (best effort)
4. Ответит списком предыдущих непривязанных Telegram-сеансов, которые пользователь может восстановить

После активации **корневой DM превращается в лобби**: обычные запросы отклоняются с подсказкой про кнопку **All Messages**. Системные команды (`/status`, `/sessions`, `/usage`, `/help` и т. д.) по-прежнему работают в корне.

### Создание новой темы (поток для конечного пользователя)

1. Откройте DM бота в Telegram
2. Нажмите **All Messages** вверху интерфейса бота, затем отправьте любое сообщение
3. Telegram создаст новую topic для этого сообщения
4. Hermes ответит внутри этой topic - теперь это самостоятельный сеанс

У каждой topic своя история разговоров, состояние модели, выполнение инструментов и session ID. Ключ изоляции - `agent:main:telegram:dm:{chat_id}:{thread_id}`, то же самое, что и у config-driven DM topics.

### Автоматическое переименование topics

Когда Hermes генерирует заголовок сеанса для topic через auto-title pipeline после первого обмена, сама Telegram topic переименовывается под него - например, `New Topic` станет `Database migration plan`. Переименование делается best effort: ошибки попадают в лог, но не ломают сеанс.

Чтобы отключить это и оставить ваши вручную выбранные имена topics нетронутыми, задайте:

```yaml
gateway:
  platforms:
    telegram:
      extra:
        disable_topic_auto_rename: true
```

Когда этот флаг включён, Hermes всё равно генерирует внутренний заголовок сеанса, который используется в `hermes sessions`, TUI и т. п., но никогда не редактирует имя Telegram topic. Это удобно, когда вы вручную организуете topics в BotFather Threaded Mode и не хотите, чтобы первый ответ каждый раз перезаписывал заголовок.

### `/new` внутри topic

Сбрасывает сеанс текущей topic (новый session ID, свежая история), не трогая другие topics. Hermes отвечает напоминанием, что для параллельной работы обычно нужно создать ещё одну topic через **All Messages**.

### Восстановление предыдущего сеанса

Внутри topic отправьте:

```
/topic <session-id>
```

Это привяжет текущую topic к уже существующему сеансу Hermes вместо создания нового. Полезно, если нужно продолжить разговор, начатый до включения topic mode. Ограничения:

- Целевой сеанс должен принадлежать тому же Telegram-пользователю
- Целевой сеанс не должен быть уже привязан к другой topic

Hermes подтверждает действие заголовком сеанса и повторяет последнее сообщение ассистента для контекста.

Чтобы узнать session ID, отправьте `/topic` без аргумента в root DM - Hermes покажет непривязанные Telegram-сеансы пользователя.

### `/topic` внутри topic без аргумента

Показывает текущую привязку topic: заголовок сеанса, session ID и подсказки для `/new` по сравнению с созданием ещё одной topic.

### Как это устроено внутри

- Активация сохраняется в `telegram_dm_topic_mode(chat_id, user_id, enabled, ...)` в `state.db`
- Каждая привязка topic хранится в `telegram_dm_topic_bindings(chat_id, thread_id, session_id, ...)` с `ON DELETE CASCADE` по `session_id` - при удалении сеанса привязка topic очищается автоматически
- Миграция SQLite для topic mode выполняется по запросу: она запускается при первом вызове `/topic`, а не при старте шлюза. Пока пользователь не выполнит `/topic` в этом профиле, `state.db` не меняется
- Каждое входящее DM-сообщение ищет привязку по `(chat_id, thread_id)`. Если она есть, сообщение маршрутизируется в привязанный сеанс через `SessionStore.switch_session()`, чтобы mapping от ключа сеанса к session_id оставался согласованным на диске
- `/new` внутри topic переписывает строку привязки на новый session ID, чтобы следующее сообщение осталось в свежем сеансе
- Topics, объявленные в `extra.dm_topics`, **никогда не переименовываются автоматически** - имя, заданное оператором, сохраняется даже при включённом multi-session mode
- `extra.disable_topic_auto_rename: true` выключает авто-переименование **всех** topics в чате, включая ad-hoc topics, созданные через Threaded Mode
- Тема General вверху DM с поддержкой форумов считается корневым лобби, независимо от того, доставляет ли Telegram её сообщения с `message_thread_id=1` или без `thread_id`
- Напоминания о корневом лобби ограничиваются одним сообщением каждые 30 секунд на чат - пользователь, который забыл, что topic mode включён, и пишет десять запросов в корень, не получит десять ответов
- Скриншоты настройки BotFather ограничиваются одной отправкой каждые 5 минут на чат - повторные попытки `/topic`, пока Threads Settings ещё выключены, не будут каждый раз загружать одну и ту же картинку
- `/background <prompt>`, запущенный внутри topic, возвращает результат в ту же topic; background sessions не вызывают авто-переименование родительской topic
- Сам `/topic` проходит через проверку пользовательской авторизации бота - неавторизованные DM получают отказ вместо активации

### Отключение multi-session mode

Отправьте `/topic off` в root DM. Hermes выключит соответствующую строку, очистит привязки `(thread_id → session_id)` для этого чата, и root DM вернётся к обычному чату Hermes. Существующие topics в Telegram не удаляются - они просто перестают обрабатываться как независимые сеансы. Позже `/topic` можно запустить снова.

Если нужно почистить вручную, например для массового сброса по множеству чатов, удалите строки напрямую:

```bash
sqlite3 ~/.hermes/state.db \
  "UPDATE telegram_dm_topic_mode SET enabled = 0 WHERE chat_id = '<your_chat_id>'; \
   DELETE FROM telegram_dm_topic_bindings WHERE chat_id = '<your_chat_id>';"
```

### Понижение версии Hermes

Если вы откатитесь на версию Hermes, в которой ещё нет `/topic`, функция просто перестанет работать - таблицы `telegram_dm_topic_mode` и `telegram_dm_topic_bindings` останутся в `state.db`, но старый код их игнорирует. DMs вернутся к нативной изоляции по тредам, то есть каждый `message_thread_id` по-прежнему получит свой сеанс через `build_session_key`, и ваши существующие Telegram topics продолжат работать как параллельные сеансы. Root DM перестанет быть лобби - сообщения там снова будут попадать в агента, как раньше. Повторное обновление активирует multi-session mode ровно там, где он был.

## Привязка навыка к topic в групповом форуме {#group-forum-topic-skill-binding}

Супергруппы с включённым **Topics mode** (их ещё называют forum topics) и так получают изоляцию сеанса по каждой topic - каждый `thread_id` маппится на свой разговор. Но иногда полезно **автоматически загружать навык**, когда сообщение приходит в конкретную topic группы, так же как это работает для привязки навыка к DM topic.

### Сценарий использования

Групповая супергруппа команды с forum topics для разных потоков работы:

- Topic **Engineering** - автоматически загружает навык `software-development`
- Topic **Research** - автоматически загружает навык `arxiv`
- Topic **General** - без навыка, универсальный ассистент

### Конфигурация

Добавьте привязки topic под `platforms.telegram.extra.group_topics` в `~/.hermes/config.yaml`:

```yaml
platforms:
  telegram:
    extra:
      group_topics:
      - chat_id: -1001234567890       # Supergroup ID
        topics:
        - name: Engineering
          thread_id: 5
          skill: software-development
        - name: Research
          thread_id: 12
          skill: arxiv
        - name: General
          thread_id: 1
          # Без навыка — общий сценарий
```

**Поля:**

| Поле | Обязательно | Описание |
|------|-------------|----------|
| `chat_id` | Yes | Числовой ID супергруппы (отрицательное число, начинающееся с `-100`) |
| `name` | No | Человекочитаемая метка topic, только для информации |
| `thread_id` | Yes | ID forum topic Telegram - виден в ссылках вида `t.me/c/<group_id>/<thread_id>` |
| `skill` | No | Навык, который нужно автоматически загружать в новых сеансах этой topic |

### Как это работает

1. Когда сообщение приходит в сопоставленную topic группы, Hermes ищет `chat_id` и `thread_id` в конфигурации `group_topics`
2. Если у найденной записи есть поле `skill`, этот навык автоматически загружается для сеанса - ровно как при привязке навыка к DM topic
3. Topics без ключа `skill` получают только изоляцию сеанса, без изменений в поведении
4. Несопоставленные значения `thread_id` или `chat_id` тихо проходят мимо - без ошибки и без загрузки навыка

### Отличия от DM Topics

| | DM Topics | Group Topics |
|---|---|---|
| Ключ конфигурации | `extra.dm_topics` | `extra.group_topics` |
| Создание topic | Hermes создаёт topics через API, если `thread_id` отсутствует | Администратор создаёт topics в интерфейсе Telegram |
| `thread_id` | Автоматически заполняется после создания | Нужно задавать вручную |
| `icon_color` / `icon_custom_emoji_id` | Поддерживаются | Неприменимо (внешний вид контролирует администратор) |
| Привязка навыка | ✓ | ✓ |
| Изоляция сеанса | ✓ | ✓ (у forum topics она уже встроена) |

:::tip
Чтобы найти `thread_id` topic, откройте её в Telegram Web или Desktop и посмотрите на URL: `https://t.me/c/1234567890/5` - последнее число (`5`) и есть `thread_id`. `chat_id` для супергрупп - это ID группы с префиксом `-100` (например, группа `1234567890` превращается в `-1001234567890`).
:::

## Последние возможности Bot API

- **Bot API 9.4 (февраль 2026):** Private Chat Topics - боты могут создавать forum topics в DM-чате один на один через `createForumTopic`. Hermes использует это для двух разных функций: управляемых оператором [Private Chat Topics](#private-chat-topics-bot-api-94) (конфигурация задаёт фиксированный список topics) и управляемого пользователем [Многосессионный DM-режим](#multi-session-dm-mode-topic) (активируется через `/topic`, безлимитные темы, создаваемые пользователем).
- **Политика приватности:** Telegram теперь требует, чтобы у ботов была privacy policy. Задайте её через BotFather с помощью `/setprivacy_policy`, иначе Telegram может сгенерировать заглушку автоматически. Это особенно важно, если бот доступен публично.
- **Bot API 9.5 (март 2026): нативная потоковая передача через `sendMessageDraft`.** Hermes поддерживает нативную потоковую передачу черновиков в Telegram как подключаемый режим для личных чатов. По умолчанию остаётся старый путь `editMessageText`, потому что предпросмотр черновиков в некоторых клиентах Telegram может заметно схлопываться и перерисовываться.

### Streaming transport (`gateway.streaming.transport`)

Когда streaming включён (`gateway.streaming.enabled: true`), Hermes выбирает один из четырёх режимов передачи:

| Значение | Поведение |
|---|---|
| `auto` | Нативная потоковая передача черновиков в поддерживаемых чатах (сейчас это Telegram DMs); в остальных случаях - старый режим редактирования. Если шаг черновика завершается ошибкой, происходит корректный переход на резервный режим. |
| `draft` | Принудительно использовать нативные черновики. Если чат не поддерживает черновики, например группы или topics, логируется понижение режима и используется редактирование сообщения. |
| `edit` (по умолчанию) | Классический пошаговый `editMessageText`-опрос для всех типов чатов. |
| `off` | Полностью отключить streaming - только финальный ответ, без промежуточных обновлений. |

В `~/.hermes/config.yaml`:

```yaml
gateway:
  streaming:
    enabled: true
    transport: edit    # edit | auto | draft | off
```

**Что вы увидите в DM при `edit` (по умолчанию)** - шлюз отправляет обычное текстовое сообщение и плавно обновляет его через `editMessageText`, избегая эффекта схлопывания и сброса предпросмотра черновика Telegram.

**Что вы увидите в DM при `auto` или `draft`** - Telegram показывает анимированный предпросмотр черновика, который обновляется по символу. Когда ответ заканчивается, он доставляется как обычное сообщение, а предпросмотр черновика естественным образом исчезает у клиента. Черновики не имеют message id, поэтому в истории чата остаётся именно финальный ответ.

**А как быть с группами, супергруппами и forum topics?** Telegram ограничивает `sendMessageDraft` только личными чатами (DM). Для всего остального шлюз прозрачно переходит на режим редактирования сообщения - UX остаётся тем же, что и раньше.

**Что если кадр черновика падает?** Любая ошибка - временная сетевая, отказ сервера, старый `python-telegram-bot` - переводит этот ответ на режим редактирования сообщения до конца потока. Следующий ответ получит свежую попытку.

## Рендеринг: таблицы и превью ссылок

У Telegram MarkdownV2 нет нативного синтаксиса таблиц - если передать pipe-таблицу как есть, она превратится в шум из экранированных обратных слэшей. Hermes автоматически нормализует markdown-таблицы:

- **Небольшие таблицы** сворачиваются в **списки с группировкой по строкам** - каждая строка превращается в читаемый bullet list под заголовками столбцов. Хорошо подходит для 2-4 колонок и коротких ячеек.
- **Более крупные или широкие таблицы** откатываются к **fenced code block** с выровненными колонками, чтобы ничего не разваливалось. В ответ добавляется однострочная подсказка, чтобы агент предпочитал дальнейшие пояснения в прозе, а не новые таблицы в Telegram.

Ничего настраивать не нужно - адаптер сам выбирает подходящий резервный режим для каждого сообщения. Если вам нужно старое поведение `always code-block`, отключите нормализацию таблиц через `telegram.pretty_tables: false` в `config.yaml` (по умолчанию `true`).

**Предпросмотр ссылок.** Telegram автоматически создаёт предпросмотр ссылок для URL в сообщениях бота. Если вы хотите его подавить, например для длинного вывода `/tools` или ответа агента с десятком ссылок:

```yaml
gateway:
  platforms:
    telegram:
      extra:
        disable_link_previews: true
```

Когда опция включена, Hermes добавляет `LinkPreviewOptions(is_disabled=True)` к каждому исходящему сообщению Telegram и откатывается к legacy-параметру `disable_web_page_preview` на старых версиях `python-telegram-bot`.

## Список разрешений для групп {#group-allowlisting}

У Telegram-групп и forum-чатов есть два независимых шлюза доступа, которые можно настроить:

- **Sender user IDs** (`group_allow_from` / `TELEGRAM_GROUP_ALLOWED_USERS`) — список разрешений на уровне отправителя, который применяется только к групповым или forum-сообщениям. Используйте его, когда хотите разрешить конкретным пользователям вызывать бота в группах, не добавляя их в `TELEGRAM_ALLOWED_USERS`, который дал бы им ещё и доступ в DM.
- **Chat IDs** (`group_allowed_chats` / `TELEGRAM_GROUP_ALLOWED_CHATS`) — список разрешений на уровне чата. Любой участник этих групп или forums может взаимодействовать с ботом. Это удобно для командных ботов или ботов поддержки, где сам факт членства в группе и есть сигнал доступа.

```yaml
gateway:
  platforms:
    telegram:
      extra:
        # Глобальный доступ (DM + группы). Пользователи здесь всегда могут вызывать бота.
        allow_from:
          - "123456789"
        # Sender ID, разрешённые только в группах/форумах. Доступ к DM НЕ дают.
        group_allow_from:
          - "987654321"
        # Целые группы/форумы — авторизован любой участник.
        group_allowed_chats:
          - "-1001234567890"
```

Эквивалентные env vars:

```bash
TELEGRAM_ALLOWED_USERS="123456789"
TELEGRAM_GROUP_ALLOWED_USERS="987654321"
TELEGRAM_GROUP_ALLOWED_CHATS="-1001234567890"
```

Поведение:

- `TELEGRAM_ALLOWED_USERS` покрывает все типы чатов - DM, группы, forums.
- `TELEGRAM_GROUP_ALLOWED_USERS` авторизует только перечисленных отправителей в группах и форумах. В DM они всё равно не смогут писать боту, если не добавлены в `TELEGRAM_ALLOWED_USERS`.
- Чат из `TELEGRAM_GROUP_ALLOWED_CHATS` авторизует каждого участника этого чата, независимо от отправителя.
- Используйте `*` в любом из этих списков, чтобы разрешить любого отправителя или любой чат.
- Это работает поверх уже существующих триггеров по упоминаниям и шаблонам, а также поверх `group_topics` и `ignored_threads`.

### Миграция до PR #17686

До этого разделения `TELEGRAM_GROUP_ALLOWED_USERS` был единственным переключателем, и пользователи записывали туда **chat ID**. Для обратной совместимости значения, похожие на chat ID и начинающиеся с `-`, в `TELEGRAM_GROUP_ALLOWED_USERS` по-прежнему воспринимаются как chat ID, а предупреждение о deprecated-формате логируется один раз. Миграция:

```bash
# Старый вариант (всё ещё работает, но устарел)
TELEGRAM_GROUP_ALLOWED_USERS="-1001234567890"

# Новый вариант
TELEGRAM_GROUP_ALLOWED_CHATS="-1001234567890"
```

### Обход упоминания для гостей (`guest_mode`)

В типичной конфигурации `group_allowed_chats` — это жёсткий шлюз: сообщения из групп вне списка молча отбрасываются, даже если участник явно упоминает бота. Для ботов поддержки или командных ботов это правильное поведение по умолчанию.

Для более неформальных сценариев - например, дружеских чатов, где бот должен быть **в основном молчаливым**, но **иногда доступным по явному пингу** - включите `guest_mode`:

```yaml
gateway:
  platforms:
    telegram:
      extra:
        group_allowed_chats:
          - "-1001234567890"   # ваша основная разрешённая группа
        guest_mode: true       # группы вне списка разрешений: разрешать только по упоминанию
```

Эквивалент в env:

```bash
TELEGRAM_GUEST_MODE=true
```

По умолчанию: `false`.

При `guest_mode: true` сообщение из неразрешённой группы обрабатывается **только если** оно явно упоминает бота. Упоминание требуется на каждом ходе — никакого устойчивого сеанса для гостевых взаимодействий нет, поэтому бот никогда не подхватывает дружеский group thread сам по себе.

DM и разрешённые группы продолжают работать как раньше.

## Контроль доступа к слэш-командам {#slash-command-access-control}

По умолчанию любой разрешённый пользователь может запускать любую слэш-команду. Чтобы разделить список разрешений на **администраторов** с полным доступом к слэш-командам и **обычных пользователей**, у которых есть только явно разрешённые команды, добавьте `allow_admin_from` и `user_allowed_commands` в блок `extra` платформы:

```yaml
gateway:
  platforms:
    telegram:
      extra:
        # Существующие списки разрешений (без изменений)
        allow_from:
          - "123456789"     # admin
          - "555555555"     # regular user
          - "777777777"     # regular user

        # NEW — администраторы получают все slash-команды (встроенные + плагиновые)
        allow_admin_from:
          - "123456789"

        # NEW — пользователи без прав администратора могут запускать только эти слэш-команды.
        # /help и /whoami всегда разрешены, чтобы пользователь мог увидеть свой доступ.
        user_allowed_commands:
          - status
          - model
          - history

        # Optional: separate admin/command lists for groups
        group_allow_admin_from:
          - "123456789"
        group_user_allowed_commands:
          - status
```

**Поведение:**

- Пользователь, перечисленный в `allow_admin_from` для конкретного scope (DM или group), может запускать **все** зарегистрированные слэш-команды - встроенные и плагинные - через живой registry
- Пользователь в `allow_from`, но **не** в `allow_admin_from`, может запускать только команды из `user_allowed_commands`, плюс всегда разрешённый минимум: `/help` и `/whoami`
- Обычный чат, не являющийся slash-командой, не затрагивается. Не-админы по-прежнему могут разговаривать с агентом обычно, они просто не могут запускать произвольные команды
- **Backward compat:** если `allow_admin_from` не задан для scope, gating слэш-команд для этого scope отключён. Существующие установки продолжают работать без изменений
- Админ-статус в DM не означает админ-статус в группе. У каждого scope свой список админов
- Если задан только `group_allow_admin_from`, DM scope остаётся в режиме без ограничений, как в старой схеме

Используйте `/whoami`, чтобы увидеть активный scope, свой уровень (admin / user / unrestricted) и доступные вам слэш-команды.

## Интерактивный выбор модели

Когда вы отправляете `/model` без аргументов в Telegram-чате, Hermes показывает интерактивную inline keyboard для переключения моделей:

1. **Выбор провайдера** - кнопки с каждым доступным провайдером и количеством моделей, например `OpenAI (15)` или `✓ Anthropic (12)` для текущего провайдера
2. **Выбор модели** - пагинированный список моделей с навигацией **Prev**/**Next**, кнопкой **Back** для возврата к провайдерам и кнопкой **Cancel**

Текущие модель и провайдер отображаются вверху. Вся навигация происходит через редактирование одного и того же сообщения на месте, без захламления чата.

:::tip
Если вы знаете точное имя модели, просто введите `/model <name>` и пропустите picker. Можно также написать `/model <name> --global`, чтобы сохранить изменение между сеансами.
:::

## Fallback IP через DNS-over-HTTPS

В некоторых ограниченных сетях `api.telegram.org` может резолвиться в недоступный IP. Telegram-адаптер включает механизм **резервных IP**, который прозрачно повторяет попытку подключения по альтернативным IP, сохраняя правильный TLS hostname и SNI.

### Как это работает

1. Если задан `TELEGRAM_FALLBACK_IPS`, используются именно они
2. Иначе адаптер автоматически опрашивает **Google DNS** и **Cloudflare DNS** через DNS-over-HTTPS (DoH), чтобы найти альтернативные IP для `api.telegram.org`
3. IP, возвращённые DoH и отличающиеся от результата системного DNS, используются как резервный вариант
4. Если DoH тоже заблокирован, как последний резерв используется зашитый seed IP `149.154.167.220`
5. Как только один из резервных IP сработает, он становится закреплённым - последующие запросы сразу идут на него, без повторной попытки сначала идти через основной путь

### Конфигурация

```bash
# Явные резервные IP (через запятую)
TELEGRAM_FALLBACK_IPS=149.154.167.220,149.154.167.221
```

Или в `~/.hermes/config.yaml`:

```yaml
platforms:
  telegram:
    extra:
      fallback_ips:
        - "149.154.167.220"
```

:::tip
Обычно вручную это настраивать не нужно. Автообнаружение через DoH закрывает большинство сценариев с ограниченной сетью. Переменная `TELEGRAM_FALLBACK_IPS` нужна только если у вас в сети заблокирован и DoH.
:::

## Поддержка прокси

Если вашей сети нужен HTTP proxy для выхода в интернет, что часто бывает в корпоративной среде, Telegram-адаптер автоматически читает стандартные переменные окружения proxy и направляет все соединения через него.

### Поддерживаемые переменные

Адаптер проверяет эти переменные окружения по порядку и берёт первую заданную:

1. `HTTPS_PROXY`
2. `HTTP_PROXY`
3. `ALL_PROXY`
4. `https_proxy` / `http_proxy` / `all_proxy` (нижний регистр)

### Конфигурация

Задайте proxy в окружении перед запуском шлюза:

```bash
export HTTPS_PROXY=http://proxy.example.com:8080
hermes gateway
```

Или добавьте в `~/.hermes/.env`:

```bash
HTTPS_PROXY=http://proxy.example.com:8080
```

Proxy применяется и к основному режиму передачи, и ко всем резервным IP-режимам. Дополнительная конфигурация Hermes не нужна - если переменная окружения задана, она используется автоматически.

:::note
Это покрывает пользовательский резервный слой передачи, который Hermes использует для Telegram-соединений. Обычный клиент `httpx`, который используется в других местах, уже нативно уважает переменные окружения прокси.
:::

## Реакции на сообщения

Бот может добавлять emoji reactions к сообщениям как визуальную обратную связь о процессе обработки:

- 👀 когда бот начинает обрабатывать ваше сообщение
- ✅ когда ответ успешно доставлен
- ❌ если во время обработки произошла ошибка

Reactions **отключены по умолчанию**. Включите их в `config.yaml`:

```yaml
telegram:
  reactions: true
```

Или через переменную окружения:

```bash
TELEGRAM_REACTIONS=true
```

:::note
В отличие от Discord, где reactions добавляются поверх друг друга, Telegram Bot API заменяет все реакции бота одним вызовом. Переход с 👀 на ✅/❌ происходит атомарно - вы не увидите их одновременно.
:::

:::tip
Если у бота нет права добавлять реакции в группе, вызовы реакции молча провалятся, а обработка сообщения продолжится как обычно.
:::

## Пер-канальные промпты

Назначайте временные системные промпты конкретным Telegram-группам или темам форумов Telegram. Промпт подставляется во время выполнения на каждом ходе и **никогда** не сохраняется в историю транскрипта, поэтому изменения вступают в силу сразу.

```yaml
telegram:
  channel_prompts:
    "-1001234567890": |
      You are a research assistant. Focus on academic sources,
      citations, and concise synthesis.
    "42":  |
      This topic is for creative writing feedback. Be warm and
      constructive.
```

Ключи - это chat ID (группы и супергруппы) или ID forum topic. Для forum-групп topic-level промпты перекрывают group-level промпт:

- Сообщение в topic `42` внутри группы `-1001234567890` → используется промпт topic `42`
- Сообщение в topic `99` без явной записи → fallback на промпт группы `-1001234567890`
- Сообщение в группе без записи → channel prompt не применяется

Числовые YAML-ключи автоматически нормализуются в строки.

## Диагностика

| Проблема | Решение |
|---------|---------|
| Бот вообще не отвечает | Проверьте, что `TELEGRAM_BOT_TOKEN` указан верно. Посмотрите логи `hermes gateway` на наличие ошибок. |
| Бот отвечает `unauthorized` | Ваш user ID не входит в `TELEGRAM_ALLOWED_USERS`. Сверьте его через @userinfobot. |
| Бот игнорирует сообщения группы | Скорее всего, включён privacy mode. Отключите его (шаг 3) или сделайте бота group admin. **Не забудьте удалить и снова добавить бота после смены privacy.** |
| Голосовые сообщения не транскрибируются | Проверьте, доступен ли STT: установите `faster-whisper` для локальной транскрипции или задайте `GROQ_API_KEY` / `VOICE_TOOLS_OPENAI_KEY` в `~/.hermes/.env`. |
| Голосовые ответы приходят файлами, а не bubbles | Установите `ffmpeg` (нужно для конвертации Edge TTS в Opus). |
| Bot token отозван или недействителен | Сгенерируйте новый токен через `/revoke`, затем `/newbot` или `/token` в BotFather. Обновите `.env`-файл. |
