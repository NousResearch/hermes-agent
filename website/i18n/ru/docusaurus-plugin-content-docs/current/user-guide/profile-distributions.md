# Дистрибутивы профилей: делитесь полноценным агентом

**Дистрибутив профиля** упаковывает полноценного Hermes-агента — его личность, навыки, cron-задачи, MCP-подключения и конфигурацию — в git-репозиторий. Любой, у кого есть доступ к репозиторию, может установить такого агента одной командой, обновлять его на месте и при этом не трогать свою память, сеансы и ключи API.

Если [профиль](./profiles.md) — это локальный агент, то дистрибутив делает его удобным для совместного использования.

## Что это значит

Раньше, до появления дистрибутивов, чтобы поделиться Hermes-агентом, приходилось вручную отправлять человеку:

1. свой `SOUL.md`
2. список навыков, которые нужно установить
3. свой `config.yaml`, но без секретов
4. описание подключённых MCP-серверов
5. все запланированные cron-задачи
6. инструкции по тому, какие переменные окружения нужно задать

...и надеяться, что он всё соберёт правильно. Любое обновление версии или исправление ошибки означало, что весь набор материалов приходилось собирать и отправлять заново.

С дистрибутивами всё это живёт в одном git-репозитории:

```
my-research-agent/
├── distribution.yaml    # манифест: имя, версия и требования к переменным окружения
├── SOUL.md              # личность агента / системная инструкция
├── config.yaml          # модель, температура, уровень рассуждений и настройки инструментов по умолчанию
├── skills/              # встроенные навыки, поставляемые с агентом
├── cron/                # запланированные задачи агента
└── mcp.json             # MCP-серверы, к которым подключается агент
```

Получатель запускает:

```bash
hermes profile install github.com/you/my-research-agent --alias
```

...и у него уже есть весь агент. Он подставляет свои API-ключи (`.env.EXAMPLE` → `.env`) и может запускать `my-research-agent chat` либо обращаться к нему через Telegram / Discord / Slack или любую другую платформу обмена сообщениями. Когда вы публикуете новую версию, он запускает `hermes profile update my-research-agent` и подтягивает ваши изменения, а его память и сеансы остаются на месте.

## Почему git?

Мы рассматривали tarball, HTTP-архивы и собственный формат. Ни один вариант не оказался лучше git:

- **Авторам не нужен отдельный этап сборки.** Отправили изменения в GitHub — пользователи сразу получили новую версию. Никакого цикла "упакуй, загрузи, обнови индекс".
- **Теги, ветки и коммиты уже выполняют роль системы версионирования.** Публикация тега делает для нас то же, что у других инструментов делает связка "упаковать + загрузить релиз".
- **Обновление сводится к fetch.** А не к повторной загрузке всего архива.
- **Прозрачно.** Пользователь может просматривать репозиторий, сравнивать версии, открывать issues, форкать его и настраивать под себя.
- **Приватные репозитории работают без дополнительной настройки.** Ключи SSH, помощники учётных данных git, сохранённые учётные данные GitHub CLI - всё, что уже настроено в терминале, подхватывается автоматически.
- **Воспроизводимость задаётся SHA коммита.** Точно так же, как это делают pip и npm.

Компромисс простой: у получателя должен быть установлен git. На любой машине, где Hermes работает в 2026 году, это уже и так есть.

## Когда стоит использовать дистрибутив?

Подходит, если:

- **Вы делитесь специализированным агентом** - агентом для контроля соответствия, ботом для ревью кода, исследовательским помощником или ботом поддержки - с командой или сообществом.
- **Вы развёртываете одного и того же агента на нескольких машинах** и не хотите каждый раз копировать файлы вручную.
- **Вы итеративно улучшаете агента** и хотите, чтобы получатели подхватывали новые версии одной командой.
- **Вы создаёте агента как продукт** - с продуманными настройками по умолчанию, отобранными навыками и выверенными системными инструкциями - и хотите, чтобы другие использовали его как стартовую точку.

Не подходит, если:

- **Вам нужно просто сделать резервную копию профиля на своей машине.** Используйте [`hermes profile export` / `import`](../reference/profile-commands.md#hermes-profile-export) - именно для этого они и нужны.
- **Вы хотите распространять вместе с агентом и API-ключи.** `auth.json` и `.env` намеренно исключены из дистрибутивов. Каждый пользователь приносит свои учётные данные.
- **Вы хотите передать вместе с агентом память, сеансы или историю разговоров.** Это пользовательские данные, а не содержимое дистрибутива. Они никогда не входят в поставку.

## Жизненный цикл: от автора к установке и обновлению

Ниже показан полный сквозной сценарий. Выберите ту сторону, которая вам нужна.

---

## Для авторов: публикация дистрибутива

### Шаг 1 — начните с рабочего профиля

Соберите и настройте агента так же, как любой другой профиль:

```bash
hermes profile create research-bot
research-bot setup                    # настроить модель и API-ключи
# Отредактируйте ~/.hermes/profiles/research-bot/SOUL.md
# Установите навыки, подключите MCP-серверы, задайте cron-задачи и т. д.
research-bot chat                     # обкатывайте агента, пока он не начнёт работать как надо
```

### Шаг 2 — добавьте `distribution.yaml`

Создайте `~/.hermes/profiles/research-bot/distribution.yaml`:

```yaml
name: research-bot
version: 1.0.0
description: "Автономный исследовательский помощник с инструментами arXiv и веб-поиска"
hermes_requires: ">=0.12.0"
author: "Your Name"
license: "MIT"

# Укажите, какие переменные окружения нужны агенту. Установщик сверит их
# со своей shell-средой и существующим `.env`, чтобы не спрашивать про
# ключи, которые уже настроены.
env_requires:
  - name: OPENAI_API_KEY
    description: "API-ключ OpenAI для доступа к модели"
    required: true
  - name: SERPAPI_KEY
    description: "API-ключ SerpAPI для веб-поиска"
    required: false
    default: ""
```

На этом манифест заканчивается. У каждого поля, кроме `name`, есть разумное значение по умолчанию.

### Шаг 3 — отправьте в git-репозиторий

```bash
cd ~/.hermes/profiles/research-bot
git init
git add .
git commit -m "v1.0.0"
git remote add origin git@github.com:you/research-bot.git
git tag v1.0.0
git push -u origin main --tags
```

Теперь этот репозиторий стал дистрибутивом. Любой, у кого есть доступ, может установить его.

:::note
В git-репозитории хранится **всё из каталога профиля, кроме того, что уже исключено из дистрибутивов**: `auth.json`, `.env`, `memories/`, `sessions/`, `state.db*`, `logs/`, `workspace/`, `*_cache/`, `local/`. Всё это остаётся у вас на машине. При желании можно добавить ещё и свой `.gitignore`, чтобы исключить дополнительные пути.
:::

### Шаг 4 — тегируйте стабильные версии

Каждый раз, когда агент доходит до стабильной точки, повышайте версию и ставьте тег:

```bash
# Измените distribution.yaml: version: 1.1.0
git add distribution.yaml SOUL.md skills/
git commit -m "v1.1.0: tighter research SOUL, add arxiv skill"
git tag v1.1.0
git push --tags
```

Получатели, которые запускают `hermes profile update research-bot`, подтянут последнюю версию.

### Как выглядит репозиторий

Полный пример авторского дистрибутива:

```
research-bot/
├── distribution.yaml            # обязательно
├── SOUL.md                      # настоятельно рекомендуется
├── config.yaml                  # модель, провайдер и настройки инструментов по умолчанию
├── mcp.json                     # MCP-подключения
├── skills/
│   ├── arxiv-search/SKILL.md
│   ├── paper-summarization/SKILL.md
│   └── citation-lookup/SKILL.md
├── cron/
│   └── weekly-digest.json       # запланированные задачи
└── README.md                    # описание для людей (необязательно)
```

### Что принадлежит дистрибутиву, а что пользователю

При обновлении часть данных заменяется, а часть остаётся на месте. По умолчанию:

| Категория | Пути | Что происходит при обновлении |
|---|---|---|
| **Принадлежит дистрибутиву** | `SOUL.md`, `config.yaml`, `mcp.json`, `skills/`, `cron/`, `distribution.yaml` | Заменяется из новой копии |
| **Переопределение конфигурации** | `config.yaml` | По умолчанию оно сохраняется: пользователь мог уже настроить модель или провайдера. Чтобы сбросить его, используйте `--force-config`. |
| **Принадлежит пользователю** | `memories/`, `sessions/`, `state.db*`, `auth.json`, `.env`, `logs/`, `workspace/`, `plans/`, `home/`, `*_cache/`, `local/` | Никогда не трогается |

Перечень путей, принадлежащих дистрибутиву, можно переопределить в манифесте:

```yaml
distribution_owned:
  - SOUL.md
  - skills/research/
  - cron/digest.json
```

Когда список не задан, используются значения по умолчанию, приведённые выше. Именно такого поведения обычно и ожидают авторы большинства дистрибутивов.

## Для устанавливающих: как использовать дистрибутив

### Установка

```bash
hermes profile install github.com/you/research-bot --alias
```

Что происходит:

1. Репозиторий клонируется во временный каталог.
2. Hermes читает `distribution.yaml` и показывает манифест: имя, версию, описание, автора и необходимые переменные окружения.
3. Каждая обязательная переменная окружения проверяется по вашей shell-среде и по существующему `.env` целевого профиля. Установщик помечает всё как `✓ set` или `needs setting`, чтобы было сразу видно, что нужно подготовить.
4. Затем он запрашивает подтверждение. Если нужно, можно передать `-y` / `--yes` и пропустить этот шаг.
5. Файлы, принадлежащие дистрибутиву, копируются в `~/.hermes/profiles/research-bot/` (или в тот путь, который соответствует `name` из манифеста).
6. Создаётся `.env.EXAMPLE` с закомментированными обязательными ключами - его нужно скопировать в `.env` и заполнить.
7. При `--alias` создаётся обёртка, чтобы можно было запускать `research-bot chat` напрямую.

### Источники

Подойдёт любой git URL:

```bash
# сокращённая запись GitHub
hermes profile install github.com/you/research-bot

# полный HTTPS
hermes profile install https://github.com/you/research-bot.git

# SSH
hermes profile install git@github.com:you/research-bot.git

# самостоятельно размещённый репозиторий, GitLab, Gitea, Forgejo — любой git-хост
hermes profile install https://git.example.com/team/research-bot.git

# приватный репозиторий с уже настроенной git-аутентификацией
hermes profile install git@github.com:your-org/internal-bot.git

# локальный каталог во время разработки (git push не нужен)
hermes profile install ~/my-profile-in-progress/
```

### Переопределить имя профиля

Два пользователя могут поставить один и тот же дистрибутив под разными локальными именами:

```bash
# Алиса
hermes profile install github.com/acme/support-bot --name support-us --alias
# Боб (тот же дистрибутив, но другое локальное имя)
hermes profile install github.com/acme/support-bot --name support-eu --alias
```

### Заполнить переменные окружения

После установки в профиле появляется файл `.env.EXAMPLE`:

```
# Переменные окружения, необходимые этому дистрибутиву Hermes.
# Скопируйте в `.env` и заполните своими значениями перед запуском.

# Ключ API OpenAI для доступа к модели
# (обязательно)
OPENAI_API_KEY=

# Ключ SerpAPI для веб-поиска
# (необязательно)
# SERPAPI_KEY=
```

Скопируйте его:

```bash
cp ~/.hermes/profiles/research-bot/.env.EXAMPLE ~/.hermes/profiles/research-bot/.env
# Отредактируйте .env и вставьте свои реальные ключи
```

Обязательные ключи, которые уже есть в вашей shell-среде, например `OPENAI_API_KEY`, экспортированный в `~/.zshrc`, помечаются при установке как `✓ set` - дублировать их в `.env` не нужно.

### Проверить, что именно установлено

```bash
hermes profile info research-bot
```

Вывод показывает:

```
Distribution: research-bot
Version:      1.0.0
Description:  Автономный исследовательский помощник с инструментами arXiv и веб-поиска
Author:       Your Name
Requires:     Hermes >=0.12.0
Source:       https://github.com/you/research-bot
Installed:    2026-05-08T17:04:32+00:00

Environment variables:
  OPENAI_API_KEY (required) — ключ API OpenAI (для доступа к модели)
  SERPAPI_KEY (optional) — ключ SerpAPI (для веб-поиска)
```

`hermes profile list` также показывает имя и версию дистрибутива в колонке `Distribution`, а `hermes profile show <name>` / `delete <name>` выводят URL источника, чтобы сразу было видно, пришёл ли профиль из git-репозитория или был создан локально:

```
 Profile          Model                        Шлюз         Alias        Distribution
 ───────────────    ───────────────────────────    ───────────    ───────────    ────────────────────
 ◆default         claude-sonnet-4              stopped      —            —
  coder           gpt-5                        stopped      coder        —
  research-bot    claude-opus-4                stopped      research-bot research-bot@1.0.0
  telemetry       claude-sonnet-4              running      telemetry    telemetry@2.3.1
```

### Обновление

```bash
hermes profile update research-bot
```

Что происходит:

1. Репозиторий заново клонируется из записанного URL источника.
2. Заменяются файлы, принадлежащие дистрибутиву (`SOUL`, `skills`, `cron`, `mcp.json`).
3. **`config.yaml` сохраняется** - возможно, вы уже подстроили модель, температуру или другие параметры. Чтобы перезаписать его, передайте `--force-config`.
4. **Пользовательские данные не трогаются**: память, сеансы, учётные данные, `.env`, журналы, состояние.

Никакой повторной загрузки всего архива. Никакого затирания локальных изменений в конфиге. Никакого удаления истории разговоров.

### Удаление

```bash
hermes profile delete research-bot
```

Перед подтверждением удаления сначала показывается информация о дистрибутиве, а уже потом Hermes просит подтвердить действие:

```
Profile: research-bot
Path:    ~/.hermes/profiles/research-bot
Model:   claude-opus-4 (anthropic)
Skills:  12
Distribution: research-bot@1.0.0
Installed from: https://github.com/you/research-bot

This will permanently delete:
  • Вся конфигурация, API-ключи, память, сеансы, навыки, cron-задачи
  • Псевдоним команды (~/.local/bin/research-bot)

Введите `research-bot`, чтобы подтвердить:
```

Так вы не удалите агента случайно и всегда будете видеть, откуда он был установлен.

## Сценарии использования и типовые схемы

### Личное: синхронизировать одного агента между машинами

Вы собрали исследовательского помощника на ноутбуке и хотите иметь того же самого агента на рабочей станции.

```bash
# Ноутбук
cd ~/.hermes/profiles/research-bot
git init && git add . && git commit -m "initial"
git remote add origin git@github.com:you/research-bot.git
git push -u origin main

# Рабочая станция
hermes profile install github.com/you/research-bot --alias
# Заполните `.env`. Готово.
```

Любые изменения на ноутбуке (`git commit && push`) затем подтягиваются на рабочую станцию через `hermes profile update research-bot`. Память остаётся локальной для каждой машины: ноутбук помнит свои разговоры, рабочая станция - свои, и они не конфликтуют.

### Командное: выпустить проверенного внутреннего агента

Инженерной команде нужен общий бот для ревью PR со своим `SOUL`, своим набором навыков и cron-задачей, которая прогоняет через него каждый пул-реквест.

```bash
# Ведущий инженер
cd ~/.hermes/profiles/pr-reviewer
# ... собрать и настроить ...
git init && git add . && git commit -m "v1.0 PR reviewer"
git tag v1.0.0
git push -u origin main --tags    # отправить в внутренний git-хост компании

# Каждый инженер
hermes profile install git@github.com:your-org/pr-reviewer.git --alias
# Заполните `.env` своим API-ключом (учётка оплачивается вами); `.env.EXAMPLE` показывает, что требуется
pr-reviewer chat
```

Когда ответственный за профиль выпускает v1.1 - с более удачным `SOUL` или новым навыком - инженеры просто запускают `hermes profile update pr-reviewer`, и через несколько минут у всех уже новая версия.

### Сообщество: опубликовать публичного агента

Вы сделали что-то новое - например, "Polymarket trader", "суммаризатор академических статей" или "помощник для администрирования Minecraft-сервера" - и хотите этим поделиться.

```bash
# Вы
cd ~/.hermes/profiles/polymarket-trader
# Напишите качественный README.md в корне репозитория — GitHub показывает его на странице репозитория
git init && git add . && git commit -m "v1.0"
git tag v1.0.0
# Опубликуйте в публичном репозитории GitHub
git remote add origin https://github.com/you/hermes-polymarket-trader.git
git push -u origin main --tags

# Любой
hermes profile install github.com/you/hermes-polymarket-trader --alias
```

Опубликуйте команду установки в соцсетях - и люди начнут пробовать, писать issues и присылать PR. Если кому-то нужно что-то поменять под себя, он просто форкает репозиторий - обычный git-процесс, к которому все уже привыкли.

### Продукт: выпустить агента с продуманными настройками по умолчанию

Вы собрали надстройку поверх Hermes - например, контур для контроля соответствия, платформу поддержки пользователей или отраслевую исследовательскую систему. Вам нужно распространять это как продукт.

```yaml
# distribution.yaml
name: telemetry-harness
version: 2.3.1
description: "Телеметрическая обвязка для compliance — отслеживает и проверяет регламентированные сценарии"
hermes_requires: ">=0.13.0"
author: "Acme Compliance Inc."
license: "Commercial"

env_requires:
  - name: ACME_API_KEY
    description: "Ваш лицензионный ключ Acme Compliance (пишите на support@acme.com)"
    required: true
  - name: OPENAI_API_KEY
    description: "Ключ OpenAI API для доступа к модели"
    required: true
  - name: GRAPHITI_MCP_URL
    description: "URL вашего экземпляра Graphiti knowledge graph"
    required: false
    default: "http://127.0.0.1:8000/sse"
```

Ваши клиенты устанавливают всё одной командой; предпросмотр установки заранее показывает, какие ключи нужно подготовить; обновления приходят сразу после нового тега; при этом их пользовательские данные (`memories/`, `sessions/`) остаются на месте.

### Временное: одноразовые скрипты на общей инфраструктуре

Вы руководите дежурной сменой. Вам нужен временный агент, который разбирает боевой инцидент — с заранее подготовленным `SOUL`, нужными инструментами и MCP-подключениями — и который будет работать на трёх ноутбуках дежурных инженеров в течение следующей недели.

```bash
# Вы
# Соберите профиль, сделайте commit и отправьте в приватный репозиторий
git push -u origin main

# Каждый дежурный инженер
hermes profile install git@github.com:your-org/incident-2026-q2.git --alias

# Инцидент закрыт — удалите профиль
hermes profile delete incident-2026-q2
```

Цикл установки и удаления настолько дешёвый, что такой агент можно сделать почти одноразовым.

## Рецепты

### Зафиксировать конкретную версию

:::note
Привязка к git ref (`#v1.2.0`) запланирована, но в первой версии её нет — сейчас установка отслеживает основную ветку. Отслеживайте установленную версию через `hermes profile info <name>` и не обновляйтесь, пока сами не будете готовы.
:::

### Проверить текущую версию и сравнить с последней

```bash
# Установленная версия
hermes profile info research-bot | grep Version

# Последняя версия upstream (без установки)
git ls-remote --tags https://github.com/you/research-bot | tail -5
```

### Сохранять локальные изменения конфигурации при обновлениях

Поведение по умолчанию уже делает именно это: `config.yaml` сохраняется. Если хотите подстраховаться, держите локальные правки в файле, который дистрибутив не владеет:

```yaml
# ~/.hermes/profiles/research-bot/local/my-overrides.yaml
# (дистрибутив никогда не трогает local/)
```

...и подключайте его из `config.yaml` или из `SOUL`, если это нужно.

### Сделать чистую переустановку

```bash
# Полная переустановка с нуля (потеряются и память, и сеансы)
hermes profile delete research-bot --yes
hermes profile install github.com/you/research-bot --alias

# Обновиться до текущего main, но сбросить config.yaml к значению дистрибутива
hermes profile update research-bot --force-config --yes
```

### Форкнуть и доработать под себя

Обычный git-процесс - дистрибутивы, по сути, просто репозитории:

```bash
# Форкните репозиторий на GitHub, а затем установите свой fork
hermes profile install github.com/yourname/forked-research-bot --alias

# Итерации локально в ~/.hermes/profiles/forked-research-bot/
# Редактируйте SOUL.md, делайте commit и push в свой fork
# Изменения upstream: подтягивайте их в свой fork обычным способом
```

### Проверить дистрибутив перед публикацией

На машине автора:

```bash
# Установка из локального каталога (git push не нужен)
hermes profile install ~/.hermes/profiles/research-bot --name research-bot-test --alias

# Дорабатывайте, удаляйте и ставьте заново, пока результат не устроит
hermes profile delete research-bot-test --yes
hermes profile install ~/.hermes/profiles/research-bot --name research-bot-test
```

## Что никогда не входит в дистрибутив

Установщик жёстко исключает эти пути, даже если автор случайно положил их в репозиторий. Переопределить это нельзя ни одной настройкой - это защитное ограничение, покрытое регрессионными тестами:

- `auth.json` — OAuth-токены, учётные данные платформ
- `.env` — API-ключи, секреты
- `memories/` — память разговора
- `sessions/` — история разговоров
- `state.db`, `state.db-shm`, `state.db-wal` — метаданные сеансов
- `logs/` — журналы агента и ошибок
- `workspace/` — сгенерированные рабочие файлы
- `plans/` — черновые планы
- `home/` — домашний каталог пользователя в Docker-бэкендах
- `*_cache/` — кэши изображений, аудио и документов
- `local/` — пространство для пользовательских доработок

Когда вы клонируете дистрибутив, этих путей там просто нет. Когда обновляете его, они остаются на месте. Если вы поставили один и тот же дистрибутив на пять машин, у вас будет пять независимых наборов этих данных - по одному на каждую машину.

## Безопасность и доверие

Дистрибутивы профилей по умолчанию не подписаны. По сути, вы доверяете:

- **git-хосту** (GitHub, GitLab или любому другому) - в том, что он отдаёт именно те байты, которые автор отправил;
- **автору** - в том, что он не положил внутрь вредоносный `SOUL`, навыки или cron-задачи.

Cron-задачи из дистрибутива **не планируются автоматически** - установщик выводит `hermes -p <name> cron list`, а включаете их уже вы вручную. `SOUL.md` и навыки начинают работать сразу, как только вы начинаете чат с профилем, так что перед первым запуском стоит прочитать их, если вы ставите профиль от незнакомого автора.

Грубая аналогия: установка дистрибутива похожа на установку расширения для браузера или VS Code. Минимум трения, максимум возможностей, но источник нужно доверять. Для внутренних корпоративных дистрибутивов используйте приватный репозиторий и обычную git-аутентификацию - ничего нового настраивать не придётся.

В будущих версиях могут появиться подпись, lockfile (`.distribution-lock.yaml`) с разрешённым SHA коммита и флаг `--dry-run`, который покажет diff перед применением обновления. Пока этого ещё нет.

## Под капотом

За точными деталями реализации, поведением CLI и полным набором флагов см. [Справочник команд профилей](../reference/profile-commands.md).

Коротко:

- `install`, `update` и `info` живут внутри `hermes profile`, а не в отдельном дереве команд.
- Формат манифеста - YAML с минимальной обязательной схемой (`name`).
- Установщик использует ваш локальный `git`, поэтому любая аутентификация, которую уже умеет ваш shell (SSH keys, credential helpers), работает как есть.
- После клонирования `.git/` удаляется - установленный профиль не остаётся полноценным git-репозиторием, и вы не рискуете случайно закоммитить свой `.env` в историю дистрибутива.
- Зарезервированные имена профилей (`hermes`, `test`, `tmp`, `root`, `sudo`) отклоняются на этапе установки, чтобы не конфликтовать с распространёнными бинарниками.
