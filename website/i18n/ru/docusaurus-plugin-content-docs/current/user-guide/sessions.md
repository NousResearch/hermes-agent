# Сеансы

Hermes Agent автоматически сохраняет каждый разговор как сеанс. Сеансы позволяют возобновлять разговоры, искать по истории прошлых сеансов и управлять полной историей переписки.

## Как устроены сеансы

Каждый разговор — будь то CLI, Telegram, Discord, Slack, WhatsApp, Signal, Matrix, Teams или любая другая платформа обмена сообщениями — сохраняется как сеанс с полной историей сообщений. Сеансы отслеживаются в двух взаимодополняющих системах:

1. **SQLite-база** (`~/.hermes/state.db`) — структурированные метаданные сеанса с полнотекстовым поиском FTS5
2. **JSONL-транскрипты** (`~/.hermes/sessions/`) — сырые транскрипты разговоров, включая вызовы инструментов через шлюз

В SQLite-базе хранятся:
- ID сеанса, исходная платформа, ID пользователя
- **Заголовок сеанса** (уникальное, удобочитаемое имя)
- Имя модели и конфигурация
- Снимок системной инструкции
- Полная история сообщений (роль, содержимое, вызовы инструментов, результаты инструментов)
- Счётчики токенов (ввод/вывод)
- Временные метки (started_at, ended_at)
- ID родительского сеанса (для разбиения сеансов при сжатии)

### Что учитывается в контексте

Hermes хранит историю сеанса, чтобы можно было возобновить разговор, но не подставляет в контекст каждый байт из прошлых запросов. На каждом ходу модель видит только системную инструкцию, текущее окно разговора и то, что Hermes явно добавил именно для этого шага.

Вложения обрабатываются как входы, привязанные к конкретному ходу:

- Изображения могут быть прикреплены нативно к следующему вызову модели или предварительно разобраны в текстовое описание, если активная модель не поддерживает нативное зрение.
- Аудио транскрибируется в текст, когда настроено распознавание речи.
- Текстовые документы могут попадать в контекст в виде извлечённого текста; другие типы документов обычно представлены сохранённым локальным путём и короткой пометкой.
- Пути к вложениям и извлечённый / производный текст могут появляться в транскрипте, но исходные байты изображения, аудио или двоичного файла не копируются в будущие запросы снова и снова.

Например, если пользователь отправляет изображение и просит сделать по нему мем, Hermes может один раз посмотреть на это изображение через vision и запустить скрипт обработки изображения. Последующие ходы не подставляют исходный JPEG в контекст автоматически. В них остаётся только то, что было записано в разговоре: сам запрос пользователя, короткое описание изображения, локальный путь к кешу или итоговый ответ ассистента.

Самая частая причина разрастания контекста — не сами медиафайлы. Это многословный текст: вставленные транскрипты, полные логи, большие выводы инструментов, длинные diff-блоки, повторяющиеся отчёты о статусе и подробные дампы доказательств. По возможности используйте сводки, пути к файлам, точечные фрагменты и поиск через инструменты вместо того, чтобы копировать в чат большие артефакты целиком.

:::tip
Используйте `/compress`, когда сеанс становится слишком длинным, `/new` — для нового чистого диалога, а `hermes sessions prune` — только когда хотите удалить из хранилища старые завершённые сеансы. Сжатие уменьшает активный контекст; это не удаление ради приватности. Передайте имя в `/new` (например, `/new payments-refactor`), чтобы заранее задать заголовок нового сеанса — так потом его проще найти через `/resume <name>` или в списке `/sessions`.
:::

## Возобновление CLI-сеансов

Возобновляйте прошлые разговоры из CLI с помощью `--continue` или `--resume`:

### Возобновить последний сеанс

```bash
# Возобновить последний CLI-сеанс
hermes --continue
hermes -c

# Или через подкоманду chat
hermes chat --continue
hermes chat -c
```

Это ищет самый последний `cli`-сеанс в SQLite-базе и загружает всю его историю разговора.

### Возобновить по имени

Если вы дали сеансу заголовок (см. [Именование сеансов](#session-naming) ниже), его можно возобновить по имени:

```bash
# Возобновить сеанс по имени
hermes -c "my project"

# Если есть варианты по линии наследования (my project, my project #2, my project #3),
# команда автоматически возобновит самый последний из них
hermes -c "my project"   # → возобновит "my project #3"
```

### Возобновить конкретный сеанс

```bash
# Возобновить конкретный сеанс по ID
hermes --resume 20250305_091523_a1b2c3d4
hermes -r 20250305_091523_a1b2c3d4

# Возобновить по заголовку
hermes --resume "refactoring auth"

# Или через подкоманду chat
hermes chat --resume 20250305_091523_a1b2c3d4
```

ID сеанса отображается, когда вы выходите из CLI-сеанса, а также доступен через `hermes sessions list`.

### Сводка разговора при возобновлении {#conversation-recap-on-resume}

Когда вы возобновляете сеанс, Hermes показывает компактную сводку предыдущего разговора в оформленной панели перед строкой ввода:

<img className="docs-terminal-figure" src="/img/docs/session-recap.svg" alt="Схематичный вид панели «Предыдущий разговор», которая показывается при возобновлении сеанса Hermes." />
<p className="docs-figure-caption">При возобновлении сеанса показывается компактная панель со сводкой последних ходов пользователя и ассистента, после чего вы возвращаетесь к активному диалогу.</p>

Сводка:
- Показывает **сообщения пользователя** (золотой `●`) и **ответы ассистента** (зелёный `◆`)
- **Обрезает** длинные сообщения (300 символов для пользователя, 200 символов / 3 строки для ассистента)
- **Сворачивает** вызовы инструментов в счётчик с именами инструментов (например, `[3 tool calls: terminal, web_search]`)
- **Скрывает** системные сообщения, результаты инструментов и внутренние рассуждения
- **Ограничивается** последними 10 обменами с индикатором `"... N earlier messages ..."`
- Использует **приглушённый стиль**, чтобы отделить сводку от активного разговора

Чтобы отключить сводку и вернуть минималистичный однострочный вывод, задайте в `~/.hermes/config.yaml`:

```yaml
display:
  resume_display: minimal   # по умолчанию: full
```

:::tip
ID сеансов имеют формат `YYYYMMDD_HHMMSS_<hex>` — CLI/TUI-сеансы используют 6-символьный hex-суффикс (например, `20250305_091523_a1b2c3`), сеансы шлюза — 8-символьный суффикс (например, `20250305_091523_a1b2c3d4`). Возобновлять можно по ID (полностью или по уникальному префиксу) либо по заголовку — оба варианта работают и с `-c`, и с `-r`.
:::

## Передача между платформами {#cross-platform-handoff}

Используйте `/handoff <platform>` из CLI-сеанса, чтобы передать живой разговор в домашний канал платформы обмена сообщениями. Агент подхватывает его ровно там же, где CLI остановился: тот же ID сеанса, полный транскрипт с учётом ролей, все вызовы инструментов.

```bash
# Внутри CLI-сеанса
/handoff telegram
```

Что происходит:

1. CLI проверяет, что `<platform>` включена и для неё задан домашний канал (один раз его настраивают через `/sethome` в чате назначения).
2. CLI помечает сеанс как ожидающий передачи и **блокирующе опрашивает шлюз**. Если агент ещё в середине хода, команда отклоняется — дождитесь завершения текущего ответа.
3. Наблюдатель шлюза захватывает передачу и просит адаптер назначения открыть новый тред:
   - **Telegram** — создаёт новую тему в форуме (либо DM topic, если в чате включён Topics mode Bot API 9.4+, либо тему в forum supergroup).
   - **Discord** — создаёт тред с автоархивацией на 1440 минут под домашним текстовым каналом.
   - **Slack** — публикует стартовое сообщение и использует его `ts` как якорь треда.
   - **WhatsApp / Signal / Matrix / SMS** — нативных тредов нет, поэтому используется прямо домашний канал.
4. Шлюз перепривязывает ключ назначения к вашему существующему CLI ID сеанса, а затем создаёт синтетический ход пользователя с просьбой подтвердить и кратко подытожить. Ответ приходит в новый тред.
5. Когда шлюз подтверждает успех, CLI печатает подсказку `/resume` и корректно завершает работу:

   ```
   ↻ Передача завершена. Сеанс теперь активен в Telegram.
     Вернуться к нему в CLI позже можно так: /resume my-session-title
   ```

6. После этого разговор живёт на платформе. Отвечайте в новом треде — любой авторизованный участник этого канала работает в одном и том же сеансе, а любое последующее реальное сообщение пользователя в треде подхватывается без разрыва, потому что ключ сеанса треда не зависит от `user_id`.

**Возврат в CLI:** когда захотите вернуться к десктопу, просто выполните `/resume <title>` (или `hermes -r "<title>"` из shell) и продолжите разговор с того места, где он остался на платформе.

**Режимы отказа:**
- Нет настроенного домашнего канала → CLI отклоняет передачу с подсказкой `/sethome`.
- Платформа не включена / шлюз не запущен → CLI ждёт до 60 секунд и выводит понятную ошибку, при этом CLI-сеанс остаётся целым.
- Создание треда не удалось (права, выключенный Topics mode) → происходит откат на домашний канал, и передача всё равно завершается; изоляции треда не будет, но сама передача сработает.
- Сбой `adapter.send` (rate limit, временная ошибка API) → передача помечается как неудачная с причиной, а строка очищается, чтобы можно было повторить попытку.

**Ограничение, о котором полезно знать:** для платформ без тредов и с общими групповыми чатами синтетический ход привязывается как сеанс в формате DM. Для личных DM это нормально (а это и есть типичный сценарий), но для по-настоящему общих групповых чатов это не идеальный вариант. Работа с тредами покрывает Telegram, Discord и Slack — то есть большую часть типичных сценариев, — поэтому в большинстве установок с этой особенностью не сталкиваются.

## Именование сеансов {#session-naming}

Давайте сеансам человекочитаемые заголовки, чтобы их было проще искать и возобновлять.

### Автоматически сгенерированные заголовки

Hermes автоматически генерирует краткий описательный заголовок (3–7 слов) для каждого сеанса после первого обмена. Это выполняется в фоне через быструю вспомогательную модель и не добавляет задержки. Автогенерированные заголовки видны в `hermes sessions list` и `hermes sessions browse`.

Автоматическая генерация заголовка срабатывает только один раз для сеанса и пропускается, если вы уже задали заголовок вручную.

### Задать заголовок вручную

Используйте слэш-команду `/title` внутри любого чата (CLI или шлюза):

```
/title my research project
```

Заголовок применяется сразу. Если сеанс ещё не успел появиться в базе данных (например, вы вызываете `/title` до первого сообщения), команда ставится в очередь и применяется, когда сеанс начнётся.

Вы также можете переименовывать существующие сеансы из командной строки:

```bash
hermes sessions rename 20250305_091523_a1b2c3d4 "refactoring auth module"
```

### Правила заголовков

- **Уникальность** — два сеанса не могут иметь одинаковый заголовок
- **Не длиннее 100 символов** — так список остаётся аккуратным
- **Санитизация** — управляющие символы, zero-width символы и RTL-override удаляются автоматически
- **Обычный Unicode разрешён** — emoji, CJK, символы с диакритикой и т. п. работают нормально

### Автоматическое наследование при сжатии

Когда контекст сеанса сжимается (вручную через `/compress` или автоматически), Hermes создаёт новый продолженный сеанс. Если у исходного сеанса был заголовок, новый сеанс автоматически получает нумерованный вариант:

```
"my project" → "my project #2" → "my project #3"
```

При возобновлении по имени (`hermes -c "my project"`) он автоматически выбирает самый свежий сеанс в этой линии.

### `/title` на платформах мессенджеров

Команда `/title` работает на всех платформах обмена сообщениями (Telegram, Discord, Slack, WhatsApp):

- `/title My Research` — задать заголовок сеанса
- `/title` — показать текущий заголовок

## Команды управления сеансами

Hermes предоставляет полный набор команд управления сеансами через `hermes sessions`:

### Показать список сеансов

```bash
# Показать недавние сеансы (по умолчанию: последние 20)
hermes sessions list

# Фильтровать по платформе
hermes sessions list --source telegram

# Показать больше сеансов
hermes sessions list --limit 50
```

Когда у сеансов есть заголовки, вывод показывает заголовки, предпросмотры и относительное время последней активности:

```
Title                  Preview                                  Last Active   ID
────────────────────────────────────────────────────────────────────────────────────────────────
refactoring auth       Help me refactor the auth module please   2h ago        20250305_091523_a
my project #3          Can you check the test failures?          yesterday     20250304_143022_e
—                      What's the weather in Las Vegas?          3d ago        20250303_101500_f
```

Когда у сеансов нет заголовков, используется более простой формат:

```
Preview                                            Last Active   Src    ID
──────────────────────────────────────────────────────────────────────────────────────
Help me refactor the auth module please             2h ago        cli    20250305_091523_a
What's the weather in Las Vegas?                    3d ago        tele   20250303_101500_f
```

### Экспорт сеансов

```bash
# Экспортировать все сеансы в JSONL-файл
hermes sessions export backup.jsonl

# Экспортировать сеансы с конкретной платформы
hermes sessions export telegram-history.jsonl --source telegram

# Экспортировать один сеанс
hermes sessions export session.jsonl --session-id 20250305_091523_a1b2c3d4
```

Экспортированные файлы содержат по одному JSON-объекту на строку со всей метаинформацией о сеансе и всеми сообщениями.

### Удалить сеанс

```bash
# Удалить конкретный сеанс (с подтверждением)
hermes sessions delete 20250305_091523_a1b2c3d4

# Удалить без подтверждения
hermes sessions delete 20250305_091523_a1b2c3d4 --yes
```

### Переименовать сеанс

```bash
# Задать или изменить заголовок сеанса
hermes sessions rename 20250305_091523_a1b2c3d4 "debugging auth flow"

# Заголовки из нескольких слов не требуют кавычек в CLI
hermes sessions rename 20250305_091523_a1b2c3d4 debugging auth flow
```

Если заголовок уже используется другим сеансом, выводится ошибка.

### Очистить старые сеансы

```bash
# Удалить завершённые сеансы старше 90 дней (по умолчанию)
hermes sessions prune

# Пользовательский порог возраста
hermes sessions prune --older-than 30

# Очищать только сеансы с конкретной платформы
hermes sessions prune --source telegram --older-than 60

# Пропустить подтверждение
hermes sessions prune --older-than 30 --yes
```

:::info
Очистка удаляет только **завершённые** сеансы (то есть явно завершённые или автоматически сброшенные). Активные сеансы никогда не очищаются.
:::

### Статистика сеансов

```bash
hermes sessions stats
```

Вывод:

```
Total sessions: 142
Total messages: 3847
  cli: 89 sessions
  telegram: 38 sessions
  discord: 15 sessions
Database size: 12.4 MB
```

Для более глубокой аналитики — расход токенов, оценка стоимости, разбор инструментов и паттерны активности — используйте [`hermes insights`](/reference/cli-commands#hermes-insights).

## Инструмент поиска по сеансам {#session-search-tool}

У агента есть встроенный инструмент `session_search`, который выполняет полнотекстовый поиск по всем прошлым разговорам через движок FTS5 SQLite — и позволяет прокручивать любой найденный сеанс. Никаких вызовов LLM, никакого суммирования, никакого усечения. Любая форма возвращает реальные сообщения из БД.

### Три варианта вызова

Инструмент сам понимает, что вам нужно, по тем аргументам, которые вы передали. Параметра `mode` нет.

**1. Обзор — передайте `query`:**

```python
session_search(query="auth refactor", limit=3)
```

Запускает FTS5, убирает дубликаты по линии сеанса и возвращает top N сеансов. Каждый результат содержит:

- `session_id`, `title`, `when`, `source`
- `snippet` — фрагмент совпадения с подсветкой FTS5
- `bookend_start` — первые 3 сообщения пользователя и ассистента сеанса (начальный блок)
- `messages` — ±5 сообщений вокруг совпадения FTS5, с пометкой якорного сообщения (совпадение в контексте)
- `bookend_end` — последние 3 сообщения пользователя и ассистента сеанса (финальный блок)
- `match_message_id`, `messages_before`, `messages_after`

Начальный и финальный блоки вместе с окном позволяют восстановить путь цель → совпадение → решение без необходимости загружать весь транскрипт. Типичное время выполнения: 15–50 мс на реальной базе сеансов.

**2. Прокрутка — передайте `session_id` + `around_message_id`:**

```python
session_search(session_id="20260510_174648_805cc2", around_message_id=590803, window=10)
```

Возвращает окно из ±`window` сообщений вокруг якоря. Никакого FTS5, никаких начальных и финальных блоков — только нужный фрагмент. Используйте после вызова обзора, если контекста из стандартного окна ±5 недостаточно.

- Чтобы прокрутить **вперёд**: передайте `messages[-1].id` обратно как `around_message_id`
- Чтобы прокрутить **назад**: передайте `messages[0].id` обратно как `around_message_id`
- Граничное сообщение появляется в обоих окнах как ориентир
- Когда `messages_before` или `messages_after` меньше `window`, вы находитесь в начале или в конце сеанса

Типичное время выполнения: 1–2 мс на один вызов прокрутки.

**3. Просмотр — без аргументов:**

```python
session_search()
```

Возвращает недавние сеансы в хронологическом порядке (заголовки, предпросмотры, временные метки). Полезно, когда пользователь спрашивает «над чем я работал», но не называет тему.

### Синтаксис запросов FTS5

Режим поиска по ключевым словам поддерживает стандартный синтаксис запросов FTS5:

- Простые ключевые слова: `docker deployment` (по умолчанию FTS5 использует AND)
- Фразы: `"exact phrase"`
- Логика: `docker OR kubernetes`, `python NOT java`
- Префиксы: `deploy*`

### Дополнительные параметры

- `sort` — `newest` или `oldest`, поверх ранжирования FTS5. Если не указывать, используется порядок только по релевантности (по умолчанию; подходит для exploratory recall). `newest` удобен для вопросов «где мы остановились», `oldest` — для вопросов «с чего это началось».
- `role_filter` — разделённые запятыми роли, которые нужно включить. По умолчанию discovery ищет по `user,assistant` (tool output обычно шумный). Передайте `user,assistant,tool`, чтобы включить вывод инструментов (для отладки поведения tool) или `tool`, чтобы искать только по выводу инструментов.

### Когда это используется

Агенту автоматически предлагается использовать session search:

> *«Когда пользователь ссылается на что-то из прошлого разговора или вы подозреваете, что есть релевантный предыдущий контекст, используйте `session_search`, чтобы восстановить его, прежде чем просить пользователя повторить себя.»*

Типичные триггеры: «мы уже делали это», «помнишь когда», «в прошлый раз», «как я уже говорил» или любая ссылка на проект / человека / концепцию, которая не находится в текущем окне.

## Отслеживание сеансов по платформам

### Сеансы шлюза

На платформах обмена сообщениями сеансы привязываются к детерминированному ключу сеанса, построенному из источника сообщения:

| Тип чата | Формат ключа по умолчанию | Поведение |
|-----------|--------------------|----------|
| Telegram DM | `agent:main:telegram:dm:<chat_id>` | Один сеанс на DM-чат |
| Discord DM | `agent:main:discord:dm:<chat_id>` | Один сеанс на DM-чат |
| WhatsApp DM | `agent:main:whatsapp:dm:<canonical_identifier>` | Один сеанс на DM-пользователя (LID / телефонные псевдонимы сворачиваются в одну сущность, если есть сопоставление) |
| Group chat | `agent:main:<platform>:group:<chat_id>:<user_id>` | По одному сеансу на пользователя внутри группы, если платформа отдаёт user ID |
| Group thread/topic | `agent:main:<platform>:group:<chat_id>:<thread_id>` | Общий сеанс для всех участников треда (по умолчанию). По пользователю, если `thread_sessions_per_user: true`. |
| Channel | `agent:main:<platform>:channel:<chat_id>:<user_id>` | По одному сеансу на пользователя внутри канала, если платформа отдаёт user ID |

Когда Hermes не может получить идентификатор участника для общего чата, он откатывается к одному общему сеансу для этой комнаты.

### Общие и изолированные групповые сеансы

По умолчанию в `config.yaml` у Hermes включено `group_sessions_per_user: true`. Это значит:

- Алиса и Боб могут писать Hermes в одном Discord-канале, не делясь историей переписки
- длинная инструментальная задача одного пользователя не засоряет окно контекста другого
- обработка прерываний тоже остаётся на уровне пользователя, потому что ключ running-agent совпадает с изолированным ключом сеанса

Если вам нужен один общий «мозг комнаты», задайте:

```yaml
group_sessions_per_user: false
```

Это вернёт для групп и каналов один общий сеанс на комнату. Так сохраняется общий контекст разговора, но вместе с ним делятся затраты токенов, состояние прерывания и рост контекста.

### Политики сброса сеансов

Сеансы шлюза автоматически сбрасываются по настраиваемым политикам:

- **idle** — сброс после N минут бездействия
- **daily** — сброс в определённый час каждый день
- **both** — сброс по тому условию, которое наступит первым (idle или daily)
- **none** — автоматический сброс отключён

Перед авто-сбросом сеансу даётся ход, чтобы он мог сохранить важные memory или skill из разговора.

Сеансы с **активными фоновыми процессами** никогда не сбрасываются автоматически, независимо от политики.

## Места хранения

| Что | Путь | Описание |
|------|------|-------------|
| SQLite database | `~/.hermes/state.db` | Все метаданные и сообщения сеансов с FTS5 |
| Транскрипты шлюза | `~/.hermes/sessions/` | JSONL-транскрипты по сеансам + индекс `sessions.json` |
| Индекс шлюза | `~/.hermes/sessions/sessions.json` | Соответствие ключа сеанса активным ID сеансов |

SQLite-база работает в режиме WAL для одновременных читателей и одного писателя, что хорошо подходит для мультиплатформенной архитектуры шлюза.

### Схема базы данных

Ключевые таблицы в `state.db`:

- **sessions** — метаданные сеанса (id, source, user_id, model, title, timestamps, token counts). У заголовков есть уникальный индекс (NULL titles разрешены, уникальными должны быть только не-NULL).
- **messages** — полная история сообщений (role, content, tool_calls, tool_name, token_count)
- **messages_fts** — виртуальная таблица FTS5 для полнотекстового поиска по содержимому сообщений

## Истечение срока и очистка сеансов

### Автоматическая очистка

- Сеансы шлюза автоматически сбрасываются по выбранной политике
- Перед сбросом агент сохраняет память и навыки из завершающегося сеанса
- Опциональная автоочистка: когда `sessions.auto_prune` равен `true`, завершённые сеансы старше `sessions.retention_days` (по умолчанию 90) удаляются при старте CLI/шлюза
- После очистки, которая действительно удалила строки, `state.db` проходит через `VACUUM`, чтобы вернуть место на диске (SQLite не уменьшает файл после обычного `DELETE`)
- Очистка запускается не чаще одного раза в `sessions.min_interval_hours` (по умолчанию 24); время последнего запуска хранится прямо в `state.db`, так что оно общее для всех процессов Hermes внутри одного `HERMES_HOME`

По умолчанию эта функция **выключена** — история сеансов ценна для поиска через `session_search`, а тихое удаление могло бы удивить пользователей. Включите её в `~/.hermes/config.yaml`:

```yaml
sessions:
  auto_prune: true          # включите вручную — по умолчанию false
  retention_days: 90        # хранить завершённые сеансы столько дней
  vacuum_after_prune: true  # освобождать место на диске после очистки
  min_interval_hours: 24    # не запускать очистку чаще этого интервала
```

Активные сеансы никогда не очищаются автоматически, независимо от возраста.

### Ручная очистка

```bash
# Очистить сеансы старше 90 дней
hermes sessions prune

# Удалить конкретный сеанс
hermes sessions delete <session_id>

# Экспортировать перед очисткой (резервная копия)
hermes sessions export backup.jsonl
hermes sessions prune --older-than 30 --yes
```

:::tip
База растёт медленно (обычно 10–15 МБ на сотни сеансов), а история сеансов полезна для поиска через `session_search`, поэтому автоочистка по умолчанию выключена. Включайте её, если у вас высокая нагрузка шлюза/cron и `state.db` заметно начинает влиять на производительность. Например, наблюдался случай, когда база `state.db` размером 384 МБ и примерно 1000 сеансов замедляла вставки в FTS5 и список `/resume`. Для разовой очистки используйте `hermes sessions prune`, не включая автоматическую очистку.
:::
