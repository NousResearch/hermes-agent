# Память Honcho для Hermes

Honcho обеспечивает ИИ-ориентированное межсеансовое моделирование пользователя. Он учится понимать пользователя по ходу разговоров и даёт каждому профилю Hermes собственную идентичность участника, сохраняя при этом единое представление о пользователе.

## Когда использовать

- при настройке Honcho (облачного или самохостируемого);
- при устранении проблем с памятью или синхронизацией участников;
- при создании мультипрофильных конфигураций, где у каждого агента есть собственный участник Honcho;
- при настройке глубины наблюдения, воспоминаний, диалектического рассуждения или частоты записи;
- когда нужно понять, что делают 5 инструментов Honcho и в каких случаях их использовать;
- при настройке бюджетов контекста и внедрения сводок сеансов.

## Архитектура

### Внедрение базового контекста

Когда Honcho внедряет контекст в системную инструкцию (в режимах `hybrid` или `context`), он собирает блок базового контекста в следующем порядке:

1. **Сводка сеанса** — краткий срез текущего сеанса (ставится первой, чтобы модель сразу получила непрерывность диалога)
2. **Представление пользователя** — накопленная модель пользователя у Honcho (предпочтения, факты, паттерны)
3. **Карточка ИИ-участника** — карточка идентичности для ИИ-участника этого профиля Hermes

Сводка сеанса формируется автоматически Honcho в начале каждого хода, если предыдущий сеанс уже существует. Она даёт модели тёплый старт, не заставляя её заново проигрывать всю историю.

### Выбор холодного/тёплого промпта

Honcho автоматически выбирает между двумя стратегиями промпта:

| Условие | Стратегия | Что происходит |
|-----------|--------------|--------------|
| Нет предыдущего сеанса или представление пустое | **Холодный старт** | Лёгкий вводный промпт; сводка не внедряется; модель поощряется к изучению пользователя |
| Есть представление и/или история сеанса | **Тёплый старт** | Полное внедрение базового контекста (сводка → представление → карточка); более насыщенная системная инструкция |

Это не требует ручной настройки — выбор происходит автоматически в зависимости от состояния сеанса.

### Участники

Honcho моделирует разговор как взаимодействие между **участниками**. Hermes создаёт двух участников на каждый сеанс:

- **Участник пользователя** (`peerName`): представляет человека. Honcho строит модель пользователя на основе наблюдаемых сообщений.
- **ИИ-участник** (`aiPeer`): представляет текущий экземпляр Hermes. У каждого профиля есть свой собственный ИИ-участник, поэтому агенты формируют независимые представления.

### Наблюдение

У каждого участника есть два переключателя наблюдения, которые определяют, из чего Honcho делает выводы:

| Переключатель | Что делает |
|--------|-------------|
| `observeMe` | Наблюдаются собственные сообщения участника (строится самопредставление) |
| `observeOthers` | Наблюдаются сообщения других участников (строится межучастниковое понимание) |

По умолчанию все четыре переключателя **включены** (полное двунаправленное наблюдение).

Настройте это для каждого участника в `honcho.json`:

```json
{
  "observation": {
    "user": { "observeMe": true, "observeOthers": true },
    "ai":   { "observeMe": true, "observeOthers": true }
  }
}
```

Или используйте сокращённые пресеты:

| Пресет | Пользователь | ИИ | Сценарий использования |
|--------|------|----|----------------|
| `"directional"` (по умолчанию) | me:on, others:on | me:on, others:on | Многоагентный сценарий, полная память |
| `"unified"` | me:on, others:off | me:off, others:on | Один агент, моделирование только пользователя |

Настройки, изменённые в [панели управления Honcho](https://app.honcho.dev), синхронизируются при инициализации сеанса — серверная конфигурация имеет приоритет над локальными значениями по умолчанию.

### Сеансы

Сеансы Honcho определяют, куда попадают сообщения и наблюдения. Доступны такие стратегии:

| Стратегия | Поведение |
|----------|-------------|
| `per-directory` (по умолчанию) | Один сеанс на рабочий каталог |
| `per-repo` | Один сеанс на корень git-репозитория |
| `per-session` | Новый сеанс Honcho при каждом запуске Hermes |
| `global` | Один общий сеанс для всех каталогов |

Ручное переопределение: `hermes honcho map my-project-name`

### Режимы воспоминаний

Как агент получает доступ к памяти Honcho:

| Режим | Автоматически внедряет контекст? | Доступны инструменты? | Сценарий использования |
|------|----------------------|-----------------|-------------|
| `hybrid` (по умолчанию) | Да | Да | Агент сам решает, когда использовать инструменты, а когда автоконтекст |
| `context` | Да | Нет (скрыты) | Минимальная стоимость токенов, без вызовов инструментов |
| `tools` | Нет | Да | Агент полностью управляет доступом к памяти вручную |

## Три независимых регулятора

Диалектическое поведение Honcho задаётся тремя независимыми параметрами. Каждый можно настраивать, не влияя на остальные:

### Частота (когда)

Определяет, **как часто** вызываются диалектические и контекстные вызовы.

| Ключ | Значение по умолчанию | Описание |
|-----|---------|-------------|
| `contextCadence` | `1` | Минимальное число ходов между вызовами API контекста |
| `dialecticCadence` | `2` | Минимальное число ходов между вызовами API диалектики. Рекомендуется 1–5 |
| `injectionFrequency` | `every-turn` | `every-turn` или `first-turn` для внедрения базового контекста |

Чем выше значение cadence, тем реже диалектическая LLM срабатывает. `dialecticCadence: 2` означает, что движок запускается через каждый второй ход. Если поставить `1`, он будет срабатывать на каждом ходе.

### Глубина (сколько)

Определяет, **сколько раундов** диалектического рассуждения Honcho выполняет на каждый запрос.

| Ключ | Значение по умолчанию | Диапазон | Описание |
|-----|---------|-------|-------------|
| `dialecticDepth` | `1` | 1-3 | Число раундов диалектического рассуждения на запрос |
| `dialecticDepthLevels` | -- | array | Необязательные переопределения уровней для каждого раунда (см. ниже) |

`dialecticDepth: 2` означает, что Honcho запускает два раунда диалектического синтеза. Первый раунд формирует первичный ответ, второй его уточняет.

`dialecticDepthLevels` позволяет задать уровень рассуждения для каждого раунда отдельно:

```json
{
  "dialecticDepth": 3,
  "dialecticDepthLevels": ["low", "medium", "high"]
}
```

Если `dialecticDepthLevels` опущен, раунды используют **пропорциональные уровни**, выведенные из `dialecticReasoningLevel` (базового значения):

| Глубина | Уровни проходов |
|-------|-------------|
| 1 | [base] |
| 2 | [minimal, base] |
| 3 | [minimal, base, low] |

Так ранние проходы остаются дешёвыми, а полная глубина используется только на финальном синтезе.

**Глубина на старте сеанса.** Предпрогрев при старте сеанса выполняет полный заданный `dialecticDepth` в фоне ещё до первого хода. Однопроходный предпрогрев на холодном участнике часто даёт слабый результат — многошаговая глубина прогоняет цикл аудита/согласования ещё до того, как пользователь что-то скажет. Первый ход напрямую использует результат предпрогрева; если он не успел завершиться, первый ход переходит к синхронному вызову с ограниченным таймаутом.

### Уровень (насколько глубоко)

Определяет **интенсивность** каждого раунда диалектического рассуждения.

| Ключ | Значение по умолчанию | Описание |
|-----|---------|-------------|
| `dialecticReasoningLevel` | `low` | `minimal`, `low`, `medium`, `high`, `max` |
| `dialecticDynamic` | `true` | Если `true`, модель может передать `reasoning_level` в `honcho_reasoning`, чтобы переопределить значение по умолчанию для конкретного вызова. `false` = всегда использовать `dialecticReasoningLevel`, переопределения от модели игнорируются |

Более высокие уровни дают более богатый синтез, но обходятся дороже по токенам на бэкенде Honcho.

## Мультипрофильная настройка

Каждый профиль Hermes получает своего ИИ-участника Honcho, при этом рабочая область (пользовательский контекст) остаётся общей. Это означает:

- все профили видят одно и то же представление пользователя;
- каждый профиль строит собственную ИИ-идентичность и собственные наблюдения;
- выводы, записанные одним профилем, видны другим через общую рабочую область.

### Создать профиль с участником Honcho

```bash
hermes profile create coder --clone
# создаёт блок хоста hermes.coder, ИИ-участника "coder", наследует конфигурацию из default
```

Что делает `--clone` для Honcho:
1. Создаёт блок хоста `hermes.coder` в `honcho.json`
2. Устанавливает `aiPeer: "coder"` (имя профиля)
3. Наследует `workspace`, `peerName`, `writeFrequency`, `recallMode` и т. д. от `default`
4. Заранее создаёт участника в Honcho, чтобы он существовал ещё до первого сообщения

### Дозаполнение существующих профилей

```bash
hermes honcho sync    # создаёт блок хоста для всех профилей, у которых его ещё нет
```

### Конфигурация на уровне профиля

Переопределяйте любые параметры в блоке хоста:

```json
{
  "hosts": {
    "hermes.coder": {
      "aiPeer": "coder",
      "recallMode": "tools",
      "dialecticDepth": 2,
      "observation": {
        "user": { "observeMe": true, "observeOthers": false },
        "ai": { "observeMe": true, "observeOthers": true }
      }
    }
  }
}
```

## Инструменты

У агента есть 5 двунаправленных инструментов Honcho (скрыты в режиме `context` recall mode):

| Инструмент | LLM-вызов? | Стоимость | Когда использовать |
|------|-----------|------|----------|
| `honcho_profile` | Нет | минимальная | Быстрый фактологический снимок в начале разговора или для быстрого поиска имени/роли/предпочтений |
| `honcho_search` | Нет | низкая | Получить конкретные прошлые факты, чтобы рассуждать самостоятельно — сырые фрагменты без синтеза |
| `honcho_context` | Нет | низкая | Полный снимок контекста сеанса: сводка, представление, карточка, недавние сообщения |
| `honcho_reasoning` | Да | средняя–высокая | Вопрос на естественном языке, синтезированный диалектическим движком Honcho |
| `honcho_conclude` | Нет | минимальная | Записать или удалить постоянный факт; для самопознания ИИ-участника передавайте `peer: "ai"` |

### `honcho_profile`
Прочитать или обновить карточку участника — отобранные ключевые факты (имя, роль, предпочтения, стиль общения). Передайте `card: [...]`, чтобы обновить; не передавайте ничего, чтобы прочитать. Без LLM-вызова.

### `honcho_search`
Семантический поиск по сохранённому контексту для конкретного участника. Возвращает сырые фрагменты, отсортированные по релевантности, без синтеза. По умолчанию 800 токенов, максимум 2000. Подходит, когда нужны конкретные прошлые факты, а не синтезированный ответ.

### `honcho_context`
Полный снимок контекста сеанса из Honcho — сводка сеанса, представление участника, карточка участника и недавние сообщения. Без LLM-вызова. Используйте, когда нужно увидеть всё, что Honcho знает о текущем сеансе и участнике, одним запросом.

### `honcho_reasoning`
Вопрос на естественном языке, на который отвечает диалектический движок Honcho (LLM-вызов на бэкенде Honcho). Более высокая стоимость, более высокое качество. Передайте `reasoning_level`, чтобы управлять глубиной: `minimal` (быстро/дёшево) → `low` → `medium` → `high` → `max` (максимально подробно). Если не передавать значение, используется настроенный уровень по умолчанию (`low`). Используйте для синтезированного понимания паттернов пользователя, его целей или текущего состояния.

### `honcho_conclude`
Записать или удалить постоянный вывод о пользователе. Передайте `conclusion: "..."`, чтобы создать его. Передайте `delete_id: "..."`, чтобы удалить вывод. Для удаления персональных данных Honcho со временем сам корректирует ошибочные выводы, поэтому удаление нужно только там, где это действительно необходимо.

### Двунаправленное таргетирование участников

Все 5 инструментов принимают необязательный параметр `peer`:
- `peer: "user"` (по умолчанию) — работает с участником пользователя
- `peer: "ai"` — работает с ИИ-участником этого профиля
- `peer: "<explicit-id>"` — любой ID участника в workspace

Примеры:
```
honcho_profile                        # читает карточку пользователя
honcho_profile peer="ai"              # читает карточку ИИ-участника
honcho_reasoning query="Что для этого пользователя важнее всего?"
honcho_reasoning query="Каковы мои типичные паттерны взаимодействия?" peer="ai" reasoning_level="medium"
honcho_conclude conclusion="Предпочитает краткие ответы"
honcho_conclude conclusion="Я склонен чересчур подробно объяснять код" peer="ai"
honcho_conclude delete_id="abc123"    # удаление персональных данных
```

## Паттерны использования агентом

Рекомендации для Hermes, когда память Honcho активна.

### В начале разговора

```
1. honcho_profile                  → быстрый прогрев, без затрат на LLM
2. Если контекста мало → honcho_context  (полный снимок, всё ещё без LLM)
3. Если нужен глубокий синтез → `honcho_reasoning` (LLM-вызов, используйте экономно)
```

Не вызывайте `honcho_reasoning` на каждом ходе. Автовнедрение уже занимается регулярным обновлением контекста. Используйте инструмент рассуждения только тогда, когда вам действительно нужен синтез, которого нет в базовом контексте.

### Когда пользователь просит что-то запомнить

```
honcho_conclude conclusion="<specific, actionable fact>"
```

Хорошие выводы: "Предпочитает примеры кода, а не развёрнутые текстовые объяснения", "Работает над асинхронным Rust-проектом до апреля 2026 года"
Плохие выводы: "Пользователь что-то говорил про Rust" (слишком расплывчато), "Пользователь технически подкован" (это уже есть в представлении)

### Когда пользователь спрашивает о прошлом контексте / нужны точные детали

```
honcho_search query="<topic>"       → fast, no LLM, good for specific facts
honcho_context                       → full snapshot with summary + messages
honcho_reasoning query="<question>"  → synthesized answer, use when search isn't enough
```

### Когда использовать `peer: "ai"`

Используйте таргетирование ИИ-участника, чтобы строить и проверять собственные знания агента:
- `honcho_conclude conclusion="Я склонен быть многословным, когда объясняю архитектуру" peer="ai"` — самокоррекция
- `honcho_reasoning query="Как я обычно работаю с неоднозначными запросами?" peer="ai"` — самоаудит
- `honcho_profile peer="ai"` — просмотр собственной карточки идентичности

### Когда НЕ вызывать инструменты

В режимах `hybrid` и `context` базовый контекст (представление пользователя + карточка + сводка сеанса) внедряется автоматически перед каждым ходом. Не запрашивайте повторно то, что уже было внедрено. Вызывайте инструменты только когда:
- вам нужно что-то, чего нет во внедрённом контексте;
- пользователь прямо просит вспомнить что-то или проверить память;
- вы записываете новый вывод.

### Осознание cadence

`honcho_reasoning` на уровне инструментов использует ту же стоимость, что и диалектическое авто-внедрение. После явного вызова инструмента cadence авто-внедрения сбрасывается — это помогает не платить дважды за один и тот же ход.

## Справочник конфигурации

Файл конфигурации: `$HERMES_HOME/honcho.json` (локальный для профиля) или `~/.honcho/config.json` (глобальный).

### Основные настройки

| Ключ | Значение по умолчанию | Описание |
|-----|---------|-------------|
| `apiKey` | -- | API-ключ ([получить](https://app.honcho.dev)) |
| `baseUrl` | -- | Базовый URL для self-hosted Honcho |
| `peerName` | -- | Идентичность пользовательского участника |
| `aiPeer` | ключ хоста | Идентичность ИИ-участника |
| `workspace` | ключ хоста | Общий ID workspace |
| `recallMode` | `hybrid` | `hybrid`, `context` или `tools` |
| `observation` | all on | Переключатели `observeMe`/`observeOthers` для каждого участника |
| `writeFrequency` | `async` | `async`, `turn`, `session` или целое N |
| `sessionStrategy` | `per-directory` | `per-directory`, `per-repo`, `per-session`, `global` |
| `messageMaxChars` | `25000` | Максимум символов на сообщение (при превышении фрагментируется) |

### Настройки диалектики

| Ключ | Значение по умолчанию | Описание |
|-----|---------|-------------|
| `dialecticReasoningLevel` | `low` | `minimal`, `low`, `medium`, `high`, `max` |
| `dialecticDynamic` | `true` | Автоматически повышает уровень рассуждения в зависимости от сложности запроса. `false` = фиксированный уровень |
| `dialecticDepth` | `1` | Число раундов диалектики на запрос (1-3) |
| `dialecticDepthLevels` | -- | Необязательный массив уровней для каждого раунда, например `["low", "high"]` |
| `dialecticMaxInputChars` | `10000` | Максимум символов для входных данных диалектического запроса |

### Бюджет контекста и внедрение

| Ключ | Значение по умолчанию | Описание |
|-----|---------|-------------|
| `contextTokens` | uncapped | Максимум токенов для объединённого внедряемого базового контекста (сводка + представление + карточка). Необязательный предел — не указывайте, чтобы оставить без лимита; задайте целое число, чтобы ограничить размер внедрения. |
| `injectionFrequency` | `every-turn` | `every-turn` или `first-turn` |
| `contextCadence` | `1` | Минимальное число ходов между вызовами API контекста |
| `dialecticCadence` | `2` | Минимальное число ходов между вызовами диалектической LLM (рекомендуется 1–5) |

Бюджет `contextTokens` проверяется в момент внедрения. Если сводка сеанса + представление + карточка превышают бюджет, Honcho сначала сокращает сводку, затем представление, сохраняя карточку. Это предотвращает раздувание контекста в длинных сеансах.

### Санитизация memory-context

Honcho санитизирует блок `memory-context` перед внедрением, чтобы предотвратить prompt-инъекцию и повреждённый контент:

- Удаляет XML/HTML-теги из пользовательских выводов
- Нормализует пробелы и управляющие символы
- Обрезает отдельные выводы, превышающие `messageMaxChars`
- Экранирует последовательности разделителей, которые могут сломать структуру системной инструкции

Это исправление закрывает крайние случаи, когда сырые пользовательские выводы с разметкой или спецсимволами могут повредить внедряемый блок контекста.

## Устранение неполадок

### "Honcho не настроен"
Запустите `hermes honcho setup`. Убедитесь, что `memory.provider: honcho` указан в `~/.hermes/config.yaml`.

### Память не сохраняется между сеансами
Проверьте `hermes honcho status` -- убедитесь, что `saveMessages: true`, а `writeFrequency` не равен `session` (при таком значении запись происходит только при выходе).

### У профиля не создаётся собственный участник
Используйте `--clone` при создании: `hermes profile create <name> --clone`. Для уже существующих профилей используйте `hermes honcho sync`.

### Изменения наблюдения в панели не отражаются
Конфигурация наблюдения синхронизируется с сервера при каждой инициализации сеанса. После изменения настроек в интерфейсе Honcho начните новый сеанс.

### Сообщения обрезаются
Сообщения длиннее `messageMaxChars` (по умолчанию 25k) автоматически фрагментируются с маркерами `[continued]`. Если это происходит часто, проверьте, не раздувают ли сообщение результаты инструментов или содержимое навыков.

### Бюджет контекста слишком велик
Если видите предупреждения о превышении бюджета контекста, уменьшите `contextTokens` или сократите `dialecticDepth`. При жёстком бюджете первым сокращается именно сводка сеанса.

### Сводка сеанса отсутствует
Для сводки сеанса нужен хотя бы один предыдущий ход в текущем сеансе Honcho. На холодном старте (новый сеанс, без истории) сводка не вставляется, и Honcho использует стратегию холодного старта.
