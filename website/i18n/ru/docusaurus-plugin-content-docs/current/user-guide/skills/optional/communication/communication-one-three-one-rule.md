# Правило общения 1-3-1

Структурированный формат принятия решений, когда у задачи есть несколько жизнеспособных подходов и пользователю нужна ясная рекомендация. Навык выдаёт компактную постановку проблемы, три варианта с компромиссами и практичный план для рекомендуемого пути.

## Когда использовать

- Пользователь прямо просит ответ в формате "1-3-1".
- Пользователь говорит "дай варианты" или "какие у меня есть пути" в техническом контексте.
- У задачи есть несколько жизнеспособных подходов с заметными компромиссами (архитектура, выбор инструментов, стратегия миграции).
- Пользователю нужен текст, который можно переслать команде или стейкхолдерам.

Не используйте его для простых вопросов с очевидным ответом, отладки или задач, где пользователь уже выбрал подход.

## Процедура

1. **Проблема** — одно предложение.
   - Сформулируйте ключевое решение или желаемый результат одним коротким предложением.
   - Фокусируйтесь на *что*, а не на *как* — без деталей реализации, названий инструментов и конкретных технологий.
   - Держите формулировку сжатой. Если вам приходится ставить "и", скорее всего, вы описываете уже две проблемы.

2. **Варианты** — ровно три.
   - Представьте три различающихся жизнеспособных подхода с метками A, B и C.
   - У каждого варианта должно быть короткое описание, плюсы и минусы.
   - Варианты должны быть действительно разными стратегиями, а не незначительными вариациями одного и того же решения.

3. **Рекомендация** — один вариант.
   - Ясно укажите, какой вариант вы рекомендуете и почему, исходя из контекста и приоритетов пользователя.
   - Отвечайте прямо: это ваш профессиональный вывод, а не уклончивое мнение.

4. **Определение готовности**
   - Перечислите конкретные критерии успеха для рекомендованного варианта.
   - Это должны быть проверяемые результаты, а не расплывчатые пожелания.
   - Если пользователь выберет другой вариант, обновите этот раздел под него.

5. **План внедрения**
   - Конкретные шаги для реализации рекомендованного варианта.
   - При необходимости указывайте точные команды, инструменты и действия.
   - Если пользователь выберет другой вариант, обновите этот раздел под него.

## Проверка

- В ответе ровно одно предложение в разделе "Проблема".
- В ответе ровно три варианта (A, B, C) с плюсами и минусами для каждого.
- В ответе есть одна рекомендация, которая выбирает один вариант и объясняет почему.
- Разделы "Определение готовности" и "План внедрения" соответствуют рекомендованному варианту.
- Если пользователь выбирает другой вариант, обновите рекомендацию, критерии готовности и план.

## Пример

Пользователь: "Сделай мне 1-3-1 для добавления логики повторных попыток в наш API-клиент."

Агент:

**Проблема:** API-запросы к внешним сервисам периодически падают под нагрузкой и нуждаются в корректной повторной отправке.

**Варианты:**

- **Вариант A: экспоненциальная задержка с jitter в общей утилите.**
  Плюсы: простой и понятный паттерн; одна реализация для всех вызовов.
  Минусы: нет circuit breaker; вызывающий код всё равно ждёт повторов.

- **Вариант B: библиотека повторных попыток (tenacity) как декоратор для каждого вызова endpoint.**
  Плюсы: настраивается точечно; есть встроенная поддержка circuit breaker и rate limits.
  Минусы: добавляет зависимость; декоратор может скрывать поток управления.

- **Вариант C: асинхронные повторы через очередь с dead letter handling.**
  Плюсы: не блокирует поток; ошибки отслеживаются и можно повторно проигрывать задания; хорошо масштабируется.
  Минусы: заметно усложняет архитектуру; требует очередь; для малой нагрузки это избыточно.

**Рекомендация:** Вариант A. Объём вызовов не оправдывает инфраструктуру очередей, а общая утилита с экспоненциальной задержкой и jitter закрывает этот сценарий без новых зависимостей.

**Определение готовности:**
- Утилита `api_retry()` обрабатывает временные HTTP-ошибки (429, 502, 503, 504).
- Повторы выполняются не более 3 раз с экспоненциальной задержкой и jitter.
- Все внешние API-вызовы проходят через эту утилиту.
- Каждая попытка повторной отправки логируется с указанием endpoint и номера попытки.
- Тесты покрывают исчерпание повторов, мгновенный успех и сценарий "ошибка, затем успех".

**План внедрения:**
1. Создайте `utils/api_retry.py` с настраиваемым максимумом повторов, базовой задержкой и списком повторяемых кодов ответа.
2. Добавьте jitter через `random.uniform(0, base_delay)`, чтобы избежать эффекта "стада".
3. Оберните существующие API-вызовы в `api_client.py` через эту утилиту.
4. Добавьте модульные тесты с моками HTTP-ответов для каждого сценария повторов.
5. Проверьте поведение под нагрузкой простым стресс-тестом против имитатора нестабильного endpoint.
