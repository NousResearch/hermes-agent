# Субагентно-ориентированная разработка

## Обзор

Выполняйте планы внедрения следующим образом: на каждую задачу запускайте нового субагента и проводите двухэтапную проверку.

**Основной принцип:** новый субагент на каждую задачу плюс двухэтапная проверка (сначала спецификация, потом качество) обеспечивают высокое качество и быструю итерацию.

## Когда использовать

Используйте этот навык, когда:
- У вас есть план внедрения (из навыка `plan` или из требований пользователя)
- Задачи в основном независимы
- Важны качество и соответствие спецификации
- Нужна автоматическая проверка между задачами

**По сравнению с ручной работой:**
- Свежий контекст для каждой задачи (никакой путаницы из накопленного состояния)
- Автоматическая проверка ловит проблемы раньше
- Единые проверки качества для всех задач
- Субагенты могут задавать вопросы до начала работы

## Процесс

### 1. Прочитайте и разберите план

Прочитайте файл плана. Извлеките ВСЕ задачи вместе с полным текстом и контекстом заранее. Создайте список todo:

```python
# Прочитать план
read_file("docs/plans/feature-plan.md")

# Создать список todo со всеми задачами
todo([
    {"id": "task-1", "content": "Создать модель User с полем email", "status": "pending"},
    {"id": "task-2", "content": "Добавить утилиту хеширования паролей", "status": "pending"},
    {"id": "task-3", "content": "Создать endpoint входа", "status": "pending"},
])
```

**Важно:** прочитайте план ОДИН РАЗ. Извлеките всё. Не заставляйте субагентов читать файл плана — передавайте им полный текст задачи напрямую в контексте.

### 2. Рабочий процесс для каждой задачи

Для КАЖДОЙ задачи из плана:

#### Шаг 1: Запустите субагента-исполнителя

Используйте `delegate_task` с полным контекстом:

```python
delegate_task(
    goal="Реализовать задачу 1: создать модель User с полями email и password_hash",
    context="""
    ЗАДАЧА ИЗ ПЛАНА:
    - Создать: src/models/user.py
    - Добавить класс User с полями email (str) и password_hash (str)
    - Использовать bcrypt для хеширования паролей
    - Добавить __repr__ для отладки

    СЛЕДУЙТЕ TDD:
    1. Написать падающий тест в tests/models/test_user.py
    2. Запустить: pytest tests/models/test_user.py -v (проверить FAIL)
    3. Написать минимальную реализацию
    4. Запустить: pytest tests/models/test_user.py -v (проверить PASS)
    5. Запустить: pytest tests/ -q (проверить, что регрессий нет)
    6. Коммит: git add -A && git commit -m "feat: add User model with password hashing"

    КОНТЕКСТ ПРОЕКТА:
    - Python 3.11, Flask-приложение в src/app.py
    - Существующие модели находятся в src/models/
    - Тесты используют pytest, запускать из корня проекта
    - bcrypt уже есть в requirements.txt
    """,
    toolsets=['terminal', 'file']
)
```

#### Шаг 2: Запустите проверяющего на соответствие спецификации

После завершения работы исполнителя проверьте соответствие исходной спецификации:

```python
delegate_task(
    goal="Проверить, что реализация соответствует спецификации из плана",
    context="""
    ИСХОДНАЯ СПЕЦИФИКАЦИЯ ЗАДАЧИ:
    - Создать src/models/user.py с классом User
    - Поля: email (str), password_hash (str)
    - Использовать bcrypt для хеширования паролей
    - Добавить __repr__

    ПРОВЕРЬТЕ:
    - [ ] Реализованы все требования из спецификации?
    - [ ] Пути к файлам соответствуют спецификации?
    - [ ] Сигнатуры функций соответствуют спецификации?
    - [ ] Поведение соответствует ожидаемому?
    - [ ] Ничего лишнего не добавлено (нет расползания scope)?

    ВЫВОД: PASS или список конкретных пробелов в спецификации, которые нужно исправить.
    """,
    toolsets=['file']
)
```

**Если найдены проблемы по спецификации:** исправьте пробелы и снова запустите проверку. Продолжайте только после соответствия спецификации.

#### Шаг 3: Запустите проверяющего на качество кода

После прохождения проверки спецификации:

```python
delegate_task(
    goal="Проверить качество кода для реализации задачи 1",
    context="""
    ФАЙЛЫ ДЛЯ ПРОВЕРКИ:
    - src/models/user.py
    - tests/models/test_user.py

    ПРОВЕРЬТЕ:
    - [ ] Код следует соглашениям и стилю проекта?
    - [ ] Есть корректная обработка ошибок?
    - [ ] Имена переменных и функций понятны?
    - [ ] Достаточно ли покрытия тестами?
    - [ ] Нет очевидных багов или упущенных краевых случаев?
    - [ ] Нет проблем безопасности?

    ФОРМАТ ОТВЕТА:
    - Критические проблемы: [обязательно исправить до продолжения]
    - Важные проблемы: [желательно исправить]
    - Небольшие замечания: [по желанию]
    - Вердикт: APPROVED or REQUEST_CHANGES
    """,
    toolsets=['file']
)
```

**Если обнаружены проблемы качества:** исправьте их и повторите проверку. Продолжайте только после одобрения.

#### Шаг 4: Пометьте задачу выполненной

```python
todo([{"id": "task-1", "content": "Создать модель User с полем email", "status": "completed"}], merge=True)
```

### 3. Финальная проверка

После выполнения ВСЕХ задач запустите финального проверяющего на интеграцию:

```python
delegate_task(
    goal="Проверить всю реализацию на согласованность и проблемы интеграции",
    context="""
    Все задачи из плана завершены. Проверьте полную реализацию:
    - Все ли компоненты работают вместе?
    - Есть ли несоответствия между задачами?
    - Все ли тесты проходят?
    - Готово ли к слиянию?
    """,
    toolsets=['terminal', 'file']
)
```

### 4. Проверка и коммит

```bash
# Запустить полный набор тестов
pytest tests/ -q

# Просмотреть все изменения
git diff --stat

# Финальный коммит, если нужен
git add -A && git commit -m "feat: complete [feature name] implementation"
```

## Размер задач

**Каждая задача = 2-5 минут сосредоточенной работы.**

**Слишком крупно:**
- "Реализовать систему аутентификации пользователей"

**Правильный размер:**
- "Создать модель User с полями email и password"
- "Добавить функцию хеширования пароля"
- "Создать endpoint входа"
- "Добавить генерацию JWT-токена"
- "Создать endpoint регистрации"

## Красные флаги — никогда так не делайте

- Начинать реализацию без плана
- Пропускать проверки (соответствие спецификации ИЛИ качество кода)
- Продолжать, если остались неустранённые критические или важные проблемы
- Запускать несколько субагентов-исполнителей для задач, которые затрагивают одни и те же файлы
- Заставлять субагента читать файл плана (вместо этого передавайте полный текст в контексте)
- Пропускать вводный контекст (субагент должен понимать, где находится задача в общей картине)
- Игнорировать вопросы субагента
- Принимать вариант "и так сойдёт" на проверке спецификации
- Пропускать цикл повторной проверки (проверяющий нашёл проблемы → исполнитель исправил → проверка повторяется)
- Подменять реальную проверку самопроверкой исполнителя (нужны обе)
- **Начинать проверку качества кода до того, как спецификация получила PASS** (неверный порядок)
- Переходить к следующей задаче, пока у любой из проверок есть открытые проблемы

## Работа с проблемами

### Если субагент задаёт вопросы

- Отвечайте чётко и полностью
- При необходимости добавляйте контекст
- Не подгоняйте его слишком быстро к реализации

### Если проверяющий находит проблемы

- Исполнитель-субагент (или новый субагент) исправляет их
- Проверяющий запускается снова
- Повторяйте, пока не будет одобрено
- Не пропускайте повторную проверку

### Если субагент не справился с задачей

- Запустите нового субагента на исправление с конкретным описанием ошибки
- Не пытайтесь исправлять вручную в управляющем сеансе (это загрязняет контекст)

## Примечания по эффективности

**Почему нужен новый субагент на каждую задачу:**
- Не возникает загрязнения контекста накопленным состоянием
- Каждый субагент получает чистый, сфокусированный контекст
- Нет путаницы из-за предыдущего кода или рассуждений

**Почему нужна двухэтапная проверка:**
- Проверка спецификации рано ловит недореализацию и переразвитие
- Проверка качества гарантирует, что реализация добротная
- Проблемы обнаруживаются до того, как успеют накопиться между задачами

**Компромисс по стоимости:**
- Больше вызовов субагентов (исполнитель + 2 проверяющих на каждую задачу)
- Но ошибки ловятся раньше, а это дешевле, чем потом разбирать накопившиеся проблемы

## Интеграция с другими навыками

### С `plan`

Этот навык ВЫПОЛНЯЕТ планы, созданные навыком `plan`:
1. Требования пользователя → `plan` → план внедрения
2. План внедрения → `subagent-driven-development` → рабочий код

### С `test-driven-development`

Субагенты-исполнители должны следовать TDD:
1. Сначала написать падающий тест
2. Реализовать минимальный код
3. Проверить, что тест проходит
4. Закоммитить

Добавляйте инструкции по TDD в контекст каждого субагента-исполнителя.

### С `requesting-code-review`

Двухэтапная проверка — это и есть code review. Для финальной интеграционной проверки используйте критерии проверки из навыка `requesting-code-review`.

### С `systematic-debugging`

Если субагент сталкивается с багом во время реализации:
1. Следуйте процессу `systematic-debugging`
2. Найдите первопричину до исправления
3. Напишите регрессионный тест
4. Возобновите выполнение

[Запущен субагент-исполнитель]

[Исполнитель исправляет проблему]

[Запущен проверяющий на соответствие спецификации снова]
  Проверка спецификации: ✅ PASS

[Запущен проверяющий качества]
  Проверка качества: Важно: магическое число 8, вынесите его в константу
  Исполнитель: значение MIN_PASSWORD_LENGTH вынесено в константу
  Проверка качества: ✅ APPROVED

[Задача 2 помечена выполненной]

... (продолжайте для всех задач)

[После всех задач: запустить финального проверяющего на интеграцию]
[Запустить полный набор тестов: всё проходит]

## Запомните

```
Свежий субагент на каждую задачу
Двухэтапная проверка каждый раз
Сначала соответствие спецификации
Потом качество кода
Не пропускать проверки
Ловить проблемы раньше
```

**Качество не возникает случайно. Это результат системной работы.**
