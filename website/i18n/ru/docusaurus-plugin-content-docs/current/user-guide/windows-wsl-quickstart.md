# Руководство по Windows (WSL2)

Hermes Agent теперь поддерживает **и нативную Windows, и WSL2**. Эта страница описывает именно путь через WSL2; для нативной установки через PowerShell откройте отдельное **[руководство по нативной установке Windows](/user-guide/windows-native)**.

**Когда стоит выбрать WSL2 вместо нативной установки:**
- вам нужна встроенная терминальная вкладка веб-панели (`/chat`) — эта панель требует POSIX PTY и доступна только в WSL2
- вы ведёте POSIX-ориентированную разработку и хотите, чтобы сеансы Hermes разделяли ту же файловую систему и те же пути, что и ваши инструменты разработки
- у вас уже есть WSL2, и вам не хочется поддерживать ещё одну отдельную установку

**Когда нативный Windows вполне подходит (или даже лучше):**
- для интерактивного чата, шлюза (Telegram/Discord и т. п.), планировщика cron, браузерного инструмента, MCP-серверов и большинства возможностей Hermes нативная Windows тоже работает нормально
- вам не хочется каждый раз думать, через какую сторону границы WSL↔Windows вы сейчас ходите

В WSL2 по сути есть две изолированные среды: ваш хост Windows и Linux VM под управлением WSL. Большая часть путаницы возникает тогда, когда вы перестаёте понимать, на какой именно стороне находитесь в конкретный момент.

Этот гид разбирает именно те части этой границы, которые важны для Hermes: установка WSL2, перенос файлов между Windows и Linux, сеть в обе стороны и реальные проблемы, на которые люди натыкаются чаще всего.

## Зачем WSL2, а не нативный Windows

Нативная установка Windows работает прямо в Windows: ваш Windows-терминал (PowerShell, Windows Terminal и т. п.), пути файловой системы Windows (`C:\Users\…`) и процессы Windows. Hermes использует Git Bash для выполнения команд оболочки; именно так сегодня работают Claude Code и похожие агенты на Windows — это позволяет обойти разрыв между POSIX и Windows без полной переработки.

WSL2 запускает полноценное Linux-ядро в лёгкой виртуальной машине, поэтому Hermes внутри него по сути ведёт себя так же, как на Ubuntu. Это ценно, когда вам нужна настоящая POSIX-среда: `fork`, `/tmp`, UNIX sockets, семантика сигналов, терминалы на базе PTY, `bash`/`zsh` и инструменты вроде `rg`, `git`, `ffmpeg`, которые работают именно так, как вы привыкли на Linux.

Практические следствия WSL2:

- Hermes CLI, шлюз, сеансы, память, навыки и рантаймы инструментов живут внутри Linux VM
- Приложения Windows (браузеры, нативные приложения, Chrome с вашим залогиненным профилем) живут снаружи
- Каждый раз, когда вам нужно, чтобы они взаимодействовали — поделиться файлами, открыть URL, управлять Chrome, достучаться до локального сервера моделей, дать телефону доступ к шлюзу Hermes — вы пересекаете границу. Именно о таких границах и идёт речь в этом руководстве

## Установите WSL2

В PowerShell от имени администратора или Windows Terminal выполните:

```powershell
wsl --install
```

На свежем Windows 10 22H2+ или Windows 11 это установит ядро WSL2, Virtual Machine Platform и Ubuntu по умолчанию. После установки перезагрузитесь. Затем Ubuntu спросит имя и пароль Linux-пользователя — это **новая Linux-учётная запись**, не связанная с вашей учётной записью Windows.

Проверьте, что у вас именно WSL2, а не устаревший WSL1:

```powershell
wsl --list --verbose
```

В списке должна быть `VERSION  2`. Если какая-то дистрибуция показывает `VERSION  1`, переведите её:

```powershell
wsl --set-version Ubuntu 2
wsl --set-default-version 2
```

Hermes не работает надёжно на WSL1 — там системные вызовы Linux транслируются на лету, и часть поведения (procfs, signals, network) заметно отличается от настоящего Linux.

### Выбор дистрибутива

Ubuntu LTS — та версия, на которой мы тестируем. Debian тоже подходит. Arch и NixOS работают у тех, кому они нужны, но однострочный установщик предполагает Debian-подобный `apt` — для Nix-пути смотрите [руководство по настройке Nix](/getting-started/nix-setup).

### Включите systemd (рекомендуется)

Шлюз Hermes и любые долгоживущие процессы удобнее запускать под systemd. В современных сборках WSL включите его один раз внутри дистрибутива:

```bash
sudo tee /etc/wsl.conf >/dev/null <<'EOF'
[boot]
systemd=true

[interop]
enabled=true
appendWindowsPath=true

[automount]
options = "metadata,umask=22,fmask=11"
EOF
```

Затем из PowerShell:

```powershell
wsl --shutdown
```

Снова откройте терминал WSL. Команда `ps -p 1 -o comm=` должна показать `systemd`.

Опция `metadata` выше важна: без неё файлы на `/mnt/c/...` не могут хранить настоящие биты прав доступа Linux, и из-за этого ломаются вещи вроде `chmod +x` для скриптов на путях Windows.

### Установите Hermes внутри WSL

Как только у вас открыт shell в WSL2:

```bash
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash
source ~/.bashrc
hermes
```

Установщик воспринимает WSL2 как обычный Linux — ничего специфичного для WSL делать не нужно. Полная схема установки описана в [Установке](/getting-started/installation).

## Файловая система: как пересекать границу Windows ↔ WSL2

Вот здесь чаще всего и случается путаница. У вас есть **две файловые системы**, и то, куда вы положите файлы, влияет на производительность, корректность и на то, какие инструменты вообще их увидят.

### Два направления

| Направление | Вид изнутри | Что использовать |
|---|---|---|
| Диск Windows, видимый из WSL | `C:\Users\you\Documents` | `/mnt/c/Users/you/Documents` |
| Диск WSL, видимый из Windows | `/home/you/code` | `\\wsl$\Ubuntu\home\you\code` (или `\\wsl.localhost\Ubuntu\...` на новых сборках) |

Оба варианта реальны и рабочие, но это **не одна и та же файловая система** — под капотом их связывает сетевой протокол 9P. У этого есть серьёзные последствия по скорости и семантике.

### Где хранить Hermes и проекты

**Практическое правило: всё Linux-подобное держите внутри файловой системы Linux.**

- Установка Hermes (`~/.hermes/`) — на стороне Linux. Установщик уже делает именно так.
- Git-репозитории, с которыми вы работаете из WSL, — на стороне Linux (`~/code/...`, `~/projects/...`).
- Модели, датасеты, venv — тоже на стороне Linux.

Что вы выигрываете, если следуете этому правилу:

- **Быстрый I/O.** Операции над `/mnt/c/...` идут через 9P и могут быть в 10–100 раз медленнее, чем на нативном ext4. `git status` на репозитории с 10k файлов, который под `~/code` кажется мгновенным, под `/mnt/c` легко превращается в 15+ секунд.
- **Корректные права.** Права доступа Linux на `/mnt/c` эмулируются приблизительно. Проблемы вроде `bad permissions` у SSH-ключа или незаметно не сработавшего `chmod +x` встречаются постоянно.
- **Надёжные наблюдатели за файлами.** inotify через 9P работает нестабильно — серверы разработки и тест-раннеры часто пропускают изменения на `/mnt/c`.
- **Без сюрпризов с чувствительностью к регистру.** Windows-пути обычно не чувствительны к регистру; Linux — чувствителен. Проекты с `Readme.md` и `README.md` ведут себя по-разному в зависимости от стороны.

Кладите всё на `/mnt/c` только если вам **обязательно** нужно, чтобы файл жил на стороне Windows — например, вы хотите открыть его в графическом приложении Windows или чтобы Chrome DevTools MCP работал из пути, доступного из Windows.

### Как переносить файлы туда и обратно

**Из Windows в WSL:** проще всего открыть Explorer и ввести `\\wsl.localhost\Ubuntu` в адресную строку. Дальше можно перетаскивать файлы в `\home\<you>\...`. Или из PowerShell:

```powershell
wsl cp /mnt/c/Users/you/Downloads/file.pdf ~/incoming/
```

**Из WSL в Windows:** скопируйте файл в `/mnt/c/Users/<you>/...`, и он сразу появится в Windows Explorer:

```bash
cp ~/reports/output.pdf /mnt/c/Users/you/Desktop/
```

**Открыть файл из WSL в Windows-приложении** (графический редактор, браузер и т. п.): используйте `explorer.exe` или `wslview`:

```bash
sudo apt install wslu     # один раз: установит wslview, wslpath, wslopen и т. д.
wslview ~/reports/output.pdf    # откроет через приложение по умолчанию в Windows
explorer.exe .                  # откроет текущий каталог WSL в Windows Explorer
```

**Конвертировать пути между двумя мирами:**

```bash
wslpath -w ~/code/project        # → \\wsl.localhost\Ubuntu\home\you\code\project
wslpath -u 'C:\Users\you'        # → /mnt/c/Users/you
```

### Переносы строк, BOM и git

Если вы редактируете файлы на стороне Windows в Windows-редакторе, они могут получить `CRLF`-переносы строк. Когда `bash` или Python на стороне Linux читает такие файлы, shell-скрипты ломаются с `bad interpreter: /bin/bash^M`, а Python может споткнуться о `.env` с BOM.

Лечится это нормальной конфигурацией git внутри WSL (не на Windows):

```bash
git config --global core.autocrlf input
git config --global core.eol lf
```

Если файлы уже с CRLF:

```bash
sudo apt install dos2unix
dos2unix path/to/script.sh
```

### Что лучше: клонировать внутри WSL или на `/mnt/c`?

Клонируйте внутри WSL. Всегда, если только у вас нет конкретной причины делать иначе. Типичный рабочий процесс Hermes (`hermes chat`, вызовы инструментов, которые `rg`-ают репозиторий, наблюдатели за файлами, фоновый шлюз) будет намного быстрее и надёжнее на `~/code/myrepo`, чем на `/mnt/c/Users/you/myrepo`.

Единственное исключение — **MCP-мосты, которые запускают исполняемые файлы Windows**. Если вы используете `chrome-devtools-mcp` через `cmd.exe` (см. [руководство MCP: WSL → Windows Chrome](/guides/use-mcp-with-hermes#wsl2-bridge-hermes-in-wsl-to-windows-chrome)), Windows может выдать предупреждение `UNC`, если текущая директория Hermes находится в `~`. В этом случае запускайте Hermes из `/mnt/c/`, чтобы Windows-процесс видел текущий каталог с буквой диска.

## Сеть: WSL ↔ Windows

WSL2 живёт в лёгкой VM со своим сетевым стеком. Это значит, что `localhost` внутри WSL — **не то же самое**, что `localhost` на Windows. Для каждого сервиса нужно отдельно решить, в какую сторону идёт трафик, и выбрать подходящий мост.

Чаще всего встречаются два сценария.

### Сценарий 1 — Hermes в WSL ходит к сервису на Windows

Самый частый случай: у вас на Windows запущен **Ollama, LM Studio или llama-server**, а Hermes внутри WSL должен к нему обращаться.

Подробная инструкция лежит в гайде по провайдерам: **[WSL2 Networking for Local Models →](/integrations/providers#wsl2-networking-windows-users)**

Коротко:

- **Windows 11 22H2+**: включите mirrored networking mode (`networkingMode=mirrored` в `%USERPROFILE%\.wslconfig`, затем `wsl --shutdown`). Тогда `localhost` работает в обе стороны.
- **Windows 10 и более старые сборки**: используйте IP хоста Windows (шлюз по умолчанию в виртуальной сети WSL) и убедитесь, что сервер на Windows слушает `0.0.0.0`, а не только `127.0.0.1`. Обычно ещё нужно правило брандмауэра на порт.

Подробная таблица с адресами привязки для Ollama / LM Studio / vLLM / SGLang, однострочными командами для брандмауэра, помощниками для динамического IP и обходным вариантом для Hyper-V firewall — в ссылке выше. Здесь мы не дублируем всё это заново.

### Сценарий 2 — что-то на Windows (или в локальной сети) ходит к Hermes в WSL

Это обратное направление, и про него обычно говорят меньше, но оно нужно для:

- использования веб-панели Hermes из браузера Windows;
- использования **API-сервера, совместимого с OpenAI** (который предоставляет `hermes gateway`, если `API_SERVER_ENABLED=true`) из инструмента на стороне Windows. См. [страницу API Server](/user-guide/features/api-server);
- тестирования **шлюза сообщений** (Telegram, Discord и т. п.), когда платформа присылает webhook на локальный URL — тут чаще всего лучше использовать `cloudflared`/`ngrok`, а не прямой проброс портов.

#### Подсценарий 2a: на самом хосте Windows

На **Windows 11 22H2+ с включённым mirrored mode** ничего делать не нужно. Процесс в WSL, который слушает `0.0.0.0:8080` (или даже `127.0.0.1:8080`), доступен из браузера Windows по `http://localhost:8080`. WSL автоматически пробрасывает привязку обратно на хост.

В **NAT mode** (Windows 10 / старые Windows 11) стандартный проброс localhost в WSL2 обычно перенаправляет `127.0.0.1` на стороне Linux в Windows `localhost`, так что служба Hermes, запущенная с `--host 127.0.0.1`, обычно доступна как `http://localhost:PORT` из Windows. Если нет:

- явно привяжитесь к `0.0.0.0` внутри WSL;
- узнайте IP WSL VM через `ip -4 addr show eth0 | grep inet` и обращайтесь к нему из Windows.

#### Подсценарий 2b: с другого устройства в вашей локальной сети (телефон, планшет, другой ПК)

Вот это и есть самый неприятный случай. Трафик идёт **устройство в локальной сети → хост Windows → WSL VM**, и вам нужно настроить оба перехода:

1. **Привяжитесь ко всем интерфейсам внутри WSL.** Процесс, слушающий `127.0.0.1`, никогда не будет доступен снаружи VM. Используйте `0.0.0.0`.

2. **Настройте проброс порта Windows → WSL VM.** В mirrored mode это делается автоматически. В NAT mode всё нужно прописать вручную, для каждого порта, в PowerShell от имени администратора:

   ```powershell
   # Получите текущий IP WSL VM (он меняется при каждом перезапуске WSL в NAT-режиме)
   $wslIp = (wsl hostname -I).Trim().Split(' ')[0]

   # Пробросьте порт Windows 8080 → WSL:8080
   netsh interface portproxy add v4tov4 `
     listenaddress=0.0.0.0 listenport=8080 `
     connectaddress=$wslIp connectport=8080

   # Разрешите его в брандмауэре Windows
   New-NetFirewallRule -DisplayName "Hermes WSL 8080" `
     -Direction Inbound -Protocol TCP -LocalPort 8080 -Action Allow
   ```

   Удалить правило потом можно так: `netsh interface portproxy delete v4tov4 listenaddress=0.0.0.0 listenport=8080`.

3. **Направьте устройство в LAN на `http://<windows-lan-ip>:8080`.**

Поскольку IP WSL VM при NAT mode меняется после каждого `wsl --shutdown`, одноразовое правило действует только до следующего рестарта WSL. Для постоянной схемы либо используйте mirrored mode, либо вынесите port-proxy в скрипт, который запускается при входе в Windows.

Для webhook-ов от облачных платформ обмена сообщениями (Telegram `setWebhook`, Slack events и т. п.) не мучайте себя пробросом портов — используйте `cloudflared`-туннели. См. [руководство по webhook-ам](/user-guide/messaging/webhooks).

## Как долго держать службы Hermes на Windows

Шлюз инструментов Hermes и API-сервер — долго живущие процессы. В WSL2 у вас есть несколько вариантов, как держать их живыми.

### Ярлык на рабочем столе для быстрого запуска Hermes

Если вам нужен запуск интерактивной оболочки Hermes двойным щелчком, создайте ярлык на стороне Windows и пусть он сразу переходит в WSL:

1. Щёлкните правой кнопкой по рабочему столу Windows и выберите **Создать -> Ярлык**.
2. В поле «Объект» укажите имя вашего дистрибутива (замените `Ubuntu`, если нужно):

   ```text
   wt.exe -w 0 -p "Ubuntu" wsl.exe -d Ubuntu --cd ~ -- bash -ic "hermes"
   ```

3. Назовите ярлык понятно, например `Hermes`.

Он откроет Windows Terminal, запустит WSL-дистрибутив, перейдёт в домашний каталог Linux и запустит Hermes. Если `hermes` ещё не виден в PATH, один раз откройте WSL вручную и выполните `source ~/.bashrc`, либо замените команду на `uv run hermes` внутри рабочей копии репозитория.

Дополнительные штрихи:

- **Свой значок:** откройте **Свойства -> Сменить значок** и укажите `.ico`, например favicon Hermes из репозитория.
- **Закрепление:** когда ярлык заработает, закрепите его в меню «Пуск» или на панели задач, чтобы не искать его снова.

### Внутри WSL с systemd (рекомендуется)

Если вы включили systemd по инструкции выше, `hermes gateway` и API-сервер работают так же, как на любом Linux. Используйте мастер настройки шлюза:

```bash
hermes gateway setup
```

Он предложит установить пользовательский systemd-юнит, чтобы шлюз стартовал автоматически вместе с WSL.

### Чтобы сам WSL запускался при входе в Windows

WSL VM живёт только пока её кто-то использует. Чтобы шлюз оставался доступен даже без открытого окна терминала, запустите процесс WSL через Планировщик заданий при логине Windows:

- **Триггер:** при входе в систему (ваш пользователь)
- **Действие:** Запустить программу
  - Программа: `C:\Windows\System32\wsl.exe`
  - Аргументы: `-d Ubuntu --exec /bin/sh -c "sleep infinity"`

Так VM будет жить, а шлюз под управлением systemd останется поднятым. На Windows 11 также работают более новые потоки автозапуска WSL, но `sleep infinity` — самый переносимый вариант.

## Прямой доступ к GPU (локальные модели)

WSL2 нативно поддерживает **NVIDIA** GPU начиная с WSL kernel 5.10.43+. Поставьте обычный драйвер NVIDIA на Windows (драйвер NVIDIA для Linux внутри WSL ставить **не нужно**), и `nvidia-smi` внутри WSL увидит GPU. После этого наборы CUDA, `torch`, `vllm`, `sglang` и `llama-server` собираются и работают на реальном GPU как обычно.

Поддержка AMD ROCm и Intel Arc внутри WSL2 всё ещё развивается и находится вне тестовой матрицы Hermes — возможно, оно и работает с текущими драйверами, но мы не можем дать надёжную рекомендуемую схему.

Если вы запускаете **нативный Windows-сервер локальной модели** (Ollama for Windows, LM Studio), который уже использует ваш GPU через драйвер Windows, прямой доступ к GPU через WSL вам не нужен вовсе — просто следуйте Сценарию 1 и обращайтесь к нему по сети из WSL.

## Типовые проблемы

**"Connection refused" к моему Ollama / LM Studio, размещённым на Windows.**
Смотрите [WSL2 Networking](/integrations/providers#wsl2-networking-windows-users). В девяти случаях из десяти сервер привязан к `127.0.0.1` и его нужно перевести на `0.0.0.0` (для Ollama: `OLLAMA_HOST=0.0.0.0`), либо у вас не хватает правила брандмауэра.

**Сильная медлительность `git status` / `hermes chat` в репозитории.**
Скорее всего, проект лежит под `/mnt/c/...`. Перенесите его в `~/code/...` (на стороне Linux). Разница по скорости обычно на порядок.

**`bad interpreter: /bin/bash^M` у скриптов.**
Это переносы строк CRLF из Windows-редактора. Прогоните `dos2unix script.sh` и задайте `core.autocrlf input` в git config внутри WSL.

**Предупреждение "UNC paths are not supported" от исполняемых файлов Windows, запущенных через MCP.**
Текущий каталог Hermes находится внутри файловой системы Linux, а `cmd.exe` на Windows не знает, что с ним делать. Запускайте Hermes из `/mnt/c/...` в этой сессии либо используйте обёртку, которая перед вызовом исполняемого файла Windows делает `cd` в путь, доступный Windows.

**Clock drift после sleep / hibernate.**
Часы WSL2 могут отставать на несколько минут после пробуждения хоста, а это ломает всё, что завязано на сертификаты и OAuth. Исправляется так:

```bash
sudo hwclock -s
```

Или установите `ntpdate` и запускайте его при логине.

**После включения mirrored mode или при VPN пропадает DNS.**
В mirrored mode настройки сети хоста проксируются в WSL — если DNS на Windows сломан (VPN split-tunnel, корпоративный DNS-резолвер), WSL унаследует ту же проблему. Обходной путь: вручную переопределите `resolv.conf` (поставьте `generateResolvConf=false` в `/etc/wsl.conf`, затем пропишите свой `/etc/resolv.conf` с `1.1.1.1` или DNS вашего VPN).

**`hermes` не найден после установки.**
Установщик добавляет `~/.local/bin` в PATH через `~/.bashrc`. Чтобы это заработало в текущей сессии, выполните `source ~/.bashrc` (или откройте новый терминал).

**Windows Defender тормозит WSL-файлы.**
Defender сканирует файлы через 9P-bridge при обращении из Windows, и из-за этого замедление `/mnt/c` только усиливается. Если вы работаете с WSL-файлами только изнутри WSL, это не критично. Если вы часто открываете `\\wsl$\...` из Windows-инструментов, подумайте об исключении пути дистрибутива WSL из сканирования в реальном времени.

**Заканчивается диск.**
WSL2 хранит диск ВМ как разреженный VHDX в `%LOCALAPPDATA%\Packages\...`. Он растёт, но не уменьшается автоматически после удаления файлов. Чтобы вернуть место: `wsl --shutdown`, затем из PowerShell от имени администратора выполните `Optimize-VHD -Path <path-to-ext4.vhdx> -Mode Full` (нужны Hyper-V tools) — либо используйте более простой путь через `diskpart`, который описан в документации WSL.
