# 持久目标

Hermes Agent 支持**持久目标**——一种让智能体在多个对话轮次中自动工作以完成多步骤任务的模式。受 Codex CLI 0.128.0 启发，Ralph 循环允许你设定一个目标，然后让智能体自主执行直到完成、暂停或达到预算限制。

## 快速开始

在对话中使用 `/goal` 命令：

```
/goal 为这个项目编写一个完整的 README，包括安装说明、使用示例和 API 文档
```

智能体将：
1. 分析目标并制定计划
2. 自动执行步骤（工具调用、文件操作、代码编写）
3. 每 20 轮检查一次进度
4. 完成后报告结果

## Ralph 循环

Ralph 循环（以 Codex CLI 的自主模式命名）是驱动目标执行的引擎：

```
用户设定目标
    ↓
智能体制定计划
    ↓
执行步骤（工具调用）
    ↓
每 N 轮评估进度
    ↓
完成？→ 报告结果
    ↓
否 → 继续执行
    ↓
阻塞/错误？→ 暂停或请求帮助
```

### 预算系统

目标使用**轮次预算**来防止无限运行：

- **默认预算**：20 轮
- 每轮包括一次 LLM 调用和任意数量的工具调用
- 预算耗尽时，智能体暂停并报告当前进度
- 你可以用 `/goal resume` 继续

### Judge Model

每 N 轮（默认 20），一个**judge model**评估进度：

```json
{"done": <bool>, "reason": "..."}
```

- `done: true` —— 目标完成，智能体报告结果
- `done: false` —— 继续执行，附带原因说明
- Judge 使用保守策略 —— 宁愿多执行几轮也不愿过早宣布完成

### Fail-open 语义

如果 judge model 调用失败或返回无效 JSON，系统采用**fail-open**语义：

- 假设 `done: false`
- 记录警告但继续执行
- 这防止了因 judge 故障而卡住

## 命令

| 命令 | 描述 |
|------|-------------|
| `/goal <描述>` | 设定一个新目标并开始执行 |
| `/goal status` | 显示当前目标的状态和进度 |
| `/goal pause` | 暂停当前目标执行 |
| `/goal resume` | 恢复暂停的目标 |
| `/goal clear` | 清除当前目标 |

## 配置

在 `~/.hermes/config.yaml` 中配置目标行为：

```yaml
goals:
  budget: 20                    # 默认轮次预算
  judge_model: ""               # Judge 模型（空 = 使用当前模型）
  auto_pause_on_error: true     # 错误时自动暂停
  show_reason: true             # 显示 judge 的推理过程
```

### 配置选项

| 选项 | 默认值 | 描述 |
|------|---------|-------------|
| `budget` | `20` | 默认轮次预算 |
| `judge_model` | `""` | 用于进度评估的模型。空字符串使用当前活动模型 |
| `auto_pause_on_error` | `true` | 工具调用失败时是否自动暂停 |
| `show_reason` | `true` | 是否在状态更新中显示 judge 的推理 |

## 与 Delegation 和 Cron 的交互

目标系统与 Hermes 的其他自动化功能交互：

- **Delegation**：目标可以在子智能体中运行。父智能体可以监控子智能体的目标进度
- **Cron**：定时任务可以设定目标。例如，每日目标"检查并更新依赖项"
- **互斥**：一个会话一次只能有一个活动目标。设定新目标会覆盖旧目标

## 使用场景

### 代码重构

```
/goal 将 utils.py 中的所有字符串操作函数重构到新的 string_utils.py 文件中，
并更新所有导入语句
```

### 文档编写

```
/goal 为所有公共 API 函数生成 JSDoc 注释，并创建一个 API 参考文档
```

### 测试生成

```
/goal 为 src/auth/ 目录中的所有模块编写单元测试，确保覆盖率 > 80%
```

### 依赖更新

```
/goal 检查所有过时的 npm 包，更新到最新兼容版本，并运行测试套件验证
```

## 最佳实践

1. **具体明确**：目标描述越具体，执行效果越好
2. **合理预算**：复杂任务可能需要 50+ 轮。在配置中调整预算
3. **监控进度**：定期使用 `/goal status` 检查执行情况
4. **错误处理**：启用 `auto_pause_on_error` 以便在出错时介入
5. **分解大目标**：对于非常大的任务，考虑分解为多个较小目标

## 限制

- 一次只能有一个活动目标
- 目标不会跨会话持久化（会话结束即丢失）
- Judge 评估消耗额外的 LLM 调用（每 N 轮一次）
- 某些工具调用（如 `delegate_task`）可能不受目标系统监控
