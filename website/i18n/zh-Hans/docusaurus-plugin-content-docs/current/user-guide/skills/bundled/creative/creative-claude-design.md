# CLI/API代理的Claude Design

当用户请求通常适合Claude Design的设计工作，但代理运行在CLI/API环境中而不是托管的Claude Design Web UI中时，使用此技能。

目标是保留Claude Design有用的设计行为和品味，同时移除在正常代理环境中不存在的托管工具管道。

**在开始之前，检查其他网页设计技能，如`popular-web-designs`（Stripe、Linear、Vercel、Notion等可直接粘贴的设计系统）和`design-md`（Google的DESIGN.md令牌规范格式）。** 如果用户想要已知品牌的外观，将`popular-web-designs`与此技能一起加载，让它提供视觉词汇。如果交付物是令牌规范文件而不是渲染的产物，请使用`design-md`。以下是完整的决策表。

## 设计原则：从上下文开始，而不是从氛围开始

好的高保真设计不是从零开始的。

5. UI套件

8. 副本文档

- 主题文件

- 全局样式表

- 路由/页面文件

文件树只是菜单。阅读定义视觉词汇的文件后再设计。

## 工作流程

1. **理解简报**
   - 要设计什么？
   - 给谁用的？
   - 最终应该存在什么产物？
   - 哪些约束是锁定的？

2. **收集上下文**
   - 阅读提供的文档、截图、仓库文件或设计资产。
   - 在编写代码之前识别视觉词汇。

3. **为此产物定义设计系统**
   - 颜色
   - 字体
   - 间距
   - 圆角
   - 阴影或层级
   - 运动姿态
   - 组件处理
   - 交互规则

4. **选择正确的格式**
   - 静态视觉比较：一个HTML画布上的选项并排。
   - 交互/流程：可点击的原型。
   - 演示：固定尺寸的HTML演示文稿，带幻灯片导航。
   - 组件探索：带变体的组件实验室。
   - 运动：时间线或基于状态的动画。

5. **构建产物**
   - 除非任务要求仓库实现，否则首选单个自包含HTML文件。
   - 对于重大修订保留之前的版本。
   - 避免不必要的依赖。

6. **验证**
   - 确认文件存在。
   - 运行任何可用的语法/静态检查。
   - 如果浏览器工具可用，打开文件并检查控制台错误。
   - 如果视觉保真度重要且截图工具可用，至少检查主视口。

7. **简要报告**
   - 确切的文件路径
   - 创建了什么
   - 注意事项
   - 下一个决定或下一个迭代

## 产物格式规则

默认为本地文件。

对于独立产物：

- 创建描述性文件名，例如`Landing Page.html`、`Command Palette Prototype.html`、`Design System Board.html`
- 将CSS嵌入`<style>`
- 将JS嵌入`<script>`
- 保持产物可以直接在浏览器中打开
- 除非明确有用且稳定，否则避免远程依赖
- 除非格式有意固定大小，否则包含响应式行为

对于重大修订：

- 将之前的版本保留为`Name.html`
- 创建`Name v2.html`、`Name v3.html`等
- 或者如果任务是变体探索，在一个文件内保留切换

对于仓库实现：

- 遵循仓库的实际技术栈
- 尽可能使用现有组件和令牌
- 如果用户要求生产代码，不要创建独立产物

## HTML / CSS / JS标准

善用现代CSS：

- CSS变量用于令牌
- CSS grid用于布局
- 容器查询在有帮助时使用
- 在支持的地方使用`text-wrap: pretty`
- 真实的焦点状态
- 真实的悬停状态
- 非平凡运动的`prefers-reduced-motion`处理
- 响应式缩放
- 实际可行的语义HTML

避免：

- 当预期真实仓库结构时，大量单体文件
- 脆弱的硬编码视口假设
- 难以访问的小点击目标
- 与可用性对抗的装饰性JS
- 除非没有更安全的选项，否则不要使用`scrollIntoView`

移动点击目标至少应为44px。

对于打印文档，文本至少应为12pt。

对于1920×1080幻灯片演示文稿，文本通常应为24px或更大。

## 独立HTML的React指导

默认使用普通HTML/CSS/JS。

仅在以下情况下使用React：

- 产物需要有意义的状态
- 变体/切换作为组件更容易
- 交互复杂性需要它
- 目标实现是React/Next.js且保真度重要

如果在独立HTML中使用CDN的React：

- 固定确切版本
- 避免未固定的`react@18`样式URL
- 除非必要，否则避免`type="module"`
- 避免多个名为`styles`的全局对象
- 给全局样式对象特定名称，例如`commandPaletteStyles`、`deckStyles`
- 如果拆分Babel脚本，明确地将共享组件附加到`window`

如果在真实仓库中构建，使用仓库的包管理器和组件架构代替。

## 演示文稿规则

对于幻灯片演示，使用固定尺寸画布并缩放以适应视口。

默认幻灯片尺寸：1920×1080，16:9。

要求：

- 键盘导航
- 可见的幻灯片计数
- localStorage持久化当前幻灯片
- 实际可行时使用打印友好布局
- 重要幻灯片的屏幕标签或稳定ID
- 除非用户明确要求，否则不要包含演讲者注释

不要用Markdown项目符号敷衍演示文稿。如果要求演示文稿，创建一个设计的产物。

除非品牌系统要求更多，否则最多使用1-2种背景颜色。

保持幻灯片稀疏。如果幻灯片感觉空旷，用布局、节奏、规模或图像占位符解决，而不是填充文本。

## 原型规则

对于交互式原型：

- 使主路径可点击
- 包括关键状态：默认、悬停/焦点、加载、空、错误、成功（相关时）
- 在有帮助时在页面内显示变体
- 将控件保持在最终构图之外，除非它们有意是原型的一部分
- 当刷新连续性重要时，在localStorage中持久化重要状态

如果原型意在模拟产品流程，设计流程，而不仅仅是第一个屏幕。

## 变体规则

在探索时，默认至少三种选项：

1. **保守** — 最接近现有模式/最低风险
2. **强适配** — 对简报的最佳诠释
3. **分歧** — 更新颖，有助于发现品味边界

变体可以探索：

- 布局
- 层级
- 字体大小
- 密度
- 颜色姿态
- 表面处理
- 运动
- 交互模型
- 复制结构
- 组件形状

除非颜色是实际问题，否则不要创建仅仅是颜色交换的变体。

当用户选择方向时，整合。不要永远把项目当作一堆选项。

## CLI/API模式中的可调整设计

这里不存在托管Claude Design编辑模式工具栏。

仍然保留这个想法：当有用时，添加名为`Tweaks`的页面内控件。

一个好的`Tweaks`面板可以控制：

- 主题模式
- 布局变体
- 密度
- 强调色
- 字体大小
- 运动开/关
- 复制变体
- 组件变体

保持小而不显眼。当调整隐藏时，设计应该看起来像最终版。

用localStorage持久化调整值（当有帮助时）。

## 内容纪律

不要添加填充内容。

每个元素都必须有其存在的理由。

避免：

- 虚假指标
- 装饰性统计数据
- 通用功能网格
- 不必要的图标
- 占位符推荐
- AI生成的 fluff 部分
- 改变策略或声明的虚构内容

如果附加部分、页面、复制或声明会改进产物，询问后再添加。

当复制必要但非最终时，标记为草稿或占位符。

## 反Slop规则

避免常见的AI设计 sludge：

- 激进的渐变背景
- 默认的玻璃拟态
- 除非品牌使用，否则不要用emoji
- 到处都是图标的通用SaaS卡片
- 左边框强调调用卡片
- 充满任意数字的虚假仪表板
- 库存照片英雄部分
- 作为层级替代品的超大圆角矩形
- 彩虹调色板
- 像"Inights"、"Growth"、"Scale"、"Optimize"这样没有内容的模糊标签
- 假装是产品图像的装饰性SVG插图

极简不自动等于好。密集不自动等于杂乱。有意选择。

## 排版

如果存在现有的类型系统，使用它。

如果没有，有意根据产物选择类型：

- 编辑：衬线或人文标题 + 约束无衬线正文
- 软件/生产力：精确无衬线 + 强数字处理
- 奢侈品/极简：更少字重，更多间距纪律
- 技术：仅mono强调，不是到处都是mono
- 演示：大大、清晰、高对比度

当更强的选择合适时，避免过度使用的默认值。

如果使用网络字体，保持字体族和字重数量低。

在添加框、图标或颜色之前，先用排版作为层级。

## 颜色

首先使用品牌/设计系统颜色。

如果没有调色板存在：

- 定义一个小系统
- 包括中性色、表面、墨水、柔和文本、边框、强调色、必要时的危险/成功
- 除非任务要求更广泛的调色板，否则只使用一种主要强调色
- 当浏览器支持可接受时，在和谐的自创调色板中使用oklch
- 检查重要文本和控件的对比度

不要从头开始发明大量颜色。

## 布局和构图

用节奏设计：

- 规模
- 空白
- 密度
- 对齐
- 重复
- 对比
- 中断

避免将每个部分做成相同的卡片网格。

对于产品UI，优先考虑理解速度而不是装饰。

对于营销界面，每个部分让一个想法落地。

对于仪表板，避免"数据slop"。只显示帮助用户决定或行动的数据。

## 运动

用纪律而不是戏剧使用运动。

好的运动：

- 澄清状态变化
- 减少加载时的焦虑
- 显示表面之间的连续性
- 给控件触觉
- 保持微妙

坏的运动：

- 无目的循环
- 延迟用户
- 吸引注意力
- 隐藏糟糕的层级

对于非平凡动画，遵守`prefers-reduced-motion`。

## 图片和图标

可用时使用提供的真实图像。

如果资产缺失：

- 使用干净的占位符
- 使用排版、布局或抽象纹理代替
- 当保真度重要时，询问真实材料

除非任务明确是插图工作，否则不要画复杂的假SVG插图。

除非图标改善扫描或匹配设计系统，否则避免图标。

## 源代码保真度

当从仓库重新创建或扩展UI时：

1. 检查仓库树
2. 识别实际的UI源文件
3. 阅读主题/令牌/全局样式/组件文件
4. 在适当时提升确切值
5. 匹配间距、圆角、阴影、复制语气、密度和交互模式
6. 然后才设计或修改

当源文件可用时，不要凭记忆构建。

对于GitHub URL，正确解析owner/repo/ref/path，在设计前检查相关文件。

## 阅读文档和资产

可用时直接阅读Markdown、HTML、CSS、JS、TS、JSX、TSX、JSON、SVG和纯文本。

对于DOCX/PPTX/PDF，如果存在则使用可用的本地提取工具。如果不可用，请用户提供导出的文本/图片或使用其他可用工具路径。

对于草图，优先考虑缩略图或截图，而不是原始绘图JSON，除非JSON是唯一可用的源。

## 版权和参考模型

除非用户明显有权使用该来源，否则不要重新创建公司的独特UI、专有命令结构、品牌屏幕或确切视觉身份。

提取一般设计原则是可以接受的：

- 无杂乱的密度
- 命令优先交互
- 单色加一种强调色
- 编辑层级
- 清晰的空状态
- 强大的键盘启示

克隆专有布局、复制确切品牌表面或复制版权内容是不可接受的。

使用参考资料时，将姿态和原则转化为原创设计。

## 验证

在最终响应之前，尽可能多地验证环境允许的内容。

最低要求：

- 文件存在于声明的路径
- HTML已完全保存
- 检查了明显的语法问题

更好：

- 在浏览器工具中打开并检查控制台错误
- 在主视口检查截图
- 测试关键交互
- 如果存在，测试亮/暗或变体
- 如果相关，测试响应式断点

如果验证受环境限制，精确说明验证了什么和没有验证什么。

如果文件实际上没有写入，永远不要说"完成"。

## 最终响应格式

保持最终响应简短。

包括：

- 产物路径
- 包含什么
- 验证状态
- 如果有用，下一步建议操作

示例：

```text
Created: /path/to/Prototype.html
包含3个布局变体、一个用于密度/主题的Tweaks面板，以及响应式行为。
验证：文件存在并在浏览器中干净打开，无控制台错误。
下一步：选择最强方向，我将精简复制+运动。
```

## 便携式打开提示模式

将Claude Design样式请求适配到CLI/API模式时，使用此心理翻译：

```text
你运行在CLI/API模式，而不是托管的Claude Design。忽略对托管工具或预览窗格的引用。生成完整的本地设计产物，通常是自包含的HTML，带嵌入式CSS/JS，并在返回之前用可用的本地工具验证。保留设计过程：收集上下文、定义系统、产生选项、避免填充，并达到高的视觉标准。
```

## 陷阱

- 不要将托管工具模式粘贴到技能中。它们会导致假的工具调用。
- 不要将技能指向巨大的外部提示作为必需的运行时上下文。那会产生漂移。
- 不要在移除工具管道时剥离设计原则。
- 当用户已经给出足够方向时，不要过度提问。
- 对于没有品牌上下文的高保真工作，不要提问不足。
- 不要产生通用SaaS布局并称它们为设计。
- 除非实际上发生了，否则不要声称浏览器验证。
