# 研究论文写作流水线

面向 **NeurIPS、ICML、ICLR、ACL、AAAI 和 COLM** 的端到端流水线，用于产出可发表的机器学习/人工智能研究论文。此技能涵盖完整的研究生命周期：实验设计、执行、监控、分析、论文写作、审阅、修改和投稿。

这不是一个**线性流水线** — 它是一个迭代循环。结果触发新实验。审阅触发新分析。agent 必须处理这些反馈循环。

## 何时使用此技能

在以下情况下使用此技能：
- **从头开始新的研究论文** 基于现有代码库或想法
- **设计和运行实验** 以支持论文声明
- **撰写或修改** 研究论文的任何部分
- **准备向** 特定会议或研讨会投稿
- **回复审阅意见** 附带额外实验或修改
- **在会议格式之间转换** 论文
- **撰写非实证论文** — 理论、综述、基准测试或立场文件（参见[超越实证机器学习的论文类型](#超越实证机器学习的论文类型)）
- **设计人工评估** 用于自然语言处理、人机交互或对齐研究
- **准备录用后的交付物** — 海报、演讲、代码发布

## 核心哲学

1. **主动出击。** 提供完整草稿，而不是问题。科学家很忙 — 产出具体的东西让他们反馈，然后迭代。
2. **永远不要编造引用。** AI 生成的引用错误率约为 40%。始终通过程序化方式获取。将无法验证的引用标记为 `[CITATION NEEDED]`。
3. **论文是一个故事，而不是实验的集合。** 每篇论文需要一句话中陈述的一个明确贡献。如果您做不到，论文还没准备好。
4. **实验为声明服务。** 每个实验必须明确说明它支持哪个声明。永远不要运行与论文叙事无关的实验。
5. **尽早提交，经常提交。** 每个完成的实验批次、每次论文草稿更新 — 用描述性消息提交。Git log 是实验历史。

### 主动性和协作

**默认：主动出击。先草稿，带着草稿询问。**

| 置信度级别 | 行动 |
|-----------------|--------|
| **高**（清晰的仓库，明显的贡献） | 写完整草稿，交付，迭代反馈 |
| **中**（有些模糊） | 带着标记的不确定性写草稿，继续 |
| **低**（重大未知） | 通过 `clarify` 提出 1-2 个有针对性的问题，然后写草稿 |

| 部分 | 自主起草？ | 带草稿标记 |
|---------|-------------------|-----------------|
| 摘要 | 是 | "贡献框架为 X — 如果需要请调整" |
| 引言 | 是 | "强调问题 Y — 如果错误请纠正" |
| 方法 | 是 | "包含了细节 A、B、C — 添加缺失部分" |
| 实验 | 是 | "突出了结果 1、2、3 — 如果需要请重新排序" |
| 相关工作 | 是 | "引用了论文 X、Y、Z — 添加我遗漏的任何论文" |

**仅在以下情况下阻塞以获取输入**：目标会议不明确、多个矛盾框架、结果似乎不完整、明确要求先审阅。

在实验运行时跟踪实际支出：
```python
# 简单成本跟踪模式
import json, os
from datetime import datetime

COST_LOG = "results/cost_log.jsonl"

def log_cost(experiment: str, model: str, input_tokens: int, output_tokens: int, cost_usd: float):
    entry = {
        "timestamp": datetime.now().isoformat(),
        "experiment": experiment,
        "model": model,
        "input_tokens": input_tokens,
        "output_tokens": output_tokens,
        "cost_usd": cost_usd,
    }
    with open(COST_LOG, "a") as f:
        f.write(json.dumps(entry) + "\n")
```

**预算紧张时**：在承诺全面扫描之前运行试点实验（1-2 个种子，任务子集）。使用更便宜的模型调试流水线，然后在最终运行时切换到目标模型。

### 步骤 0.7：多作者协调

大多数论文有 3-10 位作者。尽早建立工作流程：

| 工作流程 | 工具 | 何时使用 |
|----------|------|-------------|
| **Overleaf** | 基于浏览器 | 多位作者同时编辑，无 git 经验 |
| **Git + LaTeX** | `git` 配合 `.gitignore` 处理辅助文件 | 技术团队，需要基于分支的审阅 |
| **Overleaf + Git 同步** | Overleaf 高级版 | 两全其美 — 实时协作与版本历史 |

**部分所有权**：将每个部分分配给一位主要作者。其他人评论但不直接编辑。防止合并冲突和风格不一致。

```
作者协调检查清单：
- [ ] 同意部分所有权（谁写什么）
- [ ] 设置共享工作区（Overleaf 或 git 仓库）
- [ ] 在任何人写作之前建立符号约定
- [ ] 安排内部审阅轮次（不只是最后）
- [ ] 指定一人进行最终格式处理
- [ ] 在创建图表之前同意图表风格（颜色、字体、大小）
```

**尽早同意的 LaTeX 约定**：
- `\method{}` 宏用于一致的方法命名
- 引用样式：`\citet{}` vs `\citep{}` 用法
- 数学符号：小写粗体表示向量，大写粗体表示矩阵等
- 英式 vs 美式拼写

**目标**：找到相关工作，确定基线，收集引用。

# 通过终端：

### 步骤 1.2：搜索相关工作

**加载 `arxiv` 技能** 以进行结构化论文发现：`skill_view("arxiv")`。它提供 arXiv REST API 搜索、Semantic Scholar 引文图、作者资料和 BibTeX 生成。

# 通过 web_extract（用于特定论文）：

尝试的其他搜索查询：

```

- "[主要技术] + [应用领域]"
- "[基线方法] 比较"
- "[问题名称] 最新技术"
- 现有引用中的作者姓名
```

### 步骤 1.2b：深化搜索（广度优先，然后深度）

平面搜索（一轮查询）通常会错过重要的相关工作。使用迭代的**广度-然后-深度**模式：

```
迭代文献搜索：

第 1 轮（广度）：4-6 个覆盖不同角度的并行查询
  - "[方法] + [领域]"
  - "[问题名称] 最新技术 2024 2025"
  - "[基线方法] 比较"
  - "[替代方法] vs [你的方法]"
  → 收集论文，提取关键概念和术语

第 2 轮（深度）：从第 1 轮学习中生成后续查询
  - 第 1 轮论文中发现的新术语
  - 第 1 轮最相关结果引用的论文
  - 需要调查的矛盾发现
  → 收集论文，识别剩余差距

第 3 轮（定向）：填补特定差距
  - 第 1-2 轮确定的缺失基线
  - 近 6 个月的同期工作（相同问题）
  - 关键负面结果或失败尝试
  → 当新查询大多返回你已经看过的论文时停止
```

**何时停止**：如果一轮返回 >80% 已在收藏中的论文，搜索已饱和。通常 2-3 轮就足够了。对于综述论文，期望 4-5 轮。

**对于基于 agent 的工作流程**：通过 `delegate_task` 并行委托每轮查询。收集结果，去重，然后从组合学习中生成下一轮查询。

### 步骤 1.3：验证每个引用

**永远不要从记忆中生成 BibTeX。始终通过程序化方式获取。**

对于每个引用，遵循强制性的 5 步流程：

```
引用验证（每个引用强制）：
1. 搜索 → 使用特定关键词查询 Semantic Scholar 或 Exa MCP
2. 验证 → 在 2+ 个来源确认论文存在（Semantic Scholar + arXiv/CrossRef）
3. 获取 → 通过 DOI 内容协商获取 BibTeX（程序化，不从记忆）
4. 验证 → 确认你引用的声明实际出现在论文中
5. 添加 → 将已验证的 BibTeX 添加到参考文献
如果任何步骤失败 → 标记为 [CITATION NEEDED]，通知科学家
```

```python
# 通过 DOI 获取 BibTeX
import requests

def doi_to_bibtex(doi: str) -> str:
    response = requests.get(
        f"https://doi.org/{doi}",
        headers={"Accept": "application/x-bibtex"}
    )
    response.raise_for_status()
    return response.text
```

如果您无法验证引用：

```latex
\cite{PLACEHOLDER_author2024_verify_this}  % TODO：验证此引用存在
```

**始终告知科学家**："我已将 [X] 个引用标记为需要验证的占位符。"

### 步骤 1.4：组织相关工作

按方法论分组论文，而不是逐篇：

**好**："一系列工作使用 X 的假设 [refs]，而我们使用 Y 的假设因为..."
**坏**："Smith 等人引入了 X。Jones 等人引入了 Y。我们结合了两者。"

## 阶段 2：实验设计

**目标**：设计直接支持论文声明的实验。每个实验必须回答一个具体问题。

### 步骤 2.1：将声明映射到实验

创建明确的映射：

| 声明 | 实验 | 预期证据 |
|-------|-----------|-------------------|
| "我们的方法优于基线" | 主要比较（表 1） | 胜率，统计显著性 |
| "效果对于较弱模型更大" | 模型扩展研究 | 单调改进曲线 |
| "收敛需要范围约束" | 约束 vs 非约束 | 收敛率比较 |

**规则**：如果实验不映射到声明，不要运行它。

### 步骤 2.2：设计基线

强基线是区分录用论文和拒稿论文的关键。审阅者会问："他们与 X 比较了吗？"

标准基线类别：
- **简单基线**：最简单可能的方法
- **强基线**：最佳已知现有方法
- **消融基线**：你的方法减去一个组件
- **计算匹配基线**：相同计算预算，不同分配

### 步骤 2.3：定义评估协议

在运行任何东西之前，指定：
- **指标**：你在测量什么，方向符号（越高/越低越好）
- **聚合**：结果如何跨运行/任务组合
- **统计检验**：什么检验将建立显著性
- **样本量**：多少次运行/问题/任务

### 步骤 2.4：编写实验脚本

遵循成功研究流水线的这些模式：

**增量保存** — 每步后保存结果以便崩溃恢复：
```python
# 每个问题/任务后保存
result_path = f"results/{task}/{strategy}/result.json"
if os.path.exists(result_path):
    continue  # 跳过已完成的工作
# ... 运行实验 ...
with open(result_path, 'w') as f:
    json.dump(result, f, indent=2)
```

**Artifacts 保存** — 保存所有中间输出：
```
results/<experiment>/
  <task>/
    <strategy>/
      final_output.md          # 最终结果
      history.json             # 完整轨迹
      pass_01/                 # 每次迭代 artifacts
        version_a.md
        version_b.md
        critic.md
```

**关注点分离** — 保持生成、评估和可视化分离：
```
run_experiment.py              # 核心实验运行器
run_baselines.py               # 基线比较
run_comparison_judge.py        # 盲评估
analyze_results.py             # 统计分析
make_charts.py                 # 可视化
```

### 步骤 2.5：设计人工评估（如适用）

许多自然语言处理、人机交互和对齐研究需要人工评估作为主要或补充证据。在运行自动化实验之前设计这个 — 人工评估通常有更长的前置时间（IRB 批准、标注员招募）。

**何时需要人工评估：**
- 自动化指标无法捕获你关心的内容（流畅性、有帮助性、安全性）
- 你的贡献是关于面向人类的品质（可读性、偏好、信任）
- 自然语言处理场所的审阅者（ACL、EMNLP）期望生成任务的人工评估

**关键设计决策：**

| 决策 | 选项 | 指导 |
|----------|---------|----------|
| **标注员类型** | 专家、众包工作者、最终用户 | 与你的声明要求匹配 |
| **量表** | 李克特 (1-5)、成对比较、排序 | 对于 LLM 输出，成对比较比李克特更可靠 |
| **样本量** | 每标注员和总项目数 | 功效分析或最少 100 项，3+ 标注员 |
| **一致性指标** | Cohen's kappa、Krippendorff's alpha、ICC | 3+ 标注员用 Krippendorff's alpha；也报告原始一致性 |
| **平台** | Prolific、MTurk、内部团队 | 质量用 Prolific；规模用 MTurk；领域专业知识用内部 |

**标注指南检查清单：**
```
- [ ] 清晰的任务描述与示例（好的和坏的）
- [ ] 模糊情况的决策标准
- [ ] 每个类别至少 2 个完整示例
- [ ] 注意力检查/黄金标准项目（总数的 10-15%）
- [ ] 资格任务或筛选轮次
- [ ] 每项估计时间和公平报酬（>= 当地最低工资）
- [ ] 如有需要，机构审查委员会/伦理审查
```

**报告要求**（审阅者检查所有这些）：
- 标注员数量及其资质
- 带具体指标和值的人与人之间的一致性
- 报酬详情（金额、估计时薪）
- 标注界面描述或截图（附录）
- 总标注时间

## 阶段 3：实验执行与监控

**目标**：可靠运行实验，监控进度，从失败中恢复。

### 步骤 3.1：启动实验

使用 `nohup` 运行长时间实验：

```bash
nohup python run_experiment.py --config config.yaml > logs/experiment_01.log 2>&1 &
echo $!  # 记录 PID
```

**并行执行**：同时运行独立实验，但要注意 API 速率限制。同一 API 上 4+ 个并发实验会相互减慢。

### 步骤 3.2：设置监控（Cron 模式）

对于长时间运行的实验，设置定期状态检查。Cron 提示应遵循此模板：

```
监控提示模板：
1. 检查进程是否仍在运行：ps aux | grep <pattern>
2. 读取日志最后 30 行：tail -30 <logfile>
3. 检查已完成的结果：ls <result_dir>
4. 如果结果存在，读取并报告：cat <result_file>
5. 如果全部完成，提交：git add -A && git commit -m "<descriptive message>" && git push
6. 以结构化格式报告（带关键指标的表格）
7. 回答这个实验的关键分析问题
```

**静默模式**：如果自上次检查以来没有任何变化，用 `[SILENT]` 响应以抑制通知给用户。只有在有新闻时才报告。

### 步骤 3.3：处理失败

常见失败模式和恢复：

| 失败 | 检测 | 恢复 |
|---------|-----------|----------|
| API 速率限制 / 信用耗尽 | 日志中的 402/429 错误 | 等待，然后重新运行（脚本跳过已完成的工作） |
| 进程崩溃 | PID 消失，结果不完整 | 从最后一个检查点重新运行 |
| 难题超时 | 进程卡住，日志无进展 | 杀死并跳过，在结果中注明 |
| 错误的模型 ID | 引用模型名的错误 | 修复 ID 并重新运行 |

**关键**：脚本应始终检查现有结果并跳过已完成的工作。这使重新运行安全高效。

### 步骤 3.4：提交已完成的结果

每个实验批次完成后：

```bash
git add -A
git commit -m "添加 <实验名称>：<一句话关键发现>"
git push
```

### 步骤 3.5：维护实验日志

Git 提交跟踪发生了什么，但不是**探索树** — 基于你所学的关于下一步尝试什么的决定。维护一个结构化实验日志来捕获这棵树：

```json
// experiment_journal.jsonl — 每实验尝试追加一条记录
{
  "id": "exp_003",
  "parent": "exp_001",
  "timestamp": "2025-05-10T14:30:00Z",
  "hypothesis": "添加范围约束将修复 exp_001 的收敛失败",
  "plan": "使用 max_tokens=2000 和固定结构模板重新运行 autoreason",
  "config": {"model": "haiku", "strategy": "autoreason", "max_tokens": 2000},
  "status": "completed",
  "result_path": "results/exp_003/",
  "key_metrics": {"win_rate": 0.85, "convergence_rounds": 3},
  "analysis": "范围约束修复了收敛。胜率从 0.42 跳到 0.85。",
  "next_steps": ["在 Sonnet 上尝试相同约束", "在没有结构模板的情况下测试"],
  "figures": ["figures/exp003_convergence.pdf"]
}
```

**为什么需要日志，而不仅仅是 git？** Git 跟踪文件更改。日志跟踪推理：你为什么尝试 X，你学到了什么，这对你下一步意味着什么。写论文时，这棵树对于方法部分（"我们观察到 X，这促使我们做 Y"）和诚实的失败报告都是无价的。

**选择最佳路径**：当日志显示分支树（exp_001 → exp_002a、exp_002b、exp_003）时，识别最能支持论文声明的路径。在附录中将死胡同记录为消融或负面结果。

**每次实验快照代码**：每次运行后复制实验脚本：
```bash
cp experiment.py results/exp_003/experiment_snapshot.py
```
这使得即使在后续代码更改后也能精确重现。

## 阶段 4：结果分析

**目标**：提取发现，计算统计，识别故事。

### 步骤 4.1：聚合结果

编写分析脚本：
1. 从批次加载所有结果文件
2. 计算每个任务和聚合指标
3. 生成汇总表

```python
# 标准分析模式
import json, os
from pathlib import Path

results = {}
for result_file in Path("results/").rglob("result.json"):
    data = json.loads(result_file.read_text())
    strategy = result_file.parent.name
    task = result_file.parent.parent.name
    results.setdefault(strategy, {})[task] = data

# 计算聚合指标
for strategy, tasks in results.items():
    scores = [t["score"] for t in tasks.values()]
    print(f"{strategy}: mean={np.mean(scores):.1f}, std={np.std(scores):.1f}")
```

### 步骤 4.2：统计显著性

始终计算：
- **误差线**：标准差或标准误差，指定是哪个
- **置信区间**：关键结果的 95% CI
- **成对检验**：比较两种方法的 McNemar 检验
- **效应量**：Cohen's d 或 h 用于实际显著性

### 步骤 4.3：识别故事

分析后，明确回答：
1. **主要发现是什么？** 一句话陈述。
2. **什么让你惊讶？** 意外结果通常是最好的论文。
3. **什么失败了？** 失败实验通常是最有见地的。诚实地报告失败加强了论文。
4. **需要哪些后续实验？** 结果经常引发新问题。

#### 处理负面或无效结果

当你的假设是错误的或结果不确定时，你有三个选择：

| 情况 | 行动 | 会议适合性 |
|-----------|--------|-----------|
| 假设错误但**为什么**有信息量 | 以分析为什么的论文为中心 | NeurIPS、ICML（如果分析严谨） |
| 方法没有击败基线但**揭示了新事物** | 将贡献重新定位为理解/分析 | ICLR（重视理解）、研讨会论文 |
| 流行声明的干净负面结果 | 写出来 — 学术界需要知道 | NeurIPS 数据集与基准测试、TMLR、研讨会 |
| 结果不确定，没有清晰故事 | 转向 — 运行不同实验或重新框架 | 不要强求一篇不存在的论文 |

**如何写负面结果论文：**
- 从社区相信什么以及为什么测试它很重要开始
- 描述你严谨的方法论（必须无懈可击 — 审阅者会仔细审查）
- 清晰地呈现带有统计证据的无效结果
- 分析**为什么**预期结果没有出现
- 讨论对领域的影响

**明确欢迎负面结果的会议**：NeurIPS（数据集与基准测试 track）、TMLR、ML 重现性挑战、主要会议的研讨会。一些研讨会专门征集负面结果。

### 步骤 4.4：创建图表和表格

**图表**：
- 对所有图表使用矢量图形（PDF）：`plt.savefig('fig.pdf')`
- 色盲安全调色板（Okabe-Ito 或 Paul Tol）
- 自包含标题 — 读者无需正文就能理解
- 图表内不要有标题 — 标题提供此功能

**表格**：
- 使用 `booktabs` LaTeX 包
- 每个指标加粗最佳值
- 包含方向符号（越高/越低越好）
- 一致的小数精度

### 步骤 4.5：决定：更多实验还是写作？

| 情况 | 行动 |
|-----------|--------|
| 核心声明得到支持，结果显著 | 进入阶段 5（写作） |
| 结果不确定，需要更多数据 | 返回阶段 2（设计） |
| 意外发现暗示新方向 | 返回阶段 2（设计） |
| 缺少审阅者会要求的消融之一 | 运行它，然后阶段 5 |
| 所有实验完成但有些失败 | 记录失败，进入阶段 5 |

### 步骤 4.6：编写实验日志（连接写作）

在进入论文写作之前，创建一个结构化实验日志，连接结果和文本。这是从实验到文本的最重要的连接组织 — 没有它，写作 agent 必须从原始结果文件重新推导故事。

**创建 `experiment_log.md`** 包含以下结构：

```markdown
# 实验日志

## 贡献（一句话）
[论文的主要声明]

## 运行的实验

### 实验 1：[名称]
- **测试的声明**：支持哪个论文声明
- **设置**：[模型、数据集、配置、运行次数]
- **关键结果**：[一句话带数字]
- **结果文件**：results/exp1/final_info.json
- **生成的图表**：figures/exp1_comparison.pdf
- **意外发现**：[任何意外情况]

### 实验 2：[名称]
...

## 图表
| 文件名 | 描述 | 属于哪一部分 |
|----------|-------------|---------------------------|
| figures/main_comparison.pdf | 所有方法在基准 X 上的条形图比较 | 结果，图 2 |
| figures/ablation.pdf | 移除组件 A、B、C 的消融 | 结果，图 3 |
...

## 失败实验（记录以保持诚实）
- [尝试了什么，为什么失败，告诉我们什么]

## 开放问题
- [结果引发的任何论文应该解决的问题]
```

**为什么这很重要**：起草时，agent（或委托的子 agent）可以加载 `experiment_log.md` 以及 LaTeX 模板并产出扎根于实际结果的第一稿。没有这个桥梁，写作 agent 必须解析原始 JSON/CSV 文件并推断故事 — 这是产生虚构或错误报告数字的常见来源。

**Git 纪律**：将此日志与它描述的结果一起提交。

## 迭代精炼：策略选择

此流水线中的任何输出 — 论文草稿、实验脚本、分析 — 都可以迭代精炼。Autoreason 研究为每种精炼策略何时有效、何时失败提供了经验证据。使用本节选择正确的方法。

### 快速决策表

| 你的情况 | 策略 | 为什么 |
|---------------|----------|-----|
| 中等层模型 + 约束任务 | **Autoreason** | 最佳点。生成-评估差距最大。基线积极破坏弱模型输出。 |
| 中等层模型 + 开放任务 | **Autoreason** 加范围约束 | 添加固定事实、结构或交付物以绑定改进空间。 |
| 前沿模型 + 约束任务 | **Autoreason** | 即使在前沿也能赢得 2/3 约束任务。 |
| 前沿模型 + 开放任务 | **Critique-and-revise** 或**单次通过** | Autoreason 最后来。模型自评足够好。 |
| 具体技术任务（系统设计） | **Critique-and-revise** | 直接查找-修复循环更高效。 |
| 模板填充任务（一个正确结构） | **单次通过** 或**保守** | 决策空间最小。迭代无价值。 |
| 带测试用例的代码 | **Autoreason（代码变体）** | 在修复前结构化分析*为什么*失败。恢复率 62% vs 43%。 |
| 很弱的模型（Llama 8B 类） | **单次通过** | 模型太弱无法生成多样化候选。投资于生成质量。 |

### 生成-评估差距

**核心洞察**：Autoreason 的价值取决于模型生成能力和自评估能力之间的差距。

```
模型层        │ 生成    │ 自评估   │ 差距    │ Autoreason 价值
──────────────────┼────────────┼───────────┼────────┼─────────────────
弱 (Llama 8B)    │ 差       │ 差       │ 小     │ 无 — 无法生成多样化候选
中 (Haiku 3.5)   │ 不错     │ 差       │ 大     │ 最大 — 42/42 完美 Borda
中 (Gemini Flash)│ 不错     │ 中等     │ 大     │ 高 — 赢得 2/3
强 (Sonnet 4)    │ 好       │ 不错     │ 中等   │ 中等 — 赢得 3/5
前沿 (S4.6)      │ 优秀     │ 好       │ 小     │ 仅加约束有效
```

这个差距是结构性的，不是暂时的。随着成本下降，今天的前沿成为明天的中等层。最佳点移动但永不消失。

### Autoreason 循环（摘要）

每轮从新鲜、隔离的 agent 生成三个候选：

1. **批评者** → 找出现任 A 的问题（不修复）
2. **作者 B** → 根据批评修改 A
3. **综合器** → 合并 A 和 B（随机标签）
4. **评判小组** → 3 个盲 CoT 评判通过 Borda 计数对 A、B、AB 排名
5. **收敛** → A 连续赢 k=2 轮 → 完成

**关键参数：**
- k=2 收敛（k=1 过早，k=3 太贵，无质量提升）
- CoT 评判始终（3 倍更快收敛）
- 温度 0.8 作者，0.3 评判
- 保守平局决胜：现任赢平局
- 每个角色是一个没有共享上下文的新鲜 agent

### 应用于论文草稿

当通过 autoreason 精炼论文本身时：
- **向批评者提供 ground truth**：实际实验数据、结果 JSON、统计输出。没有这些，模型会虚构消融研究和虚假置信区间。
- **使用至少 3 个工作评判**：破损的评判解析器不会增加噪音 — 它完全阻止平衡。
- **范围约束修订**："解决这些具体弱点"而不是"改进论文。"

### 失败模式

| 失败 | 检测 | 修复 |
|---------|-----------|-----|
| 无收敛（A 永不赢） | 20+ 轮后 A 赢 <15% | 为任务添加范围约束 |
| 综合漂移 | 字数无限增长 | 约束结构和交付物 |
| 低于单次通过 | 基线得分高于迭代输出 | 切换到单次通过；模型可能太弱 |
| 过拟合（代码） | 高公开测试通过，低私密测试通过 | 使用结构化分析，而不仅仅是测试反馈 |
| 破损评判 | 解析失败将小组降至 3 以下 | 继续前修复解析器 |

## 阶段 5：论文起草

**目标**：写出一篇完整的、可发表的论文。

### 大型项目的上下文管理

一个包含 50+ 实验文件、多个结果目录和大量文献笔记的论文项目很容易超过 agent 的上下文窗口。主动管理：

**每个起草任务加载到上下文中：**

| 起草任务 | 加载到上下文 | 不要加载 |
|---------------|------------------|-------------|
| 写引言 | `experiment_log.md`、贡献声明、5-10 个最相关论文摘要 | 原始结果 JSON、所有实验脚本、所有文献笔记 |
| 写方法 | 实验配置、伪代码、架构描述 | 原始日志、其他实验结果 |
| 写结果 | `experiment_log.md`、结果汇总表、图表列表 | 完整分析脚本、中间数据 |
| 写相关工作 | 整理的引用笔记（步骤 1.4 输出）、.bib 文件 | 实验文件、原始 PDF |
| 修改轮次 | 完整论文草稿、具体审阅者关注点 | 其他一切 |

**原则：**
- **`experiment_log.md` 是主要的上下文桥梁** — 它总结写作所需的一切，无需加载原始数据文件（参见步骤 4.6）
- **一次加载一个部分的上下文** 当委托时。起草方法的子 agent 不需要文献综述笔记。
- **总结，不要包含原始文件。** 对于 200 行结果 JSON，加载 10 行汇总表。对于 50 页相关论文，加载 5 句摘要 + 2 行关于其相关性的笔记。
- **对于非常大的项目**：创建 `context/` 目录与预压缩摘要：
  ```
  context/
    contribution.md          # 1 句话
    experiment_summary.md    # 关键结果表（来自 experiment_log.md）
    literature_map.md        # 整理的引用笔记
    figure_inventory.md      # 带描述的图表列表
  ```

### 叙事原则

**最关键的洞察**：你的论文不是实验的集合 — 它是一个故事，由有证据支持的一个明确贡献为中心。

每篇成功的机器学习论文都围绕 Neel Nanda 所说的"叙事"展开：一个简短的、严谨的、基于证据的技术故事，带有读者关心的要点。

**三个支柱（引言结尾必须清晰）：**

| 支柱 | 描述 | 测试 |
|--------|-------------|------|
| **是什么** | 1-3 个具体新声明 | 你能用一句话陈述它们吗？ |
| **为什么** | 严谨的经验证据 | 实验是否区分你的假设与替代方案？ |
| **那又怎样** | 为什么读者应该关心 | 这与公认的社区问题有关吗？ |

**如果你不能用一句话陈述你的贡献，你还没有论文。**

### 时间分配

大约**平均分配**时间给：
1. 摘要
2. 引言
3. 图表
4. 其他所有组合

**为什么？** 大多数审阅者在到达你的方法之前就形成判断。读者遇到你的论文方式是：标题 → 摘要 → 引言 → 图表 → 也许其他部分。

### 写作工作流程

```
论文写作检查清单：
- [ ] 步骤 1：定义一句话贡献
- [ ] 步骤 2：起草图 1（核心思想或最令人信服的结果）
- [ ] 步骤 3：起草摘要（5 句公式）
- [ ] 步骤 4：起草引言（最多 1-1.5 页）
- [ ] 步骤 5：起草方法
- [ ] 步骤 6：起草实验与结果
- [ ] 步骤 7：起草相关工作
- [ ] 步骤 8：起草结论与讨论
- [ ] 步骤 9：起草局限性（所有场所要求）
- [ ] 步骤 10：计划附录（证明、额外实验、细节）
- [ ] 步骤 11：完整论文检查清单
- [ ] 步骤 12：最终审阅
```

### 两轮精炼模式

使用 AI agent 起草时，使用**两轮**方法（在 SakanaAI 的 AI-Scientist 流水线中证明有效）：

**第 1 轮 — 写作 + 每部分即时精炼：**
对于每个部分，编写完整草稿，然后在同一上下文中立即精炼。这在本地问题（清晰度、流程、完整性）新鲜时捕获它们。

**第 2 轮 — 全局精炼与完整论文上下文：**
在所有部分起草后，在了解完整论文的情况下重新访问每个部分。这捕获跨部分问题：冗余、不一致术语、叙事流程，以及一个部分承诺但另一个未交付的空白。

### LaTeX 错误检查清单

将此检查清单附加到每个精炼提示。LLM 写 LaTeX 时最常见的错误：

```
LaTeX 质量检查清单（每次编辑后验证）：
- [ ] 无未封闭的数学符号（$ 符号平衡）
- [ ] 只引用存在的图表/表格（\ref 匹配 \label）
- [ ] 无虚构引用（\cite 匹配 .bib 中的条目）
- [ ] 每个 \begin{env} 有匹配的 \end{env}（尤其是 figure、table、algorithm）
- [ ] 无 HTML 污染（</end{figure}> 而非 \end{figure}）
- [ ] 数学模式外无不转义的下划线（文本中使用 \_）
- [ ] 无重复 \label 定义
- [ ] 无重复章节标题
- [ ] 文本中的数字与实际实验结果匹配
- [ ] 所有图表都有标题和标签
- [ ] 无导致 overfull hbox 警告的长行
```

### 步骤 5.0：标题

标题是论文中阅读最多的元素。它决定是否有人点击摘要。

**好标题**：
- 陈述贡献或发现："Autoreason：迭代 LLM 精炼何时有效及为何失败"
- 突出令人惊讶的结果："扩展数据约束语言模型"（暗示你能做到）
- 命名方法 + 它做什么："DPO：语言模型的直接偏好优化"

**坏标题**：
- 太通用："一种改进语言模型输出的方法"
- 太长：超过约 15 个词的任何内容
- 仅行话："迭代随机策略精炼的渐近收敛"（这是给谁的？）

**规则**：
- 如果你有方法名，包含它（为了可引用性）
- 包含 1-2 个审阅者会搜索的关键词
- 除非两边都有意义，否则避免冒号
- 测试：审阅者能从标题单独知道领域和贡献吗？

### 步骤 5.1：摘要（5 句公式）

来自 Sebastian Farquhar (DeepMind)：

```
1. 你完成了什么："我们引入..."，"我们证明..."，"我们演示..."
2. 为什么这很难且重要
3. 你如何做（带专业关键词以便发现）
4. 你有什么证据
5. 你最显著的数字/结果
```

删除"大型语言模型取得了显著成功..."等通用开场。

### 步骤 5.2：图 1

图 1 是大多数读者第二眼看的东西（仅次于摘要）。在写引言之前起草它 — 它迫使你澄清核心思想。

| 图 1 类型 | 何时使用 | 示例 |
|---------------|-------------|---------|
| **方法图** | 新架构或流水线 | 显示系统的 TikZ 流程图 |
| **结果预告** | 一个令人信服的结果说明整个故事 | 条形图："我们 vs 基线" 清晰差距 |
| **问题说明** | 问题不直观 | 显示你修复的失败模式的之前/之后 |
| **概念图** | 抽象贡献需要视觉锚定 | 方法属性的 2x2 矩阵 |

**规则**：图 1 必须无需阅读任何文本就能理解。标题 alone 应该传达核心思想。有目的地使用颜色 — 不要只是装饰。

### 步骤 5.3：引言（最多 1-1.5 页）

必须包括：
- 清晰的问题陈述
- 简要方法概述
- 2-4 个要点贡献列表（双栏格式每项最多 1-2 行）
- 方法应在第 2-3 页开始

### 步骤 5.4：方法

实现可复现性：
- 概念大纲或伪代码
- 列出所有超参数
- 足以复现的架构细节
- 呈现最终设计决策；消融放在实验部分

### 步骤 5.5：实验与结果

对于每个实验，明确陈述：
- **它支持哪个声明**
- 它如何连接到主要贡献
- 要观察什么："蓝线显示 X，这证明 Y"

要求：
- 带方法学的误差线（标准差 vs 标准误差）
- 超参数搜索范围
- 计算基础设施（GPU 类型、总小时数）
- 种子设置方法

### 步骤 5.6：相关工作

按方法论组织，而不是逐篇。慷慨引用 — 审阅者可能写了相关论文。

### 步骤 5.7：局限性（要求）

所有主要会议都要求这个。诚实有帮助：
- 审阅者被指示不要因为诚实的局限性承认而惩罚
- 通过首先识别弱点来抢先批评
- 解释局限性为什么不削弱核心声明

### 步骤 5.8：结论与讨论

**结论**（必需，0.5-1 页）：
- 用不同于摘要的措辞重述贡献一句话（不同措辞）
- 总结关键发现（2-3 句，不是列表）
- 影响：这对领域意味着什么？
- 未来工作：2-3 个具体下一步（不是模糊的"我们将 X 留待未来工作"）

**讨论**（可选，有时与结论合并）：
- 直接结果之外的更广泛影响
- 与其他子领域的联系
- 方法何时有效何时无效的诚实评估
- 实际部署考虑

**不要**在结论中引入新结果或声明。

### 步骤 5.9：附录策略

所有主要场所的附录是无限的，对可复现性必不可少。结构：

| 附录部分 | 这里放什么 |
|-----------------|---------------|
| **证明与推导** | 主文本太长的完整证明。主文本可以陈述定理并"见附录 A"。 |
| **额外实验** | 消融、扩展曲线、每个数据集分解、超参数敏感性 |
| **实现细节** | 完整超参数表、训练详情、硬件规格、随机种子 |
| **数据集文档** | 数据收集过程、标注指南、许可、预处理 |
| **提示与模板** | 使用的确切提示（对于基于 LLM 的方法）、评估模板 |
| **人工评估** | 标注界面截图、给标注员的说明、IRB 详情 |
| **额外图表** | 每个任务分解、轨迹可视化、失败案例示例 |

**规则**：
- 主论文必须自包含 — 审阅者不需要阅读附录
- 永远不要只在附录中放关键证据
- 交叉引用："完整结果在表 5（附录 B）"而不仅仅是"见附录"
- 使用 `\appendix` 命令，然后 `\section{A: 证明}` 等

### 页面预算管理

超出页面限制时：

| 削减策略 | 节省 | 风险 |
|-------------|-------|------|
| 将证明移到附录 | 0.5-2 页 | 低 — 标准做法 |
| 精简相关工作 | 0.5-1 页 | 中 — 可能错过关键引用 |
| 用子图组合表格 | 0.25-0.5 页 | 低 — 通常提高可读性 |
| 谨慎使用 `\vspace{-Xpt}` | 0.1-0.3 页 | 低调则低风险，明显则高风险 |
| 删除定性示例 | 0.5-1 页 | 中 — 审阅者喜欢示例 |
| 减小图表尺寸 | 0.25-0.5 页 | 高 — 图表必须保持可读 |

**不要**：减小字体大小、改变边距、删除要求的部分（局限性、更广泛影响）或对正文使用 `\small`/`\footnotesize`。

### 步骤 5.10：伦理与更广泛影响声明

大多数场所现在要求或强烈鼓励伦理/更广泛影响声明。这不是模板 — 审阅者会阅读它，可以标记引发直接拒绝的伦理问题。

**要包括的内容：**

| 组件 | 内容 | 要求场所 |
|-----------|---------|-------------|
| **正面社会影响** | 你的工作如何惠及社会 | NeurIPS、ICML |
| **潜在负面影响** | 滥用风险、双重使用关注、失败模式 | NeurIPS、ICML |
| **公平与偏见** | 你的方法/数据有已知偏见吗？ | 所有场所（含蓄地） |
| **环境影响** | 大规模训练的计算碳足迹 | ICML、越来越 NeurIPS |
| **隐私** | 你的工作使用或启用个人数据处理吗？ | ACL、NeurIPS |
| **LLM 披露** | 写作或实验中使用了 AI 吗？ | ICLR（强制）、ACL |

**写声明：**

```latex
\section*{更广泛影响声明}
% NeurIPS/ICML：结论后，不计入页数限制

% 1. 积极应用（1-2 句）
这项工作实现了 [具体应用]，可能惠及 [具体群体]。

% 2. 风险和缓解（1-3 句，要具体）
[方法/模型] 可能被滥用于 [具体风险]。我们通过 [具体缓解，例如，仅发布大于 X 的模型权重，包含安全过滤器，记录失败模式] 来缓解这一点。

% 3. 影响声明的局限性（1 句）
我们的评估仅限于 [具体领域]；更广泛部署需要 [具体额外工作]。
```

**常见错误：**
- 写"我们预见不到负面影响"（几乎从来不是真的 — 审阅者不信任这）
- 含糊其辞："这可能被滥用"而没有具体说明如何
- 忽略大规模工作的计算成本
- 在要求它的场所忘记披露 LLM 使用

**计算碳足迹**（训练密集型论文）：
```python
# 使用 ML CO2 Impact 工具方法论估算
gpu_hours = 1000  # 总 GPU 小时
gpu_tdp_watts = 400  # 例如 A100 = 400W
pue = 1.1  # 电力使用效率（数据中心开销）
carbon_intensity = 0.429  # kg CO2/kWh（美国平均；因地区而异）

energy_kwh = (gpu_hours * gpu_tdp_watts * pue) / 1000
carbon_kg = energy_kwh * carbon_intensity
print(f"Energy: {energy_kwh:.0f} kWh, Carbon: {carbon_kg:.0f} kg CO2eq")
```

### 步骤 5.11：数据集与模型卡片（如适用）

如果你的论文引入了**新数据集**或**发布模型**，包括结构化文档。审阅者越来越期望这个，NeurIPS 数据集与基准测试 track 要求它。

**数据集文档** (Gebru et al., 2021) — 放在附录：

```
数据集文档（附录）：
- 动机：为什么创建这个数据集？支持什么任务？
- 组成：实例是什么？多少？什么数据类型？
- 收集：数据如何收集？来源是什么？
- 预处理：应用了什么清洁/过滤？
- 分布：数据集如何分发？什么许可？
- 维护：谁维护它？如何报告问题？
- 伦理考虑：包含个人数据吗？获得同意了吗？有害可能？已知偏见？
```

**模型卡片** (Mitchell et al., 2019) — 模型发布的附录：

```
模型卡片（附录）：
- 模型详情：架构、训练数据、训练程序
- 预期用途：主要用例、非预期用途
- 指标：评估指标和基准测试结果
- 伦理考虑：已知偏见、公平性评估
- 局限性：已知失败模式、方法表现不佳的领域
```

### 写作风格

**句子层面的清晰度（Gopen & Swan 的 7 个原则）：**

| 原则 | 规则 |
|-----------|------|
| 主语-动词接近 | 保持主语和动词靠近 |
| 强调位置 | 将强调放在句末 |
| 话题位置 | 先放上下文，新信息放后面 |
| 旧先于新 | 熟悉信息 → 不熟悉信息 |
| 一个单元，一个功能 | 每个段落做一个要点 |
| 动词语义 | 使用动词，不用名词化 |
| 上下文先于新 | 先设定舞台再呈现 |

**词汇选择（Lipton、Steinhardt）：**
- 要具体："accuracy" 而非 "performance"
- 消除委婉：除非真正不确定，否则去掉 "may"
- 全文术语一致
- 避免增量词汇："develop" 而非 "combine"

### 使用 LaTeX 模板

**始终先复制整个模板目录，然后在其中写作。**

```
模板设置检查清单：
- [ ] 步骤 1：将整个模板目录复制到新项目
- [ ] 步骤 2：验证模板原样编译（在任何更改之前）
- [ ] 步骤 3：阅读模板的示例内容以了解结构
- [ ] 步骤 4：逐部分替换示例内容
- [ ] 步骤 5：使用模板宏（检查 preamble 中的 \newcommand 定义）
- [ ] 步骤 6：只在最后清理模板 artifacts
```

**步骤 1：复制完整模板**

```bash
cp -r templates/neurips2025/ ~/papers/my-paper/
cd ~/papers/my-paper/
ls -la  # 应该看到：main.tex、neurips.sty、Makefile 等
```

复制**整个**目录，而不仅仅是 .tex 文件。模板包括样式文件（.sty）、参考文献样式（.bst）、示例内容和 Makefile。

**步骤 2：首先验证模板编译**

在**进行任何更改之前**：
```bash
latexmk -pdf main.tex
# 或手动：pdflatex main.tex && bibtex main && pdflatex main.tex && pdflatex main.tex
```

如果未修改的模板不编译，首先修复（通常缺少 TeX 包 — 通过 `tlmgr install <package>` 安装）。

**步骤 3：保留模板内容作为参考**

不要立即删除示例内容。将其注释掉并用作格式参考：

```latex
% 模板示例（保留供参考）：
% \begin{figure}[t]
%   \centering
%   \includegraphics[width=0.8\linewidth]{example-image}
%   \caption{Template shows caption style}
% \end{figure}

% 你的实际图表：
\begin{figure}[t]
  \centering
  \includegraphics[width=0.8\linewidth]{your-figure.pdf}
  \caption{你的标题遵循相同的风格。}
\end{figure}
```

**步骤 4：逐部分替换内容**

系统地处理：标题/作者 → 摘要 → 引言 → 方法 → 实验 → 相关工作 → 结论 → 参考文献 → 附录。每部分后编译。

**步骤 5：使用模板宏**

```latex
\newcommand{\method}{YourMethodName}  % 一致的方法命名
\newcommand{\eg}{e.g.,\xspace}        % 正确的缩写
\newcommand{\ie}{i.e.,\xspace}
```

### 模板陷阱

| 陷阱 | 问题 | 解决方案 |
|---------|---------|----------|
| 只复制 .tex 文件 | 缺少 .sty，无法编译 | 复制整个目录 |
| 修改 .sty 文件 | 破坏会议格式 | 永远不要编辑样式文件 |
| 添加随机包 | 冲突，破坏模板 | 仅在必要时添加 |
| 过早删除模板内容 | 失去格式参考 | 保留为注释直到完成 |
| 不经常编译 | 错误累积 | 每部分后编译 |
| 图表用栅格 PNG | 论文中模糊 | 始终使用矢量 PDF 通过 `savefig('fig.pdf')` |

### 表格和图表

**表格** — 使用 `booktabs` 以获得专业格式：

```latex
\usepackage{booktabs}
\begin{tabular}{lcc}
\toprule
Method & Accuracy $\uparrow$ & Latency $\downarrow$ \\
\midrule
Baseline & 85.2 & 45ms \\
\textbf{Ours} & \textbf{92.1} & 38ms \\
\bottomrule
\end{tabular}
```

规则：
- 每个指标加粗最佳值
- 包含方向符号（$\uparrow$ 越高越好，$\downarrow$ 越低越好）
- 数值列右对齐
- 一致的小数精度

**图表**：
- 所有图表和图表使用**矢量图形**（PDF、EPS）— `plt.savefig('fig.pdf')`
- **栅格**（PNG 600 DPI）仅用于照片
- **色盲安全调色板**（Okabe-Ito 或 Paul Tol）
- 验证**灰度可读性**（8% 的男性有色觉缺陷）
- **图表内不要有标题** — 标题提供此功能
- **自包含标题** — 读者无需正文就能理解

## 阶段 6：自我审阅与修改

**目标**：在投稿前模拟审阅过程。尽早捕获弱点。

### 步骤 6.1：模拟审阅（集成模式）

从多个角度生成审阅。关键洞察来自自动化研究流水线（特别是 SakanaAI 的 AI-Scientist）：**带元审阅者的集成审阅比单次审阅产生更校准的反馈。**

**步骤 1：生成 N 个独立审阅**（N=3-5）

使用不同模型或温度设置。每个审阅者只看到论文，不看其他审阅。**默认使用负面偏见** — LLMs 在评估中有充分记录的积极偏见。

```
你是 [VENUE] 的专家审阅者。你是批评性和彻底的。
如果论文有弱点或你对声明不确定，清楚地标记它并将其反映在你的分数中。不要给予怀疑的好处。

根据官方审阅者指南审阅这篇论文。评估：

1. 合理性（声明有良好支持吗？基线公平且强吗？）
2. 清晰度（论文写得好吗？专家可以复现吗？）
3. 重要性（这对社区重要吗？）
4. 原创性（新见解，而不仅仅是增量组合？）

以结构化 JSON 形式提供你的审阅：
{
  "summary": "2-3 句摘要",
  "strengths": ["优势 1", "优势 2", ...],
  "weaknesses": ["弱点 1（最关键）", "弱点 2", ...],
  "questions": ["给作者的 问题 1", ...],
  "missing_references": ["应该引用的论文", ...],
  "soundness": 1-4,
  "presentation": 1-4,
  "contribution": 1-4,
  "overall": 1-10,
  "confidence": 1-5
}
```

**步骤 2：元审阅（区域主席聚合）**

将所有 N 个审阅反馈给元审阅者：

```
你是 [VENUE] 的区域主席。你收到了论文的 [N] 个独立审阅。你的工作是：

1. 识别审阅者之间的共识优势和弱点
2. 通过直接检查论文解决分歧
3. 产生代表集体判断的元审阅
4. 对所有审阅的平均数值分数

保守：如果审阅者不同意弱点是否严重，
将其视为严重的，直到作者解决它。

审阅：
[review_1]
[review_2]
...
```

**步骤 3：反思循环**（可选，2-3 轮）

每个审阅者可以在看到元审阅后细化他们的审阅。使用早期终止哨兵：如果审阅者回复"我完成了"（无更改），停止迭代。

**用于审阅的模型选择**：审阅最好用可用的最强模型，即使你用更便宜的模型写论文。审阅模型应该独立于写作模型选择。

**少样本校准**：如果有的话，包括目标场所的 1-2 个真实发布审阅作为示例。这显著改善分数校准。

### 步骤 6.2：视觉审阅轮次（VLM）

纯文本审阅遗漏了一整类问题：图表质量、布局问题、视觉一致性。如果你有权访问视觉模型，在编译的 PDF 上运行单独的**视觉审阅**：

```
你正在审阅这篇研究论文 PDF 的视觉呈现。
检查：
1. 图表质量：图表可读吗？标签清晰吗？颜色可区分吗？
2. 图表-标题对齐：每个标题准确描述其图表吗？
3. 布局问题：孤立的章节标题、尴尬的页面中断、图表远离其引用
4. 表格格式：列对齐、一致的小数精度、最佳结果加粗
5. 视觉一致性：所有图表使用相同配色方案、一致的字体大小
6. 灰度可读性：如果黑白打印图表可理解吗？

对于每个问题，指定页码和确切位置。
```

这捕获了基于文本的审阅无法捕获的问题：图表带有不可读的轴标签、图表放置在首次引用 3 页之后、图 2 和图 5 之间不一致的配色方案，或者明显比列宽的表格。

### 步骤 6.3：声明验证轮次

在模拟审阅后，运行单独的验证轮次。这捕获审阅者可能遗漏的事实错误：

```
声明验证协议：
1. 从论文中提取每个事实声明（数字、比较、趋势）
2. 对于每个声明，将其追踪到支持它的特定实验/结果
3. 验证论文中的数字与实际结果文件匹配
4. 将任何无法追踪的声明标记为 [VERIFY]
```

对于基于 agent 的工作流程：委托验证给**新鲜子 agent**，它只接收论文文本和原始结果文件。新鲜上下文防止确认偏见 — 验证者不"记住"结果应该是什么。

### 步骤 6.4：优先处理反馈

收集审阅后，分类：

| 优先级 | 行动 |
|----------|--------|
| **关键**（技术缺陷、缺失基线） | 必须修复。可能需要新实验 → 返回阶段 2 |
| **高**（清晰度问题、缺失消融） | 应该在这次修改中修复 |
| **中**（轻微写作问题、额外实验） | 时间允许则修复 |
| **低**（风格偏好、切向建议） | 记下供将来工作 |

### 步骤 6.5：修改周期

对于每个关键/高优先级问题：
1. 识别受影响的特定部分
2. 起草修复
3. 验证修复不会破坏其他声明
4. 更新论文
5. 重新检查审阅者的担忧

### 步骤 6.6：回复信写作

回复实际审阅（投稿后）时，回复是不同于修改的技能：

**格式**：逐点。对于每个审阅者担忧：
```
> R1-W1："论文缺少与方法 X 的比较。"

感谢审阅者的建议。我们添加了在 [指标] 上比 X 高 3.2pp 的比较（p<0.05）。我们注意到 X 需要 2 倍我们的计算预算。
```

**规则**：
- 解决每个担忧 — 审阅者注意到你是否跳过了一个
- 用最强的回复领先
- 简洁直接 — 审阅者阅读大量回复
- 如果在回复期间运行了实验，附上新结果
- 永远不要防御性或轻视，即使是很弱的批评
- 使用 `latexdiff` 生成显示更改的标记 PDF（参见专业 LaTeX 工具部分）
- 对于具体、可操作的反馈感谢审阅者（不是通用赞扬）

**不要做**：没有证据的"我们恭敬地不同意"。没有解释的"这超出范围"。忽略弱点，只回复优势。

### 步骤 6.7：论文演变追踪

在关键里程碑保存快照：
```
paper/
  paper.tex                    # 当前工作版本
  paper_v1_first_draft.tex     # 第一完整草稿
  paper_v2_post_review.tex     # 审阅后
  paper_v3_pre_submission.tex  # 投稿前最终
  paper_v4_camera_ready.tex    # 录用后最终
```

## 阶段 7：投稿准备

**目标**：最终检查、格式和投稿。

### 步骤 7.1：会议检查清单

每个场所都有强制性检查清单。仔细完成它们 — 不完整的检查清单可能导致直接拒绝。

### 步骤 7.2：匿名化检查清单

双盲审阅意味着审阅者不能知道谁写了论文。检查所有这些：

```
匿名化检查清单：
- [ ] PDF 中任何地方都没有作者姓名或机构
- [ ] 没有致谢部分（录用后添加）
- [ ] 自我引用用第三人称写："Smith 等人 [1] 表明..."而非"我们之前表明 [1]..."
- [ ] 没有指向你个人仓库的 GitHub/GitLab URL
- [ ] 对代码链接使用匿名 GitHub (https://anonymous.4open.science/)
- [ ] 图表中没有机构徽标或标识符
- [ ] PDF 属性中没有包含作者姓名的文件元数据
- [ ] 没有"我们之前的工作"或"在我们较早的论文中"的措辞
- [ ] 数据集名称不透露机构（如需要重命名）
- [ ] 补充材料不包含身份信息
```

**常见错误**：补充代码中可见的 Git 提交消息、带机构工具水印的图表、从先前草稿保留的致谢、匿名期间的 arXiv 预印本。

### 步骤 7.3：格式验证

```
投稿前格式检查：
- [ ] 遵守页数限制（不含参考文献和附录）
- [ ] 所有图表是矢量（PDF）或高分辨率栅格（600 DPI PNG）
- [ ] 所有图表在灰度下可读
- [ ] 所有表格使用 booktabs
- [ ] 参考文献正确编译（引用中无"？"）
- [ ] 关键区域无 overfull hbox
- [ ] 附录清晰标记和分隔
- [ ] 存在要求的部分（局限性、更广泛影响等）
```

### 步骤 7.4：预编译验证

在尝试 `pdflatex` 之前运行这些自动化检查。在这里捕获错误比调试编译器输出更快。

```bash
# 1. 用 chktex lint（捕获常见 LaTeX 错误）
# 抑制嘈杂警告：-n2（句子结尾）、-n24（括号）、-n13（句间）、-n1（命令终止）
chktex main.tex -q -n2 -n24 -n13 -n1

# 2. 验证所有引用存在于 .bib
# 从 .tex 提取 \cite{...}，对照 .bib 检查每个
python3 -c "
import re
tex = open('main.tex').read()
bib = open('references.bib').read()
cites = set(re.findall(r'\\\\cite[tp]?{([^}]+)}', tex))
for cite_group in cites:
    for cite in cite_group.split(','):
        cite = cite.strip()
        if cite and cite not in bib:
            print(f'WARNING: \\\\cite{{{cite}}} not found in references.bib')
"

# 3. 验证所有引用的图表在磁盘上存在
python3 -c "
import re, os
tex = open('main.tex').read()
figs = re.findall(r'\\\\includegraphics(?:\[.*?\])?{([^}]+)}', tex)
for fig in figs:
    if not os.path.exists(fig):
        print(f'WARNING: Figure file not found: {fig}')
"

# 4. 检查重复 \label 定义
python3 -c "
import re
from collections import Counter
tex = open('main.tex').read()
labels = re.findall(r'\\\\label{([^}]+)}', tex)
dupes = {k: v for k, v in Counter(labels).items() if v > 1}
for label, count in dupes.items():
    print(f'WARNING: Duplicate label: {label} (appears {count} times)')
"
```

修复任何警告后再继续。对于基于 agent 的工作流程：将 chktex 输出反馈给 agent 并指示进行最小修复。

### 步骤 7.5：最终编译

```bash
# 干净构建
rm -f *.aux *.bbl *.blg *.log *.out *.pdf
latexmk -pdf main.tex

# 或手动（三次 pdflatex + bibtex 用于交叉引用）
pdflatex -interaction=nonstopmode main.tex
bibtex main
pdflatex -interaction=nonstopmode main.tex
pdflatex -interaction=nonstopmode main.tex

# 验证输出存在且有内容
ls -la main.pdf
```

**如果编译失败**：解析 `.log` 文件的第一个错误。常见修复：
- "Undefined control sequence" → 缺少包或命令名拼写错误
- "Missing $ inserted" → 数学符号在数学模式外
- "File not found" → 图表路径错误或缺少 .sty 文件
- "Citation undefined" → .bib 条目缺失或未运行 bibtex

### 步骤 7.6：会议特定要求

| 场所 | 特殊要求 |
|-------|---------------------|
| **NeurIPS** | 附录中的论文检查清单，录用后如果有 |
| **ICML** | 更广泛影响声明（结论后，不计入限制） |
| **ICLR** | 需要 LLM 披露，互惠审阅协议 |
| **ACL** | 强制局限性部分，负责任自然语言处理检查清单 |
| **AAAI** | 严格样式文件 — 绝对不要修改 |
| **COLM** | 为语言模型社区重新定位贡献 |

### 步骤 7.7：会议重新投稿和格式转换

在场所之间转换时，**永远不要在模板之间复制 LaTeX 前言**：

```bash
# 1. 从目标模板全新开始
cp -r templates/icml2026/ new_submission/

# 2. 只复制内容部分（不是前言）
#    - 摘要文本、部分内容、图表、表格、bib 条目

# 3. 调整页数限制
# 4. 添加场所特定要求部分
# 5. 更新参考文献
```

| 从 → 到 | 页面更改 | 关键调整 |
|-----------|-------------|-------------------|
| NeurIPS → ICML | 9 → 8 | 削减 1 页，添加更广泛影响 |
| ICML → ICLR | 8 → 9 | 扩展实验，添加 LLM 披露 |
| NeurIPS → ACL | 9 → 8 | 为自然语言处理惯例重新结构，添加局限性 |
| ICLR → AAAI | 9 → 7 | 大量削减，严格遵循样式 |
| 任何 → COLM | 各异 → 9 | 为语言模型焦点重新定位 |

削减页面：将证明移到附录，精简相关工作，组合表格，使用子图。
扩展：添加消融，扩展局限性，添加额外基线，添加定性示例。

**拒绝后**：在新版本中解决审阅者担忧，但不要包含"更改"部分或引用先前投稿（盲审）。

### 步骤 7.8：终稿准备（录用后）

录用后，准备终稿版本：

```
终稿检查清单：
- [ ] 去匿名化：添加作者姓名、机构、电子邮件地址
- [ ] 添加致谢部分（资助、计算补助、有帮助的审阅者）
- [ ] 添加公开代码/数据 URL（真实 GitHub，不是匿名的）
- [ ] 解决元审阅者的任何强制修订
- [ ] 如果适用，将模板切换到终稿模式（例如 AAAI \anon → \camera）
- [ ] 添加任何必要的版权声明
- [ ] 更新文本中任何"匿名"占位符
- [ ] 验证最终 PDF 编译干净
- [ ] 检查终稿的页数限制（有时与投稿不同）
- [ ] 将补充材料（代码、数据、附录）上传到场所门户
```

### 步骤 7.9：arXiv 和预印本策略

在 arXiv 上发布是机器学习中的标准做法，但有重要的时机和匿名考虑。

**时机决策树：**

| 情况 | 建议 |
|-----------|---------------|
| 投向双盲场所（NeurIPS、ICML、ACL） | 在投稿截止日期**之后**发布到 arXiv，不要之前。在此之前发布在技术上可能违反匿名政策，尽管执行情况各异。 |
| 投向 ICLR | ICLR 明确允许在投稿前发布 arXiv。但不要在投稿本身中放作者姓名。 |
| 论文已在 arXiv，投向新场所 | 大多数场所可接受。不要在审阅期间用引用审阅反馈的更改更新 arXiv 版本。 |
| 研讨会论文 | 任何时候发布 arXiv 都可以 — 研讨会通常不是双盲的。 |
| 想建立优先权 | 如果担心被抢先立即发布 — 但接受匿名权衡。 |

**arXiv 类别选择**（机器学习/人工智能论文）：

| 类别 | 代码 | 最适合 |
|----------|------|----------|
| 机器学习 | `cs.LG` | 通用机器学习方法 |
| 计算与语言 | `cs.CL` | 自然语言处理、语言模型 |
| 人工智能 | `cs.AI` | 推理、规划、agent |
| 计算机视觉 | `cs.CV` | 视觉模型 |
| 信息检索 | `cs.IR` | 搜索、推荐 |

**列出主要 + 1-2 个交叉列表类别。** 更多类别 = 更多可见度，但只在真正相关的地方交叉列表。

**版本策略：**
- **v1**：初始投稿（与会议投稿匹配）
- **v2**：录用后带终稿更正（在摘要中添加"accepted at [VENUE]"）
- 不要在审阅期间用明显回应审阅反馈的更改发布 v2

### 步骤 7.10：研究代码打包

发布干净、可运行的代码显著增加引用和审阅者信任。在终稿提交时打包代码。

**仓库结构：**

```
your-method/
  README.md              # 设置、使用、再现说明
  requirements.txt       # 或用于 conda 的 environment.yml
  setup.py               # 用于 pip 可安装包
  LICENSE                # 推荐 MIT 或 Apache 2.0 用于研究
  configs/               # 实验配置
  src/                   # 核心方法实现
  scripts/               # 训练、评估、分析脚本
    train.py
    evaluate.py
    reproduce_table1.sh  # 每个主要结果一个脚本
  data/                  # 小数据或下载脚本
    download_data.sh
  results/               # 用于验证的预期输出
```

**研究代码的 README 模板：**

```markdown

"[论文标题]"（场所 年份）的官方实现。

## 设置

## Hermites Agent 集成

此技能专为 Hermites agent 设计。它使用 Hermites 工具、委托、调度和记忆进行完整研究生命周期。

### 相关技能

组合此技能与其他 Hermites 技能用于特定阶段：

| 技能 | 何时使用 | 如何加载 |
|-------|-------------|-------------|
| **arxiv** | 阶段 1（文献综述）：搜索 arXiv、生成 BibTeX、通过 Semantic Scholar 找相关论文 | `skill_view("arxiv")` |
| **subagent-driven-development** | 阶段 5（起草）：通过两阶段审阅（规格合规然后质量）并行部分写作 | `skill_view("subagent-driven-development")` |
| **plan** | 阶段 0（设置）：在执行前创建结构化计划。写入 `.hermes/plans/` | `skill_view("plan")` |
| **qmd** | 阶段 1（文献）：通过混合 BM25+向量搜索搜索本地知识库（笔记、转录、文档） | 安装：`skill_manage("install", "qmd")` |
| **diagramming** | 阶段 4-5：创建基于 Excalidraw 的图表和架构图 | `skill_view("diagramming")` |
| **data-science** | 阶段 4（分析）：用于交互式分析和可视化的 Jupyter 实时内核 | `skill_view("data-science")` |

**此技能取代了 `ml-paper-writing`** — 它包含 ml-paper-writing 的全部内容，加上完整的实验/分析流水线和 autoreason 方法论。

### Hermites 工具参考

| 工具 | 此流水线中的用法 |
|------|----------------------|
| **`terminal`** | LaTeX 编译（`latexmk -pdf`）、git 操作、启动实验（`nohup python run.py &`）、进程检查 |
| **`process`** | 后台实验管理：`process("start", ...)`、`process("poll", pid)`、`process("log", pid)`、`process("kill", pid)` |
| **`execute_code`** | 运行 Python 进行引用验证、统计分析、数据聚合。有工具通过 RPC 访问。 |
| **`read_file`** / **`write_file`** / **`patch`** | 论文编辑、实验脚本、结果文件。使用 `patch` 对大型 .tex 文件进行有针对性的编辑。 |
| **`web_search`** | 文献发现：`web_search("transformer attention mechanism 2024")` |
| **`web_extract`** | 获取论文内容、验证引用：`web_extract("https://arxiv.org/abs/2303.17651")` |
| **`delegate_task`** | **并行部分起草** — 为每个部分生成隔离的子 agent。也用于并发引用验证。 |
| **`todo`** | 跨会话的主要状态跟踪器。每阶段转换后更新。 |
| **`memory`** | 跨会话持久化关键决策：贡献框架、场所选择、审阅者反馈。 |
| **`cronjob`** | 安排实验监控、截止日期倒计时、自动化学术检查。 |
| **`clarify`** | 当被阻塞时向用户提出有针对性的问题（场所选择、贡献框架）。 |
| **`send_message`** | 当实验完成或草稿准备好审阅时通知用户，即使用户不在聊天中。 |

### 工具使用模式

**实验监控**（最常见）：
```
terminal("ps aux | grep <pattern>")
→ terminal("tail -30 <logfile>")
→ terminal("ls results/")
→ execute_code("分析结果 JSON，计算指标")
→ terminal("git add -A && git commit -m '<descriptive message>' && git push")
→ send_message("实验完成：<summary>")
```

**并行部分起草**（使用委托）：
```
delegate_task("根据这些实验脚本和配置起草方法部分。
  包括：伪代码、所有超参数、足以复现的架构细节。
  使用 neurips2025 模板约定用 LaTeX 编写。")

delegate_task("起草相关工作部分。使用 web_search 和 web_extract
  找论文。验证每个引用通过 Semantic Scholar。按方法论分组。")

delegate_task("起草实验部分。读取 results/ 中的所有结果文件。
  陈述每个实验支持哪个声明。包括误差线和显著性。")
```

每个委托作为**新鲜子 agent** 运行，没有共享上下文 — 在提示中提供所有必要信息。收集输出并整合。

### 使用 `memory` 和 `todo` 进行状态管理

**`memory` 工具** — 持久化关键决策（有限：~2200 字符用于 MEMORY.md）：

```
memory("add", "论文：autoreason。场所：NeurIPS 2025（9 页）。
  贡献：在生成-评估差距大时结构化精炼有效。
  关键结果：Haiku 42/42，Sonnet 3/5，S4.6 约束 2/3。
  状态：阶段 5 — 起草方法部分。")
```

在主要决策或阶段转换后更新记忆。这在会话之间持久化。

**`todo` 工具** — 跟踪细粒度进度：

```
todo("add", "为 Sonnet 4.6 设计约束任务实验")
todo("add", "运行 Haiku 基线比较")
todo("add", "起草方法部分")
todo("update", id=3, status="in_progress")
todo("update", id=1, status="completed")
```

    1. ps aux | grep run_experiment

    3. ls results/haiku_baselines/
    4. 如果完成：读取结果，计算 Borda 分数，
       git add -A && git commit -m '添加 Haiku 结果' && git push
    5. 报告：结果表、关键发现、下一步
    6. 如果没有变化：用 [SILENT]"
})
```

**`[SILENT]` 协议**：当自上次检查以来没有任何变化时，精确响应 `[SILENT]`。这抑制向用户传递通知。只有在有真正需要知道的更改时才报告。

## 常见问题和解决方案

| 问题 | 解决方案 |
|-------|----------|
| 摘要太通用 | 删除第一句如果它可以是任何机器学习论文的前缀。从你的具体贡献开始。 |
| 引言超过 1.5 页 | 将背景拆分到相关工作。将贡献要点前置。 |
| 实验缺少明确声明 | 添加："这个实验测试是否 [具体声明]..." 在每个实验之前。 |
| 审阅者发现论文难以理解 | 添加路标，使用一致术语，使图表标题自包含。 |
| 缺少统计显著性 | 添加误差线、运行次数、统计检验、置信区间。 |
| 实验范围蔓延 | 每个实验必须映射到特定声明。削减不映射的实验。 |
| 论文被拒，需要重新投稿 | 参见会议重新投稿部分。解决审阅者担忧而不引用审阅。 |
| 缺少更广泛影响声明 | 参见步骤 5.10。大多数场所要求。"没有负面影响"几乎从来不可信。 |
| 人工评估被批为弱 | 参见步骤 2.5 和人工评估参考。报告一致性指标、标注员详情、报酬。 |
| 审阅者质疑可复现性 | 发布代码（步骤 7.10）、记录所有超参数、包括种子和计算详情。 |
| 理论论文缺乏直觉 | 在形式证明之前添加证明草图和通俗语言解释。 |
| 结果为负面/无效 | 参见处理负面结果部分。考虑研讨会、TMLR 或重新定位为分析。 |
