# Simplify Code — 并行审核与清理

使用三个并行运行的专注审核者审核你最近的代码变更，汇总他们的发现，并应用值得应用的修复。

**核心原则：** 三个窄范围审核者胜过一个宽范围审核者。每个审核者深入搜索代码库中某一类问题——复用、质量、效率——而不将注意力分散到所有三个类别上。它们并发运行，因此你只需支付一次审核的延迟，而非三次。

## 何时使用

当用户说出以下任何内容时触发此技能：

- "simplify" / "simplify my changes" / "simplify these changes"
- "review my code" / "review my recent changes" / "clean up my changes"
- "/simplify"（如果他们习惯了 Claude Code 的用法）

用户可能添加的可选修饰符——尊重它们：

- **Focus：** "simplify focus on efficiency" → 仅运行效率审核者
  （或将汇总偏向它）。已识别的焦点：`reuse`、`quality`、`efficiency`。
- **Dry run：** "simplify but don't change anything" / "just report" → 运行三个审核者，展示发现，不应用任何内容。在应用前询问。
- **Scope：** "simplify the last commit" / "simplify staged" / "simplify src/foo.py" → 相应缩小差异来源（见阶段 1）。

不要在每次编辑后自动运行。它会消耗三个子代理的 token——仅在用户明确要求时调用。

## 流程

### 阶段 1 — 识别变更

捕获要审核的差异。根据用户的要求选择来源，默认顺序如下：

```bash
# 1. 默认：未提交的工作树变更（已跟踪文件）
git diff

# 2. 如果为空，包含暂存变更
git diff HEAD

# 3. 用户可能请求的范围变体：
git diff --staged                 # "暂存变更"
git diff HEAD~1                    # "上次提交"
git diff main...HEAD              # "此分支" / "我的 PR"
git diff -- src/foo.py            # 特定文件
```

如果 `git diff` 和 `git diff HEAD` 都为空且没有 git 仓库或没有变更，回退到用户明确命名的文件或此会话中最近创建/编辑的文件。如果确实找不到任何变更的代码，说明并停止——没有可以简化的内容。

捕获完整的差异文本。注意其大小：如果非常大（例如 >2000 行变更），警告用户三个子代理各自携带完整差异会消耗大量 token，并提供范围缩减方案（按目录、按提交）后再继续。

### 阶段 2 — 并行启动三个审核者

使用 `delegate_task` **批处理模式**——在一个 `tasks` 数组中传递所有三个任务，使它们并发运行。三个是此模式的正确扇出；在任何默认安装上都在 `delegation.max_concurrent_children` 预算范围内。

给**每个**审核者提供**完整差异**（不是片段——跨文件问题隐藏在空白处）加上绝对仓库路径，以便他们搜索更广泛的代码库。每个审核者获得 `terminal`、`file` 和 `search` 工具集（以便他们可以 `git`、`read_file` 和 `search_files`/grep）。

告诉每个审核者：
- 搜索现有代码库中的证据（不要仅从差异推理）。
- **应用 Chesterton's Fence 原则：** 在标记任何内容删除之前，对该行运行 `git blame` 以理解其存在的原因。如果无法确定原始目的，标记为 `confidence: low`——不要猜测。
- 以结构化输出报告发现，包含置信度和风险：
  ```
  file:line → problem → suggested fix | confidence: high/medium/low | risk: SAFE/CAREFUL/RISKY
  ```
  - **SAFE** = 证明不影响行为（未使用的导入、注释掉的代码、透传包装器）。自动应用。
  - **CAREFUL** = 改善但不改变语义（重命名局部变量、展平嵌套三元运算符、提取辅助函数）。应用后进行测试验证。
  - **RISKY** = 可能改变行为或破坏公共契约（N+1 重构、公共 API 重命名、内存生命周期更改）。标记供人工审核——不要自动应用。
- 跳过格式和风格问题。仅标记实质性改善代码的内容。

传递这三个目标（排除用户焦点排除的内容）：

**审核者 1 — 代码复用**
> 审核此差异中重复代码库中已有功能的代码。搜索工具模块、共享辅助函数和相邻文件（使用 search_files / grep）查找现有函数、常量或模式，新代码可以调用它们而非重新实现。标记：重复现有功能的新函数；已由现有工具实现的手写逻辑（手动字符串/路径操作、自定义环境检查、临时类型守卫、重新实现的解析）。对于每个，命名要使用的现有工具及其位置。

**审核者 2 — 代码质量**
> 审核此差异中的质量问题。寻找：冗余状态（重复或可以从现有状态派生的值；不需要存在的缓存）；参数膨胀（在函数应该重构的地方添加新参数）；带变化的复制粘贴（应该共享抽象的近乎重复块）；泄漏的抽象（暴露内部、破坏现有封装边界）；字符串类型化代码（已存在常量/枚举/注册表的原始字符串——在标记前检查规范注册表）；AI 生成的粗劣模式（在明显代码上方添加重新陈述的额外注释如 `// increment counter`；对已验证输入的不必要的防御性空检查；绕过类型系统的 `as any` 转换；与文件其余部分不一致的模式）。对于每个，给出具体的重构。

**审核者 3 — 效率**
> 审核此差异中的效率问题。寻找：不必要的工作（冗余计算、重复文件读取、重复 API 调用、N+1 访问模式）；错过的并发（独立操作顺序运行）；热路径膨胀（启动或每个请求路径上的重量/阻塞操作）；TOCTOU 反模式（操作前的存在预检查而非直接执行操作并处理错误）；内存问题（无界增长、缺少清理、监听器/句柄泄漏）；过于宽泛的读取（加载整个文件而非仅需要的切片）；静默失败（空 catch 块、忽略的错误返回、`except: pass`、`.catch(() => {})` 无处理、错误传播缺口——这些隐藏 bug，至少应该在吞没前记录）。对于每个，给出具体修复及其更快或更安全的原因。

### 阶段 3 — 汇总和应用

等待所有三个返回（批处理模式一起返回它们）。

1. **合并** 发现到一个列表中，在审核者重叠的地方去重。
2. **丢弃误报**——你有最多上下文；你不需要与审核者争论，只需静默丢弃弱或错误的建议。
3. **解决冲突。** 审核者可能不同意（审核者 1："使用现有工具 X"；审核者 3："X 很慢，内联它"）。默认解决顺序：**正确性 > 用户声明的焦点 > 可读性/复用 > 微观性能。** 不要应用损害清晰度的性能"修复"，除非路径确实是热路径。当两个建议互斥且都合理时，选择改动更少代码的那个并注明替代方案。
4. **按风险层级顺序应用：**
   - **SAFE 先**（自动应用）：未使用的导入、注释掉的代码、透传包装器、冗余类型断言。之后运行测试。
   - **CAREFUL 接着**（带验证应用，一次一个文件）：重命名局部变量、展平三元运算符、提取辅助函数、合并重复。每个文件后运行测试。回退任何破坏的。
   - **RISKY 最后**（标记供审核——不要自动应用）：N+1 重构、公共 API 更改、并发修复、错误处理更改。每个都附带风险描述和测试覆盖状态。如果用户选择干运行，展示所有三个层级且不应用任何内容。
5. **验证** 你没有破坏任何内容：运行项目针对已修改文件的定向测试（不是完整套件），并重新运行仓库使用的任何 linter/类型检查。如果修复破坏了测试，回退该修复并报告。
6. **总结** 你更改了什么：按审核者类别和风险层级分组的已应用修复简短列表，加上你有意跳过的任何发现及其原因。

## 陷阱

- **不要扇出超过约 3 个。** 更多审核者意味着更多成本和更多需要协调的冲突建议，而非更好的覆盖。三个类别已覆盖该空间。
- **给每个审核者完整差异。** 将差异拆分给审核者会违背设计——跨文件重复和 N+1 只有在完整视图中才会出现。
- **审核者搜索，不猜测。** 没有指向现有工具指针的复用发现（"可能有一个辅助函数"）是噪音。要求 `file:line` 证据；丢弃缺少证据的发现。
- **应用 ≠ 重写。** 这是对用户最近变更的清理，而非重构整个模块的许可。编辑保持在差异涉及的范围加上修复所需的最小周围更改。
- **尊重项目约定。** 如果仓库有 AGENTS.md / CLAUDE.md / HERMES.md 或 linter 配置，将这些规则融入审核者提示，使建议匹配代码风格而非与之对抗。
- **大差异撑爆上下文。** 如果差异很大，在委派前缩减范围——三个子代理各自携带 5000 行差异既昂贵又可能被截断。
- **过度信任死代码工具。** `knip`、`ts-prune` 和 `depcheck` 会标记确实被动态使用的导出（基于字符串的导入、反射）。在移除前始终 grep 符号名——干净的工具报告不是证明。
- **未检查公共契约就重命名。** 导出名、API 路由路径、数据库列名和配置键是契约——即使名称不好，重命名也会破坏消费者。将公共契约更改标记为 RISKY；永远不要自动重命名。
- **移除"不必要"的错误处理。** 空 catch 块或忽略的错误可能是有意的——在该上下文中错误是预期且良性的。标记它，不要移除；让人来决定。
