# 1-3-1 法则

技术提案和权衡分析的结构化决策框架。当用户面临多种方案选择时（架构决策、工具选择、重构策略、迁移路径），此技能生成 1-3-1 格式：一个清晰的问题陈述、三个各具优缺点的选项，以及一个包含完成定义和实施计划的明确建议。当用户请求"1-3-1"、说"给我选项"或需要帮助在竞争方案之间做出选择时使用。

# 1-3-1 沟通法则

当任务有多种可行方案且用户需要明确建议时使用的结构化决策格式。生成简洁的问题框架、三个带权衡的选项，以及推荐路径的可操作计划。

## 使用场景

- 用户明确请求"1-3-1"响应。
- 用户说"给我选项"或"我的选择是什么"以进行技术决策。
- 任务有多种可行方案，具有有意义的权衡（架构、工具、迁移策略）。
- 用户需要可以转发给团队或利益相关者的提案。

不要用于有单一明显答案的简单问题、调试会话，或用户已决定方案的任务。

## 步骤

1. **问题**（一句话）
   - 用一句简洁的话陈述核心决策或期望结果。
   - 关注*是什么*，而非*怎么做* — 不含实现细节、工具名称或特定技术。
   - 保持简洁。如果需要"和"，说明您在描述两个问题。

2. **选项**（恰好三个）
   - 提出三个标记为 A、B、C 的不同可行方案。
   - 每个选项有简要描述、优点和缺点。
   - 选项应代表真正不同的策略，而非同一方法的微小变体。

3. **建议**（一个选项）
   - 陈述您建议哪个选项及原因，基于用户的上下文和优先级。
   - 要直接 — 这是您的专业判断，不是含糊其辞。

4. **完成定义**
   - 列出推荐选项的具体成功标准。
   - 这些是具体、可验证的结果 — 不是模糊的愿望。
   - 如果用户选择不同选项，相应修改此部分。

5. **实施计划**
   - 执行推荐选项的具体步骤。
   - 在适用时包括具体命令、工具或操作。
   - 如果用户选择不同选项，相应修改此部分。

## 验证

- 响应恰好有一个问题句子。
- 响应恰好有三个选项（A、B、C），每个都有优缺点。
- 响应有一个单一建议，选择一个选项并附有理由。
- 完成定义和实施计划与推荐的选项一致。
- 如果用户选择不同选项，建议、完成定义和实施计划相应更新。

## 示例

用户："给我一个为 API 客户端添加重试逻辑的 1-3-1。"

代理：

**问题：** 对外部服务的 API 调用在负载下间歇性失败，需要优雅的重试处理。

**选项：**

- **选项 A：共享工具中的指数退避加抖动。**
  优点：简单、众所周知的模式；所有调用方共享单一实现。
  缺点：无熔断；调用方在重试期间仍会阻塞。

- **选项 B：将重试库（tenacity）作为每个端点调用的装饰器。**
  优点：可按端点配置；内置支持熔断和速率限制。
  缺点：增加依赖；装饰器模式可能掩盖控制流。

- **选项 C：基于队列的异步重试与死信处理。**
  优点：非阻塞；失败的调用被跟踪并可重放；扩展性好。
  缺点：显著增加复杂性；需要队列后端；对于低容量服务过度。

**建议：** 选项 A。调用量不足以证明队列基础设施的合理性，共享工具中的指数退避加抖动可以干净地覆盖故障模式，且无需新依赖。

**完成定义：**
- `api_retry()` 工具处理瞬态 HTTP 错误（429、502、503、504）。
- 使用指数退避和抖动重试最多 3 次。
- 所有外部 API 调用通过重试工具路由。
- 记录每次重试尝试，包含端点和尝试次数。
- 测试覆盖重试耗尽、直接成功和瞬态后成功场景。

**实施计划：**
1. 创建 `utils/api_retry.py`，包含可配置的最大重试次数、基础延迟和可重试状态码。
2. 使用 `random.uniform(0, base_delay)` 添加抖动以防止雷鸣般的群效应。
3. 使用重试工具包装 `api_client.py` 中的现有 API 调用。
4. 为每个重试场景添加模拟 HTTP 响应的单元测试。
5. 针对不稳定的端点模拟进行简单压力测试验证。
