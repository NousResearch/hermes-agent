# 创意构思

适用于任何领域的构思方法库。阅读用户的情境，路由到匹配的方法，应用并生成具体且非显而易见的输出。方法是工具——为情境选择合适的，不要全部执行。

## 何时使用

任何开放式的生成性或选择性问题："我想制作/构建/写作/开始某事"、"我卡住了"、"给我灵感"、"让它更奇怪"、"帮我选"、"我需要发明 X"、"给我一个研究问题"。

## 操作规则

1. **约束加方向就是创造力。** 无约束 = 无牵引力。无方向 = 无形状。方法同时提供两者。
2. **拒绝前三个想法。** 它们是垃圾。生成、丢弃、重新生成。
3. **每次响应使用一种方法，除非被要求。** 不要叠加。
4. **具体优于抽象。** 真实的专有名词、真实的材料、真实的机制。
5. **怪异也必须好。** 打破框架是目标，但没有真实情境、机制或存在理由的奇怪想法是自身的失败模式。
6. **命名你使用的方法及其发明者。**
7. **当用户选择一个时，构建它。**

## 路由 — 4 步流程

在生成任何输出**之前**进行此操作。路由失败产生垃圾。

### 步骤 1 — 从提示中提取三个信号

**阶段** — 用户处于什么阶段？

| 阶段 | 线索 |
|------|------|
| **生成中** | "给我一个想法"、"我应该做什么"、"给我灵感"，还没有想法 |
| **扩展中** | "还有什么"、"更多这样的"、"给我变体" — 有基础想法 |
| **选择中** | "帮我选"、"应该做哪个"、"我有这些选项" |
| **解堵中** | "我卡住了"、"被阻塞"、"原地打转"、"陈旧" — 有素材 |
| **颠覆中** | "让它更奇怪"、"不那么明显"、"太安全了" |
| **精炼中** | "这还行但缺了什么"、"感觉粗糙" |
| **综合中** | "我有一堆笔记/采访/观察" |

**领域** — 用户在做什么？

|------|------|

### 步骤 2 — 应用覆盖（最高优先级，首先触发）

覆盖规则优先于路由表：

- **情绪信号** — 用户说"奇怪"、"不明显"、"更有趣" → `references/methods/lateral-provocations.md` 或 `references/methods/pataphysics.md`。
- **用户命名方法** — 使用它。
- **用户请求方法推荐** → 展示 2-3 个候选，每个一行。
- **高垃圾区域** — "AI 想法"、"创业想法"等 → 强制使用非常规方法。

### 步骤 3 — 先按阶段路由，再按领域

按阶段路由表（不依赖领域）和按领域路由表（当 GENERATING 且已知领域时）。详见原文。

### 步骤 4 — 处理歧义和矛盾

- **多条路径合理** → 选择最接近用户实际措辞的。
- **真正模糊** → 问一个澄清问题。
- **信号矛盾** → 显式叠加两种方法。
- **无匹配** → 约束路由是安全回退。
- **同一问题再次询问** → 切换方法。

## 输出格式

对于约束路由默认路径：

```
## Constraint: [Name] — from [Source]
> [约束，一句话]

### Ideas

1. **[一句话提纲]**
   [2-3 句——具体制作什么，为什么有趣]
   ⏱ [周末/周/月]  •  🔧 [技术栈/媒介/材料]

2. ...
3. ...
```

每组想法，无论使用何种方法：
- 命名使用的方法。
- 给每个想法其具体机制和诚实的失败模式/权衡/受众。
- 至少标记一个想法为**落地**——可立即构建/追求、非显而易见但有真实第一步。
