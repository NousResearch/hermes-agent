# 对抗性用户体验测试

扮演您产品的最坏情况用户 — 一个讨厌技术、不想要您的软件、会找各种理由抱怨的人。然后通过务实过滤器过滤他们的反馈，以区分真正的用户体验问题与"我讨厌电脑"的噪音。

可以将其视为自动化的"妈妈测试" — 但更愤怒。

## 为什么这有效

大多数 QA 发现错误。这发现的是**摩擦**。技术正确的产品对真实人类可能仍然不可用。对抗角色发现：
- 对开发者有意义但对用户无意义的令人困惑的术语
- 完成基本任务步骤太多
- 缺少入职引导或"顿悟时刻"
- 可访问性问题（字体大小、对比度、点击目标）
- 冷启动问题（空状态、无演示内容）
- 付费墙/注册摩擦导致转化率下降

**务实过滤器**（第 3 阶段）是使这有用而非只是有趣的关键。没有它，您会为每个屏幕添加"打印此页面"按钮，因为爷爷搞不懂 PDF。

## 如何使用

告诉代理：
```
"在 [URL] 上运行对抗性用户体验测试"
"做一个暴躁的 [角色类型] 测试 [应用名称]"
"在我的预发布网站上做一个刺头用户测试"
```

您可以提供角色或让代理根据您产品的目标受众生成一个。

## 步骤 1：定义角色

如果没有提供角色，通过回答以下问题生成一个：

1. **谁是此产品最难对付的用户？**（50 岁以上、非技术角色、数十年的"老办法"经验）
2. **他们的技术舒适度如何？**（越低越好 — 只用 WhatsApp、纸质笔记本、妻子帮他们设置了邮箱）
3. **他们需要完成的一件事是什么？**（他们的核心工作，而非您的功能列表）
4. **什么会让他们放弃？**（点击太多、行话、慢、令人困惑）
5. **他们沮丧时如何说话？**（直白、骂人、漠视、叹气）

### 好的角色示例
> **"大个子米克"·麦克阿里斯特** — 58 岁的体能教练。只用 WhatsApp 就这样。他的"电子表格"是一本纸质笔记本。"如果 10 秒内搞不懂我就回去用笔记本。"需要为 25 名球员记录训练结果。讨厌小字体、行话和密码。

### 糟糕的角色示例
> "一个不喜欢这个应用的用户" — 太模糊，无约束，无声音。

角色必须**足够具体**，以保持 20 分钟测试的角色一致性。

## 步骤 2：成为刺头（以角色身份浏览）

1. 阅读任何可用的项目文档以获取应用上下文和 URL
2. **完全沉浸于角色** — 他们的挫败感、局限性、目标
3. 使用浏览器工具导航到应用
4. **尝试角色真正的任务**（而非功能导览）：
   - 他们能完成他们来要做的事吗？
   - 完成它需要多少点击/屏幕？
   - 什么让他们困惑？
   - 什么让他们生气？
   - 他们在哪里迷路？
   - 什么会让他们放弃并回到老办法？
5. 测试这些摩擦类别：
   - **第一印象** — 他们会在登录页面之后继续吗？
   - **核心工作流** — 他们最常需要做的一件事
   - **错误恢复** — 当他们做错事时会发生什么？
   - **可读性** — 字体大小、对比度、信息密度
   - **速度** — 是否比他们的当前方法更快？
   - **术语** — 有他们不理解的行话吗？
   - **导航** — 他们能找到回去的路吗？他们知道自己在哪里吗？
6. 为每个痛点截图
7. 在每个页面上检查浏览器控制台的 JS 错误

## 步骤 3：吐槽（以角色身份写反馈）

以角色的身份写反馈 — 用他们的声音、带着他们的挫败感。这不是错误报告。这是一个真实的人在发泄。

```
[角色名称] 对 [产品] 的评价

总体：他们会继续使用吗？是的/不是/有条件地是

好的方面（勉强承认）：
- [即使是也不得不承认有效的东西]

坏的方面（合法的用户体验问题）：
- [会阻止他们使用产品的真正问题]

丑陋的方面（阻碍者）：
- [会让他们立即卸载/取消的事情]

具体投诉：
1. [页面/功能]："[以角色声音引用]" — [发生了什么，期望什么]
2. ...

结论："[一句话角色引用总结他们的体验]"
```

## 步骤 4：务实过滤器（关键 — 不要跳过）

走出角色。以产品人员的身份评估每个投诉：

- **红色：真正的用户体验错误** — 任何用户都会有这个问题，而非仅仅是刺头。修复它。
- **黄色：有效但低优先级** — 真正的问题，但仅适用于极端用户。记录它。
- **白色：角色噪音** — "我讨厌电脑"的言论，不是产品问题。跳过它。
- **绿色：功能请求** — 隐藏在投诉中的好主意。考虑它。

### 过滤标准
1. 一个 35 岁的、能干但忙碌的用户会有同样的投诉吗？→ 红色
2. 这是真正的可访问性问题（字体大小、对比度、点击目标）吗？→ 红色
3. 这是"我希望它像纸一样工作"对数字化的抵触吗？→ 白色
4. 这是角色偶然发现的真正工作流低效吗？→ 黄色或红色
5. 修复这会为 80% 没问题的人增加复杂性吗？→ 白色
6. 投诉揭示了缺失的入职引导时刻吗？→ 绿色

**此过滤器是强制性的。** 永远不要将原始角色投诉作为工单交付。

## 步骤 5：创建工单

仅针对**红色**和**绿色**项目：
- 清晰、可操作的标题
- 包含角色的原话引用（有趣且令人难忘）
- 下面的真正用户体验问题（客观）
- 建议的修复（可操作）
- 标签：`ux-review`

对于**黄色**项目：一个综合工单包含所有笔记。

**白色**项目仅出现在报告中。不创建工单。

**每个会话最多 10 个工单** — 专注于最严重的问题。

## 步骤 6：报告

交付：
1. 角色吐槽（第 3 步）— 有趣且生动
2. 过滤后的评估（第 4 步）— 务实且可操作
3. 创建的工单（第 5 步）— 带有链接
4. 关键问题的截图

## 提示

- **每个会话一个角色。** 不要混合视角。
- **在第 2-3 步保持角色。** 仅在第 4 步走出角色。
- **首先测试核心工作流。** 不要被设置页面分散注意力。
- **空状态是金矿。** 新用户体验揭示最多的摩擦。
- **最好的发现是角色在做其他事情时偶然发现的红色项目。**
- **如果角色零投诉，您的角色太懂技术了。** 让他们的年龄更大、更没耐心、更固执。
- **在演示、发布后或发布一批功能后运行此测试。**
- **尽可能注册为新用户。** 不要使用预填充的管理员账户 — 冷启动体验是大多数摩擦所在。
- **零白色项目是信号，不是失败。** 如果务实过滤器没有发现噪音，您的产品有真正的用户体验问题，而非仅仅是一个刺头角色。
- **在测试后检查项目文档中的已知问题。** 如果角色发现的问题已在已知问题列表中，那实际上是最有力的发现 — 这意味着团队知道但从未感受到用户的痛苦。
- **订阅/付费墙测试很关键。** 使用过期账户而非仅活跃账户进行测试。"当您无法付款时会发生什么"体验揭示产品是尊重用户还是扣留他们的数据。
- **计算角色完成一件事所需的点击次数。** 如果超过 5 次，无论角色技术水平如何，这几乎总是红色发现。

## 按行业的角色示例

这些是起点 — 根据您的特定产品进行定制：

| 产品类型 | 角色 | 年龄 | 关键特征 |
|-------------|---------|-----|-----------|
| CRM | 养老院院长 | 68 | 档案柜是当前的 CRM |
| 摄影 SaaS | 农村婚礼摄影师 | 62 | 通过电话预约客户，纸质开发票 |
| AI/ML 工具 | 百货公司采购员 | 55 | 被 3 家失败的科技初创公司烧伤 |
| 健身应用 | 老派健身房教练 | 58 | 纸质笔记本、粗手指、视力差 |
| 会计 | 家庭面包店老板 | 64 | 一鞋盒收据，讨厌订阅 |
| 电商 | 市场摊位 vendor | 60 | 现金为王，智能手机只用来打电话 |
| 医疗 | 资深全科医生 | 63 | 口述笔记，护士处理电脑 |
| 教育 | 老牌教师 | 57 | 粉笔加讲解，活页夹中的工作表 |

## 规则

- 在第 2-3 步保持角色
- 真正刻薄但公平 — 发现真正的问题，而非捏造的问题
- 务实过滤器（第 4 步）是**强制性的**
- 每个投诉需要截图
- 每个会话最多 10 个工单
- 在预发布/已部署的应用上测试，而非本地开发
- 一个角色，一个会话，一份报告
